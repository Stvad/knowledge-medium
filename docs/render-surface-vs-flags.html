Render surface vs. context flag bag

Decision aid for the embed/backlinks rendering bug — 2026-05-14

The bug, restated

When a block embed targets the panel's focal block (id collision), the embed inherits the "top-level" designation and renders backlinks/breadcrumbs/force-open/no-bullet. Symptom is the backlinks list, but the same id-equality check drives four other affordances.

Three axes are currently tangled together. Any clean fix needs to name them separately.
  1. Layout dispatch — "should the renderer pick TopLevelRenderer, PanelRenderer, or DefaultBlockRenderer here?"
  2. Focal-block identity — "is this the block the panel is currently zoomed to?"
  3. Render surface / mount role — "am I being mounted as the document body, as an embed, as a backlink entry, as a breadcrumb segment?"

The conflation you flagged: the name topLevel is overloaded between axis 1 (layout dispatch) and the intuition of axis 2 (focal block). They look like the same thing but aren't — BlockContext.topLevel is already false for the focal block by the time DefaultBlockRenderer runs (PanelRenderer flips it before recursing). The flag is really "this mount point is a layout-dispatch boundary," nothing about focal-ness.

All five collapse to the same conceptual check: "this is the focal block, rendered as the document body." They differ only in what affordance they gate.

Option A — Add another boolean (isEmbedded)

Smallest diff. Extend the existing flag bag; combine with the id check at every site.

// BlockContextType
interface BlockContextType {
  topLevel?: boolean        // layout dispatch (unchanged, still confusingly named)
  panelId?: string
  isBacklink?: boolean
  isBreadcrumb?: boolean
  isEmbedded?: boolean      // NEW
}

// BlockEmbed.tsx
<NestedBlockContextProvider overrides={{isEmbedded: true}}>
  <BlockComponent blockId={blockId}/>
</NestedBlockContextProvider>

// Each of the 5 sites:
const isFocal = block.id === topLevelBlockId
              && !ctx.isEmbedded
              && !ctx.isBacklink
              && !ctx.isBreadcrumb

+ Minimal churn (one new flag).
− Doesn't fix the topLevel naming conflation.
− Every consumer has to remember the full conjunction !isEmbedded && !isBacklink && !isBreadcrumb. Each new mount role adds a term everywhere.
− Models mutually-exclusive states as orthogonal booleans (a render can't be both a backlink AND an embed at the same nesting level — the booleans pretend it could).

Option B — Single renderSurface enum

Replace the surface booleans with one tagged field. Default 'document'.

type BlockRenderSurface = 'document' | 'embed' | 'backlink' | 'breadcrumb'

interface BlockContextType {
  topLevel?: boolean        // layout dispatch (still confusingly named)
  panelId?: string
  renderSurface?: BlockRenderSurface  // default 'document'
}

// BlockEmbed.tsx
<NestedBlockContextProvider overrides={{renderSurface: 'embed'}}>...</>

// BacklinkEntry.tsx
<NestedBlockContextProvider overrides={{renderSurface: 'backlink'}}>...</>

// useIsFocalRender hook centralizes the policy:
const useIsFocalRender = (block: Block) => {
  const [topLevelBlockId] = useUIStateProperty(topLevelBlockIdProp)
  const {renderSurface = 'document'} = useBlockContext()
  return block.id === topLevelBlockId && renderSurface === 'document'
}

// All 5 sites:
const isFocal = useIsFocalRender(block)

+ One axis, one field. New surfaces (preview pane, sidebar peek) cost a string, not a flag-conjunction-update.
+ Mutually-exclusive states modeled as mutually exclusive.
+ Aligns with the direction in renderer-resolution.md (explicit mount intent).
+ Centralizing in useIsFocalRender means the rule lives in one place.
○ Migration touches the two existing surface-flag setters + adds isEmbedded-equivalent. Still small.
− Doesn't fix the topLevel naming conflation either.
− Nesting question: backlink containing an embed — surface is 'embed' or 'backlink'? Innermost-wins is natural React-context semantics and fine if no consumer cares about specific outer surface (current code only cares about truthiness).

Option C — Option B + rename topLevel

Same as B, plus fix the axis-1 naming. topLevel becomes layoutBoundary (or similar): a flag whose only job is gating which top-level renderer fires. Now the three axes have three names with no overlap.

interface BlockContextType {
  layoutBoundary?: boolean             // was: topLevel
  panelId?: string
  renderSurface?: BlockRenderSurface   // default 'document'
}

// TopLevelRenderer.canRender:
({context}) => !!(context?.layoutBoundary && !context.panelId)

+ Resolves the conflation directly. After this, "top-level" in code means exactly one thing in each place: id === topLevelBlockId (focal) vs. layoutBoundary (dispatch).
+ Future readers don't have to learn the gotcha that context.topLevel isn't what they think.
○ Rename touches ~6 setter/reader sites. Mechanical.
− Bundles two concerns in one PR (surface refactor + rename). Could be split.

Option D — Flag bag + derived hook helpers (with umbrella flag)

Keep the bag of context flags open and additive, but move the policy ("what counts as document- surface?") into hooks so consumers ask the question, not the implementation. Each non- document mount also sets an umbrella isNestedSurface flag, so the hook has nothing to forget when new surfaces are added.

interface BlockContextType {
  topLevel?: boolean         // layout dispatch (still mis-named — orthogonal)
  panelId?: string
  isNestedSurface?: boolean  // umbrella: set by every non-document mount
  isEmbedded?: boolean       // specific descriptors, for consumers that discriminate
  isBacklink?: boolean
  isBreadcrumb?: boolean
}

// BlockEmbed.tsx
<NestedBlockContextProvider overrides={{isNestedSurface: true, isEmbedded: true}}>...</>

// BacklinkEntry.tsx
<NestedBlockContextProvider overrides={{isNestedSurface: true, isBacklink: true}}>...</>

// Policy lives in one hook:
const useIsFocalRender = (block: Block) => {
  const [topLevelBlockId] = useUIStateProperty(topLevelBlockIdProp)
  const ctx = useBlockContext()
  return block.id === topLevelBlockId && !ctx.isNestedSurface
}

// All 5 sites:
const isFocal = useIsFocalRender(block)

+ Open surface set — new mount role = new flag, no closed union to amend.
+ Composes in nested cases — backlink containing embed has BOTH specific flags true; either consumer asks its specific question correctly. Enum forces innermost-wins and loses info.
+ Consumer code reads as the question (useIsFocalRender), not the implementation. Policy is in one hook; all 5 sites inherit changes automatically.
+ Umbrella flag (isNestedSurface) eliminates the disjunction footgun of a naive flag bag — adding a new surface only requires setting the umbrella, the gate hook needs no update.
○ Two flags per setter (umbrella + specific). Mild redundancy, but each carries different info.
− Discoverability slightly worse than enum: "what surfaces exist?" means grepping setters rather than reading a union type.
− Doesn't fix the topLevel conflation either (compatible with the rename from C if desired).

Updated read: D wins on the merits. The enum's main "feature" (mutual exclusivity) turns into a bug in nested cases like a backlink-containing-embed, where you actually want both descriptors true; D's umbrella+specific flags compose naturally. D's hook layer gives consumers the same ergonomics as B (one call, no conjunction visible at the site), but the surface set stays open and adding new ones doesn't require type amendments or hook updates.

If the topLevel naming conflation also bothers you, D is compatible with C's rename — they're orthogonal changes.

Open questions before committing

  • Nesting policy for surface: innermost-wins (simple, matches context semantics) vs. a stack (preserves outer context). Innermost-wins works iff no future consumer needs to ask "am I anywhere inside a backlink, even if also inside an embed?"
  • Do we want 'document' to be explicit or implicit? Explicit (default empty, sites check === 'document') reads cleaner; implicit (!surface means document) is one less field at root. Lean explicit.
  • Should useIsFocalRender live with the other render hooks or with the context module? Doesn't matter much; co-locating with context module keeps the policy next to the type it depends on.
