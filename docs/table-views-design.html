Table Views

Tana-style query tables over the type system — a proposed tables plugin: editable table views fed by a text query language compiling onto TypedBlockQuery.

Status: proposed design — no implementation.
Last verified against code: 2026-07-12 (master @ 44d8631d; every engine/UI claim below was checked against source at that revision, with file:symbol pointers inline). Written for the properties-as-blocks world (PR #288, which lands first); the seams chosen here (cell-projection reads, setProperty writes) are correct on both sides of that migration regardless.

Decisions proposed in this doc (each argued in its section):

  • Ship as a core plugin (src/plugins/tables/): a table block type plus table:view child blocks, real npm dep on TanStack Table (headless), reusing the property panel's ValuePreset editors for cells (§4.2, §4.6). No bespoke embed type — a table is a block; embedding a table (or one specific view) is the generic ((id)) block-embed affordance, the media model (§4.4).
  • One table substrate, two row sources: a query, or a block's own children ("view any node as a table"). Children-as-rows is a query term (childof=self), not a second mechanism — the compiler routes it to childrenOf (§4.1).
  • The query is text, and text is the only stored artifact — no shadow copy of resolved ids. Names resolve against the schema registry at parse time; a name^fieldId exact form exists for stability-critical terms; schema renames rewrite query texts as part of the rename action; unresolvable names render as visible term errors, never silent empties (§4.3).
  • Compile to TypedBlockQuery where the engine can express a term; evaluate the residue in JS over the pre-filtered set, with client-side terms marked in the UI (§4.3, §4.7).
  • Columns are property schemas keyed by stable definition-block id, never display name. Schema unification (docs/schema-unification.html, accepted; Slice A landed) makes every schema — kernel and plugin included — a definition block, so one header affordance covers all columns (§4.2).
  • Cells read the cell projection and write through tx.setProperty — correct pre- and post-#288. Commit on Enter/blur (never per keystroke); sort/group re-evaluation is held while a cell editor is open; a new row stays pinned until commit, with query-condition back-fill (§4.5).
  • Render cap + "show more" first; row virtualization deferred until it is designed together with the focus model it constrains (§4.6).
  • View config is layered: durable view definitions live as props on table:view child blocks, plus a UiState-scoped tweak layer; casual interactions write the tweak layer, an explicit "save view" promotes. Both layers sync — a true per-user layer is future work, stated honestly (§4.4).

1. Why: the type system has no collection surface

KM already has the Tana-shaped half of a database feature: additive multi-typing (types on any block), user-defined types as blocks (block-type), property schemas as blocks (property-schema) with codecs and ValuePreset editors, and a SQL-compiled typed query engine. What it lacks is the surface that makes a type system legible: nowhere to see all #book rows with their fields as columns, sort by a property, edit a value without opening each block's property panel.

The design deliberately follows Tana's model rather than Notion's, because KM's data model already made that choice. Notion bundles schema and views on a database container; a page has properties only by living inside one. Tana (and KM) attach schema to the type — instances live anywhere in the outline and are gathered by search. The consequence worth stating up front: collection views are node-anchored, not type-anchored — a table hangs off whichever block displays it, exactly like Tana's view config and unlike Notion's database views.

2. Ground truth #1: the properties-as-blocks world (PR #288)

The migration plan makes properties field-row children ([[Schema Name]] content, denormalized reference_target_id, values as children). Four of its guarantees are the seams this design builds on:

  • properties_json survives as a derived projection — block.get(schema) and json_extract queries stay correct and synchronous. A table reads the cell; it never walks field-row children on the read path.
  • tx.setProperty dual-writes children + cell in one tx. A cell editor that writes through setProperty is correct in both worlds, unchanged.
  • tx.childrenOf excludes field rows by default — children-as-rows tables see only content children, for free.
  • Field identity is fieldId — the schema's definition block id (uniformly, post-unification; #288's original property:<name> class for kernel/plugin schemas is superseded by the seeded definition blocks — §2 below); renames touch name only.

Two caveats the plan itself carries, inherited here rather than dropped:

  • Stale bootstrap-hint window: on a freshly-synced device the cell can be per-key stale until the projection pass sweeps — a table there can transiently show wrong cell values and wrong row membership. Bounded and self-healing, but an editing surface should not treat cold-device query results as authoritative (#288 §5, §12).
  • Plugin schemas default to cell-backed until their durable opt-in — so field-row-derived affordances apply to user schemas first, not to e.g. a todo-status column.

(E2EE is not a caveat here: encryption seals content columns at the upload boundary — services/powersync.ts: encryptUploadOps — so cell reads, setProperty writes, and compiled SQL all operate on local plaintext in any workspace mode. The #288 e2ee gate keeps such workspaces cell-only, whose only consequence for this plugin is that the §4.2 schema-block-backlinks nicety doesn't accrue there.)

Column identity is the definition block id, uniformly. Schema unification (docs/schema-unification.html — accepted; Slice A landed) dissolves the kernel/plugin-vs-user split this doc originally special-cased: every schema resolves to a definition block, seeded ones at deterministic per-workspace ids (uuidv5(workspaceId:seedKey); handles carry seedKey and resolve ids through the registry primitive, never cache them). #288's fieldId is that same id, and #288 lands before this plugin — columns simply consume the resolution primitive; there is no synthetic property:<name> class to special-case. Unification also keeps all schema renames blocked until #288's slice B2 — the same machinery §4.3's query-text rewrite pass rides.

3. Ground truth #2: the query engine, exactly as it is

3.1 What TypedBlockQuery gives today

Verified against src/data/api/typedBlockQuery.ts and src/data/internals/typedBlockQuery.ts:

  • types (contains-any), where over codec-encoded properties — eq / lt / lte / gt / gte / between / exists, NULL-safe excludes (IS NOT TRUE), JSON paths inlined so expression indexes match.
  • where.<field>.target traverses a ref (equality join) or refList (json_each fan-out) into inner predicates on the referenced block (compileTargetTraversal).
  • referencedBy: {id, sourceField?} on a predicate compiles to EXISTS(… block_references br WHERE br.source_id = <candidate>.id AND br.target_id = ?) (compileReferencedByExists) — i.e. "the candidate references X (optionally via field F)". This is both ref-equality and refList-contains, already production-used (typeExtraction.ts: findCandidatesByPropertyShape). Ref codecs are deliberately not where-comparable; the compiler's own error message routes to referencedBy. Caveat: block_references is a projection maintained for schemas registered as ref/refList at write time — values written while a defining plugin was disabled aren't in it.
  • match (AND) / exclude (NOR) predicate lists with self / ancestor scope; the ancestor walk is a recursive CTE that is both depth-bounded and gated: assertAncestorWalkBounded rejects the query unless a candidate-bounding filter exists (types / referencedBy / selective self-where). A bare "everything under X" is not compilable — the grammar must account for this (§4.3).
  • Reactivity: per-dimension deps (type channel, per-property channel, reference channels) for membership, plus per-row deps from hydration — one subscribed handle re-fires on any change to any matched row, including properties and content not mentioned in the query. Honesty note: it also over-fires — property channels are keyed by (workspace, property name), so any block writing that property re-resolves the view, and any member-row edit re-runs the query + full rehydrate. Direction is right (no under-fire; a table's cells stay live through one subscription); cost is budgeted in §4.7.
  • Surfaced as repo.queryBlocks / subscribeBlocks (+ active-workspace shorthands); hydrated rows include content.
  • Free-text is SQL-able too: blocks_fts is a trigger-maintained trigram FTS5 index over live blocks.content (clientSchema.ts) — true case-insensitive substring semantics on an index. A content term compiles as an FTS join intersected with the typed candidates (exhaustive — core.searchByContent's top-N limit is that query's choice, not the index's), and the narrow KERNEL_CONTENT_CHANNEL dep (searchByContent's own pattern) provides exactly the invalidation an SQL-pushed content term needs: it fires when a non-member starts matching via a content edit, which per-row deps alone would miss. Residue: sub-trigram-length terms (< 3 chars, isTrigramSearchable) fall back to the JS tail.

4. Design

4.1 One substrate, two row sources

A table block renders a query's results as rows. "View any node's children as a table" — the pure outliner↔table flip — is the same substrate with a one-term query: childof=self. The compiler routes children-terms to childrenOf (the engine has no direct-children predicate) and everything else to subscribeBlocks; the view definition, the embed mechanism, and the table component see one shape either way. Rows are always live references to real blocks — an editing view, not a report.

Embedding follows the media model: a table is a block, so pinning it elsewhere is the generic ((id)) block embed — of the table block (its default/active view) or of one table:view child block (that view, pinned). No bespoke embed type (§4.4).

4.2 Column model

  • A column = a property schema, identified by its definition block id (§2; resolved through the unified registry — handles carry seedKey, ids are per-workspace) — never by display name. The leftmost column is the block itself (title/content; navigates on click).
  • Default column set: for a type-backed query (#book), the type's declared properties — the same set the outline will render as ghost slots post-#288; union-of-present-fields for untyped queries and mixed children.
  • Every column header links to the schema's definition block — post-unification, kernel/plugin schemas have seeded definition blocks too (read-only metadata mirrors in v1), so there is no popover carve-out. "Where is this field used" backlinks accrue on the definition block post-#288, for child-backed schemas.
  • Add column = pick an existing schema (typeahead over the registry + schema blocks) or create one (reuse AddPropertyForm / the ValuePreset picker). Adding a column does not touch the type; when the query is type-backed, offer — never auto-apply — "add to #type's template", Tana's exact split.
  • Cell display/editing resolves by codec shape through the existing property-panel machinery (resolvePropertyDisplay, ValuePreset editors) — a table cell editing property P on row B is semantically the property panel's row for (B, P). Ref cells render as chips that navigate.

4.3 The query language — text-first, agent-authorable

Requirements: a readable one-liner; writable by coding agents and by the UI (the UI writes text — text is canonical); compiles onto TypedBlockQuery; degrades to the JS tail for the residue, honestly marked.

The grammar has three term shapes plus grouping — small enough that an outline-style query builder can round-trip it later:

Terms AND by juxtaposition — matching the engine's shape (match=AND, exclude=NOR, types=OR) — so the compiler's job is a clean split: every SQL-marked term joins the TypedBlockQuery; or-groups, pseudo-fields, and any term the engine rejects join the JS tail, and the UI marks tail terms ("evaluated client-side"). The or-group syntax exists precisely so boolean trees have a surface, not just an escape hatch — field evidence (§6) shows real corpora use them.

Name resolution & renames — the one rule. The text is the sole stored artifact. Names resolve against the schema registry at parse time (which is cheap and reactive). There is no shadow copy of resolved ids — a second storage location would need a sync invariant and a which-wins rule, and hand edits (CLI, agents) would bypass it. Instead:

  • A name^fieldId exact form is part of the grammar for stability-critical terms (Tana Paste's ^nodeID:: precedent). The UI's query builder emits it when two registered schemas share a display name.
  • Renaming a schema rewrites query texts: the rename action (today propertyPanel/actions.ts: renameProperty; post-#288, slice B2's rename machinery) gains a pass over table blocks rewriting the renamed term. Same class of work as #288's rename-reproject, and it composes with it.
  • An unresolvable name is a visible term error on the table (that term shows as errored; the rest of the query still runs) — never a silent empty result and never a silent fallback to a different schema.

4.4 View definition storage

  • Content = the query, in a ```table fence — an unambiguous boundary (the plain-text alternative leaves "is line 2 more query or prose?" undefined), edit-mode-friendly (edit = raw text, parse errors render as an error placeholder, never a crash), and greppable/authorable by agents.
  • Named views = table:view child blocks of the table block, each holding its view config in props (columns, sort, group, view type, column widths; the block's content is the view's name). Views thereby have block identity — addressable by ((id)) (which is how per-view embedding works, no bespoke embed type), referenceable, commentable — and the stable-id-not-name lesson comes free. The table block's own props hold only the default-view ref. View children are typed and excluded from row sources: a childof= row set skips table:view blocks, the same exclusion class as #288's field rows — otherwise a node viewed as a table would show its own view-config blocks as rows.
  • Tweak layer = UiState-scoped props: active view on the table block; unsaved sort/filter/column tweaks on the view block they tweak. Casual interactions (header-click sort) write this layer; an explicit "save view" promotes tweaks into the view block's durable props. Honesty required by the code: ChangeScope.UiState uploads and syncs — it is a second synced layer distinguished by undo-exclusion and convention, not a per-user store. In a single-user workspace that behaves exactly like Tana's personal layer; true per-user tweaks need a per-user ui-state home and are future work, not claimed now.
  • Children-as-rows mode works identically: "view as table" gives the viewed block its own table:view child (excluded from its row set, per above). The config is block-anchored: every pane showing that block sees the same table (per-pane divergence, if ever wanted, belongs to the tweak layer keyed by panel — out of scope).

4.5 Editing model

  • Write path: cells write setProperty(rowId, schema, value), one undo group per gesture. Never raw cell JSON, never field-row children — the dual-write keeps both worlds converged.
  • Commit cadence: Enter/blur, not keystrokes. A per-keystroke setProperty would be a per-keystroke transaction and a per-keystroke re-resolution of the view (§4.7). The editor holds local state until commit.
  • Sort/group hold while editing. Editing the sorted column would otherwise move the row under the open editor (the subscription re-fires on your own write; remounting an active editor mid-typing is a known bug class). While any cell editor is open, membership/sort changes are applied to the model but the open row's position is pinned; the view settles on commit.
  • New row lifecycle: the bottom placeholder creates the block, applies back-fill, and stays pinned (out of sort order, exempt from membership filtering) until the user commits the row — otherwise a row that doesn't yet satisfy the query vanishes mid-typing, and a sorted view teleports it away as soon as it matches. Back-fill taxonomy, explicit: type terms → add the type; = predicates → set the value; childof=/under= → choose the parent (direct child for childof; for under, create under that ancestor); refs=[[X]] → not back-filled; comparisons, negations, or-groups, content terms → not back-filled. If non-back-fillable positive terms remain unsatisfied at commit, the row keeps its pin with a "doesn't match this view's query" marker rather than disappearing.
  • Row expansion: an expanded row mounts the real nested block renderer (the row's sub-outline, editable) — the thing Tana tables can't do. Column-cell editing and expansion are independent.
  • Row-level actions reuse block actions (open in panel, etc.) rather than inventing table-specific ones.

4.6 Grid implementation

Survey of React table/spreadsheet libraries (2026-07-12; sizes from bundlephobia, licenses/peers from the repos): TanStack Table v8 — headless — is the fit. MIT, ~15 KB gz, React-19-native, actively maintained; sorting, grouping, row expansion, column sizing/order/pinning all exist as headless state and the DOM is entirely ours. That last property is the decisive one three times over: cells are ValuePreset editors and ref chips (real React components with app context), expanded rows mount the nested block renderer, and keyboard handling must compose with KM's shortcut-surface dispatch and class-based logical focus rather than fight them with a packaged grid's global handlers. Runner-up: adazzle's react-data-grid (15 KB gz, zero deps, MIT, React ^19.2, excellent built-in keyboard nav) if build-it-yourself proves too costly. Eliminated: AG Grid Community & MUI X (row grouping / master-detail paywalled), Glide Data Grid (canvas: React 18 peer cap, custom cells are hand-drawn, grouping half-shipped), Handsontable (non-commercial license), react-datasheet-grid (no sort/group/expand).

Cap + "show more" first; virtualization deferred — deliberately. KM's focus restoration and scroll-to-focused read the DOM; with windowed rows the focused block's row may not be mounted at all. Virtualizing in v1 would re-open that class of bug while keyboard nav doesn't even exist yet. v1 renders a capped page (grouping-aware) with "show more"; row virtualization (TanStack Virtual, ~7 KB) lands only together with the keyboard/focus work, designed as one unit (scroll-to-index before focus application).

Focus model: KM logical focus stays on the table block itself (the viewport); cell focus is table-internal, in-memory state. Table rows are not KM-focusable blocks in v1 — the alternative (every row a focusable block location) collides with panel-level focus arbitration and is exactly the seam virtualization breaks; revisit with evidence.

Other rendering rules, inherited from hard-won list-view experience downstream: never subscribe to a second block per row (ref-chip labels come from a shared lookup built once per view, cover images are IntersectionObserver-gated); keep per-row work O(1). Mobile: v1 tables are read-only on narrow viewports (horizontal-scroll container; cell editing is desktop-first) — a dense editable grid is the worst iOS surface to debug and shouldn't gate the rest.

4.7 Reactivity & cost budget

One subscribeBlocks handle per table. Per-row hydration deps mean out-of-band edits (another pane, another device) reach cells with no per-row subscriptions — verified plumbing, and the reason cells can be "live" cheaply. The costs to budget, not hide:

  • Any write to any matched row re-runs the query (SQL + full rehydrate + JS tail). At thousands of matched rows this is the dominant cost — mitigations, in order: the render cap bounds React work (the tail sort still runs over the full set); commit-on-Enter bounds self-inflicted re-runs; LIMIT/ORDER pushdown (§3.2) if profiling says the tail hurts.
  • Property channels over-fire (name-keyed per workspace) — a busy property on unrelated blocks re-resolves the view. Acceptable at current scales; the fix, if ever needed, is engine-side channel narrowing, not view-side caching.

5. Phasing

  1. Read-only substrate. Fenced query text → parser → TypedBlockQuery compiler + JS-tail evaluator; one subscription; TanStack table with cap+show-more; named views as table:view children + tweak layer; embedding via the generic block-embed affordance. (A tested JS evaluator for filter trees + sort/group/limit exists downstream — §6 — and ports directly.)
  2. Editing. ValuePreset cell editors + setProperty; commit/hold/pin semantics (§4.5); add-column flow with the "add to type template" suggestion; new-row back-fill; keyboard nav (cell-focus store + shortcut context).
  3. Children-as-rows. childof=self routing, "view as table" affordance, row expansion mounting the nested renderer.
  4. Depth. Grouping UI (kanban drag in a cards variant writes the grouped property), query-builder UI emitting the text grammar, engine pushdowns as warranted, virtualization + focus model as one unit, per-type default views only with evidence.

6. Field evidence: the downstream corpus

A downstream deployment (spolakh's workspace) has run a predecessor for a month: an Obsidian-Bases port ("bases v1") rendering read-only tables from .base YAML over a workspace-wide byType('page') scan with JS filtering. Its corpus is the best available evidence for what real queries need:

  • 116 live table definitions (124 at import). 18 of 116 use or: groups across predicates — or-groups are not a theoretical need; a grammar without an OR surface cannot express ~16% of a real corpus. (Hence §4.3's group syntax, and hence "JS tail" rather than "unsupported".)
  • Constructs in live use that shaped the grammar beyond types + properties: title-contains (KM's content term, §4.3), mtime/ctime comparisons (the timestamp pseudo-fields), string-list contains, and a path-prefix folder test (import-provenance-specific; maps to under= only where vault folders became ancestry — a per-corpus migration judgment, not a grammar rule).
  • Most of that corpus filters on unregistered (import-provenance) props, which compileWhereClause rejects by design — so migrated-corpus terms evaluate in the JS tail unless schemas are minted for them. The SQL-pushdown win applies fully to typed queries (#task status=open), which are the design center here; the honest statement for legacy corpora is "same evaluation as before, behind a real grammar".
  • Operational lessons already folded in above: render caps beat naive full renders (its 100-row cap); one shared query, never per-row subscriptions; edit mode = raw text with error placeholders is a fine authoring UX.

7. Open questions

  • OR pushdown: is the JS tail enough long-term, or should match gain anyOf groups? Proposal: ship with the tail, measure; the grammar doesn't change either way.
  • Per-user tweak layer: worth a real per-user ui-state home for multi-user workspaces, or does the synced tweak layer suffice until multi-user view usage is real?
  • Ghost-slot alignment: default columns for #type should equal the outline's post-#288 ghost-slot set — align the two default computations when that registry shape lands (slice B) rather than guessing now.
  • Timestamp pushdown: created/updated terms are JS-tail v1; a where-on-columns engine extension would make several common queries fully SQL. Worth it only with profiling evidence.
