Shortcut context precedence

Options & decision aid for cross-context shortcut collisions — 2026-05-30

Companion to docs/shortcut-conflict-resolution.md (2026-05-25), which introduced the modal flag (now shipped). This doc revisits the area with fresh eyes for a new class of report and proposes a precedence model. Where the older doc still holds, it says so; where it constrains the design wrongly, it says that too.

The report

A user built a new display surface for blocks. The surface registers its own action context with a handful of bindings. When you click a block inside that surface, the block gains focus and normal-mode activates on top of the surface context. Both contexts are now active, and any chord they share fires both handlers. Their current workaround is a pile of imperative "skip" decorators that no-op the host action — one per colliding action.

Two things the user wants that today's primitives don't give:

  1. Additive coexistence for the keys the surface doesn't care about (modal is too broad — it shadows everything).
  2. The surface to win the chords it does bind, even though normal-mode activated after it.

Root cause: the double-fire is incidental, not designed

HotkeyReconciler installs one window listener per active action (HotkeyReconciler.tsx:241–264). On a shared chord, both listeners match and both run; preventDefault() stops the browser default and other listeners' propagation in some cases, but it does not stop sibling listeners registered separately on the same target/phase. So both handlers fire. Nothing in the codebase intends two actions on one chord — if you want two effects you write one action. The double-fire is a side effect of the per-listener install model, not a feature. Treat it as a latent bug to close, not a behavior to preserve.

We already resolve to a single winner — just not on the keyboard path. getActiveActionById (effectiveActions.ts:77–93), used by runActionById (command palette, programmatic dispatch), walks active contexts in reverse activation order and returns the first match. So "one winner, latest-activated context wins" is already the law for one dispatch path. The keyboard path is the inconsistent one. Any fix should make the two paths agree.
But the double-fire accidentally provides one real capability: conditional fall-through. Because both handlers run today, a context can bind a chord, check a condition in its handler, and no-op — and the lower context's handler still ran. The event "fell through" by accident. A naïve single-winner fix removes this: if only the winner installs the chord and its handler bails, nothing else happens — the lower layer never had a listener. Whether that capability is worth keeping is the hinge between the two implementation strategies (activation-time vs fire-time resolution) and gets its own option below — see Option D. It is not a footnote; it changes the recommendation.

What to steal from the old doc

  • Keep modal. Precedence resolves collisions; it does not cover the palette case (palette binds no chords but still must stop normal-mode's p reaching the search box). modal = "own all chords, even unbound." The two are orthogonal tiers and should coexist.
  • The global carve-out is load-bearing. Whatever wins, app-wide chords (Cmd+K, Escape) must survive. Keep global always-installed.
  • Resolve at the context level, not per binding. The old doc rejected binding-level priority: number (Model C) as non-local and full of magic-number cliffs. That critique still stands — and it's specifically about per-binding numbers. Context-level precedence (one number per context, ~9 of them) dodges it.
  • Teach the conflict UI about intentional shadowing. findKeybindingConflicts (keybindingConflicts.ts) will flag a deliberately-shadowed chord as a collision. Whatever we ship, the UI should render "shadowed by X while active," not "conflict."

Where the old doc constrains wrongly: it framed the choice as modal-vs-additive and scoped selective precedence out as a "known limit." The report is that limit. We're picking it back up.

The shared decision underneath every option

Whatever we choose, the bug fix is the same: stop installing both colliding bindings; install a single winner per chord. The reconcile effect already re-runs on every active change and already installs/uninstalls per action (HotkeyReconciler.tsx:194–271), so this is a dedup step inside the loop that already runs — not a new coordinator. The options below differ only in how the winner is chosen.

Options

Option 0 — Last-activated-context wins (no new config)

Make the keyboard path obey the same rule the dispatch path already does: on a chord collision among active contexts, the latest-activated context's binding installs; the rest are skipped. Pure reuse of the existing comparator.

  • Closes the double-fire with zero new surface area; unifies keyboard and runActionById paths under one rule.
  • Most collisions today are "stacked modes" where latest is the right winner.
  • Gets the report's case wrong. Surface activates early, normal-mode late → the generic layer wins the collision. Exactly the inversion the user is fighting.
  • Precedence = timing, which is incidental when two contexts activate off the same focus event. Non-local: you can't read a context and know if it'll win.

Verdict: necessary foundation, insufficient alone. Ship it as the default, but it needs an escape hatch for the inversion.

Option A — Context-level priority

Add one optional field to ActionContextConfig; default 0. Winner = highest priority, ties broken by activation recency (Option 0 as the tiebreak).

actionContextsFacet.of({
  type: SURFACE_CONTEXT,
  priority: 100,          // > normal-mode's default (0)
  // ...
})
  • Fixes the inversion directly. The surface declares it outranks the baseline; click timing no longer decides. Additive for every chord it doesn't bind.
  • One number per context (~9 total), most at default. This is the Emacs major/minor-mode model — precedence by layer specificity, which is what the user means by "normal mode shouldn't win."
  • Ownership sits in the right place: the specific layer declares its rank; normal-mode stays dumb and knows nothing about the surface.
  • Supersedes the per-decorator workaround with a single line.
  • Resolves per exact chord. The external agenda's d would not shadow delete_block's d d (different chords). For in-repo cases this is fine — no same-key precedence gap exists today; flag if one appears.
  • Absolute numbers need a known baseline (0). Mitigate with a tiny named tier set (high/default/low) instead of bare integers if desired.
  • A high-priority context silently shadows lower bindings on shared chords — powerful, needs the conflict-UI surfacing noted above.

Option B — Explicit supersession (declared suppression)

A context lists the actions/contexts it suppresses while active. (This is the user's sketch, and the old doc's Model B.)

actionContextsFacet.of({
  type: SURFACE_CONTEXT,
  supersedes: {
    [ActionContextTypes.NORMAL_MODE]: {
      actionIds: ['delete_block', 'toggle_properties', /* ... */],  // or '*'
    },
  },
})
  • Most explicit — the conflict is documented at the site that introduces it.
  • Targets action ids, so it suppresses regardless of binding shape — handles the d d sequence case that exact-chord priority misses.
  • Suppression, not precedence. It only turns things off. For a key both contexts want where the surface should simply win, you still enumerate host actions — drifting back toward A with more typing.
  • Name-coupled to host action ids; brittle if normal-mode renames or adds an action (new colliding action fires until you add it to the list — silent).
  • Verbose (the report needs ~8 entries). The '*' wildcard tames verbosity but then it's modal with extra steps.

Option C — Selective per-chord shadowing (shadowChordsFrom)

Middle ground: a context shadows listed contexts only for chords it itself binds.

shadowChordsFrom: [ActionContextTypes.NORMAL_MODE]
  • Additive by default, surgical on conflict; no per-action bookkeeping.
  • Still name-coupled to the suppressed context.
  • It's just priority with a target list. "I beat normal-mode specifically" is the coupled special case of "I beat anything lower." A is the decoupled version. No reason to ship both.

Option D — Declinable resolution (fall-through)

The previous options answer "who wins a chord." This one answers a different, partly orthogonal question: "when does the winner get to say not me — let the next layer have it?" It's the one capability today's accidental double-fire actually provides (a higher context's handler can no-op and the lower one still ran), and a naïve single-winner fix silently deletes it.

The model: dispatch consults active contexts in precedence order (whatever 0/A picks). The first that binds the chord runs — but its handler may report "not handled," and dispatch then tries the next candidate in order. No double-fire (at most one handler ultimately acts), but conditional yielding works. This is the dispatch-gating sibling of canRun: a predicate that, when it declines, defers instead of suppressing.

// handler returns false / "not handled" => dispatch falls through to next candidate
handler: (deps, event) => {
  if (!surfaceWantsThisKeyRightNow(deps)) return false   // let normal-mode have it
  doSurfaceThing(deps)
  return true
}

Precedent: Emacs (a command keymap entry of nil / a command that signals it didn't handle falls through to the next keymap), CodeMirror (a keymap run returning false tries the next handler). This is the standard layered-keymap escape valve.

  • Preserves the only genuine capability the double-fire gave us, but cleanly (one effect, not two).
  • Composes with A — precedence picks the order; decline walks down that order. They're different axes.
  • Lets a high-priority surface own a key conditionally without re-implementing the lower layer's behavior for the cases it wants to pass through.
  • Forces fire-time resolution. The winner is now per-event (depends on what handlers return), so you can't pre-resolve at activation time by skipping installs. That means a real dispatch coordinator — a single keydown listener that owns chord → ordered-candidates → run-until-handled — replacing today's independent per-action listeners. Bigger change than the activation-time dedup.
  • Handler contract grows a return value (boolean / "handled"). Existing handlers return void | Promise<void>; "no return = handled" keeps them working, but async handlers that want to decline must resolve to the sentinel, and "did it handle?" for an async handler is awkward (you'd likely restrict decline to a synchronous pre-check).
  • Only worth it if surfaces actually need to conditionally yield a key they bind. If every case is "this layer wins these keys, full stop," D is unnecessary machinery and activation-time wins on simplicity.
Two axes, not five rivals. Winner-selection (0 / A / B / C) and resolution-strategy (activation-time vs fire-time/declinable, i.e. whether D is in) are independent. The real choice is a 2×2: {last-wins | priority} × {activation-time | fire-time-declinable}. modal sits above both as the "own everything, even unbound" tier and is unaffected.

Recommendation hinges on one question: do surfaces need to conditionally yield a chord they bind?

Common to both answers: fix the double-fire (single winner per chord), winner = shared comparator (priority desc, then activation-recency desc) used by both the keyboard path and getActiveActionById, add priority?: number (default 0) so the reporting surface outranks normal-mode and the 8 skip-decorators collapse to one line, and keep modal as the extreme tier. Drop B (suppression, not precedence; name-coupled) and C (A with coupling).

If NO — every case is "this layer wins these keys, full stop": ship A + activation-time resolution (dedup in the reconcile loop that already runs). Smallest change; no handler-contract churn. This is the default recommendation absent evidence otherwise.

If YES — a surface wants to own a key sometimes and pass it through otherwise: ship A + D (fire-time, declinable). Costs a real dispatch coordinator and a handler return value, but it's the clean version of the capability the double-fire gives us by accident, and retrofitting it later means redoing the dispatch path. This is the open decision for the reader — see the question at the top of Open questions.

Skip-decorators were the wrong layer regardless: imperative no-ops fighting the architecture, and canRun can't gate dispatch anyway. Note that Option D is the principled replacement for exactly what the skip-decorators were faking — "run my handler, decide it doesn't apply, let the host act" is precisely conditional fall-through, done in dispatch instead of by mutating the action catalog. If the 8 decorators are unconditional no-ops (always suppress host), A alone replaces them; if any are conditional (suppress host only sometimes), that's the in-repo evidence that D is needed. Worth auditing those decorators before deciding. Separately, for a genuine static "remove this action everywhere" need, the clean tool is letting ActionDecorator.decorate return null (parity with ActionOverride.apply).

Open questions / risks

  • THE decision: is conditional fall-through (Option D) needed? Everything else is settled; this isn't. Decider: audit the 8 skip-decorators on the user's surface. If they always no-op the host action when the surface is active → unconditional → A alone suffices, ship activation-time. If any no-op the host only under a condition (and otherwise want the host to run) → that's conditional fall-through in disguise → ship D (fire-time). Picking activation-time and discovering a conditional case later is a dispatch-path rewrite, so decide this deliberately, not by default.
  • The eventFilter invariant is load-bearing for activation-time. Pre-resolution is correct only because no context can decline-and-defer a chord today. If we ship activation-time and someone later writes a filter (or handler) meant to scope a chord down and hand it to a lower layer, pre-resolution breaks (the loser is already uninstalled). That's the same wall as the D question — Option D is the supported way to do declinable handling. Document the invariant next to the dedup so a future reader knows shipping declinable handling means moving to fire-time, not bolting onto the filter.
  • Chord canonicalization. Grouping by chord needs the same normalization keybindingConflicts.ts already does (alternatives like ['$mod+z','ctrl+z'], code-form chords). Reuse it; don't reinvent. (Not unique to this approach — any single-winner scheme needs it.)
  • Equal-priority fragility persists. When two same-priority contexts activate off one focus event, the tiebreak is incidental timing. Acceptable for stacked modes; if a real case wants determinism, that's the moment to reconsider — not now.
  • Conflict-UI follow-up. Extend findKeybindingConflicts to mark priority-shadowed chords as intentional rather than warnings. Can ship after the core change.
  • Granularity escape hatch. If an in-repo same-key-vs-sequence case ever needs B's action-targeted suppression, add it then as a narrow supplement — don't pre-build it.

Rough surface area

Shared by both strategies (A core):

  • types.ts — add priority?: number to ActionContextConfig (+ doc comment).
  • effectiveActions.ts — extract compareContexts; rewrite getActiveActionById to use it.

If activation-time (A only):

  • HotkeyReconciler.tsx — per-chord dedup in the install loop using the shared comparator; install only the winner; document the eventFilter invariant.
  • Tests — higher-priority context wins a same-key collision even when activated earlier; deactivating it restores the lower binding; equal priority falls back to last-wins; modal still suppresses non-colliding chords; global survives.

If fire-time (A + D):

  • types.ts — widen ActionHandler return to allow a "not handled" sentinel (e.g. boolean; void = handled, for back-compat). Decide async semantics (likely: decline must be a synchronous decision).
  • HotkeyReconciler.tsx — replace independent per-action window listeners with a single keydown/keyup coordinator: one listener per phase that maps the event's chord to candidates in precedence order and runs until one reports handled. Preserve withRecoveredLetterKey, shouldHandleEvent, hold-binding handling, and eventOptions precedence.
  • Tests — all of the activation-time tests, plus: a declining winner falls through to the next candidate; a handling winner suppresses the rest; decline order respects priority then recency.

Per repo testing guidance: no tests that merely re-state default binding strings.
