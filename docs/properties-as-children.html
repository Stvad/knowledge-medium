SKETCH

Properties as child blocks — sync-merge angle

Visualizes how the deferred "properties are children" model dissolves the per-key properties_json merge problem that drove docs/field-level-sync-merge.md. Not a new architecture proposal — the structural design already lives in docs/properties-model-evolution.md (Thread 2, "Option A + D"). This doc is the visual sketch + sync-side framing now that #52 has landed and the merge plan is on hold. Written 2026-05-27.

→ properties-model-evolution.md (full design, 2026-05-08) → field-level-sync-merge.md (parked merge plan) → follow-ups.md

1. The problem this dissolves

Post-#52, uploads are column-narrow: a content edit no longer wipes a concurrent properties_json edit. The remaining LWW failure mode lives inside properties_json itself — two clients writing different keys still PATCH the same column, and the later write strips the earlier key. Today's data model (src/data/blockSchema.ts:43: properties_json TEXT NOT NULL DEFAULT '{}') has no per-key grain, so no amount of envelope plumbing makes this a true per-key write.

Per-key clobber under column-LWW on properties_json.

2. The structural move

Every property becomes a child block of its owner. There is no shared properties_json column being concurrently written — there are N separate rows, one per property, each its own LWW grain. The merge problem doesn't get solved; it stops existing.

Today: properties as JSON blob

One row. One column. Concurrent key writes share LWW grain.

After: properties as children

Three rows. Each property has its own updated_at. No shared grain.

3. The same scenario, after

Two-client concurrent edit on different properties. Different rows ⇒ no merge needed.

Note the id derivation. The child id has to be a function of (parent_id, property_name) — typically uuidv5(parent_id, name) — so the same property writen from two clients converges to the same row. Otherwise we'd get duplicates per client rather than a conflict on one row, which is worse.

5. What the model actually looks like

The structural decision is already made in properties-model-evolution.md §Thread 2: Option A + D. Brief refresher (full reasoning in that doc):

  • Option A — cache on parent. properties_json on the parent block becomes a derived projection of the field-tagged children. Children are source of truth; the cache exists so reads stay sync and free, and so cross-block queries (json_extract(properties_json, '$.foo'), block_aliases index at src/data/internals/clientSchema.ts:134-142) don't change shape.
  • Option D — hybrid system fields. A small set of well-known fields (system:collapsed, types, title, maybe content) stays as cache-only system fields — no child block exists. Honest about which fields actually want first-class renamable/refable identity vs. which are just well-known per-block flags.

Call-site shape stays uniform — the asymmetry only surfaces at schema-definition time:

// System field — value lives in the parent's properties_json cache only.
const collapsed = defineSystemField({ id: 'system:collapsed', codec: codecs.boolean })

// User field — backed by a property-definition block in the propertiesPage
// (already exists today, see src/data/propertiesPage.ts).
const status = defineUserField({
  fieldBlockId: 'status-field-id',
  shape: 'scalar',
  codec: codecs.string,
})

// Reads — uniform, always sync, always from the cached projection:
block.get(collapsed)  // boolean
block.get(status)     // string

Notable: user-defined property schemas already live on property-definition blocks (children of propertiesPage; see src/data/propertiesPage.ts + src/data/repo.ts:169-197 propertySchemasFacet). The model is half-built today — what we'd add is property values living on per-(owner, field) child blocks, with the parent's properties_json as a cache.

The cache projection is what makes the read story not regress. Every existing reader continues to read properties_json on the parent — they don't learn that there's a child block underneath. The only code that knows the difference is the mutator-tail projector and (eventually) the property panel UI for user fields.

7. The sync echo, walked through

Scenario: Client A sets alias = ['Foo']; Client B sets isCollapsed = true on the same parent block. Both txes commit locally, both upload concurrently.
  1. Client A's tx writes two rows in one transaction:
    • UPSERT child "abc:alias" with value ["Foo"]
    • PATCH parent "abc"'s properties_json to project the new alias key
  2. Client B's tx writes two rows in one transaction:
    • UPSERT child "abc:isCollapsed" with value true
    • PATCH parent "abc"'s properties_json to project the new isCollapsed key
  3. On the server:
    • The two child UPSERTs touch different rows — both land cleanly.
    • The two parent PATCHes do collide on properties_json. Same column-LWW failure mode as today's property writes — and that's OK, because the cache is now derived.
  4. Sync arrives on each peer. The peer sees the two child rows it didn't have. A cache-recompute hook (same one the local mutator-tail uses) reprojects the parent's properties_json from all field-tagged children of that parent — both keys present. The collided parent PATCH is overwritten by the locally-recomputed projection.

This is the key invariant: parent.properties_json is never the source of truth — only a cache. A collision on the cache is recoverable by recomputing from the child rows. A collision on the children doesn't happen because the children are per-property rows.

One subtle thing. The peer-side recompute has to fire on sync-arrived property-child changes (insert, update, delete, re-parent), not just local edits. That's a smaller version of Layer C from the merge plan — but pleasantly bounded: the input is "list of field-tagged children of parent P", the output is the projected properties_json, and the entire recompute is deterministic given the inputs. No skipMetadata shenanigans, no clamp-bypass needed: the cache write echoes back to the original sender, which recomputes again and gets the same answer. Idempotent.

8. The 8 consistency cases (from the existing design)

From properties-model-evolution.md §Thread 2:

  1. Direct property write
  2. Child created already-tagged
  3. Untagged child → tagged (user converts a regular child into a field-value)
  4. Tagged child → untagged
  5. Tagged child's content edited
  6. Tagged child deleted
  7. Tagged child re-parented (drop from old parent, add to new)
  8. Remote sync apply

All eight reduce to a single rule: any mutation to a row where field_id IS NOT NULL (before or after) must also patch the relevant parent's properties_json cache in the same transaction. Implementable as one helper invoked from the tail of each kernel mutator. Case 8 (sync apply) is the case the merge plan was specifically struggling with — it falls out of the recompute-on-sync-arrival hook.

10. Open questions for a "do it now" version

  1. Property-child schema indicator. Three options:
    • New kind column on blocks ('normal' | 'property') — explicit, indexable, schema change
    • Reserved key on the child's own properties_json ({ "_field_id": "status-field-id" }) — zero schema change, slightly recursive
    • Derived from parent_id + a deterministic id-derivation (id === uuidv5(parent_id, field_name)) — no marker, but every read has to do the derivation to recognize a property child as such
    Leaning toward the reserved-key option for minimum migration cost; revisit if the "every property-child query needs a marker filter" overhead bites.
  2. Backfill shape. For each block with non-empty properties_json, create child blocks for the field-tagged values, leave the cache in place. Idempotent helper; runnable as a one-off migration over repo.allBlocks. Order of operations: backfill children first (cache already matches), then switch the setProperty mutator to write children, then the cache becomes derived rather than authoritative.
  3. System-fields list (Option D). The hard call: which fields are "system" (cache-only, no child) vs "user" (backed by a child block)? Conservative starter: only system:* ids are system; everything else is user. Aggressive starter: also types, alias, renderer, createdAt are system because they're per-block flags with no first-class renamable/refable identity. The aggressive version means fewer child blocks but loses some of the model's appeal for those fields specifically.
  4. Sync envelope for the cache PATCH. Do we ship it explicitly (parent block's properties_json column gets a PATCH on every property change), or rely on the peer-side recompute and skip uploading the cache entirely? Skipping is tempting (less wire traffic, no cache-collision concerns) but it means every peer pays the recompute cost on every property-child arrival. Probably ship the cache PATCH for the local sender's bandwidth/latency, recompute on peer for correctness — i.e., both, with peer recompute as the tiebreak.
  5. Property panel rendering. Property panel today reads the parent's data.properties dict. Unchanged for the cache-based reads. The "convert this child block into a field-value" interaction (case 3 above) is new UI — defer until the model lands.
  6. Migration: dual-write or one-shot? The cache projection makes dual-write nearly free — the cache is already what readers see. Probably: backfill children → flip setProperty → keep the cache updated from children forevermore. No flag, no transition window.

11. What this isn't

  • An implementation plan. Sizing and sequencing are in properties-model-evolution.md §Defer ("~1–2 weeks when we do it later").
  • A commitment to do this now. It's the alternative being weighed against finishing the parked merge plan in field-level-sync-merge.md.
  • A claim about Tana's actual storage internals. "Tana-style" is a surface-model reference — fields as first-class, properties as children. The design above is justified by our read/write/sync workload, not by appeals to what Tana does.
