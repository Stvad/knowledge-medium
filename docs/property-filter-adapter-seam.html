Property Filter Seam

The typed query engine already has richer semantics than the backlinks filter UI exposes. The filter UI needed a way to learn about a property's filter affordance without hard-coding domain knowledge into BacklinkFilters.

Background

  • codec.where means the stored JSON value can be directly compared in SQL.
  • where[name].target traverses a ref or refList and filters by the referenced block's properties — generic in the query compiler, no daily-notes special case.
  • All registered schemas can support is set / is unset, because those compile to JSON presence checks.

Problem

The UI used to know that a ref to a daily note should behave like a date. Useful, but it leaked daily-notes semantics into a generic backlinks surface, and it missed Roam-imported date attributes that arrive as refList.

What shipped

A small two-piece seam rather than a plugin-style adapter registry. The two pieces hang on facilities the codebase already has (a facet for cross-plugin contributions, a codec capability for storage-layer answers), so neither side has to grow new arbitration logic.

  • refTargetFilterDefaultsFacet in src/data/facets.ts: keyed by target block type. Daily notes contributes {targetType: 'daily-note', property: 'daily-note:date'}. Any ref or refList property whose target type appears in the facet — kernel-defined, user-defined, or Roam-imported — gets the same affordance without per-property registration.
  • resolvePropertyFilter in src/plugins/backlinks/propertyFilter.ts: derives a single PropertyFilterAffordance (operators, input kind, parser, predicate builder) from (schema, schemas, refTargetDefaults). For ref/refList with a matching target entry, it recursively resolves the inner property's affordance and wraps the builder in a target traversal. Otherwise it derives a scalar affordance from the codec's type and where presence.

Why not a filterAffordance field on PropertySchema?

Considered but deferred. The proposed field — letting a schema declare its own operators, parser, and builder — has no concrete consumer in the tree today. Adding a speculative extension point would have violated the project's "no design for hypothetical requirements" guideline. The door stays open: if a future schema needs per-property custom UX, slotting a filterAffordance field in front of resolvePropertyFilter is a small change.

Why not the originally proposed adapter registry?

The original sketch had a PropertyFilterAdapter interface with match(schema) and priority. That arbitration only earns its keep when ownership of a property is ambiguous between multiple contributors. Here it isn't — the target type owns the affordance for refs pointing at it, and codec type owns the scalar case. A direct facet lookup beats match() when there's no contention to resolve.

Layering

Query capability stays in codecs and typed query. The facet expresses a target-type-level fact ("filtering a ref to me defaults to filtering on this inner property") rather than a per-property UI hint, so the contributing plugin owns one entry regardless of how many ref properties downstream point at its type.
