The one thing to take away

The individual bugs are already fixed (c2df661e, 21494fdb, 979ca0fd, 4e4037fd, plus the matrix content-ref case). What we lacked — and what every incident exposed — is early detection: each corruption sat invisible until a human noticed "fewer backlinks." This document defines the layered detection/prevention/recovery system to close that gap, sized for a small coordinatable fleet, plus an explicit map of the blind spots it doesn't close yet (§4). Read-only detection is the priority; write/recovery tooling is approval-gated. Status: the client eval (L3+L4) is committed (read-only, run on demand via the bridge); the server detector (L5) is now written and committed (validated against a seeded Postgres exercising every detector + the e2ee/durability guards) — it goes live on the prod hourly cron once the commit is pushed to the default branch.

0Scope — what this is and isn't

knowledge-medium runs on a small, coordinatable fleet (≈2 users / 5 devices). The defense is sized for that: cheap read-only detection a human can act on, not auto-healing infrastructure or mixed-version-window machinery. We do not gold-plate for scale.

The single most valuable property the incidents revealed we lack is early detection. So detection leads; recovery/write tooling is authored but approval-gated (standing rule: no execution against shared infra without an explicit ask).

Hard constraint — no hot-path cost

No guard here may add latency to the synchronous read or write tail. Audited per layer: L1 is compile/PR-time; L3/L4 run on demand or on an idle schedule (the idleMarkerJobs pattern), never on a read/write; L5 is a separate cron process on its own connection (zero client cost); L6 is manual/gated. the only layer that touches app code is L2, and those are dev/test-only assertions compiled out of prod (not shipped at all) — so there is no prod runtime variant to sample or bound. This is also why class D (below) gets no sync-gate change — its defense is operational, in L6.

3The layers — defense in depth

Seven concerns across six layers, ordered from the point of derivation (catch the instant corruption is introduced) outward to fleet-wide at rest (last-resort catch). The earlier a layer catches, the cheaper the blast radius.

L1 — Prevention guardrails (codify what we learned)

These principles are now law; the job is to keep them from regressing.

  1. A property-derived ref is tied to its VALUE, not its schema. Schema changes only ever ADD projections; removal is value-driven (21494fdb). Reprojection is add-only (repo.ts:1485) and never strips. Schema absence ⇒ retain the ref (referencesProcessor.ts:181), unless this write changed that field's own value.
  2. Reprojection is workspace-scoped (post 24971b31) — never touches sibling workspaces.
  3. Bulk timestamp bumps are forbidden as a "make-X-win" tool unless the data forced to win is independently verified correct (incident #4). Recovery drives off the converged/server view and re-verifies — see L6.

Gap to close — class B

Per-column patch consistency. A references-only write can still win the references_json column independently of the content version it was derived from. The gate is whole-row LWW on updated_at (reconcile.ts:88) — it doesn't know a given references_json was computed from stale content. This is the matrix incident's root and is not structurally prevented yet — see Decision D1.

L2 — Write / processor-time assertions

Invariant checks at the point of derivation, so inconsistency is caught the instant it's introduced. These are dev/test-only assertions — a build-time DEV gate (process.env.NODE_ENV !== 'production', which both the Vite build and the non-Vite kernel/agent-cli build statically eliminate in prod; not import.meta.env.DEV in kernel modules like repo.ts, which compile outside Vite), not a user-facing flag. They hard-throw in dev and test runs and are compiled out of prod entirely. Two reasons there's no prod variant (and no sampling): you must never throw on a user's write, and even an async recompute is needless cost on every derivation. The in-the-wild signal is L3 + L5's job (at-rest detection); L2 exists purely to fail a regression in CI before it ships.

  • References processor: assert the references being written equal a fresh projection of the committed content + properties. (Not a self-comparison — the point is the basis is committed state, so a plan built from a read that went stale under it fails the check.) R1/R3 · A
  • Reprojection: assert add-only (no key in the prior set is missing from the next set) — a cheap set comparison. Turns a future "I made it strip again" into a failing test. R1/R4 · A
  • decideStagingRow/apply: flag a references-only apply whose content differs from the version the refs imply (derived/source desync, class B). R3 · B · Decision D1

Dev/test only — never on the prod path, synchronous or async. roadmap P2

L3 — Client background consistency check

A periodic/idle self-audit, committed as a read-only agent-bridge eval (scripts/data-integrity/consistency-check.eval.js, run on demand). The ad-hoc checks used during this week's investigation, as a maintained suite — all read-only. It runs on the decrypted client, so unlike the server probe it can audit the e2ee workspace (see §4):

  1. R2 mirror — references_json ↔ block_references, both directions (pure SQL).
  2. R7 dangling — refs whose target is missing/deleted, split content vs property (reports baseline; for trend not absolute).
  3. R4 property-ref projection — two complementary checks. (1) Precise: for each loaded ref-typed property, recompute via projectPropertyReferences and diff vs stored (a missing projection is a strip; an extra is flagged only when the schema is present but the value no longer encodes it — absent-schema refs are retained-by-design, excluded). This is sound only for the active workspace (repo.propertySchemas is active-only) and only for currently-loaded props, so it's gated to the active workspace. (2) At-rest heuristic: schema-independent — enumerates believed-ref props from block_references.source_field ∪ a curated list (next-review-date) and flags value-present/ref-absent blocks. This is what catches a strip of a toggled-off prop's refs (the exact 06-09 SRS-off condition the precise check is blind to), so a clean precise report doesn't falsely reassure.
  4. R1/R3 content-link strip recompute — parser-precise: parse each candidate's content with the processor's own parser (parseReferences + parseBlockRefs) and flag a block with ≥1 real content mark but zero stored content refs (a strip). Keys on presence not per-ref identity, so it's sound without alias resolution (content refs dedup by target id). Replaces the old LIKE heuristic — no more code-block / escaped-bracket noise. The full per-id alias-resolving diff (catching a partial content-ref strip) needs the processor's read-only seat resolution and is deferred (§6).

The aim: a built-in idle self-audit. The same logic becomes a scheduled idle job (the runAnalyzeIfStale / idleMarkerJobs.ts pattern) emitting a metric + optional UI health indicator — giving the checks a real cadence instead of running only on demand. The on-demand eval is the bootstrap: it validates the checks are low-noise on real data, then we promote. This is the committed direction the §4 blind spots and decision D3 point at — not an "if only," but the planned next step.

L4 — Client ↔ server divergence detection

The lesson of incidents #3 and #5: per-client scans miss divergence. Detection must compare blocks against blocks_synced, ps_crud-aware. Same eval; categories:

  • Stranded local-only — blocks row, no blocks_synced row, no pending ps_crud (will never upload). R6 · C
  • Equal-stamp standoff — both exist, updated_at equal & nonzero, content differs (violates R8; neither side yields). R8 · C/D
  • Local-richer-than-server — blocks newer, no pending ps_crud (at-risk local data that won't sync). R6 · C
  • Server-ahead-undrained — blocks_synced newer & differs, nothing queued. Normally transient; persistent ⇒ a Layout-B drain problem. R6

ps_crud membership read via SELECT json_extract(data,'$.id') FROM ps_crud WHERE json_extract(data,'$.type')='blocks'.

L5 — Server-side anomaly detection (extend the hourly probe)

blocks_history is the forensic backbone: append-only, with op/semantic_op/changed_columns/actor/event_time/before_diff/after_diff. The existing hourly probe (scripts/supabase-health.mjs, cron :17) already holds a privileged pooler connection. It extends it with a runDataIntegrityChecks() pass (implemented — signatures + starting thresholds below) over a 90-minute window (overlaps the hourly cadence so no gap). These detectors read the server, so they are plaintext-only and see nothing in a history-trigger-off window — see §4 for those blind spots:

Detector nuance — the column set is not a discriminator (emptied vs shrunk tradeoff)

It's tempting to key the strip detector on changed_columns ⊆ {references_json, updated_at, user_updated_at}. That set is not a discriminator: every normal references-processor write and add-only reprojection writes refs with skipMetadata (referencesProcessor.ts:234, repo.ts:1513), which produces exactly {references_json, updated_at} via metadataPatch (txEngine.ts:759). So the column shape matches routine editing too. Two candidate value-signals, with a real tradeoff:

  • Emptied (after = '[]', non-empty before) — reliable and low-noise; normal editing rarely empties a block's entire ref set. This is the signal the catastrophe hard-fail uses.
  • Shrunk (ref count merely decreased) — catches a partial strip, but also fires on every ordinary link removal. On a 5-device fleet a single cleanup session can remove 200+ links in 90 min, so a shrunk-based alarm trips benignly.

Decision: the hard-fail keys on emptied refs that are still empty at window end (durable), not a raw in-window count — a burst that refills (a self-healing reprojection, a bulk import) is at most a WARN, so the alarm isn't trained into noise on the very bulk-op class (D) it guards (the 06-08 event stripped 52k then self-healed 99.4% — a raw-count fail would have screamed on a transient). Shrunk stays informational only (high threshold / off), since routine link removal shrinks ref sets; a slow/partial strip is instead caught at rest by R4 (§4, slow-drip row), not by this rate detector.

Alerting posture — Decision D2

The probe's failure channel is GitHub-Actions-failure email; only result.failures exits non-zero. Data-integrity detectors are warn-by-default (visible in the JSON snapshot, no email) so an untuned detector can't spam on first deploy — except the mass references-only strip catastrophe, which hard-fails above a conservative threshold (the now-add-only reprojection should never strip, so a large count is unambiguously wrong). Set DATA_INTEGRITY_STRICT=1 to promote all warns to fails once thresholds are baselined.

L6 — Recovery tooling

Turn the ad-hoc bridge evals into a maintained, idempotent, dry-run-first harness. The pattern already exists and is good (scripts/daily-note-date-recovery/recover.eval.js): default dry-run returns a report; --data-json '{"apply":true}' writes through repo.tx (so rows upload and the fleet converges); in-tx recheck makes re-runs safe. Rules learned:

  • Pre-flight verify before any "make-X-win" op — diff the copy you're about to force to win against the converged view first. This, not the L5 warn, is the real defense against incident #4: a bulk bump propagates immediately, so a post-hoc detector is always too late. The harness must refuse to bump rows it hasn't confirmed correct.
  • Drive off the converged/server view, or write from the client where the row is actually stale (a byte-identical write on the healthy client produces no ps_crud — the #5 trap).
  • Verify convergence after (re-run the detector; confirm ps_crud drained).
  • Never blind-bump updated_at to force unverified data to win (incident #4).

These are conventions + the existing harness, not a new framework. A shared scripts/data-integrity/ README plus the verify-before-bump pre-flight is the cheap win — and the actual class-D prevention (L5 only detects D after the fact, and only if the history trigger was left on). P1

These gaps were surfaced by adversarial review; for the small fleet they are accepted-with-awareness, with the noted mitigations. The recurring theme — and the most important caveat — is that no integrity detector runs automatically today: the hourly probe still does infra checks only (L5 is unwritten), and the client eval (the one integrity check that exists) is manual. Net — the honest headline

No automatic integrity detector runs in prod yet. The committed eval (L3/L4) is manual-only, and the L5 server detector is now written and committed but not yet deployed — until the commit is pushed to the default branch the hourly cron still does infra checks only. So "caught within hours, not a week" is the goal, not the current state: it becomes true for the plaintext mass-strip the moment L5 is pushed to the cron, and for the e2ee workspace / mobile / slow drip once the scheduled built-in L3 lands. Until L5 is deployed, the only integrity detection that exists is a manual eval someone has to remember to run.

5Convergence SLO

A row is "diverged" if blocks ≠ blocks_synced with no pending ps_crud. Target: a row diverged > 30 minutes is an anomaly. An online client drains a staging batch in seconds; 30 min covers an offline/backgrounded device's next foreground.

Not measured yet — this is a target, not a live SLO. The committed L4 eval takes a point-in-time snapshot: it flags a row diverged at scan time, with no notion of how long it's been diverged; and L5 (server) can't observe client-local divergence at all. Measuring the age of a divergence needs either repeated snapshots or a per-row "first seen diverged" timestamp — neither is built. So the 30-minute bound is what a scheduled built-in L3 should eventually enforce; today nothing evaluates it.

7Design decisions

Resolved during review; deferred structural work is tracked in linked issues.

D1 · Per-column patch / derived-vs-source desync at the sync gate (class B, the matrix incident) resolved

The gate is whole-row LWW on updated_at; a references-only patch from a sync-lagged client can win the references_json column carrying refs derived from stale content. Options were (a) tie derived columns to a content version, (b) re-derive on apply, (c) accept + detect.

Decision: start with detection (c) — ship the L3/L5 detectors (and the scheduled L3, §3) and see whether class B actually recurs before touching sync-gate semantics. Promote to (a)/(b) only if detection shows recurrence. Tracked in #165. Caveat (be honest): "(c)" only means "detect" once detection is actually running — the on-demand eval covers class B at rest, but until the L5 server detector and a scheduled L3 ship, class B is undetected between manual runs, and the "recurrence" trigger for promoting to (a)/(b) can't fire on its own.

D2 · L5 alerting posture resolved

Decision: share the infra — the data-integrity detectors run on the existing hourly probe's cron (once the commit is pushed) and use its failure-email channel (one cron, one channel; right for the small fleet). Posture stays warn-by-default with the one catastrophe hard-fail, promotable via DATA_INTEGRITY_STRICT (§3 L5). Calibration caveat: WARN is email-silent (snapshot-only), but for a 2-user fleet the realistic incident is a few-hundred-row mistake, which lands in the WARN band — so the dangerous-but-not-catastrophic middle is muted. Once the detector is built and baselined, lower the FAIL thresholds toward fleet-realistic counts (a 200-row unintended delete should page), rather than leaving them at catastrophe-only ceilings.

D3 · Built-in self-audit vs eval-only (L3) resolved

Decision: the built-in scheduled idle self-audit is the aim (§3 L3) — a metric + optional UI health indicator on a real cadence. The on-demand eval is the bootstrap that validates the checks are low-noise on real data; once it does, promote. Not "stay eval-only" indefinitely.
