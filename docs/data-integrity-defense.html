The one thing to take away

The individual bugs are already fixed (c2df661e, 21494fdb, 979ca0fd, 4e4037fd, plus the matrix content-ref case). What we lacked — and what every incident exposed — is early detection: each corruption sat invisible until a human noticed "fewer backlinks." This document defines the layered detection/prevention/recovery system to close that gap, sized for a small coordinatable fleet, plus an explicit map of the blind spots it doesn't close yet (§4). Read-only detection is the priority; write/recovery tooling is approval-gated. Status: the client eval (L3+L4) is authored and held in a stash pending this review; the server detector (L5) is designed but not yet written; nothing is committed alongside this doc.

0Scope — what this is and isn't

knowledge-medium runs on a small, coordinatable fleet (≈2 users / 5 devices). The defense is sized for that: cheap read-only detection a human can act on, not auto-healing infrastructure or mixed-version-window machinery. We do not gold-plate for scale.

The single most valuable property the incidents revealed we lack is early detection. So detection leads; recovery/write tooling is authored but approval-gated (standing rule: no execution against shared infra without an explicit ask).

Hard constraint — no hot-path cost

No guard here may add latency to the synchronous read or write tail. Audited per layer: L1 is compile/PR-time; L3/L4 run on demand or on an idle schedule (the idleMarkerJobs pattern), never on a read/write; L5 is a separate cron process on its own connection (zero client cost); L6 is manual/gated. The only layer that touches app code is L2 — dev-only by default, and any prod variant confined to the already-asynchronous post-commit derivation points (the references processor deliberately runs off the typing tail) and sampled. Nothing synchronous on the hot path. This is also why class D (below) gets no sync-gate change — its defense is operational, in L6.

3The layers — defense in depth

Seven concerns across six layers, ordered from the point of derivation (catch the instant corruption is introduced) outward to fleet-wide at rest (last-resort catch). The earlier a layer catches, the cheaper the blast radius.

L1 — Prevention guardrails (codify what we learned)

These principles are now law; the job is to keep them from regressing.

  1. A property-derived ref is tied to its VALUE, not its schema. Schema changes only ever ADD projections; removal is value-driven (21494fdb). Reprojection is add-only (repo.ts:1485) and never strips. Schema absence ⇒ retain the ref (referencesProcessor.ts:181), unless this write changed that field's own value.
  2. Reprojection is workspace-scoped (post 24971b31) — never touches sibling workspaces.
  3. Bulk timestamp bumps are forbidden as a "make-X-win" tool unless the data forced to win is independently verified correct (incident #4). Recovery drives off the converged/server view and re-verifies — see L6.

Gap to close — class B

Per-column patch consistency. A references-only write can still win the references_json column independently of the content version it was derived from. The gate is whole-row LWW on updated_at (reconcile.ts:88) — it doesn't know a given references_json was computed from stale content. This is the matrix incident's root and is not structurally prevented yet — see Decision D1.

L2 — Write / processor-time assertions

Cheap invariant checks at the point of derivation, so inconsistency is caught the instant it's introduced. Subject to the no-hot-path constraint (§0): dev = hard throw; prod = off by default, and any prod variant is sampled and confined to already-async paths.

  • References processor write phase (already off the synchronous typing tail): recompute references from the same source snapshot used for the write and assert equality (catches a plan built from a stale read). Cheap, async — safe to sample in prod. R1/R3
  • Reprojection: assert add-only (no prior key missing from the next set) — a set comparison on an already-async pass. Turns a future "I made it strip again" into a dev-time failure. R1/R4 · A
  • decideStagingRow/apply: flag a references-only apply whose content differs from the version the refs imply (derived/source desync). Per-synced-row re-derivation is too expensive for prod — keep this one dev-only, or reduce it to a stamp/column check. R3 · B · Decision D1

Dev flag first; any prod metric is sampled and async-only. Never on the synchronous read or write tail. roadmap P2

L3 — Client background consistency check

A periodic/idle self-audit, authored now as a read-only agent-bridge eval (scripts/data-integrity/consistency-check.eval.js — held in a stash pending this review, not yet committed). The ad-hoc checks used during this week's investigation, as a maintained suite — all read-only. It runs on the decrypted client, so unlike the server probe it can audit the e2ee workspace (see §4):

  1. R2 mirror — references_json ↔ block_references, both directions (pure SQL).
  2. R7 dangling — refs whose target is missing/deleted, split content vs property (reports baseline; for trend not absolute).
  3. R4 property-ref projection — for every ref-typed property in repo.propertySchemas, recompute via projectPropertyReferences and diff vs stored. The generic next-review-date detector (covers all ~83 ref props). Degrades gracefully if schemas can't resolve.
  4. R1/R3 content-link coarse check — heuristic flag for blocks whose content has link syntax but zero content-refs. Informational (escaping/code-blocks cause benign hits).

Promotion path: the same logic can become a built-in idle self-audit (the runAnalyzeIfStale / idleMarkerJobs.ts pattern) emitting a metric + optional UI health indicator. This is what would give the checks a real cadence (today they run only on demand — see §4); deferred until the eval proves low-noise.

L4 — Client ↔ server divergence detection

The lesson of incidents #3 and #5: per-client scans miss divergence. Detection must compare blocks against blocks_synced, ps_crud-aware. Same eval; categories:

  • Stranded local-only — blocks row, no blocks_synced row, no pending ps_crud (will never upload). R6 · C
  • Equal-stamp standoff — both exist, updated_at equal & nonzero, content differs (violates R8; neither side yields). R8 · C/D
  • Local-richer-than-server — blocks newer, no pending ps_crud (at-risk local data that won't sync). R6 · C
  • Server-ahead-undrained — blocks_synced newer & differs, nothing queued. Normally transient; persistent ⇒ a Layout-B drain problem. R6

ps_crud membership read via SELECT json_extract(data,'$.id') FROM ps_crud WHERE json_extract(data,'$.type')='blocks'.

L5 — Server-side anomaly detection (extend the hourly probe)

blocks_history is the forensic backbone: append-only, with op/semantic_op/changed_columns/actor/event_time/before_diff/after_diff. The existing hourly probe (scripts/supabase-health.mjs, cron :17) already holds a privileged pooler connection. The plan extends it with a runDataIntegrityChecks() pass (designed; not yet written — only thresholds drafted) over a 90-minute window (overlaps the hourly cadence so no gap). These detectors read the server, so they are plaintext-only and see nothing in a history-trigger-off window — see §4 for those blind spots:

Detector nuance — the column set is not a discriminator (emptied vs shrunk tradeoff)

It's tempting to key the strip detector on changed_columns ⊆ {references_json, updated_at, user_updated_at}. That set is not a discriminator: every normal references-processor write and add-only reprojection writes refs with skipMetadata (referencesProcessor.ts:234, repo.ts:1513), which produces exactly {references_json, updated_at} via metadataPatch (txEngine.ts:759). So the column shape matches routine editing too. Two candidate value-signals, with a real tradeoff:

  • Emptied (after = '[]', non-empty before) — reliable and low-noise; normal editing rarely empties a block's entire ref set. This is the signal the catastrophe hard-fail uses.
  • Shrunk (ref count merely decreased) — catches a partial strip, but also fires on every ordinary link removal. On a 5-device fleet a single cleanup session can remove 200+ links in 90 min, so a shrunk-based alarm trips benignly.

Decision: ship emptied + count-in-window as the alerting signal; keep shrunk only as an informational snapshot field with a deliberately high threshold (or off), accepting that a slow/partial strip is the cost — and is instead caught at rest by the R4 check (§4, slow-drip row), not by this rate detector.

Alerting posture — Decision D2

The probe's failure channel is GitHub-Actions-failure email; only result.failures exits non-zero. Data-integrity detectors are warn-by-default (visible in the JSON snapshot, no email) so an untuned detector can't spam on first deploy — except the mass references-only strip catastrophe, which hard-fails above a conservative threshold (the now-add-only reprojection should never strip, so a large count is unambiguously wrong). Set DATA_INTEGRITY_STRICT=1 to promote all warns to fails once thresholds are baselined.

L6 — Recovery tooling

Turn the ad-hoc bridge evals into a maintained, idempotent, dry-run-first harness. The pattern already exists and is good (scripts/daily-note-date-recovery/recover.eval.js): default dry-run returns a report; --data-json '{"apply":true}' writes through repo.tx (so rows upload and the fleet converges); in-tx recheck makes re-runs safe. Rules learned:

  • Pre-flight verify before any "make-X-win" op — diff the copy you're about to force to win against the converged view first. This, not the L5 warn, is the real defense against incident #4: a bulk bump propagates immediately, so a post-hoc detector is always too late. The harness must refuse to bump rows it hasn't confirmed correct.
  • Drive off the converged/server view, or write from the client where the row is actually stale (a byte-identical write on the healthy client produces no ps_crud — the #5 trap).
  • Verify convergence after (re-run the detector; confirm ps_crud drained).
  • Never blind-bump updated_at to force unverified data to win (incident #4).

These are conventions + the existing harness, not a new framework. A shared scripts/data-integrity/ README plus the verify-before-bump pre-flight is the cheap win — and the actual class-D prevention (L5 only detects D after the fact, and only if the history trigger was left on). P1

4Coverage limits & blind spots

Defense in depth only earns trust if it's honest about what it can't see. These gaps were surfaced by adversarial review; for the small fleet they are accepted-with-awareness, with the noted mitigations. The recurring theme: the only continuous watcher today is the hourly server probe, and it has structural blind spots that the (on-demand) client eval covers — so the real fix for several rows below is giving the client eval a cadence (a built-in scheduled L3).

Net — the honest headline

The "caught within hours, not a week" claim holds today only for the plaintext mass-strip catastrophe (the hourly probe's emptied-refs hard-fail). For the e2ee workspace, slow drips, history-trigger-off bulk ops, and mobile client-local divergence, coverage depends on the client eval — which currently runs only on demand. Closing the "runs only on demand" gap (a scheduled built-in L3) and adding L2 are what make early detection true for every class.

5Convergence SLO

A row is "diverged" if blocks ≠ blocks_synced with no pending ps_crud. Target: a row diverged > 30 minutes is an anomaly. An online client drains a staging batch in seconds; 30 min covers an offline/backgrounded device's next foreground. This SLO is what L4 (client) and L5 (server, indirectly) alarm against. For the small fleet it's a human-actionable signal, not an auto-remediation trigger.

7Open design decisions — need you

D1 · Per-column patch / derived-vs-source desync at the sync gate (class B, the matrix incident)

Today the gate is whole-row LWW on updated_at; a references-only patch from a sync-lagged client can win the references_json column carrying refs derived from stale content, independent of the content no-clobber. Options:

  • (a) Tie derived columns to a content version — stamp references_json writes with the updated_at of the content they were derived from; reject a references apply whose basis is older than local content. Strongest, but adds a column/semantic to the gate.
  • (b) Re-derive on apply — when applying a references-only staging row, re-run projection locally instead of trusting incoming refs. Moves derivation to the consumer; cost on the sync path.
  • (c) Accept + detect — leave the gate, rely on L3/L5 to catch desync after the fact. Cheapest; matches the small-fleet posture.

Recommendation: (c) for now, revisit (a) if L5 shows recurrence. Caveat: "accept + detect" is only as good as the detection — until the L5 detector and a scheduled L3 actually ship, "(c)" currently means just "accept", so class B is undetected, not detected-after-the-fact. This touches sync-gate semantics — flagging for explicit sign-off before any gate change.

D2 · L5 alerting posture

Warn-by-default with one catastrophe hard-fail, promotable via DATA_INTEGRITY_STRICT (see §3 L5). Confirm this matches how you want the email channel used, and whether data-integrity checks should share the infra-probe's failure channel or get a separate one. Recommendation: share it — one cron, one channel, small fleet.

D3 · Built-in self-audit vs eval-only (L3)

A built-in idle self-audit means a scheduled task + metric in the app bundle. Recommendation: stay eval-only until the eval proves low-noise across a couple of weeks, then promote.
