The one thing to take away

The individual bugs are already fixed (c2df661e, 21494fdb, 979ca0fd, 4e4037fd, plus the matrix content-ref case). What we lacked — and what every incident exposed — is early detection: each corruption sat invisible until a human noticed "fewer backlinks." This document defines the layered detection/prevention/recovery system that closes that gap, sized for a small coordinatable fleet, and a prioritized roadmap. Read-only detection is the priority; write/recovery tooling is authored but approval-gated.

0Scope — what this is and isn't

knowledge-medium runs on a small, coordinatable fleet (≈2 users / 5 devices). The defense is sized for that: cheap read-only detection a human can act on, not auto-healing infrastructure or mixed-version-window machinery. We do not gold-plate for scale.

The single most valuable property the incidents revealed we lack is early detection. So detection leads; recovery/write tooling is authored but approval-gated (standing rule: no execution against shared infra without an explicit ask).

The week produced six incidents. They collapse into five recurring failure classes — naming them is the point, because each needs its own defense and any future bug will be a new instance of one of them.

3The layers — defense in depth

Seven concerns across six layers, ordered from the point of derivation (catch the instant corruption is introduced) outward to fleet-wide at rest (last-resort catch). The earlier a layer catches, the cheaper the blast radius.

L1 — Prevention guardrails (codify what we learned)

These principles are now law; the job is to keep them from regressing.

  1. A property-derived ref is tied to its VALUE, not its schema. Schema changes only ever ADD projections; removal is value-driven (21494fdb). Reprojection is add-only (repo.ts:1485) and never strips. Schema absence ⇒ retain the ref (referencesProcessor.ts:181), unless this write changed that field's own value.
  2. Reprojection is workspace-scoped (post 24971b31) — never touches sibling workspaces.
  3. Bulk timestamp bumps are forbidden as a "make-X-win" tool unless the data forced to win is independently verified correct (incident #4). Recovery drives off the converged/server view and re-verifies — see L6.

Gap to close — class B

Per-column patch consistency. A references-only write can still win the references_json column independently of the content version it was derived from. The gate is whole-row LWW on updated_at (reconcile.ts:88) — it doesn't know a given references_json was computed from stale content. This is the matrix incident's root and is not structurally prevented yet — see Decision D1.

L2 — Write / processor-time assertions

Cheap invariant checks at the point of derivation, so inconsistency is caught the instant it's introduced. Dev = hard throw; prod = soft log + metric counter (never block a user write in prod).

  • References processor write phase: recompute references from the same source snapshot used for the write and assert equality (catches a plan built from a stale read). R1/R3
  • decideStagingRow/apply: flag a references-only apply whose content column differs from the version the refs imply — derived/source desync. R3 · B · Decision D1
  • Reprojection: assert add-only (no prior key missing from the next set). Turns a future "I made it strip again" into a dev-time failure. R1/R4 · A

Small and high-leverage but they touch hot/sync-gate paths, so they ship behind a dev flag first, prod metric second. roadmap P2

L3 — Client background consistency check

A periodic/idle self-audit, implemented now as a committed read-only agent-bridge eval (scripts/data-integrity/consistency-check.eval.js). The ad-hoc checks used during this week's investigation, as a maintained suite — all read-only:

  1. R2 mirror — references_json ↔ block_references, both directions (pure SQL).
  2. R7 dangling — refs whose target is missing/deleted, split content vs property (reports baseline; for trend not absolute).
  3. R4 property-ref projection — for every ref-typed property in repo.propertySchemas, recompute via projectPropertyReferences and diff vs stored. The generic next-review-date detector (covers all ~83 ref props). Degrades gracefully if schemas can't resolve.
  4. R1/R3 content-link coarse check — heuristic flag for blocks whose content has link syntax but zero content-refs. Informational (escaping/code-blocks cause benign hits).

Promotion path: the same logic can become a built-in idle self-audit (mirror runAnalyzeIfDue) emitting a metric + optional UI health indicator. Deferred until the eval proves low-noise.

L4 — Client ↔ server divergence detection

The lesson of incidents #3 and #5: per-client scans miss divergence. Detection must compare blocks against blocks_synced, ps_crud-aware. Same eval; categories:

  • Stranded local-only — blocks row, no blocks_synced row, no pending ps_crud (will never upload). R6 · C
  • Equal-stamp standoff — both exist, updated_at equal & nonzero, content differs (violates R8; neither side yields). R8 · C/D
  • Local-richer-than-server — blocks newer, no pending ps_crud (at-risk local data that won't sync). R6 · C
  • Server-ahead-undrained — blocks_synced newer & differs, nothing queued. Normally transient; persistent ⇒ a Layout-B drain problem. R6

ps_crud membership read via SELECT json_extract(data,'$.id') FROM ps_crud WHERE json_extract(data,'$.type')='blocks'.

L5 — Server-side anomaly detection (extend the hourly probe)

blocks_history is the forensic backbone: append-only, with op/semantic_op/changed_columns/actor/event_time/before_diff/after_diff. The existing hourly probe (scripts/supabase-health.mjs, cron :17) already holds a privileged pooler connection. We extend it with runDataIntegrityChecks() over a 90-minute window (overlaps the hourly cadence so no gap):

Alerting posture — Decision D2

The probe's failure channel is GitHub-Actions-failure email; only result.failures exits non-zero. Data-integrity detectors are warn-by-default (visible in the JSON snapshot, no email) so an untuned detector can't spam on first deploy — except the mass references-only strip catastrophe, which hard-fails above a conservative threshold (the now-add-only reprojection should never strip, so a large count is unambiguously wrong). Set DATA_INTEGRITY_STRICT=1 to promote all warns to fails once thresholds are baselined.

L6 — Recovery tooling

Turn the ad-hoc bridge evals into a maintained, idempotent, dry-run-first harness. The pattern already exists and is good (scripts/daily-note-date-recovery/recover.eval.js): default dry-run returns a report; --data-json '{"apply":true}' writes through repo.tx (so rows upload and the fleet converges); in-tx recheck makes re-runs safe. Rules learned:

  • Drive off the converged/server view, or write from the client where the row is actually stale (a byte-identical write on the healthy client produces no ps_crud — the #5 trap).
  • Verify convergence after (re-run the detector; confirm ps_crud drained).
  • Never blind-bump updated_at to force unverified data to win (incident #4).

These are conventions + the existing harness, not a new framework. A shared scripts/data-integrity/ README is the cheap win. P1

4Convergence SLO

A row is "diverged" if blocks ≠ blocks_synced with no pending ps_crud. Target: a row diverged > 30 minutes is an anomaly. An online client drains a staging batch in seconds; 30 min covers an offline/backgrounded device's next foreground. This SLO is what L4 (client) and L5 (server, indirectly) alarm against. For the small fleet it's a human-actionable signal, not an auto-remediation trigger.

6Open design decisions — need you

D1 · Per-column patch / derived-vs-source desync at the sync gate (class B, the matrix incident)

Today the gate is whole-row LWW on updated_at; a references-only patch from a sync-lagged client can win the references_json column carrying refs derived from stale content, independent of the content no-clobber. Options:

  • (a) Tie derived columns to a content version — stamp references_json writes with the updated_at of the content they were derived from; reject a references apply whose basis is older than local content. Strongest, but adds a column/semantic to the gate.
  • (b) Re-derive on apply — when applying a references-only staging row, re-run projection locally instead of trusting incoming refs. Moves derivation to the consumer; cost on the sync path.
  • (c) Accept + detect — leave the gate, rely on L3/L5 to catch desync after the fact. Cheapest; matches the small-fleet posture.

Recommendation: (c) for now, revisit (a) if L5 shows recurrence. This touches sync-gate semantics — flagging for explicit sign-off before any gate change.

D2 · L5 alerting posture

Warn-by-default with one catastrophe hard-fail, promotable via DATA_INTEGRITY_STRICT (see §3 L5). Confirm this matches how you want the email channel used, and whether data-integrity checks should share the infra-probe's failure channel or get a separate one. Recommendation: share it — one cron, one channel, small fleet.

D3 · Built-in self-audit vs eval-only (L3)

A built-in idle self-audit means a scheduled task + metric in the app bundle. Recommendation: stay eval-only until the eval proves low-noise across a couple of weeks, then promote.
