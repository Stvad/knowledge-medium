Malleable-systems digest → KM experiment catalog

2026-07-12

The Malleable Systems Collective posts a "Collective Digest" roundup each year — 2025 and 2024 each link off to the forum threads that made up that year's conversation, ~44 across the two posts, all read in full for this doc (essays, reply chains, and — where a thread pointed at a real repo/spec — the artifact itself). The filter throughout: concrete, transferable mechanisms — data models, UI affordances, APIs, live-customization patterns — over philosophy and manifesto. A thread that's purely "software should empower users" gets a one-line appendix entry; a thread with a rewrite-rule syntax, a projector interface, or a capability model gets mined for a KM-shaped design sketch — the eight numbered experiments below, plus four cheaper one-offs, a validates-existing-design note, and a full thread index.

If building one thing first: #1 (block inspector) — highest utility-to-effort ratio, a pure extension with no core change, and it feeds the dev loop that produces everything else in this repo. If building a second: #3 (rewrite rules) — the first surface in KM that would be genuinely end-user-programmable without going through the agent bridge, and it fits naturally with the existing pattern of an agent drafting something the user then owns and edits.

1. Block inspector — "reveal codes" for KM

Sources: Transparent UIs · The Mystery of Moose · Locality, modular analysis · Brutalist HTML quine

Glamorous Toolkit's moldable inspector lets you right-click any widget and jump straight to the object or class rendering it — views are summoned per object type on demand, not fixed to one debugger. The Locality thread names the general problem this solves: when remote code (AOP advice, or in KM an extension's actionTransform/decorator) silently modifies behavior at a site, the fix belongs in tooling, not language restriction — annotate the affected site inline with a pointer back to the source. Konrad Hinsen's "local understandability" (you should understand an element by looking only at it, via Gabriel's Patterns of Software) and the HTML-quine thread's "reveal codes" (WYSIWYG layered over an always-inspectable substrate) both converge on the same affordance.

KM shape: an inspector extension — a keybind/action on the focused block opens a modal (house modal pattern, gotcha #19) showing the block's types and typed props with their contributing schemas; which renderers, decorators, content-decorators, and actionTransforms are currently active on it and from which extension; and which keybind contexts are armed. Every entry links back to the contributing extension's source block. Phase 2 borrows Locality's tooling fix directly: a passive "overridden here" gutter badge on any block whose behavior is being rewritten by another extension's facet contribution.

Value/effort: high value, low effort — buildable today as a plain extension reading the facet registry via the runtime, no core change. With agenda, marginalia, theme-tweaks, and editing-tweaks all stacking decorators on the same rows, "which extension did this" is already a recurring weekly debugging question this would answer directly.

2. Schema lenses — Cambria-style read/write decoupling

Sources: Ink & Switch malleable software essay (Cambria)

Cambria, one of the Ink & Switch research prototypes surveyed in the essay, decouples write-schema from read-schema: each tool writes in its own preferred document shape, and a lens — a declarative schema-transformation layer — projects the document into whatever shape a reading tool expects. Multiple tool versions coexist on one document with no forced migration moment.

KM shape: today every schema change ships as a one-shot idempotent migration eval — project-type-to-flag, tasks-from-dailies, categories→types, wheel→compass — and each carries the same failure mode: stale devices or scripts still writing the old shape after the flag day, plus a window where an old pinned extension reads new data it doesn't understand. A lens layer registered alongside propertySchemasFacet — declarative prop rename/reshape/move rules that decoders (agenda's dataSources.ts, books, bases' evaluate.ts) consult on read, with an optional write-back shim — turns migrations into soft deprecations: ship the lens, migrate lazily or never, retire the lens once telemetry shows the old shape is gone.

Open questions: lens composition order when more than one applies to the same prop, and where lenses should live — block-stored so they sync across devices immediately, or facet contributions so they version with the extension's own code (and are subject to the same install/enable/reload lifecycle, gotcha #2).

3. End-user rewrite rules — a declarative automation layer

Sources: Tote · Nova · Modal · Ink & Switch essay (Potluck)

Three threads from the same rewriting-language milieu share one substrate: programs as user-visible rewrite rules — "when these facts co-occur, replace with those" — evaluated to quiescence. Modal's entire standard library is itself user-space rules, nothing privileged in the kernel. Tote encodes order inside an unordered bag via anchor symbols. Nova gets negation-as-failure for free from rule ordering (a fallback rule only fires if the specific-case rules above it didn't match). Potluck, from the Ink & Switch essay, adds the AI angle directly relevant here: an AI can draft the detector rules, but they persist as inspectable, user-editable artifacts — not opaque model behavior baked into a prompt.

KM shape: a rules extension. Each rule is an ordinary rule-typed block: a match pattern over blocks (type + prop predicates, extending the agenda saved-filter grammar's +tag/!tag family with prop comparisons) plus an effect (set/clear props, add type, refile, create child). An evaluator subscribes to the relevant queries and runs matched effects in repo.tx under ChangeScope.Automation — off the undo stack, the same scope the archivedAt watcher already uses — with a per-rule "ran N times / last run" audit prop and a dry-run preview mode. Example: "task with a past deadline and no snooze → add an urgent tag," or "block tagged +listen → refile under the Listen page."

This is the sharpest genuinely end-user-programmable surface available: today every automation goes through the agent bridge and a hand-authored JS extension; rules would be the first affordance that doesn't. The interesting design surface is the guardrails — termination (an iteration cap, no rewrite-of-own-output), loop detection between rules, and keeping the Automation scope so a runaway rule is fixable by an inverse rule rather than by polluting the undo stack.

4. Assertions with auto-retract — structured ephemeral state

Sources: The Syndicated Actor Model · Spaces of Data

In Syndicate, an actor publishes portions of its own state as assertions into a dataspace; peers subscribe by asserting interest — subscriptions are themselves first-class assertions, so "who is watching what" is queryable data, not a side-channel API call. When an actor or facet terminates, including via crash, all its assertions are atomically retracted — fault tolerance is structural: a crash looks like "assertions retracted," not silence. Facets nest into a tree, giving structured-concurrency-style scoped teardown for free.

KM shape: a small in-memory assertions module shared across extensions, growing directly out of the module-store pattern already used for high-frequency UI state (gotcha #14): publish(scope, fact), where scope ties to a real lifecycle — React unmount, pane close, bridge session disconnect, extension reload — and retraction on scope death is automatic. Candidate users: an agent-bridge session-presence indicator that can never go stale; multi-viewport focus arbitration (the pattern SectionedTaskListView's ownsScope already hand-rolls); cross-view hover channels (see #6); "extension X is mid-operation" indicators.

Value: replaces ad hoc cleanup code and stale-state bugs with one audited lifecycle, and is cheap to prototype since it's purely in-memory — no persistence, no facet system change. A side benefit: interest-as-assertion makes "what's watching this block" itself inspectable, which feeds directly into #1's inspector.

5. Capability-tiered extensions

Sources: Fearless Extensibility Challenge (announcement, thread) and its submissions: capabilities and effects · Draupnir OCap · Smalltalk and object capabilities · modular malleability · Decker (Danger Zone)

The challenge's framing is, almost word for word, KM's own trust-gate problem: powerful customization versus security and maintenance risk, with Firefox's XUL→WebExtensions retreat as the cautionary arc — safety bought by amputating capability. The submissions cluster around object capabilities and effect handlers: authority flowing only through references explicitly handed over (attenuable at each hop), or typed interception of the effects an extension is allowed to perform. Decker ships the pragmatic version already working in production: scripts sandboxed by default, with dangerous surface (file access, etc.) gated behind an explicit opt-in "Danger Zone." Konrad Hinsen's modular-malleability submission adds a structural constraint worth keeping: an extension dependency graph should stay a tree, to bound blast radius.

KM shape: today the per-device pin (upstream #67; gotcha #16) is binary — approved source runs with full power, the whole repo API, every block type, network access. A declared capability manifest per extension (contributes: [facets…], writes: [block types / prop namespaces…], network: yes/no) shown at enable-time, diffed on update ("this update now also wants write access to book:*"), and eventually enforced by handing extensions a wrapped repo scoped to their manifest.

Even an unenforced manifest is a real win: it makes the enable-extension re-approval step (currently a blind re-pin) actually mean something. Enforcement is a bigger lift — proxying the repo handle — but the facet system already centralizes contribution registration, so it's not architecturally foreign. Open question: how an agent authoring a new extension mid-development requests manifest expansion without turning every iteration into a fresh approval prompt.

6. Bidirectional cross-view highlighting

Sources: Ink & Switch essay (Embark)

Embark embeds typed structured objects — places, dates — in an outline and keeps embedded views (map, calendar) bidirectionally synchronized with it: hover a location in the outline and the map pin highlights, and vice versa. The embedded view is a projection of outline data sharing identity with it, not a separate app holding its own copy.

KM shape: this structure already exists in KM — agenda rows, bases table/card rows, timeline bars, and book cards are all projections of blocks that also live in outlines and pages, sharing block ids. A shared hover/peek channel — an in-memory store keyed by block id, either the module-store pattern or built on #4's assertions — that any view extension can publish "hovered/focused" into and subscribe from: an agenda row hover highlights the same task in a visible outline pane and its timeline bar; hovering a bases row flashes the source block. The same channel covers "peek": hold a key on an agenda row and other views scroll-to and halo the block, riding the scroll-to-focused machinery that already exists for focus restoration.

Cheap experiment with outsized cross-extension payoff — no persistence, no schema change, purely an in-memory pub/sub. The design question is throttling: publish/subscribe needs to be opt-in per view so it never taxes a large section, in keeping with the O(rows) discipline already required by unvirtualized agenda sections (gotcha #24).

7. Time-travel — surfacing per-block history

Sources: History as a First-class Citizen · Reversibility

The thread's thesis (Yihan Wu): history should be compulsory and system-level, not something users opt into. vnstore is the reply that grounds it at the storage layer: never mutate in place — timestamp every tuple's creation and retraction so any past state is reconstructible by a timestamp filter. The same thread's Permissive Navigation essay contributes a UI vocabulary: volatile versus pinned panes (a single click opens a replaceable preview, an explicit pin makes it persistent) plus a recents list that can be kept on-screen. The Reversibility thread points at the same shape from a different angle: an invertible-operations log is the right shape for undo — which is what KM's undo stack already is.

KM shape: KM already keeps a full local change log at the block layer — the trigger-written row_events audit table (before/after row JSON on every write, intentionally unbounded) (src/data/internals/clientSchema.ts) — but nothing surfaces it. A block-history inspector view — "what did this task's status/schedule look like on date X," a per-prop change list (when, from-what to-what, which ChangeScope, so Automation writes are distinguishable from user edits) — rides on row_events joined with command_events for scope, no new storage. Two caveats shape the design: the log is client-only (per-device history, never synced — a fresh device has no past), and scope attribution exists only for locally-made writes (sync-applied changes arrive as undifferentiated 'sync'). It pairs naturally with #1's inspector as a second tab. Separately, the Permissive Navigation vocabulary (volatile/pinned panes, visible recents) is ready-made design language worth carrying into the window-management extension's next iteration, independent of the history feature.

Open question: read cost on a block with a very long history — likely needs the same size-capping discipline the container-scale query freeze already forced on subtree() (gotcha #28).

8. Leaf Projectors — the reference model for in-editor block projections

Sources: Hazel Leaf Projectors PR, via the phone-target thread; compositional window management essay, same thread

Hazel's projectors wrap a convex chunk of syntax: the text layer renders an inert atomic placeholder (the cursor treats it as a single tile) while a decoration layer draws the live GUI positioned over it. Multiple projector kinds — fold, live value, text area — implement one common interface and are swappable per-site via keyboard. Structure editing keeps working because the projection is a view over unchanged underlying syntax, never a fork of it.

KM shape: this is the sharpest available architecture for "an editable block that presents swappable live projections" — raw markdown, a rendered widget, a collapsed summary — without losing cursor and selection sanity. It is not buildable in-editor today: CodeMirror internals are unreachable from extensions until upstream CM re-exports land (PR #359, gotcha #27), so this is filed as the reference design for when that lands, not a near-term build. At block-renderer granularity a lite version already exists — the agenda, video-player, and base renderers are each a per-type projection — but Leaf Projectors' contract adds something KM doesn't have: per-instance swappable projection behind one common interface, worth adopting as the target shape whenever the CM blocker clears.

The companion essay from the same thread — containers (tabs, splits, canvas, grid) as arbitrarily nestable primitives, with apps exposing their internal layout structure to the manager rather than each hand-rolling tabs — is separately noted as direct input to the perspectives extension's next iteration, independent of the CM blocker.

Cheaper one-offs

Self-contained HTML export

Lopecode (thread) serializes its entire live runtime — kernel, userspace modules, notebook state — into one dependency-free HTML file, Smalltalk-image-like. A KM version: export a page or agenda snapshot as one offline HTML artifact, in the same spirit as this repo's own docs/*.html design-doc convention. Useful for sharing a view outside KM without granting access to the live workspace.

Glitch-freedom audit

The Data-First movement thread's (thread) sharpest technical point: most reactive-signal implementations leak transient inconsistent intermediate states. A KM version: a test asserting that a multi-block reactive update never exposes an intermediate state to a renderer — loader invalidation counters (AGENTS.md's testing notes) already make this observable. A correctness gut-check, not a feature.

Point-free relation queries

Binate's algebra (thread) — composition, converse ~, transitive closure * over binary relations — is a plausible growth path for the bases/saved-filter grammar: "tasks whose project's track is X" expressed as a composed relation (track ~project) instead of hand-rolled traversal code.

Per-type merge functions

The distributed-as-in-Git thread (thread, via Irmin) attaches domain-specific merge semantics to a data type where last-write-wins sync is the wrong answer. File under "if offline conflicts ever actually bite" — not a current KM pain point.

Where the community validates KM's existing design

Several threads read less like new ideas for KM and more like independent arguments for choices this repo already made:
  • Data-first / no app silos / one shared typed-block substrate: the Data-First thread and the Alan Kay Quora thread's critique of app-sandboxed data (data dying with the app, control ceded to the app's author) is an argument for KM's shared substrate, not against it.
  • akkartik's "unbundle OO into primitives" (thread) — treat namespaces, dispatch, aspects, and registries as independently composable rather than one fused paradigm — reads as a near-literal description of the facet system (renderers, actions, property schemas, keybind contexts as separate contribution points).
  • CSS-style additive override as the good malleability model (On UI as Code's Model 1 fork-and-modify versus Model 2 composable-layer) — KM's facets are Model 2.
  • Aldrich's first-class interoperability (Objects as Records vs Activities) is the formal account of why shared prop contracts across independently-authored extensions work at all — the same property why a stranger's widget can nest inside a stranger's container.
  • Lithification (thread) describes this repo's exact practice — agent-extensions/shared/ compiled by relative import into each consuming bundle rather than published as a dependency, and theme-tweaks' Tweak #8 upstreamed then deleted once core absorbed it — including the thread's unsolved cost: propagating a fix across diverged copies once you've forked instead of depending.
  • Hinsen's data-formats-outlive-APIs argument (from the Rethinking OOP thread) directly supports the vault-mirror/export strategy: a documented byte format survives the program that wrote it; a bespoke API does not.
