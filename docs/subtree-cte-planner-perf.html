Status: proposal (diagnosis verified, fix measured; not yet implemented) · last verified against code: master 51a8d2d7, and empirically against a live production workspace (117,645-row blocks table), 2026-07-13.

SUBTREE_SQL planner miss: every subtree load costs O(table), not O(subtree)

Felt as: holding j/k advances focus at ~5 rows/sec. Actual bug: SQLite builds a transient index over the whole blocks table on every core.subtree execution.

TL;DR. SQLite cannot estimate a recursive CTE's cardinality, so for SUBTREE_SQL's downward join (child.parent_id = subtree.id) it ignores idx_blocks_parent_order and instead builds an AUTOMATIC PARTIAL COVERING INDEX over every live row — on every execution. A 3-row subtree load measures 190.8 ms on a 117k-row table; the identical query with INDEXED BY idx_blocks_parent_order measures 1.6 ms (~120×). A fresh ANALYZE does not change the plan — the fix is a one-line INDEXED BY hint in treeQueries.ts:SUBTREE_SQL, plus a regression test on the query plan.

1 · The symptom, and the measured chain behind it

Navigating a long list with held j/k (key autorepeat, ~30 presses/sec) recently became unusably slow — the cursor advances at roughly 5 rows/sec and keeps crawling long after the key is released. Instrumenting a live client (via repo.metrics() + a scripted probe) attributes the entire budget to one query:

  1. Each press writes focus. focusBlock (src/data/properties.ts) runs one small repo.tx setting focusedBlockLocation + isEditing on the panel's ui-state block. The very first press measures 10 ms tx-to-commit — the write itself is fine.
  2. The panel block sits inside a subscribed subtree. LayoutRenderer.tsx and PanelLayoutProjection.start() both hold a live core.subtree(layoutSessionBlock) handle, and core.subtree registers per-row deps (via hydrateBlocks) for every block in its result — including the panel row the focus write just touched. So every keypress invalidates the layout-session subtree handle (confirmed: loaderInvalidations +1 per press, 10/10).
  3. The re-run is absurdly expensive for its size. The layout-session subtree is 3 live rows (session root + two panels), yet each core.subtree loader run measures ~185 ms (9 runs = 1,668.9 ms during a 10-press burst).
  4. The next press queues behind it. All queries and txs share the one serialized PowerSync connection, so each focus tx waits out the in-flight subtree re-load: writeTransaction timings escalate 26 → 58 → 88 → 120 → 151 → 177 ms across a burst. Steady state is ~190 ms/press; autorepeat at ~30 Hz can never drain the queue.

2 · Root cause: the planner never uses idx_blocks_parent_order

EXPLAIN QUERY PLAN for the shipped SUBTREE_SQL (src/data/internals/treeQueries.ts):

CO-ROUTINE subtree
SETUP
  SEARCH blocks USING INDEX sqlite_autoindex_blocks_1 (id=?)
RECURSIVE STEP
  SCAN subtree
  BLOOM FILTER ON child (deleted=?)
  SEARCH child USING AUTOMATIC PARTIAL COVERING INDEX (deleted=?)   ← the bug
SCAN subtree
USE TEMP B-TREE FOR ORDER BY

An automatic index is a transient index SQLite builds per statement execution. Here it covers every live row of blocks (deleted = 0, ~75k rows of 117,645 total on the measured workspace), so every subtree load — regardless of how small the subtree is — pays an O(table · log table) index build. That is the ~190 ms.

Why the planner hedges: SQLite has no cardinality estimate for the recursive self-reference (SCAN subtree), so it assumes the outer side of the join may be large; amortizing one covering-index build across many probes then looks cheaper than per-probe lookups on idx_blocks_parent_order followed by full-row fetches (SELECT child.*; the partial index is not covering). The estimate is badly wrong for typical subtrees, and the hedge re-runs on every execution.

Ruled out: stale statistics

sqlite_stat1 was indeed stale (it recorded 51,079 rows against an actual 117,645), but a fresh ANALYZE blocks (96.7 ms, stats now current) does not change the plan — same automatic index, 240.2 ms after re-analyze. The recursive-CTE cardinality blind spot, not statistics, drives the choice. So there is no ops-side fix; the SQL must pin the index.

Why it surfaced "recently"

The cost scales with total live rows, not subtree size. The measured workspace roughly doubled its blocks table in recent large imports; every core.subtree call site (page open, copy, navigation, layout projection) has been paying ~200 ms per load, and the layout-session subscription wires that cost to every keypress, which is where it finally became impossible to ignore. This planner behavior also plausibly contributed to earlier reports of container-scale subtree() loads starving the connection.

3 · Proposed fix

Pin the join index in SUBTREE_SQL:

  WITH RECURSIVE subtree AS (
    SELECT *, '!' || hex(id) || '/' AS path, 0 AS depth
      FROM blocks
     WHERE id = ? AND deleted = 0
    UNION ALL
    SELECT child.*, subtree.path || child.order_key || '!' || hex(child.id) || '/', subtree.depth + 1
      FROM subtree
-     JOIN blocks AS child ON child.parent_id = subtree.id
+     JOIN blocks AS child INDEXED BY idx_blocks_parent_order ON child.parent_id = subtree.id
     WHERE child.deleted = 0
       AND subtree.depth < 100
       AND INSTR(subtree.path, '!' || hex(child.id) || '/') = 0
  )
  SELECT * FROM subtree ORDER BY path

Plan after (verified live): SEARCH child USING INDEX idx_blocks_parent_order (parent_id=?) — no bloom filter, no automatic index. Measured on the same workspace, same 3-row subtree: 190.8 ms → 1.6 ms.

Safety of INDEXED BY

  • The index is created by our own client DDL (src/data/blockSchema.ts — CREATE INDEX IF NOT EXISTS idx_blocks_parent_order ON blocks (parent_id, order_key, id) WHERE deleted = 0), so it is guaranteed present on every client that can run this query. INDEXED BY fails loudly (prepare-time error) if the index is ever renamed/dropped — which is the failure mode we want, caught immediately by any test that runs the query.
  • INDEXED BY does not change semantics, only plan choice; the partial-index predicate (deleted = 0) is implied by the query's own child.deleted = 0.
  • Results are unchanged (same rows, same ORDER BY path).

Regression test

Alongside the existing treeQueries.test.ts coverage, assert the plan itself so a future schema/SQL edit can't silently regress:

const plan = await db.getAll(`EXPLAIN QUERY PLAN ${SUBTREE_SQL}`, [rootId])
const details = plan.map(r => r.detail).join('\n')
expect(details).toContain('USING INDEX idx_blocks_parent_order')
expect(details).not.toContain('AUTOMATIC')

4 · Audit: other recursive CTEs are safe

Every other WITH RECURSIVE over blocks in core walks upward (ON blocks.id = chain.parent_id and variants) — a primary-key lookup per step, which the planner handles fine. Only the downward parent→children join hits the blind spot:

SiteJoin directionVerdict
treeQueries.ts — SUBTREE_SQLdownward (child.parent_id = subtree.id)affected — this fix
treeQueries.ts — ANCESTORS_SQL, many-ancestors, depth walkupward (PK lookup)safe
kernelQueries.ts — ancestor walk (walk CTE)upward (PK lookup)safe
kernelQueries.ts — childrenOfflat WHERE parent_id = ?safe (direct index use)
typedBlockQuery.ts — ancestor-chain CTEupward (PK lookup)safe
grouped-backlinks/query.ts — ancestor_chainupward (PK lookup)safe

Rule of thumb going forward: any new recursive CTE that joins blocks downward on parent_id needs the INDEXED BY hint (or a plan-asserting test); check EXPLAIN QUERY PLAN for the string AUTOMATIC.

5 · Secondary observation (separate follow-up, not part of this change)

Even at 1.6 ms, re-running the layout-session subtree on every focus keystroke is busywork: the focus write only changes focusedBlockLocation/isEditing on a panel row, which the layout projection and LayoutRenderer never read — they consume structure, topLevelBlockId, and the panel-stack type. The per-row deps from hydrateBlocks are property-name-blind, so any UiState property write on any row of any subscribed subtree re-runs the whole query.

With the index fix in place this is no longer a performance problem, so it's flagged here only as a design note: if per-row deps ever grow property-name granularity (as the narrow-dep channels in kernelInvalidation.ts already have for typedBlocks.property), the layout subscription is the first consumer that would benefit. Not proposing anything concrete in this doc.
