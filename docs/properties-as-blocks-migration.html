Properties as blocks — rev 2

Migrate user-defined property values from the properties_json cell to first-class block children ("field rows" + "value children"), keeping the cell as a derived projection. Re-grounded against the current data layer and the working prototype in PR #285. Supersedes rev 1 (PR #55).

Status: current (planning). Rewritten 2026-07-01 against master 8a6f22cd and the spike merge in PR #285 (props-as-childrent-spike-merge, d33393a9). Rev 1 (PR #55, May 27) targeted a pre-Layout-B data layer and is superseded by this document; §3 lists every rev-1 claim that no longer holds. Mechanism descriptions in §5 describe code that exists on the spike branch, not on master. Last verified against code: 2026-07-01.

1. Why move

Property schemas are blocks ('property-schema' type, projected into the schema registry by userSchemasProjector). Property values are entries in a JSON cell — the properties_json column on the consuming block, read through block.get(schema) and written through tx.setProperty. Schemas get the full block treatment — refs, comments, history, sync, undo. Values get a row-local property bag with custom plumbing for every operation that needs to cross the bag boundary.

The asymmetry leaks into a half-dozen features. Copy-paste serializes blocks and copies the JSON cell separately — round-tripping through markdown loses property data entirely. Subtree-sharing has to special-case the cell. The Roam importer writes schemas and values through different code paths. Comments on a field value, references to a specific field's setting, embedding a single field elsewhere — all require new surface every time.

Since rev 1, the sync-layer motivation has sharpened. PR #52 made the upload pipeline column-narrow (apply_block_patches): a content edit no longer clobbers a concurrent property edit. What remains is per-key last-write-wins inside properties_json itself — client A sets foo, client B sets bar, both PATCH the same column, one key silently disappears. docs/field-level-sync-merge.md designed a server-side per-key merge for exactly this, and put it on hold in favor of this project: each round of review surfaced another way per-key merge semantics fight the row-LWW + cache-acceptance + invalidation pipeline. Properties-as-children dissolves the problem structurally — each field is its own row, and rows already merge per-column. This document is now the only live plan for closing that gap.

The motivating reframe stands: the user-facing properties UX is "fields are children of the block, rendered with structure" — that's what every Tana-style editor does. We've been emulating it with a sidecar panel because the data model said no.

2. The target shape

A property on a block becomes a field row (rev 1 called this a "carrier"): a regular block whose content is a reference to the property schema, with a denormalized reference_target_id column carrying the schema's stable fieldId. The field row's children are the property's values — one for scalars today, many for lists later; ref-typed values are themselves block references. Same primitive everywhere.

Things to notice, updated from rev 1:

  • Field-row content is [[Schema Name]], not ((schema-id)). The spike made this call and it's the right one — see §7 for the tradeoff. Detection never parses content at read sites; it reads the reference_target_id column, derived in the same transaction by a processor.
  • Identity is PropertySchema.fieldId: the 'property-schema' block's row id for user-defined schemas, a deterministic property:<name> string for kernel/plugin schemas. name stays the display label and the projection key in the cell; renames change name without touching stored children.
  • Scalar-first. The prototype projects one primary value child per field row; list values serialize as JSON into that one child for now. Multi-value-as-N-children is a follow-up slice (§11), not part of the first cut. This is a deliberate narrowing from rev 1, which promised list-cardinality-as-children up front.
  • Ref-typed values reuse the block-reference primitive: a value child whose content is ((target-id)) is the value and the backlink edge in one shape.

4. Why this shape, not another

Alternatives, re-scored against the current pipeline:

Approach Refs & comments on fields Per-key conflict story Read-path cost Verdict
Stay on JSON cell no per-key LWW clobber unchanged doesn't solve the motivating problems
Server-side per-key JSON merge (field-level-sync-merge.md) no merged unchanged explored in depth; on hold — merge semantics fight row-LWW + cache-acceptance (applyIfNewer) + derived-references staleness; none of the block-y affordances
CRDT-typed JSON cell (Yjs/Automerge) no field-grained unchanged layering nightmare with PowerSync + e2ee envelopes; save Yjs for content
Children + new projection table yes row-per-field new compiler path more code than needed; queries and hot reads already speak properties_json
Children + cell kept as derived projection yes row-per-field unchanged chosen — and prototyped in PR #285

The chosen shape mirrors the existing references[] pattern: the source of truth lives where the user writes (children for properties, content markdown for references), and same-tx machinery maintains a denormalized cell that downstream consumers (typed queries, block.get, panel, search predicates) already read. The same-tx processor registry this relies on (defineSameTxProcessor, commit-pipeline step 3.5, core.normalizeReferences as prior art) exists on master and is exactly the seam rev 1 hoped for.

5. Mechanism — as built in the spike

Three same-tx processors plus a dual-writing tx.setProperty. All of this exists on the spike branch with tests; names below are the real ones.

core.deriveReferenceTarget (permanent)
Watches content. When a row's content trims to exactly one reference token (((uuid)) or [[alias]]), resolves it — schema-name map first, then the tx-aware alias lookup — and writes the target id into reference_target_id in the same transaction (skipMetadata: bumps the sync clock, not "last edited"). Detection everywhere downstream is a column read, never a content parse.
core.projectPropertyChildren (permanent)
Watches content / referenceTargetId / parentId / orderKey / deleted. For each changed row, walks up at most two levels to find affected (parent, fieldId) pairs — both before and after sides of a move — and rebuilds that parent's properties_json key from the field row's value children. This is what keeps manual tree edits of field/value rows convergent with the cell.
core.materializePropertyChildren (permanent, direction: cell → children)
Watches properties. For changed keys with a registered schema: find-or-create the field row + primary value child, update stale content, delete children for removed keys, collapse duplicates. This keeps raw cell writes (importers, plugins, old clients' synced rows via the catch-up pass) convergent with the children. Unlike rev 1's plan, this is not deletable scaffolding — it is the permanent convergence path for the cell-write direction.

Dual-write: tx.setProperty(id, schema, value) writes the field/value children and the cell in one transaction. Rev 1 wanted "write children, let a processor derive the cell"; the spike's shape is better on the hot path — no processor-ordering dependency for the common write, readers (block.get, typed queries, the panel) stay synchronous against the cell, and idempotent processors turn the round-trip into no-ops. Loop-avoidance is unchanged from rev 1: every processor's write is skipped when output already equals input.

Sync arrival

Sync-applied rows never pass through repo.tx, so processors don't fire on them. The spike hooks the Layout-B observer: onRowsAccepted(snapshots) fires once per drain window after applySyncInvalidation, and the repo schedules a row-targeted materialize pass (cell → children) for the accepted rows, through the serialized backfill queue. Because every current-code writer dual-writes, a remote value-child edit always arrives together with its updated cell; the catch-up pass exists for rows written by older clients (cell-only writes) and for backfill convergence. This invariant — no client writes children without the cell, or the cell without children materialized behind it — is what lets the receiving side avoid a full re-projection pass on every drain window. State it in code comments; it's load-bearing.

Denoted-value rule (unchanged, implemented). Mutations deep in a value child's subtree — a comment under a long-text value — do not touch the parent's cell. The projection watches field rows and their direct value children only. Without this rule the processor becomes a reactive query engine and write amplification explodes.
Upload reality. Every write above uploads (column-narrow, but it uploads): field/value child rows are ordinary synced rows; the derived cell write ships the properties_json column. There is no "derived, don't upload" tag and we shouldn't build one for this — peers need both sides. Budget roughly 2 extra rows per (block, property) pair on first materialization plus one cell-column PATCH per property write. Backfill of a large workspace is a real upload burst (§11, slice C).

6. The scope decision: which schemas get children

This is the fork in the road, and the spike and the standing docs disagree. properties-model-evolution.md (May 8) chose A + D: children for user-defined fields, cache-only for system fields. The spike flipped it to A only — every registered schema materializes, including types, system:collapsed, system:aliases, and the property-schema:* config props on schema blocks themselves — arguing that a carve-out "preserves two write models and makes 'properties are children' false in the kernel/plugin paths that most need to exercise the invariant."

Running the spike surfaced what A-only costs in practice:

  • Row and write amplification on hot kernel props. Collapsing a block writes a [[system:collapsed]] field row + value child (2 synced rows) the first time, and a value-child content update on every toggle after. Type-tagging writes [[types]] rows behind every typed block. These are the highest-frequency property writes in the system, and they gain none of the block-y affordances — nobody comments on their collapse state.
  • Tree pollution as default. Field rows are ordinary children, so every consumer of tree shape sees them: 8 master test suites fail on tree-shape assertions; panelLayoutProjection and the orphan-alias cleanup needed field-row special cases; the outline needs a per-schema hidden UI flag plus a "Show Hidden Fields" toggle just to look normal.
  • UiState props live on dedicated state rows (per-user ui-state children under the user page) — materializing focusedBlockLocation or editorSelection as grandchildren of a state row is pure noise, and their write rate is the highest in the app.
  • Synthetic fieldIds leak. Kernel schemas' property:<name> pseudo-ids aren't block ids, but the references processor's exact-reference branch happily projects them into references_json, putting non-existent target ids into the backlink index.

The counterargument deserves a real answer, though. The spike is right that a hardcoded "system fields" set is a smell, and right that kernel paths exercising the invariant is worth something. The resolution:

Recommendation: make child-backing a per-schema property, not a global invariant. Add childBacked: boolean to PropertySchema (alongside fieldId). Defaults: true for user-defined schemas built from 'property-schema' blocks; false for kernel and plugin schemas — any of which can opt in explicitly (a plugin field that wants comments/refs on its values is one defineProperty flag away). ChangeScope.UiState schemas are never child-backed. All machinery (dual-write, three processors, backfill) branches on the flag in exactly one place each; there is still one write model (setProperty), one detection rule, one projection. What varies is data placement per schema — which is a schema property, the same way changeScope already is.

This restores master's tree-shape expectations for kernel props (the 8 failing tests pass unmodified), keeps collapse/type hot paths at today's write cost, keeps state-row subtrees clean, and keeps pseudo-ids out of the references index — while every benefit in §1 still lands, because the benefits were always about user-defined fields. It also un-blocks the UIState question PR #285 explicitly left open ("should UIState props materialize?" — no).

types is worth a special note: it already has a denormalized local index (block_types) and kernel mutator integration (TypeTagger). If types-as-children is ever wanted (type tags as referenceable rows), that's its own design with its own doc; don't get it as a side effect of this migration.

7. Field-row content: [[name]] + derived column

Rev 1 specified field-row content as ((schema-id)). The spike ships [[Schema Name]] with the resolved fieldId in the synced reference_target_id column, derived same-tx from content. Keep the spike's shape:

  • Markdown round-trip is a headline motivation — and [[priority]] exports as meaningful markdown where ((7f3a…)) doesn't. Copy-paste into another workspace can resolve by name (§9); a uuid never resolves.
  • Renames don't corrupt data. The column, not the label, is authoritative; a stale [[old name]] label is display drift, repaired opportunistically by the materialize processor (it rewrites field-row content when it doesn't match the schema's current name) and by the rename-reproject pass (§11 slice E).
  • Typing a field by hand works. A user (or an importer, or an agent) can type [[status]] as a child and put a value under it; the derive processor resolves the alias to the schema and the project processor folds it into the cell. That's the "it's just blocks" promise made real.
  • The cost: name collisions need a resolution rule (registered schema map wins over generic alias lookup — implemented), and the exact-reference grammar is scoped to whole-block content so prose containing ((…)) never becomes a field (implemented in referenceBlock.ts).

8. E2EE workspaces — new constraint

E2EE workspaces (May 29) encrypt content, properties_json, and references_json into enc:v1 envelopes; a server trigger rejects plaintext in those columns. Reference edges are therefore part of the encrypted surface today. A synced plaintext reference_target_id would leak, per row: "this row is a field instance", "of this schema", and for exact block-ref rows "this row points at that row" — precisely the edge metadata the current design encrypts.

Options, in increasing effort:

  1. Gate: e2ee workspaces stay cell-only for now. The childBacked machinery checks the workspace's encryption mode and behaves as if every schema were cell-backed. Zero leak, zero new code beyond the gate. E2EE users keep today's behavior. Recommended for the first ship.
  2. Derive locally, never upload. Exclude the column from BLOCK_UPLOAD_COLUMNS for e2ee workspaces and run the derive pass on sync arrival (the observer materializes decrypted plaintext locally, so content is available; onRowsAccepted is the hook). Needs a first-sync full derivation pass per device (marker-gated, same pattern as the backfill). This is the real fix; it's mostly plumbing that reuses existing machinery.
  3. Encrypt the column. Defeats the point — the column exists to be an indexable, queryable detection key on the client; the server never queries it. (2) achieves the same privacy with less ceremony.

Related server-surface loose ends whichever option lands: the apply_block_patches RPC's column whitelist doesn't carry reference_target_id (the spike didn't extend it — verify the upload envelope path for the new column end-to-end), and the spike's Supabase migration is timestamped 20260527123000, which sorts before already-applied migrations (20260612…, 20260625…) — rename it to a current timestamp before any db push.

9. Detection rules and edge cases — as implemented, with divergences

Field-row recognition.
A child of P is a field row iff reference_target_id equals a registered schema's fieldId. Column read, no content parse. Unregistered targets render as plain reference blocks — a pasted field from another workspace degrades to a visible, honest "unknown field" row.
Value decoding.
Spike: first parseable value child wins (order-key order); unparseable children are skipped. If no child parses, the projection removes the key from the cell. Rev 1 said "log + omit, keep data visible" — the spike keeps the rows (data is never destroyed) but the cell-side key vanishes, which downstream readers see as "unset". Acceptable; document that "field with only-invalid values" reads as unset while remaining visible/fixable in the tree.
Duplicates.
Divergence from rev 1 ("keep extras, pick lowest orderKey"): the materialize path deletes duplicate field rows and duplicate value children (soft-delete, so recoverable via history). Projection reads the first field row's first parseable value. The delete-duplicates behavior is the better call under sync races — two devices materializing concurrently self-heal to one row — but it means "N children under one field row" is reserved for the future multi-value slice and must not be user-visible affordance until then.
Default-value vs. unset.
Unchanged from rev 1: zero value children ⇒ key omitted from the cell ⇒ block.get falls back to defaultValue. No reader change.
Ref-typed values → parent references[].
Two paths exist now: (a) the cell-side projection (projectPropertyReferences, sourceField = schema.name) still runs off the bag — unchanged, keeps backlinks-by-field working; (b) field rows and value rows are themselves reference-bearing blocks, so the references plugin indexes them as ordinary rows — meaning a user field's schema block accrues backlinks from every block that uses the field ("where is this field used?" for free). Verify the two paths don't double-count in backlink UIs (dedup key is (sourceField, id); path (b) rows carry sourceField: '' on their own row — the parent's entry stays single).
Merge, orphan cleanup, panel layout.
Implemented in the spike: mergeBlocksInTx reparents only non-field children and re-materializes the merged bag; orphan-alias cleanup deletes generated field rows under alias seats; panel-layout projection ignores field rows. Under the §6 scoping these special cases shrink (kernel props stop generating field rows) but the merge rule stays — user-field rows must not be reparented as content.
Type-add field slots vs ghosts.
Divergence, decision needed. The spike materializes empty field rows ("slots") when a type is added (TypeTagger → materializePropertyFieldSlotsForExistingRow). Rev 1 argued for render-only ghost rows that materialize on first edit — no storage until data exists, nothing to clean up on type removal. The spike has no slot cleanup on type removal, so removing a type strands empty [[field]] rows. Recommendation: render-only ghosts; drop the TypeTagger materialization. If empty slots prove necessary (e.g., for cross-device consistency of field order), add explicit cleanup on type removal.

10. UI: in-tree rendering exists; the panel is not sunset

The spike ships FieldBlockRenderer (field-row line: shape glyph + label linking to the schema block; value editing via the preset's editor; empty-value editor for slot rows) and PropertyValueBlockRenderer, registered in the default renderer chain, plus hidden-field filtering in useChildIds and a "Show Hidden Fields" context-menu toggle. The property panel remains fully functional alongside — both surfaces read the same cell and write through the same setProperty, so they coexist safely during the transition. Panel sunset is deliberately last (§11) and follows usage, not a schedule. Rev 1's inventory of what survives (value presets, schema registry, resolvePropertyDisplay, PropertySchemaBlockRenderer) matches what the spike actually reuses.

11. Shipping plan — extract slices from the spike, don't land the merge

PR #285 proved the design end-to-end, but as a shippable unit it has structural problems: it's a 37-commit merge branch that is already in conflict with master again (five files, including one that git treats as binary), it carries the un-decided A-only scope, and its test suite disagrees with master's by design. Treat it as the reference implementation and land master-first slices extracted from it. Each slice is independently shippable and reversible.

A
Foundations: column + derive processor

reference_target_id through all three layers (Supabase migration — re-timestamped; BLOCK_STORAGE_COLUMNS/parse/upload; BlockData.referenceTargetId; regenerate sync-config), referenceBlock.ts grammar, core.deriveReferenceTarget, the partial index, e2ee gate decision from §8 (recommended: derive skips e2ee workspaces for now, column excluded from their upload). Also the standalone prerequisite fix: replace the literal NUL bytes in src/data/api/blockData.ts with escapes so git stops treating it as binary (it silently corrupted two auto-merges in the spike merge already).

No behavior change for existing data — the column stays NULL except on exact-reference rows. Reversible by dropping the processor.

B
Child-backed write path, scoped

PropertySchema.fieldId + childBacked (§6 defaults), dual-writing setProperty, core.projectPropertyChildren + core.materializePropertyChildren gated on the flag, childrenOf option, merge/orphan/panel-layout handling. Because kernel/plugin schemas default to cell-only, master's tree-shape tests pass unmodified — the spike's 8 failures don't need porting decisions.

Every user-defined property write is child-backed. Rollback: flip user-schema default to cell-only; already-written children stay inert (project processor keeps them convergent), cleanup script optional.

C
Backfill + sync catch-up

The spike's backfill machinery (markers, candidate scan, bulk insert, retry) and the observer onRowsAccepted catch-up, scoped to child-backed schemas. Sequence the first production run deliberately: it's an upload burst of 2 rows per (block, property) — run workspace-by-workspace, watch ps_crud drain. Before flipping it on, size it: run select count(*) from blocks where properties_json != '{}' and deleted = 0 (and a key histogram) against production via yarn agent sql / Supabase.

100% of user-defined property data is child-backed; cell fully derived. Fall-back per block unchanged from rev 1: on backfill failure the cell stays authoritative for that block.

D
In-tree UI

FieldBlockRenderer / PropertyValueBlockRenderer, hidden-field filtering, ghost rows for type-expected fields (replacing the spike's materialized slots), add-field affordance on the block (slash command → schema autocomplete → field row + value in one tx). Panel stays.

Users see and edit user-defined properties in the tree. Panel remains the fallback; no flag needed since both surfaces coexist.

E
Rename reproject + polish

Wire reprojectPropertyValueChildren (exists, currently untriggered — the spike lost the auto-trigger when the stateful schema service became a projector) to a projector post-rebuild hook: on user-schema rebuild, diff old/new names per fieldId and reproject. Fix the brittle kernelQueries query-plan assertion. Multi-value-as-children and block-valued fields get their own slice when a concrete UX needs them.

Renames converge cell keys and field-row labels. Remaining spike deltas are accounted for.

F
Panel sunset (later, usage-driven)

When the in-tree surface has clearly won: delete BlockProperties.tsx, propertyPanel/*, AddPropertyForm. E2EE derive-locally path (§8 option 2) lands here or earlier if e2ee users want child-backed fields sooner.

One rendering surface. The processors stay — they're the convergence machinery, not scaffolding.

12. Risks to monitor

  • Write amplification at parents. Same shape as rev 1, now measurable: the spike added per-processor timing to the commit pipeline (TxTimingDiagnostics.sameTxProcessorRuns). Watch it during slice B.
  • Backfill upload burst per device. Completion markers are local (client_schema_state), so every device scans; the NOT-EXISTS candidate filter makes re-materialization a no-op once one device's rows sync down, but two devices racing before convergence briefly duplicate field rows (self-healing via duplicate deletion — verify with a two-client test). Consider holding the first full backfill until the initial sync drain completes, not just schema readiness.
  • Schema-registry timing. Derivation quality depends on which schemas are registered when a pass runs. The spike's gating (backfill held until projector publish, generation-coalesced rescans, row-targeted catch-up bypassing markers) is the answer; keep its tests.
  • findSchemaByFieldId is O(schemas) per lookup and sits in child-filtering paths that run per rendered row. Fine at today's schema counts; add a fieldId-keyed map alongside the facet if profiles say otherwise.
  • Old-client interop during rollout. A device on pre-slice-B code writes cell-only; current devices converge it via the sync catch-up pass. But an old device editing a property whose children already exist produces a cell/child divergence resolved by materialize-on-arrival — last-writer-wins per key, children follow. Coordinated upgrade (small fleet) makes this a non-issue; note it for any future where it isn't.
  • Undo/replay. Same-tx processors are skipped on undo/redo replay by design (snapshot restoration is authoritative). Backfill writes are system-scoped and undo-free (tested in the spike). Verify undo of a value-child edit restores both the child and the projected cell (they're one tx, so snapshots cover both — the existing test should assert it explicitly).
  • Doc drift. field-level-sync-merge.md points at docs/properties-as-children.html, which never merged — it means this document. properties-model-evolution.md's Thread-2 decision section is superseded by §6 here (the spike edited it to "A only"; this doc re-opens that to per-schema scoping). Re-stamp both when slices land.

Open questions for the slice PRs: exact ordering of field rows among children (the spike inserts at keyAtStart — field rows first; rev 1 wanted carriers-then-content; pick one and make materialization respect it); whether ghost rows need a per-type declared order; the add-field slash-command UX; whether types ever becomes child-backed (out of scope here, §6); and when to run the e2ee derive-locally path (§8 option 2).
