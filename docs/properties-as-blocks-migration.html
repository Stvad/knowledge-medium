Decisions made in this doc (each argued in its section):
  • Children are the only property truth that crosses sync. The cell is never a merge input for child-backed keys; arrival re-projects from children (per-row LWW = per-key merge). The cell survives as local read surface, bootstrap hint, and carrier for non-child-backed keys (§5).
  • Materialize everything — all schemas, all scopes — kernel, plugin, and user alike, with system/machine field rows hidden by default and orphan keys promoted to synthesized definitions. No permanent cell-backed category: partial migration would keep the column (and its sync/RPC/e2ee surface) alive forever. The flip lever is a per-workspace column, not a per-schema flag (§6).
  • Field-row shape: [[Schema Name]] content + derived synced reference_target_id, as the spike built it (§7).
  • setProperty dual-writes children and cell in one tx; the materialize/project processor pair is permanent convergence machinery (§5).
  • The arrival pass becomes a reconcile: project-from-children for current-client rows, materialize only for old-client convergence, dedup at parents (§5, slice C).
  • Stopping cell sync is a staged, measurement-gated optimization, per-workspace once backfill-complete and sweep-clean — see the tiered real-DB measurement plan and decision rule; the column is ultimately droppable (§11, cell end-state). The flip itself (slice C, all schemas) is what closes the collapsed clobber — §6 correctness, not optimization.
  • E2EE workspaces stay cell-only until the derive-locally path lands (§8).
  • Machinery inserts field rows first among children (orderKey stays user-owned); render-only ghosts replace the spike's materialized empty slots (§9).
  • Ship as master-first slices extracted from the spike; dormant-flag landing is the fallback; PR #285 is the reference implementation (§11).

1. Why move

Property schemas are blocks — since schema unification, all of them: user-authored 'property-schema' blocks projected into the registry, and kernel/plugin schemas seeded as definition blocks with deterministic per-workspace ids. Property values are entries in a JSON cell — the properties_json column on the consuming block, read through block.get(schema) and written through tx.setProperty. Schemas get the full block treatment — refs, comments, history, sync, undo. Values get a row-local property bag with custom plumbing for every operation that needs to cross the bag boundary.

The asymmetry leaks into a half-dozen features. Copy-paste serializes blocks and copies the JSON cell separately — round-tripping through markdown loses property data entirely. Subtree-sharing has to special-case the cell. The Roam importer writes schemas and values through different code paths. Comments on a field value, references to a specific field's setting, embedding a single field elsewhere — all require new surface every time.

Since rev 1, the sync-layer motivation has sharpened. PR #52 made the upload pipeline column-narrow (apply_block_patches): a content edit no longer clobbers a concurrent property edit. What remains is per-key last-write-wins inside properties_json itself — client A sets foo, client B sets bar, both PATCH the same column, one key silently disappears. docs/field-level-sync-merge.md designed a server-side per-key merge for exactly this, and put it on hold in favor of this project: each round of review surfaced another way per-key merge semantics fight the row-LWW + cache-acceptance + invalidation pipeline. Properties-as-children dissolves the problem structurally — each field is its own row, and rows already merge per-column. This document is now the only live plan for closing that gap.

The motivating reframe stands: the user-facing properties UX is "fields are children of the block, rendered with structure" — that's what every Tana-style editor does. We've been emulating it with a sidecar panel because the data model said no.

2. The target shape

A property on a block becomes a field row (rev 1 called this a "carrier"): a regular block whose content is a reference to the property schema, with a denormalized reference_target_id column carrying the schema's stable fieldId. The field row's children are the property's values — one for scalars today, many for lists later; ref-typed values are themselves block references. Same primitive everywhere.

Things to notice, updated from rev 1:

  • Field-row content is [[Schema Name]], not ((schema-id)). The spike made this call and it's the right one — see §7 for the tradeoff. Detection never parses content at read sites; it reads the reference_target_id column, derived in the same transaction by a processor.
  • Identity is the definition block id. fieldId = the 'property-schema' block's row id for every schema, kernel and plugin included: schema unification seeds code-declared schemas as definition blocks with deterministic per-workspace ids (uuidv5(workspaceId:seedKey)), and identity reaches write paths only through its tx/workspace-bound resolution primitive (Repo.propertySchemaResolverFor(workspaceId) → branded ResolvedPropertySchema; code handles carry seedKey, never a fieldId). The spike's synthetic property:<name> ids are superseded and never ship — extraction swaps them for resolved ids (§11, slice B). name stays the display label and the projection key in the cell; renames change name without touching stored children.
  • Scalar-first. The prototype projects one primary value child per field row; list values serialize as JSON into that one child for now. Multi-value-as-N-children is a follow-up slice (§11), not part of the first cut. This is a deliberate narrowing from rev 1, which promised list-cardinality-as-children up front.
  • Ref-typed values reuse the block-reference primitive: a value child whose content is ((target-id)) is the value and the backlink edge in one shape.

5. Mechanism — as built in the spike

Three same-tx processors plus a dual-writing tx.setProperty. All of this exists on the spike branch with tests; names below are the real ones.

core.deriveReferenceTarget (permanent)
Watches content. When a row's content trims to exactly one reference token (((uuid)) or [[alias]]), resolves it — schema-name map first, then the tx-aware alias lookup — and writes the target id into reference_target_id in the same transaction (skipMetadata: bumps the sync clock, not "last edited"). Detection everywhere downstream is a column read, never a content parse.
core.projectPropertyChildren (permanent)
Watches content / referenceTargetId / parentId / orderKey / deleted. For each changed row, walks up at most two levels to find affected (parent, fieldId) pairs — both before and after sides of a move — and rebuilds that parent's properties_json key from the field row's value children. This is what keeps manual tree edits of field/value rows convergent with the cell.
core.materializePropertyChildren (permanent, direction: cell → children)
Watches properties. For changed keys with a registered schema: find-or-create the field row + primary value child, update stale content, delete children for removed keys, collapse duplicates. This keeps raw cell writes (importers, plugins, old clients' synced rows via the catch-up pass) convergent with the children. Unlike rev 1's plan, this is not deletable scaffolding — it is the permanent convergence path for the cell-write direction.

Dual-write: tx.setProperty(id, schema, value) writes the field/value children and the cell in one transaction. Rev 1 wanted "write children, let a processor derive the cell"; the spike's shape is better on the hot path — no processor-ordering dependency for the common write, readers (block.get, typed queries, the panel) stay synchronous against the cell, and idempotent processors turn the round-trip into no-ops (the callout below states the invariants that make that safe).

Merge semantics: children are the only truth that crosses sync

This subsection is the heart of the design, and it is a correction — the spike gets it wrong, and rev 2 as first reviewed inherited the mistake. The requirement: the cell is never a merge input for child-backed keys. Children are the only property state that crosses the sync boundary as truth.

Why: watch two current, fully-dual-writing clients clobber each other under the spike's arrival handling. Client A sets status on block P; client B concurrently sets priority. Each uploads its value-child row plus P's full re-projected cell. The child rows are separate rows — row-level LWW keeps both, which is exactly the per-key merge we want. But properties_json is one column carrying the whole bag, so column-LWW picks one writer's cell wholesale — say B's, which carries a stale copy of status. Then the spike's catch-up runs materialize (cell → children) over the arrived parent on every client, sees cell status: "old" vs child "done", "repairs" the child to match the stale cell, and uploads that. A watches its own edit revert, fleet-wide. The dual-write invariant holds for each writer individually; the server's column-vs-row LWW recombines A's children with B's cell, and cell-authoritative arrival then enforces the wrong winner. Effective merge semantics: whole-cell LWW — the exact failure this migration exists to fix.

The fix is to flip arrival authority. On sync arrival, the incoming cell's child-backed keys are ignored; those keys are re-projected from local children, which converge per-row. The cell's roles shrink to exactly three, none of them merge:

  • Local read/query surface — block.get and json_extract queries keep their sync, free reads; the value they read is locally derived (same-tx by the writer, arrival-hook for remote changes).
  • Bootstrap hint — a fresh device reads downloaded cells before its projection pass has swept each block. A hint can be per-key stale (it's whichever writer's bag last won column-LWW); the projection pass corrects it. Bounded, self-healing — and it applies to json_extract queries on a fresh device too, not just reads.
  • Carrier for not-yet-migrated keys — before a workspace flips (and, transitionally, for keys whose backfill hasn't reached a block) the cell keeps today's cell-LWW semantics. Under §6 this is a transitional role only: no key category stays cell-backed permanently, so post-backfill the cell is pure derived projection everywhere.

Convergence is two-layered. Locally: projection is deterministic, so once the child rows have arrived, every client derives a byte-identical cell. Server-side: the stored cell re-converges the next time anyone writes a property on that block (their upload carries a freshly-projected bag); between writes it may sit per-key stale, and that's fine because nothing consumes it except the bootstrap-hint path. Deletes need no special case among current clients: a property delete is a child soft-delete row, which syncs per-row and re-projects on arrival like any other child change.

The flip must be total. Materialize-on-arrival (cell → children) must run only where the cell is genuinely the newer truth: rows written by old, pre-migration clients (cell-only writers) and backfill convergence. Any path that lets an arriving cell overwrite existing children for a dual-writing fleet re-opens the clobber through the back door — this is precisely what the spike's current row-targeted catch-up does and the single most important thing to change when extracting it.

Sync arrival — mechanics

Sync-applied rows never pass through repo.tx, so processors don't fire on them. The spike hooks the Layout-B observer: onRowsAccepted(snapshots) fires once per drain window after applySyncInvalidation, and the repo schedules a row-targeted pass over the accepted rows through the serialized backfill queue. One ordering requirement that hook placement misses (PR-review P2): firing after invalidation means the raw column-LWW bag is reader-visible between materialization and the pass's repair — a stale-read window on every concurrent remote edit among current clients, not just the upload stop's frozen-cell case. The cell repair for child-backed keys (re-project from local children) therefore runs inside the materializer seam, before the affected invalidations fan out — the same single sync-write seam the §11 upload stop already modifies; the heavier reconcile work below (dedup, checksum branches, the materialize direction) stays in the async pass. As built in the spike that pass runs materialize (cell-authoritative — wrong per the callout above). As specified here it becomes a reconcile pass, per arrived row:

  • Arrived field/value rows → walk up to the parent, re-project its child-backed keys from local children, and dedup duplicate field rows at the parent while there.
  • Arrived parent cells → classify the cell first (the discriminator in the callout below). A current-client cell (projection checksum matches) is never a merge input: re-project child-backed keys from local children; apply the cell only for keys that are not child-backed. Child-backed is per-client actionable here (PR-review P2): a key whose schema this client cannot build — a custom-codec plugin schema without its plugin, a user schema on an absent plugin preset — is treated as cell-backed on this client: the arriving cell applies, and this client's own projections never cover it (no codec ⇒ it can neither decode children nor re-project; ignoring the cell would replace its only usable value with an empty local projection). Fleet-wide the key stays child-backed — peers with the behavior project it. One hazard rides this skew (PR-review P2, resolved by fleet policy rather than machinery — see the unbuildable-writer note in the callout below): a client in this state still uploads whole bags, so its stale copy of the unbuildable key can travel under a fresh parent timestamp. In this fleet the state itself is exceptional — every codec-bearing plugin is static (shipped in the app bundle), so it requires a disabled plugin or a stale tab, both coordination-visible. A child-backed key with no local child rows at all is left untouched as pending materialization — children-in-flight from the writer's dual-write, landing next drain (under the id-bound checksum a copied bag can't reach this branch: it mismatches and materializes); removing a cell key because children are absent is forbidden everywhere — arrival, projection rebuilds, and the sweep alike. An old-client cell (checksum mismatched or absent) is the one place the full materialize direction still runs — sets, updates to keys that already have children, and clears (the deletion diff below) — because that bag genuinely is its writer's newest truth. Two guards apply even there: a child is overwritten only if its own updated_at predates the arrived bag's (the per-row clock the pipeline already trusts), and a key whose local children are all soft-deleted is not resurrected past a newer deletion.
  • Arrived rows whose cell carries a key no registered schema matches, on a block whose children carry a known fieldId → re-project the cell from those children (the offline-rename race; §7, slice C). Still project-direction: the registry is consulted by fieldId instead of by the stale name.
Two gaps the spike's catch-up leaves open (folded into the reconcile pass above; stated separately because each has a concrete failure):
  • Old-client clears resurrect — and the discriminator that closes them.

    The failure: a pre-migration client (cell-only writer) removes key status; the PATCH arrives at client B, whose [[status]] field row is untouched — the spike's set-only catch-up sees no present key to process. B's tree still shows the value, and the next touch of that subtree re-projects it into the cell and re-uploads: the delete is undone fleet-wide. The reconcile's deletion diff closes this — but it must never fire on cell shape alone. "Arrived cell omits key K while local children for K exist" is also the signature of an innocent current-client bag (a writer that never had K, winning column-LWW); soft-deleting on that re-opens the clobber for deletes.

    The discriminator (shaped by three review rounds): a presence marker fails — an old client's setProperty is a read-modify-write of the whole bag, so a reserved marker key written by the last current client survives the old client's edit untouched, and column-narrow upload means a separate stamp column is never touched either; a marker masks exactly the writes it must expose. The discriminator is therefore a projection checksum with three properties. Content-bound: a canonical hash over the covered keys' values, so any non-projecting edit breaks it. Id-bound: the block id is hashed in, so a wholesale-copied bag mismatches on the new block and takes the old-client branch instead of being mis-certified. Self-describing: the stored record is {hash, coveredKeys}, and verification recomputes over exactly the listed keys — never over "what this client's registry considers child-backed", which varies with plugin/version skew and would let a client missing a plugin mis-classify a peer's fresh projection as an old-client bag (PR-review P1). The registry decides what to re-project; it plays no part in classification. A containment rule completes verification (PR-review P1): a child-backed key present in the bag but missing from coveredKeys is itself a mismatch. On a current client's projection, present-child-backed ⊆ covered always holds — projection stamps exactly what it covers, in the same tx — so an uncovered present key proves a non-projecting writer added it after the stamp; without the rule, an old client's added key rides the still-matching checksum into the current-client branch and sits pending forever (readable from the cell, but never materialized until the sweep) instead of materializing on arrival. The containment check classifies bag keys by this client's actionable set — a key it cannot build is cell-backed for it (per-client rule above) and exempt. The unbuildable-writer note (PR-review P2, resolved by fleet policy): the exemption protects the reader's own branch but not its uploads — a client that cannot build a child-backed key still re-ships the whole bag on unrelated edits, and on a codec-bearing peer that stale key is present, actionable, and uncovered: the materialize branch, whose per-child clock guard has only the arrived bag's timestamp, could overwrite a newer value child — and unrepairably, since the sweep then sees children and cell agreeing. This fleet closes the state rather than armoring the merge: every codec-bearing plugin is static, so an unbuildable current writer requires a deliberately disabled plugin or a stale tab — coordination-visible, and the flip runbook says so. If plugin asymmetry ever becomes a real fleet state, the designed remedy is a third checksum class (preservedKeys: verify-gated verbatim carries, hashed like covered keys, never merge inputs, enumerable from synced metadata without the codec — full sketch in this PR's review thread, round 20) — build it then, not now. Match (hash and containment) ⇒ a current client's projection ⇒ never a merge input. Mismatch or absent ⇒ last modified by a non-projecting writer ⇒ the full materialize direction (set / update / clear) is licensed for that row's child-backed keys, clock-guarded per child (previous list).

    Storage (PR-review P2): the record must be invisible to every generic property surface — the panel's unregistered-keys section, copy/extract/import paths, key enumeration — or every block grows a visible synthetic property. Two sound homes, slice C picks: a reserved, filtered bag key (atomic with the bag by construction — it rides the same column through LWW and e2ee envelopes — but every enumerator must filter the reserved namespace through one shared helper, and filtering a namespace is only sound once no real schema name can occupy it (PR-review P2): bag keys are schema names and users pick those freely, so a colliding name would either vanish from every property surface or be misread as checksum metadata. Choose the reserved key from the space §7's name-validity rule already forbids — a key containing ]] cannot round-trip the wikilink grammar, so addSchema/rename can never mint it — and the same slice-B preflight that sweeps existing names for round-trip violations doubles as the occupancy proof for pre-existing names: a schema found squatting in the reserved space is rejected/renamed there, before any filter ships), or a dedicated synced column (invisible to property consumers by construction; atomicity holds because current clients write bag + checksum in the same patch, so per-column LWW picks both from the same winner, while an old client's bag-only patch leaves the checksum column behind — exactly the mismatch signal; costs a migration and an upload-whitelist entry). One e2ee constraint on the column option (PR-review P2): it must never carry plaintext for e2ee rows — coveredKeys names the schemas present on an encrypted row, and hashes over low-cardinality values invite guessing; exactly the edge metadata §8 encrypts. Under the §8 option-1 gate this is moot (e2ee rows run no child machinery and stamp no checksum); if option 2 later child-backs e2ee workspaces, their checksum rides the encrypted reserved-bag-key form — the column, if it exists, stays NULL for e2ee rows, enforced by the same server-trigger family as reference_target_id.

    Residual and fallback: an old-client bag is a whole-bag write, so accepting it can carry stale copies of other keys on that block — inherent to mixed-version operation (the old client has no per-key channel), bounded by the per-child clock guard and the upgrade window. Among current clients the cell is never consulted — deletes travel exclusively as child soft-delete rows. If the checksum slips out of slice C, ship without the old-client materialize branches and close the whole mixed-version gap by coordinated fleet upgrade instead (§12): a wrong-direction overwrite is worse than a temporarily unconverged edit.

  • Duplicate field rows from concurrent backfills persist. Two devices backfilling offline each create their own [[status]] row; after sync, the arrived rows are field/value rows whose own cells are empty, so the spike's row-targeted catch-up no-ops on them and never visits the shared parent — where the dedup lives. The reconcile pass walks arrived field rows (recognizable by reference_target_id) up to their parent and dedups there. Until then, duplicate rows persist until some unrelated local edit happens to touch the parent.
Why the processor pair doesn't ping-pong. Materialize (cell → children) and project (children → cell) watch each other's outputs; convergence rests on two load-bearing properties, currently implicit in the spike: every write is skipped when output already equals input (idempotence), and duplicate resolution is deterministic (ORDER BY order_key, id), so all replicas pick the same survivor. State both in code comments next to the processors; a refactor that breaks either turns the pair into a loop or a divergence engine.
Consistency sweep — fsck for the projection. The design's main standing risk is invariant density: correctness rests on every write path honoring the dual-write and arrival rules, and a future path that doesn't (importer, plugin, server feature) fails silently — the cell and the children just drift. Deterministic projection makes that cheap to police: a background sweep recomputes each block's child-backed cell keys from its children, byte-compares against the stored cell, and repairs divergence — with the arrival reconcile's projection-side branch rules, not blanket children-win (PR-review P2; checksum classification is arrival-only — the sweep repairs local projection state and never consumes a remote bag): keys with live children → project, children win; keys whose children are all soft-deleted → the deletion won, drop the key; cell keys with no child rows at all → pending materialization, not divergence — that is exactly the slice-C backfill-failure fallback where the cell stays authoritative for the block, so the sweep materializes them (a free backfill retry) or leaves them untouched and reports, but never deletes. Divergence counts are telemetry. Reuse the backfill harness (marker-gated per workspace, chunked transactions, idle scheduling) in verify mode; ship it with slice C. A nonzero counter is the alarm that some write path broke the invariant — it converts "discipline held in code" into "violations are detected and self-heal", which is the difference between an invariant and a hope.
Denoted-value rule (unchanged, implemented). Mutations deep in a value child's subtree — a comment under a long-text value — do not touch the parent's cell. The projection watches field rows and their direct value children only. Without this rule the processor becomes a reactive query engine and write amplification explodes.
Upload reality — and the cell's exit path. Every write above uploads (column-narrow, but it uploads): field/value child rows are ordinary synced rows; the derived cell write ships the properties_json column. There is no "derived, don't upload" tag on master and we don't need one yet — under the merge semantics above the uploaded cell is harmless (never a merge input) and useful (bootstrap hint). Budget roughly 2 extra rows per (block, property) pair on first materialization plus one cell-column PATCH per property write; the backfill of a large workspace is a real upload burst (§11, slice C). The end state goes further: once children are the fleet-wide source, the cell can stop syncing and become a locally-derived column (per-workspace, once backfill is complete and sweep-clean — §11; with §6's no-carve-out decision and §9's orphan synthesis there is no permanent key category that keeps a row on the channel) — at which point a property update is a small value-child PATCH plus the parent's metadata PATCH (setProperty bumps updated_at/updated_by on the parent, and those columns sync), still well under re-shipping the whole bag, i.e. cheaper on the wire than today — but that step is deliberately staged and gated on the hydration measurements (§11). One column is deliberately untouched by all of this: references_json (PR-review P2). Ref-typed property edits re-project the parent's by-field backlinks (sourceField = the schema name) from the locally-derived cell, and that column keeps uploading through and beyond the cell sync-off — existing parent/sourceField backlink queries ride it unchanged, and the value child's own reference row (sourceField: '') is not a substitute for it. Only properties_json ever leaves the channel.

6. The scope decision: every schema gets children

This is the fork in the road. properties-model-evolution.md (May 8) chose A + D: children for user-defined fields, cache-only for system fields. The spike flipped it to A only — every registered schema materializes, including types, system:collapsed, alias, and the property-schema:* config props on schema blocks themselves. An earlier draft of this doc argued for restoring the A+D boundary as a per-schema flag; the §5 merge-semantics analysis reversed that — and an earlier revision of this section still kept the three non-document scopes (UiState, UserPrefs, Automation) as a permanent cell-backed carve-out. That carve-out is now removed too (owner decision, stated repeatedly across design rounds and finally landed here — the split had survived on drafting inertia plus review-loop entrenchment, not argument). Decision: materialize everything — kernel, plugin, and user schemas, all five ChangeScopes (BlockDefault, UiState, UserPrefs, Automation, References — the count is enumerated deliberately; PR-review P2 caught "all four" leaving References-scoped schemas outside the backfill and blocking the column drop) — with system and machine field rows hidden by default behind the existing per-schema hidden override and "Show Hidden Fields" toggle. No schema category stays cell-backed permanently. Two reasons, both structural. First, §5's own argument admits no boundary: whatever stays cell-synced stays clobberable, and a per-user UiState row's bag is still whole-bag LWW across that user's own devices — the same cross-key clobber, just lower-stakes. Second and decisive: partial migration doesn't finish the job — any permanent cell-backed category keeps properties_json alive forever (schema, sync rules, RPC bindings, e2ee envelope handling), so the end-state column drop (§11) would be unreachable. Hidden-by-default is what makes this free at the UI layer: machine props materialize like everything else and simply never render outside the debug/hidden surfaces.

What flipped the call: whatever stays cell-synced stays clobberable. Under §5, per-key merge exists only for keys that are rows; a key that lives solely in the cell still rides whole-bag column-LWW, and every dual-write of any property re-uploads the whole cell — carrying stale copies of every cell-backed key with it. A concurrent system:collapsed toggle would be clobbered by an unrelated property write on the same block, and no child exists to repair it from. The one contended kernel prop (collapsed) is precisely the one that most needs row-granular merge. Child-backing it also clears the path on wire cost — though only at the end state: while the cell still syncs, each toggle ships the value-child PATCH plus the re-projected whole bag (more wire than today, not less); once the cell stops uploading (§11's per-workspace cutover — backfill complete, sweep clean), a toggle is the value-child PATCH plus the parent's metadata bump — cheaper than re-shipping the bag. The wire argument is a consequence of the sync-off, not an immediate saving; the merge argument above is what decides the flip.

The costs the spike surfaced are real but turned out to be presentation-layer, and the spike itself has since built the answers: hidden-field filtering in the outline (per-schema hidden + toggle — since schema unification a synced definition field, not a curated code list, though §10's outline-vs-panel question remains open for the slice-C flip), tx.childrenOf default-excluding field rows so structural callers see only visible siblings (f4d0b447), and panel-layout/orphan-alias special cases. One genuine residual to carry, eyes open — plus one that schema unification has since dissolved:

  • Master's ~8 tree-shape tests need porting. Under materialize-everything they assert a pre-feature tree shape that is now wrong by design. Note the failure list predates the childrenOf default-exclude flip — re-run the spike suite to get the true residual set before porting (several of the 8 may already pass).
  • Resolved — synthetic fieldIds no longer exist. The spike's kernel/plugin property:<name> pseudo-ids would have leaked non-block ids into references_json, and slice B would have had to special-case them out of reference projection. Schema unification took the parenthetical alternative — real definition blocks for every schema — so a field row's reference_target_id is always a valid block id and the backlink entry is correct: "where is this field used?" backlinks for kernel/plugin fields come free instead of needing a guard.

The non-document scopes migrate too — this section previously carved out UiState/UserPrefs/Automation as permanent cell-backed, and a review round even completed the carve-out to all three (PR-review P1, "§6 silently decided only two of the four"). Both moves are reversed. The carve-out's rationale ("per-user, effectively uncontended, cell-LWW harmless") was true per-key but answered the wrong question: the cost of the carve-out was never those keys' merge quality — it was that their rows keep the cell channel, and therefore the column, permanently. What migrating them actually costs, stated for the record: (a) keystroke-frequency UiState props (editorSelection, focus) pay double wire transitionally — value-child PATCH plus whole-bag reproject while the cell still syncs — the same shape this section already accepted for collapsed; it ends at the workspace sync-off. (b) history churn: high-frequency machine props ride the row_events provenance knob (that optimization is already a slice-C prerequisite, §12) so cursor state doesn't flood history. Their field rows live where the props live today — on the per-user state/prefs/automation rows, hidden by default — not in anyone's document tree.

Mechanism: a per-workspace flip column, not a per-schema flag. With §6's no-carve-out decision and one uniform flip, per-definition state has nothing left to vary over — so the lever is a single synced bit: workspaces.properties_migration (an enum column on the workspaces table — 'cell' → 'children' → eventually 'cell-off', the encryption_mode pattern: operator-written, synced to every client, synchronously readable after workspace sync, and server-visible, which is precisely what the §11 sync-rules split needs). The states are ordered, and "flipped" everywhere in this doc means at or past 'children' — IN ('children','cell-off'), never an equality test (PR-review P2: an equality gate would un-recognize every field row the moment a workspace advances to 'cell-off' — exactly when the cell is no longer a synced fallback). And "operator-written" is enforced server-side, not by convention (PR-review P2): the workspaces table is directly updatable by workspace writers — the e2ee migration added workspaces_prevent_e2ee_field_change for exactly this exposure — so the column ships with the same trigger family: client writes rejected, transitions forward-only, advanced only through a SECURITY DEFINER operator RPC that can refuse 'cell-off' until the backfill/sweep gates have been attested. The predicate every consumer shares — recognition (§9), dual-write, projection, reconcile — is child-backed = workspace flipped. Nothing else: no scope term (§6 migrates all scopes), no per-schema stamp, no buildability term at flip time (§5's unbuildable-writer note is fleet policy), and no per-key definition-existence term either — "every key resolves a definition" is an invariant, not a predicate branch: established before the flip by §9's orphan synthesis, maintained after it by addSchema (the only normal key-minting path creates the definition), and repaired by the sweep, which synthesizes on encountering a definition-less key from a raw writer or import. Two runbook consequences: synthesis runs before the flip, and unconverted dynamic extensions convert to seeds before it too — otherwise synthesis mints a user-origin definition for their keys and the later seed conversion collides with it in the winner machinery, stranding field rows on the loser's fieldId. (A key that momentarily lacks a definition anyway is mechanically inert, not wrong: no field row can exist for it — recognition points at definition ids — so it rides the cell until the sweep promotes it.) This deletes outright: the reserved property-schema:child-backed definition field (schema unification reserved it for this doc — record the divergence there), the slice-C bulk stamp across user definitions and its idempotent re-run rule, the consistency-sweep re-stamp repair for flag-less schemas created by old clients (a pre-flip client's new schema is simply covered by the workspace bit like every other definition), and the per-plugin seed-revision-bump flip choreography. A per-definition override (exempting one schema) is deliberately not built — if a real exception ever appears, adding an override field then is a small additive change; its natural home remains the definition block. What survives unchanged: all machinery branches on the one predicate in exactly one place each; one write model (setProperty), one detection rule, one projection. E2EE composes by the same column: an e2ee workspace simply never leaves 'cell' (§8 option 1). One window this rollout must tolerate (and does — the flip is a coordinated event here): the column propagates via sync, not atomically, so a device may briefly run pre-flip against a flipped workspace; its cell-only writes are classified by the §5 checksum exactly like any old-client writer's, and the backfill sweeps them in.

Two hard requirements the flip lever carries, both from adversarial review — both met (one upstream by schema unification, one restated for the workspace column), so the slices here consume them rather than build them:

  • The lever must be durable and fleet-consistent where it can vary — met. The workspace column is one synced server-side value — a code-constant fork between client versions is not expressible, and every device reads the same bit after workspace sync. The definitions-exist invariant rides synced data for every origin since unification; a dynamic extension not yet converted to seed declarations (the transitional dual-path, until its B′ deletion slice) has no definition block — the runbook converts it before the flip, and a straggler's keys are inert-until-swept (callout above) — the safe default either way. What remains ours: moving the column in either direction is a migration — run a final projection pass, then materialize or delete children under the new regime (rollback: drain then delete, §11 slice B) — never a silent payload change.
  • Name collisions become a hard error — met. The old registry resolved collisions last-wins-by-load-order with a console.warn — per-client, exactly what child-backing can't tolerate (name is the cell key, fieldId the child identity; a per-client winner strands the loser's field rows). The unified registry resolves names deterministically fleet-wide and enforces it centrally: code-owned seeds are unshadowable in v1, the rest resolve by a deterministic order over synced state, and every cell-writing boundary resolves through the one resolution primitive, which returns a structured shadowed/ambiguous status instead of touching the winner's key (unification §7; buildPropertyDefinitionRegistry + SnapshotPropertySchemaResolver). A shadowed definition stays resolvable by fieldId — its child rows keep classifying and its backlinks keep resolving; it is excluded from the name map and the cell projection. Slice B inherits this instead of building collision handling.

This answers the question PR #285 explicitly left open ("should UIState props materialize?" — yes, and likewise UserPrefs and Automation: everything materializes), fully restoring the spike's original instinct, now for a load-bearing reason: the merge story plus the column's eventual removal, not aesthetic uniformity. types keeps its denormalized local index (block_types) and TypeTagger integration regardless — the child rows are its synced source like any other prop, the index stays the query surface.

7. Field-row content: [[name]] + derived column

Rev 1 specified field-row content as ((schema-id)). The spike ships [[Schema Name]] with the resolved fieldId in the synced reference_target_id column, derived same-tx from content. Keep the spike's shape:

  • Markdown round-trip is a headline motivation — and [[priority]] exports as meaningful markdown where ((7f3a…)) doesn't. Copy-paste into another workspace can resolve by name (§9); a uuid never resolves.
  • Renames don't corrupt stored data — but rename support must land with the write path, not after it. The column, not the label, is authoritative, so children survive a rename untouched. The cell does not: it's keyed by name, so after a rename every consuming block still carries the old key, the registry no longer knows that key (materialize skips unknown names — it cannot repair a key it no longer recognizes), and every schema-aware reader falls back to defaultValue while the tree still shows the real value. Mass silent unset. The rename-reproject pass (reprojectPropertyValueChildren, built in the spike, currently untriggered) is therefore part of slice B2, not polish — or renames must be blocked until it lands. And it rewrites field-row content too (PR-review P2): field rows store [[Schema Name]], so the pass retitles [[old]] → [[new]] across the schema's field rows (indexed by reference_target_id — one targeted sweep); left stale, markdown export and the §9 cross-workspace re-derive — which resolves by content — would bind the dead name or a wrong same-named schema.
  • Typing a field by hand works. A user (or an importer, or an agent) can type [[status]] as a child and put a value under it; the derive processor resolves the alias to the schema and the project processor folds it into the cell. That's the "it's just blocks" promise made real.
  • The cost: name collisions need a resolution rule. The two-tier lookup stands — schema-name map first, generic alias lookup second (implemented in the spike) — but the schema-name map itself is now the unification's deterministic fleet-wide winner map (§6), replacing the spike-era per-client "registered map wins by load order". And the exact-reference grammar is scoped to whole-block content so prose containing ((…)) never becomes a field (implemented in referenceBlock.ts). A second cost surfaced late (PR-review P2): schema names must round-trip the wikilink grammar — a name containing ]] renders lossy (renderWikilink documents foo]]bar → foo] ]bar), and every re-derive-by-content path (§9 cross-workspace copy, §8 de-structure, markdown round-trip) then binds the wrong schema or none. addSchema and the rename flow reject non-round-trippable names; existing names get checked in the slice-B preflight (one more entry alongside the unification's §7 name preflight).

8. E2EE workspaces — new constraint

E2EE workspaces (May 29) encrypt content, properties_json, and references_json into enc:v1 envelopes; a server trigger rejects plaintext in those columns. Reference edges are therefore part of the encrypted surface today. A synced plaintext reference_target_id would leak, per row: "this row is a field instance", "of this schema", and for exact block-ref rows "this row points at that row" — precisely the edge metadata the current design encrypts.

Options, in increasing effort:

  1. Gate: e2ee workspaces flip later — sequencing, not an endpoint. An e2ee workspace's properties_migration column stays at 'cell' until option 2 below is built (§6 — the gate and the flip ride the same lever), so the machinery behaves as if every schema were cell-backed. Zero leak, zero new code beyond the gate; e2ee users keep today's behavior for now. Recommended for the first ship — but option 2 is required work, not optional: §11's column drop needs every workspace at 'cell-off', e2ee included, so a permanently-gated e2ee workspace would keep the column alive exactly the way §6's rejected carve-outs would have.
  2. Derive locally, never upload. The exclusion cannot live in BLOCK_UPLOAD_COLUMNS — that is a static, trigger-generated list with no per-workspace variance (PR-review P2). Strip the column at the upload connector instead, keyed by workspace mode, when assembling envelopes (suppressing column-only PATCHes outright): otherwise a local re-derive in an e2ee workspace enqueues an envelope the slice-A server trigger rejects into the upload quarantine. Run the derive pass on sync arrival (the observer materializes decrypted plaintext locally, so content is available; onRowsAccepted is the hook). Needs a first-sync full derivation pass per device (marker-gated, same pattern as the backfill). This is the real fix; it's mostly plumbing that reuses existing machinery.
  3. Encrypt the column. Defeats the point — the column exists to be an indexable, queryable detection key on the client; the server never queries it. (2) achieves the same privacy with less ceremony.

The gate in option 1 needs its scope stated precisely — and one piece of it must move server-side (PR-review P1). As shipped, the e2ee ciphertext trigger (blocks_require_ciphertext_for_e2ee) checks only the three envelope columns and would happily accept a plaintext reference_target_id from a stale client, a missed hook, or a direct PostgREST write — exactly the untrusted paths that trigger exists to stop. Slice A must extend it to also require reference_target_id IS NULL for e2ee-workspace rows (same migration as the column add; note encryption_mode is immutable server-side today — workspaces_prevent_e2ee_field_change rejects flips — so no transition path exists to race the check; a future flip flow's bulk rewrite nulls the column and passes). The rest of the gate is client-side:

  • What the gate disables: all of it — the dual-write, both property processors, the backfill, and the derive processor's uploads. Disabling only the write path while leaving projection live re-creates the stale children problem (§11 rollback discussion); the gate means "this workspace has no child-backed machinery running, full stop."
  • Cross-workspace moves/copies de-structure, not just strip. A field row moved or pasted from a plaintext workspace into an e2ee workspace carries its plaintext reference_target_id; nulling the column is necessary but not sufficient (PR-review P2) — detection is column-based and the machinery is gated off in e2ee, so the [[field]] rows and value children would linger as ordinary visible content while the destination cell is the property store. The move path must run the same de-structure step as the mode-transition flow below: project the subtree's field/value children into the destination cell, then null the column and hide or soft-delete the de-structured rows. (Owner: slice B3's move/merge integration.)
  • Mode transitions are migrations. Flipping a backfilled plaintext workspace to e2ee means the column values for its rows were already uploaded — the flip must null them server-side (the encryption migration is already a bulk rewrite; add the column to it), run a project-then-freeze pass over existing children locally, and then hide or soft-delete the de-structured rows: with the column nulled, §9 detection fails and every former field row would otherwise render as a plain [[name]] block with its values as loose children, fleet-wide. The cell is authoritative post-flip; the rows must not linger as clutter. Flipping e2ee → plaintext re-enables the machinery; the ordinary backfill covers materialization. And the flip must be history-safe (PR-review P2): today encryption_mode is immutable by trigger, and blocks_history retains full plaintext INSERT rows and UPDATE diffs — a live-row rewrite alone would leave every pre-flip reference_target_id and plaintext column readable in history after the flip. Either the mode stays immutable, or the transition is a SECURITY DEFINER migration that also purges/encrypts the affected history rows. (Owner: the workspace-encryption flip flow itself — these requirements gate that feature, not the §11 slices.)

A side benefit of the §11 cell end-state — for e2ee workspaces, sequenced behind option 2: under the option-1 gate the cell is an e2ee workspace's only property store, so the sync-off cannot apply to it. Once derive-locally child-backing (option 2) replaces the gate, stopping the cell upload removes the envelope column from the wire (the §11 workspace-level cutover applies in e2ee too, gated on the same backfill-complete condition); property data then travels only as ordinary encrypted child-row content.

Two server-surface items move out of "verify later" into hard slice-A deliverables (§11): extend both server write RPCs (PR-review P1) — creates now batch through apply_block_creates (20260709000000, postdating this paragraph's first draft; its SQL inserts an explicit column list, superseding the carries-all-columns PostgREST upsert), and later changes ship as PATCHes through apply_block_patches, which silently drops unknown keys. Left unextended, a newly created field row reaches the server with reference_target_id NULL — peers never recognize it until some later PATCH — and a change to an existing row is dropped server-side, with no client-side re-derive on arrival to compensate; the synced detection column would be broken over sync in both directions. The extension must also be null-aware (PR-review P2): the column must be clearable — a row edited back to ordinary content, or stripped by the e2ee/move paths, must stop being a field row on peers — so the RPC needs the presence-based handling used for nullable parent_id, not the COALESCE(patch, col) pattern that silently ignores JSON null. And re-timestamp every spike-introduced Supabase migration — both 20260527123000 (reference_target_id) and 20260530143000 (recent-content index) sort before already-applied master migrations (20260612…, 20260625…); wrap them in begin; … commit; per the supabase skill while at it.

9. Detection rules and edge cases — as implemented, with divergences

Field-row recognition.
A child of P is a field row iff reference_target_id equals a definition block's id — a fieldId-keyed lookup the unified registry ships (resolveField) — the workspace is flipped (properties_migration at or past 'children', §6; PR-review P2: recognition off the raw column alone activates the moment slice A/B starts stamping — an ordinary whole-block [[some property schema]] link would be hidden and reinterpreted even while the workspace is still cell-only, breaking the "machinery lands dormant" claim; the workspace column is synced and server-visible, so the gate is fleet-consistent) — and P is a content block: not itself a field row and not inside a property subtree. The subtree half generalizes the direct-parent guard (PR-review P2): a [[status]] link written as a comment below a value child has a parent that is no field row, yet it is user content, not a field of the value block. Role is positional and inherits: children of field rows are values; everything beneath a value child is ordinary content; recognition rejects any row whose ancestry passes through a field row. Renderers and traversals get the walk free from context (they arrive from above and already know they're inside a property subtree); SQL callers bind the same recursive rule. The parent guard is load-bearing (PR-review P2): a ref-typed value pointing at a definition block gets stamped with that definition's id by the derive processor exactly as a field row would (block-type:properties values are the standing example — a refList whose targets are all property-schema blocks); without the guard, render/filter/projection would hide or reinterpret those values as nested fields. Children of field rows are values and comments, never field rows, whatever their target. Column read plus flag read, no content parse. Classification needs the definition, not the owning plugin (an uninstalled plugin's definitions sync like any block), and works for shadowed definitions too (§6 — losers stay fieldId-resolvable). Unresolvable targets render as plain reference blocks — a pasted field from another workspace degrades to a visible, honest "unknown field" row.
Cross-workspace copies re-derive the column, never carry it.
Seeded fieldIds are workspace-scoped (unification §4.2), so every field row copied or moved across workspaces — kernel schemas included — carries a foreign reference_target_id (PR-review P2). The structural copy/move path must clear the column and let core.deriveReferenceTarget re-derive from the [[name]] content against the destination's name-winner map, same tx — the seam where the unification already strips seed:*/type-id props on paste (its §12). Without the strip, §7's resolve-by-name promise silently fails: detection never parses content, so the copied rows degrade to unresolvable reference blocks. The e2ee de-structure rule (§8) covers encrypted destinations; this is the plaintext general case. One arrival-order hole in the re-derive (PR-review P2): if the destination's definition hasn't synced or been imported yet, the same-tx re-derive clears the foreign id and resolves nothing — honest degradation to a plain reference row — but core.deriveReferenceTarget watches content, so the definition's later arrival never revisits the unchanged [[name]] rows, and a no-cell copy's fields would stay unprojected forever. Two repairs, both cheap: definition-block arrival/creation enqueues a targeted re-derive over NULL-column whole-block-wikilink rows matching the new name, and the §5 consistency sweep carries the same rule as the backstop. Matching by name is sound because rename-retitle keeps field-row content in lock-step with schema names (§7).
Orphan keys — synthesized definitions, not a permanent bucket.
A cell key with no definition block of any origin (historic residue, keys from plugins disabled before the unification's cutover, dormant dynamic extensions) gets a definition synthesized by the slice-C flip tooling (owner decision — a permanent definition-less bucket would keep rows on the cell channel forever, defeating the §11 column drop): scan the workspace's distinct definition-less keys, create a user-origin definition per key — name = key, codec inferred from all observed values of that key (all-boolean → boolean, all-number → number, ISO-date strings → date, else string; anything mixed or structured falls back to the raw-json preset — conservative on ambiguity), hidden by default until triaged. From that point they are ordinary schemas riding the normal machinery. Until the flip runs (and for keys a still-unconverted dynamic extension owns), definition-less keys retain today's cell semantics — a transitional state, not an end state. Synthesized definitions carry a provenance marker, because a synthesized key's real owner can arrive later (PR-review P2): re-enabling or converting a plugin seeds an unshadowable code-owned definition under the same name, the synthesized definition drops to shadowed loser — excluded from the name map and cell projection — and the already-backfilled field rows still point at the synthetic fieldId, so the migrated values would vanish from block.get/queries under the real schema. The plan therefore includes a retarget pass: on a seed claiming a name held by a marker-carrying synthesized definition, re-point that definition's field rows to the seed's fieldId (the §7 rename-reproject machinery is the precedent — same re-stamp + re-project shape), then retire the synthesized block; the sweep carries the same rule as backstop. The unification doc counts disabled-plugin residue as this problem's easy sub-case (its §4.3); e2ee workspaces synthesize client-side at their own (option-2) flip, since the server can never enumerate their keys.
Value decoding.
Spike: first parseable value child wins (order-key order); unparseable children are skipped. If no child parses, the projection removes the key from the cell. (This rule — and default-vs-unset below — presupposes a live field row; a cell key with no child rows at all is pending materialization and is never removed, per the §5 global rule.) Rev 1 said "log + omit, keep data visible" — the spike keeps the rows (data is never destroyed) but the cell-side key vanishes, which downstream readers see as "unset". Acceptable; document that "field with only-invalid values" reads as unset while remaining visible/fixable in the tree.
Duplicates.
Divergence from rev 1 ("keep extras, pick lowest orderKey"): the materialize path deletes duplicate field rows and duplicate value children (soft-delete, so recoverable via history). Projection reads the first field row's first parseable value. Deterministic dedup is the better call under sync races, but slice B3 must tighten two things the spike leaves sharp: (a) the survivor is picked by (order_key, id) — arbitrary relative to content — and deleteSubtree recursively removes the loser including any user-authored sub-children (a comment thread under the losing value silently goes with it). Before deleting: for duplicate value children, relocate the loser's non-value sub-children (comments, elaboration) under the survivor; for duplicate field rows whose surviving values differ, preserve the losing value visibly — e.g. relocate it under the surviving value child (not directly under the field row, where the value-child dedup would just re-collapse it) — silently hiding a divergent value behind first-wins is data loss in effect if not in storage. (b) the two dedup call sites disagree on deletion semantics: materialize subtree-deletes, while writePropertyValueChild shallow-deletes duplicate value children — orphaning their sub-children live under a tombstone. Unify on one (relocate, then subtree-delete). Until multi-value lands, "N children under one field row" stays a reserved shape, not a user affordance.
Default-value vs. unset.
Unchanged from rev 1: zero value children under a live field row ⇒ key omitted from the cell ⇒ block.get falls back to defaultValue. No reader change.
Codec / preset changes are migrations, not edits.
§9's "only-invalid values reads as unset" is fine for one malformed child; it is a mass event when a user changes a field's preset or config (text → number): every existing value child becomes unparseable under the new codec, each consuming block's next projection removes the key, and readers see defaultValue everywhere — silent fleet-wide unset presented as default. Worse, until reprojection runs, the cell still holds the old encoding and the new codec decodes it on the read path. A preset/config change must run an explicit re-encode/validate pass over existing values (with a user-visible "N values can't convert" outcome), not lean on the projection's remove-on-invalid behavior. Same machinery as the rename reproject; build them together in slice B.
Field-row ordering (decided).
Machinery inserts field rows first among children: every creation path uses keyAtStart, which is what the spike ships — so fields cluster above content children as an emergent default, not an enforced invariant. Rev 1's "carriers-then-content with ghosts between" is dropped. Materialization, backfill, and the add-field affordance all follow the insertion rule; orderKey remains authoritative and user-owned, so a field row manually reordered below content children stays where the user put it.
Reparenting a field row = assigning the property (decide in slice D).
The project processor reprojects both sides of a move, so dragging a [[status]] row under a different block gives that block a status property — and removes it from the old parent. That's a coherent "it's just blocks" semantic, but as a silent side effect of an ordinary indent/drag it's surprising (and works on any host, including schema/system blocks). Slice D must either embrace it as the explicit field-assignment affordance (with UI signal on drop) or guard plain structural moves (e.g. reject, or clear reference_target_id on cross-parent moves that aren't the assignment gesture). Recommendation: embrace with signal — guards that silently strip identity create their own confusion.
Ref-typed values → parent references[].
Two paths exist now: (a) the cell-side projection (projectPropertyReferences, sourceField = schema.name) still runs off the bag — unchanged, keeps backlinks-by-field working; (b) field rows and value rows are themselves reference-bearing blocks, so the references plugin indexes them as ordinary rows — meaning a user field's schema block accrues backlinks from every block that uses the field ("where is this field used?" for free). Verify the two paths don't double-count in backlink UIs (dedup key is (sourceField, id); path (b) rows carry sourceField: '' on their own row — the parent's entry stays single).
Merge, orphan cleanup, panel layout, tree primitives.
Implemented in the spike: mergeBlocksInTx reparents only non-field children, subtree-deletes from's field children (fixed at f4d0b447 — a bare delete had orphaned value rows), and re-materializes the merged bag — but slice B3 must add the same preservation rule as the dedup above (PR-review P2): before the subtree-delete, relocate user-authored descendants of the source's value children (comments, elaboration) under the merged result's corresponding value children — a bare subtree-delete silently tombstones them; orphan-alias cleanup deletes generated field rows under alias seats; panel-layout projection ignores field rows; and tx.childrenOf now excludes field rows by default (structural callers must see only visible siblings — the machinery opts in with includePropertyChildren). Copy, export, import, and delete/purge traversals count as machinery, not outline callers (PR-review P2 ×2): they opt in with includePropertyChildren — markdown round-trip is a headline motivation (§7), and a copied subtree missing its field/value rows silently loses the properties wherever the destination path doesn't rebuild them from the carried bag; and softDeleteSubtree discovers descendants via tx.childrenOf (verified, mutators.ts), so under the default exclusion a delete would strand live field/value rows attached beneath a deleted parent. Under the §6 scoping these special cases apply everywhere — kernel props materialize too, so hidden field rows exist on most blocks and the exclusion rules get more load-bearing, not less. Field rows, user and kernel alike, must not be reparented as content.
Type-add field slots vs ghosts (decided: render-only ghosts).
The spike materializes empty field rows ("slots") when a type is added (TypeTagger → materializePropertyFieldSlotsForExistingRow; since 03236460 a slot whose key already has a cell value materializes the value too — an empty slot had let the reverse projection strip it). Rev 1 argued for render-only ghost rows that materialize on first edit — no storage until data exists, nothing to clean up on type removal. The spike still has no slot cleanup on type removal (_removeTypeInTx only rewrites the types array), so removing a type strands empty [[field]] rows. Decision: render-only ghosts; drop the TypeTagger materialization. If empty slots prove necessary (e.g., for cross-device consistency of field order), add explicit cleanup on type removal.

11. Shipping plan — slices extracted from the spike (dormant-flag landing as fallback)

PR #285 proved the design end-to-end, and the branch is actively maintained — it re-merged master again on 2026-07-01 and has been taking review fixes since (03236460, f4d0b447). But as a shippable unit it has structural problems: it's a merge branch carrying ~35 spike commits (40+ on the PR) whose history has already needed two hand-repaired merges against a fast-moving master (one merge obstacle is blockData.ts being binary-to-git, which defeats line-level auto-merging outright), its arrival handling has the §5 clobber inversion baked in, and its test suite disagrees with master's by design (~8 tree-shape tests — which §6 now says get ported deliberately, not reverted).

Three ways to ship it:

  • Land the merge as-is, fix forward. One review of a known-working integrated tree (3667/3677 green at the PR's last stated run, ~880 lines of convergence tests). But it lands the review-identified gaps — above all the §5 arrival-authority inversion that re-creates the clobber, plus deletion convergence, dedup semantics (§9), flag durability (§6), RPC (§8) — as production behavior, then re-touches the same code to fix each.
  • Land the merge behind the dormant flip column. Ship the whole mechanism gated on workspaces.properties_migration = 'children' with every workspace still at 'cell', so it lands inert, master's tests stay green, and the flip follows per-workspace. Cheapest path to "mechanism on master"; preserves the spike's tested integration; the natural home for the §8 fixes. Its real cost is review quality: one ~4k-line PR whose inert-ness must be trusted, with the §5/§9 convergence gaps still to be fixed in-place afterward.
  • Extract master-first slices (below), copying spike code verbatim wherever possible — extraction that re-derives the mechanism would add semantic-drift risk for no gain. Each slice lands with its review fixes built in and is reviewed against master's suite.

Recommended: slices, second choice: dormant column. The deciding factor is that the adversarial review found changes needed inside the core mechanism (the deletion-convergence catch-up, dedup semantics, lever durability), not just around it — landing first and fixing after means shipping convergence machinery with known gaps, and the fix commits churn the same files a reviewer just read. If shipping speed dominates, the dormant-column path is legitimate — but it must fold in the full review-fix set, not just the gate: the §5 deletion-convergence catch-up, the §6 durability/collision rules, the §8 gate + RPC + migration re-timestamps, and the §9 dedup tightening. A dormant landing that defers those ships the same gaps one workspace-column write away from production. Either way, treat PR #285 as the reference implementation and keep it open until its content is fully accounted for.

Cell end-state: measure first, then stage the sync-off

With §5's merge semantics in place and §6's uniform flip (all schemas, all scopes, at slice C — there is no separate kernel or plugin flip step anymore), the synced cell is harmless everywhere in a flipped workspace: pure derived projection, never a merge input. What remains is the pure-optimization tail — stop syncing properties_json, then drop it (wire savings, e2ee simplification, the cell becomes an ordinary local derived column). Because no key category stays cell-backed permanently (§6) and orphans are synthesized (§9), the old per-row composition rule collapses: eligibility is per-workspace — a workspace whose backfill is complete and whose §5 sweep reports zero divergence can leave the channel wholesale (properties_migration = 'cell-off', the same server-visible column the flip rides, which is what the sync-rules split keys on). The per-row fail-safe marker this section used to specify survives only as the transitional guard while a backfill is in flight. The staged sequence, each step gated on the one before:

  1. Arrival-authority flip ships (slice C) and the fleet fully dual-writes — every scope, every origin. Zero cold-start change; cell still syncs as bootstrap hint.
  2. Backfill completes per workspace (slice C) — every property has child rows behind it; the sweep's divergence counter reads zero over a full pass. Settle §10's outline-visibility question for system field rows along the way (the synced hidden definition field replaced the curated code list; what's left is deciding the outline's bit for panel-visible props like types).
  3. Stop uploading the cell — per workspace, once backfill-complete and sweep-clean. The rule (PR-review P1, simplified by §6's no-carve-out decision): in a 'cell-off' workspace no per-edit cell PATCH ships — with every key child-backed and orphans synthesized, the old per-row key-composition test has nothing left to discriminate. Two conditions gate the workspace transition, both checkable by tooling: the backfill has completed (no definition-less keys, no property without child rows) and a full §5 sweep pass reports zero divergence. A transitional per-row guard remains while a backfill is in flight: rows the backfill hasn't reached keep PATCHing — this is the fail-safe direction (unswept rows keep the channel). Fleet-buildability, previously a per-definition predicate here, is a runbook check under §6's uniform flip: all codec-bearing plugins are static, so "every client can re-project every key" means "all devices current, no codec-bearing plugin disabled" — verified at flip time, not encoded per row (§5's unbuildable-writer note). "Stops uploading" also means the per-edit PATCH channel only (PR-review P2): creates keep sending the projected bag — apply_block_creates binds an explicit properties_json into a NOT NULL server column, so a bagless create would bind NULL and fail the insert; the one-time create payload was never the wire cost this step chases. Semantics for rows that stop: the server copy freezes — and because downloads continue at this step, the materializer seam change moves up from the download stop to here (PR-review P2): a remote property edit re-delivers the parent row carrying the frozen stale cell, and the Layout-B materializer writes every downloaded column into blocks before invalidating readers — stomping the receiver's freshly-projected cell, readable as stale until the reconcile restores it, on every remote edit. The materializer must preserve (or re-project before invalidation) the local properties_json for rows whose cell has stopped uploading. A fresh device's downloaded hint also decays (every post-flip edit is missing until the projection pass sweeps the block). Status: must come after (1)–(2) — a remaining cell-only writer still needs the channel — and it is a transition, not a stopping point: it shares the next step's projection-readiness gate (PR-review P2).
  4. Stop downloading it — same workspace condition: workspaces not yet 'cell-off' keep the column. Mechanically a stream split in the sync rules — which also requires relaxing the Layout-B staging schema (PR-review P2: blocks_synced.properties_json is TEXT NOT NULL and the raw-table put binds every storage column, so a missing stream column arrives as an explicit NULL and fails the insert before the materializer can act. Make the column nullable in the same migration, or bind a non-null placeholder in the put itself — a column DEFAULT alone does nothing, since SQLite never applies defaults to explicitly-bound NULLs. And nullable staging alone only moves the failure one hop (PR-review P2): the materializer's UPSERT_BLOCK_SQL binds every storage column into blocks, whose properties_json is equally TEXT NOT NULL DEFAULT '{}' (blockSchema.ts), so the staged NULL fails at the live table before the first-sync projection can repair anything. Choosing nullable staging therefore also extends the step-4 materializer seam: a NULL staged cell is the "column stopped" case — preserve the local properties_json / re-project instead of writing the NULL through. The placeholder bind avoids the cascade entirely, at the cost of making '{}' ambiguous between "empty bag" and "column stopped" — fine only because the checksum, not cell shape, licenses merge decisions). The split's predicate must be server-visible (PR-review P2): sync rules select over public.blocks and can see neither a client's registry nor, for e2ee rows, any plaintext keys — which is exactly why the eligibility state lives in the workspaces.properties_migration column (§6): operator-written, fail-safe (only workspaces explicitly at 'cell-off' leave the stream), and already the thing the sync rules join on. The per-row client-written marker this step used to specify — with its buildable-on-every-client stamping rule (PR-review P2 ×2) — is no longer needed as global server state; what survives of the buildability concern is the runbook check in the upload step. Or keep downloading the column fleet-wide and consume it only where the workspace state says it is live; the download bytes are the smallest of the wins. Fresh devices now need the first-sync projection pass (below) before cells are readable. The measurements gate the upload and download steps together: take both or neither.
  5. Deprecate, then drop, the server column (stale from the upload stop onward; a future server-side feature that wants properties derives them from the child rows with a SQL view/aggregate — the data is all there for plaintext workspaces, which is the only case server features could read anyway). Under §6's no-carve-out decision plus orphan synthesis the column is genuinely droppable once every workspace reaches 'cell-off' — the end this project actually aims at; before that point it survives only as the transitional carrier for un-flipped workspaces. The drop has its own prerequisite (PR-review P2): the create path still binds the column — apply_block_creates binds an explicit NOT NULL properties_json, and the client's generated storage/upload column lists include it — so before any server-side drop, rework creates end-to-end (RPC signature, upload payloads, BLOCK_STORAGE_COLUMNS-derived lists) to omit the column while the local table keeps it as a derived, never-uploaded column. Dropping first would fail every new-block create in 'cell-off' workspaces.

The measurement that gates the download stop — run against the real database before committing to the no-hint cold start. Known scale (2026-07-01, primary workspace): ~300–350k blocks; the spike's backfill over it took 5–10 minutes (expected — it created ~650k rows through repo.tx, paying row_events + ps_crud envelopes per insert, then uploaded them) and lands at ~1M rows total. The projection direction is much lighter — one indexed join scan, ~325k trivial decodes, ~325k local cell UPDATEs, no inserts, no upload — but at this scale it is not free, and in-browser wa-sqlite/OPFS runs 2–5× slower than native. Measure it in tiers, no infrastructure needed:

  1. Read-side floor (zero setup, live client): yarn agent sql all timing the aggregate join — field rows (reference_target_id restricted to registered schema fieldIds; plain IS NOT NULL also catches ordinary exact-reference rows the derive processor stamps, over-counting the join and skewing the gate) joined to their value children, grouped by parent — over the already-materialized live DB. Real engine, real data, read-only.
  2. Full pass offline (~40-line Node script): export the local DB (the corruption-recovery backup zip yields the .db), open with better-sqlite3, run join → decode (the pure helpers in src/data/propertyChildren.ts; kernel/plugin declarations import directly — their per-workspace fieldIds are computable offline from seedKeys — and user schemas rebuild from the property-schema rows in the same file) → chunked UPDATEs of ~1k rows/tx. Wall-clock it; multiply by ~3× for the browser.
  3. Browser-fidelity write test (live client, safe): yarn agent eval a chunked pass writing to a scratch copy (CREATE TABLE bench_projection AS SELECT id, properties_json FROM blocks, then UPDATE the copy) — real OPFS write throughput without touching live columns.
  4. The independent cost — 1M-row initial sync: if the spike run's backfill drained its upload queue, the children already exist server-side: open the workspace in a clean browser profile on the spike branch and time first sync. This measures the fresh-device cost of materialize-everything itself — it exists no matter what the cell does, and it triples the row count PowerSync moves on first sync.

Decision rule: if the gated, visible-panel-prioritized projection pass (tiers 2–3) lands under ~10–15 seconds, take the upload and download stops — the cell becomes fully local, and wire cost for property updates drops below today's. If it doesn't, stop after step 2 and keep the cell uploading: a frozen server cell is not a usable long-term hint (it goes stale from the moment uploads stop), so the upload stop without the download stop is a transition, not a destination. Cold start then stays at today's behavior, receivers still never merge from the cell, and the only cost kept is the bag upload per property write. (A server-side derived hint — SQL aggregation over child rows refreshing the column for plaintext workspaces — could reopen the upload stop as a stopping point later; that's future server work, outside this plan.) Nothing about correctness rides on this choice — that's the point of the staging.

Implementation note for the upload/download stops: the Layout-B materializer applies sync rows with an explicit-column INSERT OR REPLACE into blocks; for rows whose cell has stopped uploading it must preserve (or immediately re-project) the local properties_json instead of wiping it on every arrival — needed from the upload stop on, not the download stop (the still-downloading frozen cell would otherwise stomp the projected cell on every remote edit). It's the single seam that applies sync writes — one place to get right. The first-sync projection pass reuses the backfill's harness (schema-readiness gating, per-workspace markers, chunked transactions, throughput logging), pointed the other direction — with one structural wrinkle (PR-review P2): the pass must be two-phase, because user-schema discovery itself reads the cell. userSchemasProjector finds schema blocks through types: ['property-schema'], and block_types is populated from properties_json.$.types — with no downloaded cell those entries don't exist yet. Schema unification dissolved the phase's hard dependency (its §4.4): phase 1's recognizer — the fieldIds of code-declared schemas — is a pure computation, uuidv5(workspaceId:seedKey) over the loaded declarations, needing zero DB state; and registry behavior needs no bootstrap at all (declaration synthesis serves every code-declared schema with zero rows synced). Phase 1 projects those schemas' keys (above all types and the property-schema:* config props) from field rows, which repopulates types/block_types and lets the projector register user definitions; phase 2 then projects user-schema keys. A single schema-readiness-gated pass would deadlock — waiting on schemas only the pass itself can surface — or complete early with user schemas unregistered. And the pass's cell writes must not ride repo.tx (PR-review P2): txEngine advances updated_at even under skipMetadata — that's the LWW clock, and advancing it locally can leave the cached row "newer" than genuine later server snapshots, which the acceptance path would then reject — and its upload triggers would re-ship the very cell channel the upload stop removed. The projection rewrites properties_json through the same sanctioned raw path the sync materializer uses (source-null, no upload enqueue, metadata untouched); it borrows the backfill's harness for gating and chunking, not its write path. And the raw write needs an explicit invalidation fan-out (PR-review P2): precisely because updated_at is preserved, the row-version never advances — the cache/hydrate LWW gate (applyIfNewer) sees nothing newer and raw SQL emits no invalidations — so a block already hydrated or query-cached before the pass reaches it would keep serving the stale cell until an unrelated edit. The pass must evict-or-force-refresh the touched blocks' handles per chunk (the observer's invalidation fan-out pointed at the projected ids), never rely on row-version deltas to propagate its repairs.

12. Risks to monitor

  • Write amplification at parents. Same shape as rev 1, now measurable: the spike added per-processor timing to the commit pipeline (TxTimingDiagnostics.sameTxProcessorRuns). Watch it during slice B.
  • Backfill upload burst per device. Completion markers are local (client_schema_state), so every device scans; the NOT-EXISTS candidate filter makes re-materialization a no-op once one device's rows sync down. Two devices racing before convergence duplicate field rows, and — per the §5 finding — the dedup does not fire passively on arrival as the spike stands (the catch-up never visits the shared parent); slice C's parent-dedup fix plus the hold-until-first-sync-drained requirement close this. Gate slice C on a two-client concurrent-backfill test.
  • Local history storage multiplies — optimize row_events before slice C (history is kept; events shrink). row_events is an unbounded, write-only audit log — full before+after row snapshots per update, trigger-maintained for local and sync-applied writes, no runtime consumer, no client-side VACUUM (clientSchema.ts) — already at ~1.17M rows × ~2 KB on a dev instance. Unmitigated, the backfill's ~650k creates would append a snapshot each on the authoring device and again per receiving device as the rows sync down. Direction (no pruning — history stays; a dedicated design task owns the full spec, this bullet records the constraints #288 needs): (1) compact encoding with sampled anchors — store only changed columns for updates; before_json becomes an optional field written periodically (~1-in-50) as a full-snapshot recovery anchor, so reconstruction is anchored rather than full-chain replay back to the insert. One invariant stands regardless (PR-review P2): every chain starts full — a compact update targeting a row with no prior event promotes itself to a full after snapshot, so no exemption ever leaves a later edit without a base. (2) derived-write logging level — open, owned by the design task. Skipping derived-machinery writes (backfill, children→cell projection, sweep, catch-up) entirely may be too harsh — derived-write history is exactly what debugging the reconcile wants; the task weighs skip vs compact-log vs sample. Two constraints are firm: the backfill burst must not write ~650k full snapshots per device (the slice-C gate), and any exemption is by provenance, never by column — skipping properties_json wholesale would lose primary history for un-flipped workspaces and the whole pre-flip period (and §6's migrate-everything decision adds a knob the task should carry: UiState-scoped keystroke-frequency props want a provenance-level exemption of their own). (3) sync-source logging decided per workspace mode — redundant with the server-side blocks_history for plaintext workspaces; the only plaintext history for e2ee ones (starting position: drop for plaintext, keep compact for e2ee; the design task verifies blocks_history's coverage before relying on it). (4) migration of the existing ~1.17M events into the new format — chunked, resumable, idle-scheduled, with old before_json values sampled into anchors rather than dropped wholesale — plus idle VACUUM to reclaim the bytes. Re-stamp docs/row-events-retention.md when touching this — it references the deleted rowEventsTail path, and its prune design is superseded by this direction. The blocks_synced/ps_oplog full copies tripling with row count are separate — load-bearing sync machinery, priced by the tier-4 measurement, not an optimization target here.
  • Schema-registry timing. Derivation quality depends on which schemas are registered when a pass runs. Since schema unification the repo supplies the backbone: active-workspace transactions queue at a shared readiness boundary until the definition projector's first fresh snapshot (its §5.1 prime gate), and declaration behavior is available with zero rows. The spike's backfill-specific gating (held until projector publish, generation-coalesced rescans, row-targeted catch-up bypassing markers) layers on top; keep its tests where they still apply.
  • findSchemaByFieldId-style scans are retired. The spike's O(schemas)-per-lookup helper sat in child-filtering paths that run per rendered row. The unified registry is fieldId-keyed (projected definitions key by metadata.fieldId; the resolver exposes resolveField) — extraction consumes that instead of porting the scan.
  • Old-client interop during rollout. A device on pre-slice-B code writes cell-only. Only its new-key sets converge without help (no children exist for the projection to fight with). Its updates to already-backfilled keys and its clears converge only once the §5 checksum discriminator and the old-client materialize branches land with slice C — before that, an old client's update to a child-backed key is silently reverted by the next projection, and its deletes resurrect on the next touch (PR-review P1 caught the update half of this). Coordinated upgrade (small fleet) is the honest mitigation for the pre-slice-C window, not the reconcile pass; the §5 fix closes it properly.
  • Stale bootstrap-hint window. A fresh device reads downloaded cells (and runs json_extract queries against them) before its projection pass has swept each block; a hint cell can be per-key stale from server column-LWW. Bounded and self-healing, but real: prioritize projection for visible blocks, and don't treat early query results on a cold device as authoritative. Goes away entirely at the download stop — and would worsen unboundedly under an upload-only stop (frozen server cell), which is why the two stops share one gate (§11).
  • Fresh-device initial sync moves ~3× the rows. Materialize-everything takes the primary workspace from ~350k to ~1M rows; PowerSync first-sync time scales with row count even though property bytes mostly moved rather than grew. This cost is independent of the cell question — measure it (tier 4 in §11) before the flip, and treat prioritized/lazy sync buckets as the eventual mitigation if it hurts.
  • Undo/replay. Same-tx processors are skipped on undo/redo replay by design (snapshot restoration is authoritative). Backfill writes are system-scoped and undo-free (tested in the spike). Verify undo of a value-child edit restores both the child and the projected cell (they're one tx, so snapshots cover both — the existing test should assert it explicitly).
