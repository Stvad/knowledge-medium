Migrate user-defined property values from the properties_json cell to first-class block children ("field rows" + "value children"), keeping the cell as a derived projection. Re-grounded against the current data layer and the working prototype in PR #285. Supersedes rev 1 (PR #55).

The load-bearing corrections, stated as final positions:
  • Children-authoritative arrival — the spike's cell-authoritative catch-up re-creates the per-key clobber between current clients; arrival is a reconcile pass and the cell is never a merge input for child-backed keys (§5).
  • The old-client discriminator is a self-describing, id-bound projection checksum — presence markers fail because an old client's whole-bag read-modify-write preserves them; no-children keys are pending materialization, never deleted (§5).

  • The cell sync-off is measurement-gated and composition-scoped: a row leaves the cell channel only when every key on it is known child-backed; steps 4+5 go together, the fallback is step 3, and the column survives as the carrier for carve-out, un-flipped-plugin, and unclassifiable keys (§11).

1. Why move

Property schemas are blocks ('property-schema' type, projected into the schema registry by userSchemasProjector). Property values are entries in a JSON cell — the properties_json column on the consuming block, read through block.get(schema) and written through tx.setProperty. Schemas get the full block treatment — refs, comments, history, sync, undo. Values get a row-local property bag with custom plumbing for every operation that needs to cross the bag boundary.

The asymmetry leaks into a half-dozen features. Copy-paste serializes blocks and copies the JSON cell separately — round-tripping through markdown loses property data entirely. Subtree-sharing has to special-case the cell. The Roam importer writes schemas and values through different code paths. Comments on a field value, references to a specific field's setting, embedding a single field elsewhere — all require new surface every time.

Since rev 1, the sync-layer motivation has sharpened. PR #52 made the upload pipeline column-narrow (apply_block_patches): a content edit no longer clobbers a concurrent property edit. What remains is per-key last-write-wins inside properties_json itself — client A sets foo, client B sets bar, both PATCH the same column, one key silently disappears. docs/field-level-sync-merge.md designed a server-side per-key merge for exactly this, and put it on hold in favor of this project: each round of review surfaced another way per-key merge semantics fight the row-LWW + cache-acceptance + invalidation pipeline. Properties-as-children dissolves the problem structurally — each field is its own row, and rows already merge per-column. This document is now the only live plan for closing that gap.

The motivating reframe stands: the user-facing properties UX is "fields are children of the block, rendered with structure" — that's what every Tana-style editor does. We've been emulating it with a sidecar panel because the data model said no.

2. The target shape

A property on a block becomes a field row (rev 1 called this a "carrier"): a regular block whose content is a reference to the property schema, with a denormalized reference_target_id column carrying the schema's stable fieldId. The field row's children are the property's values — one for scalars today, many for lists later; ref-typed values are themselves block references. Same primitive everywhere.

Things to notice, updated from rev 1:

  • Field-row content is [[Schema Name]], not ((schema-id)). The spike made this call and it's the right one — see §7 for the tradeoff. Detection never parses content at read sites; it reads the reference_target_id column, derived in the same transaction by a processor.
  • Identity is PropertySchema.fieldId: the 'property-schema' block's row id for user-defined schemas, a deterministic property:<name> string for kernel/plugin schemas. name stays the display label and the projection key in the cell; renames change name without touching stored children.
  • Scalar-first. The prototype projects one primary value child per field row; list values serialize as JSON into that one child for now. Multi-value-as-N-children is a follow-up slice (§11), not part of the first cut. This is a deliberate narrowing from rev 1, which promised list-cardinality-as-children up front.
  • Ref-typed values reuse the block-reference primitive: a value child whose content is ((target-id)) is the value and the backlink edge in one shape.

The chosen shape mirrors the existing references[] pattern: the source of truth lives where the user writes (children for properties, content markdown for references), and same-tx machinery maintains a denormalized cell that downstream consumers (typed queries, block.get, panel, search predicates) already read. The same-tx processor registry this relies on (defineSameTxProcessor, commit-pipeline step 3.5, core.normalizeReferences as prior art) exists on master and is exactly the seam rev 1 hoped for.

5. Mechanism — as built in the spike

Three same-tx processors plus a dual-writing tx.setProperty. All of this exists on the spike branch with tests; names below are the real ones.

core.deriveReferenceTarget (permanent)
Watches content. When a row's content trims to exactly one reference token (((uuid)) or [[alias]]), resolves it — schema-name map first, then the tx-aware alias lookup — and writes the target id into reference_target_id in the same transaction (skipMetadata: bumps the sync clock, not "last edited"). Detection everywhere downstream is a column read, never a content parse.
core.projectPropertyChildren (permanent)
Watches content / referenceTargetId / parentId / orderKey / deleted. For each changed row, walks up at most two levels to find affected (parent, fieldId) pairs — both before and after sides of a move — and rebuilds that parent's properties_json key from the field row's value children. This is what keeps manual tree edits of field/value rows convergent with the cell.
core.materializePropertyChildren (permanent, direction: cell → children)
Watches properties. For changed keys with a registered schema: find-or-create the field row + primary value child, update stale content, delete children for removed keys, collapse duplicates. This keeps raw cell writes (importers, plugins, old clients' synced rows via the catch-up pass) convergent with the children. Unlike rev 1's plan, this is not deletable scaffolding — it is the permanent convergence path for the cell-write direction.

Dual-write: tx.setProperty(id, schema, value) writes the field/value children and the cell in one transaction. Rev 1 wanted "write children, let a processor derive the cell"; the spike's shape is better on the hot path — no processor-ordering dependency for the common write, readers (block.get, typed queries, the panel) stay synchronous against the cell, and idempotent processors turn the round-trip into no-ops (the callout below states the invariants that make that safe).

Merge semantics: children are the only truth that crosses sync

This subsection is the heart of the design, and it is a correction — the spike gets it wrong, and rev 2 as first reviewed inherited the mistake. The requirement: the cell is never a merge input for child-backed keys. Children are the only property state that crosses the sync boundary as truth.

Why: watch two current, fully-dual-writing clients clobber each other under the spike's arrival handling. Client A sets status on block P; client B concurrently sets priority. Each uploads its value-child row plus P's full re-projected cell. The child rows are separate rows — row-level LWW keeps both, which is exactly the per-key merge we want. But properties_json is one column carrying the whole bag, so column-LWW picks one writer's cell wholesale — say B's, which carries a stale copy of status. Then the spike's catch-up runs materialize (cell → children) over the arrived parent on every client, sees cell status: "old" vs child "done", "repairs" the child to match the stale cell, and uploads that. A watches its own edit revert, fleet-wide. The dual-write invariant holds for each writer individually; the server's column-vs-row LWW recombines A's children with B's cell, and cell-authoritative arrival then enforces the wrong winner. Effective merge semantics: whole-cell LWW — the exact failure this migration exists to fix.

The fix is to flip arrival authority. On sync arrival, the incoming cell's child-backed keys are ignored; those keys are re-projected from local children, which converge per-row. The cell's roles shrink to exactly three, none of them merge:

  • Local read/query surface — block.get and json_extract queries keep their sync, free reads; the value they read is locally derived (same-tx by the writer, arrival-hook for remote changes).
  • Bootstrap hint — a fresh device reads downloaded cells before its projection pass has swept each block. A hint can be per-key stale (it's whichever writer's bag last won column-LWW); the projection pass corrects it. Bounded, self-healing — and it applies to json_extract queries on a fresh device too, not just reads.
  • Carrier for non-child-backed keys — any schema left cell-backed keeps today's cell-LWW semantics. Per §6 that means the three non-document scopes (UiState / UserPrefs / Automation — per-user or machine-authored, effectively uncontended) and plugin BlockDefault schemas that haven't yet made their durable opt-in, for which cell-LWW is simply today's semantics retained until the flip.

Convergence is two-layered. Locally: projection is deterministic, so once the child rows have arrived, every client derives a byte-identical cell. Server-side: the stored cell re-converges the next time anyone writes a property on that block (their upload carries a freshly-projected bag); between writes it may sit per-key stale, and that's fine because nothing consumes it except the bootstrap-hint path. Deletes need no special case among current clients: a property delete is a child soft-delete row, which syncs per-row and re-projects on arrival like any other child change.

The flip must be total. Materialize-on-arrival (cell → children) must run only where the cell is genuinely the newer truth: rows written by old, pre-migration clients (cell-only writers) and backfill convergence. Any path that lets an arriving cell overwrite existing children for a dual-writing fleet re-opens the clobber through the back door — this is precisely what the spike's current row-targeted catch-up does and the single most important thing to change when extracting it.

Sync arrival — mechanics

Sync-applied rows never pass through repo.tx, so processors don't fire on them. The spike hooks the Layout-B observer: onRowsAccepted(snapshots) fires once per drain window after applySyncInvalidation, and the repo schedules a row-targeted pass over the accepted rows through the serialized backfill queue. As built in the spike that pass runs materialize (cell-authoritative — wrong per the callout above). As specified here it becomes a reconcile pass, per arrived row:

  • Arrived field/value rows → walk up to the parent, re-project its child-backed keys from local children, and dedup duplicate field rows at the parent while there.
  • Arrived parent cells → classify the cell first (the discriminator in the callout below). A current-client cell (projection checksum matches) is never a merge input: re-project child-backed keys from local children; apply the cell only for keys that are not child-backed. A child-backed key with no local child rows at all is left untouched as pending materialization — children-in-flight from the writer's dual-write, landing next drain (under the id-bound checksum a copied bag can't reach this branch: it mismatches and materializes); removing a cell key because children are absent is forbidden everywhere — arrival, projection rebuilds, and the sweep alike. An old-client cell (checksum mismatched or absent) is the one place the full materialize direction still runs — sets, updates to keys that already have children, and clears (the deletion diff below) — because that bag genuinely is its writer's newest truth. Two guards apply even there: a child is overwritten only if its own updated_at predates the arrived bag's (the per-row clock the pipeline already trusts), and a key whose local children are all soft-deleted is not resurrected past a newer deletion.
  • Arrived rows whose cell carries a key no registered schema matches, on a block whose children carry a known fieldId → re-project the cell from those children (the offline-rename race; §7, slice C). Still project-direction: the registry is consulted by fieldId instead of by the stale name.
Two gaps the spike's catch-up leaves open (folded into the reconcile pass above; stated separately because each has a concrete failure):
  • Old-client clears resurrect — and the discriminator that closes them.

    The failure: a pre-migration client (cell-only writer) removes key status; the PATCH arrives at client B, whose [[status]] field row is untouched — the spike's set-only catch-up sees no present key to process. B's tree still shows the value, and the next touch of that subtree re-projects it into the cell and re-uploads: the delete is undone fleet-wide. The reconcile's deletion diff closes this — but it must never fire on cell shape alone. "Arrived cell omits key K while local children for K exist" is also the signature of an innocent current-client bag (a writer that never had K, winning column-LWW); soft-deleting on that re-opens the clobber for deletes.

    The discriminator (shaped by three review rounds): a presence marker fails — an old client's setProperty is a read-modify-write of the whole bag, so a reserved marker key written by the last current client survives the old client's edit untouched, and column-narrow upload means a separate stamp column is never touched either; a marker masks exactly the writes it must expose. The discriminator is therefore a projection checksum with three properties. Content-bound: a canonical hash over the covered keys' values, so any non-projecting edit breaks it. Id-bound: the block id is hashed in, so a wholesale-copied bag mismatches on the new block and takes the old-client branch instead of being mis-certified. Self-describing: the stored record is {hash, coveredKeys}, and verification recomputes over exactly the listed keys — never over "what this client's registry considers child-backed", which varies with plugin/version skew and would let a client missing a plugin mis-classify a peer's fresh projection as an old-client bag (PR-review P1). The registry decides what to re-project; it plays no part in classification. Match ⇒ a current client's projection ⇒ never a merge input. Mismatch or absent ⇒ last modified by a non-projecting writer ⇒ the full materialize direction (set / update / clear) is licensed for that row's child-backed keys, clock-guarded per child (previous list).

    Storage (PR-review P2): the record must be invisible to every generic property surface — the panel's unregistered-keys section, copy/extract/import paths, key enumeration — or every block grows a visible synthetic property. Two sound homes, slice C picks: a reserved, filtered bag key (atomic with the bag by construction — it rides the same column through LWW and e2ee envelopes — but every enumerator must filter the reserved namespace through one shared helper), or a dedicated synced column (invisible to property consumers by construction; atomicity holds because current clients write bag + checksum in the same patch, so per-column LWW picks both from the same winner, while an old client's bag-only patch leaves the checksum column behind — exactly the mismatch signal; costs a migration and an upload-whitelist entry).

    Residual and fallback: an old-client bag is a whole-bag write, so accepting it can carry stale copies of other keys on that block — inherent to mixed-version operation (the old client has no per-key channel), bounded by the per-child clock guard and the upgrade window. Among current clients the cell is never consulted — deletes travel exclusively as child soft-delete rows. If the checksum slips out of slice C, ship without the old-client materialize branches and close the whole mixed-version gap by coordinated fleet upgrade instead (§12): a wrong-direction overwrite is worse than a temporarily unconverged edit.

  • Duplicate field rows from concurrent backfills persist. Two devices backfilling offline each create their own [[status]] row; after sync, the arrived rows are field/value rows whose own cells are empty, so the spike's row-targeted catch-up no-ops on them and never visits the shared parent — where the dedup lives. The reconcile pass walks arrived field rows (recognizable by reference_target_id) up to their parent and dedups there. Until then, duplicate rows persist until some unrelated local edit happens to touch the parent.
Why the processor pair doesn't ping-pong. Materialize (cell → children) and project (children → cell) watch each other's outputs; convergence rests on two load-bearing properties, currently implicit in the spike: every write is skipped when output already equals input (idempotence), and duplicate resolution is deterministic (ORDER BY order_key, id), so all replicas pick the same survivor. State both in code comments next to the processors; a refactor that breaks either turns the pair into a loop or a divergence engine.
Consistency sweep — fsck for the projection. The design's main standing risk is invariant density: correctness rests on every write path honoring the dual-write and arrival rules, and a future path that doesn't (importer, plugin, server feature) fails silently — the cell and the children just drift. Deterministic projection makes that cheap to police: a background sweep recomputes each block's child-backed cell keys from its children, byte-compares against the stored cell, and repairs divergence — with the arrival reconcile's projection-side branch rules, not blanket children-win (PR-review P2; checksum classification is arrival-only — the sweep repairs local projection state and never consumes a remote bag): keys with live children → project, children win; keys whose children are all soft-deleted → the deletion won, drop the key; cell keys with no child rows at all → pending materialization, not divergence — that is exactly the slice-C backfill-failure fallback where the cell stays authoritative for the block, so the sweep materializes them (a free backfill retry) or leaves them untouched and reports, but never deletes. Divergence counts are telemetry. Reuse the backfill harness (marker-gated per workspace, chunked transactions, idle scheduling) in verify mode; ship it with slice C. A nonzero counter is the alarm that some write path broke the invariant — it converts "discipline held in code" into "violations are detected and self-heal", which is the difference between an invariant and a hope.
Denoted-value rule (unchanged, implemented). Mutations deep in a value child's subtree — a comment under a long-text value — do not touch the parent's cell. The projection watches field rows and their direct value children only. Without this rule the processor becomes a reactive query engine and write amplification explodes.
Upload reality — and the cell's exit path. Every write above uploads (column-narrow, but it uploads): field/value child rows are ordinary synced rows; the derived cell write ships the properties_json column. There is no "derived, don't upload" tag on master and we don't need one yet — under the merge semantics above the uploaded cell is harmless (never a merge input) and useful (bootstrap hint). Budget roughly 2 extra rows per (block, property) pair on first materialization plus one cell-column PATCH per property write; the backfill of a large workspace is a real upload burst (§11, slice C). The end state goes further: once children are the fleet-wide source, the cell can stop syncing for fully child-backed rows and become a locally-derived column there (the §11 composition rule — rows carrying carve-out, un-flipped-plugin, or unclassifiable keys keep the channel) — at which point a property update is a small value-child PATCH plus the parent's metadata PATCH (setProperty bumps updated_at/updated_by on the parent, and those columns sync), still well under re-shipping the whole bag, i.e. cheaper on the wire than today — but that step is deliberately staged and gated on the hydration measurements (§11).

6. The scope decision: which schemas get children

This is the fork in the road. properties-model-evolution.md (May 8) chose A + D: children for user-defined fields, cache-only for system fields. The spike flipped it to A only — every registered schema materializes, including types, system:collapsed, alias, and the property-schema:* config props on schema blocks themselves. An earlier draft of this doc argued for restoring the A+D boundary as a per-schema flag; the §5 merge-semantics analysis reversed that. Decision: materialize everything in ChangeScope.BlockDefault — kernel, plugin, and user schemas alike (plugins per-schema, behind the durable opt-in below) — with system field rows hidden by default behind the existing per-schema hidden override and "Show Hidden Fields" toggle. The non-document scopes (UiState, UserPrefs, Automation) stay cell-backed.

What flipped the call: whatever stays cell-synced stays clobberable. Under §5, per-key merge exists only for keys that are rows; a key that lives solely in the cell still rides whole-bag column-LWW, and every dual-write of any property re-uploads the whole cell — carrying stale copies of every cell-backed key with it. A concurrent system:collapsed toggle would be clobbered by an unrelated property write on the same block, and no child exists to repair it from. The one contended kernel prop (collapsed) is precisely the one that most needs row-granular merge. Child-backing it also clears the path on wire cost — though only at the end state: while the cell still syncs, each toggle ships the value-child PATCH plus the re-projected whole bag (more wire than today, not less); once the cell stops uploading for a row (§11, step 4 — the row must pass the composition test: every key on it known child-backed), a toggle is the value-child PATCH plus the parent's metadata bump — cheaper than re-shipping the bag. The wire argument is a consequence of the sync-off, not an immediate saving; the merge argument above is what decides the flip.

The costs the spike surfaced are real but turned out to be presentation-layer, and the spike itself has since built the answers: hidden-field filtering in the outline (per-schema hidden + toggle — though the spike's hidden set is a curated list that must grow to cover every system schema at the kernel flip; §10), tx.childrenOf default-excluding field rows so structural callers see only visible siblings (f4d0b447), and panel-layout/orphan-alias special cases. Two genuine residuals to carry, eyes open:

  • Master's ~8 tree-shape tests need porting. Under materialize-everything they assert a pre-feature tree shape that is now wrong by design. Note the failure list predates the childrenOf default-exclude flip — re-run the spike suite to get the true residual set before porting (several of the 8 may already pass).
  • Synthetic fieldIds leak into references. Kernel schemas' property:<name> pseudo-ids aren't block ids, but the references processor's exact-reference branch projects them into references_json, putting non-existent target ids into the backlink index. Slice B must special-case non-block fieldIds out of reference projection (or give kernel schemas real definition blocks).

The UiState carve-out stays, and it is not a violation of "properties are children" so much as a recognition that those aren't shared properties: they live on per-user state rows (ui-state children under the user page), are written at keystroke frequency (editorSelection, focus, selection), and have no cross-user contention — cell-LWW is harmless there, and pushing cursor movement through the materialize/project machinery buys nothing. If a UiState prop ever becomes contended across a user's devices in a way that matters, child-back it then. The same reasoning extends to the other two non-document scopes the codebase actually has — §6 as first written silently decided only two of the four (PR-review P1): ChangeScope.UserPrefs (user-owned preferences, e.g. plugin prefs) and ChangeScope.Automation (machine-authored records — diagnostics, startup metrics — panel-surfaced but never hand-edited as documents). Both sync through the same cell path, both are per-user or program-written with negligible cross-writer contention, and neither wants field rows in a document tree. All three non-document scopes stay cell-backed permanently; only BlockDefault materializes — and their rows keep the cell sync channel under the §11 sync-off scoping.

Mechanism: keep the per-schema childBacked flag — as rollout gate and UiState carve-out, not as a kernel/user boundary. Add childBacked: boolean to PropertySchema (alongside fieldId). End-state defaults: true for kernel and user BlockDefault schemas; plugin schemas default false and flip per-schema only once the opt-in is durably persisted (below) — defaulting them true from code would flip before any synced decision exists, which is the data fork this callout forbids (PR-review P2); false — permanently — for UiState, UserPrefs, and Automation. During rollout the flag is the staging lever: user schemas flip first, kernel props follow at the §11 kernel flip (tier-4-informed and test-ported, but required regardless of the sync-off decision). All read/write machinery branches on the flag in exactly one place each; one write model (setProperty), one detection rule, one projection. (Sourcing the flag durably where it can vary, and migrating data when it flips, is separate plumbing — the requirements below.)

Two hard requirements the flag carries, both from adversarial review:

  • The flag must be durable where it can vary. For user schemas it is: it derives from the 'property-schema' block, which syncs, so every client agrees. A plugin opt-in expressed only as a code constant is a data fork waiting to happen — two clients on different plugin versions would disagree on whether schema S is child-backed, one maintaining children the other ignores. Plugin opt-ins must persist the decision in synced data (the natural home: the same projector-fed registry mechanism user schemas use), and flipping the flag in either direction is a migration — run a final projection pass, then materialize or delete children under the new regime — never a silent constant change.
  • Name collisions become a hard error. The registry is a last-wins Map<name, schema> with a console.warn (mergeLiftedSchemas). That was tolerable when a collision meant one codec silently won; under child-backing, name is the cell key but fieldId is the child identity, so a user schema and a plugin schema sharing a name strand the loser's field rows (unrecognized fieldId) while both fight over one cell key — and which side wins can differ per client with load order. Slice B must make cross-source name collisions fail loudly (or namespace them), not warn-and-proceed.

This answers the question PR #285 explicitly left open ("should UIState props materialize?" — no, and likewise UserPrefs and Automation; everything BlockDefault — yes), and aligns the doc with the spike's original instinct, now for a load-bearing reason: the merge story, not aesthetic uniformity. types keeps its denormalized local index (block_types) and TypeTagger integration regardless — the child rows are its synced source like any other prop, the index stays the query surface.

7. Field-row content: [[name]] + derived column

Rev 1 specified field-row content as ((schema-id)). The spike ships [[Schema Name]] with the resolved fieldId in the synced reference_target_id column, derived same-tx from content. Keep the spike's shape:

  • Markdown round-trip is a headline motivation — and [[priority]] exports as meaningful markdown where ((7f3a…)) doesn't. Copy-paste into another workspace can resolve by name (§9); a uuid never resolves.
  • Renames don't corrupt stored data — but rename support must land with the write path, not after it. The column, not the label, is authoritative, so children survive a rename untouched. The cell does not: it's keyed by name, so after a rename every consuming block still carries the old key, the registry no longer knows that key (materialize skips unknown names — it cannot repair a key it no longer recognizes), and every schema-aware reader falls back to defaultValue while the tree still shows the real value. Mass silent unset. The rename-reproject pass (reprojectPropertyValueChildren, built in the spike, currently untriggered) is therefore part of slice B2, not polish — or renames must be blocked until it lands.
  • Typing a field by hand works. A user (or an importer, or an agent) can type [[status]] as a child and put a value under it; the derive processor resolves the alias to the schema and the project processor folds it into the cell. That's the "it's just blocks" promise made real.
  • The cost: name collisions need a resolution rule (registered schema map wins over generic alias lookup — implemented), and the exact-reference grammar is scoped to whole-block content so prose containing ((…)) never becomes a field (implemented in referenceBlock.ts).

8. E2EE workspaces — new constraint

E2EE workspaces (May 29) encrypt content, properties_json, and references_json into enc:v1 envelopes; a server trigger rejects plaintext in those columns. Reference edges are therefore part of the encrypted surface today. A synced plaintext reference_target_id would leak, per row: "this row is a field instance", "of this schema", and for exact block-ref rows "this row points at that row" — precisely the edge metadata the current design encrypts.

Options, in increasing effort:

  1. Gate: e2ee workspaces stay cell-only for now. The childBacked machinery checks the workspace's encryption mode and behaves as if every schema were cell-backed. Zero leak, zero new code beyond the gate. E2EE users keep today's behavior. Recommended for the first ship.
  2. Derive locally, never upload. Exclude the column from BLOCK_UPLOAD_COLUMNS for e2ee workspaces and run the derive pass on sync arrival (the observer materializes decrypted plaintext locally, so content is available; onRowsAccepted is the hook). Needs a first-sync full derivation pass per device (marker-gated, same pattern as the backfill). This is the real fix; it's mostly plumbing that reuses existing machinery.
  3. Encrypt the column. Defeats the point — the column exists to be an indexable, queryable detection key on the client; the server never queries it. (2) achieves the same privacy with less ceremony.

The gate in option 1 needs its scope stated precisely — and one piece of it must move server-side (PR-review P1). As shipped, the e2ee ciphertext trigger (blocks_require_ciphertext_for_e2ee) checks only the three envelope columns and would happily accept a plaintext reference_target_id from a stale client, a missed hook, or a direct PostgREST write — exactly the untrusted paths that trigger exists to stop. Slice A must extend it to also require reference_target_id IS NULL for e2ee-workspace rows (same migration as the column add; the plaintext→e2ee flip already nulls the column in its bulk rewrite, so mode transitions pass the check). The rest of the gate is client-side:

  • What the gate disables: all of it — the dual-write, both property processors, the backfill, and the derive processor's uploads. Disabling only the write path while leaving projection live re-creates the stale children problem (§11 rollback discussion); the gate means "this workspace has no child-backed machinery running, full stop."
  • Cross-workspace moves/copies de-structure, not just strip. A field row moved or pasted from a plaintext workspace into an e2ee workspace carries its plaintext reference_target_id; nulling the column is necessary but not sufficient (PR-review P2) — detection is column-based and the machinery is gated off in e2ee, so the [[field]] rows and value children would linger as ordinary visible content while the destination cell is the property store. The move path must run the same de-structure step as the mode-transition flow below: project the subtree's field/value children into the destination cell, then null the column and hide or soft-delete the de-structured rows. (Owner: slice B3's move/merge integration.)
  • Mode transitions are migrations. Flipping a backfilled plaintext workspace to e2ee means the column values for its rows were already uploaded — the flip must null them server-side (the encryption migration is already a bulk rewrite; add the column to it), run a project-then-freeze pass over existing children locally, and then hide or soft-delete the de-structured rows: with the column nulled, §9 detection fails and every former field row would otherwise render as a plain [[name]] block with its values as loose children, fleet-wide. The cell is authoritative post-flip; the rows must not linger as clutter. Flipping e2ee → plaintext re-enables the machinery; the ordinary backfill covers materialization. (Owner: the workspace-encryption flip flow itself — these requirements gate that feature, not the §11 slices.)

A side benefit of the §11 cell end-state — for e2ee workspaces, sequenced behind option 2: under the option-1 gate the cell is an e2ee workspace's only property store, so step 4 cannot apply to it. Once derive-locally child-backing (option 2) replaces the gate, stopping the cell upload removes the envelope column from the wire for fully-child-backed rows (the §11 composition rule applies in e2ee too — carve-out and un-flipped-plugin rows keep their encrypted cells); property data for those rows then travels only as ordinary encrypted child-row content.

Two server-surface items move out of "verify later" into hard slice-A deliverables (§11): extend the apply_block_patches whitelist — today the RPC silently drops unknown keys, so while a fresh field row propagates fine (creates go through the PUT/upsert path, which carries all columns), any later change to reference_target_id on an existing row ships as a PATCH and is dropped server-side, with no client-side re-derive on arrival to compensate; the synced detection column would be insert-only over sync. The extension must also be null-aware (PR-review P2): the column must be clearable — a row edited back to ordinary content, or stripped by the e2ee/move paths, must stop being a field row on peers — so the RPC needs the presence-based handling used for nullable parent_id, not the COALESCE(patch, col) pattern that silently ignores JSON null. And re-timestamp every spike-introduced Supabase migration — both 20260527123000 (reference_target_id) and 20260530143000 (recent-content index) sort before already-applied master migrations (20260612…, 20260625…); wrap them in begin; … commit; per the supabase skill while at it.

9. Detection rules and edge cases — as implemented, with divergences

Field-row recognition.
A child of P is a field row iff reference_target_id equals a registered schema's fieldId. Column read, no content parse. Unregistered targets render as plain reference blocks — a pasted field from another workspace degrades to a visible, honest "unknown field" row.
Value decoding.
Spike: first parseable value child wins (order-key order); unparseable children are skipped. If no child parses, the projection removes the key from the cell. (This rule — and default-vs-unset below — presupposes a live field row; a cell key with no child rows at all is pending materialization and is never removed, per the §5 global rule.) Rev 1 said "log + omit, keep data visible" — the spike keeps the rows (data is never destroyed) but the cell-side key vanishes, which downstream readers see as "unset". Acceptable; document that "field with only-invalid values" reads as unset while remaining visible/fixable in the tree.
Duplicates.
Divergence from rev 1 ("keep extras, pick lowest orderKey"): the materialize path deletes duplicate field rows and duplicate value children (soft-delete, so recoverable via history). Projection reads the first field row's first parseable value. Deterministic dedup is the better call under sync races, but slice B3 must tighten two things the spike leaves sharp: (a) the survivor is picked by (order_key, id) — arbitrary relative to content — and deleteSubtree recursively removes the loser including any user-authored sub-children (a comment thread under the losing value silently goes with it). Before deleting: for duplicate value children, relocate the loser's non-value sub-children (comments, elaboration) under the survivor; for duplicate field rows whose surviving values differ, preserve the losing value visibly — e.g. relocate it under the surviving value child (not directly under the field row, where the value-child dedup would just re-collapse it) — silently hiding a divergent value behind first-wins is data loss in effect if not in storage. (b) the two dedup call sites disagree on deletion semantics: materialize subtree-deletes, while writePropertyValueChild shallow-deletes duplicate value children — orphaning their sub-children live under a tombstone. Unify on one (relocate, then subtree-delete). Until multi-value lands, "N children under one field row" stays a reserved shape, not a user affordance.
Default-value vs. unset.
Unchanged from rev 1: zero value children under a live field row ⇒ key omitted from the cell ⇒ block.get falls back to defaultValue. No reader change.
Codec / preset changes are migrations, not edits.
§9's "only-invalid values reads as unset" is fine for one malformed child; it is a mass event when a user changes a field's preset or config (text → number): every existing value child becomes unparseable under the new codec, each consuming block's next projection removes the key, and readers see defaultValue everywhere — silent fleet-wide unset presented as default. Worse, until reprojection runs, the cell still holds the old encoding and the new codec decodes it on the read path. A preset/config change must run an explicit re-encode/validate pass over existing values (with a user-visible "N values can't convert" outcome), not lean on the projection's remove-on-invalid behavior. Same machinery as the rename reproject; build them together in slice B.
Field-row ordering (decided).
Machinery inserts field rows first among children: every creation path uses keyAtStart, which is what the spike ships — so fields cluster above content children as an emergent default, not an enforced invariant. Rev 1's "carriers-then-content with ghosts between" is dropped. Materialization, backfill, and the add-field affordance all follow the insertion rule; orderKey remains authoritative and user-owned, so a field row manually reordered below content children stays where the user put it.
Reparenting a field row = assigning the property (decide in slice D).
The project processor reprojects both sides of a move, so dragging a [[status]] row under a different block gives that block a status property — and removes it from the old parent. That's a coherent "it's just blocks" semantic, but as a silent side effect of an ordinary indent/drag it's surprising (and works on any host, including schema/system blocks). Slice D must either embrace it as the explicit field-assignment affordance (with UI signal on drop) or guard plain structural moves (e.g. reject, or clear reference_target_id on cross-parent moves that aren't the assignment gesture). Recommendation: embrace with signal — guards that silently strip identity create their own confusion.
Ref-typed values → parent references[].
Two paths exist now: (a) the cell-side projection (projectPropertyReferences, sourceField = schema.name) still runs off the bag — unchanged, keeps backlinks-by-field working; (b) field rows and value rows are themselves reference-bearing blocks, so the references plugin indexes them as ordinary rows — meaning a user field's schema block accrues backlinks from every block that uses the field ("where is this field used?" for free). Verify the two paths don't double-count in backlink UIs (dedup key is (sourceField, id); path (b) rows carry sourceField: '' on their own row — the parent's entry stays single).
Merge, orphan cleanup, panel layout, tree primitives.
Implemented in the spike: mergeBlocksInTx reparents only non-field children, subtree-deletes from's field children (fixed at f4d0b447 — a bare delete had orphaned value rows), and re-materializes the merged bag — but slice B3 must add the same preservation rule as the dedup above (PR-review P2): before the subtree-delete, relocate user-authored descendants of the source's value children (comments, elaboration) under the merged result's corresponding value children — a bare subtree-delete silently tombstones them; orphan-alias cleanup deletes generated field rows under alias seats; panel-layout projection ignores field rows; and tx.childrenOf now excludes field rows by default (structural callers must see only visible siblings — the machinery opts in with includePropertyChildren). Under the §6 scoping these special cases apply everywhere — kernel props materialize too, so hidden field rows exist on most blocks and the exclusion rules get more load-bearing, not less. Field rows, user and kernel alike, must not be reparented as content.
Type-add field slots vs ghosts (decided: render-only ghosts).
The spike materializes empty field rows ("slots") when a type is added (TypeTagger → materializePropertyFieldSlotsForExistingRow; since 03236460 a slot whose key already has a cell value materializes the value too — an empty slot had let the reverse projection strip it). Rev 1 argued for render-only ghost rows that materialize on first edit — no storage until data exists, nothing to clean up on type removal. The spike still has no slot cleanup on type removal (_removeTypeInTx only rewrites the types array), so removing a type strands empty [[field]] rows. Decision: render-only ghosts; drop the TypeTagger materialization. If empty slots prove necessary (e.g., for cross-device consistency of field order), add explicit cleanup on type removal.

10. UI: in-tree rendering exists; the panel is not sunset

The spike ships FieldBlockRenderer (field-row line: shape glyph + label linking to the schema block; value editing via the preset's editor; empty-value editor for slot rows) and PropertyValueBlockRenderer, registered in the default renderer chain, plus hidden-field filtering in useChildIds and a "Show Hidden Fields" context-menu toggle. Under §6's materialize-everything decision the hidden mechanism is load-bearing, not cosmetic — and it is not yet complete: the spike's hidden set is a curated list (hiddenKernelPropertyUis in typesPropertyUi.ts), not a default on every kernel schema. system:collapsed and createdAt are in it; types is not — a materialized [[types]] row would be visible in today's outline. Extending hidden: true to every child-backed system schema that should stay out of the outline is a kernel-flip deliverable (§11, end-state step 3), with the toggle as the escape hatch. The property panel remains fully functional alongside — both surfaces read the same cell and write through the same setProperty, so they coexist safely during the transition. Panel sunset is deliberately last (§11) and follows usage, not a schedule. Rev 1's inventory of what survives (value presets, schema registry, resolvePropertyDisplay, PropertySchemaBlockRenderer) matches what the spike actually reuses.

11. Shipping plan — slices extracted from the spike (dormant-flag landing as fallback)

PR #285 proved the design end-to-end, and the branch is actively maintained — it re-merged master again on 2026-07-01 and has been taking review fixes since (03236460, f4d0b447). But as a shippable unit it has structural problems: it's a merge branch carrying ~35 spike commits (40+ on the PR) whose history has already needed two hand-repaired merges against a fast-moving master (one merge obstacle is blockData.ts being binary-to-git, which defeats line-level auto-merging outright), its arrival handling has the §5 clobber inversion baked in, and its test suite disagrees with master's by design (~8 tree-shape tests — which §6 now says get ported deliberately, not reverted).

Three ways to ship it:

  • Land the merge as-is, fix forward. One review of a known-working integrated tree (3667/3677 green at the PR's last stated run, ~880 lines of convergence tests). But it lands the review-identified gaps — above all the §5 arrival-authority inversion that re-creates the clobber, plus deletion convergence, dedup semantics (§9), flag durability (§6), RPC (§8) — as production behavior, then re-touches the same code to fix each.
  • Land the merge behind a dormant flag. Add childBacked defaulting false for all schemas first, so the whole mechanism lands inert, master's tests stay green, and scope flips per-schema in follow-ups. Cheapest path to "mechanism on master"; preserves the spike's tested integration; the natural home for the §8 fixes. Its real cost is review quality: one ~4k-line PR whose inert-ness must be trusted, with the §5/§9 convergence gaps still to be fixed in-place afterward.
  • Extract master-first slices (below), copying spike code verbatim wherever possible — extraction that re-derives the mechanism would add semantic-drift risk for no gain. Each slice lands with its review fixes built in and is reviewed against master's suite.

Recommended: slices, second choice: dormant flag. The deciding factor is that the adversarial review found changes needed inside the core mechanism (the deletion-convergence catch-up, dedup semantics, flag durability), not just around it — landing first and fixing after means shipping convergence machinery with known gaps, and the fix commits churn the same files a reviewer just read. If shipping speed dominates, the dormant-flag path is legitimate — but it must fold in the full review-fix set, not just the flag: the §5 deletion-convergence catch-up, the §6 durability/collision rules, the §8 gate + RPC + migration re-timestamps, and the §9 dedup tightening. A dormant-flag landing that defers those ships the same gaps one flag-flip away from production. Either way, treat PR #285 as the reference implementation and keep it open until its content is fully accounted for.

A
Foundations: column + derive processor

reference_target_id through all three layers (Supabase migration — re-timestamped and wrapped in begin;…commit; per §8; BLOCK_STORAGE_COLUMNS/parse/upload; BlockData.referenceTargetId; regenerate sync-config), referenceBlock.ts grammar, core.deriveReferenceTarget, the partial index, and the §8 items: the e2ee gate (derive skips e2ee workspaces, column excluded from their upload), the server-side trigger extension (reference_target_id IS NULL enforced for e2ee rows — client gates alone are not a guarantee), and the apply_block_patches whitelist extension — null-aware, per §8: clears must propagate (presence-based handling, as for nullable parent_id). Two hard-won requirements from the spike's own review history: the existing-install local migration must ALTER both blocks and blocks_synced (copy the spike's ensureBlockStorageColumns as fixed at 03236460, run before the sync observer can stage rows — the single-table version broke sync-apply on every upgrading device), and decide BlockData.referenceTargetId's optionality up front (the spike made it required, which touches every BlockData literal in the test suite; optional-in, normalized-to-null in parseBlockRow avoids that churn).

Deploy ordering is load-bearing: (1) db push the column, (2) powersync deploy the regenerated sync rule, (3) roll clients. Deploying the sync rule before the Postgres column exists fails the entire blocks stream for everyone; rolling new clients before the rule deploy just reads NULLs (degraded, harmless). Old clients tolerate an extra stream column (raw-table puts bind by name) — that tolerance is what makes the rollout non-atomic-safe; canary one pinned old client before the fleet.

Also the standalone prerequisite fix: replace the literal NUL bytes in src/data/api/blockData.ts with \u0000 escapes so git stops treating the file as binary and can line-merge it again (its binary-ness defeats auto-merging outright — PR #285's merges had to be repaired by hand around it).

No behavior change for existing data — the column stays NULL except on exact-reference rows. Behavior is reversible by dropping the processor; the schema/sync surface is effectively one-way (the deployed sync rule and already-uploaded envelopes reference the column) — acceptable for an additive nullable column, but say it out loud.

"Children stay inert but convergent" is not a coherent state: if the processors are flag-gated off, nothing keeps inert children convergent, and the tree UI would render stale values. Rollback = drain + freeze, not flip and hope.

C
Backfill + sync catch-up

The spike's backfill machinery (markers, candidate scan, bulk insert, retry — including its txEngine bulk-insert/undo-free plumbing, a large and subtle surface worth its own review attention) and the observer onRowsAccepted catch-up, scoped to child-backed schemas — rebuilt as the §5 reconcile pass — this slice IS the clobber fix: arrival re-projects child-backed keys from local children (a current client's cell is never a merge input); materialize direction runs only for genuinely old-client cells, gated on the §5 projection-checksum discriminator — which this slice introduces (current clients embed the self-describing {hash, coveredKeys} record when they project; a mismatched or absent checksum marks an old-client bag) — never on cell shape (mismatched bags get the full set/update/clear materialize, clock-guarded per child and soft-delete-aware — no resurrection past a newer deletion; the deletion diff must not ship without the checksum); child-backed keys with no child rows at all are pending materialization, never deleted; and dedup happens at the parent of arrived field rows. The §5 consistency sweep ships here too — it reuses this slice's harness in verify mode, and its divergence counter is the acceptance metric for the reconcile pass itself. It must also reconcile the rename race B2's one-shot reproject can't reach: a block edited offline across a schema rename syncs up with a cell key no registry knows; the reconcile recognizes that case from the children's side — field rows carry the stable fieldId — and re-projects the cell from them. Two more requirements, not suggestions: the backfill inherits the §8 e2ee gate explicitly (the server trigger won't catch a plaintext-column leak), and the first full backfill on a device is held until the initial sync drain completes (backfilling a partially-synced cell state creates children for stale values).

Sequence the first production run deliberately: it's an upload burst of 2 rows per (block, property) — run workspace-by-workspace, watch ps_crud drain. Before flipping it on, size it: run select count(*) from blocks where properties_json != '{}' and deleted = 0 (and a key histogram) against production via yarn agent sql / Supabase.

100% of user-defined property data is child-backed; the cell is fully derived for user-defined keys (kernel keys stay cell-primary until the end-state step-3 flip — itself part of the §6 correctness fix, not optional). Cell end-state: measure first, then stage the sync-off

With §5's merge semantics in place, the synced cell is harmless for child-backed keys — the clobber fix for those keys does not wait on the sync-off. One item below is still correctness, not optimization: step 3, the kernel flip — until it lands, kernel keys (collapsed above all) are cell-only and still ride whole-bag column-LWW (§6). Steps 4–6 are the pure-optimization tail: whether to stop syncing properties_json at all (wire savings, e2ee simplification, and the cell becomes an ordinary local derived column). The staged sequence, each step gated on the one before:

  1. Arrival-authority flip ships (slice C) and the fleet fully dual-writes. Zero cold-start change; cell still syncs as bootstrap hint.
  2. Backfill completes fleet-wide (slice C) — every property has child rows behind it.
  3. Kernel-prop flip + test porting: flip childBacked defaults for kernel schemas (§6 end state). Plugin schemas flip separately, each behind the durable persisted opt-in §6 requires (PR-review P2): plugin code varies independently of app version (dynamic/agent extensions), so a code-constant flip is exactly the data fork §6 forbids — until the persistence mechanism lands, plugin schemas stay cell-backed. Run the schema-scoped backfill for the newly flipped schemas (markers make this incremental), port the residual tree-shape tests as deliberate behavior changes, extend the curated hidden-override set to every system schema that should stay out of the outline (§10), ship the references-projection guard for synthetic fieldIds. This step completes the §6 correctness fix — it is sequenced here for rollout pacing (tier-4 measurement informs it), but it is required regardless of the step-5 decision.
  4. Stop uploading the cell — per row, by key composition, never globally. The rule (PR-review P1): a row's cell stops uploading only when every key in its bag is known to be child-backed. Three kinds of key therefore keep a row on the channel: §6 carve-out-scope keys (UiState, UserPrefs, Automation — cell-backed permanently); cell-backed plugin BlockDefault keys (§6 defaults plugins false until their durable opt-in, and plugin props like todo status live on ordinary content blocks — freezing those rows would orphan the plugin's only synced store); and keys a client's registry can't classify (a plugin not installed here may be live elsewhere). Practical consequence: opting-in the content-block plugin schemas is effectively a prerequisite for this step delivering much. Semantics for rows that stop: the server copy freezes; receivers never merged from it, so nothing changes for current clients — but a fresh device's downloaded hint decays (every post-flip edit is missing until the projection pass sweeps the block). Status: must come after (1)–(3) — a remaining cell-only writer still needs the channel — and it is a transition, not a stopping point: it shares step 5's projection-readiness gate (PR-review P2).
  5. Stop downloading it — same composition rule: rows not yet fully child-backed keep the column. Mechanically a stream split in the sync rules — which also requires relaxing the Layout-B staging schema (PR-review P2: blocks_synced.properties_json is TEXT NOT NULL and the raw-table put binds every storage column, so a missing stream column fails the insert before the materializer can act; make it nullable or default-'{}' in the same migration) — or keep downloading the column fleet-wide and consume it only where the composition rule says it is live; the download bytes are the smallest of the wins. Fresh devices now need the first-sync projection pass (below) before the cells of fully-child-backed rows are readable. The measurements gate steps 4 and 5 together: take both or neither.
  6. Deprecate the server column for fully-child-backed rows (stale from step 4 on for the rows that stopped uploading; a future server-side feature that wants properties derives them from the child rows with a SQL view/aggregate — the data is all there for plaintext workspaces, which is the only case server features could read anyway). It cannot be dropped: the column survives as the carrier for carve-out-scope, un-flipped-plugin, and unclassifiable keys unless those are someday migrated off blocks.

The measurement that gates step 5 — run against the real database before committing to the no-hint cold start. Known scale (2026-07-01, primary workspace): ~300–350k blocks; the spike's backfill over it took 5–10 minutes (expected — it created ~650k rows through repo.tx, paying row_events + ps_crud envelopes per insert, then uploaded them) and lands at ~1M rows total. The projection direction is much lighter — one indexed join scan, ~325k trivial decodes, ~325k local cell UPDATEs, no inserts, no upload — but at this scale it is not free, and in-browser wa-sqlite/OPFS runs 2–5× slower than native. Measure it in tiers, no infrastructure needed:

  1. Read-side floor (zero setup, live client): yarn agent sql all timing the aggregate join — field rows (reference_target_id restricted to registered schema fieldIds; plain IS NOT NULL also catches ordinary exact-reference rows the derive processor stamps, over-counting the join and skewing the gate) joined to their value children, grouped by parent — over the already-materialized live DB. Real engine, real data, read-only.
  2. Full pass offline (~40-line Node script): export the local DB (the corruption-recovery backup zip yields the .db), open with better-sqlite3, run join → decode (the pure helpers in src/data/propertyChildren.ts; kernel schemas import directly, user schemas rebuild from the property-schema rows in the same file) → chunked UPDATEs of ~1k rows/tx. Wall-clock it; multiply by ~3× for the browser.
  3. Browser-fidelity write test (live client, safe): yarn agent eval a chunked pass writing to a scratch copy (CREATE TABLE bench_projection AS SELECT id, properties_json FROM blocks, then UPDATE the copy) — real OPFS write throughput without touching live columns.
  4. The independent cost — 1M-row initial sync: if the spike run's backfill drained its upload queue, the children already exist server-side: open the workspace in a clean browser profile on the spike branch and time first sync. This measures the fresh-device cost of materialize-everything itself — it exists no matter what the cell does, and it triples the row count PowerSync moves on first sync.

Decision rule: if the gated, visible-panel-prioritized projection pass (tiers 2–3) lands under ~10–15 seconds, take steps 4 and 5 — the cell becomes fully local for fully-child-backed rows, and wire cost for property updates on those rows drops below today's. If it doesn't, stop at step 3 and keep the cell uploading: a frozen server cell is not a usable long-term hint (it goes stale from the flip moment on — see step 4), so step 4 without step 5 is a transition, not a destination. Cold start then stays at today's behavior, receivers still never merge from the cell, and the only cost kept is the bag upload per property write. (A server-side derived hint — SQL aggregation over child rows refreshing the column for plaintext workspaces — could reopen step 4 as a stopping point later; that's future server work, outside this plan.) Nothing about correctness rides on this choice — that's the point of the staging.

Implementation note for step 5: the Layout-B materializer applies sync rows with an explicit-column INSERT OR REPLACE into blocks; once the cell is local-only it must preserve (or immediately re-project) the local properties_json instead of wiping it on every arrival. It's the single seam that applies sync writes — one place to get right. The first-sync projection pass reuses the backfill's harness (schema-readiness gating, per-workspace markers, chunked transactions, throughput logging), pointed the other direction — with one structural wrinkle (PR-review P2): the pass must be two-phase, because schema discovery itself reads the cell. userSchemasProjector finds schema blocks through types: ['property-schema'], and block_types is populated from properties_json.$.types — with no downloaded cell those entries don't exist yet. Phase 1 projects the code-defined schemas' keys (kernel/plugin — above all types and the property-schema:* config props, recognizable by their code-known synthetic fieldIds) from field rows, which repopulates types/block_types and lets the projector register user schemas; phase 2 then projects user-schema keys. A single schema-readiness-gated pass would deadlock — waiting on schemas only the pass itself can surface — or complete early with user schemas unregistered. And the pass's cell writes must not ride repo.tx (PR-review P2): txEngine advances updated_at even under skipMetadata — that's the LWW clock, and advancing it locally can leave the cached row "newer" than genuine later server snapshots, which the acceptance path would then reject — and its upload triggers would re-ship the very cell channel step 4 removed. The projection rewrites properties_json through the same sanctioned raw path the sync materializer uses (source-null, no upload enqueue, metadata untouched); it borrows the backfill's harness for gating and chunking, not its write path.

12. Risks to monitor

  • Write amplification at parents. Same shape as rev 1, now measurable: the spike added per-processor timing to the commit pipeline (TxTimingDiagnostics.sameTxProcessorRuns). Watch it during slice B.
  • Backfill upload burst per device. Completion markers are local (client_schema_state), so every device scans; the NOT-EXISTS candidate filter makes re-materialization a no-op once one device's rows sync down. Two devices racing before convergence duplicate field rows, and — per the §5 finding — the dedup does not fire passively on arrival as the spike stands (the catch-up never visits the shared parent); slice C's parent-dedup fix plus the hold-until-first-sync-drained requirement close this. Gate slice C on a two-client concurrent-backfill test.
  • Schema-registry timing. Derivation quality depends on which schemas are registered when a pass runs. The spike's gating (backfill held until projector publish, generation-coalesced rescans, row-targeted catch-up bypassing markers) is the answer; keep its tests.
  • findSchemaByFieldId is O(schemas) per lookup and sits in child-filtering paths that run per rendered row. Fine at today's schema counts; add a fieldId-keyed map alongside the facet if profiles say otherwise.
  • Old-client interop during rollout. A device on pre-slice-B code writes cell-only. Only its new-key sets converge without help (no children exist for the projection to fight with). Its updates to already-backfilled keys and its clears converge only once the §5 checksum discriminator and the old-client materialize branches land with slice C — before that, an old client's update to a child-backed key is silently reverted by the next projection, and its deletes resurrect on the next touch (PR-review P1 caught the update half of this). Coordinated upgrade (small fleet) is the honest mitigation for the pre-slice-C window, not the reconcile pass; the §5 fix closes it properly.
  • Stale bootstrap-hint window. A fresh device reads downloaded cells (and runs json_extract queries against them) before its projection pass has swept each block; a hint cell can be per-key stale from server column-LWW. Bounded and self-healing, but real: prioritize projection for visible blocks, and don't treat early query results on a cold device as authoritative. Goes away entirely at sync-off step 5 — and would worsen unboundedly under a step-4-only stop (frozen server cell), which is why step 4 shares step 5's gate (§11).
  • Fresh-device initial sync moves ~3× the rows. Materialize-everything takes the primary workspace from ~350k to ~1M rows; PowerSync first-sync time scales with row count even though property bytes mostly moved rather than grew. This cost is independent of the cell question — measure it (tier 4 in §11) before the kernel flip, and treat prioritized/lazy sync buckets as the eventual mitigation if it hurts.
  • Undo/replay. Same-tx processors are skipped on undo/redo replay by design (snapshot restoration is authoritative). Backfill writes are system-scoped and undo-free (tested in the spike). Verify undo of a value-child edit restores both the child and the projected cell (they're one tx, so snapshots cover both — the existing test should assert it explicitly).

Re-stamp it when slices land. (field-level-sync-merge.md's stale pointer to the never-merged properties-as-children.html was fixed alongside this doc.)
