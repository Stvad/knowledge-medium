Decisions made in this doc (each argued in its section):
  • Children are the only property truth that crosses sync. The cell is never a merge input for child-backed keys; arrival re-projects from children (per-row LWW = per-key merge). The cell survives as local read surface, bootstrap hint, and carrier for non-child-backed keys (§5).
  • Materialize everything — all schemas, all scopes — kernel, plugin, and user alike, with system/machine field rows hidden by default and orphan keys promoted to synthesized definitions. No permanent cell-backed category — and no depth carve-out either: field rows' and value rows' own bags materialize too, since the :: grammar makes machinery recognizable at any depth (§9). The flip lever is a per-workspace column, not a per-schema flag (§6).
  • Field-row shape: ::((fieldId)) — the :: field marker + exact-ref content — with derived local reference_target_id + is_field_form — never synced, derived purely textually on every device (no name-resolution tier), which also erases the column's e2ee fork; unmarked [[name]]/((id)) rows are never machinery — hand-authoring is the :: input flow (§7, §8).
  • setProperty dual-writes children and cell in one tx; the materialize/project processor pair is permanent convergence machinery (§5).
  • The arrival pass becomes a reconcile: project-from-children, always — arriving bags are never merge inputs, no classification exists, and non-projecting writers are closed by policy; dedup at parents (§5, slice C).
  • Stopping cell sync is a staged, measurement-gated optimization, per-workspace once backfill-complete and sweep-clean — see the tiered real-DB measurement plan and decision rule; the column is ultimately droppable (§11, cell end-state). The flip itself (slice C, all schemas) is what closes the collapsed clobber — §6 correctness, not optimization.
  • E2EE workspaces stay cell-only until the derive-locally path lands (§8).
  • Machinery inserts field rows first among children (orderKey stays user-owned); render-only ghosts replace the spike's materialized empty slots (§9).
  • Ship as master-first slices extracted from the spike; dormant-flag landing is the fallback; PR #285 is the reference implementation (§11).

1. Why move

Property schemas are blocks — since schema unification, all of them: user-authored 'property-schema' blocks projected into the registry, and kernel/plugin schemas seeded as definition blocks with deterministic per-workspace ids. Property values are entries in a JSON cell — the properties_json column on the consuming block, read through block.get(schema) and written through tx.setProperty. Schemas get the full block treatment — refs, comments, history, sync, undo. Values get a row-local property bag with custom plumbing for every operation that needs to cross the bag boundary.

The asymmetry leaks into a half-dozen features. Copy-paste serializes blocks and copies the JSON cell separately — round-tripping through markdown loses property data entirely. Subtree-sharing has to special-case the cell. The Roam importer writes schemas and values through different code paths. Comments on a field value, references to a specific field's setting, embedding a single field elsewhere — all require new surface every time.

Since rev 1, the sync-layer motivation has sharpened. PR #52 made the upload pipeline column-narrow (apply_block_patches): a content edit no longer clobbers a concurrent property edit. What remains is per-key last-write-wins inside properties_json itself — client A sets foo, client B sets bar, both PATCH the same column, one key silently disappears. docs/field-level-sync-merge.md designed a server-side per-key merge for exactly this, and put it on hold in favor of this project: each round of review surfaced another way per-key merge semantics fight the row-LWW + cache-acceptance + invalidation pipeline. Properties-as-children dissolves the problem structurally — each field is its own row, and rows already merge per-column. This document is now the only live plan for closing that gap.

The motivating reframe stands: the user-facing properties UX is "fields are children of the block, rendered with structure" — that's what every Tana-style editor does. We've been emulating it with a sidecar panel because the data model said no.

2. The target shape

A property on a block becomes a field row (rev 1 called this a "carrier"): a regular block whose content is the :: field marker followed by a reference to the property schema (::((fieldId))), with a denormalized reference_target_id column carrying the schema's stable fieldId. The field row's children are the property's values — one for scalars today, many for lists later; ref-typed values are themselves block references. Same primitive everywhere.

Things to notice, updated from rev 1:

  • Field-row content is ::((fieldId)) — the :: field marker + the definition addressed by id. (The spike used [[Schema Name]]; the implementation went to bare id-content; the 2026-07-22 grammar decision added the marker so field-ness is carried by content itself — §7 has both arguments. Fig. 1's ::priority labels show the rendered tree: the marker renders with the target's label, so the display is the name even though the stored ref is the id.) Detection never parses content at read sites; it reads the reference_target_id + is_field_form columns, derived in the same transaction by a processor.
  • Identity is the definition block id. fieldId = the 'property-schema' block's row id for every schema, kernel and plugin included: schema unification seeds code-declared schemas as definition blocks with deterministic per-workspace ids (uuidv5(workspaceId:seedKey)), and identity reaches write paths only through its tx/workspace-bound resolution primitive (Repo.propertySchemaResolverFor(workspaceId) → branded ResolvedPropertySchema; code handles carry seedKey, never a fieldId). The spike's synthetic property:<name> ids are superseded and never ship — extraction swaps them for resolved ids (§11, slice B). name stays the display label and the projection key in the cell; renames change name without touching stored children.
  • Scalar-first. The prototype projects one primary value child per field row; list values serialize as JSON into that one child for now. Multi-value-as-N-children is a follow-up slice (§11), not part of the first cut. This is a deliberate narrowing from rev 1, which promised list-cardinality-as-children up front.
  • Ref-typed values reuse the block-reference primitive: a value child whose content is ((target-id)) is the value and the backlink edge in one shape.

5. Mechanism — as built

Three same-tx processors plus a dual-writing tx.setProperty. Originally described from the spike branch; slices A–B landed this machinery on the implementation branch (PR #386, with the §7 id-content divergence folded in below) — code + tests there are authoritative.

core.deriveReferenceTarget (permanent)
Watches content. When a row's trimmed content is exactly one reference token — or the :: marker + one block-ref token, the §7 field form — resolves it and writes the target id into reference_target_id in the same transaction, and stamps is_field_form for both marked bracket forms — the marked forms derive exactly like their unmarked twins (owner decision, 07-22 second round: no marked-alias exclusion branch; a ::[[name]] row's target binds via the same alias lookup/late-binding as any link, and classification follows resolution — §9 recognition). As shipped (PR #386, pre-grammar — the two-column condition is on the branch delta): ((id)) resolves purely textually — the id is the content — and [[alias]] resolves through the one tx-aware generic alias lookup; there is no schema-name tier (§7). Write metadata splits by path (PR-review P2, adapted to the as-built): the same-tx processor uses skipMetadata — the row is already uploading with the content edit that triggered it, at the cost of one extra updated_at in that PATCH — while the bulk paths (the per-open catch-up sweep, derive-at-arrival) write the column raw (CAS, no updated_at advance, no uploads), which is where the no-op-upload / false-recency hazard actually lived. Detection everywhere downstream is a column read, never a content parse.
core.projectPropertyChildren (permanent)
Watches content / referenceTargetId / isFieldForm / parentId / orderKey / deleted — the bit is a watched field (PR #417 review: projection's classification and value-set both read it, so a bit-only change — an arrival repair, the catch-up sweep stamping existing marked rows after the column ships — must re-project; bulk repair paths that write the bit raw enqueue projection for the affected parents explicitly). For each changed row, walks up at most two levels to find affected (parent, fieldId) pairs — both before and after sides of a move — and rebuilds that parent's properties_json key from the field row's value children — its is_field_form IS NOT 1 children only, per §9's binding selection discipline (a nested marked row materialized under the field row is that row's own machinery, never a value candidate). This is what keeps manual tree edits of field/value rows convergent with the cell. Registration order is load-bearing (PR-review P2): the same-tx pass is single-pass in registration order, no fixpoint (commitPipeline.ts) — a later processor sees an earlier one's amendments, never the reverse — so derive registers before project, or editing a field row's content (re-pointing it at a different definition, or back to plain text) projects against the stale column: old key kept, new key missed until the next unrelated edit. Slice B pins the order with a test.
core.materializePropertyChildren (permanent, direction: cell → children)
Watches properties. For changed keys with a registered schema: find-or-create the field row + primary value child, update stale content, delete children for removed keys, collapse duplicates — every find/primary/duplicate step selecting by the bit per §9's discipline (field row = is_field_form = 1 child targeting the fieldId; values = the is_field_form IS NOT 1 children; a duplicate field row's own marked children fold as field rows, never as values). Minted field-row content is the marked form, ::((fieldId)) (§7 — the branch's bare-ref mint predates the 07-22 grammar and is on the delta checklist). Rows it creates are born classified: both creation paths (setProperty's writePropertyValueChild and this processor) stamp reference_target_id — and now is_field_form — directly in the create — no reliance on the derive processor running afterwards, so materializer-minted rows classify and project within the same single pass (PR-review P2 asked; verified in the PR #386 code). It runs for every block, machinery rows included — the branch's interior-ancestry gate (bag writes on field/value rows stayed cell-only, "recognition could never reclaim the nested rows") is deleted with the positional model: under the :: grammar, recognition reclaims nested machinery at any depth, so field rows' and value rows' own bags materialize like everyone else's (§9). This keeps raw local cell writes (importers, plugins — any bag-shaped write through repo.tx) convergent with the children; sync arrivals never enter this direction (the one-direction rule below — an earlier revision also fed it old clients' synced rows via a catch-up pass). Unlike rev 1's plan, this is not deletable scaffolding — it is the permanent convergence path for local cell-shaped writes, and it is exactly why repo.tx writers are exempt from the policy-violating writer class below.

Dual-write: tx.setProperty(id, schema, value) writes the field/value children and the cell in one transaction. Rev 1 wanted "write children, let a processor derive the cell"; the spike's shape is better on the hot path — no processor-ordering dependency for the common write, readers (block.get, typed queries, the panel) stay synchronous against the cell, and idempotent processors turn the round-trip into no-ops (the callout below states the invariants that make that safe).

Merge semantics: children are the only truth that crosses sync

This subsection is the heart of the design, and it is a correction — the spike gets it wrong, and rev 2 as first reviewed inherited the mistake. The requirement: the cell is never a merge input for child-backed keys. Children are the only property state that crosses the sync boundary as truth.

Why: watch two current, fully-dual-writing clients clobber each other under the spike's arrival handling. Client A sets status on block P; client B concurrently sets priority. Each uploads its value-child row plus P's full re-projected cell. The child rows are separate rows — row-level LWW keeps both, which is exactly the per-key merge we want. But properties_json is one column carrying the whole bag, so column-LWW picks one writer's cell wholesale — say B's, which carries a stale copy of status. Then the spike's catch-up runs materialize (cell → children) over the arrived parent on every client, sees cell status: "old" vs child "done", "repairs" the child to match the stale cell, and uploads that. A watches its own edit revert, fleet-wide. The dual-write invariant holds for each writer individually; the server's column-vs-row LWW recombines A's children with B's cell, and cell-authoritative arrival then enforces the wrong winner. Effective merge semantics: whole-cell LWW — the exact failure this migration exists to fix.

The fix is to flip arrival authority. On sync arrival, the incoming cell's child-backed keys are ignored; those keys are re-projected from local children, which converge per-row. The cell's roles shrink to exactly three, none of them merge:

  • Local read/query surface — block.get and json_extract queries keep their sync, free reads; the value they read is locally derived (same-tx by the writer, arrival-hook for remote changes).
  • Bootstrap hint — a fresh device reads downloaded cells before its projection pass has swept each block. A hint can be per-key stale (it's whichever writer's bag last won column-LWW); the projection pass corrects it. Bounded, self-healing — and it applies to json_extract queries on a fresh device too, not just reads.
  • Carrier for not-yet-migrated keys — before a workspace flips (and, transitionally, for keys whose backfill hasn't reached a block) the cell keeps today's cell-LWW semantics. Under §6 this is a transitional role only: no key category stays cell-backed permanently, so post-backfill the cell is pure derived projection everywhere.

Convergence is two-layered. Locally: projection is deterministic, so once the child rows have arrived, every client derives a byte-identical cell. Server-side: the stored cell re-converges the next time anyone writes a property on that block (their upload carries a freshly-projected bag); between writes it may sit per-key stale, and that's fine because nothing consumes it except the bootstrap-hint path. Deletes need no special case among current clients: a property delete is a child soft-delete row, which syncs per-row and re-projects on arrival like any other child change.

The flip must be total: no materialize-on-arrival, for anyone. Sync-arrival cell→children materialization exists nowhere after the one-time backfill — not even for rows written by old, pre-migration cell-only clients. Those are a policy-closed writer class (the one-direction callout below): their property updates and deletes are reverted by the next projection, logged by the sweep, recoverable from row_events. An earlier revision of this callout sanctioned an old-client materialize branch (behind a checksum discriminator); that survives only in review-thread history — any path that lets an arriving cell overwrite existing children re-opens the clobber through the back door, which is precisely what the spike's row-targeted catch-up does and the single most important thing to change when extracting it. The only cell→children machinery that survives is local: the same-tx materializer for repo.tx bag writes, which is not a sync path.

Sync arrival — mechanics

Sync-applied rows never pass through repo.tx, so processors don't fire on them. The spike hooks the Layout-B observer: onRowsAccepted(snapshots) fires once per drain window after applySyncInvalidation, and the repo schedules a row-targeted pass over the accepted rows through the serialized backfill queue. One ordering requirement that hook placement misses (PR-review P2): firing after invalidation means the raw column-LWW bag is reader-visible between materialization and the pass's repair — a stale-read window on every concurrent remote edit among current clients, not just the upload stop's frozen-cell case. The cell repair for child-backed keys (re-project from local children) therefore runs inside the materializer seam, before the affected invalidations fan out — the same single sync-write seam the §11 upload stop already modifies; the heavier reconcile work below (dedup, the pending-keys handling) stays in the async pass. As built in the spike that pass runs materialize (cell-authoritative — wrong per the callout above). As specified here it becomes a reconcile pass, per arrived row:

  • Arrived field/value rows → walk up to the parent, re-project its child-backed keys from local children — the cell and the property-sourced references_json entries (§6's wire callout: that column is column-LWW too, and a repair that fixes only the cell leaves the by-field backlinks carrying the losing writer's list) — and dedup duplicate field rows at the parent while there.
  • Arrived parent cells → never a merge input, without classification (owner decision — see the one-direction callout below; the earlier revisions of this doc classified arriving bags with a projection checksum and ran a materialize direction for old-client bags): re-project keys that have local children. A key with no local child rows at all keeps the arrived cell value as pending materialization — a read fallback, children-in-flight from the writer's dual-write or awaiting the backfill; removing a cell key because children are absent is forbidden everywhere — arrival, projection rebuilds, and the sweep alike. The same pending treatment covers per-client buildability degradation (PR-review P2): a key whose schema this client cannot build — a custom-codec plugin schema with the plugin disabled — keeps the arrived value as its read surface (no codec ⇒ it can neither decode children nor re-project; projecting it away would replace its only usable value with nothing). The round-20 stale-carry hazard is gone by construction: such a client's whole-bag uploads still carry its stale copy, but no peer reads an arriving bag as a merge input, so the stale carry can overwrite nothing.
  • Arrived rows whose cell carries a key no registered schema matches, on a block whose children carry a known fieldId → re-project the cell from those children (the offline-rename race; §7, slice C). Still project-direction: the registry is consulted by fieldId instead of by the stale name.
One direction, by policy — and the dedup gap (folded into the reconcile pass above):
  • Sync is children→cell only; the cell is never read back.

    Rounds 14–21 of this PR's review built a full two-way design: a projection checksum ({hash, coveredKeys}, id-bound, containment rule, a storage scheme, a preservedKeys extension) classifying every arriving bag so that non-projecting writers' edits — old clients, raw-SQL writers, buildability-degraded clients — could be honored via a clock-guarded materialize direction, including a deletion diff. Owner decision: delete all of it. The fleet closes every non-projecting writer class by coordination instead of armoring the merge, one closure per class: stale tabs (the update path never force-reloads, so this is social policy — update before continuing to use; a stale tab's property edits are reverted); pre-flip offline queues (runbook: every device online and drained before the flip); raw writers (agent-bridge SQL, ad-hoc server SQL — post-flip convention: write children, not the cell; note that anything going through repo.tx is not in this class, because the same-tx materializer creates children for bag-shaped local writes). After the one-time backfill, cell→children materialization never runs on the sync path again: arrival re-projects from children, period. The same-tx materializer itself stays — it is local write machinery, not sync machinery (PR-review P2 caught the two sentences reading as contradictory): "one-way" constrains what an arriving cell may do, not how a local bag-shaped write becomes children.

    Accepted consequences, stated plainly: a policy-violating writer's property edits — updates and deletes — are reverted by the next projection, silently from that writer's perspective; the sweep below logs the divergence it repairs, and row_events history holds the reverted values for recovery. That recovery claim pins an ordering (PR-review P2): the materializer seam applies the arrived row first and re-projects second — two writes, so the discarded bag lands in row_events ('sync'-sourced) on every device that materialized it before the repair overwrites it. A seam that rewrote the bag before the upsert would discard the value tracelessly. Server-side raw writes are additionally captured by blocks_history for plaintext workspaces — the runbook's second place to look. Deletes among current clients travel exclusively as child soft-delete rows (an old client's cell-key removal is not honored — the old "clears resurrect" failure is resolved by refusing the writer class rather than by classifying it). What this deletes from the design: the checksum record and its verification, the containment rule, the reserved-bag-key / dedicated-column storage question with its e2ee constraints, the old-client materialize branches, the per-child clock guard, and the no-resurrection rules — the full record lives in this PR's review threads (rounds 14–21) as the remedy if the fleet ever outgrows coordination.

  • Duplicate field rows from concurrent dual-writes persist. Two devices offline-editing the same not-yet-materialized property each create their own ::((fieldId)) row (the backfill itself no longer races — it is operator-run-once per workspace, §11); after sync, the arrived rows are field/value rows whose own cells are empty, so the spike's row-targeted catch-up no-ops on them and never visits the shared parent — where the dedup lives. The reconcile pass walks arrived field rows (recognizable by is_field_form + reference_target_id) up to their parent and dedups there — union-with-dedupe (§9): the duplicate field rows fold into one, content-equal values fold with them, and divergent values persist as peer siblings, so a genuine concurrent conflict is surfaced in the tree rather than silently resolved — except view-state-flagged keys (§9's merge entry): there the reconcile keeps the newest value — updated_at LWW, matching the cell era's same-key semantics (PR #417 review: an order-key survivor would let an older collapse toggle win arbitrarily), with order_key, id only as the deterministic tiebreak for equal clocks (no target exists at this site; the merge gesture stays target-wins — an explicit user act, not a clock race) — and drops the rest silently. Until the reconcile visits the parent, duplicate rows persist until some unrelated local edit happens to touch it.
Why the processor pair doesn't ping-pong. Materialize (cell → children) and project (children → cell) watch each other's outputs; convergence rests on two load-bearing properties, currently implicit in the spike: every write is skipped when output already equals input (idempotence), and duplicate resolution is deterministic (ORDER BY order_key, id), so all replicas pick the same survivor. State both in code comments next to the processors; a refactor that breaks either turns the pair into a loop or a divergence engine.
Consistency sweep — fsck for the projection. The design's main standing risk is invariant density: correctness rests on every write path honoring the dual-write and arrival rules, and a future path that doesn't (importer, plugin, server feature) fails silently — the cell and the children just drift. Deterministic projection makes that cheap to police: a background sweep recomputes each block's child-backed cell keys from its children, byte-compares against the stored cell, and repairs divergence — children-win, with two carve-outs that were never about classification (the sweep repairs local projection state and never consumes a remote bag): keys with live children → project, children win; keys whose children are all soft-deleted → the deletion won, drop the key; cell keys with no child rows at all → pending materialization, not divergence — that is exactly the slice-C backfill-failure fallback where the cell stays authoritative for the block, so the sweep materializes them (a free backfill retry) or leaves them untouched and reports, but never deletes. Divergence counts are telemetry. Reuse the backfill harness (marker-gated per workspace, chunked transactions, idle scheduling) in verify mode; ship it with slice C. A nonzero counter is the alarm that some write path broke the invariant — it converts "discipline held in code" into "violations are detected and self-heal", which is the difference between an invariant and a hope.
Denoted-value rule (unchanged, implemented). Mutations deep in a value child's subtree — a comment under a long-text value — do not touch the parent's cell. The projection watches field rows and their direct value children only. Without this rule the processor becomes a reactive query engine and write amplification explodes.
Upload reality — and the cell's exit path. Every write above uploads (column-narrow, but it uploads): field/value child rows are ordinary synced rows; the derived cell write ships the properties_json column. There is no "derived, don't upload" tag on master and we don't need one yet — under the merge semantics above the uploaded cell is harmless (never a merge input) and useful (bootstrap hint). Budget roughly 2 extra rows per (block, property) pair on first materialization plus one cell-column PATCH per property write; the backfill of a large workspace is a real upload burst (§11, slice C). The end state goes further: once children are the fleet-wide source, the cell can stop syncing and become a locally-derived column (per-workspace, once backfill is complete and sweep-clean — §11; with §6's no-carve-out decision and §9's orphan synthesis there is no permanent key category that keeps a row on the channel) — at which point a property update is a small value-child PATCH plus the parent's metadata PATCH (setProperty bumps updated_at/updated_by on the parent, and those columns sync), still well under re-shipping the whole bag, i.e. cheaper on the wire than today — but that step is deliberately staged and gated on the hydration measurements (§11). One column deliberately keeps its channel through all of this: references_json (PR-review P2). Ref-typed property edits re-project the parent's by-field backlinks (sourceField = the schema name) from the locally-derived cell, and that column keeps uploading through and beyond the cell sync-off — existing parent/sourceField backlink queries ride it unchanged, and the value child's own reference row (sourceField: '') is not a substitute for it. "Keeps its channel" does not mean the reconcile may skip it (PR-review P2, round 25): the column is column-LWW like the cell was, so when two clients concurrently set different ref-typed fields, the value children union under §5 but one writer's whole references_json wins — the losing by-field backlink goes missing until an unrelated local edit recomputes the parent. The arrival repair therefore recomputes the parent's property-sourced references_json entries from the repaired bag along with the cell — through the projection pipeline, never applyRaw (the same rule B2's rename re-key already pins) — and the sweep verifies both columns. The value row's own raw block_references edge is the maintenance-level backstop in the window, not the query surface. Only properties_json ever leaves the channel.

6. The scope decision: every schema gets children

This is the fork in the road. properties-model-evolution.md (May 8) chose A + D: children for user-defined fields, cache-only for system fields. The spike flipped it to A only — every registered schema materializes, including types, system:collapsed, alias, and the property-schema:* config props on schema blocks themselves. An earlier draft of this doc argued for restoring the A+D boundary as a per-schema flag; the §5 merge-semantics analysis reversed that — and an earlier revision of this section still kept the three non-document scopes (UiState, UserPrefs, Automation) as a permanent cell-backed carve-out. That carve-out is now removed too (owner decision, stated repeatedly across design rounds and finally landed here — the split had survived on drafting inertia plus review-loop entrenchment, not argument). Decision: materialize everything — kernel, plugin, and user schemas, all five ChangeScopes (BlockDefault, UiState, UserPrefs, Automation, References — the count is enumerated deliberately; PR-review P2 caught "all four" leaving References-scoped schemas outside the backfill and blocking the column drop) — with system and machine field rows hidden by default behind the existing per-schema hidden override and "Show Hidden Fields" toggle. No schema category stays cell-backed permanently. Two reasons, both structural. First, §5's own argument admits no boundary: whatever stays cell-synced stays clobberable, and a per-user UiState row's bag is still whole-bag LWW across that user's own devices — the same cross-key clobber, just lower-stakes. Second and decisive: partial migration doesn't finish the job — any permanent cell-backed category keeps properties_json alive forever (schema, sync rules, RPC bindings, e2ee envelope handling), so the end-state column drop (§11) would be unreachable. Hidden-by-default is what makes this free at the UI layer: machine props materialize like everything else and simply never render outside the debug/hidden surfaces.

Re-raised and re-affirmed (owner design round, 2026-07-22). The display-tier complexity (§10's hidden-row gesture rules) prompted a fresh look at two alternatives: re-tiering storage by hidden-ness, and a narrower domain split (view-state scopes never materialize). Both rejected. Re-tiering makes hidden a storage bit — marking a property hidden/unhidden becomes a data migration instead of a reversible display toggle — and resurrects the dual write/merge paths §5 killed. The scope split fails more quietly: a schema's changeScope would become a storage decision (scope changes = data migrations), every consumer keeps a bag path alive forever, and "children are the only truth" gains an asterisk. What legitimized both was the belief that full uniformity was unreachable anyway (the field-row-own-bag carve-out); the :: grammar (§7/§9) deletes that carve-out, so uniformity wins outright. The sync analysis backs it: per-row LWW is strictly finer than the bag's column-LWW — the high-frequency UI writes are precisely the ones most exposed to cross-key clobber (this section's collapsed argument) — a toggle rewrites one value row in place (no new rows, no tombstone churn), and same-key concurrent writes converge on the same row under content-LWW, matching bag behavior where it was fine and beating it where it wasn't. View state needs exactly one special case, and it lives in merge policy, not storage: the view-state divergence rule (per-definition merge-policy flag, §9).

What flipped the call: whatever stays cell-synced stays clobberable. Under §5, per-key merge exists only for keys that are rows; a key that lives solely in the cell still rides whole-bag column-LWW, and every dual-write of any property re-uploads the whole cell — carrying stale copies of every cell-backed key with it. A concurrent system:collapsed toggle would be clobbered by an unrelated property write on the same block, and no child exists to repair it from. The one contended kernel prop (collapsed) is precisely the one that most needs row-granular merge. Child-backing it also clears the path on wire cost — though only at the end state: while the cell still syncs, each toggle ships the value-child PATCH plus the re-projected whole bag (more wire than today, not less); once the cell stops uploading (§11's per-workspace cutover — backfill complete, sweep clean), a toggle is the value-child PATCH plus the parent's metadata bump — cheaper than re-shipping the bag. The wire argument is a consequence of the sync-off, not an immediate saving; the merge argument above is what decides the flip.

The costs the spike surfaced are real but turned out to be presentation-layer, and the spike itself has since built the answers: hidden-field filtering in the outline (per-schema hidden + toggle — since schema unification a synced definition field, not a curated code list, though §10's outline-vs-panel question remains open for the slice-C flip), the visible-view/structural-view split in the traversal primitives (as shipped the polarity is everything-by-default with the outline opting into hidePropertyChildren — §9), and panel-layout/orphan-alias special cases. One genuine residual to carry, eyes open — plus one that schema unification has since dissolved:

  • Master's ~8 tree-shape tests need porting — at the flip, not before. Under materialize-everything they assert a pre-feature tree shape that is wrong by design in a flipped workspace. The A/B landing (PR #386) confirmed the dormant half: with every workspace at 'cell', master's suite passes unmodified (mechanical param-count adaptations aside); the deliberate port happens with slice C.
  • Resolved — synthetic fieldIds no longer exist. The spike's kernel/plugin property:<name> pseudo-ids would have leaked non-block ids into references_json, and slice B would have had to special-case them out of reference projection. Schema unification took the parenthetical alternative — real definition blocks for every schema — so a field row's reference_target_id is always a valid block id and the backlink entry is correct: "where is this field used?" backlinks for kernel/plugin fields come free instead of needing a guard.

The non-document scopes migrate too — this section previously carved out UiState/UserPrefs/Automation as permanent cell-backed, and a review round even completed the carve-out to all three (PR-review P1, "§6 silently decided only two of the four"). Both moves are reversed. The carve-out's rationale ("per-user, effectively uncontended, cell-LWW harmless") was true per-key but answered the wrong question: the cost of the carve-out was never those keys' merge quality — it was that their rows keep the cell channel, and therefore the column, permanently. What migrating them actually costs, stated for the record: (a) keystroke-frequency UiState props (editorSelection, focus) pay double wire transitionally — value-child PATCH plus whole-bag reproject while the cell still syncs — the same shape this section already accepted for collapsed; it ends at the workspace sync-off. (b) history churn: high-frequency machine props ride the row_events provenance knob (that optimization is already a slice-C prerequisite, §12) so cursor state doesn't flood history. Their field rows live where the props live today — on the per-user state/prefs/automation rows, hidden by default — not in anyone's document tree. (c) scope integrity for structural edits (PR-review P2): once materialized, those field/value rows are ordinary blocks reachable by tree operations (visible under Show Hidden Fields), but tree edits run under BlockDefault while ChangeScope partitions undo and read-only policy — tx.setProperty enforces scopePoliciesEquivalent today, and an outline edit of a UserPrefs field row would route the equivalent write around it, landing in the wrong undo bucket. Block-level ops on a recognized property-subtree row must resolve their scope from the owning definition — via the shadow-tolerant schema-returning metadata lookup (§9 recognition, condition 3's extension), never bare resolveField, whose winner-enforcing miss on a shadowed loser would land the write in the wrong policy bucket — the routing shim ships in slice B3.

Mechanism: a per-workspace flip column, not a per-schema flag. With §6's no-carve-out decision and one uniform flip, per-definition state has nothing left to vary over — so the lever is a single synced bit: workspaces.properties_migration (an enum column on the workspaces table — 'cell' → 'children' → eventually 'cell-off', the encryption_mode pattern: operator-written, synced to every client, synchronously readable after workspace sync, and server-visible, which is precisely what the §11 sync-rules split needs). The states are ordered, and "flipped" everywhere in this doc means at or past 'children' — IN ('children','cell-off'), never an equality test (PR-review P2: an equality gate would un-recognize every field row the moment a workspace advances to 'cell-off' — exactly when the cell is no longer a synced fallback). And "operator-written" is enforced server-side, not by convention (PR-review P2): the workspaces table is directly updatable by workspace writers — the e2ee migration added workspaces_prevent_e2ee_field_change for exactly this exposure — so the column ships with the same trigger family. As shipped (PR #386's migration, stricter than an earlier revision of this paragraph): client writes rejected (errcode 23514 so a stale client PATCH quarantines instead of retrying forever), transitions forward-only for everyone, service role included, and e2ee workspaces refused outright — §8's gate is encoded, not left to the runbook ("one fat-fingered service-role UPDATE away from a leak"; the check is stable because encryption_mode is itself immutable). Slice B's rollback is still executable, as a migration rather than a column write: the runbook drops the trigger for the duration of the drain-and-delete pass — a deliberate, visible act where a bare backward UPDATE would silently strand recognized field rows. The e2ee refusal is revisable the same way (PR-review P2, round 26: as written it would block e2ee workspaces from ever leaving 'cell', contradicting §8's option-2 path to the end state): when the client-side option-2 flip ships, its rollout includes a successor trigger migration that permits e2ee transitions under option-2's conditions — until that migration exists, the refusal is the correct encoding of "option 2 is not built yet." Advancing is a service-role SQL write per the supabase skill — the runbook is the gate (no attestation RPC; encoding the checklist in an RPC is armor for an operator this fleet doesn't have). The predicate every consumer shares — recognition (§9), dual-write, projection, reconcile — is child-backed = workspace flipped. Nothing else: no scope term (§6 migrates all scopes), no per-schema stamp, no buildability term at flip time (under §5's one-direction rule a degraded writer's stale bag can overwrite nothing — buildability degradation is read-side only, the pending fallback), and no per-key definition-existence term either — "every key resolves a definition" is an invariant, not a predicate branch: established before the flip by §9's orphan synthesis, maintained after it by addSchema (the only normal key-minting path creates the definition), and repaired by the sweep, which synthesizes on encountering a definition-less key from a raw writer or import. Two runbook consequences: synthesis runs before the flip, and unconverted dynamic extensions convert to seeds before it too — a class the types cutover has since narrowed (master 62096388: an enabled extension's dynamic type seeds and inline properties auto-contribute definition blocks, so "convert" is mostly "enable"; what's left is extensions that stay disabled, and bag-writers that never declare schemas) — otherwise synthesis mints a user-origin definition for their keys and the later seed conversion collides with it in the winner machinery, stranding field rows on the loser's fieldId. (A key that momentarily lacks a definition anyway is mechanically inert, not wrong: no field row can exist for it — recognition points at definition ids — so it rides the cell until the sweep promotes it. "Rides the cell" is safe exactly as long as the cell uploads — the §11 upload stop must close this, see its step 3.) This deletes outright: the reserved property-schema:child-backed definition field (schema unification reserved it for this doc — record the divergence there), the slice-C bulk stamp across user definitions and its idempotent re-run rule, the consistency-sweep re-stamp repair for flag-less schemas created by old clients (a pre-flip client's new schema is simply covered by the workspace bit like every other definition), and the per-plugin seed-revision-bump flip choreography. A per-definition override (exempting one schema) is deliberately not built — if a real exception ever appears, adding an override field then is a small additive change; its natural home remains the definition block. What survives unchanged: all machinery branches on the one predicate in exactly one place each; one write model (setProperty), one detection rule, one projection. E2EE composes by the same column: an e2ee workspace simply never leaves 'cell' (§8 option 1). One window this rollout must tolerate (and does — the flip is a coordinated event here): the column propagates via sync, not atomically, so a device may briefly run pre-flip against a flipped workspace; its cell-only writes in that window are stale-writer writes under §5's one-direction rule — reverted on the next projection, logged by the sweep — which is why the flip runbook drains every device before moving the column.

Two hard requirements the flip lever carries, both from adversarial review — both met (one upstream by schema unification, one restated for the workspace column), so the slices here consume them rather than build them:

  • The lever must be durable and fleet-consistent where it can vary — met. The workspace column is one synced server-side value — a code-constant fork between client versions is not expressible, and every device reads the same bit after workspace sync. The definitions-exist invariant rides synced data for every origin since unification; a dynamic extension not yet converted to seed declarations (B′ has since landed on master, deleting the transitional dual-path; enabled extensions now self-seed via dynamic type seeds + inline-property harvest) has no definition block — the runbook converts it before the flip, and a straggler's keys are inert-until-swept (callout above) — the safe default either way. What remains ours: moving the column in either direction is a migration — run a final projection pass, then materialize or delete children under the new regime (rollback: drain then delete, §11 slice B) — never a silent payload change.
  • Name collisions become a hard error — met. The old registry resolved collisions last-wins-by-load-order with a console.warn — per-client, exactly what child-backing can't tolerate (name is the cell key, fieldId the child identity; a per-client winner strands the loser's field rows). The unified registry resolves names deterministically fleet-wide and enforces it centrally: code-owned seeds are unshadowable in v1, the rest resolve by a deterministic order over synced state, and every cell-writing boundary resolves through the one resolution primitive, which returns a structured shadowed/ambiguous status instead of touching the winner's key (unification §7; buildPropertyDefinitionRegistry + SnapshotPropertySchemaResolver). A shadowed definition stays resolvable by fieldId — its child rows keep classifying and its backlinks keep resolving; it is excluded from the name map and the cell projection. Slice B inherits this instead of building collision handling.

This answers the question PR #285 explicitly left open ("should UIState props materialize?" — yes, and likewise UserPrefs and Automation: everything materializes), fully restoring the spike's original instinct, now for a load-bearing reason: the merge story plus the column's eventual removal, not aesthetic uniformity. types keeps its denormalized local index (block_types) and TypeTagger integration regardless — the child rows are its synced source like any other prop, the index stays the query surface.

7. Field-row content: ::((fieldId)) + derived columns

Marker
:: + one whole-block block-ref → ::((fieldId)); matched on trimmed content, no space between marker and ref; the embedded ref inherits the exact-ref grammar's id canonicalization. Either bracket form is legal: ::((fieldId)) is canonical (machinery + autocomplete mint it; rename-stable); ::[[name]] resolves live through the alias index like any link — follow-the-name semantics, dormant until names resolve to definitions (§9 condition 1, §11 AC).
Columns
reference_target_id (stamped for both unmarked ref forms — purely textual for ((id)), the generic alias lookup for [[name]] — and for the marked block-ref form) · is_field_form (both marked bracket forms — each derives through the same branch as its unmarked twin: textual for ::((id)), alias lookup + late-binding for ::[[name]]). Both columns local, derived, never synced, rebuilt from content by the same repair paths.
Predicate
is_field_form = 1 ∧ parent_id IS NOT NULL ∧ shadow-tolerant-resolves ∧ workspace flipped (§9 has each condition's rationale and contract).
Roles
:: children of any block = its field rows · a field row's value set = its is_field_form IS NOT 1 children (a binding selection discipline, §9) · non-:: children of value rows = ordinary content (a :: child of a value row is that value's own field row — the first rule wins at every depth).

Rev 1 specified field-row content as ((schema-id)); the spike switched to [[Schema Name]] with a two-tier resolution (schema-name winner map first, generic alias lookup second), and earlier revisions of this doc endorsed that. The A/B implementation (PR #386) went back to bare id-content; the 2026-07-22 owner decision keeps id-addressing and adds the :: field marker — the canonical shape is ::((fieldId)): the marker carries field-ness in the content itself, the embedded exact ref carries identity, and the local reference_target_id still derives purely textually (block-ref → id — no name lookup of any kind), now alongside a second local is_field_form bit recording that the marked grammar matched (§9 — recognition and SQL discovery read the bits, never the text). The marker is typed and stored identically (owner ruling — no trigger/storage decoupling): typing :: at block start opens field-name autocomplete over the schema registry, and the committed row is ::((fieldId)), with an unresolvable name creating the definition inline — create-on-write lives only here, in the watched input flow (owner ruling: paste never auto-creates — importing fifty ::[[whatever]] lines must not silently mint fifty definitions; a pasted unresolvable row sits visibly unbound with a per-row "create property?" affordance). Beyond that there is no seam at all (owner decision, 07-22 second round, superseding this revision's earlier import-canonicalization step): a pasted or hand-written ::[[name]] is not rewritten — the embedded ref resolves live exactly as unmarked [[name]] links resolve (the alias lookup + late-binding the derive already has), and the row binds when the name resolves to a definition, sits visibly unbound otherwise. The id form stays canonical — machinery and autocomplete always mint ::((fieldId)), which is rename-stable; a name-form row is the pasted/hand-edited minority and carries follow-the-name semantics by construction (§9). The match is strict whole-block on trimmed content — trim(content) = '::' + blockRef, no space between marker and ref, and the embedded ref inherits the exact-ref grammar's id canonicalization (PR #417 review: outer-whitespace and id-case policy must be pinned or classification flickers per path — a pasted trailing newline is the common artifact) — so prose beginning with :: never classifies. :: won over the Tana-style > because > is a markdown blockquote at line start — a standing escape tax on every serializer and paste path — while :: has no line-start meaning in markdown (no escape rule anywhere) and is the property glyph of the Roam/Logseq lineage; single : lost on trigger false-positives (emoticons, prose colons).

The human-readable name is recovered from the definition wherever it's needed: the cell key at projection time, the rendered label. One as-built caveat sharpens this (PR-review round 28): today the recovery runs through the registry (property-schema:name on the definition row) — a definition block's own content is its name only for seeded definitions (seeds write content: seed.name); the user-schema creation path (addSchema → userSchemasService) leaves content empty and stores the label solely in the name field. Registry-reading surfaces (projection, slice D's FieldBlockRenderer) are unaffected, but a generic render of ::((fieldId)) — backlink previews, pre-D surfaces, readable markdown export — would show a blank label. B2 therefore carries a name→content mirror for user definitions: addSchema writes content = name, renames write the mirror through (the same B2 registry-diff funnel that re-keys the cell), and a one-shot backfill aligns existing user definitions.

Why the reversal stuck: the name-resolution tier only existed because properties weren't blocks. Post-unification, definitions are blocks, and blocks already have exactly one name mechanism — the alias index. Types are the standing precedent: a type block claims its label as an alias and [[Person]] resolves through the generic index with no "type-name tier". The precedent is narrower than "universal", though (PR-review round 29, correcting this doc's first rebase note): it holds for user-authored type blocks — seeded backing blocks are deliberately bare, no PAGE_TYPE and no alias (definitionSeeds.ts, pinned by test: the C4 cutover kept them invisible to avoid a behavior change and alias.collision on real type labels — and there is no transition out of bare for them: seed-backed rows are read-only and typeify skips seed rows, so the label-claim path exists only for user-authored type blocks; PR #417 review caught an earlier draft implying seeds could later claim via a label write). So auto-claim cannot assume seeded definitions "already work like types" — its bulk claim must cover seeded and user property definitions explicitly (§11 AC). [[name]] therefore stops being the identity carrier — and under the :: grammar it stops being a recognition input entirely: an unmarked [[name]] or ((id)) row is never machinery, at any depth. What auto-claim still buys is navigation and readable markdown ([[name]] links resolving to definition pages; ::[[name]] as the readable export form of a field row) — an affordance, not a classification prerequisite (§11 AC).

  • What id-content deletes. The two-tier lookup and its fleet-determinism obligations; the derive-at-arrival deferral (schemaTierReady) and the definition-arrival [[name]] re-derive/reclaim (the surviving re-derive is strictly additive generic alias late-binding — [[Foo]] NULL→Foo when a page named Foo is minted; the reclaim of already-bound rows was dead code until auto-claim and is rebuilt with auto-claim, together with cell reprojection); the rename-retitle pass and its content/name lock-step reasoning; and name-validity-as-identity. Renames are content-stable: children and field-row content survive a rename untouched, and only the name-keyed cell re-keys — B2's migration pass, detected by a per-fieldId registry-snapshot diff so every rename source (panel edit, outline edit of the definition block, a synced-in rename) funnels through one seam. references_json re-keys with it: sourceField = schema.name, so the cell re-key runs through the processor pipeline (the references reprojection recomputes sourceField), never applyRaw. The offline-rename race — a bag written under the old key before the rename arrived — is still slice C's fieldId-side reconcile; the one-shot B2 pass can't reach it (verified in the B2 header comment).
  • Markdown round-trip: same-workspace works today, readable/cross-workspace is a named follow-up. Within a workspace, ::((fieldId)) pastes back and resolves — ids are workspace-valid, so the round-trip promise holds where it's most used (property rows enter the copied markdown at all via §9's WYSIWYG copy rule — visible property subtrees serialize once slice D's tier-aware predicate lands; today's shipped copy still prunes all machinery, correctly, while every workspace reads 'cell'). What id-content defers is readable markdown and cross-workspace transfer: the designed home is "copy resolves references" — a generic copy-boundary mechanism (not property-specific) that renders id refs as [[name]] on the way out, paired with auto-claim on the import side so [[name]] resolves back to the definition. Until both exist, a cross-workspace paste of property rows carries uuid refs that degrade to honest unknown-field rows (§9's existing rule). Follow-up, not a blocker (owner call).
  • Typing a field by hand — un-deferred: the :: input flow works without auto-claim. An earlier revision sequenced hand-authoring behind auto-claim (recognition was form-agnostic, so a [[status]] row would "light up" once definitions claimed aliases). The :: grammar inverts this: name resolution happens at input time — ::status autocompletes against the client-side schema registry (no alias index involved) and commits as ::((fieldId)); an unresolvable name offers to create the definition inline. Nothing waits for auto-claim, and — the other half — a plain [[status]] link can never light up as machinery later, so claiming a name is retroactively safe (the AC trap list collapses, §11 AC). The sync same-tx prospective probe checks the marked form only; its deferred async [[alias]] branch is deleted rather than pending.
  • Names are hygiene now, not identity. addSchema and the rename flow reject ]]-lossy names (isRoundTrippableReferenceLabel, shipped) so names stay clean for the future [[name]] affordance and for export — but an invalid name can no longer corrupt field-row identity, so the same-tx definition-name validator an earlier revision required (PR-review P2, rounds 22) is deleted; hygiene at the two flows plus a preflight report over existing names suffices. Name collisions ride the alias machinery's semantics at auto-claim; the unification winner gate survives as the name→cell-key authority (projection and addSchema's structured shadowed rejection), not as an identity-resolution tier. The exact-reference grammar stays scoped to whole-block content, and only the ::-marked form is ever a field — prose containing ((…)), plain whole-block refs, and [[name]] links never classify (referenceBlock.ts + the §9 form bit).

8. E2EE workspaces — new constraint

E2EE workspaces (May 29) encrypt content, properties_json, and references_json into enc:v1 envelopes; a server trigger rejects plaintext in those columns. Reference edges are therefore part of the encrypted surface today. A synced plaintext reference_target_id would have leaked, per row: "this row is a field instance", "of this schema" (concretely readable — seeded definition ids are deterministic uuidv5 values anyone can precompute), and for exact block-ref rows "this row points at that row" — precisely the edge metadata the current design encrypts. This analysis is what drove the column local for every workspace (owner decision, §11 slice A): derivation is deterministic on every device — under §7's id-content shape, purely textual — so nothing needed to cross the wire in the first place — the derive-at-arrival and catch-up derive pass that an e2ee-only fix would have required turned out to be the right universal design, and the e2ee fork for this column no longer exists. (Encrypting the column instead was considered and rejected: it would maintain a second synced copy of derivable state, and the server never queries it — ciphertext or plaintext.)

What remains e2ee-specific is only when the workspace flips:

  1. Gate: e2ee workspaces flip later — sequencing, not an endpoint. An e2ee workspace's properties_migration column stays at 'cell' until option 2 below is built (§6 — the gate and the flip ride the same lever), so the machinery behaves as if every schema were cell-backed. Zero new code beyond the gate; e2ee users keep today's behavior for now. Recommended for the first ship — but option 2 is required work, not optional: §11's column drop needs every workspace at 'cell-off', e2ee included, so a permanently-gated e2ee workspace would keep the column alive exactly the way §6's rejected carve-outs would have.
  2. Flip with client-side tooling. Field/value children are ordinary blocks — their content encrypts like any content, so the child machinery needs nothing new. What must run client-side because the server cannot read e2ee keys: the orphan-definition synthesis and the one-time backfill (§9, §11), on a synced device, per the same runbook. The cell keeps its envelope until the workspace reaches the ordinary sync-off. (2) achieves the same privacy with less ceremony.

With the column local, the gate needs no e2ee-specific machinery at all — it is simply the workspace column not having been advanced, same as any un-flipped workspace. The derive processor runs everywhere, including e2ee (a local column over decrypted content leaks nothing). Two edges survive, neither e2ee-specific in form:

  • Pastes from a flipped workspace into an un-flipped one de-structure, not just strip (e2ee-at-'cell' is the standing case, but the rule is general — PR-review P2, restated for the local column): copied field/value rows would linger as ordinary visible content in a workspace whose cell is the property store, so the move path projects the subtree's field/value children into the destination cell and soft-deletes the de-structured rows. Satisfied by construction on current master (implementation finding, PR #386): cross-workspace copy/paste is markdown-mediated — parse → fresh creates — so no structural carrier exists to de-structure; this rule gets a code path only when a structural cross-workspace move ships (it lands with that feature, alongside §7's "copy resolves references" follow-up).
  • Encryption-mode transitions are the e2ee feature's problem, with one property-shaped constraint. Today encryption_mode is immutable by trigger (workspaces_prevent_e2ee_field_change). If a flip flow ever lands, a flipped workspace transitions cleanly — children are ordinary blocks, and the encryption bulk-rewrite covers them like all content — but it must be history-safe (PR-review P2): blocks_history retains full plaintext INSERT rows and UPDATE diffs, so the transition is a SECURITY DEFINER migration that also purges/encrypts the affected history rows, or the mode stays immutable. (Owner: the workspace-encryption flip flow, not the §11 slices.)

A side benefit of the §11 cell end-state — for e2ee workspaces, sequenced behind option 2: under the option-1 gate the cell is an e2ee workspace's only property store, so the sync-off cannot apply to it. Once derive-locally child-backing (option 2) replaces the gate, stopping the cell upload removes the envelope column from the wire (the §11 workspace-level cutover applies in e2ee too, gated on the same backfill-complete condition); property data then travels only as ordinary encrypted child-row content.

No server surface remains for the column (the earlier synced design needed both write RPCs extended null-aware, a Supabase migration, and the ciphertext-trigger extension — all deleted with the local-column decision; clearability is now derive-at-arrival's job: content changed → re-derive → NULL locally, on every device independently). One housekeeping item survives for slice E: the spike's recent-content index migration (20260530143000) sorts before already-applied master migrations — re-timestamp it and wrap in begin; … commit; per the supabase skill if it's kept.

9. Detection rules and edge cases — as implemented, with divergences

Field-row recognition.
A row is a field row iff all four hold. (1) Its content is the marked field form — :: + a whole-block block-ref (::((fieldId)); §7), recorded at derive time in the local is_field_form bit — either bracket form (owner decision, 07-22 second round, superseding this revision's earlier block-ref-only rule and its seam apparatus): the embedded ref resolves exactly as refs resolve everywhere, so a marked alias ::[[name]] binds through the alias index and late-binding like any link. The retroactivity concern that motivated the old exclusion was mis-transferred from unmarked prose: a plain [[status]] comment becoming machinery at claim time is reinterpretation of link-intent — still structurally impossible, unmarked rows never classify — but a ::-marked row lighting up when its name resolves is the system catching up to declared field intent; nobody types ::[[status]] as prose. Pre-AC no name resolves to a definition, so the alias half is naturally dormant (id-only-at-first falls out as sequencing, no flag). Two semantics follow and are deliberate: id rows are identity-pinned (the round-29 shadow tolerance applies to them — losers stay resolvable); name rows follow the name — they re-bind to whatever wins it (a rename or winner flip re-binds or unbinds them; values re-decode under the new codec, failures landing in the invalid-values story). (2) Its parent is non-null (root half: a workspace-root row has no owner to be a field of — its marker is just content; the SQL twins carry the same parent_id IS NOT NULL clause). (3) reference_target_id resolves via the shadow-tolerant classification checker, not bare resolveField (PR-review round 29: resolveField alone is winner-enforcing — a shadowed fieldId returns identity-unavailable/shadowed — so classifying through it would unclassify every shadowed loser's rows, contradicting §6; the as-built checker, isFieldDefinitionCheckerFor, wraps it and classifies on resolved OR shadowed, every other reason unclassifies — that disjunction is the contract, pin it so a resolver refactor can't silently narrow it; the checker lives on PR #386 (txEngine.ts), this doc's authoritative implementation branch — it is not on master until A/B merges). One extension pinned with it (PR #417 review): consumers that need the definition's metadata, not a boolean — B3 scope routing needs the owning schema's changeScope, merge needs its merge policy, value decode needs the codec — use a shadow-tolerant schema-returning lookup: a fieldId-keyed read of the definition row and its parsed metadata, winner-independent (shadowing is name authority, not row readability). Routing a shadowed loser's rows through bare resolveField would classify them as machinery while losing the very scope that decides their write-policy bucket. (4) The workspace is flipped (properties_migration at or past 'children', §6; PR-review P2: recognition off the raw bits alone activates the moment slice A starts stamping — a marked row typed or pasted while the workspace is still cell-only would be reinterpreted as machinery — and hidden outright for hidden-tier definitions — breaking the "machinery lands dormant" claim; the workspace column is synced and server-visible, so the gate is fleet-consistent). No positional condition exists (owner decision, 2026-07-22): recognition is content-intrinsic, identical at every depth, and move-proof at any non-root position — a row's classification cannot change because merge, indent, or a hoist relocated it; the root half is the one positional residue, by design (moving a marked row to or from the workspace root is an ordinary embraced role change, not an exception to guard). This supersedes the whole positional lineage (PR-review P2 ×2's direct-parent guard, the alternating role grammar that followed it, and the blanket interior-ancestry walk the A/B branch actually shipped): isInsidePropertySubtreeWalk, its per-tx memos and SQL recursive twins, and the materialize/project/re-key interior gates are all deleted as recognition inputs. What survives is the narrower ownership question — "which field row owns this descendant" — which is containment, not identity, and is inherently positional. Both walks start at the row itself (PR #417 review: a field row is its own owner — a structural op on the field row takes its scope from its own definition, and a hidden field row suppresses itself; ancestor-only walks would fall back to BlockDefault for a top-level UserPrefs field row and fail to suppress a hidden field row hit by search): B3's scope routing (a structural edit deep in a property subtree routes under the owning schema's scope) then walks to the nearest :: row — self or ancestor, the directly-owning field; hidden-subtree pruning's upward form (search/recents filtering classifying an arbitrary hit) checks self and every field ancestor — any hidden one suppresses, since a visible field materialized inside a hidden field's subtree must not leak its rows into default surfaces (PR #417 review; top-down render pruning gets this free by never descending into a hidden field row). Backlink presentation needs no walk — it is a single parent hop, the References entry's rule. (PR #417 review: deleting the walk outright would silently re-route value-descendant edits under the caller's BlockDefault scope.) The surviving walk is much simpler than the deleted one: each hop is independently classified by the flat predicate, so it carries none of the memo-invalidation coupling to the tx engine that made the recognition walk expensive to keep honest. Roles fall out of content alone: :: children of any block are that block's field rows — so field rows and value rows materialize their own bags like every other block, and no cell-only depth carve-out exists anywhere (the §6/§11 "every key leaves the cell" promise finally holds without exception). A field row's value set is exactly its children with is_field_form IS NOT 1 (the bit is NULL for every underived/unmarked row — ordinary value children are never stamped 0, so an equality test against 0 silently excludes every real value; PR #417 review caught an earlier draft's = 0 phrasing, and the JS predicate treats null/undefined as false the same way) — including plain ((id)) refs to definition blocks, which are unambiguous now that field-ness is marked (block-type:properties values — a refList whose targets are all property-schema blocks, materialized on every type block since the types cutover — render as values by construction; the parent guard that existed to protect them, and the escape-rule alternative briefly considered for them, are both unnecessary). Non-:: children of value rows are ordinary content — a :: child of a value row is the value's own field row, the any-depth rule again — and a content block anywhere in a property subtree carries field rows of its own with full standing. The value-set definition is a binding selection discipline, not display guidance (PR #417 areview — the highest-leverage item on the branch delta): every site that enumerates a field row's values or selects "the field row for this fieldId" filters by the bit — the projection's first-parseable-value scan, setProperty's primary-value pick and SELECT_PROPERTY_FIELD_CHILD_SQL (which must also require is_field_form = 1, or an unmarked ((fieldId)) link row can be selected as the field row and its first child overwritten with the encoded value), the materializer's primary/duplicate logic, both dedup helpers, merge's destination field map, and rename's computePlan. The failure without it is concrete and non-deterministic: a nested :: row materialized under a field row shares keyAtStart(null) with the primary value and sorts by id tiebreak, so it randomly projects as the property's value into the synced cell — and the next setProperty overwrites the machinery row's content with the encoded value. Duplicate-field-row folding routes a duplicate's own marked children through field-row folding recursively, never through value folding. Discovery never parses content at read time: the same textual derive pass stamps reference_target_id (for marked and unmarked ref forms alike — backlinks, the B2 used-by count, and merge-retarget stay uniform over machinery and plain refs) plus is_field_form, both local, never synced, rebuilt by the same arrival-repair that already recomputes the target column — content remains the only truth crossing the wire, so the bits are rebuildable forever. "All field rows under X" is an indexed scan (partial index WHERE is_field_form = 1); the classification predicate is flat — bit + parent_id IS NOT NULL + checker + flip. Classification needs the definition, not the owning plugin (an uninstalled plugin's definitions sync like any block), and works for shadowed definitions too (§6 — losers stay fieldId-resolvable). An unresolvable marked row degrades to a visible, honest "unknown field" row (a pasted field from another workspace, a foreign fieldId); an unmarked ref targeting a definition is a plain reference block, full stop.
Cross-workspace copies — satisfied by construction today; the rules wake with structural moves.
Seeded fieldIds are workspace-scoped (unification §4.2), so a field row that structurally crossed workspaces would carry a foreign reference_target_id and must re-derive at the destination (PR-review P2). Implementation finding (PR #386): on current master this entire family has no code path — cross-workspace copy/paste is markdown-mediated (parse → fresh creates), so nothing carries a column to clear or strip; a pasted ::((fieldId)) row simply fails to resolve at the destination and degrades to an honest unknown-field row (§7's markdown follow-up — "copy resolves references" + auto-claim — is what will make these pastes meaningful). The clear-and-re-derive rule, the seed:*-strip seam, and §8's de-structure rule all activate together when a structural cross-workspace move ships. What survives now is deliberately additive-only (PR #386, replacing an earlier definition-arrival/reclaim design that was dead code until auto-claim): the re-derive watches for generic alias late-binding — a [[Foo]] row whose column is NULL binds when a page named Foo is minted or arrives — and the reclaim of already-bound rows is rebuilt with auto-claim, together with cell reprojection. An earlier revision's name-collision-paste edge (same-named destination schema, incompatible codec) is likewise an auto-claim-era concern; its general mitigation — the sweep reports a field row none of whose children decode — ships regardless, because codec changes produce the same state.
Orphan keys — synthesized definitions, not a permanent bucket.
A cell key with no definition block of any origin (historic residue, keys from plugins disabled before the unification's cutover, dormant dynamic extensions) gets a definition synthesized by the slice-C flip tooling (owner decision — a permanent definition-less bucket would keep rows on the cell channel forever, defeating the §11 column drop): scan the workspace's distinct definition-less keys, create a user-origin definition per key — name = key, codec inferred from all observed values of that key (all-boolean → boolean, all-number → number, ISO-date strings → date, else string; anything mixed or structured falls back to the raw-json preset — conservative on ambiguity), hidden by default until triaged. Two rules the synthesis must apply, the first now much weaker under §7's id-content shape (PR-review P2 ×2, revised): names are hygiene-checked, not rewritten — field rows are ::((fieldId)), so a historical key that fails the wikilink round-trip (trim-changing, containing ]]) can still be a definition name and cell key verbatim; it merely can't gain the future [[name]] affordance, so synthesis keeps name = key and flags such keys in the flip report for optional cleanup (the earlier deterministic-rename requirement is deleted; only a genuinely unusable key — the empty string — cannot be synthesized at all, and its report is a hard flip blocker, not an FYI (PR-review round 29): the definition parser and addSchema both reject empty names, so an empty-string cell key can never be definition-backed — left in place it makes the "no definition-less keys" gate unsatisfiable forever, and at cell-off the column drop would orphan the value's only carrier; the runbook resolves it before the flip proceeds — delete the key or remap its value under a named key); and ids are deterministic — uuidv5(workspaceId, key), the seeded-definition pattern — so a re-run converges on the same blocks, and two devices independently running an e2ee option-2 flip mint the same definition instead of same-named duplicates whose loser strands its backfilled field rows. For e2ee workspaces the derivation must not be dictionary-testable (PR-review P2, round 26: ids sync in plaintext, so uuidv5(workspaceId, key) would let the server confirm a guessed private property name — health, client:ssn — against §8's edge-metadata sensitivity line): e2ee synthesis derives the namespace from workspace secret key material (an HMAC the key-holders can all compute) — still deterministic across the workspace's devices, opaque to anyone without the secret. Plaintext workspaces keep plain uuidv5; their keys are server-readable anyway. From that point they are ordinary schemas riding the normal machinery. Until the flip runs (and for keys a still-unconverted dynamic extension owns), definition-less keys retain today's cell semantics — a transitional state, not an end state. Synthesized definitions carry a provenance marker, because a synthesized key's real owner can arrive later (PR-review P2): re-enabling or converting a plugin seeds an unshadowable code-owned definition under the same name, the synthesized definition drops to shadowed loser — excluded from the name map and cell projection — and the already-backfilled field rows still point at the synthetic fieldId, so the migrated values would vanish from block.get/queries under the real schema. Handled at runbook + detection level, not as shipped machinery (simplification): the flip runbook's primary rule is enable and convert everything you intend to use before flipping, so the collision should not occur; if it does, the sweep detects a shadowed marker-carrying synthesized definition and reports it, and the operator runs the retarget manually — re-point the field rows to the seed's fieldId and retire the synthesized block, reusing B2's rename re-key machinery plus the codec-change re-encode (PR-review P2: the seed's preset can differ from the synthesized definition's inferred codec, and a retarget without re-encoding leaves value children the real schema decodes as invalid — the migrated values would read as unset). Build a shipped pass only if this ever actually fires. The unification doc counts disabled-plugin residue as this problem's easy sub-case (its §4.3); e2ee workspaces synthesize client-side at their own (option-2) flip, since the server can never enumerate their keys.
Value decoding.
Spike: first parseable value child wins (order-key order over the is_field_form IS NOT 1 children — §9's selection discipline); unparseable children are skipped. If no child parses, the projection removes the key from the cell. (This rule — and default-vs-unset below — presupposes a live field row; a cell key with no child rows at all is pending materialization and is never removed, per the §5 global rule.) Rev 1 said "log + omit, keep data visible" — the spike keeps the rows (data is never destroyed) but the cell-side key vanishes, which downstream readers see as "unset". Acceptable; document that "field with only-invalid values" reads as unset while remaining visible/fixable in the tree — with one addition (PR-review P2, via the cross-workspace paste case): "visible/fixable" is weak comfort for hidden-tier rows, so the §5 sweep reports the class — a live field row none of whose children decode — alongside its divergence log, surfacing what would otherwise read as unset forever. Ref-typed values are reference-shaped (PR #386, divergence item 6): a ref-typed value child holds ((id)) — the same editable block-reference affordance as anywhere else, unmarked (the field row's own content embeds the same exact-ref grammar behind its :: marker, so a ref value targeting a definition block is unambiguous, §9 recognition) — while the cell keeps a bare id (codecs.ref unchanged). The read path returns the row's derived reference_target_id, which stays NULL unless the content is a resolvable exact ref — so prose typed into a ref property ("people type text into a ref property, like a log") is unparseable, the projection skips it, and the key reads unset while the row keeps its text, instead of a log silently coercing into a fake block id as bare-id storage did. It also puts property refs on the reference-maintenance rails: merge-retarget rewrites ((A))→((B)) inside a value and the cell follows. refList stays JSON for now — the direction is "a refList is N ((id)) sibling values," deferred with the multi-value design pass (it needs list-aggregate projection semantics).
Deleting a referenced block keeps the value dangling — never inline it.
Decided in #404 (item 4), widened by PR-review round 28: inlineDeletedBlockReferences must skip recognized property machinery — ref-typed value children and field rows — keyed on the marked form (the bit) and value position, not on live classification (owner round, 07-22: a bound ::[[name]] row whose definition dies in the same delete tx has already lost resolution when the processor asks — form-keying makes the exemption order-independent, or the tx inlines the name into the row and destroys the marker shape). The field-row half is what makes the B2 definition-deletion rule (below, §11) coherent at all: field rows parse into block_references (the "used by" edge above), so deleting a definition would inline the definition's text into every ::((fieldId)) field row in the same delete tx — destroying the marked exact-ref shape before tombstone-resolvable recognition ever gets a say. Inlining a deleted target rewrites the row's ((deletedId)) into prose — which clears its derived column, silently drops the key at the next projection, and is irreversible (the id is gone). Left dangling, the value keeps identity: soft-delete means the target is a tombstone, so restoring it snaps the property back — exactly the cell era's semantics, where a dangling ref left the property retained. The principle is the same one inlining serves for prose, applied to a different content type: prose wants readable text preserved; a typed value wants identity preserved, and inlining destroys exactly that. (Derivation already cooperates: deriveReferenceTargetId resolves purely textually with no existence check, so the stamp survives the target's deletion and the property survives re-materialization.)
Duplicates and divergence — union-with-dedupe (as shipped).
PR #386 (divergence item 7) replaced both rev 1's "keep extras, pick lowest orderKey" and the spike's delete-all-duplicates: a field row holds a set of value children — its is_field_form IS NOT 1 children (§9's selection discipline) — deduped by content. Content-equal duplicates fold (soft-delete; the relocate rule survives for their user-authored sub-children — a comment thread under a folding duplicate moves to the survivor, and both dedup call sites must fold identically); divergent values are never deleted — they persist as peer siblings under the field row (view-state-flagged keys excepted: the merge entry's silent-resolve rule, below). Projection reads the first parseable value, so the visibly-first value owns the cell while the others stay in the tree as a surfaced conflict; concurrent dual-writes from two devices converge to sibling values rather than one silently winning. This dissolves the earlier tightening list this entry carried (relocating a losing value under the survivor to keep it visible): divergent values were exactly the case that made deletion delicate, and they are no longer deleted at all. It also composes with movement being ordinary (§10): since projection reads the first value and reorder is a normal gesture, dragging a value to the top is the conflict-resolution affordance — no new mechanism; slice D may badge a field row carrying more than one divergent value ("+1 conflicting"). Duplicate field rows for one fieldId collapse to a single row whose value set is the union, deduped by the same content rule. Divergent siblings are thus a real, expected shape — a surfaced conflict — but still not the user-facing multi-value affordance; that (and refList-as-children) gets its own design pass.
Default-value vs. unset.
Unchanged from rev 1: zero value children under a live field row ⇒ key omitted from the cell ⇒ block.get falls back to defaultValue. No reader change.
Codec / preset changes are migrations, not edits.
§9's "only-invalid values reads as unset" is fine for one malformed child; it is a mass event when a user changes a field's preset or config (text → number): every existing value child becomes unparseable under the new codec, each consuming block's next projection removes the key, and readers see defaultValue everywhere — silent fleet-wide unset presented as default. Worse, until reprojection runs, the cell still holds the old encoding and the new codec decodes it on the read path. A preset/config change must run an explicit re-encode/validate pass over existing values (with a user-visible "N values can't convert" outcome), not lean on the projection's remove-on-invalid behavior. Same machinery as the rename reproject; build them together in slice B.
Field-row placement (decided: no special logic).
Owner decision (2026-07-19): there is no placement policy for property rows at all. A user who types a property into a block has chosen where it lives — nothing repositions, redirects, or normalizes it, ever. Programmatic creation paths (setProperty, materialization, backfill) still need some deterministic key and use keyAtStart as shipped — an implementation default, not a UX invariant (machinery-created rows therefore cluster at the head, in reverse creation order among themselves; accepted cosmetics, freely reordered since movement is ordinary — §10). This supersedes both rev 1's "carriers-then-content with ghosts between" and the interim "insert after the last property child" intent from the divergence-note round. orderKey remains authoritative and user-owned: a field row reordered below content children stays where the user put it, and the display renders it there (§10 — position-honest, no grouping).
Reparenting a field row = transferring the property (decided: embrace).
The project processor reprojects both sides of a move, so moving a status field row under a different block gives that block the status property — and removes it from the old parent. Owner decision (2026-07-19): this is the intended everything-is-a-block semantic, not a hazard — properties "behave like normal children," including transfer across hierarchies. #404 reached the same conclusion independently and withdrew its movement item: the transfer is what the spike pins as consistency case 7 ("reprojects both parents when a property child is moved"), and the destructive variant it feared isn't reachable. No guard, no reference_target_id clearing on cross-parent moves; what makes the semantic safe is the anchor rule below — every transfer is a gesture the user performed on a row they could see, and undo covers it. A drop-time UI signal ("moving property status") is optional slice-D polish, not a correctness requirement. The same embrace covers role changes: indenting a content block under a visible field row makes it that property's value — visible, undoable, and the natural "add another value" gesture.
Movement anchors on the list the caller sees (decided).
The one rule that makes hidden rows safe (owner decision, 2026-07-19): every structural gesture resolves its anchors — the moveVertical neighbour, the indent target, the drop target, the insertion position — against the sibling list the caller sees; the physical list is used only for order-key arithmetic around the chosen anchor. For outline gestures that is the visible view: non-hidden property rows and any hidden rows currently revealed are in the list (so indent/outdent/reorder work on them like any child — resolving the asymmetry #404 noted, where outdent reads the raw sibling list while indent/moveVertical read the visible one); hidden-tier rows are not. For programmatic callers (the agent bridge, plugins) the anchor list is the structural view — a deliberate move of machinery just works. Consequences, each pinned by a test in slice D: move-up skips a hidden physical neighbour — physical [hidden, C1, C2], move C2 up → C2 lands visibly above C1, and whether it physically lands before or after the hidden row is unobservable (hidden rows' position carries no semantics); a visible gesture can never appear to do nothing. Indent targets the previous visible sibling, never a hidden row — a block can't be captured as a secret property's value. Value-row edges no-op cleanly — the spike's -1-fallback path that silently deleted a property on outdent/moveVertical of a value row must not be inherited; the current branch avoids it by accident, the rule makes it deliberate. Split has no property special-casing (owner decision, 2026-07-19): whatever the existing split logic does with children applies to property rows unchanged — they are children. No property-specific movement guard exists anywhere: with visible anchoring, every role change is something the user watched themselves do.
References: field rows carry a "used by" edge (as shipped).
Parse-level property-machinery suppression has been removed (PR #386 at 12609823): field and value rows parse into block_references like any other row — and this is a pinned requirement on the :: grammar: the reference parser treats the marked form's embedded exact ref identically to a bare one, or definitions lose their used-by edges. A property definition's backlinks therefore answer "which blocks use this property?" — a deliberate feature, and also what lets definition merge/rename retargeting reach field rows at all (the maintenance passes walk the raw reference table). Backlink display filtering is role-based, not depth-based (re-specified for the :: grammar — the earlier "value rows and deeper" rule predated nested machinery, and PR #417 review caught that it would drop or misattribute a machinery row's own property uses): what's filtered is value rows' duplicate edges — they duplicate the owner's reprojected edge — while every field row's edge, at any depth, is unique and is kept; the inline backlink count applies the same role rule. Kept, but presented as the parent (PR-review P2, round 27, generalized): the raw source is the ::((fieldId)) field row itself, and a backlink surface or watcher that returned it verbatim would be listing machinery — display maps a field-row source to its parent block, which is the thing that "uses" the property. That mapping is uniform under nesting: a nested field row's edge presents as its machinery parent (accurate — the property is used by that field/value row; the surface renders the parent with its property-row presentation rather than walking further up). The raw table keeps the true source row for the maintenance passes. The cell-side projection (projectPropertyReferences, sourceField = schema.name) still runs off the bag unchanged, so backlinks-by-field keep working through the transition.
Merge, orphan cleanup, panel layout, tree primitives.
Merge is union-with-dedupe as shipped (PR #386, divergence item 7, superseding the spike's subtree-delete-and-rematerialize): mergeBlocksInTx is property-aware — for a property both blocks carry, a content-equal value folds; a divergent value is kept as a peer sibling under the target's field row (projection reads the first value, so the target's winner keeps the cell while the conflict stays visible and resolvable — see the dedup entry above); a property only the source has is adopted, its field row moved intact. Nothing is silently dropped and nothing litters the outline, so the earlier preservation rule this entry carried (relocate user-authored descendants before the subtree-delete) is superseded — nothing but content-equal duplicates is deleted, and those fold under the dedup entry's relocate rule. The merge relocate-stamp-clear is gone: the old path relocated a losing value into ordinary content and had to null its definition-shaped reference_target_id to stop it classifying as a field row — unsound, because the column is content-derived and device-local (the clear evaporates on the next edit and never synced). Under intrinsic recognition (§9, 07-22) the reasoning is stronger than the old positional exemption: a relocated value is simply not ::-marked, so it can never classify as a field row anywhere — no clear, no position to reason about. The same move-proof-ness kills the positional-reclassification family outright (a hoisted :: row stays a field row through any merge or move and folds via the normal property-aware routing — the A/B branch's merge-hoist finding dissolves structurally, and any interim refuse-guard on the branch becomes unnecessary once the grammar lands). One view-state merge exception (owner decision, 07-22, with the §6 re-affirmation; re-specified after PR #417 areview caught the first keying missing its own motivating case): divergence preservation is for document content — for view-state keys merge resolves silently instead of keeping conflict siblings; collapse state must never demand adjudication. The predicate is not bare changeScope = UiState: system:collapsed is declared BlockDefault today (properties.ts — it is shared document-adjacent state, deliberately), and the genuinely UiState-scoped keys live on per-user state blocks no merge gesture reaches — keyed on scope alone the rule would cover approximately nothing. View-state-ness is a per-definition merge-policy flag (seeded true for system:collapsed and friends; UiState-scoped schemas get it by default), read through the shadow-tolerant metadata lookup (§9 recognition, condition 3's extension). Semantics, fully: both sides hold a value → target's wins, the source's is dropped (no sibling); only the source holds one → it is adopted (absence is not a conflict — consistent with the source-only adoption rule below); a dropped loser's user-authored sub-children relocate exactly as the content-equal fold relocates them, never silently deleted. And the exception binds every dedup site — silent resolution, no conflict siblings, for view-state keys everywhere — while the winner rule is per-site (PR #417 review: "same policy" overclaimed): target-wins at the merge gesture (an explicit user act), newest-updated_at at the §5 sync-arrival reconcile, where no target exists and a clock-blind rule would let an older collapse toggle win (two devices' concurrent toggles are the motivating case, and they meet in the reconcile, not in mergeBlocksInTx; §5 states the clocked rule). It lives in merge/dedup policy — not in storage, recognition, or traversal. Orphan-alias cleanup deletes generated field rows under alias seats; panel-layout projection ignores field rows. Traversal polarity is inverted from earlier revisions (PR #386, deliberate divergence): every traversal primitive (tx.childrenOf, core.children/childIds, subtree) returns everything by default — the structural view — and the display-visible view is opt-in via hidePropertyChildren: true (the outline hooks pass it). The earlier hide-by-default polarity made the dangerous mistake silent: a structural caller that forgot the include flag stranded machinery rows under a delete (softDeleteSubtree discovers descendants via tx.childrenOf — verified in mutators.ts). Everything-by-default makes the forgotten flag merely cosmetic — a display surface briefly showing hidden machinery rows — and is the more coherent primitive. Delete/purge/backup are therefore correct with no flag. Copy is WYSIWYG (owner decision, 2026-07-19): default copy serializes exactly the visible view — non-hidden property rows travel with their subtrees, hidden-tier subtrees prune whole — and an explicit "copy with hidden properties" command is the full-tree variant. As shipped today serializeBlock prunes all machinery (hidePropertyChildren: true) — correct while every workspace reads 'cell'; the tier-aware predicate lands with slice D, and switching to it also closes #404's copy gap by construction (user content nested under a visible property's value stops being silently dropped, because visible subtrees are no longer pruned). Content nested under a hidden property's value still doesn't travel on default copy — an accepted WYSIWYG consequence (it was authored under a revealed row; the full-copy command carries it). Under the §6 scoping these rules apply everywhere — kernel props materialize too, so hidden-tier field rows exist on most blocks and the copy/display predicates get more load-bearing, not less. Reparenting a field row as "content" is not a case to guard: by intrinsic recognition it simply is a field row of its new parent — the marker travels with the content; transfer, embraced above.
Type-add field slots vs ghosts (decided: render-only ghosts).
The spike materializes empty field rows ("slots") when a type is added (TypeTagger → materializePropertyFieldSlotsForExistingRow; since 03236460 a slot whose key already has a cell value materializes the value too — an empty slot had let the reverse projection strip it). Rev 1 argued for render-only ghost rows that materialize on first edit — no storage until data exists, nothing to clean up on type removal. The spike still has no slot cleanup on type removal (_removeTypeInTx only rewrites the types array), so removing a type strands empty [[field]] rows. Decision: render-only ghosts; drop the TypeTagger materialization. If empty slots prove necessary (e.g., for cross-device consistency of field order), add explicit cleanup on type removal.

10. UI: the display model — two tiers, in place; the panel is not sunset

The display model is settled (owner decisions, divergence-note round 3 + 2026-07-19), and it is two tiers, rendered in place. Properties whose definition is not hidden render inline in the outline at their true position among the children, as a two-column name/value row (not a nested indent) — the spike's FieldBlockRenderer direction: the label links to the definition block, the value cell renders the property type's real editor, and each row is individually collapsible (PropertyValueBlockRenderer covers the expanded value/subtree view). For every gesture they are ordinary children — reorder, indent/outdent, transfer, copy — per §9's movement policy. Properties whose definition is hidden: true (system:collapsed and friends — the ones that would appear on nearly every block) stay out of the default view and are revealed per block, on demand (the spike's "Show/Hide Hidden Fields" context-menu entry); while revealed they join the visible list with full gesture standing. The display is position-honest: a grouped "property section" pinned above the content — considered in this design round — was rejected (2026-07-19), because grouping makes property rows lie about their position, recreating the display-vs-structure divergence for every row that the two-tier model confines to hidden ones. The invariant: what you see is the true order, minus hidden rows. One caution the audit quantified so the model isn't oversold: of ~30 machinery-exclusion call sites, only ~9 are render-motivated and collapse under this display; ~20 are structural tree math that is indifferent to whether a row is drawn (the spike's final commit flipped its childrenOf default for exactly this reason). Visibility does not remove the machinery — property children stay structurally special even once visible.

Hidden-ness is the synced property-schema:hidden definition field carried on ResolvedPropertySchema (schema unification replaced the spike's curated code list); extraction reads that field. What the one bit does not yet answer: it gates the property panel, while the outline needs its own answer for panel-visible props — types above all is (rightly) not hidden in the panel, yet a materialized types row must not clutter the outline. Deciding whether outline filtering reuses hidden, adds a per-surface bit to the definition, or defaults by origin is a slice-C flip deliverable (§11, end-state step 2), with the per-block reveal as the escape hatch. The property panel remains fully functional alongside — both surfaces read the same cell and write through the same setProperty, so they coexist safely during the transition. Panel sunset is deliberately last (§11) and follows usage, not a schedule. Rev 1's inventory of what survives (value presets, schema registry, resolvePropertyDisplay, PropertySchemaBlockRenderer) matches what the spike actually reuses.

  • Extract master-first slices (below), copying spike code verbatim wherever possible — extraction that re-derives the mechanism would add semantic-drift risk for no gain. Each slice lands with its review fixes built in and is reviewed against master's suite.

Recommended: slices, second choice: dormant column. And the fourth lifecycle event, definition merge (PR #417 areview — reachable today via the merge picker, and the designed remedy for alias collisions once AC claims names): the retarget rewrite itself works by construction (::((A))→::((B)) — the ref regex is a substring match, the marker survives, the column recomputes inline), but the retarget is a plugin processor registered after project in the single-pass tx, so the rewritten field rows are never re-projected in the merge tx: every consuming owner's cell keeps name(A) — a key whose definition is tombstoned, which nothing may delete (§5's never-delete rule) and nothing can re-back, permanently blocking the cell-off gate. The rule: a definition merge is rename and potentially codec change combined (PR #417 review: A's values were encoded under A's preset — retargeted under B with a different codec, the next projection would skip them as invalid and the property reads unset) — so it runs the rename-style cell re-key plus B2's codec-change re-encode/validate pass over the retargeted rows (unconvertible values report via the user-error channel, never silently unset) in the same tx through B2's funnel, with the ordering pinned; incompatible presets refuse the merge up front. Until that exists, definition-vs-definition merges are refused, not silently degraded. B3 (integration): merge/orphan/panel-layout handling — as shipped, merge is union-with-dedupe (§9): content-equal values fold (one canonical deleteSubtreeInTx, with the relocate rule preserving a folding duplicate's user-authored sub-children), divergent values persist as peer siblings, and source-only properties are adopted intact — nothing else is deleted, so the old relocate-then-subtree-delete preservation choreography is superseded. Cell end-state: measure first, then stage the sync-off

With §5's merge semantics in place and §6's uniform flip (all schemas, all scopes, at slice C — there is no separate kernel or plugin flip step anymore), the synced cell is harmless everywhere in a flipped workspace: pure derived projection, never a merge input. What remains is the pure-optimization tail — stop syncing properties_json, then drop it (wire savings, e2ee simplification, the cell becomes an ordinary local derived column). Because no key category stays cell-backed permanently (§6) and orphans are synthesized (§9), the old per-row composition rule collapses: eligibility is per-workspace — a workspace whose backfill is complete and whose §5 sweep reports zero divergence can leave the channel wholesale (properties_migration = 'cell-off', the same server-visible column the flip rides, which is what the sync-rules split keys on). No per-row marker exists in any form — 'cell-off' is only entered after the gates, so no state mixes an unfinished backfill with a stopped channel. Nothing in steps 3–5 is slice work: do not implement any of it until the tier measurements below say the sync-off is worth taking. The trap details are recorded so the eventual implementation doesn't rediscover them, not as an invitation to build. The staged sequence, each step gated on the one before:

  1. Arrival-authority flip ships (slice C) and the fleet fully dual-writes — every scope, every origin. Zero cold-start change; cell still syncs as bootstrap hint.
  2. Backfill completes per workspace (slice C) — every property has child rows behind it; the sweep's divergence counter reads zero over a full pass. Settle §10's outline-visibility question for system field rows along the way (the synced hidden definition field replaced the curated code list; what's left is deciding the outline's bit for panel-visible props like types).
  3. Stop uploading the cell — per workspace, once backfill-complete and sweep-clean. The rule (PR-review P1, simplified by §6's no-carve-out decision): in a 'cell-off' workspace no per-edit cell PATCH ships — with every key child-backed and orphans synthesized, the old per-row key-composition test has nothing left to discriminate. Two conditions gate the workspace transition, both checkable by tooling: the backfill has completed (no definition-less keys, no property without child rows) and a full §5 sweep pass reports zero divergence. (No per-row transitional guard exists — there is no state in which the upload stop coexists with an unfinished backfill: 'cell-off' is only ever entered after the gates above, and blocks created post-flip are born child-backed.) Fleet-buildability, previously a per-definition predicate here, is a runbook check under §6's uniform flip: all codec-bearing plugins are static, so "every client can re-project every key" means "all devices current, no codec-bearing plugin disabled" — verified at cutover time, not encoded per row. The check protects readers, not the merge (§5's per-client degradation): a degraded client lives off the downloaded cell as its read fallback for keys it can't build, and the sync-off removes exactly that fallback. "Stops uploading" also means the per-edit PATCH channel only (PR-review P2): creates keep sending the projected bag — apply_block_creates binds an explicit properties_json into a NOT NULL server column, so a bagless create would bind NULL and fail the insert; the one-time create payload was never the wire cost this step chases. And a second seam change joins the gates when this step is built (PR-review P2): definition-less keys stop being "mechanically inert" here. Pre-cutover, a raw repo.tx bag write with an unknown key rides the cell until the sweep synthesizes its definition (§6) — safe, because the cell uploads meanwhile. In 'cell-off' that interim copy is local-only: close the tab before the sweep runs and the write is stranded on one device. Before the upload stop ships, the same-tx materializer synthesizes definitions inline for unknown keys (the §9 inference, hidden, provenance-marked) so the children — which do sync — exist from the write itself. Semantics for rows that stop: the server copy freezes — and because downloads continue at this step, the materializer seam change moves up from the download stop to here (PR-review P2): a remote property edit re-delivers the parent row carrying the frozen stale cell, and the Layout-B materializer writes every downloaded column into blocks before invalidating readers — stomping the receiver's freshly-projected cell, readable as stale until the reconcile restores it, on every remote edit. The materializer must preserve (or re-project before invalidation) the local properties_json for rows whose cell has stopped uploading. A fresh device's downloaded hint also decays (every post-flip edit is missing until the projection pass sweeps the block). Status: must come after (1)–(2) — a remaining cell-only writer still needs the channel — and it is a transition, not a stopping point: it shares the next step's projection-readiness gate (PR-review P2).
  4. Stop downloading it — same workspace condition: workspaces not yet 'cell-off' keep the column. Mechanically a stream split in the sync rules — which also requires relaxing the Layout-B staging schema (PR-review P2: blocks_synced.properties_json is TEXT NOT NULL and the raw-table put binds every storage column, so a missing stream column arrives as an explicit NULL and fails the insert before the materializer can act. Make the column nullable in the same migration, or bind a non-null placeholder in the put itself — a column DEFAULT alone does nothing, since SQLite never applies defaults to explicitly-bound NULLs. And nullable staging alone only moves the failure one hop (PR-review P2): the materializer's UPSERT_BLOCK_SQL binds every storage column into blocks, whose properties_json is equally TEXT NOT NULL DEFAULT '{}' (blockSchema.ts), so the staged NULL fails at the live table before the first-sync projection can repair anything. Choosing nullable staging therefore also extends the step-4 materializer seam: a NULL staged cell is the "column stopped" case — preserve the local properties_json / re-project instead of writing the NULL through. The placeholder bind avoids the cascade entirely, at the cost of making '{}' ambiguous between "empty bag" and "column stopped" — harmless under §5's one-direction rule, since no merge decision ever reads cell shape. The placeholder is plaintext-workspaces-only, though (PR-review P2, round 25): e2ee materialization decrypts all three content columns, and a locally bound plaintext '{}' is not a valid envelope for the row's AAD — it fails or quarantines before any preserve/re-project logic runs. An e2ee workspace reaching cell-off (option-2 era, §8) takes the nullable/preserve-before-decrypt path — treat the staged NULL as "column stopped" and skip that column in the decrypt transform). The split's predicate must be server-visible (PR-review P2): sync rules select over public.blocks and can see neither a client's registry nor, for e2ee rows, any plaintext keys — which is exactly why the eligibility state lives in the workspaces.properties_migration column (§6): operator-written, fail-safe (only workspaces explicitly at 'cell-off' leave the stream), and already the thing the sync rules join on. The per-row client-written marker this step used to specify — with its buildable-on-every-client stamping rule (PR-review P2 ×2) — is no longer needed as global server state; what survives of the buildability concern is the runbook check in the upload step. Or keep downloading the column fleet-wide and consume it only where the workspace state says it is live; the download bytes are the smallest of the wins. Fresh devices now need the first-sync projection pass (below) before cells are readable. The measurements gate the upload and download steps together: take both or neither.
  5. Deprecate, then drop, the server column (stale from the upload stop onward; a future server-side feature that wants properties derives them from the child rows with a SQL view/aggregate — the data is all there for plaintext workspaces, which is the only case server features could read anyway). Under §6's no-carve-out decision plus orphan synthesis the column is genuinely droppable once every workspace reaches 'cell-off' — the end this project actually aims at; before that point it survives only as the transitional carrier for un-flipped workspaces. The drop has its own prerequisite (PR-review P2): the create path still binds the column — apply_block_creates binds an explicit NOT NULL properties_json, and the client's generated storage/upload column lists include it — so before any server-side drop, rework creates end-to-end (RPC signature, upload payloads, BLOCK_STORAGE_COLUMNS-derived lists) to omit the column while the local table keeps it as a derived, never-uploaded column. Dropping first would fail every new-block create in 'cell-off' workspaces.

Decision rule: if the gated, visible-panel-prioritized projection pass (tiers 2–3) lands under ~10–15 seconds, take the upload and download stops — the cell becomes fully local, and wire cost for property updates drops below today's. Implementation note for the upload/download stops: the Layout-B materializer applies sync rows with an explicit-column INSERT OR REPLACE into blocks; for rows whose cell has stopped uploading it must preserve (or immediately re-project) the local properties_json instead of wiping it on every arrival — needed from the upload stop on, not the download stop (the still-downloading frozen cell would otherwise stomp the projected cell on every remote edit). It's the single seam that applies sync writes — one place to get right. The first-sync projection pass reuses the backfill's harness (schema-readiness gating, per-workspace markers, chunked transactions, throughput logging), pointed the other direction — with one structural wrinkle (PR-review P2): the pass must be two-phase, because user-schema discovery itself reads the cell. userSchemasProjector finds schema blocks through types: ['property-schema'], and block_types is populated from properties_json.$.types — with no downloaded cell those entries don't exist yet. Schema unification dissolved the phase's hard dependency (its §4.4): phase 1's recognizer — the fieldIds of code-declared schemas — is a pure computation, uuidv5(workspaceId:seedKey) over the loaded declarations, needing zero DB state; and registry behavior needs no bootstrap at all (declaration synthesis serves every code-declared schema with zero rows synced). Phase 1 projects those schemas' keys (above all types and the property-schema:* config props) from field rows, which repopulates types/block_types and lets the projector register user definitions; phase 2 then projects user-schema keys. A single schema-readiness-gated pass would deadlock — waiting on schemas only the pass itself can surface — or complete early with user schemas unregistered. And the pass's cell writes must not ride repo.tx (PR-review P2): txEngine advances updated_at even under skipMetadata — that's the LWW clock, and advancing it locally can leave the cached row "newer" than genuine later server snapshots, which the acceptance path would then reject — and its upload triggers would re-ship the very cell channel the upload stop removed. The projection rewrites properties_json through the same sanctioned raw path the sync materializer uses (source-null, no upload enqueue, metadata untouched); it borrows the backfill's harness for gating and chunking, not its write path. And the raw write needs an explicit invalidation fan-out (PR-review P2): precisely because updated_at is preserved, the row-version never advances — the cache/hydrate LWW gate (applyIfNewer) sees nothing newer and raw SQL emits no invalidations — so a block already hydrated or query-cached before the pass reaches it would keep serving the stale cell until an unrelated edit. The pass must evict-or-force-refresh the touched blocks' handles per chunk (the observer's invalidation fan-out pointed at the projected ids), never rely on row-version deltas to propagate its repairs.

12. Risks to monitor

  • Write amplification at parents. Same shape as rev 1, now measurable: the spike added per-processor timing to the commit pipeline (TxTimingDiagnostics.sameTxProcessorRuns). Watch it during slice B.
  • Backfill upload burst per device. Completion markers are local (client_schema_state), so in the spike every device scans — the operator-run-once decision (§11 slice C) retires that shape: one device runs it, everyone else receives rows. The dedup concern survives in its general form — two devices offline-editing the same not-yet-materialized property mint duplicate field rows, and the dedup does not fire passively on arrival as the spike stands (the catch-up never visits the shared parent); slice C's parent-dedup fix closes this. Gate slice C on a two-client concurrent dual-write dedup test.
  • Local history storage multiplies — optimize row_events before slice C (history is kept; events shrink). row_events is an unbounded, write-only audit log — full before+after row snapshots per update, trigger-maintained for local and sync-applied writes, no runtime consumer, no client-side VACUUM (clientSchema.ts) — already at ~1.17M rows × ~2 KB on a dev instance. Unmitigated, the backfill's ~650k creates would append a snapshot each on the authoring device and again per receiving device as the rows sync down. Direction (no pruning — history stays; a dedicated design task owns the full spec, this bullet records the constraints #288 needs): (1) compact encoding with sampled anchors — store only changed columns for updates; before_json becomes an optional field written periodically (~1-in-50) as a full-snapshot recovery anchor, so reconstruction is anchored rather than full-chain replay back to the insert. One invariant stands regardless (PR-review P2): every chain starts full — a compact update targeting a row with no prior event promotes itself to a full after snapshot, so no exemption ever leaves a later edit without a base. (2) derived-write logging level — open, owned by the design task. Skipping derived-machinery writes (backfill, children→cell projection, sweep, catch-up) entirely may be too harsh — derived-write history is exactly what debugging the reconcile wants; the task weighs skip vs compact-log vs sample. Two constraints are firm: the backfill burst must not write ~650k full snapshots per device (the slice-C gate), and any exemption is by provenance, never by column — skipping properties_json wholesale would lose primary history for un-flipped workspaces and the whole pre-flip period (and §6's migrate-everything decision adds a knob the task should carry: UiState-scoped keystroke-frequency props want a provenance-level exemption of their own). (3) sync-source logging decided per workspace mode — redundant with the server-side blocks_history for plaintext workspaces; the only plaintext history for e2ee ones (starting position: drop for plaintext, keep compact for e2ee; the design task verifies blocks_history's coverage before relying on it). (4) migration of the existing ~1.17M events into the new format — chunked, resumable, idle-scheduled, with old before_json values sampled into anchors rather than dropped wholesale — plus idle VACUUM to reclaim the bytes. Re-stamp docs/row-events-retention.md when touching this — it references the deleted rowEventsTail path, and its prune design is superseded by this direction. The blocks_synced/ps_oplog full copies tripling with row count are separate — load-bearing sync machinery, priced by the tier-4 measurement, not an optimization target here.
  • Schema-registry timing. Derivation quality depends on which schemas are registered when a pass runs. Since schema unification the repo supplies the backbone: active-workspace transactions queue at a shared readiness boundary until the definition projector's first fresh snapshot (its §5.1 prime gate), and declaration behavior is available with zero rows. The spike's backfill-specific gating (held until projector publish, generation-coalesced rescans, row-targeted catch-up bypassing markers) layers on top; keep its tests where they still apply.
  • findSchemaByFieldId-style scans are retired. The spike's O(schemas)-per-lookup helper sat in child-filtering paths that run per rendered row. The unified registry is fieldId-keyed (projected definitions key by metadata.fieldId; the resolver exposes resolveField) — extraction consumes that instead of porting the scan.
  • Old-client interop during rollout — closed by policy, permanently. A device on pre-flip code writes cell-only. Its new-key sets converge without help (no children exist for the projection to fight with); its updates to already-backfilled keys and its clears do not — they are reverted by the next projection under §5's one-direction rule. This is the accepted design, not a window (owner decision; an earlier revision closed it with the checksum discriminator + materialize branches — see the §5 callout and review rounds 14–21): coordinated upgrade IS the mitigation, made mechanical by the flip runbook (all devices drained before the column moves) and made visible by the sweep's divergence log, with row_events as the recovery path for anything a straggler loses.
  • Stale bootstrap-hint window. A fresh device reads downloaded cells (and runs json_extract queries against them) before its projection pass has swept each block; a hint cell can be per-key stale from server column-LWW. Bounded and self-healing, but real: prioritize projection for visible blocks, and don't treat early query results on a cold device as authoritative. Goes away entirely at the download stop — and would worsen unboundedly under an upload-only stop (frozen server cell), which is why the two stops share one gate (§11).
  • Fresh-device initial sync moves ~3× the rows. Materialize-everything takes the primary workspace from ~350k to ~1M rows; PowerSync first-sync time scales with row count even though property bytes mostly moved rather than grew. This cost is independent of the cell question — measure it (tier 4 in §11) before the flip, and treat prioritized/lazy sync buckets as the eventual mitigation if it hurts.
  • Undo/replay. Same-tx processors are skipped on undo/redo replay by design (snapshot restoration is authoritative). Backfill writes are system-scoped and undo-free (tested in the spike). Verify undo of a value-child edit restores both the child and the projected cell (they're one tx, so snapshots cover both — the existing test should assert it explicitly).
