Properties as blocks

Migrate user-defined property values from JSON cells to first-class block children, with the existing properties_json column kept current as a derived index. A one-time read covering the target shape, the mechanism, and the migration path.

1. Why move

Today properties live in two parallel worlds. Property schemas are blocks ('property-schema' type, edited via a dedicated renderer, persisted under a workspace's Properties page). Property values are entries in a JSON cell (BlockData.properties: Record<string, unknown>) on the consuming block. Schemas get the full block treatment — refs, comments, history, sync, undo. Values get a row-local property bag with custom plumbing for every operation that needs to cross the bag boundary.

The asymmetry leaks into a half-dozen features. Copy-paste serializes blocks and copies the JSON cell separately — round-tripping through markdown loses property data entirely. Subtree-sharing has to special-case the cell. The Roam importer writes both schemas and values through different code paths. Comments on a field value, references to a specific field's setting, embedding a single field elsewhere — all require new surface every time. And cross-workspace name collisions on schemas silently coerce or drop values instead of surfacing a visible "this schema isn't here" affordance.

The motivating reframe: the user-facing properties UX is "fields are children of the block, rendered with structure" — that's what every Tana-style editor does. We've been emulating it with a sidecar panel because the data model said no. The data model can say yes.

2. The target shape

A property on a block becomes a carrier child: a regular block whose content is exactly a reference to a property-schema block, ((schema-id)). The carrier's own children are the property's values — one for scalars, many for lists, ref-typed values are themselves block references. Same primitive everywhere.

Three things to notice:

  • Carrier content is only the schema reference. No "schema + value in one string" — no parse ambiguity, no escaping rules.
  • Scalar versus list is just cardinality of value-children. The codec sees an array; scalar codecs unwrap one element, list codecs see them all.
  • Ref-typed values reuse the block-as-reference primitive: a value-child whose content is ((target-id)) is the value and the backlink edge in one shape. The existing references index already counts these.

3. The new affordances this unlocks

Putting properties on the tree gives them the same superpowers every other block already has. The motivation isn't symmetry for its own sake — each of these is a feature someone has asked for, today blocked by the JSON cell.

And the same model fixes a class of bugs around subtree operations:

  • Copy / paste / markdown round-trip. Today the clipboard carries blocks: BlockData[] and a separate markdown string; the markdown side has no slot for properties at all. Carriers are blocks — they serialize through the same path as any other child. Markdown export becomes faithful for free.
  • Subtree sharing. A shared subtree pulls properties along automatically because they're descendants. No special "also include the property cell" branch.
  • Undo of a subtree delete. Existing restore plumbing covers carriers. Today the JSON cell is restored by the row-level snapshot, which works, but ties property restoration to row restoration in ways that limit per-field undo.
  • Cross-workspace paste. Schema references either resolve (visible field) or don't (visible unresolved-ref glyph). Today the same situation silently coerces or drops. Honest failure beats invisible degradation.

4. Why this shape, not another

A few alternatives we walked and rejected, summarized so the choice is legible:

Approach Refs & comments on fields Sync conflict story Read-path cost Verdict
Stay on JSON cell no LWW on whole cell unchanged doesn't solve the motivating problems
Field handles addressing JSON entries partial still LWW on cell unchanged doesn't fix sync clobbering; partial fix
CRDT-typed JSON cell (Yjs/Automerge) no field-grained unchanged layering nightmare with PowerSync; saves Yjs for content later
Carriers + new projection table yes row-per-field new compiler path more code than needed; existing JSON path works fine
Carriers + materialize-back to JSON cell yes row-per-field unchanged chosen

The chosen shape mirrors the existing references[] pattern: the source of truth lives where the user writes (carrier children for properties, content markdown for references), and a same-tx processor maintains a denormalized cell on the row that downstream consumers (queries, indexes, UI selectors) already know how to read. We're reusing a pattern already in the codebase, not introducing a new one.

5. Mechanism: the same-tx processor

A new same-tx processor watches tree mutations and re-derives the affected parent's properties_json in the same transaction. It runs on every commit — local writes through block.set(schema, value), raw tx.create from plugins or agents, sync arrivals from other clients. All three converge on the same derivation. There is no "you must go through the mutator" rule.

Walk-up logic for any changed row C:

  • If C is itself a carrier → rebuild C.parent's cell.
  • If C.parent is a carrier → rebuild C.parent.parent's cell.
  • If a row moved between parents, run derivation for both before.parentId and after.parentId chains. The ChangedRow shape already gives both sides.

The derivation reads carrier-child content (the ((schema-id))), resolves the schema via UserSchemasService.getSchemaForBlockId, walks value-children in orderKey order, hands their content to schema.codec.decode, then writes the parent's properties_json (and the parent's projected references[] entries, with sourceField = schema.name) in the same transaction.

Determinism matters. Two clients receiving the same mutations in the same order must derive byte-identical cells. Sort property keys by schema id; sort projected refs the way normalizeReferences already does for content refs. The existing canonicalization pattern carries over.

6. Ghost carriers — type-lifted fields

A block of type Task expects fields priority, assignee, due-date (lifted via block-type:properties). When the user adds the type but hasn't set any values yet, we don't materialize empty carriers in storage — that would amplify writes, clutter the tree, and leave orphans when types change. Instead the renderer projects ghost carriers: render-only slots that materialize on first edit.

Projection rule, exactly mirroring today's syntheticProperties state in the panel: 

renderTypedBlock(P):
  expected   = union(typesFacet.get(t).properties for t in P.types)
  real       = childrenOf(P) filtered to carrier shape
  realIds    = set(carrier → schema-block-id)
  ghosts     = expected where schema-block-id ∉ realIds
  render(real ordered by orderKey, then ghosts in type-declared order)

Rule choices baked in:

  • Ghost position — after real carriers (in their orderKey order), before regular content children. Materialization picks an orderKey at the end of the real-carrier cluster so the visual position is preserved.
  • Type removal — real carriers stay (they're data the user entered); ghosts disappear because the renderer stops asking for them. No cleanup needed.
  • Schema removed from type — real carrier stays as a plain field without the "expected by type" affordance. Data persists.
  • Multi-type overlap — two types lifting the same schema dedupe by schema-block-id. Field order follows the order of P.types for grouping.
  • Materialization on focus — focusing a ghost is a UI focus, not a block focus. First keystroke runs a tx that creates the carrier + value and redirects editorSelection to the value's id in the same commit (mirrors focusBlock({edit: true}) atomicity).

7. Detection rules and edge cases

The rules the processor and renderer share, stated once:

Carrier recognition.
A child of P is a carrier iff its trimmed content matches ^\(\((uuid)\)\)$ AND the uuid resolves to a registered property-schema block. No other recognition path; no separate fieldOf property. The schema link rides the existing references[] pipeline.
Value decoding.
Walk the carrier's value-children in orderKey order. For scalar codecs: expect one value, pass its content to codec.decode; multiple children with a scalar codec → pick first, log diagnostic, keep extras in the tree. For list / refList codecs: pass the array of contents. Refs decode from ((target-id)) in the value-child's content.
Decode failure.
Log + omit from the projection. Carrier and values stay in the tree visible so the user can fix them. Mirrors today's tryBuildSchema behavior for invalid schema config.
Multiple carriers for same schema on one parent.
Pick the one with the lowest orderKey, log a diagnostic, keep the others in the tree. The data isn't lost; the projection picks one. Same shape as today's "duplicate key in JSON" would behave — except now the data is recoverable.
Default-value vs. unset.
Zero value-children = "carrier exists but unset" — not the same as the schema's defaultValue. The projection omits the key from properties_json in that case (mirror of "no key in JSON cell"). block.get(schema) still falls back to defaultValue; no behavior change for readers.
Field-projected refs into the parent.
Today ref-typed property values surface on parent's references[] with sourceField = schema.name. The processor preserves that — every value-child whose content is ((target)) contributes a parent-level reference entry tagged with the schema name. Backlinks-by-source-field queries keep working unchanged.
Carrier's own properties.
Allowed. A carrier is a normal block; whatever it carries on properties_json follows the same rules as any other block. The renderer for carriers chooses not to surface a property panel, so in practice users won't put anything there — but the model imposes no rule.
Sub-structure under value children.
Supported transparently. The codec reads value-child content; the renderer can show value-children's children as nested content (comments, elaboration). This is the new affordance — see Fig. 2.

8. The UI shift: panel sunsets, renderers take over

The property panel exists today because properties weren't blocks. Once they are, the panel's job — laying out rows, dispatching to value editors, showing add/delete affordances — moves into block-tree rendering. The renderer chooser picks CarrierBlockRenderer for carriers; the carrier renderer dispatches preset.Editor per value-child. Value blocks are regular blocks in the tree, keyboard-navigable, focusable, addressable.

Sunsets

  • BlockProperties.tsx shell
  • propertyPanel/* — model, sections, rows, layout, visibility
  • Synthetic-row state, recently-materialized flash
  • Hidden-fields toggle, metadata rows
  • AddPropertyForm (replaced by an add-field affordance on the parent block — slash command or autocomplete)

Survives, reused

  • valuePresetsFacet — carriers ask presets for the per-value Editor + Glyph
  • propertySchemasFacet — schema registry, unchanged
  • PropertySchemaBlockRenderer — schema-block editing UI, unchanged
  • resolvePropertyDisplay — single resolver consumed by the carrier renderer
  • Type registry + computeExpectedFields (extracted from propertyPanelSections) — drives ghost projection

The "add a field" UX changes shape. Today it's a form inside the panel; in the new world it's an action on the parent block — a slash command on a typed block opens the preset/schema autocomplete and creates a carrier + first value in one transaction. Same backing service (UserSchemasService.addSchema for brand-new schemas), different entry point.

9. Cross-workspace paste — the one new problem

Schema-id refs are workspace-scoped: the schema block lives in the source workspace's Properties page. Pasting a subtree into a different workspace means the ((schema-id)) in the carrier content may not resolve. Three reasonable strategies, in increasing aggressiveness:

  1. Leave dangling. Unresolved blockrefs render with the existing "unresolved ref" style. The user sees there's a missing schema; they can decide to recreate it or remove the carrier. Honest. Default.
  2. Resolve-by-name on paste. Walk the pasted subtree, find carrier contents, look up the source schema's name in the destination's registered schemas, rewrite the ref to point at the destination's matching schema block. Only safe when the destination has a schema by the same name and with an equivalent codec — name match alone risks coercing into a different value type.
  3. Create-on-paste. Same as (2), but also create the schema block in the destination's Properties page when the name isn't there. Surfaces as "this paste added 3 new fields to your workspace" — user-visible side-effect.

The problem already exists today, just invisibly. With JSON cells, properties keyed by a name the destination doesn't recognize fall through to the unknown-schema fallback path; with a same-name-different-codec schema they silently decode through the wrong codec. Carriers surface the mismatch — the unresolved-ref glyph is the signal. Pick (1) as the v1 default; (2) and (3) become explicit "import properties" flows triggered from a paste confirmation, not automatic.

10. Migration phases

The migration is staged so each phase is independently shippable and reversible. The JSON cell stays load-bearing through the whole sequence — it's the derived output that readers see; carriers and values become its source.

0
Same-tx processor, shadow mode

Ship the carrier-recognition logic and the derivation processor. Wired into the commit pipeline but writes to a shadow column (properties_json_v2) instead of properties_json. Diff the two on every commit; log mismatches.

Deliverable: processor lands, dark-rolls in production, zero user-visible change. Mismatches surface bugs before the cutover.

1
Write path: block.set creates carriers

Upgrade block.set(schema, value) (and tx.setProperty) to find-or-create a carrier-child and write value-children, instead of writing the JSON cell directly. The processor derives the cell. Existing readers — block.get, the typed-query compiler, the panel — keep reading the cell, unaware.

Deliverable: every new property write goes through carriers. Old data still lives in JSON cells until backfill.

2
Backfill

One-shot job per workspace: walk every block with non-empty properties_json, materialize carrier + value children for each entry. Idempotent — running it twice produces the same tree. The processor re-derives the JSON cells; the diff should be zero (every key already in the cell, just now backed by carriers).

Deliverable: 100% of property data lives in carriers; the JSON cell is fully derived. Fall back rule: if backfill fails on a block, the JSON cell stays authoritative for that block; the write path can still derive on top of it (the carrier shape and the cell will agree).

3
UI shift: in-tree rendering

Ship CarrierBlockRenderer. Block trees start rendering carriers and values inline. The property panel stays mounted behind a feature flag as a fallback for a release or two. Slash-command on the parent block for adding new fields. Ghost carriers project from the type registry.

Deliverable: users see and edit properties in the tree. Panel still available via flag for users who want it during transition.

4
Panel sunset, cell becomes pure derivation

Remove the panel feature flag. Delete BlockProperties.tsx, propertyPanel/*, the synthetic-rows machinery, AddPropertyForm. Direct writes to tx.update({properties}) are deprecated and route through a compatibility shim that creates carriers (or get removed if no callers remain).

Deliverable: one source of truth (carriers), one read path (JSON cell as derived index), one rendering surface (the tree). Net code deletion.

11. Risks and rollback

Each phase has a back-out:

  • Phase 0: shadow column has no consumers. Remove the processor; drop the column. Zero impact.
  • Phase 1: revert the block.set change. New writes go back to JSON cells. Carriers written under the new path remain in the tree as inert blocks (their ((schema-id)) content is still meaningful; the panel/queries just don't read from them anymore). One-shot cleanup script can collapse them back to JSON if desired.
  • Phase 2: backfill is idempotent and additive — it never removes JSON cell data, only writes carriers alongside. Rollback = stop running the processor and stop reading carriers; cells remain authoritative.
  • Phase 3: feature-flag drives renderer choice. Flip the flag workspace-wide to fall back to the panel.
  • Phase 4: point of no return. Don't ship Phase 4 until Phase 3 has been stable for at least one release cycle and the shadow-mode diff log from Phase 0 has been silent for the same period.

Concrete risks to monitor:

  • Write amplification at parents. A parent with N fields rewrites its JSON cell on every field-write. Expected to be fine (today's tx.update({properties}) is also O(N)); watch p99 tx duration during Phase 1.
  • Sync conflict resolution on the JSON cell. Two clients touching different fields each rewrite the cell; the cell could LWW-clobber. The mitigation is "on every inbound carrier mutation, re-derive locally" — the cell is treated as output, not input, of merge. Verify with a multi-client integration test during Phase 1.
  • Backfill correctness on edge schemas. Properties whose schema has no registered preset, or whose codec rejects the stored encoded value, won't backfill cleanly. Plan: log + skip per block, leave the JSON cell as authoritative for those rows. Phase 4 cutover requires the skip list to be addressable.
  • Cross-workspace paste regressions. Section 9 covers the model answer; the UX needs explicit "X fields not found in destination workspace" affordance in the paste confirmation. Belongs in Phase 3.

This is a planning document, not an authoritative spec. Open questions worth revisiting before Phase 1: exact storage of carrier orderKeys (preserve from prior cell-key order on backfill?), the schema-block-id reference resolution path for the processor when the schema block isn't yet hydrated in cache, and the shape of the slash-command UI that replaces AddPropertyForm.
