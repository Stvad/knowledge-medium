How to read this doc

The load-bearing core is §3–§4 (the event format and the anchor/reconstruction machinery) and §8 (the migration of existing history). §5 and §6 resolve the three policy questions (derived-write logging, UI-state retention, sync-source logging). If you are implementing, start from your slice in §13 and the invariants index. If you are PR #288, read §11 — it is the contract your slice C consumes. SQL in this doc is illustrative shape, not implementation.

Decisions made in this doc (each argued in its section):
  • Compact updates are invertible, column-granular patches: an update event stores the changed domain fields' before and after values (equal key sets); identical updates are not logged (§3).
  • Anchors are sampled — one in-trigger coin flip at p = 1/64, no write-path history reads (adopted per O5, superseding a deterministic probe + counter design, §15). Anchors carry full before+after and are pure read-acceleration + corruption-bounding: no correctness property depends on them (§4).
  • Every event is self-restoring; reconstruction never leans on a predecessor: walks terminate on the newer side unconditionally — live row, delete event, or anchor (§4.3). The old "every chain starts full" promotion machinery is unnecessary (§4.1).
  • Every write logs — there is no exemption mechanism (§5, revised 2026-07-17): derived writes and one-shot bulk bursts alike. The history_mode = 'skip' opt-in an earlier draft carried is deleted, which is what makes the log self-sufficient (I8). UI-state churn — measured at roughly half of a working device's update events — is logged but hard-bounded: a retention sweeper prunes each no-history-typed or identity-confirmed block (panels carry the type; identity covers rows an old client created untagged) to its newest K events. The events stay compact like any other update; keeping the newest K is a clean oldest-prefix delete, which is safe for compact events, so no full-writes are needed (user call, O6/§5.1).
  • Sync-applied events are kept, compact, for every workspace (user call 2026-07-15, superseding an earlier per-workspace-mode drop): local history stays self-sufficient instead of leaning on a server read path that doesn't exist yet. The server-redundancy analysis is retained in §6 for the record; an explicit sync-apply marker still distinguishes sync from raw writes, now for truthful labeling only.
  • Triggers remain the writer; bodies are generated from a single column spec shared with the existing snapshot/upload projections (§7).
  • The migration is reconstruction-lossless and backup-prompted: chunked, resumable via the format tag itself, sampling anchors at cadence (chain discontinuities need no special anchor — every unlogged gap is metadata-only, I9), so every state derivable today stays derivable — and it arms only after the user has been prompted with a full DB export (§8).
  • One-time full VACUUM (deep-idle + manual command), not incremental_vacuum — in-place row rewrites leave half-full pages only a rebuild compacts (§9).

1. Why now

row_events is the client-side per-row audit log: three triggers on blocks write one event per row write, for local and sync-applied writes alike (src/data/internals/clientSchema.ts:370-381). Updates store two full row snapshots — before and after — including the whole properties_json and references_json columns (blockJsonObjectSql, clientSchema.ts:386-402). Nothing reads the table at runtime (clientSchema.ts:63-65); it is intentionally unbounded and never trimmed (clientSchema.ts:67-69), and there is no client-side VACUUM anywhere in src/, so even a future deletion would not return OPFS bytes.

Observed scale and the two forcing functions:

  • It is already the biggest thing in the database. Measured live 2026-07-15 (§12): the long-lived dev instance holds 1.94M events carrying 1.92 GB of payload inside a 4.42 GB OPFS file, for a ~345k-block workspace — history bytes exceeding live-data bytes, on devices where OPFS quota is finite. (The earlier ~1.17M × ~2 KB figure from docs/row-events-retention.md:4 had grown ~65% by measurement time.)
  • PR #288 multiplies it. The properties-as-blocks backfill creates ~650k rows through repo.tx per workspace (its §12 and measurement section; the spike took 5–10 minutes largely paying row_events + ps_crud envelopes), and those rows then sync down and append events again on every receiving device. Its §12 carries a constraint record written against an earlier draft of this design — that the optimization must land before its slice C runs. That gate is gone (revised 2026-07-17): with the skip exemption deleted there is nothing left to plumb, so the two projects are independent and may land in either order. §11 is the current contract; what #288 gains by waiting is the cheaper per-event encoding, not a prerequisite.
  • Every fresh device pays on first sync. The observer materializes each downloaded block into blocks (materialize.ts:365), firing the INSERT trigger once per block: ~345k full-snapshot create events (~205 MB measured on the fresh prod instance, §12) per device, recording rows the server already has. (§6 keeps these — an accepted cost of self-sufficient local history; creates are inherently full, so v2 doesn't shrink them either.)

The consumers this format serves are all future — block history / time-travel UI, drift-tolerant undo (src/data/internals/undoManager.ts:26-28 names row_events.before_json as the substrate; today's undo is in-memory only, depth 100 at undoManager.ts:80-82), forensics after incidents (the SRS backlink-wipe class), and convergence debugging for the props migration. Because nothing reads the log yet, we are free to change the format — the design is judged by what those readers need (§10), not by compatibility.

Discovery is the flag partition ∪ live block_types ∪ computed identity (§5.1); this paragraph specifies the index that serves the first of those, not the set of sources. It then uses the full (block_id, id) index per candidate to count events and cut the oldest prefix by id (the prune decision comes from the authoritative type, not from any event flag — §5.1). The cut deletes flagged and unflagged rows of the whole block, so this step needs the all-events index, not the partial one. Ordering the cut by id (not created_at, which is not reliably monotone in id) is what makes it a safe oldest-prefix delete (§5.1). That full (block_id, id) index is the same one slice C's reader walks and the §8 rewriter need; whichever of B/C ships first creates it idempotently (CREATE INDEX IF NOT EXISTS), so neither depends on the other's ordering (slice B is independent of C). id is AUTOINCREMENT-monotonic and is the authoritative per-block order (same-millisecond events in one tx tie on created_at). The existing (block_id, created_at DESC) index is retained (review finding): per-block time-seek ("latest event ≤ T") has no sound substitute — id order is insertion order, and created_at is not reliably monotone in id under wall-clock steps, so bisecting created_at over an id-ordered index can pick a wrong event or degrade to a scan (see the subtree reader row in §10). (created_at DESC) and (tx_id) stay for time-scoped and tx-join forensics.

4. Anchors: invertible payloads, sampled full snapshots

4.1 Decision — invertible compact events; anchors by coin flip

Adopted 2026-07-15 (user call; decision record in O5, superseding the deterministic predecessor-probe design — its record is in §15). Two properties define the format:

  • Every compact event is invertible. before_json and after_json carry the same key set — the changed fields — with old and new values respectively. Applying after_json onto the pre-state, or un-applying before_json from the post-state, are both single-object merges. No event's interpretation or inverse depends on any other row existing (invariant I1).
  • Anchors are sampled, not scheduled. On each logged update the trigger flips one coin (random(), p = 1/64 — argued in §4.4); a hit writes the event full (complete before + after, full = 1) instead of compact. No probe, no counter, no history reads on the write path — only the 1-row tx_context probes invariant I6 permits. Creates and hard deletes are full by nature.

"Never references predecessors" (below, and §8.2) is a statement about dependence, not just stored pointers: an event's payload is defined against the row states the write itself transitioned — the trigger's own OLD and NEW at fire time — never against another event's content or format. The only cross-row relationship in the log is position (id order), which readers use to know where to walk, not to make any event meaningful. The old design lacked exactly this: its after-only compact events were interpretable only relative to the predecessor's reconstruction, so the trigger had to read the predecessor and promote on v1/missing — a row-to-row data dependency that priced every gap class and every rewrite. Self-containedness costs nothing at write time because the trigger already holds OLD and NEW.

What the deleted probe machinery used to buy, and why it is no longer needed:

  • Chain-start promotion (the old I1, "every chain starts full") — chains no longer have bases: reconstruction walks from the newer side, which always terminates (§4.3), and every event restores its own fields by itself. The first edit after any unlogged gap (§5 skip, trigger-suspended window) is fully usable the moment it lands, including for undo.
  • v1-predecessor promotion — v2 events never reference predecessors, so the §8 rewriter keeps its zero-coordination property with the machinery deleted rather than because of it.
  • The deterministic walk bound W — replaced by an expected bound with a geometric tail (§4.4). Acceptable because walks cost microseconds per event, anchors carry no correctness weight, and the current state is never at risk.
Implementation trap — one coin per event. The trigger body needs the anchor decision in three places (before side, after side, the full column); three separate random() calls would disagree (e.g. a full before paired with a compact after). Compute the flip once and reference it — e.g. INSERT INTO row_events ... SELECT ... FROM (SELECT (random() % 64) = 0 AS anchor). No abs(): abs(random()) raises an integer-overflow error on −2⁶³ (p = 2⁻⁶⁴) and aborts the user's whole tx from inside the trigger; plain random() % 64 hits 0 at exactly 1/64 (negative multiples of 64 included). Pin with a shape test: both payload sides of every event agree on key set / fullness. (Verified empirically on SQLite 3.51: the shape is valid in an AFTER UPDATE body alongside NEW/OLD, and the coin is not re-evaluated per reference site. Porting trap: in a multi-row INSERT … SELECT cross-joined against the coin subquery, SQLite freezes ONE coin for the whole statement — never reuse this shape for batch rewrites.)

4.2 Anchor content — decision

Anchors carry full before + full after (not after-only):

  • Integrity check for free: replaying the compact run from the previous full event must reproduce anchor.before_json. The §8 migration's equivalence test and any future integrity sweep get a built-in checkpoint at every anchor.
  • One-sided recovery: if the run before an anchor is lost, the anchor's before still yields the state just prior to it.
  • Symmetric bases: a full event serves forward walks (its after side) and backward walks (its before side) equally — §4.3 picks whichever full state is nearer. Every full event is crossable in both directions; there is no non-crossable "wall" variant, because every unlogged gap is content-invisible (§3/§4.3, I9).
  • Cost is noise: one extra full pair per 1/p events ≈ +25 B amortized per update at p = 1/64 (§4.4).

4.3 Reconstruction: bidirectional, the newer side always terminates

stateAt(blockId, eventId):                # row state just after event E
  E = event(eventId)
  if E.kind is 'delete':
      return ABSENT                        # state just after a hard delete is a tombstone.
                                           # E.before_json is NOT this state — it is only the
                                           # newer-side base for reconstructing OLDER events.
  if E.v is NULL or E.full:
      return E.after                       # create / anchor (full=1): the full after snapshot

  # nearest full state on EITHER side, whichever is closer:
  #   older side: newest event ≤ E with a full after (create, anchor)
  #               → walk FORWARD applying after_json patches
  #   newer side: oldest event ≥ E with a full before (anchor, delete),
  #               or the live blocks row itself
  #               → walk BACKWARD un-applying before_json patches
  #
  # The newer side ALWAYS terminates: a live block ends at its blocks row,
  # a purged block at its delete event (full before). ChainBroken can only
  # mean corruption inside the log — see the failure bound.
  #
  # Gap safety needs NO read-time check and NO wall marker (simplified
  # 2026-07-18). An unlogged gap could only splice stale values for fields
  # CHANGED inside the gap — but every unlogged blocks write is metadata-only
  # by invariant (I9): the sole suspend site backfills user_updated_at once
  # (ensureBlockUserUpdatedAtColumn), and every other write path — user, sync-
  # apply, raw sql, restore — fires the trigger. So content/structure is
  # continuous across every gap; any full state on either side is a valid base
  # for content/structure, in either direction, unconditionally. Engine-metadata
  # columns (updated_at / user_updated_at / updated_by) are the ONLY fields a gap
  # touches, and no reader reconstructs them by walking: undo re-stamps them
  # (§10), and a forensic "exact row at T" reads them as approximate-by-a-known-
  # transform across the one backfill point (it set user_updated_at = updated_at).
  # The pre-log era is the only other gap — a block predating logging has no
  # create, so its OLDEST state is older-side-incomplete (terminates early), never
  # a mid-history wall. (A FUTURE suspend site that mutated CONTENT would break I9
  # and need a reintroduced discontinuity marker — none exists, so none is built.)

Complexity: expected walk = distance to the nearest full state on either side — E[one-sided gap] = 1/p = 64 events, so expected walk ≲ 32, p99 one-sided ≈ 4.6/p ≈ 295 (§4.4); each step is one ~0.4 KB JSON parse + ~3-field merge, microseconds. A whole-block timeline streams once carrying running state, either direction: O(events of block). Recent history — the common UI case — is effectively free and exact: it walks backward from the live row through trigger-recorded OLD values. Content/structure is exact unconditionally (every write logs, §5; the one unlogged gap is metadata-only, I9); the only older-side incompleteness is a pre-log block whose history predates logging, which simply terminates the walk early rather than holing it.

Failure bound. Corruption severing the log confines damage to states older than the break, back to the next-older full state — expected ~1/p events, p99 ≈ 4.6/p. (One non-corruption cause of a broken newer side exists: a raw id rewrite orphans the old id's chain, which recovers via its older-side fulls only — §3's asterisk.) Everything at or newer than the break reconstructs from the live side, and the current state is never at risk (it lives in blocks). Anchor before-sides double as integrity checkpoints: replaying the span between two full states must reproduce the newer one's before_json.

4.4 Choosing p — a checkpoint-spacing problem

Under this format anchors carry no correctness weight — every property in §4.3 holds with zero anchors — so p is chosen economically, as in the classic checkpoint-interval tradeoff:

  • Cost is linear in p. An anchor adds Δ ≈ 1.6 KB over a compact event (~2 KB full pair vs ~0.4 KB patch pair). Per-update overhead = p·Δ: ~64 B (+16%) at 1/25, ~25 B (+6%) at 1/64, ~13 B (+3%) at 1/128. (Measured full pairs run 1.4–2.4 KB — §12 — so Δ spans ~1.0–2.0 KB and the amortized cost ~16–31 B/update at 1/64; noise either way.)
  • Read latency does not bind. Expected point-lookup walk ~1/(2p) events at microseconds each; even 1/p in the several hundreds keeps far-past lookups sub-millisecond, and the common recent-history reads walk from the live row and touch no anchor at all. Optimizing p for reads would push it toward one — anchor everything, and every lookup is O(1) with no walk at all. The point is that nothing pushes it there: read latency is nowhere near binding, so p is free to be set by the loss window below, which lands it far lower. The user's "plausibly rarer" instinct is right.
  • The binding constraint is the loss window (and with it, integrity-checkpoint granularity): how much of a block's older history may become unreconstructable when corruption severs its chain, and how tightly an integrity sweep can localize a divergence (one inter-anchor span). Gaps are geometric, so to guarantee a p99 window ≤ L events, set p ≈ ln(100)/L ≈ 4.6/L.

Default: p = 1/64 — expected gap 64, p99 ≈ 295, +6% update bytes: "99% of corruption-loss windows and far-past walks stay under ~300 events" at byte cost that is noise against the creates dominating §12. Rarer is defensible (1/128 → p99 ≈ 590, still microseconds to walk, still bounded per block by its create); denser buys only checkpoint granularity. The constant lives in the trigger SQL, re-installed every boot (clientSchema.ts:1170-1175); mixed rates coexist freely since readers key on full, never on cadence. Revisit with history-UI telemetry — observed walk lengths are the one number that settles this empirically.

Measured reality check (2026-07-15, §12): the per-block event distribution is flat-plus-spiky — mean 2.9 events per ever-logged block on the dev instance, yet maxima of 8,082 (dev) and 14,702 (prod — a single UI-state panel block, §5.1), with only ~80 blocks above 64 events. For 99.99% of blocks anchors never fire and never matter: their walks terminate at a create or the live row within a handful of events. Anchors are in practice a hot-block mechanism — which is exactly where the loss-window bound and integrity checkpoints are worth paying for. p = 1/64 stands.

Revision note (2026-07-17, user call): earlier drafts exempted enumerated one-shot bulk bursts (the #288 backfill) from logging via a tx_context.history_mode = 'skip' gate. That mechanism is deleted — every write logs. The argument is below; the deleted design, and what it cost, is recorded in §15.

  • Steady-state derived writes log normally (option d): projection, sweep repairs, per-row catch-up. In compact form a 2–3-write fan-out per property edit costs ~1 KB total — affordable, and this is the record convergence debugging needs — with one honest limit on granularity. No new provenance column: every local event already carries tx_id, and command_events (scope, description, mutator_calls; commitPipeline.ts:348-368) identifies the writing tx and the tx.run mutators it invoked — provenance is a join, and a later compaction pass could target it by that join if the volume ever warrants. What the join does not resolve (review finding): a same-tx processor pass (projections / normalizers running after the user fn) shares the outer tx's tx_id and is not separately named in mutator_calls, so its derived row writes attribute to the tx, not to "which pass". Per-pass attribution for those would need processor provenance the design does not add today; the claim is the tx and its mutators, from/to/when — not the internal pass.
  • So do the one-shot bulk bursts — the props backfill, the arrival catch-up pass, future whole-table column backfills. The exemption that used to cover them is deleted, for four reasons:
    • It was asymmetric, and pointed the wrong way. Because §6 keeps sync events, every other device logs the backfill's 650k creates as sync arrivals regardless. The skip saved ~230 MB on the one device that ran the migration — precisely where a bug would originate — while the devices that merely received the results kept the full record. You would have debugged the migration on a machine that never ran it, and the fleet's histories would disagree about whether it happened.
    • The riskiest write is the one you least want unlogged. This system's own incident history is exactly this class of operation — bulk passes that damaged live data and whose recovery depended on reconstructing what the pass actually did. A 650k-row migration is the single riskiest write it will ever perform; that is an argument for logging it, not against.
    • The economics moved under the decision. Skip was reasoned about against a 2.2 GB log. This document's whole job is freeing ~1.45 GB; the burst costs ~230 MB once. The room exists now.
    • It bought a permanent invariant hole for a rare event. Bulk migrations should not be routine. With no skip, every block's history opens with its create — so the log is self-sufficient (I8) and §10's parent-edge index is fully log-derivable instead of leaning on the live blocks table as a crutch.
    Cost owned: ~230 MB one-time on the authoring device. Creates are incompressible — compaction does not shrink them and VACUUM will not reclaim them; that is real data we are choosing to keep. Bulk cleanups likewise pay a tombstone per row: the dev instance's 317k-block workspace-migration purge cost ~194 MB of delete events (measured, §12) — the price of "every deleted block's last state survives locally". If some future burst is genuinely unaffordable, an exemption can be reintroduced then, against a concrete case rather than a speculative one.

With no exemption gate, the trigger WHEN clauses lose their history_mode condition entirely — the only remaining bootstrap-scoped silence is the existing withTriggerSuspended bracket (clientSchema.ts:1187-1200), which stays bootstrap-only because dropping a trigger is DB-global and would also unlog concurrent writers mid-session. Invariant I4 (exemption by provenance, never by column) survives as a standing review rule rather than a live mechanism: if an exemption is ever reintroduced, it may not condition on which columns changed — skipping properties_json wholesale would lose primary history for carve-out/un-flipped/orphan keys (PR #288 §12).

5.1 Measured: UI-state churn dominates organic updates — bounded, not skipped (O6)

Live measurement (2026-07-15, §12) surfaced a third write class this section's draft missed. On the 23-day-old prod instance, the top four blocks by event count are all panel/UI-state blocks (types: ["panel"] — focused block, scroll position, editor selection), together accounting for ~26k of ~55k update events: roughly half of a working device's update traffic is selection/scroll churn, topped by a single panel block with 14,702 events in three weeks. These writes rewrite the whole properties value each time, so compact v2 patches only shrink them ~1.7× — the changed "field" is the properties object itself.

The class is identifiable two ways, and which one the sweeper keys on is the whole design. By provenance: these writes go through repo.tx under ChangeScope.UiState = 'local-ui' ("Selection / focus / chrome state", explicitly not undoable — changeScope.ts), and tx_context.scope is populated per tx (commitPipeline.ts:263-264). And by the block itself: UiState writes land on a small, known set of view-state blocks — panels and panel-stacks (panelLayoutProjection.ts), but also the user-level ui-state child outside a panel (getUIStateBlock, stateBlocks.ts:344-362) and plugin-specific UI-state blocks (stateBlocks.ts:376-397; e.g. QuickFind's quick-find-ui-state) — each created by the UI-state machinery, not by the user (review finding: it is not panels only). The difference is decisive: scope is per-tx and lives in tx_context, which is NULL on every sync arrival (§3) — so keying retention on it strands ~half of a hot block's events (the sync-authored ones) unprunable, the defect the earlier draft of this section carried. The block's type has no such blind spot: it is part of the block's replicated state, present and identical on every device. Retention keys on the type — and every one of these view-state blocks gets tagged no-history (below), so the guarantee covers the whole class, not just panels.

Decision (user call 2026-07-15; mechanism settled 2026-07-17) — log them, hard-bound retention: keep only the newest K events per block. Key the bound on the block's type, not the event's scope, and prune the block's whole history to newest-K. Skipping entirely was considered and set aside: a short local record keeps panel-state bugs debuggable ("what did this panel hold a minute ago"), while bounding is what actually matters — the class's cost is unbounded accumulation, not per-event size. Default K = 8 (boot-tunable like p; the value is having a recent record, not depth), capping a panel block's local history at ~K × 1.4 KB ≈ 11 KB regardless of write rate — versus 35 MB measured on the hottest prod panel in three weeks.

Mechanism — three parts, each chosen so the signal survives sync:

  • Prunability is its own explicit type, no-history, carried in the block's types. Not a facet on panel, and not a per-event flag or per-block column: a distinct, purpose-built type whose name is its contract — a block tagged no-history is declaring "don't keep my history", and the sweeper prunes exactly those. Every UI-state block gets it at creation, across all creation paths (review finding — there is no single factory): the root ui-state child via getUIStateBlock / ensureUiChild, the plugin UI-state children via getPluginUIStateBlock / ensureStateChild (stateBlocks.ts:376-397 — a distinct seam, easy to miss), and panels via panelLayoutProjection, and panel-stack rows via createPanelStackRowInTx (tx.create + addTypeInTx directly, not through ensureUiChild — review finding). Each stamps no-history into types. Two implementation constraints (review findings): (a) no-history must be a registered TypeContribution (a kernel type, like panel) before anything stamps it — addTypeInTx throws on an unknown type id, so an unregistered marker would abort creation, backfill, and the promised user assignment rather than tag. (b) there is no single factory: the ensureStateChild-based seams (getUIStateBlock, getPluginUIStateBlock) can be covered by threading through the ChangeScope.UiState wrapper ensureUiChild (stateBlocks.ts:180-188), but the panel and panel-stack seams do not go through it and must stamp explicitly. The stamping is gated on ChangeScope.UiState, never in the shared ensureStateChild one layer down: ensureStateChild also builds UserPrefs blocks (ensureUserPrefsChild, getPluginPrefsBlock, stateBlocks.ts:190-198/:322-338), and tagging those would make real preference history a retention candidate — the sweeper must never touch prefs. The backfill tags on the same authorization the sweeper uses, never on ancestry alone (review finding): writing the visible no-history type onto a block merely because it sits under ui-state is the mutable-parentage gate §13 rejects — and worse than an inert misclassification, because the tag persists and turns destructive the moment B2/O7 honours user tags. So the pass tags provenance/identity-confirmed state rows and reports ambiguous descendants rather than tagging them. With that constraint, it walks every existing block in the ui-state subtree, from from every workspace member's root — not just the active user's (review finding): UI-state roots are per user (getUIStateBlock puts ui-state under that user's page, stateBlocks.ts:344-361) while sync downloads every block in the workspace (sync-config.yaml), so a collaborator on a legacy client keeps syncing untyped, unflagged UI-state churn into this DB that a single-root walk never sees. Enumeration is cheap and needs no traversal because the roots are computable: userPageBlockId(workspaceId, userId) for each member, then stateChildBlockId(userPage, 'ui-state') (stateBlocks.ts:54-58) — not an enumerated by-kind list (panels only) that a review finding showed misses the layout-sessions children and their descendants: getLayoutSessionBlock creates ui-state/layout-sessions/{id} through the same ensureUiChild primitive (stateBlocks.ts:368-374) and hot-activation writes land selection/history events on those blocks (panelLayoutProjection.ts:605-620). Walking the subtree is how candidates are found; what gets tagged is only those the authorization rule above admits — provenance or identity-confirmed — with ambiguous descendants reported instead (review finding: "tag every descendant" is ancestry-based authorization by another name, and because the type persists, a content block moved under ui-state and tagged here becomes destructive the moment B2/O7 honours user tags, long after the move is forgotten). For the descendants that do qualify, creation-time stamping threads through the shared ensureUiChild primitive, so new ones (of whatever future kind) are covered automatically — provided the creating client is upgraded. It is therefore a recurring reconciliation, not a one-time backfill (review finding): an older tab or device can still create (or sync in) a panel / layout-session / plugin UI-state block after the one-shot marker has run, and that row carries no no-history type at all — so a backfill that believed it only owed pre-slice rows would leave a brand-new hot UI-state block with neither type nor flag, invisible to the sweeper and growing unbounded. Two consequences: the subtree tagging pass re-runs (cheaply — the ui-state subtree is ~857 panels plus the root/plugin/layout-sessions rows, not the whole DB) rather than latching a completion marker; and, more importantly, B1's discovery of live UI-state blocks is structural, not tag-dependent — the sweeper already has to confirm dedicated-UI-state membership before pruning anything in B1 (§13), so it discovers those blocks by computed identity directly and an untagged old-client creation is found anyway. The type and flag stay load-bearing for the cases structure cannot reach: deleted blocks, and (post-O7) user-tagged blocks outside the subtree. The walk must not filter out deleted = 1 (review finding): a UI-state block soft-deleted before slice B has no live tag event to set the flag, so the partial-index sweeper would never discover it and its history would sit unbounded forever — contradicting the "deleted flagged blocks are bounded" claim below. A soft-deleted block is still a blocks row with its parent pointer, so computed identity — the same signal used for live ones, not a reintroduction of structural panel keying — reaches it. (Implementation note: the strict type path rejects tombstones, so this needs a tagging path that tolerates deleted = 1.) Measured legacy stock on prod: 365 soft-deleted UI-state blocks holding 8,908 events (max 3,116 on one) — static, already-deleted, ~12 MB. Hard-deleted legacy blocks are genuinely unreachable — no row to tag, and their v1 snapshots predate the no-history type so there is nothing to match on — but the measured count is 0 on prod, so the residual is currently empty and, being already-deleted, cannot grow. Named, not silent. The backfill is per-workspace, run when a workspace is active — it must not write blocks in workspaces the user hasn't opened (the sync-queue-flood rule the rest of the system holds to). For a writable, synced workspace that is sufficient, not a coverage gap, because the type rides sync: whichever device opens a workspace stamps no-history once, it replicates to every device, and each device's sweeper then prunes its own local row_events off the synced type — no device needs to run the backfill itself. But nothing guarantees any upgraded device ever opens it (review finding): PowerSync downloads every block for all user_workspaces (sync-config.yaml:64-76), so a legacy collaborator's UI-state churn materializes into this DB — and grows here — for a workspace this client never activates and therefore never tags. The resolution turns on a distinction the "don't touch unopened workspaces" rule already draws: that rule forbids writing blocks into a workspace the user hasn't opened, because that floods the sync queue. Pruning row_events writes no blocks and uploads nothing — it is purely local bookkeeping. So the two halves scope differently: tagging stays active-workspace-only (it is a block write that syncs), while discovery and pruning run locally across every synced workspace, keyed on whatever signal is available there. That bounds remote-authored churn in workspaces this device never opens without violating the flood rule at all. A read-only workspace is the one case the tag can't ride sync (review finding): the user is a member but not a writer, so the server blocks_write policy rejects the tag write (is_workspace_writer, consolidated_initial.sql:630). But it doesn't need to ride sync — with one correction to how (review finding): UiState is not a local-only scope. It source = 'user' and uploads like any other write; on a read-only workspace its readOnly: 'allow' policy keeps the local write but the server rejects the upload into the rejection quarantine (changeScope.ts:8-13/:61-64), so it doesn't block the queue but it isn't free either. So the pre-slice UI-state there is already effectively per-device (its uploads were always quarantined), and each device tags its own copy at creation — but the backfill on a read-only workspace must suppress the upload (a genuinely local-only write path), or it enqueues one quarantined reject per tagged block. This is a real corner needing a local-only write path, not the "harmlessly local" hand-wave the previous draft claimed; it is also moot for this fleet, which has no read-only shared workspaces today, so the path is specified-and-deferred. The only residual is a workspace no device ever opens again post-ship, whose legacy panels stay untagged; that workspace is also not generating new UI-state, and the first open backfills it (acceptable for this fleet, review finding). Two reasons this is the right signal, and both answer the review finding that keying on panel was unsafe:
    • It is explicit, so user-editability is a feature, not a hazard. The danger in keying on panel was implicitness — types live in user-editable properties_json.$.types (blockType.ts: even structural types stay "removable in the property panel"), so a content block tagged panel would have lost its history silently, with nothing in the type name warning of it. A type called no-history inverts that: applying it is a deliberate, legible choice a user is welcome to make on their own scratch blocks, and losing history is the thing they asked for — not a surprise. Mis-tagging panel is now harmless, because panel is no longer the key.
    • It rides sync and is stable enough. The type is part of the block's replicated properties, so it is present and identical on every device (the completeness the scope key lacked). No new synced column, no sync-config.yaml change, no migration on the hot blocks table. The sweeper doesn't scan the type at all: the trigger denormalizes it onto row_events.no_history at write time, and a partial index over that flag makes "find the no-history blocks to prune" a cheap local lookup that also reaches deleted blocks (below).
    The residual data-loss surface is a genuinely churning block (§below: only blocks with > K events are ever touched) that a user explicitly marks no-history and later regrets — the same shape as any deliberate destructive action, and far narrower than an accidental structural tag. No paternalistic guard second-guesses the mark; the explicit type is the safety.
  • The sweeper keys on a denormalized no_history flag the trigger stamps onto each event — one indexed query, uniform across every deletion state. The block's no-history type stays the authoritative, synced, user-editable signal (above); the row-event trigger copies it onto a row_events.no_history column (0/1) at write time — a JSON check on the NEW row the trigger already holds, and it fires for sync-applied writes too, so a panel that synced in from another device is flagged identically to a locally-created one (the exact completeness the scope key lacked — scope is NULL on sync). This is a derived cache of the block type, not a second source of truth: the type decides, the flag indexes. "Cheap" is an assumption the bench must check, not a given (review finding): after slice B every row-event trigger extracts from properties_json to compute this flag — including content-or-reference-only updates, which slice A's compact format otherwise lets skip the properties blob entirely. On a block with a large properties value that is a per-write tail cost on the hottest path, so it belongs in I6's final-index-set benchmark alongside the index maintenance, not in prose. If it does bind, the escape is the same change §13's open problem already wants for soundness: a persisted provenance column is a column read rather than a JSON parse, so it would fix the gate and this write-tail cost together.
    • Candidate discovery = one partial-index scan, all deletion states at once. A deep-idle job (same PendingIdleJobs machinery as §8) reads the blocks carrying any flagged event — SELECT DISTINCT block_id FROM row_events WHERE no_history = 1, served by a partial index (no_history, block_id, id) WHERE no_history = 1 (slice B). Because the flag lives on the event, this reaches live, soft-deleted, and hard-deleted blocks in one query — a hard-deleted block's events still carry the flag. That retires the whole three-source enumeration the earlier draft needed (live via block_types, soft-deleted via a delete-surviving side index, hard-deleted via a delete-event high-watermark) and, with it, the watermark's advance-after-prune atomicity concern — every review finding against the enumeration, dissolved rather than patched. The partial index holds only flagged rows, so this is not the rejected "blocks with > K events" full aggregate (a full scan of the whole table); it touches only the no-history partition. Union'd with the live type index as a safety net (review finding): the flag partition alone is not a complete discovery source, because a block tagged during a downgrade window (below) has every event defaulted to no_history = 0 and so never appears in the scan at all — the status fallback can't rescue a block it never discovered. So discovery is the flag partition ∪ the live block_types lookup for type = 'no-history' (already indexed, clientSchema.ts:861-865). The two are complementary and each is a cheap indexed read: block_types covers live tagged blocks whatever their events look like, the flag partition covers deleted ones (which block_types drops). This is not a return to the three-source enumeration — there is no delete-surviving side table and no delete-event watermark; it is one extra indexed lookup. Plus the structural identity scan, which this union previously omitted (review finding): in the old-creator window a pre-B client can create and keep churning a live computed UI-state row — a plugin or layout-session child — that has no no_history events and no no-history type, and active-workspace tagging may never run for a synced-but-unopened workspace. Neither the flag partition nor block_types would ever enqueue it, so an implementation copying just those two would leave a hot live row growing forever. The union is therefore flag partition ∪ live block_types ∪ computed identity (§13's gate set, derived not traversed), which is the same signal B1 already needs to authorize the prune. One residual hole, and the ordering requirement does not close it (review finding — an earlier draft claimed it did): shipping the trigger-version guard before slice B only binds clients that have the guard. A client older than that release runs the unconditional DROP+CREATE expansion (clientSchema.ts:1168-1175) and will never consult a body-version marker — you cannot retroactively make old code check something. The surviving failure is narrow but real: a block tagged while a stale body is installed (so none of its events carry the flag) that is then deleted (so block_types drops it) is invisible to both discovery paths, permanently. The closer is a bounded backstop, not a permanent side index: a reconciliation scan of tombstones — the 20,825-row scan §5.1 refuses to run every deep idle, affordable occasionally. Armed by evidence of a downgrade, not by a version bump (review finding): "once, on the first boot whose version exceeds the recorded one" misses the case it exists for — an upgraded client boots and scans, then a pre-guard tab opens, reinstalls the old body, and deletes a tagged block; that tombstone is created after the only scan and no later version transition rescans it. Instead, every boot probes the installed trigger body (SELECT sql FROM sqlite_master WHERE name = 'blocks_row_event_update') for no_history capability. A body lacking it is direct evidence that a pre-guard client ran since last boot — the probe does not depend on old clients cooperating with a marker they don't know about — so the client reinstalls the current body and runs the reconciliation. Cost stays bounded because the probe is one row read and the scan only follows an actual regression. Soft-deleted stale-body blocks fall between both paths (review finding): block_types is not a live+soft index — its update trigger deletes a block's entries and reinserts only when NEW.deleted = 0 (clientSchema.ts:881-885) — and a stale body wrote no no_history flags, so the partial index is empty for that block too. The block still exists, so the hard-tombstone scan does not enumerate it either. A user-tagged block soft-deleted during the old-writer window is therefore invisible to the whole union. Closing it costs one more read in the same arming window: the reconciliation also scans blocks WHERE deleted = 1 for the authoritative type. That is the tombstone-population scan §5.1 refuses to run every deep idle — affordable here for the same reason as the hard-deleted scan, because it is armed by evidence rather than scheduled. And a downgrade is not the only arming condition (review finding): a post-guard but pre-B client leaves the current trigger body untouched — so the probe sees nothing wrong — while still creating UI-state rows without the no-history type, because it predates the stamping. If it creates, churns and hard-deletes such a row before the recurring subtree pass sees it, there is no flag, no type, and no downgrade evidence. So the structural scan stays armed through the whole old-creator window, not merely after a body regression; both conditions retire together once no pre-B client remains. Ordering is load-bearing: the probe must run before the schema bootstrap recreates triggers (review finding) — that bootstrap expands to DROP TRIGGER IF EXISTS + CREATE TRIGGER unconditionally on every boot (clientSchema.ts:1170-1174), so recreating first would overwrite the very evidence the probe looks for: the client would see a current body, conclude nothing happened, skip the reconciliation, and leave the tombstone undiscoverable. Probe, then recreate. And the scan is structural, not type-keyed (review finding): it also collects tombstones identified by computed identity regardless of type or flag — delete events are full=1, so the snapshot carries both the block's own id and its parentId. That is the only thing that reaches a UI-state block an old client created untagged, churned, and hard-deleted before any upgraded reconciliation saw it: no blocks row for structural discovery, no type or flag for a type-keyed scan, and otherwise invisible forever. Three tiers, in order of certainty (review finding — a single parentId is not sufficient for nested rows, since layout-session descendants nest through ui-state/layout-sessions/{id} and panels may sit under panel-stack rows). But these tiers must be split by what they are allowed to do (review finding): ancestry may discover a candidate; only unforgeable identity may authorize the destructive prune. Treating "a parentId in the computed set" as decisive would reintroduce the exact mutable-parentage gate §13 rejects — a content block can be moved under the ui-state subtree with tx.move and then hard-deleted, leaving a delete snapshot with that parent, and B1 would classify it as dedicated UI-state and prune undoable history. So: (1) the tombstone's own id is in the computed deterministic set — authorizes the prune, no ancestry needed; (2) its parentId (or a chain walked up through surviving rows and other tombstones' snapshots) reaches the computed set — enqueues it as a candidate, but is a clue, not a certificate; on its own it does not authorize deletion. Consequence, named rather than hidden: a tombstone whose own id is a runtime id — every panel and panel-stack row — has no authorizing signal in B1, so B1 cannot prune it. That is the panel problem (§13) reappearing on the tombstone path, and it is resolved by the same thing: persisted provenance, or an accepted weaker gate. Until then those histories stay unbounded, along with the narrower residual where an old client hard-deleted an entire runtime-id chain so no link survives at all.
    • The prune decision reads the authoritative type, never an event flag. The flag's whole job is discovery; once a candidate is in hand, whether to prune it is answered from the type itself: the blocks row's properties_json.types for a live or soft-deleted block (it is right there), and the delete event's before_json snapshot for a hard-deleted one (delete events are full=1, so they carry properties; and the cut always keeps the newest K, so a hard-deleted block's delete event is never pruned away). Currently typed ⇒ prune its whole history to newest-K — all events, flagged or not (e.g. a create written before an addTypeInTx tag). Computed identity authorizes a prune on its own, without the type (review finding): the structural scan deliberately collects UI-state rows an old client created untyped, so a type-only decision would find them and then skip them, leaving exactly the histories the scan exists to repair unbounded. This is not a loophole — a computed id (stateChildBlockId/userPageBlockId) is a stronger signal than the type: it proves the block is dedicated UI-state storage by construction, whereas the type is a user-editable assertion that it is. So: typed or identity-confirmed ⇒ prune — but only within whatever B1 gate has been resolved (review finding): stated unconditionally, this rule would let an implementer delete undoable content history before that decision is made, either for a user-tagged content block or for a squatted computed id. The gate (§13) is what narrows "identity-confirmed" to something unforgeable; until it is chosen, this rule authorizes nothing on blocks outside dedicated UI-state storage. Otherwise ⇒ skip; already-pruned events stay gone (retention is one-way), nothing further is cut. In B1 the structural gate applies to tombstones too (review finding): "typed ⇒ prune" read alone would let a user apply the (visible, §13) type to an ordinary content block, hard-delete it, and have B1 prune undoable history — exactly what the B1/B2 split forbids. Since the delete snapshot carries parentId, the same dedicated-UI-state confirmation is available for a tombstone and is required before pruning it; a hard-deleted candidate that cannot be structurally confirmed is skipped until B2 resolves O7. After B2 the type alone suffices, which is precisely what B2 unlocks. The flag is therefore write-time-only and advisory: no retroactive backfill, so I7 keeps its two exceptions and event payloads stay immutable. This is deliberately not "status = the latest event's flag", which two review rounds killed: that rule made correctness depend on which trigger body happened to write the newest row. A stale tab is the hazard — boot-time withTriggerRecreate reinstalls every body from whichever client booted last, so an older tab rebooting after slice B can reinstall a body with no no_history in its INSERT list (clientSchema.ts:450-463), defaulting the flag to 0 on a tagged panel. Worse, a slice-A body writes v = 2 rows that still predate the column, so even "the latest v = 2 event" is not a capability test (review finding) — the format version and the column's existence are independent. Reading the type instead sidesteps the entire question: no capability marker, no trigger-body version in the correctness path, and immune to any future column-ordering surprise. The version guard (slice A/B) is still worth having — it protects every v2 stamp — but retention no longer depends on it.
    Deleted blocks are pruned deliberately: a panel deleted before a sweep capped it would otherwise keep its whole pre-delete history (up to the full delete-before snapshot, multi-MB) forever — and the event flag reaches them where a live-type lookup could not. For each pruned candidate it then deletes all but its newest K row_events ordered by id, not created_at: id is the authoritative per-block order (§3), and because created_at can tie within a tx or step under wall-clock skew, a cut by timestamp could retain an earlier-id event while deleting a later one — holing a survivor's backward walk, the exact corruption this is meant to avoid. The cut is simply id < (K-th largest id for the block) — delete the oldest prefix, keep the newest K. No wall-clamp and no terminal-gap reconciliation (simplified 2026-07-18, I9): a prefix delete is safe for compact events because every survivor is newer than everything deleted and reconstructs backward from the live row (§4.3), and that backward walk is unconditionally valid for content/structure since the only unlogged gap is metadata-only. An earlier draft clamped the cut at max(K-th id, latest full=2 wall) (with a COALESCE for SQLite's max(x,NULL)=NULL) and wrote a synthetic full=2 reconciliation anchor to close a "terminal gap" left by the user_updated_at backfill; with walls retired and that backfill known metadata-only, the survivors' live-row base is always content-valid, so both mechanisms are unnecessary. This also retires the sweeper's one sanctioned row_events insert — it now only deletes (I7). Discovery uses the partial (no_history, block_id, id) index; the per-candidate count and cut use the full (block_id, id) index (the cut deletes flagged and unflagged rows of the whole block, and the latest event may be unflagged, so the partial index alone won't serve them — §3). Not in-trigger: pruning from the trigger would put a read + DELETE on the hottest write path (rejected, §15) — the flag set is a single column write, the DELETE stays at idle. The bound is eventual, enforced at idle. It shares §8's machinery but not its lifecycle (review finding): §8 is a one-shot that flips v=2 and never runs again, whereas the retention sweeper is recurring — it re-arms every deep idle, because churn keeps accumulating; there is no completion marker to set.
  • Whole-block newest-K is a clean oldest-prefix delete — safe for compact events, so no full-writes are needed. Keeping the newest K deletes a contiguous oldest prefix of the block's events. Every survivor is therefore newer than everything deleted, and reconstructing a survivor walks backward from the live row (§4.3), touching only the newest K events — it never crosses the pruned region. The one corruption case §4.3 warns of — a walk crossing a hole and silently splicing — requires a surviving event older than the deleted region, which keep-newest-K has none of by construction. So UiState events are stored compact like everything else; the earlier "write them full so pruning is order-free" machinery is unnecessary, because the sweeper never deletes from a middle, only a prefix. (Forfeiting the pruned states' own reconstructability is the point of retention; forward-walk-from-create breaks for that block with its create, which is the named I8 UI-state exception.)
  • What this deletes, owned honestly. The sweep removes every old event on a flagged block, sync-authored ones included — not just locally-scoped writes. That is intended: cross-device panel view-state is the same disposable churn as local view-state, and "keep a few recent versions" wants both bounded. It is also what makes the bound complete rather than partial — no interleaved event class can freeze the cut, because the sweeper does not distinguish classes on a flagged block. This is a real behaviour change from the scope-keyed draft, which preserved sync-authored panel events; naming it so it can be vetoed. I7 is amended to match: the sweeper is the second sanctioned row_events mutation (with the §8 rewrite), and it may delete any event on a no-history-typed or identity-confirmed block keeping the newest K — never any event on a block that is neither (review finding — this previously said "without that type", which forbids the very backstop §5.1/I7 now require: an old client can create a UI-state row untyped, churn it and delete it, and the structural scan reaches it only by computed identity).
  • Pruning an undoable no-history block has an open undo-boundary cost. [1000] lets a user tag their own ordinary (undoable-scope) block no-history, not just non-undoable UI-state — so the sweeper can fully delete the row_events of a tx whose only write was that block, group_id included (group_id lives on row_events, never on command_events). A future undo-from-history reader (§10) must not let that erase a tx boundary and merge two grouped actions on other blocks across the gap. An earlier draft claimed command_events alone closes this; two review findings showed it does not (§10): the ledger has no DB-wide monotonic order (a same-millisecond boundary tx can be misplaced and cause a wrong merge) and can't distinguish a fully-pruned writing tx from a harmless zero-write one. Closing it needs durable per-tx state (persisted four columns on command_events: a SQLite-allocated sequence — not the session-seeded tx_seq — plus group_id, a wrote-a-row flag, and is_replay) — provisioning an unbuilt reader (I6) — or restricting what the sweeper will act on to dedicated non-undoable storage (a sweeper-side check, not a scope filter and not a tagging policy — §16 O7(b)), or deferring §10's guarantee. This is now O7, open. Note it does not touch the dominant case: UI-state blocks are non-undoable (ChangeScope.UiState), so pruning them — the entire 35 MB motivation — has zero undo interaction; the gap is only the user-marked-undoable-block corner [1000] opened.
  • Scope is kept, but only as provenance. The scope stamp (§3) still rides on each event for forensics and for the #210 provenance story — "what edit scope produced this" — but it no longer decides deletion, so the two uses fully decouple. Its source-gating (NULL on sync/raw) is now honest provenance rather than a correctness gate: a mislabeled scope is wrong provenance, not deletable content history. Retention correctness rests entirely on the block-type facet.
  • Stock churn is reached with no special migration handling. The §8 rewriter compacts a panel's old events like any other block's — no "keep local-ui full" carve-out — and once migrated, the sweeper prunes the panel to newest-K on its type alone. The measured 14.7k-event panel is bounded on the first sweep after migration, not only for future traffic. Pruned bytes return at VACUUM (§9); the point is bounding growth, not reclaiming it.
  • What this does not bound — UI-state props on content blocks (review finding, scope honestly narrowed). ChangeScope.UiState is a scope, not a set of blocks: some UiState writes land on ordinary content blocks, not on dedicated UI-state storage. Toggling the properties panel writes showProperties — a ChangeScope.UiState property (properties.ts:30-35) — straight onto the target content block (blockActions.ts:321-322). Whole-block no-history cannot prune these: the block also holds real content history, and the sweeper only ever deletes a block's whole oldest prefix, never a scope-subset of its events (scope is NULL on sync anyway — the reason keying moved to the block). So the honest guarantee is "bounds the dedicated UI-state storage blocks" (panels, layout-sessions, panel-stack, plugin UI-state — where the 14.7k flagship and every top-churn block on prod live), not "bounds all UiState-scoped churn". Measured residual on prod: 315 content blocks carry showProperties, 9,450 row_events among them, max 7,101 on one — smaller than the panel class but not nil, and genuinely outside this mechanism. The clean fix is a data-model one (out of scope here, flagged as follow-up): move UI-state properties off content blocks onto per-(block,user) UI-state storage that is no-history-typed and prunable, or bound those specific properties another way. Until then this is a named, measured limitation, not a silent one.

6. Sync-source logging: kept for every workspace (revised)

Revision note (2026-07-15, user call): an earlier draft of this section dropped sync-applied events for plaintext workspaces on the strength of the server-side redundancy analyzed below. The user overrode it — sync events are kept, so local history stays self-sufficient. §6.1 is retained as the record of what the server does and doesn't cover; §6.2 keeps the one mechanism that survives the revision (the sync-apply marker, now labeling-only).

6.1 The server-side redundancy analysis (kept for the record)

Position examined: sync-applied events are redundant for plaintext workspaces because the server-side blocks_history table already records those changes. Read against supabase/migrations/20260522062437_add_blocks_history.sql:

  • All columns, all writers — the trigger fires AFTER INSERT OR UPDATE OR DELETE ON public.blocks FOR EACH ROW (:167-170) and snapshots to_jsonb(NEW/OLD) (:93, :112-113), so every server-side write path — PowerSync CRUD-apply, RPCs, manual SQL — is captured, every column included. Shape is the same "changed-columns diff, full bookends" family as this design (:18-24). ✅
  • Granularity caveat — resolution is per uploaded write, and the client compacts per-id before upload (compactBlockCrudEntries, src/services/powersync.ts:211, used at :638/:672; the migration's own header says so, :5-10). But the local source='sync' events are also collapsed — PowerSync delivers checkpoint states, and the observer additionally skips no-op re-deliveries (materialize.ts:56-60). Server history is at least as fine as what any receiving device could log. ✅ (Per-edit fidelity exists only in the authoring device's source='user' events — unaffected by this section.)
  • E2EE caveat — for e2ee workspaces the server sees and records ciphertext envelopes. The history bytes are as durable as any e2ee data (server-side, ciphertext); what a key-holding device's local events provide is the only readable-today plaintext copy — an access gap, not a durability gap. ❌ redundancy does not hold at the access level. (An earlier draft called this "local-only e2ee history" / "per-device durability only" — wrong framing, corrected per user.)
  • Availability caveat — blocks_history has no client read path yet: the migration header defers read/restore RPCs to a follow-up (:2-3), RLS grants SELECT to workspace members (:177-192), and no later migration in supabase/migrations/ adds an RPC (verified against the full listing). The redundancy is "the bytes exist server-side", not "a reader exists today". This was open question O1; keeping sync events (the decision below) resolves it — local history no longer depends on the missing RPC.
  • What is genuinely lost — the local-apply record: when THIS device materialized a change and what it saw. LWW skip decisions are recorded nowhere either way (skippedStale rows never fire triggers), so the marginal forensic loss is apply-timing only.
Decision (revised 2026-07-15, user call) — sync-applied events are kept, for every workspace, in compact v2 form with anchors like any other events. Rationale: local history should be self-sufficient — readable offline, without a server round-trip, and not gated on a blocks_history read RPC that doesn't exist yet (former O1, now moot). The server-side redundancy above is real at the byte level but is an access story, not a local-history substitute.
Consequences owned:
  • Fresh-device first sync logs ~345k full creates (~205 MB/device, measured — §12) — creates are inherently full, v2 cannot shrink them. This was the second-largest byte win of the earlier draft; deliberately given back.
  • #288's backfill arrival burst logs on receiving devices (~650k full creates ≈ ~230 MB one-time per device). With the §5 revision the authoring device pays the same, so this stopped being an asymmetry to negotiate — G2 in §11 is now simply priced, with no exemption offered or needed.
  • The comment at clientSchema.ts:55-69 ("the ONLY place an INCOMING sync change is durably recorded") holds for every incoming sync change that materializes — but a delivery that loses the LWW gate (skippedStale returns before Phase 2) writes no blocks row and fires no trigger, so it is an incoming delivery that is deliberately not recorded (review finding). That is consistent, not a gap — row_events logs state transitions, and a skipped-stale delivery changes no local state (§6.1 already notes skip decisions are recorded nowhere either way) — but the word "unconditionally" would overclaim to a future forensics reader treating row_events as a complete incoming-sync audit. So slice B narrows the comment's wording to "materialized incoming sync change." No mechanism change; a one-line comment tightening.
  • Raw out-of-band local writes (agent-bridge sql execute, dev tools — a live path, §2.1) also keep logging, labeled truthfully via the §6.2 marker — a raw write never uploads, so its row_event is the only record of it anywhere.

6.2 Mechanism: the sync-apply marker (labeling only)

With no drop policy, the workspace_history_policy side-table, the materializer's Phase-2 policy writes, and the trigger drop gates of the earlier draft are all deleted from the design. What survives is the marker that distinguishes sync-apply from the other NULL-source writers (the raw path, §2.1) — now purely for truthful attribution, no longer load-bearing for any gate: the materializer's Phase-2 bracket, which today sets source = NULL (materialize.ts:344), additionally sets a new nullable tx_context.apply_kind column — added by an idempotent ALTER TABLE ensure-migration ordered before any writer touches it, not merely an edit to the CREATE TABLE IF NOT EXISTS body (which never runs once the table exists); ensureUndoGroupIdColumns already ALTERs tx_context and is the exact precedent, or the materializer's first UPDATE tx_context SET apply_kind = ? hits no such column — set to 'sync'. Unlike today's bracket — which only opens (it asserts source = NULL and relies on NULL being the resting state) — the clear must be explicit: reset apply_kind = NULL as the last statement of the same write transaction, so a rollback restores it for free and a crash cannot leave a sticky 'sync' mislabeling later raw writes (I3's labeling test pins this). The upload and workspace-invariant trigger gates key on source and are untouched (they must keep skipping sync-apply); only the row_events triggers' label projection reads apply_kind: the v1 COALESCE(source, 'sync') (clientSchema.ts:422) retires in favor of a truthful three-way source tag — 'user' (repo.tx), 'raw' (positive apply_kind = 'raw' marker — the unguarded paths, always logged, I3), and 'sync' for everything else with a NULL source, marker or not. Stating it in that order matters (review finding): defining 'raw' as "neither source nor marker" reads naturally but is wrong, because a pre-B materializer writes exactly source = NULL with no apply_kind — so "neither" is the sync case during any mixed-version window, and an implementer copying that phrasing would mislabel every old-client server change. The NULL/NULL default must stay 'sync', not 'raw' (review finding): in a mixed-version session a B client installs this trigger while an older tab still runs the current materializer, which sets only source = NULL and never apply_kind (materialize.ts:341-345). Falling through to 'raw' would permanently mislabel every materialized server change until the last writer upgrades — the common case losing to the rare one. So 'raw' requires positive evidence (a marker written by a raw path), and an unmarked NULL-source write keeps the historical 'sync' reading. Which means the raw paths must actually set that marker, or the 'raw' label never appears (review finding): the known raw writer is a direct repo.db.execute (agent-runtime/commands.ts:170-172), and with the default flipped to 'sync' an unbracketed raw write is now indistinguishable from a sync apply — the I3 promise that raw writes are labeled truthfully would be vacuous. Slice B therefore brackets those paths with apply_kind = 'raw' (same set/clear discipline as the sync marker: cleared in the same transaction, so a crash can't leave it sticky). Bracket every mutating SQL mode, not just execute (review finding): the bridge exposes all|get|optional|execute (agent-cli/src/protocol.ts:106) and dispatches the non-execute modes to repo.db.get/getAll (agent-runtime/commands.ts:168-173) — its own read-only classifier already treats a mutating statement in those modes as a write (agent-cli/src/server.ts:304-310). So sql all "UPDATE … RETURNING" is a raw block write that a execute-only bracket would miss, leaving it mislabeled 'sync'. Bracket on mutation, not on mode name — or restrict mutations to execute and reject them elsewhere. The set is small and enumerable, which is what makes this tractable; any new raw path must bracket itself, and that is the I3 review rule. Residual, named: during that window a genuine raw write from a pre-B client reads as 'sync'. That is provenance imprecision in the direction of the truth-preserving default, and it gates nothing — source/scope are provenance only, never a delete or upload gate (§3). Existing v1 rows keep their historical 'sync' ambiguity — and stock rows also carry legacy source strings from retired TxSource values (the dev instance holds ~11k 'local-ephemeral' events, §12), so readers must treat source as an open set, not a closed three-value enum; the three-way tag governs new writes only.

Two review lessons from the deleted drop gate stay on the books as review rules (they bind the label projection and any future trigger predicate — the history_mode gate they were first written for is itself gone with the §5 revision): every WHEN predicate must be NULL-safe under SQLite's three-valued logic — a bare (subquery) = 'x' against a missing row yields NULL, NOT (TRUE AND NULL) is still NULL, and a NULL WHEN silently skips the INSERT, i.e. silently drops history (hence IS / COALESCE, as §5.1 already does) — and row references are per-trigger: blocks_row_event_delete is AFTER DELETE (clientSchema.ts:471-488) and has no NEW row, so any predicate shared across the three triggers must read NEW in insert/update and OLD in delete.

8. Migrating the existing ~1.9M events

Required, not optional: the stock of old full-snapshot events is where the multi-GB sits. Direction fixed by the frame: rewrite old events into compact form, sampling old before_json values into anchors rather than dropping them wholesale; delete nothing.

8.1 The rewrite, per block

Unit of work: one block's v1 events, in id order (per-block processing keeps the running reconstruction state at one row, ~2 KB — processing in global id order would need a 350k-entry state map). For each block with any v IS NULL event:

  1. Stream all its events ascending — not only the v1 ones (review finding). Rewrite only the v1 rows, but advance runningState through every event, v2 included, because v1 and v2 can interleave within one block (§8.2: a stale tab that reinstalls v1 trigger bodies after slice A resumes writing v1-shaped rows alongside earlier v2 ones). A v1-only stream would compare one v1 event's before_json against a runningState that never saw the normal, fully-logged v2 edit in between, so that edit would read as a content-level divergence — i.e. a false I9 violation, which under the detection rule below could wrongly wall or block the migration. Maintain runningState (the reconstructed after of the previous event, kept or not) and pos (compact count since the last full event — a rewriter-local counter; nothing in the output encodes it; already-v2 rows advance it according to their own full value, so cadence stays consistent across the interleave). Initialization, because the stream can open with a compact v2 row (review finding — distinct from the interleave case above, where a v1 full precedes it): a block that predates row_events entirely has no create, so slice A logs its first-ever edit as a compact patch (§4.1/I6: the v2 trigger has no predecessor probe), and a stale tab can then append the first v IS NULL row after it. The stream therefore starts with a patch and no base to apply it to. Rule: seed runningState from an older-side full state if one exists (forward-walk from it, which cannot cross a newer gap) and proceed normally. Where no older-side base exists, the seam is unclassifiable and the block takes the conservative all-full path below — it must not be seeded from the newer side. Two review findings landed here in sequence and the second corrects the first. Leaving runningState unset and skipping classification hides the gap, which is why an earlier draft made newer-side seeding mandatory. But newer-side seeding is worse, not better: reconstructing that point walks backward from the live row through the very gap being tested for, so the seed arrives already carrying post-gap values in the unchanged fields. The later before_json ≠ runningState comparison then comes out equal and reports nothing — the seam is laundered by the check meant to expose it. An older-side base is safe because the gap lies newer than it; a newer-side one is not, for exactly the same reason. When neither exists there is no honest way to classify the seam, so the block is migrated with every event full, none compact — the same resolution as a detected content divergence, and for the same reason: a full event never walks.
  2. create: stamp v=2, full=1, payload unchanged. Reset pos. (A create opens a chain — nothing logged is older.)
  3. delete: stamp v=2, full=1, payload unchanged (a hard delete carries a full before by nature). Run the same field-wise divergence classification as the update branch (review finding — an earlier draft exempted deletes on the assumption that any pre-purge divergence is metadata-only, which is precisely the assumption I9 no longer makes unconditionally): if the first logged event after a pre-A unlogged content write happens to be a hard delete, treating its full before as a crossable newer-side base lets older stateAt walks splice post-gap content into pre-gap history — the same corruption the update branch now stops, arriving through the delete door. So: metadata-only divergence ⇒ the delete's before is a crossable base as usual; content-level divergence ⇒ migrate the block conservatively (every event full, none compact) and report — the same resolution as the update branch below, which explains why neither a marker nor an abort suffices and why a full event is the thing that actually cannot splice. Reset pos.
  4. update/soft-delete: compute changed fields by comparing the event's own before/after JSON (exact — no cross-event inference), then keep it full=1 if either of the following (there is no full=2 — discontinuity walls are retired, I9/§4.3):
    • cadence — pos ≥ 63 (deterministic every-64th in the rewriter — it runs offline, no coin needed, and readers key on full, never on spacing). Crossable base. Reset pos = 0 after emitting it (review finding): the counter's contract is "compact events since the last full event", so every full event resets it — create, delete, this cadence anchor, and the no-runningState chain start — while each compact rewrite increments it. Stating the reset only for create/delete is the bug: a literal implementation leaves pos ≥ 63 after the first cadence hit, so every subsequent update on that block is kept full and the compaction is wiped out for exactly the hot histories this migration targets.
    • no runningState — the block's history doesn't open with a create (pre-log era). A chain start: nothing logged is older, so there is nothing to splice across.
    A before_json ≠ runningState divergence (an unlogged gap precedes this event) must be classified, not assumed. The expected case is metadata-only and needs no wall: git history confirms the only write path that ever suspended the trigger is the one-time user_updated_at backfill (commit ef4862550; the sole withTriggerSuspended UPDATE across all history is SET user_updated_at = updated_at), which is content-continuous (I9). Such an event rewrites compact like any other — the rewriter may emit full=1 at the seam as a courtesy checkpoint, but it is a crossable anchor, never a wall. A content-level divergence is a different animal and must not be compacted through (review finding — an earlier draft said "any divergence is metadata-only", which contradicts the very detection I9 relies on §8.1 to perform): it is positive evidence that some write bypassed the trigger, most plausibly a pre-A mixed-version trigger-recreation window (I9's stated precondition). Compacting it away would let a later stateAt walk cross the seam and splice post-gap values into pre-gap history — silently, and exactly the corruption walls were retired on the premise that it cannot happen. So the rewriter compares the divergence field-wise: metadata-only ⇒ proceed; any content or structure field ⇒ migrate the block conservatively: every event full, none compact, then report. Neither a marker nor an abort works here (two review findings, the second correcting the first). Marking the seam non-crossable is not available: full=2 was retired and there is nothing to mark with. But aborting — "leave the block wholly v1" — does not hold either, because a block can already carry compact v2 rows written live by slice A before the later v1 row exposes the gap; a future stateAt for one of those could still pick the v1 row's full before_json as its newer-side base and splice straight across the gap. Abort cannot un-write those rows. The property that actually delivers safety is simpler than either: a full event never walks. So the rewriter makes the whole block walk-free — a v1 row is already a full snapshot, so it migrates trivially (stamp v = 2, full = 1, payload untouched, no reconstruction needed); every compact v2 row is promoted to full, each reconstructed from whichever side does not cross the gap (§4.3): rows newer than the gap from the live row backward, rows older than it forward from the nearest older full. Neither walk crosses the seam, and once every event is self-contained nothing ever needs to. The gap becomes inert rather than guarded. This also keeps the block migrated — v = 2 is stamped, so it leaves the work queue normally and the completion marker stays reachable (review finding: an abort would have left it permanently claimable under WHERE v IS NULL, either looping forever or pressuring an implementer to compact through the gap to make progress). The only thing forfeited is compaction for that block. Expected count is zero — it means I9's fleet precondition was violated — so it is reported rather than silently handled, and a rare block keeping full events costs nothing measurable. If some compact row has no valid base on either side, that single event is unreconstructable (a pre-existing condition, not one this creates): report it specifically rather than guessing. Otherwise rewrite compact: before_json = changed fields' old values, after_json = changed fields' new values (equal key sets — the invertible-patch shape of §3), v=2, full=0, and pos += 1. An all-fields-identical v1 update (the v1 trigger logged no-ops) becomes an empty compact event — kept, ~60 B, preserving "an UPDATE happened" timing (it still counts toward pos, being a compact row).
  5. Stamp workspace_id while rewriting — from whichever snapshot side is non-null: COALESCE(after_json ->> 'workspaceId', before_json ->> 'workspaceId'). Not "both sides": a create carries no before_json (measured — all 345,497 creates on the prod instance) and a hard delete no after_json (§4.3). The two sides can never disagree where both exist, because workspace_id is immutable by contract (txEngine.ts:842; a cross-workspace write throws DeterministicIdCrossWorkspaceError), so the COALESCE resolves an absence, never a conflict. This is not optional bookkeeping: ALTER TABLE … ADD COLUMN workspace_id initialises the entire v1 stock to NULL, so without this step the column is populated for new events only, and a workspace-scoped recovery over the pre-migration era — the exact era most likely to need one — would find nothing, reintroducing the problem the column exists to solve. The rewriter already holds the full snapshot in hand, so the stamp is free; it must land before the row's v flips to 2, since after the flip a compacted event may no longer carry workspaceId in its payload to recover it from. Rows the rewriter never reaches keep workspace_id IS NULL, which readers treat as "unknown, fall back to the payload/join" rather than as "no workspace".
  6. Stamp scope while rewriting: any v1 event whose tx_id joins command_events backfills scope from it (command_events has carried scope NOT NULL keyed by tx_id since before both measured instances' DBs; commitPipeline.ts:348-368; no delete path exists). Key on the join, not on source = 'user' — v1 source labels are an open set. Sync-applied events carry no tx_id and correctly stay NULL. This is provenance backfill only — v1 UI-state rows are compacted like any other block's, with no "keep local-ui full" carve-out: the §5.1 bound keys on the panel block's type, so it reaches the measured 14.7k-event panel on the first sweep after migration regardless of how those events are encoded. One count to run at implementation: v1 tx-carrying events with no command_events row would indicate a decoupled era and quantify the backfill's misses.
  7. Mark the batch done by the rewrite itself: v flips 2.
The migration is reconstruction-lossless (invariant I5): for every event e, stateAt(e) returns the same value before and after the rewrite — the only bytes dropped are ones derivable from neighboring events, and every non-derivable before is retained as an anchor. Pinned by an equivalence test: seed a DB with randomized edit histories (including deliberate gaps), snapshot stateAt for every event, migrate, compare — comparing content/structure, not raw bytes, across the one metadata gap (review finding, matching I5's own narrowing): the seeded histories include the trigger-suspended user_updated_at backfill, and after walls were retired a pre-gap event reconstructs its metadata from the post-backfill base. A full-row byte comparison would therefore fail a correct no-wall implementation, or push it to reintroduce markers purely to satisfy an assertion about fields no reader reconstructs by walking.

8.2 Resumability, scheduling, coexistence

  • Backup before the first destructive chunk (user call 2026-07-15). The rewrite is the one sanctioned mutation of history (I7) and it runs in place on live user data — so on each device the rewriter arms only after the user has been prompted to take a full database backup: the existing export machinery (src/utils/exportSqliteDb.ts — the same flow PR #281's corruption recovery uses for its pre-reset download), surfaced via the db-maintenance command. Acknowledgment (or explicit decline) is recorded in client_schema_state (row_events_migration_backup_ack); the idle job checks it before claiming its first block. I5's equivalence tests make the rewrite safe in expectation; the backup makes it safe in fact. The export is multi-GB on the big instances — hence prompted, never silent.
  • Progress marker is the format tag. WHERE v IS NULL is the work queue; a partial index ON row_events(block_id) WHERE v IS NULL makes claiming the next blocks cheap and shrinks to nothing as the job completes (dropped at the end). Interruption at any point (crash, tab close) is safe: rewritten rows are self-describingly done, unrewritten rows self-describingly not — no cursor to corrupt. A client_schema_state marker (row_events_compact_migration_v1) records only overall completion, gating the §9 VACUUM and any §3 index revisit.
  • Chunking: a few hundred blocks (or ~2–5k events) per write tx, scheduled via PendingIdleJobs + scheduleDeepIdle (idleMarkerJobs.ts) like the other catch-up passes — never on the boot or first-paint path. Each chunk holds the single writer briefly; live writes queue behind it, as they do behind any tx.
  • Live-write coexistence is free by construction: v2 events never reference predecessors at all (§4.1), so the rewriter may compact any v1 event without coordination — and I5 preserves stateAt for every event, so a reader walking across the v1/v2 boundary mid-migration stays sound. Do not assume v1 events stop at slice A: trigger bodies are shared DB state recreated at every boot from the booting code's constants (clientSchema.ts:1170-1175), so a stale tab on old code that re-boots reinstalls v1 bodies until the next new-code boot — v1 and v2 events can interleave within a block. The mechanism doesn't care (each v1 event rewrites from its own before/after, independent of any interleave), but two consequences are owned: the completion marker means "work queue observed empty once", not "no v1 rows can exist" (stragglers stay v-NULL = readable-as-full, merely uncompacted — permanently, unless the job is deliberately re-armed; the marker-gated one-shot does not re-scan on its own); and during a downgrade window the v1 bodies carry no identical-update skip, no workspace_id, and no scope stamp, so a stale tab's writes land as v IS NULL full rows until that tab reboots on new code. (This used to be sharper: while the §5 'skip' gate existed, a mid-run downgrade would also silently un-skip the #288 backfill, which is why G1 carried a per-chunk sqlite_master trigger-body probe. With no gate to lose, that whole guard is gone — a downgraded tab now costs only encoding, never a policy.) This is owned, not fixed: it stays sound (v-NULL reads as full and reconstruction never assumes a cadence), the only cost is those rows miss the skip/scope semantics until a later recompaction, and the small fleet bounds the exposure. A trigger-body version guard — refuse to install older bodies over newer, keyed on a schema-version marker — is the hardening if this ever bites in practice.
  • Ordering with the ALTERs: the column adds run in initializePowerSyncDb before the CLIENT_SCHEMA_STATEMENTS loop recreates triggers referencing them — same constraint and same slot as ensureUndoGroupIdColumns (repoProvider.ts:352-357 area; documented footgun at clientSchema.ts:1384-1391).
  • One explicit ANALYZE after the rewrite completes (the (block_id, id) add lands in this same slice — §3/§13; (block_id, created_at DESC) is retained per §3 — any measurement-driven revisit also lands here): the automatic drift check keys on blocks counts only (analyzeIsWarranted, clientSchema.ts:1438-1455) and would never notice row_events index changes.

9. VACUUM: making the bytes come back

The rewrite shrinks rows in place; SQLite reuses the slack for future inserts but the file never shrinks — and after compaction most pages are partially filled, which is exactly what incremental_vacuum cannot help with (it releases only wholly-free pages, and auto_vacuum is off on every existing DB and cannot be enabled without a full VACUUM anyway). A one-time full VACUUM is the honest tool.

Two wa-sqlite/OPFS-specific hazards to design around:
  • temp_store = MEMORY (repoProvider.ts:339): SQLite's VACUUM builds the rebuilt database in its temp store — with MEMORY that means holding a multi-GB rebuild in RAM. Bracket the VACUUM with PRAGMA temp_store = FILE (restore after), and verify at implementation that the OPFS VFS serves temp files sanely; if not, fall back to the VACUUM-INTO variant below.
  • Single worker, minutes-long stall: a multi-GB rebuild at OPFS speeds blocks every query. Run only at deep idle (scheduleDeepIdle), gated on the §8 completion marker, a one-shot client_schema_state marker of its own, and a free-space check sized against the worst case, not the target (review finding): SQLite documents VACUUM as needing up to twice the size of the original database file (the temp rebuild plus the rollback journal protecting the overwrite), so gate on navigator.storage.estimate() headroom ≥ ~2× the current file size. Gating on ~1.2× the post-compaction size would greenlight a 2+ GB→0.6 GB VACUUM with a few hundred MB of quota, stall for minutes, and die with SQLITE_FULL. If quota can't cover 2×-current, skip in-place VACUUM and take the VACUUM INTO route below — it needs only ~the compacted size and never journals the live file. (Rollback-journal mode makes an interrupted VACUUM roll back safely either way, repoProvider.ts:312-316.)

Two forms are specified here, mirroring the ANALYZE precedent (automatic drift-gated pass + manual override command in src/plugins/db-maintenance/plugin.ts:1-23): the deep-idle one-shot, and a command-palette "Compact database" action with a size/duration toast for users who want it now. Only the manual command ships in slice D; the deep-idle one-shot is specified but deferred — O2 is resolved manual-first, because an automatic minutes-long single-worker stall fired by a heuristic is the riskiest UX path in this design, so it waits for a later release with real telemetry (see slice D). Fallback variant if in-place VACUUM proves fragile on this VFS: VACUUM INTO a staging OPFS file + swap at next boot — the swap machinery already exists for DB import/recovery (src/utils/exportSqliteDb.ts, localDbRecovery.ts); costs more moving parts, bounds memory, never touches the live file. Non-negotiable freshness fence (review finding — data loss as previously worded): VACUUM INTO is a point-in-time snapshot taken at deep idle, and the recovery-swap machinery replaces the DB by design, with no freshness check — a naive next-boot swap would silently discard every write made after the snapshot (blocks, the ps_crud upload queue, row_events). The fallback must record a write fence at INTO time and the boot-time swap must verify the live file has not advanced past it — discarding the staging file otherwise, never the live one. The fence must be the SQLite file change counter (or an equivalent any-committed-write signal): a MAX(row_events.id)-plus-ps_crud-emptiness fence looks natural but is defeated by this doc's own machinery, running in exactly the deep-idle window the snapshot targets. The §8 rewriter UPDATEs existing rows in place and the §5.1 sweeper DELETEs them: both advance the file substantially while leaving MAX(row_events.id) untouched. Marker and bookkeeping writes (client_schema_state completion flags, command_events) and PowerSync's blocks_synced applies likewise advance the file without inserting a single row_events row. In every one of those cases the swap would pass its fence and silently discard the work. Capture the fence before starting the snapshot, not after it finishes (review finding — the ordering is itself a data-loss path): VACUUM INTO takes minutes on the multi-GB DBs this route serves, and a write committing after the snapshot's read view but before the fence is recorded would be baked into the fence while being absent from the staging file. The next-boot check then compares equal, passes, and swaps in a database missing that write — the precise failure the fence exists to prevent, reintroduced by recording it a moment too late. Read the change counter first, then snapshot; the swap is valid only if the live counter still equals the pre-snapshot value (equivalently: quiesce writes until both the snapshot and the fence record are complete). The fence value and the pending-swap state must live beside the staging file (or inside it), never in the live DB — an in-DB marker in this section's usual client_schema_state idiom would itself advance the counter and deterministically invalidate every swap. Cost owned: with a correct fence, the staging file survives only a write-free session tail — any user return invalidates it (interaction writes panel state, §5.1) and re-pays a multi-GB staging write on exactly the quota-tight device this route serves. If retries prove wasteful, swap immediately after INTO under a quiesced writer (close/reopen) instead of at next boot.

Reopening on restore needs the parent reconstructed, not read off the event (review finding): tx.restore is symmetric — a deleted = 1 → 0 UPDATE that likewise does not rewrite parent_id — so a compact restore event carries only deleted plus metadata and no parentId at all. A builder keying on the event payload can therefore close the interval on soft-delete and then have no edge to reopen with. It resolves the edge the same way it does for the other payload-less cases (§4.3 stateAt at that event, which walks back from the live row), rather than requiring restore events to carry a redundant parent edge. Every block written since the log era has a create event to open its interval from (§5 revised — with no skip exemption the index is fully log-derivable for that stock and needs no live-blocks crutch; I8). Upgraded DBs are the exception, and the builder must not read the line above literally: blocks predating row_events, or sitting behind a historical gap, have no create event at all — §8.1's no runningState case. A builder that opened intervals only on create/restore would drop those blocks from subtree-at-time results until their first logged parent move, or forever if they never move. So it opens an interval on the first event of a block's surviving history — which §8.1 makes full (a create, a pre-log chain-start, or a cadence anchor — all full=1) for every block the rewriter touches, i.e. it carries the parent edge outright. This is the second I8 exception (historical gaps) showing up as a concrete reader rule, not a new crutch: the boundary comes from the log, not from blocks. A first event that is a hard delete both opens and closes (review finding): an upgraded block that predates row_events and was then purged has a delete as its only surviving event, and the delete rule above says only that a hard delete closes an edge — so a subtree-at-T before the purge would omit the block entirely, even though the delete's full before_json carries its parentId. The builder therefore opens the interval from that before_json and closes it at the same event: the block correctly appears under its parent for all T before the purge, and vanishes after. Subject to the same deleted guard as a create (review finding): open this synthetic interval only when the delete's before_json.deleted is false. A pre-log row can have been soft-deleted long before it was finally purged, in which case its delete snapshot carries deleted = 1 and it was not live immediately before the purge — opening from it would resurrect a historical tombstone into every subtree-at-T result in that span. Mirroring the create rule keeps both bookends honest about liveness rather than just about existence. Two cases fall outside that guarantee, and the builder must resolve the opening parent by reconstruction rather than assuming the event carries it. (i) Zero-history pre-log stock (review finding): a block that predates logging and has no row_events at all is never seen by the §8 rewriter — §8.1's rule only shapes rows it rewrites — so its first-ever logged write, landing live after slice A, is whatever the v2 trigger produces: a compact patch, since §4.1/I6 deliberately removed any predecessor/history probe. That patch may omit parentId entirely. The builder resolves the edge with §4.3 stateAt at that event (which walks backward from the live row), rather than requiring the first event to be full — no format change and no anchor-backfill of that stock. (ii) The §5.1 retention sweeper (review finding): after a no-history block is pruned to its newest K, its first surviving event is whatever survived the cut — usually a compact update with no parentId — because §8.1's guarantee is about migration output, not post-prune state. The builder cannot open a historical interval for such a block from the log, and that is by design: retention deliberately discarded that block's older structure, so a subtree-at-T query over a no-history block can only report its current parent (from the live blocks row) — the one legitimate live-row fallback, scoped exactly to blocks whose history was chosen not to be kept. For UI-state blocks it is moot (they are not in the content tree a subtree reader walks); for a user-marked no-history content block it is the declared consequence of the tag. Per-member state is §4.3 stateAt; deleted-at-T members reconstruct fine (delete events carry full before). Per-block time-seek: the retained (block_id, created_at DESC) index (§3), read ORDER BY created_at DESC, id DESC so same-millisecond same-tx events resolve to the latest by id (bisecting created_at over the id index was considered and rejected: created_at is not reliably monotone in id under clock steps).
Drift-tolerant undo (undoManager.ts:26-28, user.ts:17-21) My action's writes (tx_id, plus group_id for the contiguous grouped run); the state to restore to; detection that someone else changed the field since My events carry their own before values (§4.1) — restore = un-apply, per block, that block's own events' before_json; no reconstruction at all, including for the first edit after an unlogged gap. Fold, don't pick: one tx can write the same block more than once and touch different fields each time (a mutator sets content, a same-tx processor later sets references/properties), so the block's earliest event alone carries befores only for that first statement's fields — restoring from it would leave the rest of the tx applied. Fold all of the block's events in id order, taking each field's first before and its final after; that folded pair is the restore target and the drift baseline. The fold unit is the undo group, not the tx. repo.undoGroup spans several repo.tx calls, and the live manager already merges them into one entry by exactly this rule — earliest before, latest after, per block (undoManager.ts:84-90, issue #306) — so a tx-scoped fold would split one user-perceived action into several undos. Only when group_id is non-null (review finding): the live manager merges only when entry.groupId !== undefined (undoManager.ts:84-90), so two adjacent ungrouped txs stay two entries even though both their group ids are NULL — a reader that expanded by "same group_id" would fold consecutive ungrouped txs into one. So: a NULL group_id tx stands alone (restore only its own tx_id); grouping applies only to a non-null id. Collect via that id, but only the contiguous run: merging targets the top-of-stack entry only (undoManager.ts:107), so a foreign tx landing mid-group splits the group rather than letting undo reorder history — leaving two stack entries that share one group_id. A reader that took every event WHERE group_id = ? would re-merge across that split and swallow the interloper's neighbours into one undo. But foreign is much narrower than "a different tx_id", and the difference is not cosmetic: the stacks are per scope (peekUndo(entry.scope)), and a tx is recorded at all only if its scope is undoable and it actually wrote (undoManager.ts:103-104). So a UiState or UserPrefs tx (undoable: false, changeScope.ts:61-75), a sync-applied or raw event, a zero-write tx, or an undoable tx on a different scope's stack all interleave without splitting anything. A reader that stopped at any foreign tx_id would shatter a grouped action that today takes one undo — and panel churn landing between two grouped edits is the everyday case, not a corner. The boundary is an undoable, writing tx on the same scope's stack carrying a different group_id. Contiguity is over the whole scope's tx order, not one block's events (review finding): the split happens on the stack, which is per scope and spans every block, so a group of edits to block A separated by a same-scope undoable tx that wrote only block B is split — even though B's tx never touched A. The intent: over commit order, take the sequence of undoable, writing, non-replay txs in the selection's scope; from the selected tx extend across adjacent such txs while their group_id equals the selection's, stopping at the first that differs; replay txs must be excluded and cannot be today (review finding): undo/redo application runs through the same commit pipeline and therefore does insert a command_events row, but is deliberately never pushed onto the live undo stack (repo.ts:1423 — runTx "records on undo stack", _replay "does not"). A reader building the run from every undoable writing tx in the ledger would count those replay writes as stack entries or boundaries and diverge from cmd-Z behaviour. The isReplay flag exists at commit time (commitPipeline.ts:177, threaded from repo.ts:1440) but is not persisted, so it must be stamped onto command_events alongside the other per-tx state below; group_id and per-field diffs join from row_events on tx_id. But this revision surfaces a correctness gap rather than hiding it (review findings): a no-history prune (§5.1) can delete a tx's row_events out from under it, and command_events — the only per-tx ledger that survives the prune — cannot reconstruct this sequence on its own, two ways. (1) No durable order. command_events carries only a millisecond created_at and a session-seeded, non-persisted tx_seq (let seq = Date.now(); () => ++seq, repo.ts:751-756) — neither a DB-wide monotonic key like row_events.id. When A(g), B(other group, only a pruned no-history block), C(g) commit in one millisecond, B can be placed outside A–C, so A and C look adjacent and same-group and merge across a real boundary — the unsafe direction, not the "extra split" an earlier draft wrongly claimed. (2) No write signal. command_events inserts a row for every repo.tx including zero-write ones (commitPipeline.ts:351-368), which live undo ignores (undoManager.ts:102-104); so "a tx with no surviving row_events is a boundary" also fires for a harmless zero-write tx and splits a group live-undo keeps, while filtering to txs-that-kept-events drops the fully-pruned writing tx we needed as a boundary. The ledger alone can't tell the two apart. (3) No group membership. Even knowing the tx existed and wrote, its group_id lived only on the deleted row_events (review finding). For A(g), B(g, writing only a pruned no-history block), C(g) — all one group — a reader that treats the unreadable B as a boundary splits an action live undo merges. So "unprovable ⇒ boundary" is not a safe default either: it over-splits as readily as the alternative under-splits. The honest fix is durable per-tx state that survives the prune — on command_events, all four values already computed at commit time: a real DB-wide monotonic tx sequence, the group_id, a wrote-a-row flag/count, and is_replay (review finding — an earlier draft listed only the first three, even though this same contract had just established that replay txs must be excluded and that isReplay is not persisted today; an implementation following the three-column list would let undo/redo replay commits appear as ordinary undoable writing txs, becoming false stack entries or group boundaries). With those, order, membership, write-ness, and replay status all reconstruct with no surviving row_events at all. Note the existing tx_context.tx_seq is not that sequence (review finding): it is seeded from Date.now() per Repo and incremented in-memory (repo.ts:751-756), so a clock rollback or a second session can place a later tx below an earlier one — persisting it would not give a total order. (a) needs a SQLite-allocated monotonic value, the way row_events.id is DB-wide monotonic — not the session counter. It cannot simply be a new AUTOINCREMENT column on command_events (review finding): that table already has tx_id TEXT PRIMARY KEY (clientSchema.ts:103-113), and SQLite ties AUTOINCREMENT to an INTEGER PRIMARY KEY, so an in-place ALTER either fails or degrades to a hand-rolled value that loses the very ordering guarantee this depends on. Specify one of: a small side sequence table the insert draws from, or a table rebuild making the sequence the rowid. That is new persisted state provisioned for a reader that does not exist yet (I6: nothing reads row_events at runtime) — the exact scope tension in the retention-vs-undo question (§16 open): pay it now, or restrict retention to non-undoable scopes so no undoable tx is ever pruned and undo never depends on one, or defer §10's guarantee until the reader is built. Flagged as open, not asserted as safe. The run is that maximal tx interval, and the folded target is every surviving event of those txs on every block they touched. Per-block walking would miss the block-B splitter; the boundary test lives at the scope-tx level. Well-defined from the ledgers alone, no undo-stack read — but which ledger supplies group_id depends on how O7 resolves (review finding): reading it from joined row_events, as an earlier draft said unconditionally, fails for exactly the txs O7(a) exists to rescue — in the A(g), B(g, touching only a pruned block), C(g) case, B has no surviving row_events to join, so its membership is lost and the fold diverges from live undo despite the durable columns being present. So under O7(a), command_events supplies the tx sequence, scope, group_id, write-ness and is_replay, and row_events supplies only the surviving per-field diffs. (Under O7(b) the question does not arise: no undoable tx is ever pruned, so every tx still has its events.) Two per-kind carve-outs: a create (before_json = NULL) undoes to absence (soft-delete/tombstone), with redo re-applying the folded final after state — not the create's own after_json: a tx that creates a block and then writes it again (QuickFind's create-page does tx.create then repo.addTypeInTx, QuickFind.tsx:534-543) leaves the create's snapshot holding only the bare row, so redoing from it would silently drop the type and aliases the same action added — the fold rule above already yields the right state, and the carve-out is only about where undo stops (absence), never about which snapshot redo replays. And when the anchor coin made an update full=1, before_json is a full snapshot rather than a changed-field patch, so restore and the drift check must diff it against after_json to recover the changed-field set — else untouched fields get clobbered or counted as drift. Drift check = my (changed) after vs live blocks row. Two more constraints a row_events-backed undo must carry from the live engine, or it corrupts (review findings): (1) engine metadata is excluded from the un-apply and the drift check. A compact event's changed-field set includes updatedAt / userUpdatedAt / updatedBy because metadataPatch advances them on every write (txEngine.ts:906-927), but replay deliberately stamps replay-time metadata rather than restoring the old author/time (txEngine.ts:770-772, :846-854). The reader folds by id for the per-field before/after values, then applies in replayApplicationOrder, not id order.

13. Rollout slices (every slice lands green)

Rollout principle — a rule shipped in slice N binds only clients that have run slice N. Several findings in review were the same mistake in different places: a mechanism was specified as if all clients would honour it the moment it shipped. They cannot. The bootstrap re-installs trigger bodies from whichever client booted last (withTriggerRecreate), old code will not consult a marker it does not contain, and an old tab can create or delete rows at any time during the mixed-version window. So every mechanism whose correctness depends on upgraded code must name one of three things, explicitly:
  • a reconciliation backstop that repairs what an old client did (§5.1's structural tombstone scan and the recurring UI-state subtree pass). Note the arming condition is not a version transition (review finding): a scan keyed to a version bump misses a downgrade that happens after it, and it also misses a post-guard client that keeps the current trigger body while still creating untyped UI-state rows. The scan is armed by the pre-recreate body probe and stays active through the old-creator window;
  • a structural fallback that does not depend on the new mechanism at all (B1 discovering UI-state blocks by computed deterministic id rather than by tag);
  • or an explicit fleet precondition — the invariant holds once no pre-slice client can still run (I9, whose transactional trigger swap only binds clients at ≥ A). A precondition is a migration consideration, legitimate to rely on for this fleet, but it must be stated rather than assumed — and where possible paired with detection, as §8.1's content-level divergence check does for I9.
"It ships in slice A/B" is not by itself an answer to "what about the tab that hasn't upgraded".

Those are found by traversing the children of the computed parents — which is discovery, not authorization, since parentage is mutable. Open problem: the gate is sound for computed-id blocks and not for panels (review finding, unresolved). Panels and panel-stack rows carry runtime ids, so the only signals available are their parent being a computed id — but parentage is exactly the mutable structure this gate was rewritten to stop trusting (tx.move) — or the panel type, which is user-editable ([1000]). systemMint is not persisted. And computed identity is not unforgeable either (review finding, which removes the last candidate signal): tx.create accepts a caller-supplied id (const id = data.id ?? newId(), txEngine.ts:340-344), and raw SQL can insert any id, so a plugin can squat a computed UI-state id before the real state block is lazily created — DuplicateIdError only prevents colliding with one that already exists. A content block with real history sitting at that id would then be classified as dedicated UI-state storage and pruned. So in the current data model no signal — parentage, type, systemMint, or computed id — soundly identifies dedicated UI-state storage, and panels are the entire 35 MB motivation. Three ways out, and this is a decision, not an oversight:
  • (i) persist creation provenance — a durable "system-minted state block" flag written at creation and not user-settable. Note this is now the only form (i) can take: deterministic panel ids alone would not do it, since computed ids are squattable (above). It also pays a second dividend — a column read replaces the per-write properties_json extraction §5.1's flag needs (I6). But it must be a synced signal, not a local flag (review finding), which is what actually sets this option's price: a panel created on device A arrives on device B through the blocks sync projection, which carries only the existing columns (sync-config.yaml:64-76). A local-only creation flag would therefore be absent on every receiving device — and since type, parentage and computed id have all just been ruled out, B would have no sound fallback and would either skip remote-authored UI-state churn (most of a shared panel's traffic) or silently drop back to an unsound gate. So (i) means a new synced column on blocks: a schema change, a sync-config.yaml change, and a PowerSync deploy — not merely a code change in the creation path. That is a materially bigger step than this option looked at first, and it is the honest comparison point for (ii) and (iii). But a creation-time flag does not exist on the stock (review finding), including the 14.7k-event panel that motivates the whole project, and the only signals available to backfill it are the unsound ones. The honest resolution is the ordinary trust-on-migration split: classify existing rows once using structure + type + computed id, and be sound for everything created after. That is defensible precisely because the stock predates the mechanism — no one could have squatted an id to defeat a gate that did not exist — but it is a judgement call, not a proof, so the migration should report its classified count for review before it runs rather than silently tagging. Alternatively, name the stock as an explicit residual and bound only new churn — which forfeits the measured 35 MB and most of the motivation.
  • (ii) accept a weaker gate — admit runtime rows on parent-plus-type and accept that a deliberate forgery (move a content block under a layout-session and apply the type) gets its history pruned. Bounded by requiring two deliberate steps, but it is real data loss and [1000] is what makes the type reachable.
  • (iii) resolve O7 — then B1/B2 and this gate cease to exist, because the gate is only scaffolding for deferring O7.
This corrects an earlier claim in this doc that O7 "blocks nothing". It blocks shipping the sweeper soundly: the byte win is real and B1 still delivers it, but only under (i) or (ii). And the deferral is not free either (review finding): B1-era tags on ordinary content produce histories that, if B2 later enables the affordance under O7(a), lack the durable tx_seq/group_id/write-count/is_replay state O7(a) requires — and for hard-deleted ones only row_events ever carried that boundary data. So B2 must first backfill or exclude pre-O7 tagged content histories, or enabling the affordance later prunes exactly the undo metadata the gate was built to protect.
Hiding the type from the property panel until B2 is worth doing too, but it is the cosmetic half — the sweeper-side restriction is the enforceable one, because it does not depend on a type-registry capability that does not exist today. The reason is asymmetry, not caution: retention is one-way, so shipping B2 first lets a prune permanently erase the only boundary data (group_id and friends) that O7(a) or (c) would later need, and no subsequent decision can recover it — whereas shipping B1 first and adding B2 after O7 resolves costs nothing. Until O7 is decided the shipping default is therefore (b)-shaped — the safe direction, but note it is not a free one: (b) is itself a sweeper-side dedicated-storage check, so it needs the same sound identification this section's open problem is about (review finding). The default protects the irreversible thing (boundary data) without by itself making B1 sound.

Slice A: a stateful suite drives random edit sequences (create / update / no-op update / soft-delete / restore / hard-delete, projected-field edits) against a model oracle — stateAt at every event must equal the model's timeline (I1/I2), and every event must pass the shape property — which is kind-dependent, and a suite that applied one rule to all kinds would fail on its own generator: for updates and soft-deletes, equal key sets on both sides, both agreeing with full (the §4.1 one-coin trap); for creates and hard-deletes the event is one-sided by construction (a create has no before_json, a hard delete no after_json — §4.3), so the property is that the present side is a complete snapshot and the absent side is NULL. Because anchors carry no correctness weight, every oracle is anchor-agnostic — SQLite's unseedable random() cannot break seed reproducibility of a failure's semantics — but the suite also runs deterministic p = 1 and p = 0 trigger variants so both the full and the compact path are exercised on every seed. Slice B adds fuzzed interleavings of user / sync-applied / raw / derived / bulk writers (oracle: every write of every class that changes a row produces its event — no class of writer is exempt — labels are truthful, and each block reconstructs forward from its create with the blocks table ignored entirely: I3/I8, excepting the raw UPDATE … SET id case below), and randomized §5.1 retention sweeps interleaved with UiState, user, AND sync-applied writers hitting the same no-history-typed blocks — including deleted flagged blocks, identity-confirmed rows an old client created untyped (the structural-backstop case, which a type-only generator would never produce), and metadata-only unlogged gaps (a trigger-suspended user_updated_at write, I9) (safety oracle: after a sweep, a flagged block retains its newest K events by id whatever their source — a plain oldest-prefix cut, no wall-clamp, which I9 makes unconditionally safe; a deleted flagged block is swept via the no_history flag its events still carry (live, soft-deleted, and hard-deleted all found by one partial-index query), no survivor is older than any pruned event, and stateAt stays exact for every survivor including compact ones — the property that proves oldest-prefix deletion is safe without full-writes; the prune decision tracks the authoritative type, never an event flag — tag then untag interleaved with edits leaves the block skipped, create-then-tag prunes the whole block including its unflagged create, and a block written by a stale trigger body that could not stamp no_history — including a slice-A body emitting v = 2 rows without the column — is still pruned, because the decision reads the type; untouched blocks are never pruned, and a pruned block with a foreign-typed sibling in the same tx does not drag it in. Scoped to what the shipping slice actually does (review finding): this is the B1 suite, so it generates dedicated UI-state storage and asserts B1 semantics — a user-applied no-history on an ordinary content block must be inert, not pruned. Asserting generic typed-block pruning here would either fail a correct B1 implementation or pressure it into shipping the gated, irreversible B2 behaviour early; arbitrary typed-block pruning is B2's oracle, behind O7. And discovery is tested, not just the decision (review finding): reading the authoritative type only settles whether to prune a candidate already in hand — it cannot make an undiscoverable block discoverable. So the suite includes the compound stale-body case — a block tagged while a stale body is installed (no event carries the flag) and then deleted (so block_types has dropped it) — and asserts it is reached by the one-shot tombstone reconciliation (§5.1), not by the partial-index scan. Without that case the tests would bless a sweeper that permanently misses deleted stale-body candidates. Live UI-state blocks are additionally asserted discoverable structurally (computed identity), so an untagged old-client creation is still found. Plus a progress oracle, because a never-prunes sweeper passes safety trivially: a currently-typed block with > K events ends the sweep with exactly K). Slice C's I5 equivalence suite is the same generator aimed at v1 histories (metadata-only gaps, pre-log openings, no-op runs): migrate, then stateAt must be identical for every event — under random interruption/resume points. It specifically pins I9: the generator's only unlogged gaps are metadata-only, so content/structure stateAt is identical across them and a pre-gap event reconstructs exactly from either side — no wall needed, because nothing content-level changed in the gap.

Two qualifiers on slice B's first oracle are load-bearing rather than hedges — an oracle stated without them fails on the suite's own generator, which is how both were found.

  • Writes that change nothing produce nothing. An identical no-op update writes no event by design (§4.1's identical-update skip — the measured 339,923 no-op updates on the dev instance are why it exists), so an oracle demanding an event per statement fails on the generator's own no-op case. I8 is a claim about provenance classes — no writer is privileged — not about SQL statements; a write that changes no state leaves no state to reconstruct, so skipping it costs the log nothing. (The asymmetry with §8 is deliberate and worth naming: the rewriter keeps v1's logged no-ops as empty compact events because deleting history rows is the worse sin (I7), not because "an UPDATE happened" timing is a value this design upholds going forward — v2 does not record it.)
  • A raw UPDATE … SET id is a rename boundary, not a create. The §2.1 unguarded path is recorded under NEW.id with full = 1 (§3), so the new id has no create event and the old id's newer-side terminator is broken by the key rewrite itself — v1 alike. "Every block reconstructs forward from its create" is therefore impossible here by construction, and an oracle that insisted would either fail on a documented case or push the implementation to hide it. The suite models it as what it is: the pre-rewrite chain reconstructs via its older-side full, the post-rewrite id opens at the full=1 rename event, and the two are asserted as a rename pair rather than one continuous history. This is the same raw-write asterisk §4.3 already carries on "ChainBroken ⇒ corruption".

15. Rejected alternatives

  • Pruning (keep-recent + grace window) — the row-events-retention.md design. Rejected by the frame: history is a product feature to keep, and pruning also would not have reduced the per-write cost that motivated that doc.
  • Server-shape adoption without anchors (changed-cols diffs, full bookends only — exactly blocks_history's Option B). Reconstruction becomes full-chain replay, and a single lost row poisons every state beyond it on the walk side. Sampled anchors cost ~1.5% of rows and bound both.
  • Deterministic probe + chain_pos cadence — the original §4 design (superseded 2026-07-15 by O5's adoption): a per-write predecessor probe fed a deterministic anchor window W=50 and promotion rules for missing/v1 predecessors. It bought deterministic walk bounds and "every chain starts full" — machinery that exists only because after-only compact events made chains base-dependent. Invertible payloads delete the dependence itself, at ~2× compact-update bytes. Full tradeoff record in O5.
  • Counter side-table cadence — mutable state that can drift from the log, an extra B-tree upsert per write, its own backfill/repair story; rejected under both cadence designs.
  • Naive probabilistic anchors over after-only payloads — unsafe: without invertible patches, chain-start and gap cases splice silently or strand undo (O5 record). A create-only-skip variant was examined and set aside with the format change.
  • App-layer event writer — §7 table; loses the catch-all property that is the log's reason to exist.
  • Values-free or sampled logging for the bulk bursts — §5 table rows (b)/(c).
  • Skipping UiState-scoped events entirely — proposed from the §5.1 measurements, set aside by the user 2026-07-15 in favor of bounded retention (keep newest K per block): a short local record keeps panel-state debugging possible while the bound removes the actual problem (unbounded accumulation).
  • history_mode = 'skip' exemption for bulk bursts — adopted 2026-07-15, deleted 2026-07-17 (user). A tx-scoped tx_context.history_mode column + RepoTxOptions plumbing let enumerated one-shot bursts (the #288 backfill) write nothing to the log, saving ~230 MB on the authoring device. Why it was wrong, in the order the reasons landed: it was asymmetric — §6 keeps sync events, so all other devices logged the same 650k creates on arrival, and the exemption blinded only the machine that ran the migration; it removed the record from the riskiest write the system performs, which is this project's own repeated failure mode (bulk passes that damaged data and whose recovery depended on reconstructing what they did); its economics were set against a 2.2 GB log that this very document shrinks by ~1.45 GB; and it bought a permanent invariant hole (I8 — the log stopped being self-sufficient, and §10's parent-edge index needed the live blocks table to enumerate skip-created rows) in exchange for a rare event that should not be routine. Deleting it also removed the gate, its NULL-safety rule, its deletion-state carve-out (OLD.deleted IS NEW.deleted — soft-delete/restore are UPDATEs and would otherwise have been skippable), the per-chunk sqlite_master trigger-body probe, and #288's dependency on slice B. Reintroduce only against a concrete burst that is genuinely unaffordable — and then keyed on provenance (I4), never columns.
  • Scope-keyed UiState retention (three dead variants) — all superseded 2026-07-17 (user), landing on an explicit no-history type + whole-block newest-K (§5.1). (a) Prefix-cut over compact events (rule R1, the original O6 mechanism): because a compact event cannot be deleted from a middle without silently splicing (§4.3), the sweeper cut only a block's oldest contiguous all-local-ui prefix, with opposite sentinels for two absent operands, and dragged a "freeze class" — any interleaved non-local-ui event pinned the cut forever, making the bound partial and needing an undesigned "phase 2". (b) Write local-ui events full for order-free pruning: dissolved the freeze but keyed the sweeper on the event's scope, which is NULL on every sync arrival — measured on prod, that left the flagship 14,768-event panel with 0 prunable events (its 244 sync-authored events are unkeyed), and cost full snapshots plus a per-write partial index on the hottest class. (c) Key on the structural panel type: fixed the sync-completeness (type rides sync) and let whole-block newest-K over compact events work, but keyed a destructive delete on panel-ness — and types are user-editable (properties_json.$.types), so a content block tagged panel would lose its history silently, the retention consequence invisible in the structural type's name (review finding). The fix is not to lock the type down but to name the consequence: a dedicated no-history type that panels carry and users may apply to their own scratch blocks — the prune becomes a legible, deliberate opt-out, and mis-tagging panel is harmless. Root cause across (a)/(b): scope does not survive sync. Across (b)/(c): the key must be both sync-complete and a non-surprising signal for a delete. Kept as the record that the keying, not the event format or the predicate, was the lever.
  • In-trigger retention pruning — enforcing the UiState bound from the trigger itself would put an indexed read + DELETE on the hottest write path in the system (violates I6, and the §4 design just removed the last write-path history read); the deep-idle sweeper enforces the same bound eventually.
  • Column-keyed exemptions (e.g. never log properties_json) — violates I4; loses the only history for cell-primary keys and the whole pre-flip period (PR #288 §12).
  • Sub-key diffs of properties_json — real wins today (property edits rewrite the whole cell), but it complicates every reader and the trigger, and the props migration is about to shrink the cell structurally; column-granular is right-sized for the post-#288 world. Revisit only if #288 stalls.
  • Compression (per-event zlib, content dedup) — wa-sqlite ships no compression extension; JS-side compression forces an app-layer writer (see above). The compact format gets the structural win without new machinery.
  • History in a separate ATTACHed database file — would make VACUUM trivial (vacuum the side file, or drop it), but SQLite triggers cannot write across databases, so it forces the app-layer writer; rejected with it.
  • Dropping sync-applied events at all (the earlier §6 draft: per-workspace-mode drop for plaintext, plus a workspace_history_policy side-table) — overridden by the user 2026-07-15: local history must stay self-sufficient; the server-side bytes (§6.1) are an access story, not a substitute. The retroactive variant — deleting existing sync events — was rejected from the start: history is kept.

Fully pruning such a tx's row_events (with its group_id) can leave a future undo-from-history reader (§10) unable to reconstruct the tx boundary: command_events survives the prune but has no DB-wide monotonic order (only ms created_at + a session-seeded, non-persisted tx_seq) and no wrote-a-row signal (it logs zero-write txs too), so a same-millisecond boundary tx can be misplaced into a wrong merge, and a fully-pruned writing tx is indistinguishable from a zero-write one. Three candidate resolutions, none yet chosen:
    • (a) Provision durable per-tx state now — on command_events, four values all already computed at commit time: a SQLite-allocated DB-wide monotonic tx sequence (monotonic the way row_events.id is — but not a plain AUTOINCREMENT column added to command_events, which its tx_id TEXT PRIMARY KEY forecloses; a side sequence table or a rowid-making rebuild, per §10), the group_id, a wrote-a-row flag/count, and is_replay. All four are load-bearing, each for a distinct failure the review surfaced: without the sequence there is no durable order (the existing tx_context.tx_seq is Date.now()-seeded per session, repo.ts:751-756, so persisting it gives no total order); without group_id a fully-pruned same-group tx can't be shown to belong to the group and over-splits an action live undo merges; without the write flag a zero-write tx is indistinguishable from a pruned writing one (and mutator_calls can't substitute — §3); without is_replay undo/redo replays are counted as stack entries (repo.ts:1423). Closes O7 exactly, and the columns sit on the low-volume per-tx table — but it is persisted state provisioned for a reader that does not exist yet (I6).
    • (b) Restrict what the sweeper will act on — dedicated non-undoable UI-state storage only. No undoable tx is ever pruned, so undo never depends on one. Costs [1000]'s "mark my own (undoable) block transient" affordance; the 35 MB UI-state motivation is unaffected. This must be enforced in the sweeper, not as a rule about who may tag (review finding — an earlier draft framed (b) as a tagging policy): no-history is a registered TypeContribution and stays manageable in the property panel, so "only these blocks may be tagged" is unenforceable. A user tagging an undoable content block would still get its row_events pruned — destroying precisely the undo-boundary data (b) exists to avoid needing. Which means (b) is not the free option it appeared to be: a sweeper-side dedicated-storage check requires the same sound identification mechanism §13's open problem is about. (b) and that problem are therefore one decision, not two — either a system-only type / persisted provenance serves both, or a knowingly weaker gate is accepted for both. Note this is a policy restriction on tagging, not a scope filter on events (review finding): an earlier draft phrased (b) as "prune only ChangeScope.UiState history", which would have reintroduced exactly the sync blind spot §5.1 removed — scope is NULL on sync-applied events (§3) and a receiving device has no command_events row for the originating tx, so it cannot tell a remote UiState write from a remote undoable one. That version of (b) would either leave sync-authored UI-state churn unbounded again (measured: 0 of the 14,768-event flagship panel prunable) or prune remote undoable boundaries without solving O7 at all. Keyed on the block, (b) is coherent; keyed on event scope, it is not.
    • (c) Defer §10's guarantee — ship retention; record that undo-from-history over a pruned undoable tx is best-effort until the reader is actually built, at which point (a) or (b) is decided with a concrete consumer in hand.

Only the undoable corner is affected — non-undoable UI-state pruning (the dominant case) is unaffected. Correction (2026-07-18): this partly does block shipping. An earlier revision claimed the B1/B2 split made O7 cost-free to defer. Two review rounds since have shown the split cannot be made sound in the current data model (§13's callout): the B1 gate keys on computed identity, which beats parentage but is unavailable for panels and panel-stack rows (runtime ids, mutable parentage and type) and is not even sound for the ensureStateChild family, since tx.create accepts a caller-supplied id and a computed id can be squatted before the real state block is lazily created. No signal in the current model — parentage, type, systemMint, or computed id — soundly identifies dedicated UI-state storage, and panels are the whole byte motivation. So B1 ships only under a code change persisting creation provenance, or under a knowingly weaker gate. Deferring also has its own cost: B1-era tags on content blocks produce histories lacking the durable state O7(a) needs, so B2 would have to backfill or exclude them. The split still sequences the work — B1 (tagging dedicated UI-state storage + the sweeper) before B2 (the user-assignable affordance) — but it does not make this decision free. The gate exists because retention is one-way — shipping B2 before deciding lets a prune permanently destroy the group_id/boundary data that (a) and (c) both depend on, and no later choice can recover it, while deferring B2 costs nothing. So the decision can be taken unhurried; what it must not be is taken implicitly, by shipping B2 first.
