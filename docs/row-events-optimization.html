row_events: compact local history with sampled anchors

Keep the full per-row history log, shrink it ~4×: changed-columns events with periodic full-snapshot anchors, provenance-gated logging for bulk derivation bursts, per-workspace-mode sync logging, and a reconstruction-lossless migration of the existing ~1.17M full-snapshot events — followed by the VACUUM that actually returns the bytes.

How to read this doc

The load-bearing core is §3–§4 (the event format and the anchor/reconstruction machinery) and §8 (the migration of existing history). §5 and §6 resolve the two policy questions (derived-write logging, sync-source logging). If you are implementing, start from your slice in §13 and the invariants index. If you are PR #288, read §11 — it is the contract your slice C consumes. SQL in this doc is illustrative shape, not implementation.

Decisions made in this doc (each argued in its section):
  • Compact updates, column-granular: an update event stores only the changed domain fields' after values; identical updates are not logged (§3).
  • Deterministic per-block anchors every W=50 events, cadence carried in the log itself (chain_pos), not in an auxiliary counter and not probabilistic. Anchors carry full before+after (§4). Cadence mechanism under re-review — O5.
  • Every chain starts full: a compact update whose predecessor is missing — or is a legacy-v1 event — promotes itself to an anchor (§4, inherited from PR #288's invariants index).
  • Derived writes log normally in compact form; only enumerated one-shot bulk bursts skip, via a tx-scoped tx_context.history_mode opt-in — provenance-keyed, never column-keyed (§5).
  • Sync-applied events drop for plaintext workspaces, stay (compact) for e2ee; unknown mode logs; the drop gate keys on an explicit sync-apply marker, so raw/unknown writers always keep logging. The redundancy claim against server-side blocks_history is verified with caveats stated (§6).
  • Triggers remain the writer; bodies are generated from a single column spec shared with the existing snapshot/upload projections (§7).
  • The migration is reconstruction-lossless: chunked, resumable via the format tag itself, sampling anchors at cadence and at chain discontinuities, so every state derivable today stays derivable (§8).
  • One-time full VACUUM (deep-idle + manual command), not incremental_vacuum — in-place row rewrites leave half-full pages only a rebuild compacts (§9).

1. Why now

row_events is the client-side per-row audit log: three triggers on blocks write one event per row write, for local and sync-applied writes alike (src/data/internals/clientSchema.ts:370-381). Updates store two full row snapshots — before and after — including the whole properties_json and references_json columns (blockJsonObjectSql, clientSchema.ts:386-402). Nothing reads the table at runtime (clientSchema.ts:63-65); it is intentionally unbounded and never trimmed (clientSchema.ts:67-69), and there is no client-side VACUUM anywhere in src/, so even a future deletion would not return OPFS bytes.

Observed scale and the two forcing functions:

  • It is already the biggest thing in the database. ~1.17M events × ~2 KB on a dev instance (docs/row-events-retention.md:4) ≈ 2+ GB of OPFS for a ~300–350k-block workspace — history bytes rivaling or exceeding live-data bytes, on devices where OPFS quota is finite.
  • PR #288 multiplies it. The properties-as-blocks backfill creates ~650k rows through repo.tx per workspace (its §12 and measurement section; the spike took 5–10 minutes largely paying row_events + ps_crud envelopes), and those rows then sync down and append events again on every receiving device. Its §12 carries the constraint record: the optimization must land before its slice C runs. §11 states the contract.
  • Every fresh device pays on first sync. The observer materializes each downloaded block into blocks (materialize.ts:365), firing the INSERT trigger once per block: ~350k full-snapshot create events (~350 MB) per device, recording rows the server already has.

The consumers this format serves are all future — block history / time-travel UI, drift-tolerant undo (src/data/internals/undoManager.ts:26-28 names row_events.before_json as the substrate; today's undo is in-memory only, depth 100 at undoManager.ts:80-82), forensics after incidents (the SRS backlink-wipe class), and convergence debugging for the props migration. Because nothing reads the log yet, we are free to change the format — the design is judged by what those readers need (§10), not by compatibility.

2.2 What reads it, and adjacent facts the design leans on

  • No runtime reader. The only statements touching row_events in non-test src/ are the three trigger INSERTs (verified by search). The old row_events→invalidation tail (rowEventsTail.ts) is deleted; only historical comments mention it (observer.ts:32, invalidate.ts:6, block.ts:118).
  • Trigger bodies self-upgrade. Every boot re-applies DROP TRIGGER + CREATE TRIGGER from source (withTriggerRecreate, clientSchema.ts:1170-1175) — a trigger-body change ships fleet-wide with no per-change migration.
  • ALTER-based column adds have precedent and a footgun. ensureUndoGroupIdColumns (clientSchema.ts:1375-1410) added group_id to row_events and documents the ordering constraint: the ALTER must run before any trigger re-creation references the new column.
  • Provenance is joinable today. command_events (clientSchema.ts:99-113) records one row per repo.tx — scope, description, mutator_calls — keyed by the same tx_id the events carry (commitPipeline.ts:348-368). It is small (one row per tx) and also unbounded; out of scope here beyond this note.
  • Idle machinery exists. Marker-gated one-shot jobs (client_schema_state, clientSchema.ts:182-187), MarkerStore/PendingIdleJobs (idleMarkerJobs.ts), deep-idle scheduling (scheduleIdle.ts), and the idle ANALYZE pattern (repoProvider.ts:412-431).
  • DB config that matters for §9: rollback journal, not WAL (repoProvider.ts:312-316); temp_store = MEMORY and a 256 MiB page cache (:338-339); single SQLite worker.
  • docs/row-events-retention.md is stale: it cites the deleted rowEventsTail.ts drain, a local-ephemeral source that no longer exists, and a prune design ("keep recent 50k + 7-day grace") superseded by the keep-history direction. Re-stamped as superseded in this PR.

3. Event format v2

Two new columns, added by an idempotent ALTER migration in the ensureUndoGroupIdColumns mold (and with its same must-run-before- trigger-recreation ordering):

ColumnMeaning
v INTEGER Format version. NULL = legacy v1 (full snapshots per §2.1); 2 = compact format below. How a reader tells formats apart, and how the migration finds unrewritten rows.
chain_pos INTEGER Position since the last full event of this block. 0 = this event carries complete row state (create, anchor, delete); k>0 = compact event, k-th since the last full one. NULL on v1 rows (every v1 row is full by construction). One column serves as both the anchor marker and the cadence counter — the log describes itself; there is no auxiliary state to drift or migrate.

Per-kind payloads in v2 (domain-shaped keys, same naming as blockJsonObjectSql today so v1 and v2 payloads read uniformly):

create0NULLfull snapshot (unchanged from v1 — creates are inherently full)
update / soft-delete, compactk>0NULLchanged domain fields only: JSON object, key present ⇔ column changed, value = new value (JSON null = cleared). The key set is the changed-columns list — no separate column.

delete (hard purge)0full snapshotNULL
identical update (no column changed)no event written — mirrors the server trigger's skip (20260522062437_add_blocks_history.sql:120-125). The v1 trigger has no such gate; this alone deletes a class of pure-noise 4 KB events (LWW echoes, clamp no-ops).

Change detection compares storage columns (OLD.x IS NOT NEW.x — the exact predicate the upload trigger already uses, clientSchema.ts:542-544) but emits domain keys. Column-granular only: a changed properties_json stores the whole new cell value, not a sub-key diff — see §15 for why, and note the props-as-blocks migration shrinks that very column structurally. One v1 nuance carries over: userUpdatedAt is serialized as coalesce(user_updated_at, updated_at) (clientSchema.ts:397); v2 keeps that projection for the field and accepts that a (never-occurring today) explicit clear is not distinguishable.

Index change — the add moves to slice A (review finding; originally bundled with the §8 migration): a new (block_id, id) index. id is AUTOINCREMENT-monotonic and is the authoritative per-block order (same-millisecond events in one tx tie on created_at); the trigger's predecessor probe (§4) and every reader walk need exactly this ordering, and the entries are smaller. The probe runs on every logged update from the first v2 write, and the existing (block_id, created_at DESC) index cannot serve ORDER BY id DESC LIMIT 1 — without the new index, a hot block pays a per-write sort over its whole event history inside the write trigger, during exactly the window (slices A/B) that must precede the #288 burst: an I6 violation. To avoid a boot stall (a synchronous CREATE INDEX over the ~1.17M-row stock is a seconds-scale single-worker pause), slice A builds the index as a marker-gated idle job and only then swaps the v2 trigger bodies in — trigger recreation is cheap and can run mid-session; until the marker is set, boots keep installing the v1 bodies. Retiring the old (block_id, created_at DESC) index stays a slice-C decision, and may resolve to "keep both": per-block time-seek ("latest event ≤ T") wants it — see the subtree reader row in §10. (created_at DESC) and (tx_id) stay for time-scoped and tx-join forensics.

4. Anchors: cadence, promotion, reconstruction

4.1 Cadence and keying — decision

On every logged update, the trigger probes the block's most recent event — one O(log n) seek on the new (block_id, id) index: 

SELECT v, chain_pos FROM row_events
 WHERE block_id = NEW.id ORDER BY id DESC LIMIT 1

and decides:

  • No prior event → this chain has no base: promote to anchor (full before+after, chain_pos = 0). This is PR #288's "every history chain starts full" invariant (I1) — it is what makes every exemption in §5/§6 safe: OLD is always available to the trigger, so the first logged edit of an unlogged row rebuilds a complete base no matter why earlier events are absent.
  • Prior event is v1 (v IS NULL) → also promote to anchor. v2 chains never count a v1 event as their base — this decouples the live log from the §8 migration entirely: the rewriter may compact any v1 event without checking whether some live chain leans on it. Cost: one anchor per block on its first post-deploy edit, only for blocks edited at all.
  • prev.chain_pos ≥ W−1 → anchor (chain_pos = 0).
  • Otherwise → compact event, chain_pos = prev.chain_pos + 1.
Decision — deterministic per-block window W = 50, position carried in the event row itself. Alternatives and why not:
  • Probabilistic (1-in-50 by random()): stateless, but the window is unbounded in the tail (P(no anchor in n events) = 0.98n; the 99th-percentile gap is ~228 events) — which makes both reconstruction cost and the corruption blast radius unbounded. Rejected.
  • Per-block counter side-table: O(1), but introduces mutable state that can drift from the log, costs an extra B-tree upsert on every write, and needs its own backfill/repair story. Rejected.
  • Byte-budget (anchor after ≥X bytes of compact payloads): bounds replay bytes rather than replay count, but needs an accumulator (same drawbacks as the counter) or an O(W) SUM per write. Deferred — W bounds bytes well enough in practice (worst case W × largest column value), and cadence is a constant we can revisit (§4.4).
The chosen probe keeps the log self-describing: cadence state cannot be lost, corrupted, or skewed independently of the history it describes, and the migration + live trigger share one rule.
Note: this mechanism (probe + deterministic cadence) is under re-review — see O5 for a probe-free candidate that also changes the compact payload shape.

4.2 Anchor content — decision

Anchors carry full before + full after (not after-only):

  • Integrity check for free: replaying the compact run from the previous full event must reproduce anchor.before_json. The §8 migration's equivalence test and any future integrity sweep get a built-in checkpoint at every anchor.
  • One-sided recovery: if the run before an anchor is lost, the anchor's before still yields the state just prior to it.
  • Cost is noise: one extra ~1 KB side per W=50 events ≈ +20 B amortized per event.

4.3 Reconstruction algorithm

stateAt(blockId, eventId):                  # row state just after event E
  E = event(eventId)
  if E.v is NULL:      return per-kind full side (v1 events are all full)
  if E.chain_pos == 0: return per-kind full side (create/anchor: after; delete: gone/before)

  # compact: walk back to the chain base — at most W−1 steps, contiguous
  # in the (block_id, id) index
  patches = [E]
  for ev in events(blockId, id < E.id, ORDER BY id DESC):
      if ev.v is NULL or ev.chain_pos == 0:
          base = fullAfterOf(ev); break
      patches.prepend(ev)
  if no base: raise ChainBroken            # bounded loss — see below

  state = base
  for ev in patches: state = {...state, ...parse(ev.after_json)}
  return state

Complexity: O(log n) seek + O(W) contiguous index scan + O(W·c) field merges (c = mean changed fields, ~3) — microseconds-scale per point lookup at W=50. A whole-block timeline streams forward once, carrying the running state: O(events of block). Workspace-wide time-travel to time T: per touched block, binary-search the event ≤ T, then stateAt — O(B·(log n + W)).

Failure bound. ChainBroken can only arise from row loss/corruption inside the log itself. The damage is confined to the compact run between two full events — at most W−1 states; everything at or after the next full event reconstructs normally, and the current state is never at risk (it lives in blocks). This is the property the sampled anchors buy: reconstruction is "nearest anchor + short replay", never full-chain replay to the insert, and an interrupted chain loses at most one window.

4.4 W is a tunable constant, not a schema property

W lives in the trigger SQL, which re-installs from source on every boot (clientSchema.ts:1170-1175). Changing it needs no migration and no version bump: chain_pos makes each chain self-describing, so mixed cadences coexist — readers never assume W, they walk to chain_pos = 0. Amortized anchor overhead ≈ (full-pair size − compact size)/W ≈ (2048−200)/50 ≈ +37 B per update (~+18% of compact-update bytes); W=100 would halve that at double the replay bound. 50 is the starting point (matching the frame's 1-in-50 suggestion), revisit with real telemetry.

5. Derived-write logging — analysis and recommendation

The props migration introduces machine writers: the one-shot backfill (~650k creates), the children→cell projection (2–3 row writes per property edit), the consistency sweep, and arrival catch-up. The question: do their writes belong in the history log? The earlier "skip them all via a tx_context-gated trigger condition" suggestion trades away exactly the record you want when the machinery misbehaves. Options considered:

OptionBytes (backfill burst)Debugging valueVerdict
(a)Skip entirely0Per-tx only (via command_events)Adopted for enumerated one-shot bursts only
(b)Log compact, values stripped~650k × ~90 B ≈ 60 MB + index churnRow-level "something wrote here", no valuesRejected — pays row+index cost for little: burst writes are fully derivable, so the values a debugger wants are reconstructable from the sources' own logged history
(c)Log compact, sampled~1–10 MBStatistical onlyRejected — a sampled forensic log answers neither "what happened to this block" nor "how much happened"
(d)Log normally, provenance-marked~650k × ~350 B ≈ 230 MB/deviceFullAdopted for steady-state derived writes — and the "mark" is free, see below
Recommendation — split by write class, keyed on provenance.
  • Steady-state derived writes log normally (option d): projection, sweep repairs, per-row catch-up. In compact form a 2–3-write fan-out per property edit costs ~0.5 KB total — affordable, and this is precisely the record convergence debugging needs ("which pass rewrote the cell, from what, to what, when"). No new provenance column: every local event already carries tx_id, and command_events (scope, description, mutator_calls; commitPipeline.ts:348-368) identifies the writer — provenance is a join, and a later compaction pass could target it by that join if the volume ever warrants.
  • Enumerated one-shot bulk bursts skip (option a): the props backfill, the arrival catch-up bulk pass, and future whole-table column backfills. Justification: their writes are fully derivable (field rows mirror cell values whose own edit history is logged), each burst still leaves a command_events row per tx, and invariant I1 guarantees the first later edit of any exempted row rebuilds a full base.

5.1 Mechanism: tx_context.history_mode

Add a nullable history_mode TEXT column to tx_context (clientSchema.ts:38-48), set by the commit pipeline from a new optional RepoTxOptions.historyMode (tx.ts:269-277) in step 1 (commitPipeline.ts:261-265) and cleared with the rest in step 5 (:375-383, rollback-safe inside the tx). One value: 'skip'. The row_events triggers add COALESCE((SELECT history_mode FROM tx_context WHERE id = 1), 'full') != 'skip' to their WHEN clause. NULL/absent means full logging — the default cannot silently drop history (invariant I3), sync-applied writes (which never set it) are unaffected, and only a caller that explicitly opts in is exempt (invariant I4: exemption by provenance, never by column — skipping properties_json wholesale would lose primary history for carve-out/un-flipped/orphan keys, per PR #288 §12).

This is the tx-scoped, concurrency-safe generalization of the existing withTriggerSuspended bracket (clientSchema.ts:1187-1200) — which stays bootstrap-only, because dropping a trigger is DB-global and would also unlog concurrent writers mid-session.

6. Sync-source logging, per workspace mode

6.1 Verifying the redundancy claim

Starting position to verify: sync-applied events are redundant for plaintext workspaces because the server-side blocks_history table already records those changes. Read against supabase/migrations/20260522062437_add_blocks_history.sql:

  • All columns, all writers — the trigger fires AFTER INSERT OR UPDATE OR DELETE ON public.blocks FOR EACH ROW (:167-170) and snapshots to_jsonb(NEW/OLD) (:93, :112-113), so every server-side write path — PowerSync CRUD-apply, RPCs, manual SQL — is captured, every column included. Shape is the same "changed-columns diff, full bookends" family as this design (:18-24). ✅
  • Granularity caveat — resolution is per uploaded write, and the client compacts per-id before upload (compactBlockCrudEntries, src/services/powersync.ts:211, used at :638/:672; the migration's own header says so, :5-10). But the local source='sync' events are also collapsed — PowerSync delivers checkpoint states, and the observer additionally skips no-op re-deliveries (materialize.ts:56-60). Server history is at least as fine as what any receiving device could log. ✅ (Per-edit fidelity exists only in the authoring device's source='user' events — unaffected by this section.)
  • E2EE caveat — for e2ee workspaces the server sees and records ciphertext envelopes. Local events on a device holding the key are the only plaintext history. ❌ redundancy does not hold.
  • Availability caveat — blocks_history has no client read path yet: the migration header defers read/restore RPCs to a follow-up (:2-3), RLS grants SELECT to workspace members (:177-192), and no later migration in supabase/migrations/ adds an RPC (verified against the full listing). The redundancy is "the bytes exist server-side", not "a reader exists today". Flagged as open question O1.
  • What is genuinely lost — the local-apply record: when THIS device materialized a change and what it saw. LWW skip decisions are recorded nowhere either way (skippedStale rows never fire triggers), so the marginal forensic loss is apply-timing only.
Decision — per-workspace-mode sync logging. Plaintext workspaces: drop source='sync' events entirely (create, update, soft-delete, delete). E2EE workspaces: keep them, in compact v2 form with anchors like any other events. Unknown/undecided mode: log (fail-safe, invariant I3). Applies to future writes only — the §8 migration never deletes existing sync events (history is kept).
Consequences owned: a plaintext workspace's cross-device timeline is served by the server (RPC pending — O1); e2ee first sync writes one full create per block per device (~1 KB each) — that IS the product cost of local-only e2ee history, and it is also per-device-loss fragile, worth a line in e2ee docs. The comment at clientSchema.ts:55-69 ("the ONLY place an INCOMING sync change is durably recorded") becomes conditional and must be updated in the same PR. Raw out-of-band local writes (agent-bridge sql execute, dev tools — a live path, §2.1) keep logging under this policy (review finding — an earlier draft dropped them as "forbidden anyway"): the drop gate keys on the explicit sync-apply marker (§6.2), never on source IS NULL, precisely because a raw write never uploads — its row_event is the only record of it anywhere, local or server.

6.2 Mechanism: a local policy side-table

Workspace mode is a JS-side fact (durable per-(user, workspace) mode pin, src/sync/keys/modePin.ts:2-16, set-once :104-124; the observer resolves materializability per workspace each pass, materialize.ts:284-291) — triggers can't call it. Bridge it with a tiny local table: 

CREATE TABLE IF NOT EXISTS workspace_history_policy (
  workspace_id TEXT PRIMARY KEY,
  sync_events  TEXT NOT NULL CHECK (sync_events IN ('log', 'drop'))
)

The materializer writes the row at the top of its Phase-2 write tx for each workspace in the batch (mode 'copy' → 'drop', 'decrypt' → 'log'; 'defer' writes nothing — deferred rows aren't materialized anyway), so the policy is committed before the first upsert of that workspace in the same transaction — no ordering race with first sync.

Sync-apply must also be distinguishable from the other NULL-source writers (the raw path, §2.1) — the drop policy applies only to the former. The materializer's Phase-2 bracket, which today sets source = NULL (materialize.ts:344), additionally sets a new nullable tx_context.apply_kind = 'sync', cleared with the bracket. The upload and workspace-invariant trigger gates key on source and are untouched (they must keep skipping sync-apply); only the row_events triggers read apply_kind. Their WHEN clause gains: 

NOT (    (SELECT apply_kind FROM tx_context WHERE id = 1) IS 'sync'
     AND COALESCE((SELECT sync_events FROM workspace_history_policy
                    WHERE workspace_id = NEW.workspace_id), 'log') = 'drop')

— one PK probe per sync write; local writes short-circuit on the IS 'sync' test. Both probes are NULL-safe by construction, and they have to be (review finding): under SQLite's three-valued logic a bare (SELECT sync_events ...) = 'drop' against a missing policy row yields NULL, NOT (TRUE AND NULL) is still NULL, and a NULL WHEN does not fire — silently dropping the first sync events of an unknown/unmigrated workspace, the exact I3 violation the fail-safe default exists to prevent. Hence IS (null-safe equality) for the marker and COALESCE(..., 'log') for the policy row. The row reference is per-trigger (review finding): insert/update triggers read NEW.workspace_id, but blocks_row_event_delete is AFTER DELETE (clientSchema.ts:471-488) where no NEW row exists — its predicate reads OLD.workspace_id, so plaintext stream-exit deletes follow the same policy instead of raising no such column: NEW.workspace_id.

With the marker in place, the v1 label projection COALESCE(source, 'sync') (clientSchema.ts:422) retires in favor of a truthful three-way source tag: 'user' (repo.tx), 'sync' (marker present), 'raw' (neither — always logged, I3). Existing v1 rows keep their historical 'sync' ambiguity. Mode pins are immutable per workspace (modePin.ts:104-124), so the policy row never flips in normal operation; a debugging escape hatch (force-log) is a one-row UPDATE documented in the implementation PR.

7. Write path: triggers stay

Both serious candidates were examined honestly — the app layer is more capable than it first looks, since both major writers already hold before/after snapshots in JS (repo.tx's snapshots map, the observer's SyncSnapshot map, materialize.ts:110-113) and both hold an open write tx, so an app-layer event writer could be atomic. Decision: triggers remain the sole writer. To pay the verbosity once instead of thrice, the implementation generates the v2 trigger bodies from a single column-spec array shared with blockJsonObjectSql (:386-402) and BLOCK_UPLOAD_COLUMNS (:518-534) — which also discharges PR #288's slice-A warning that the row_events serialization is a hand-maintained parallel of BLOCK_STORAGE_COLUMNS that must be extended in lockstep (its §11 slice A, "history/undo seam"). Per-write cost of v2: the same JSON build machinery as today minus one full snapshot side, plus one O(log n) predecessor probe and one PK policy probe — strictly less work than v1 for every non-anchor update (invariant I6).

8. Migrating the existing ~1.17M events

Required, not optional: the stock of old full-snapshot events is where the multi-GB sits. Direction fixed by the frame: rewrite old events into compact form, sampling old before_json values into anchors rather than dropping them wholesale; delete nothing.

8.1 The rewrite, per block

Unit of work: one block's v1 events, in id order (per-block processing keeps the running reconstruction state at one row, ~2 KB — processing in global id order would need a 350k-entry state map). For each block with any v IS NULL event:

  1. Stream its v1 events ascending. Maintain runningState (the reconstructed after of the previous kept event) and pos (compact count since the last full event).
  2. create/delete: stamp v=2, chain_pos=0, payload unchanged. Reset pos.
  3. update/soft-delete: compute changed fields by comparing the event's own before/after JSON (exact — no cross-event inference), then keep it full (anchor) if any of:
    • chain start — no prior event for the block (I1);
    • cadence — pos ≥ W−1;
    • discontinuity — the event's before_json ≠ runningState. This is the load-bearing sampling rule: v1 before is normally redundant with the previous event's after, but where the log has gaps (trigger-suspended backfill windows, the pre-row_events era of a block, historical bugs) it is not — anchoring exactly at discontinuities keeps those seams reconstructable and visible instead of silently splicing them.
    Otherwise rewrite compact: after_json = changed fields, before_json = NULL, v=2, chain_pos=++pos. An all-fields-identical v1 update (the v1 trigger logged no-ops) becomes an empty compact event — kept, ~60 B, preserving "an UPDATE happened" timing.
  4. Mark the batch done by the rewrite itself: v flips 2.
The migration is reconstruction-lossless (invariant I5): for every event e, stateAt(e) returns the same value before and after the rewrite — the only bytes dropped are ones derivable from neighboring events, and every non-derivable before is retained as an anchor. Pinned by an equivalence test: seed a DB with randomized edit histories (including deliberate gaps), snapshot stateAt for every event, migrate, compare.

8.2 Resumability, scheduling, coexistence

  • Progress marker is the format tag. WHERE v IS NULL is the work queue; a partial index ON row_events(block_id) WHERE v IS NULL makes claiming the next blocks cheap and shrinks to nothing as the job completes (dropped at the end). Interruption at any point (crash, tab close) is safe: rewritten rows are self-describingly done, unrewritten rows self-describingly not — no cursor to corrupt. A client_schema_state marker (row_events_compact_migration_v1) records only overall completion, gating the §9 VACUUM and the old-index retirement decision (§3).
  • Chunking: a few hundred blocks (or ~2–5k events) per write tx, scheduled via PendingIdleJobs + scheduleDeepIdle (idleMarkerJobs.ts) like the other catch-up passes — never on the boot or first-paint path. Each chunk holds the single writer briefly; live writes queue behind it, as they do behind any tx.
  • Live-write coexistence is free by construction: §4's trigger anchors whenever the predecessor is v1, so no live v2 chain ever references a v1 base — the rewriter may compact any v1 event without coordination. Within a block, all v1 events predate all v2 events (v1 stops being written the moment slice A deploys).
  • Ordering with the ALTERs: the column adds run in initializePowerSyncDb before the CLIENT_SCHEMA_STATEMENTS loop recreates triggers referencing them — same constraint and same slot as ensureUndoGroupIdColumns (repoProvider.ts:352-357 area; documented footgun at clientSchema.ts:1384-1391).
  • The old-index retirement decision (§3) runs after the rewrite completes (the (block_id, id) add itself happened in slice A), then one explicit ANALYZE: the automatic drift check keys on blocks counts only (analyzeIsWarranted, clientSchema.ts:1438-1455) and would never notice row_events index changes.

9. VACUUM: making the bytes come back

The rewrite shrinks rows in place; SQLite reuses the slack for future inserts but the file never shrinks — and after compaction most pages are partially filled, which is exactly what incremental_vacuum cannot help with (it releases only wholly-free pages, and auto_vacuum is off on every existing DB and cannot be enabled without a full VACUUM anyway). A one-time full VACUUM is the honest tool.

Two wa-sqlite/OPFS-specific hazards to design around:
  • temp_store = MEMORY (repoProvider.ts:339): SQLite's VACUUM builds the rebuilt database in its temp store — with MEMORY that means holding a multi-GB rebuild in RAM. Bracket the VACUUM with PRAGMA temp_store = FILE (restore after), and verify at implementation that the OPFS VFS serves temp files sanely; if not, fall back to the VACUUM-INTO variant below.
  • Single worker, minutes-long stall: a multi-GB rebuild at OPFS speeds blocks every query. Run only at deep idle (scheduleDeepIdle), gated on the §8 completion marker, a one-shot client_schema_state marker of its own, and a free-space check sized against the worst case, not the target (review finding): SQLite documents VACUUM as needing up to twice the size of the original database file (the temp rebuild plus the rollback journal protecting the overwrite), so gate on navigator.storage.estimate() headroom ≥ ~2× the current file size. Gating on ~1.2× the post-compaction size would greenlight a 2+ GB→0.6 GB VACUUM with a few hundred MB of quota, stall for minutes, and die with SQLITE_FULL. If quota can't cover 2×-current, skip in-place VACUUM and take the VACUUM INTO route below — it needs only ~the compacted size and never journals the live file. (Rollback-journal mode makes an interrupted VACUUM roll back safely either way, repoProvider.ts:312-316.)

Ship it in two forms, mirroring the ANALYZE precedent (automatic drift-gated pass + manual override command in src/plugins/db-maintenance/plugin.ts:1-23): the deep-idle one-shot, plus a command-palette "Compact database" action with a size/duration toast for users who want it now. Fallback variant if in-place VACUUM proves fragile on this VFS: VACUUM INTO a staging OPFS file + swap at next boot — the swap machinery already exists for DB import/recovery (src/utils/exportSqliteDb.ts, localDbRecovery.ts); costs more moving parts, bounds memory, never touches the live file.

11. Contract with PR #288 (the slice-C gate)

What this design guarantees to the props migration, in its §12 terms:

  1. G1 — authoring device: the backfill runs its repo.tx batches with historyMode: 'skip' → ~650k creates write zero row_events rows (vs ~230 MB + index churn). command_events still records every backfill tx.
  2. G2 — receiving devices: backfilled rows arrive as sync-applied creates; plaintext workspaces drop them (§6) → zero rows per device. E2EE workspaces are gated out of the backfill by #288 itself, so no e2ee arrival burst exists.
  3. G3 — no stranded chains: the first later edit of any exempted row promotes to a full anchor (I1) — "every history chain starts full" holds under every exemption above.
  4. G4 — steady-state fan-out priced: a property edit's 2–3 row writes log as compact events (~0.5 KB total), not 2–3 full snapshot pairs (~4–6 KB).
  5. G5 — ordering: #288 slice C requires this project's slices A+B only (format + provenance/sync policy). The §8 migration and §9 VACUUM (slices C/D) may trail — they concern the existing stock, not the new burst. Matches #288's "prerequisite for slice C, not slices A–B".

12. Storage arithmetic

15. Rejected alternatives

  • Pruning (keep-recent + grace window) — the row-events-retention.md design. Rejected by the frame: history is a product feature to keep, and pruning also would not have reduced the per-write cost that motivated that doc.
  • Server-shape adoption without anchors (changed-cols diffs, full bookends only — exactly blocks_history's Option B). Reconstruction becomes full-chain replay back to the create; a single lost row poisons everything after it. Anchors cost ~2% of rows and bound both.
  • Probabilistic or counter-table cadence — §4.1 callout.
  • App-layer event writer — §7 table; loses the catch-all property that is the log's reason to exist.
  • Values-free or sampled logging for the bulk bursts — §5 table rows (b)/(c).
  • Column-keyed exemptions (e.g. never log properties_json) — violates I4; loses the only history for cell-primary keys and the whole pre-flip period (PR #288 §12).
  • Sub-key diffs of properties_json — real wins today (property edits rewrite the whole cell), but it complicates every reader and the trigger, and the props migration is about to shrink the cell structurally; column-granular is right-sized for the post-#288 world. Revisit only if #288 stalls.
  • Compression (per-event zlib, content dedup) — wa-sqlite ships no compression extension; JS-side compression forces an app-layer writer (see above). The compact format gets the structural win without new machinery.
  • History in a separate ATTACHed database file — would make VACUUM trivial (vacuum the side file, or drop it), but SQLite triggers cannot write across databases, so it forces the app-layer writer; rejected with it.
  • Retroactively applying the §6 sync-drop to old events — deleting existing sync history contradicts "history is kept"; the migration compacts it instead.

16. Open questions

  1. O1 — plaintext sync-drop before a blocks_history reader exists. §6 drops local sync events on the strength of server-side bytes with no client RPC yet. Lean: accept — the server data is durable and RLS-readable the day the RPC lands, and the reclaimed per-device cost is the second-largest item in §12. Alternative if unacceptable: hold the drop (log plaintext sync compact) until the RPC ships, keeping everything else; costs ~0 for updates but re-admits the first-sync create burst. Needs a user call.
  2. O2 — auto-VACUUM at deep idle vs manual-first. A minutes-long single-worker stall triggered by a heuristic is the riskiest UX moment in this design. Lean: ship the manual command in slice D, enable the deep-idle one-shot a release later once the manual path has telemetry.
  3. O3 — W=50 confirmation — §4.4 has the tradeoff curve; changing it later is free. Default stands unless someone objects.
  4. O4 — e2ee first-sync cost — full creates per block per device is the price of local-only e2ee history (§6); confirm that's the intended product posture (it also means e2ee history has per-device durability only).
  5. O5 — cadence mechanism under challenge: probe vs probabilistic (and with it, the compact payload shape). Post-review user position: W=50 was an arbitrary starting point, and a per-write history probe feels like write-path work we shouldn't pay — a probabilistic ~1-in-25 anchor would probably be good enough. Assessment: against the §3 index the probe is genuinely cheap (one covering point-seek, microseconds next to the JSON build), so performance alone doesn't decide this — but naively swapping §4.1 for random() is not safe: the probe is also what detects missing/v1 predecessors, so I1 ("every chain starts full") and the #288 G3 guarantee die with it. The coherent probe-free variant changes the payload instead: compact events carry changed-field before AND after values (invertible patches), reconstruction runs backward from the live row (or from the nearest full event in either direction), anchors demote to pure read-acceleration + corruption bounding at probabilistic ~1-in-25, and chain_pos, the predecessor probe, the v1-lean rule, and slice A's index-timing dependency (§3) all disappear. It also fixes a soundness gap the forward design has under §6's own drops: after an unlogged sync write, forward replay silently splices stale pre-sync values into reconstructed states until the next anchor, while backward-from-live is exact at and after every gap (staleness is confined to states older than the gap — where plaintext workspaces defer to server history anyway). Bonus: undo's restore values become directly stored rather than reconstructed. Cost: ~2× compact-update bytes (~400 B vs ~200 B; §12 stock-update row ~180 MB → ~330 MB — still ~10× under v1), and deterministic walk/corruption bounds become probabilistic (p99 anchor gap ≈ 113 events at 1/25). Needs a user call; on adoption §3/§4/§8/§12 get rewritten around it.
