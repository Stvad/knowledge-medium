row_events: compact local history with sampled anchors

Keep the full per-row history log, shrink it ~3×: invertible changed-columns events (before + after per changed field) with sampled full-snapshot anchors, provenance-gated logging for bulk derivation bursts, sync-applied events kept (compact) for local self-sufficiency, and a reconstruction-lossless migration of the existing ~1.17M full-snapshot events — followed by the VACUUM that actually returns the bytes.

How to read this doc

The load-bearing core is §3–§4 (the event format and the anchor/reconstruction machinery) and §8 (the migration of existing history). §5 and §6 resolve the two policy questions (derived-write logging, sync-source logging). If you are implementing, start from your slice in §13 and the invariants index. If you are PR #288, read §11 — it is the contract your slice C consumes. SQL in this doc is illustrative shape, not implementation.

Decisions made in this doc (each argued in its section):
  • Compact updates are invertible, column-granular patches: an update event stores the changed domain fields' before and after values (equal key sets); identical updates are not logged (§3).
  • Anchors are sampled — one in-trigger coin flip at p = 1/64, no write-path reads (adopted per O5, superseding a deterministic probe + counter design, §15). Anchors carry full before+after and are pure read-acceleration + corruption-bounding: no correctness property depends on them (§4).
  • Every event is self-restoring; reconstruction never leans on a predecessor: walks terminate on the newer side unconditionally — live row, delete event, or anchor (§4.3). The old "every chain starts full" promotion machinery is unnecessary (§4.1).
  • Derived writes log normally in compact form; only enumerated one-shot bulk bursts skip, via a tx-scoped tx_context.history_mode opt-in — provenance-keyed, never column-keyed (§5).
  • Sync-applied events are kept, compact, for every workspace (user call 2026-07-15, superseding an earlier per-workspace-mode drop): local history stays self-sufficient instead of leaning on a server read path that doesn't exist yet. The server-redundancy analysis is retained in §6 for the record; an explicit sync-apply marker still distinguishes sync from raw writes, now for truthful labeling only.
  • Triggers remain the writer; bodies are generated from a single column spec shared with the existing snapshot/upload projections (§7).
  • The migration is reconstruction-lossless: chunked, resumable via the format tag itself, sampling anchors at cadence and at chain discontinuities, so every state derivable today stays derivable (§8).
  • One-time full VACUUM (deep-idle + manual command), not incremental_vacuum — in-place row rewrites leave half-full pages only a rebuild compacts (§9).

1. Why now

row_events is the client-side per-row audit log: three triggers on blocks write one event per row write, for local and sync-applied writes alike (src/data/internals/clientSchema.ts:370-381). Updates store two full row snapshots — before and after — including the whole properties_json and references_json columns (blockJsonObjectSql, clientSchema.ts:386-402). Nothing reads the table at runtime (clientSchema.ts:63-65); it is intentionally unbounded and never trimmed (clientSchema.ts:67-69), and there is no client-side VACUUM anywhere in src/, so even a future deletion would not return OPFS bytes.

Observed scale and the two forcing functions:

  • It is already the biggest thing in the database. ~1.17M events × ~2 KB on a dev instance (docs/row-events-retention.md:4) ≈ 2+ GB of OPFS for a ~300–350k-block workspace — history bytes rivaling or exceeding live-data bytes, on devices where OPFS quota is finite.
  • PR #288 multiplies it. The properties-as-blocks backfill creates ~650k rows through repo.tx per workspace (its §12 and measurement section; the spike took 5–10 minutes largely paying row_events + ps_crud envelopes), and those rows then sync down and append events again on every receiving device. Its §12 carries the constraint record: the optimization must land before its slice C runs. §11 states the contract.
  • Every fresh device pays on first sync. The observer materializes each downloaded block into blocks (materialize.ts:365), firing the INSERT trigger once per block: ~350k full-snapshot create events (~350 MB) per device, recording rows the server already has. (§6 keeps these — an accepted cost of self-sufficient local history; creates are inherently full, so v2 doesn't shrink them either.)

The consumers this format serves are all future — block history / time-travel UI, drift-tolerant undo (src/data/internals/undoManager.ts:26-28 names row_events.before_json as the substrate; today's undo is in-memory only, depth 100 at undoManager.ts:80-82), forensics after incidents (the SRS backlink-wipe class), and convergence debugging for the props migration. Because nothing reads the log yet, we are free to change the format — the design is judged by what those readers need (§10), not by compatibility.

  • No runtime reader. The only statements touching row_events in non-test src/ are the three trigger INSERTs (verified by search). The old row_events→invalidation tail (rowEventsTail.ts) is deleted; only historical comments mention it (observer.ts:32, invalidate.ts:6, block.ts:118).

  • docs/row-events-retention.md is stale: it cites the deleted rowEventsTail.ts drain, a local-ephemeral source that no longer exists, and a prune design ("keep recent 50k + 7-day grace") superseded by the keep-history direction. Re-stamped as superseded in this PR.

Change detection compares storage columns (OLD.x IS NOT NEW.x — the exact predicate the upload trigger already uses, clientSchema.ts:542-544) but emits domain keys. Column-granular only: a changed properties_json stores the whole new cell value, not a sub-key diff — see §15 for why, and note the props-as-blocks migration shrinks that very column structurally. One v1 nuance carries over with a v2-specific consequence (review finding): userUpdatedAt is serialized as coalesce(user_updated_at, updated_at) (clientSchema.ts:397), and for a projected field the diff must be projected too — when user_updated_at is NULL, an update that changes only updated_at changes the domain value while the storage column is untouched, so a storage-only OLD.x IS NOT NEW.x key set would omit userUpdatedAt and replay would reconstruct it stale. The v2 trigger keys this field on coalesce(OLD.user_updated_at, OLD.updated_at) IS NOT coalesce(NEW.user_updated_at, NEW.updated_at) and emits the coalesced value (the shared column spec carries an optional domain-projection expression used for both the diff and the emit). The projection still accepts that a (never-occurring today) explicit clear is not distinguishable.

Index changes — none on the write path: the v2 trigger reads nothing (§4.1), so slice A ships no index work at all. Slice C adds (block_id, id) for reader walks and the §8 rewriter — id is AUTOINCREMENT-monotonic and is the authoritative per-block order (same-millisecond events in one tx tie on created_at). The existing (block_id, created_at DESC) index is retained (review finding): per-block time-seek ("latest event ≤ T") has no sound substitute — id order is insertion order, and created_at is not reliably monotone in id under wall-clock steps, so bisecting created_at over an id-ordered index can pick a wrong event or degrade to a scan (see the subtree reader row in §10). (created_at DESC) and (tx_id) stay for time-scoped and tx-join forensics. (Two earlier revisions of this paragraph — moving the id-index add into slice A as an idle job gating the trigger swap — existed solely for the deterministic probe's benefit; moot since O5's adoption.)

4. Anchors: invertible payloads, sampled full snapshots

4.1 Decision — invertible compact events; anchors by coin flip

Adopted 2026-07-15 (user call; decision record in O5, superseding the deterministic predecessor-probe design — its record is in §15). Two properties define the format:

  • Every compact event is invertible. before_json and after_json carry the same key set — the changed fields — with old and new values respectively. Applying after_json onto the pre-state, or un-applying before_json from the post-state, are both single-object merges. No event's interpretation or inverse depends on any other row existing (invariant I1).
  • Anchors are sampled, not scheduled. On each logged update the trigger flips one coin (random(), p = 1/64 — argued in §4.4); a hit writes the event full (complete before + after, full = 1) instead of compact. No probe, no counter, no reads of any kind on the write path (invariant I6). Creates and hard deletes are full by nature.

What the deleted probe machinery used to buy, and why it is no longer needed:

  • Chain-start promotion (the old I1, "every chain starts full") — chains no longer have bases: reconstruction walks from the newer side, which always terminates (§4.3), and every event restores its own fields by itself. The first edit after any unlogged gap (§5 skip, trigger-suspended window) is fully usable the moment it lands, including for undo.
  • v1-predecessor promotion — v2 events never reference predecessors, so the §8 rewriter keeps its zero-coordination property with the machinery deleted rather than because of it.
  • The deterministic walk bound W — replaced by an expected bound with a geometric tail (§4.4). Acceptable because walks cost microseconds per event, anchors carry no correctness weight, and the current state is never at risk.
Implementation trap — one coin per event. The trigger body needs the anchor decision in three places (before side, after side, the full column); three separate random() calls would disagree (e.g. a full before paired with a compact after). Compute the flip once and reference it — e.g. INSERT INTO row_events ... SELECT ... FROM (SELECT (abs(random()) % 64) = 0 AS anchor). Pin with a shape test: both payload sides of every event agree on key set / fullness.

4.2 Anchor content — decision

Anchors carry full before + full after (not after-only):

  • Integrity check for free: replaying the compact run from the previous full event must reproduce anchor.before_json. The §8 migration's equivalence test and any future integrity sweep get a built-in checkpoint at every anchor.
  • One-sided recovery: if the run before an anchor is lost, the anchor's before still yields the state just prior to it.
  • Symmetric bases: a full event serves forward walks (its after side) and backward walks (its before side) equally — §4.3 picks whichever full state is nearer.
  • Cost is noise: one extra full pair per 1/p events ≈ +25 B amortized per update at p = 1/64 (§4.4).

4.3 Reconstruction: bidirectional, the newer side always terminates

stateAt(blockId, eventId):                # row state just after event E
  E = event(eventId)
  if E.v is NULL or E.full:
      return per-kind full side           # create/anchor: after; delete: before

  # nearest full state on EITHER side, whichever is closer:
  #   older side: newest event ≤ E with a full after (create, anchor)
  #               → walk FORWARD applying after_json patches
  #   newer side: oldest event ≥ E with a full before (anchor, delete),
  #               or the live blocks row itself
  #               → walk BACKWARD un-applying before_json patches
  #
  # The newer side ALWAYS terminates: a live block ends at its blocks row,
  # a purged block at its delete event (full before). ChainBroken can only
  # mean corruption inside the log — see the failure bound.

Complexity: expected walk = distance to the nearest full state on either side — E[one-sided gap] = 1/p = 64 events, so expected walk ≲ 32, p99 one-sided ≈ 4.6/p ≈ 295 (§4.4); each step is one ~0.4 KB JSON parse + ~3-field merge, microseconds. A whole-block timeline streams once carrying running state, either direction: O(events of block). Recent history — the common UI case — is effectively free and exact: it walks backward from the live row through trigger-recorded OLD values, so it is correct even across unlogged gaps (§5 skips); a gap only degrades states older than itself, back to the next-older full state.

Failure bound. Corruption severing the log confines damage to states older than the break, back to the next-older full state — expected ~1/p events, p99 ≈ 4.6/p. Everything at or newer than the break reconstructs from the live side, and the current state is never at risk (it lives in blocks). Anchor before-sides double as integrity checkpoints: replaying the span between two full states must reproduce the newer one's before_json.

4.4 Choosing p — a checkpoint-spacing problem

Under this format anchors carry no correctness weight — every property in §4.3 holds with zero anchors — so p is chosen economically, as in the classic checkpoint-interval tradeoff:

  • Cost is linear in p. An anchor adds Δ ≈ 1.6 KB over a compact event (~2 KB full pair vs ~0.4 KB patch pair). Per-update overhead = p·Δ: ~64 B (+16%) at 1/25, ~25 B (+6%) at 1/64, ~13 B (+3%) at 1/128.
  • Read latency does not bind. Expected point-lookup walk ~1/(2p) events at microseconds each; even 1/p in the several hundreds keeps far-past lookups sub-millisecond, and the common recent-history reads walk from the live row and touch no anchor at all. Optimizing p for reads would push it toward zero — the user's "plausibly rarer" instinct is right.
  • The binding constraint is the loss window (and with it, integrity-checkpoint granularity): how much of a block's older history may become unreconstructable when corruption severs its chain, and how tightly an integrity sweep can localize a divergence (one inter-anchor span). Gaps are geometric, so to guarantee a p99 window ≤ L events, set p ≈ ln(100)/L ≈ 4.6/L.

Default: p = 1/64 — expected gap 64, p99 ≈ 295, +6% update bytes: "99% of corruption-loss windows and far-past walks stay under ~300 events" at byte cost that is noise against the creates dominating §12. Rarer is defensible (1/128 → p99 ≈ 590, still microseconds to walk, still bounded per block by its create); denser buys only checkpoint granularity. The constant lives in the trigger SQL, re-installed every boot (clientSchema.ts:1170-1175); mixed rates coexist freely since readers key on full, never on cadence. Revisit with history-UI telemetry — observed walk lengths are the one number that settles this empirically.

5. Derived-write logging — analysis and recommendation

The props migration introduces machine writers: the one-shot backfill (~650k creates), the children→cell projection (2–3 row writes per property edit), the consistency sweep, and arrival catch-up. The question: do their writes belong in the history log? The earlier "skip them all via a tx_context-gated trigger condition" suggestion trades away exactly the record you want when the machinery misbehaves. Options considered:

OptionBytes (backfill burst)Debugging valueVerdict
(a)Skip entirely0Per-tx only (via command_events)Adopted for enumerated one-shot bursts only
(b)Log compact, values stripped~650k × ~90 B ≈ 60 MB + index churnRow-level "something wrote here", no valuesRejected — pays row+index cost for little: burst writes are fully derivable, so the values a debugger wants are reconstructable from the sources' own logged history
(c)Log compact, sampled~1–10 MBStatistical onlyRejected — a sampled forensic log answers neither "what happened to this block" nor "how much happened"
(d)Log normally, provenance-marked~650k × ~350 B ≈ 230 MB/deviceFullAdopted for steady-state derived writes — and the "mark" is free, see below
Recommendation — split by write class, keyed on provenance.
  • Steady-state derived writes log normally (option d): projection, sweep repairs, per-row catch-up. In compact form a 2–3-write fan-out per property edit costs ~1 KB total — affordable, and this is precisely the record convergence debugging needs ("which pass rewrote the cell, from what, to what, when"). No new provenance column: every local event already carries tx_id, and command_events (scope, description, mutator_calls; commitPipeline.ts:348-368) identifies the writer — provenance is a join, and a later compaction pass could target it by that join if the volume ever warrants.
  • Enumerated one-shot bulk bursts skip (option a): the props backfill, the arrival catch-up bulk pass, and future whole-table column backfills. Justification: their writes are fully derivable (field rows mirror cell values whose own edit history is logged), each burst still leaves a command_events row per tx, and invariant I1 guarantees the first later edit of any exempted row rebuilds a full base.

5.1 Mechanism: tx_context.history_mode

Add a nullable history_mode TEXT column to tx_context (clientSchema.ts:38-48), set by the commit pipeline from a new optional RepoTxOptions.historyMode (tx.ts:269-277) in step 1 (commitPipeline.ts:261-265) and cleared with the rest in step 5 (:375-383, rollback-safe inside the tx). One value: 'skip'. The row_events triggers add COALESCE((SELECT history_mode FROM tx_context WHERE id = 1), 'full') != 'skip' to their WHEN clause. NULL/absent means full logging — the default cannot silently drop history (invariant I3), sync-applied writes (which never set it) are unaffected, and only a caller that explicitly opts in is exempt (invariant I4: exemption by provenance, never by column — skipping properties_json wholesale would lose primary history for carve-out/un-flipped/orphan keys, per PR #288 §12).

This is the tx-scoped, concurrency-safe generalization of the existing withTriggerSuspended bracket (clientSchema.ts:1187-1200) — which stays bootstrap-only, because dropping a trigger is DB-global and would also unlog concurrent writers mid-session.

6. Sync-source logging: kept for every workspace (revised)

Revision note (2026-07-15, user call): an earlier draft of this section dropped sync-applied events for plaintext workspaces on the strength of the server-side redundancy analyzed below. The user overrode it — sync events are kept, so local history stays self-sufficient. §6.1 is retained as the record of what the server does and doesn't cover; §6.2 keeps the one mechanism that survives the revision (the sync-apply marker, now labeling-only).

6.1 The server-side redundancy analysis (kept for the record)

Position examined: sync-applied events are redundant for plaintext workspaces because the server-side blocks_history table already records those changes. Read against supabase/migrations/20260522062437_add_blocks_history.sql:

  • All columns, all writers — the trigger fires AFTER INSERT OR UPDATE OR DELETE ON public.blocks FOR EACH ROW (:167-170) and snapshots to_jsonb(NEW/OLD) (:93, :112-113), so every server-side write path — PowerSync CRUD-apply, RPCs, manual SQL — is captured, every column included. Shape is the same "changed-columns diff, full bookends" family as this design (:18-24). ✅
  • Granularity caveat — resolution is per uploaded write, and the client compacts per-id before upload (compactBlockCrudEntries, src/services/powersync.ts:211, used at :638/:672; the migration's own header says so, :5-10). But the local source='sync' events are also collapsed — PowerSync delivers checkpoint states, and the observer additionally skips no-op re-deliveries (materialize.ts:56-60). Server history is at least as fine as what any receiving device could log. ✅ (Per-edit fidelity exists only in the authoring device's source='user' events — unaffected by this section.)
  • E2EE caveat — for e2ee workspaces the server sees and records ciphertext envelopes. The history bytes are as durable as any e2ee data (server-side, ciphertext); what a key-holding device's local events provide is the only readable-today plaintext copy — an access gap, not a durability gap. ❌ redundancy does not hold at the access level. (An earlier draft called this "local-only e2ee history" / "per-device durability only" — wrong framing, corrected per user.)
  • Availability caveat — blocks_history has no client read path yet: the migration header defers read/restore RPCs to a follow-up (:2-3), RLS grants SELECT to workspace members (:177-192), and no later migration in supabase/migrations/ adds an RPC (verified against the full listing). The redundancy is "the bytes exist server-side", not "a reader exists today". This was open question O1; keeping sync events (the decision below) resolves it — local history no longer depends on the missing RPC.
  • What is genuinely lost — the local-apply record: when THIS device materialized a change and what it saw. LWW skip decisions are recorded nowhere either way (skippedStale rows never fire triggers), so the marginal forensic loss is apply-timing only.
Decision (revised 2026-07-15, user call) — sync-applied events are kept, for every workspace, in compact v2 form with anchors like any other events. Rationale: local history should be self-sufficient — readable offline, without a server round-trip, and not gated on a blocks_history read RPC that doesn't exist yet (former O1, now moot). The server-side redundancy above is real at the byte level but is an access story, not a local-history substitute.
Consequences owned:
  • Fresh-device first sync logs ~350k full creates (~350 MB/device) — creates are inherently full, v2 cannot shrink them (§12). This was the second-largest byte win of the earlier draft; deliberately given back.
  • #288's backfill arrival burst logs on receiving devices (~650k full creates ≈ ~230 MB one-time per device) unless #288 adds its own enumerated arrival exemption — G2 in §11 is renegotiated accordingly.
  • The comment at clientSchema.ts:55-69 ("the ONLY place an INCOMING sync change is durably recorded") stays unconditionally true — no update needed for this section (slice B touches it only for the format/provenance notes).
  • Raw out-of-band local writes (agent-bridge sql execute, dev tools — a live path, §2.1) also keep logging, labeled truthfully via the §6.2 marker — a raw write never uploads, so its row_event is the only record of it anywhere.

6.2 Mechanism: the sync-apply marker (labeling only)

With no drop policy, the workspace_history_policy side-table, the materializer's Phase-2 policy writes, and the trigger drop gates of the earlier draft are all deleted from the design. What survives is the marker that distinguishes sync-apply from the other NULL-source writers (the raw path, §2.1) — now purely for truthful attribution, no longer load-bearing for any gate: the materializer's Phase-2 bracket, which today sets source = NULL (materialize.ts:344), additionally sets a new nullable tx_context.apply_kind = 'sync', cleared with the bracket. The upload and workspace-invariant trigger gates key on source and are untouched (they must keep skipping sync-apply); only the row_events triggers' label projection reads apply_kind: the v1 COALESCE(source, 'sync') (clientSchema.ts:422) retires in favor of a truthful three-way source tag — 'user' (repo.tx), 'sync' (marker present), 'raw' (neither — the unguarded paths, always logged, I3). Existing v1 rows keep their historical 'sync' ambiguity.

Two review lessons from the deleted drop gate stay on the books as review rules (they still bind the §5.1 history_mode gate and any future trigger predicate): every WHEN predicate must be NULL-safe under SQLite's three-valued logic — a bare (subquery) = 'x' against a missing row yields NULL, NOT (TRUE AND NULL) is still NULL, and a NULL WHEN silently skips the INSERT, i.e. silently drops history (hence IS / COALESCE, as §5.1 already does) — and row references are per-trigger: blocks_row_event_delete is AFTER DELETE (clientSchema.ts:471-488) and has no NEW row, so any predicate shared across the three triggers must read NEW in insert/update and OLD in delete.

7. Write path: triggers stay

Both serious candidates were examined honestly — the app layer is more capable than it first looks, since both major writers already hold before/after snapshots in JS (repo.tx's snapshots map, the observer's SyncSnapshot map, materialize.ts:110-113) and both hold an open write tx, so an app-layer event writer could be atomic. 8. Migrating the existing ~1.17M events

Required, not optional: the stock of old full-snapshot events is where the multi-GB sits. Direction fixed by the frame: rewrite old events into compact form, sampling old before_json values into anchors rather than dropping them wholesale; delete nothing.

8.1 The rewrite, per block

Unit of work: one block's v1 events, in id order (per-block processing keeps the running reconstruction state at one row, ~2 KB — processing in global id order would need a 350k-entry state map). For each block with any v IS NULL event:

  1. Stream its v1 events ascending. Maintain runningState (the reconstructed after of the previous kept event) and pos (compact count since the last full event — a rewriter-local counter; nothing in the output encodes it).
  2. create/delete: stamp v=2, full=1, payload unchanged. Reset pos.
  3. update/soft-delete: compute changed fields by comparing the event's own before/after JSON (exact — no cross-event inference), then keep it full (anchor, full=1) if any of:
    • no runningState — the block's history doesn't open with a create (pre-log era, historical gaps);
    • cadence — pos ≥ 63 (deterministic every-64th in the rewriter — it runs offline, no coin needed, and readers key on full, never on spacing);
    • discontinuity — the event's before_json ≠ runningState. This is the load-bearing sampling rule: v1 before is normally redundant with the previous event's after, but where the log has gaps (trigger-suspended backfill windows, the pre-row_events era of a block, historical bugs) it is not — anchoring exactly at discontinuities keeps those seams reconstructable and visible instead of silently splicing them.
    Otherwise rewrite compact: before_json = changed fields' old values, after_json = changed fields' new values (equal key sets — the invertible-patch shape of §3), v=2, full=0. An all-fields-identical v1 update (the v1 trigger logged no-ops) becomes an empty compact event — kept, ~60 B, preserving "an UPDATE happened" timing.
  4. Mark the batch done by the rewrite itself: v flips 2.
The migration is reconstruction-lossless (invariant I5): for every event e, stateAt(e) returns the same value before and after the rewrite — the only bytes dropped are ones derivable from neighboring events, and every non-derivable before is retained as an anchor. Pinned by an equivalence test: seed a DB with randomized edit histories (including deliberate gaps), snapshot stateAt for every event, migrate, compare.

8.2 Resumability, scheduling, coexistence

  • Progress marker is the format tag. WHERE v IS NULL is the work queue; a partial index ON row_events(block_id) WHERE v IS NULL makes claiming the next blocks cheap and shrinks to nothing as the job completes (dropped at the end). Interruption at any point (crash, tab close) is safe: rewritten rows are self-describingly done, unrewritten rows self-describingly not — no cursor to corrupt. A client_schema_state marker (row_events_compact_migration_v1) records only overall completion, gating the §9 VACUUM and any §3 index revisit.
  • Chunking: a few hundred blocks (or ~2–5k events) per write tx, scheduled via PendingIdleJobs + scheduleDeepIdle (idleMarkerJobs.ts) like the other catch-up passes — never on the boot or first-paint path. Each chunk holds the single writer briefly; live writes queue behind it, as they do behind any tx.
  • Live-write coexistence is free by construction: v2 events never reference predecessors at all (§4.1), so the rewriter may compact any v1 event without coordination — and I5 preserves stateAt for every event, so a reader walking across the v1/v2 boundary mid-migration stays sound. Within a block, all v1 events predate all v2 events (v1 stops being written the moment slice A deploys).
  • Ordering with the ALTERs: the column adds run in initializePowerSyncDb before the CLIENT_SCHEMA_STATEMENTS loop recreates triggers referencing them — same constraint and same slot as ensureUndoGroupIdColumns (repoProvider.ts:352-357 area; documented footgun at clientSchema.ts:1384-1391).
  • One explicit ANALYZE after the rewrite completes (the (block_id, id) add happened in slice A; (block_id, created_at DESC) is retained per §3 — any measurement-driven revisit also lands here): the automatic drift check keys on blocks counts only (analyzeIsWarranted, clientSchema.ts:1438-1455) and would never notice row_events index changes.

9. VACUUM: making the bytes come back

The rewrite shrinks rows in place; SQLite reuses the slack for future inserts but the file never shrinks — and after compaction most pages are partially filled, which is exactly what incremental_vacuum cannot help with (it releases only wholly-free pages, and auto_vacuum is off on every existing DB and cannot be enabled without a full VACUUM anyway). A one-time full VACUUM is the honest tool.

Two wa-sqlite/OPFS-specific hazards to design around:
  • temp_store = MEMORY (repoProvider.ts:339): SQLite's VACUUM builds the rebuilt database in its temp store — with MEMORY that means holding a multi-GB rebuild in RAM. Bracket the VACUUM with PRAGMA temp_store = FILE (restore after), and verify at implementation that the OPFS VFS serves temp files sanely; if not, fall back to the VACUUM-INTO variant below.
  • Single worker, minutes-long stall: a multi-GB rebuild at OPFS speeds blocks every query. Run only at deep idle (scheduleDeepIdle), gated on the §8 completion marker, a one-shot client_schema_state marker of its own, and a free-space check sized against the worst case, not the target (review finding): SQLite documents VACUUM as needing up to twice the size of the original database file (the temp rebuild plus the rollback journal protecting the overwrite), so gate on navigator.storage.estimate() headroom ≥ ~2× the current file size. Gating on ~1.2× the post-compaction size would greenlight a 2+ GB→0.6 GB VACUUM with a few hundred MB of quota, stall for minutes, and die with SQLITE_FULL. If quota can't cover 2×-current, skip in-place VACUUM and take the VACUUM INTO route below — it needs only ~the compacted size and never journals the live file. (Rollback-journal mode makes an interrupted VACUUM roll back safely either way, repoProvider.ts:312-316.)

Ship it in two forms, mirroring the ANALYZE precedent (automatic drift-gated pass + manual override command in src/plugins/db-maintenance/plugin.ts:1-23): the deep-idle one-shot, plus a command-palette "Compact database" action with a size/duration toast for users who want it now. Fallback variant if in-place VACUUM proves fragile on this VFS: VACUUM INTO a staging OPFS file + swap at next boot — the swap machinery already exists for DB import/recovery (src/utils/exportSqliteDb.ts, localDbRecovery.ts); costs more moving parts, bounds memory, never touches the live file.

15. Rejected alternatives

  • Pruning (keep-recent + grace window) — the row-events-retention.md design. Rejected by the frame: history is a product feature to keep, and pruning also would not have reduced the per-write cost that motivated that doc.
  • Server-shape adoption without anchors (changed-cols diffs, full bookends only — exactly blocks_history's Option B). Reconstruction becomes full-chain replay, and a single lost row poisons every state beyond it on the walk side. Sampled anchors cost ~1.5% of rows and bound both.
  • Deterministic probe + chain_pos cadence — the original §4 design (superseded 2026-07-15 by O5's adoption): a per-write predecessor probe fed a deterministic anchor window W=50 and promotion rules for missing/v1 predecessors. It bought deterministic walk bounds and "every chain starts full" — machinery that exists only because after-only compact events made chains base-dependent. Invertible payloads delete the dependence itself, at ~2× compact-update bytes. Full tradeoff record in O5.
  • Counter side-table cadence — mutable state that can drift from the log, an extra B-tree upsert per write, its own backfill/repair story; rejected under both cadence designs.
  • Naive probabilistic anchors over after-only payloads — unsafe: without invertible patches, chain-start and gap cases splice silently or strand undo (O5 record). A create-only-skip variant was examined and set aside with the format change.
  • App-layer event writer — §7 table; loses the catch-all property that is the log's reason to exist.
  • Values-free or sampled logging for the bulk bursts — §5 table rows (b)/(c).
  • Column-keyed exemptions (e.g. never log properties_json) — violates I4; loses the only history for cell-primary keys and the whole pre-flip period (PR #288 §12).
  • Sub-key diffs of properties_json — real wins today (property edits rewrite the whole cell), but it complicates every reader and the trigger, and the props migration is about to shrink the cell structurally; column-granular is right-sized for the post-#288 world. Revisit only if #288 stalls.
  • Compression (per-event zlib, content dedup) — wa-sqlite ships no compression extension; JS-side compression forces an app-layer writer (see above). The compact format gets the structural win without new machinery.
  • History in a separate ATTACHed database file — would make VACUUM trivial (vacuum the side file, or drop it), but SQLite triggers cannot write across databases, so it forces the app-layer writer; rejected with it.
  • Dropping sync-applied events at all (the earlier §6 draft: per-workspace-mode drop for plaintext, plus a workspace_history_policy side-table) — overridden by the user 2026-07-15: local history must stay self-sufficient; the server-side bytes (§6.1) are an access story, not a substitute. The retroactive variant — deleting existing sync events — was rejected from the start: history is kept.
