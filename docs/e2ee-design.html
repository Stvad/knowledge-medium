End-to-End Encryption Design

Status: sketch / not yet committed to. Discussion artifact.

Per-workspace opt-in E2EE with user-managed workspace keys. Supabase and the network see only ciphertext for content in encrypted workspaces; plaintext workspaces remain fully server-readable and keep all current and future server-side features. The user holds the key, types/pastes it on each device, and shares it with collaborators over whatever channel they trust.

1. Goals & non-goals

Goals

  • Workspace owner can mark a workspace as end-to-end encrypted at creation. Mode is immutable thereafter.
  • For E2EE workspaces, Supabase (and anyone with DB access) sees only ciphertext for blocks.content, blocks.properties_json, blocks.references_json.
  • Plaintext workspaces remain bit-identical to today — same schema, same RLS, same access to server-side features (current and future).
  • Multi-device: a user can decrypt their E2EE workspaces on any device they sign in on by pasting the workspace key once per device.
  • Sharing: an owner can give another user access by sending them the workspace key over whatever channel they trust (Signal, in person, encrypted email, etc.).

Non-goals (for v1)

  • Hiding metadata (workspace structure, block tree shape, edit timing, who-edited-what).
  • Defense against a compromised local device. OS-level disk encryption is the right tool; OPFS already gives per-origin isolation.
  • Cryptographic verification of authorship (signed writes). Useful later, not in scope now.
  • Server-mediated key delivery. The app deliberately does not transmit workspace keys through its own server channel — the user picks the channel. This eliminates an entire class of MITM concerns at the cost of one extra step at share time.
  • Account recovery without the user's workspace key. Lost key = lost data; the UI surfaces this clearly at create time.
  • Server-side features over encrypted workspaces (search, AI, embeddings). They simply don't apply.

2. Threat model

The threat we're addressing is the most common one users actually ask about: "can the hosting provider read my notes?" Concretely, in v1:

  • In scope: passive or active attacker with full read access to the Supabase Postgres, the PowerSync replica, network traffic to/from Supabase, and Supabase logs/backups. They must not learn the contents of E2EE-workspace blocks.
  • Out of scope: attacker who controls the user's browser at runtime (they can read decrypted data and exfiltrate the key); attacker who can compromise the client build pipeline (they can ship a backdoored client); attacker who can perform traffic analysis on edit patterns; attacker who can intercept the user's chosen out-of-band channel for sharing the workspace key (that's a property of the channel, not of this system).

Notably, the user-managed key model means active attackers on the server gain very little power over confidentiality: the server never sees a workspace key in any form, so there's nothing to substitute, MITM, or extract. The attacker can serve ciphertext rows or refuse to serve them; they can't read them.

3. Why the current architecture is friendly to this

The codebase is already structured in a way that makes E2EE tractable rather than invasive:

  • Server is blind storage today. RLS policies filter by workspace membership, not content. No tsvector, no pg_trgm, no AI/embeddings, no webhooks read block content.
  • Workspace is already the RLS boundary. Per-workspace key material maps cleanly onto a unit that already exists.
  • Per-user OPFS SQLite via kmp-v6-<user_id>.db gives physical isolation between users on a shared device.
  • PowerSync upload compaction already coalesces edits per block, which keeps wire overhead low independent of encoding.
  • Client-side edit log already exists (tx_context, command_events) — useful if we ever need to move history off the server for encrypted workspaces.

4. Design shape

Three properties drive the design:

  1. Per-workspace opt-in. One user can have plaintext and E2EE workspaces side by side on the same device, in the same local DB.
  2. Per-column encryption with the existing schema. Encrypt the three content columns in place rather than consolidating them into one opaque payload. Preserves PowerSync column-level PATCH granularity, keeps plaintext workspaces bit-identical, and avoids a schema migration that would touch every workspace.
  3. User-managed workspace key. Each E2EE workspace has a single symmetric key. The user is responsible for storing it and transmitting it to collaborators. The server has no role in key management — no key store, no key wrapping, no per-user identity material.
Decision. v1 ships the simple model: one symmetric workspace key (WK) per encrypted workspace, generated client-side, surfaced to the user once at create, stored by the user (password manager, paper, etc.), and pasted on additional devices. Sharing is out-of-band. No server-side key store, no asymmetric crypto, no KDF, no recovery codes.

5. Workspace key: shape, lifecycle, and storage

The workspace key (WK) is the only secret in the entire design.

PropertyValue
Material256 random bits, generated client-side via crypto.getRandomValues.
LifetimeOne per workspace, generated at workspace create. Stable for the life of the workspace unless explicitly rotated.
User-facing formatkmp-wk-1:<base32(32 bytes)> — 61-character paste-friendly string (52 unpadded base32 chars + 9-char prefix). The 1 reserves a clean upgrade path.
Where it lives(a) the user's password manager / backup of their choice; (b) on each device the user has pasted it on, as a non-extractable WebCrypto CryptoKey in IndexedDB, keyed by (user_id, workspace_id). Non-extractable means JS can use the key (crypto.subtle.encrypt/decrypt) but exportKey throws, so a script that gets onto the origin (XSS, a hostile extension) can't read the key bytes out and exfiltrate them — the worst it can do is use the key while the page is open (decrypt-and-leak plaintext), not steal it for offline, forever decryption. The raw bytes therefore exist only transiently: once at generation (shown for backup) and once at paste (imported, then discarded). IndexedDB rather than localStorage because localStorage holds strings only — it can't store a non-extractable CryptoKey, and raw key bytes in localStorage are readable by any origin script. Cross-tab propagation uses BroadcastChannel instead of the storage event; IndexedDB can be evicted under storage pressure, which is fine because the model already requires the user to keep a backup. Never the server, in any form.
Validation on pasteDecrypt workspaces.wk_canary and verify the plaintext equals the workspace id. Works on empty workspaces (no blocks needed). AEAD failure or plaintext mismatch → reject with "that key doesn't decrypt this workspace's data." See §7 (canary column) and §8.2 (paste flow).

The UI shows the WK exactly once, at create time, with prominent "save this somewhere safe — there is no recovery" copy and a recommendation to paste it into a password manager. A confirmation step ("type the last 6 characters") ensures the user actually saw it rather than dismissing the dialog.

6. Cryptographic choices

ConcernChoiceWhy
Symmetric AEAD AES-256-GCM Native in WebCrypto, no dependency. XChaCha20-Poly1305 wins on two axes — platforms without AES hardware, and a 192-bit nonce that removes the random-nonce birthday bound (next row) outright — but neither is decisive at our write volumes, and AES-GCM avoids pulling in a crypto dependency. Revisit if the nonce budget or non-AES targets ever bind.
Nonce 96-bit random per ciphertext Per-column random nonce. Random 96-bit nonces are birthday-bounded: NIST SP 800-38D caps a single key at ~232 invocations to keep the collision probability below ~2-33, where one invocation is one column-write of one block, summed across all collaborators (who share the one WK) over the workspace's entire life. A personal or small-team note workspace stays many orders of magnitude under 232 (~4 billion encryptions), so reuse is negligible — but that is the stated budget and assumption, not a hand-wave: nonce reuse is catastrophic for GCM (it leaks the XOR of the two plaintexts and enables forgeries via auth-key recovery). A workspace that could ever approach the bound should move to a counter/deterministic nonce or the wider-nonce AEAD above.
AAD (associated data) length-prefixed block_id ‖ workspace_id ‖ column_name ‖ schema_version Prevents the server from swapping ciphertexts between blocks, between workspaces, or between columns of the same block. The fields must use an unambiguous canonical encoding — length-prefix each field (or a delimiter byte that cannot occur in any field, e.g. 0x1F, already used for structured trigger messages in clientSchema.ts) — so that "A"‖"BC" and "AB"‖"C" can't alias over variable-length TEXT ids and defeat the very cross-binding the AAD exists to provide. The same encoding applies to the canary's AAD (§7, §8.1). schema_version is the envelope version (ciphertext-format row below), so it is reconstructable at decrypt time from the enc:vN: tag rather than stored separately; it reserves room for future format upgrades (e.g. a hierarchy migration).
Ciphertext format "enc:v1:" + base64url(nonce ‖ ciphertext) The prefix identifies the envelope format (version dispatch for the decrypt routine), not whether to decrypt — per §6 only the durable local mode pin decides that. Within a pinned-E2EE workspace every wire value must already be an enc:v1: envelope (§7 enforces it server-side), so here the prefix is a format tag and tamper/sanity check, never a trigger that routes a row into decrypt/error handling on its own. A plaintext-workspace value that happens to start with enc:v1: is therefore left untouched, and the server can't influence first-encounter behavior by adding or stripping the prefix. The v1 reserves a clean upgrade path — and the version tag is the AAD's schema_version: the envelope carries only nonce ‖ ciphertext (no separate version field), so a decryptor reconstructs the exact AAD from the prefix alone. If a future format ever needs the payload schema version to diverge from the envelope version, that value must move into the envelope so the AAD stays reconstructable.
WK display/paste format kmp-wk-1:<base32> Base32 (Crockford or RFC 4648 without padding) is paste-friendly across messaging apps and looks reasonable on paper. The kmp-wk-1: prefix makes mis-pasted keys identifiable.
KDF None — the WK is the key. No passphrase in v1, so no derivation. If the hierarchy is added later (§13), Argon2id over the user's vault passphrase is the right primitive.
Mode authority is local and cryptographically earned, not a server flag. A workspace is E2EE for this client iff the client holds a durable local pin marking it so — a pin set the first time a WK validates against the workspace's wk_canary, never by the server. workspaces.encryption_mode is a server-maintained projection (§7) with zero downgrade authority: the durable pin, never this flag, decides whether decrypted content is treated as authoritative and whether plaintext writes are allowed. The flag is consulted only in the direction that fails safe — an unpinned membership the server marks e2ee is held key-required until a WK validates (rule 3a), which can only add a key prompt, never authorize plaintext — so a hostile server flipping it achieves at most a lockout, never a downgrade. The three rules below are corollaries of this one principle, not separate defenses bolted on top of a trusted flag.

1. The authority is a durable local mode pin — separate from the WK, which is only decrypt material. The client keeps a per-(user, workspace) mode pin in localStorage (e2ee | plaintext) — separate from the WK store (§5, now IndexedDB) and not wiped by the Lock & wipe action (below). The pin is set once and is locally immutable thereafter (mirroring §7's server-side immutability): a workspace pins to e2ee the moment a WK first validates against its canary — at create (§8.1) or first paste (§8.2) — and pins to plaintext on a plaintext create or explicit confirmation. The WK is decrypt material only: transient, stored separately, and never the record of mode. Locking is a deliberate, coarse, user-only action in v1. "Lock & wipe local data" is the only thing that performs the destructive whole-DB wipe, and it is gated behind explicit confirmation and clear "this erases all local data on this device — you'll re-download and re-paste your keys" warnings. Nothing automatic triggers it — not a sync event, not a membership revoke, not sign-out. (Revoke in particular must never escalate to this wipe: it concerns one workspace, and the wipe is all-workspaces — see §9.2.) It is distinct from sign-out (it keeps you signed in; it isn't a logout). It flushes the upload queue first so unsynced edits aren't lost; if the flush can't complete (offline, or an edit the server permanently rejects) it warns the user that locking now would discard unsynced edits and lets them choose — reconnect and retry, or proceed and lose them — rather than silently doing nothing or silently dropping work. It then drops all WKs and wipes the whole local SQLite DB file rather than enumerating tables. The file holds every plaintext-bearing surface — the blocks table, the derived indexes (block_aliases/block_types/blocks_fts), the blocks_synced staging, and PowerSync's ps_crud / ps_crud_rejected queues (whose envelopes hold pre-encryption plaintext, §9.2) — so deleting the file is exhaustive by construction: there's no per-surface checklist to keep complete, which is the right shape given how many times that checklist grew under review. (This maps onto PowerSync's supported clear-and-resync path, not a hack.) Two things live outside the file and are handled separately: the in-memory BlockCache and other live JS state, cleared the simplest way — a forced page reload after the wipe (a fresh boot re-opens the recreated DB and re-syncs; run the actual file delete on that next boot, before the DB is opened, so wa-sqlite isn't holding the OPFS file when it's removed) — and the durable pins (in localStorage, deliberately preserved so each workspace's mode stays known). Note this coarse file-wipe is the lock action only; revoke/workspace-delete is a separate, per-workspace concern that v1 defers (§9.2) and that must never trigger this wipe. On next use, plaintext workspaces re-sync and render as usual; an e2ee-pinned workspace re-enters its locked read-only state (rule 3) until its WK is re-pasted — so the wipe never downgrades, and the plaintext-confirmation path is still not offered for a pinned workspace. v1 deliberately does not support clearing one workspace's WK while you keep using others: that per-workspace selectivity is what forces chasing scattered plaintext across every local surface, and the coarse wipe sidesteps it. A non-destructive lock (drop the key, data stays sealed rather than wiped) is a post-v1 option via at-rest encryption — §14. The pin is local and earned by decrypting the canary, so the server's encryption_mode still carries zero downgrade authority — the mode authority is the durable pin, not the server flag and not the ephemeral key.

2. The enc:v1: prefix is not self-authorizing, and the check belongs at the wire boundary. A hostile server could strip the prefix and serve attacker-chosen plaintext, so the prefix never decides whether a row should be encrypted — the pin does. For a workspace pinned e2ee (rule 1), every server-origin value must carry enc:v1: on the three content columns; a wire value missing it is treated as tampering (reject, surface a warning). "Server-origin" is the scope that matters: under Layout B (§9.2) the check sits where ciphertext actually appears — the observer ingesting blocks_synced, plus the upload/decrypt boundary — not at parseBlockRow over the app-visible blocks table, which by design holds decrypted plaintext for E2EE workspaces (and plaintext local edits until uploadData encrypts them). Demanding the prefix on blocks would reject every legitimate decrypted row and break rendering. The server's mode and the prefix are both untrusted; the local pin decides whether to demand a prefix at the boundary.
3. The genuinely-uncertain case: a never-pinned membership the server marks none. Once a workspace is pinned (rule 1) its mode is settled locally and the rules above govern it — including the "locked" state of an e2ee pin whose WK was cleared, which is read-only and offers only "paste WK," never a downgrade. For a membership the client has never pinned — a brand-new device, reinstall, or freshly accepted invite — the server's encryption_mode is the only signal available, and it is trustworthy in exactly one direction.

Trust "e2ee," distrust "none." Acting on encryption_mode = 'e2ee' can only make the client stricter: it demands a WK before rendering or writing anything (fail-closed). The worst a hostile server gains by falsely flagging a plaintext workspace e2ee is a lockout — a pasted WK won't validate against a real or forged canary — which is a denial of service we can't prevent against a hostile server anyway, and which never leaks or accepts plaintext. Acting on encryption_mode = 'none', by contrast, would make the client laxer — render server plaintext and accept plaintext writes — which is exactly the downgrade a hostile server wants. So the field is trusted only in the direction that fails safe, which splits the never-pinned case in two:

(a) Server says e2ee → key-required (happy path). Render nothing and prompt "this workspace needs its key"; the user pastes a WK, and validating it against wk_canary (§8.2) sets the e2ee pin. One resolution path, no "this is plaintext" option — the server has positively (and safely) told us it's encrypted, so we just prompt for the key before loading.

(b) Server says none → encryption-uncertain quarantine. This is the one case the server can't resolve for us: a real plaintext workspace and a hostile downgrade of an e2ee one look identical. Render read-only with a banner, no local writes, and offer both resolutions, each of which sets the durable pin: paste a WK that validates against wk_canary → pin e2ee (this is what defeats a downgrade lie); or explicitly confirm "this workspace is plaintext" → pin plaintext. The plaintext-confirmation path exists only here — never-pinned and server-says-none — which is what keeps it unreachable once a pin exists and makes clearing a WK safe.

This first-encounter handling governs every entry point where a membership first becomes visible locally without a pre-existing pin (the rollout bullet is the one deliberate, bootstrap-only exception that trusts the server's none):
  • Self-create on this device: both create paths resolve immediately inside the create RPC, so the quarantine window is effectively zero. The plaintext create flow (existing today) confirms plaintext on success — otherwise plaintext workspaces would enter quarantine on creation, contradicting the "behave as today" premise. The E2EE create flow (§8.1) generates the WK and validates it against the canary it just minted.
  • Accept invitation on this device: a shared E2EE workspace reports e2ee, so the accept flow (§8.3) takes branch (a) — prompt for the out-of-band WK before loading. Only a server claiming the shared workspace is plaintext drops it into branch (b)'s quarantine.
  • Existing membership encountered on a new device or after reinstall: the membership row arrives via sync without going through accept_invitation and there's no local pin. Same split — branch (a) prompts for the WK when the server says e2ee (§8.2), branch (b) quarantines when it says none.
  • Pre-existing memberships on the rollout release — trust the server once, during the upgrade, gated outside the wiped DB. The first release that introduces mode pins finds every pre-existing workspace unpinned. We initialize those pins directly from the server's encryption_mode (all none at this point) — the one place the design trusts that field. This is safe because the rollout is a coordinated bootstrap: we're alpha with a handful of clients, so we require everyone updates before any E2EE workspace is created. At that moment no E2EE workspace exists, so there's nothing for the server to misrepresent. No seed heuristics, no confirmation prompts, no staggered-rollout handling.

    The seed must fire exactly once and never re-fire — in particular not after a §6 Lock & wipe, which recreates an empty SQLite DB while E2EE workspaces may by then exist. A DB-resident "seeded" marker (a schema-state / version row in the SQLite file, or the like) would be erased by the wipe and let the seed re-run on the empty DB — at which point a hostile server could present an unpinned e2ee membership as none and win a durable plaintext pin. Two properties prevent that, and this gating — not the prose above — is the load-bearing part of the bullet:
    • Gated in localStorage, version-keyed — not in the wiped DB. The "pins seeded" marker lives in the same wipe-surviving localStorage as the pins themselves (rule 1), and the seed is a one-time migration keyed to the pre-pin → pin-aware app-version transition. It is never triggered by "DB is empty" or "membership unpinned," so a recreated DB can't re-arm it.
    • Fallback is quarantine, never re-seed. Outside that one upgrade, an unpinned membership always takes the steady-state path above — branch (a) for server-says-e2ee, branch (b) quarantine for server-says-none. The server-trusting seed is unreachable thereafter, so even if the marker were somehow lost the worst case is a quarantine prompt, not a silent plaintext pin.
    This is what closes the post-wipe downgrade: pins survive the wipe in localStorage, so a membership that was pinned re-syncs already pinned (an e2ee pin re-enters locked read-only until its WK is re-pasted, rule 3), and a genuinely-new unpinned e2ee membership the server tries to pass off as none lands in branch (b) quarantine — neither path reaches the seed. If coordination ever broke during the initial upgrade a late-upgrading device could mis-pin; that's an accepted alpha limitation of the one-time bootstrap, not a standing exposure. The trust is scoped to this one upgrade: the steady-state entry points (new device, new invite) keep the full pin / quarantine / wire-check defense, so a later server compromise — or any post-wipe re-sync — stays contained.
The cost is one resolution step per workspace per new device — not free, but the alternative is silently accepting attacker plaintext on a workspace the user has every reason to believe is encrypted.

7. Schema additions

One small migration. No changes to blocks columns; no new tables.

-- Every object below is schema-qualified (public.*). The repo's migrations run
-- with an empty search_path (the consolidated initial migration sets
-- search_path = '' and qualifies all objects), so an unqualified DROP/CREATE/
-- ALTER/GRANT would not resolve when this sketch is copied into a migration.
--
-- encryption_mode is a server-side PROJECTION of canary presence, not the
-- source of truth: per §6, a client decides "E2EE" from holding a validated
-- WK, never from this column. It exists for server-side feature gating and
-- as a UX hint; the CHECK below keeps it in lockstep with wk_canary. It is
-- an immutable property of a workspace once set.
ALTER TABLE public.workspaces ADD COLUMN encryption_mode TEXT
  NOT NULL DEFAULT 'none'
  CHECK (encryption_mode IN ('none', 'e2ee'));

-- Immutability trigger, matching the existing blocks_prevent_workspace_change()
-- pattern in the consolidated migration (concrete function per invariant, not a
-- generic helper). ONE trigger covers BOTH e2ee fields (encryption_mode AND
-- wk_canary, added below) -- this, not a column-level privilege REVOKE/GRANT, is
-- what the shipped migration uses; see the immutability note under the canary
-- CHECK for the why. (Late-binding plpgsql lets the body name wk_canary before
-- the ADD COLUMN a few lines down; the trigger only fires at runtime, after it
-- exists.)
CREATE OR REPLACE FUNCTION public.workspaces_prevent_e2ee_field_change()
  RETURNS trigger
  LANGUAGE plpgsql
  SET search_path = '' AS $$
begin
  -- errcode 23514 (check_violation) is a PERMANENT SQLSTATE, so a stale/buggy
  -- client that PATCHes these columns is quarantined by the upload classifier
  -- rather than retrying the generic P0001 forever and jamming the queue.
  if OLD.encryption_mode is distinct from NEW.encryption_mode then
    raise exception 'workspaces.encryption_mode is immutable (% -> %)',
      OLD.encryption_mode, NEW.encryption_mode
      using errcode = 'check_violation';
  end if;
  if OLD.wk_canary is distinct from NEW.wk_canary then
    raise exception 'workspaces.wk_canary is immutable (rotation is a post-v1 SECURITY DEFINER path)'
      using errcode = 'check_violation';
  end if;
  return NEW;
end;
$$;

CREATE TRIGGER workspaces_prevent_e2ee_field_change_trg
  BEFORE UPDATE ON public.workspaces
  FOR EACH ROW
  EXECUTE FUNCTION public.workspaces_prevent_e2ee_field_change();

-- create_workspace accepts encryption_mode and a key-check canary
-- so we can validate pasted WKs even on workspaces that have no
-- block rows yet (freshly invited or just-created E2EE workspaces).
-- The canary is an AEAD-encrypted known plaintext (the workspace id);
-- decrypting it with the candidate WK both authenticates the key
-- (AEAD failure = wrong key) and confirms it's bound to this workspace
-- (right key on the wrong workspace's canary fails the plaintext check).
-- wk_canary format: enc:v1:base64url(nonce || ciphertext) of workspace_id
-- as the plaintext, with AAD = workspace_id || "canary" || schema_version
-- (length-prefixed / unambiguous encoding, same as the content AAD -- §6).
ALTER TABLE public.workspaces ADD COLUMN wk_canary TEXT;

-- NULL for plaintext workspaces; required for E2EE so new devices can
-- validate a pasted WK before any blocks exist. Two-way (biconditional)
-- cross-column CHECK: a canary is present iff the mode is e2ee. The
-- forward direction stops an e2ee row with no canary; the reverse stops a
-- 'none' row that still carries a canary (e.g. a caller passing the
-- default mode plus p_wk_canary), which would otherwise let server-side
-- gating and client prompts misclassify a "plaintext" workspace given
-- that later sections treat wk_canary IS NULL as the plaintext signal.
-- Presence alone isn't enough: an e2ee canary (and every E2EE content column,
-- see the blocks trigger below) must be a syntactically well-formed enc:v1:
-- AEAD envelope, not merely a prefixed string. Otherwise p_wk_canary =>
-- 'anything' or content => 'enc:v1:My plaintext note' is accepted -- the
-- former leaves the workspace unrecoverable (§8.2 can never validate a WK
-- against it), the latter leaves the server storing readable plaintext. The
-- server can't prove the bytes are real ciphertext (it has no key), but it can
-- require the envelope SHAPE: enc:v1: + a *decodable* base64url payload of at
-- least nonce(12B)+tag(16B) = 28B. A bare {38,} length floor isn't enough: it
-- still admits payloads that can't decode -- a data-char count of len%4==1
-- (e.g. 41 chars) is impossible in base64, and stray padding (40 data chars
-- then '==') is malformed -- so §8.2 clients would fail to decrypt and the
-- workspace/block would be just as unrecoverable as a missing canary. So
-- enforce the base64url group grammar: every 4 chars -> 3 bytes, with only a
-- 2-char (->1B, optional '==') or 3-char (->2B, optional '=') final group
-- legal. The 28B floor is then >= 9 full groups plus one of those terminals
-- (>= 38 data chars; data-char count mod 4 is never 1). One shared IMMUTABLE
-- helper keeps the canary CHECK and the blocks trigger in lockstep.
-- ReDoS: safe here -- Postgres's regex engine is DFA-based, no backtracking.
-- The inner {4} is fixed-width so the grammar has no ambiguous decomposition;
-- measured linear even in V8's backtracking engine (~3ms at 200k chars). Even
-- so, the CLIENT should not port this regex to gate server-supplied ciphertext
-- -- it validates by base64url-decode + AEAD verify, which is the real
-- integrity gate and has no regex surface at all.
CREATE OR REPLACE FUNCTION public.is_enc_v1_envelope(v text)
  RETURNS boolean
  LANGUAGE sql IMMUTABLE AS $$
  -- No padding: the client encodes with base64url-NOPAD (@scure/base), so a
  -- '='-padded payload never appears on the wire and is rejected as malformed.
  SELECT v ~ '^enc:v1:([A-Za-z0-9_-]{4}){9,}([A-Za-z0-9_-]{4}|[A-Za-z0-9_-]{2}|[A-Za-z0-9_-]{3})$'
$$;

ALTER TABLE public.workspaces
  ADD CONSTRAINT workspaces_wk_canary_matches_mode
  CHECK (
    (encryption_mode = 'e2ee') = (wk_canary IS NOT NULL)
    AND (encryption_mode <> 'e2ee' OR public.is_enc_v1_envelope(wk_canary))
  );

-- Immutable after create: an editor with write access to the workspaces row
-- could otherwise replace the canary and brick WK validation for other members.
-- The shipped migration enforces this with the BEFORE UPDATE trigger above
-- (workspaces_prevent_e2ee_field_change), which rejects any change to either
-- e2ee field. This is load-bearing, NOT defense-in-depth: the consolidated
-- migration keeps a live workspaces_update RLS policy for writers plus
-- `GRANT ALL ON TABLE public.workspaces TO authenticated`, so an authenticated
-- writer CAN issue a direct PostgREST UPDATE on this table -- and the canary
-- CHECK only requires the replacement to be a well-formed envelope, so without
-- the trigger a writer could swap in an attacker-chosen canary while keeping
-- encryption_mode = 'e2ee', bricking every other member's unlock.
--
-- Why a trigger, not a column-level privilege REVOKE/GRANT (an earlier draft):
-- both fully prevent client writes and are equally secure for v1 (there is NO
-- legitimate post-create writer of either field -- the canary is minted once in
-- create_workspace's INSERT, which a BEFORE UPDATE trigger does not gate). The
-- trigger is simpler and localized: a column REVOKE UPDATE ... GRANT
-- UPDATE(name, update_time) recarving would have to untangle that existing
-- `GRANT ALL` + per-column grant surface, and the trigger matches the repo's
-- concrete-function-per-invariant convention (cf. blocks_prevent_workspace_change()).
--
-- A session GUC bypass (an earlier SET LOCAL kmp.rotating_wk idea) was rejected
-- as forgeable: any caller running SQL as the authenticated role can set it
-- before updating wk_canary, so it can't scope a bypass to one RPC. Rotation
-- (§14) is post-v1; when it lands, the dedicated rotate_workspace_wk RPC --
-- SECURITY DEFINER, owned by postgres -- can carry a NON-forgeable bypass (the
-- trigger permits the write only when current_user is the table owner, which a
-- client on the authenticated role can never become), or switch to the column-
-- privilege model then. Either way the client never gains a canary write path.
-- (Client upload paths must still not PATCH wk_canary/encryption_mode with
-- CHANGED values, or the trigger rejects the write.)

-- Postgres function signatures include parameter count/types, so a new
-- create_workspace with extra defaulted params would *overload* the
-- existing create_workspace(p_name text) rather than replace it — and
-- create_workspace('foo') would then be ambiguous. Drop the old signature
-- explicitly, then create the new one and grant it.
DROP FUNCTION IF EXISTS public.create_workspace(text);

CREATE OR REPLACE FUNCTION public.create_workspace(
    p_name             text,
    p_encryption_mode  text DEFAULT 'none',  -- 'none' | 'e2ee'
    p_workspace_id     text DEFAULT NULL,    -- REQUIRED for e2ee (guard below);
                                             -- NULL falls back to server-generated
                                             -- UUID for plaintext (current behavior)
    p_wk_canary        text DEFAULT NULL     -- required iff p_encryption_mode = 'e2ee'
  ) RETURNS jsonb
  -- Carry over the existing RPC's attributes: SECURITY DEFINER is load-bearing
  -- (workspaces has no INSERT policy for callers, so a non-definer RPC running
  -- under the caller's RLS can't insert the workspace/member rows and creation
  -- fails for both modes). search_path is pinned per the SECURITY DEFINER norm.
  LANGUAGE plpgsql SECURITY DEFINER
  SET search_path TO 'public'
  AS $$
declare
  v_id        text := coalesce(p_workspace_id, gen_random_uuid()::text);
  v_user_id   text := auth.uid()::text;
  v_name      text := coalesce(nullif(trim(p_name), ''), 'Workspace');
  v_now       bigint := (extract(epoch from now()) * 1000)::bigint;
  v_workspace public.workspaces;
  v_member    public.workspace_members;
begin
  -- AuthZ: this RPC is SECURITY DEFINER and bypasses RLS, so it must self-guard.
  -- Reject unauthenticated callers, and bind ownership to auth.uid() rather than
  -- trusting any client-supplied owner id -- the workspace_members owner row
  -- inserted below is keyed by auth.uid(), never by a function argument.
  if auth.uid() is null then
    raise exception 'create_workspace: authentication required';
  end if;
  -- The e2ee canary is minted client-side binding the CLIENT-CHOSEN workspace id
  -- (it's the canary's plaintext and part of its AAD). If the server were to
  -- generate a different id, the canary/mode CHECK would still pass, but the
  -- canary would never validate the WK on reload or another device (it decrypts
  -- to the client's intended id, not the row's actual id). So e2ee MUST supply
  -- its own id, and the row is inserted with exactly that id.
  if p_encryption_mode = 'e2ee' and p_workspace_id is null then
    raise exception 'create_workspace: p_workspace_id is required for e2ee workspaces';
  end if;
  -- Insert the workspace with the CLIENT-CHOSEN id and the two e2ee columns.
  -- No explicit canary guard is needed: the workspaces_wk_canary_matches_mode
  -- CHECK (above) enforces canary-present iff e2ee AND canary well-formedness,
  -- so a missing or garbage p_wk_canary for an e2ee create is rejected here.
  insert into public.workspaces
      (id, name, owner_user_id, create_time, update_time, encryption_mode, wk_canary)
    values (v_id, v_name, v_user_id, v_now, v_now, p_encryption_mode, p_wk_canary)
    returning * into v_workspace;

  -- Owner membership keyed by auth.uid() (v_user_id), never a client argument.
  insert into public.workspace_members (id, workspace_id, user_id, role, create_time)
    values (gen_random_uuid()::text, v_workspace.id, v_user_id, 'owner', v_now)
    returning * into v_member;

  return jsonb_build_object(
    'workspace', to_jsonb(v_workspace),
    'member', to_jsonb(v_member)
  );
end;
$$;

GRANT EXECUTE ON FUNCTION public.create_workspace(text, text, text, text)
  TO anon, authenticated, service_role;

-- Defense-in-depth: "the server only sees ciphertext for e2ee workspaces"
-- must not rest solely on the client's upload hook. A stale client, a missed
-- hook, or a direct PostgREST write could otherwise land plaintext in an
-- e2ee workspace's content columns -- the blocks_write RLS policy only checks
-- writer membership, not payload shape. Enforce it server-side: for any block
-- whose workspace is e2ee, the three content columns must carry a well-formed
-- enc:v1: envelope -- the same is_enc_v1_envelope() shape the canary uses, so
-- a prefixed-but-readable value like 'enc:v1:My plaintext note' (which would
-- still leak plaintext to the server) is rejected, not just NULL/bare values.
-- A CHECK can't read another table, so this is a BEFORE trigger that looks the
-- workspace up (PK lookup, cheap). The client encrypts even "empty" values, so
-- an e2ee block never legitimately holds bare ''/{}/[].
CREATE OR REPLACE FUNCTION public.blocks_require_ciphertext_for_e2ee()
  RETURNS trigger
  LANGUAGE plpgsql AS $$
begin
  if exists (
    select 1 from public.workspaces w
    where w.id = NEW.workspace_id and w.encryption_mode = 'e2ee'
  ) and not (
    public.is_enc_v1_envelope(NEW.content) and
    public.is_enc_v1_envelope(NEW.properties_json) and
    public.is_enc_v1_envelope(NEW.references_json)
  ) then
    raise exception 'blocks in an e2ee workspace must carry a well-formed enc:v1: envelope in all content columns';
  end if;
  return NEW;
end;
$$;

CREATE TRIGGER blocks_require_ciphertext_for_e2ee_trg
  BEFORE INSERT OR UPDATE ON public.blocks
  FOR EACH ROW
  EXECUTE FUNCTION public.blocks_require_ciphertext_for_e2ee();

Editing WORKSPACE_COLUMNS alone would regenerate the raw-table put SQL to reference the new columns, but PowerSync's apply would then fail with a missing-column error on existing clients. Need explicit ALTER TABLE workspaces ADD COLUMN encryption_mode TEXT NOT NULL DEFAULT 'none' and ALTER TABLE workspaces ADD COLUMN wk_canary TEXT migrations. A client_schema_state marker (the existing backfill-marker pattern) is the steady-state fast-path skip, but it is not a sufficient guard on its own — the ALTERs must be idempotent on column existence. CREATE_WORKSPACES_TABLE_SQL is buildCreateTableSql('workspaces', WORKSPACE_COLUMNS) and runs at repoProvider.ts:227, before the gated backfills — so once WORKSPACE_COLUMNS carries the two new ColumnDefs, a fresh install or a §6 Lock & wipe (which recreates the DB) creates workspaces with encryption_mode/wk_canary already present — carrying their full definitions, including the NOT NULL DEFAULT 'none' — and with no marker row. A marker-gated-only ALTER would then fire on that fresh DB and throw duplicate column name, blocking startup/resync. So guard each ADD COLUMN on a PRAGMA table_info(workspaces) existence check — add the column only when absent — and write the marker afterward regardless: an upgrading device has the columns missing → they're added (and the DEFAULT backfills existing rows, below); a fresh DB has them already → the ALTERs are skipped and only the marker is recorded. Either way the migration runs at most once per device and never errors. Mirror the server's NOT NULL DEFAULT 'none' on the local encryption_mode column. A bare ADD COLUMN leaves the workspace rows that already exist on an upgrading device as NULL until PowerSync happens to replay each one — and the §6 rollout seed pins pre-existing workspaces from encryption_mode = 'none', so those NULL rows would be skipped and left encryption-uncertain on first boot after upgrade. The default backfills existing rows to 'none' at migration time (matching the server), and later replays overwrite with the real value. wk_canary stays nullable — NULL is the correct value for a plaintext workspace.

8. User-visible flows

8.1 Create an encrypted workspace

  1. User clicks "Create encrypted workspace".
  2. Client generates a 32-byte WK via crypto.getRandomValues; encodes it as kmp-wk-1:<base32>.
  3. Client picks the workspace id (UUID v4), then encrypts the id with the WK (AAD = workspace_id || "canary" || schema_version, length-prefixed per §6) to produce the wk_canary. The canary is a known-plaintext AEAD blob that any future device can decrypt to validate a pasted WK, even on a workspace that has no blocks yet.
  4. Client validates the WK against the canary it just minted, then imports it as a non-extractable CryptoKey in IndexedDB keyed by (user_id, workspace_id) (§5) and sets the durable mode pin for this workspace to e2ee (§6 rule 1). The pin is a separate, WK-independent record, so future loads never trust a server-supplied encryption_mode = 'none' and a Lock & wipe can't downgrade the workspace.
  5. Client calls create_workspace(p_name, p_encryption_mode='e2ee', p_workspace_id, p_wk_canary) with the client-chosen id so the canary's plaintext and AAD line up with the server's stored row. Server inserts workspaces with the canary and the owner's workspace_members row, exactly as the plaintext flow does. No WK bytes are sent to the server — only the canary (which is opaque ciphertext).
  6. UI: show the WK once with strong "save this securely — no recovery is possible" copy. Recommend pasting into a password manager. Confirm-by-retype on the last 6 characters before dismissing.

8.2 Open an E2EE workspace on a new device

  1. User signs in on device 2. PowerSync syncs encrypted block rows for workspaces they're a member of.
  2. Client checks the IndexedDB WK store at (user_id, workspace_id) — no WK present.
  3. UI surfaces a "this workspace needs its key" prompt instead of rendering ciphertext.
  4. User pastes their stored WK.
  5. Client validates by decrypting workspaces.wk_canary with the candidate WK and checking the result equals the workspace id. AEAD failure (wrong key) and plaintext mismatch (right key, wrong workspace) both reject. Works for freshly invited / empty workspaces because the canary is created at workspace-create time, not derived from blocks.
  6. On success: import the WK as a non-extractable CryptoKey in IndexedDB (§5) and set the durable mode pin to e2ee (§6 rule 1; the pin is in a separate store and survives a Lock & wipe), then render normally. On failure: surface "that key doesn't decrypt this workspace's data," let the user retry.

8.3 Share an E2EE workspace with another user

  1. Owner invites the collaborator via the existing invite_member_by_email → accept_invitation flow. The RPC chain doesn't change; the membership row is plaintext-only metadata.
  2. Owner sends the WK to the collaborator over a channel the user picks themselves — Signal, in person, sealed envelope, encrypted email, password manager share, whatever they trust for this contact. The app does not transmit the WK through its own server channel.
  3. Collaborator accepts the invitation server-side as today. Because the shared workspace is E2EE the server reports encryption_mode = 'e2ee', so per §6 rule 3 the collaborator takes the key-required happy path (branch a): read-only with a "paste the key you were sent" prompt, no local writes until the out-of-band WK validates against wk_canary (which sets the e2ee pin). There is no "this is plaintext" option here — the server has positively told us it's encrypted, and trusting that direction only fails closed (a key prompt), never down. The downgrade defense is the other branch: if a hostile server instead claimed this shared workspace was plaintext, the collaborator would be a never-pinned + server-says-none membership, land in branch (b)'s encryption-uncertain quarantine, and could still paste the out-of-band WK to defeat the lie.
Sharing is the user's responsibility. The app does not provide an in-app "send key to Bob" button on purpose. Any in-app key delivery routes through the server, which would re-introduce the MITM concern that motivated this design. The "annoying" extra step (using Signal / a password manager / a piece of paper) is the security property.

9. Sync boundary & per-column encryption

The encrypt/decrypt seam lives in a new module — call it src/sync/transform.ts — that sits between the app domain and the PowerSync CRUD edge. It is the single named place where "what crosses the wire" gets transformed.

The seam owns two pure functions: encodeForWire(block, mode, getCek) and decodeFromWire(row, mode, getCek). They start as identity functions and grow transforms over time. Today: per-workspace per-column AES-GCM. Later: schema-version migration of payload shapes, redaction for share-link previews, optional compression if data sizes ever justify it.

Pluggable key lookup. The seam takes getCek(workspaceId) as a callback rather than reading any global. It returns a usable CryptoKey handle, not raw bytes — in the simple model the implementation is "fetch the non-extractable CryptoKey from IndexedDB at (user_id, workspace_id)" (§5). If the hierarchy is added later (§13), the implementation becomes "look up in the in-memory unwrapped-CEK map populated at vault unlock." Same call site, two implementations. This is the single most important seam to get right for keeping the hierarchy migration cheap.

9.1 What "per-column" actually means on the wire

Each of the three content columns is sealed independently with its own random nonce. The AAD carries the column name so the server can't silently move ciphertext from one column to another and have it still decrypt as valid.

9.2 Where local plaintext lives: Layout B (staging table + observer)

The codebase leans heavily on SQL-side processing of content. Triggers in src/data/internals/clientSchema.ts and src/plugins/references/localSchema.ts maintain plaintext-derived indexes (block_aliases, block_types, blocks_fts, references-graph tables) on every write. Queries against those indexes drive core UX: parseReferences, alias autocomplete, type filters, local FTS.

Decision: Layout B — the app-visible blocks table stays plaintext, fed by a JS observer that decrypts a synced ciphertext mirror. Two alternatives were considered and rejected. Layout A (keep ciphertext in the local blocks table and let the derived indexes lapse) degrades alias search and local FTS — core UX — so it's a non-starter. Layout C (ciphertext at rest with async-UDF-decorated triggers) scatters encryption-awareness across every raw-SQL site that touches a content column, an unbounded surface one author away from silently breaking E2EE. Full mechanics of both, and the codebase survey that killed C, are in e2ee-rejected-alternatives.html. The transport-investigation box below explains why a simpler "decrypt-on-download" hook isn't available — which is what forces the mirror-table shape in the first place.

Layout B — local plaintext via staging table + JS observer

The local blocks table holds plaintext for everything (encrypted and plaintext workspaces alike). PowerSync's sync apply lands all blocks rows — for both plaintext and E2EE workspaces — in a parallel blocks_synced staging table (the row shape that matches the server). PowerSync's sync rule projects from public.blocks, so the FROM-clause-as-row_type behavior would route everything to the client blocks table by default; the gen-sync-config change re-targets the stream's row_type to blocks_synced, which means every downloaded row (plaintext or encrypted) lands in staging, not just E2EE rows.

A JS observer on PowerSync's change events processes every staging insert/update. For rows in plaintext workspaces it copies through unchanged. For rows in E2EE workspaces it decrypts the content columns with the WK and writes plaintext to blocks. Both cases end up populating blocks as the app-visible table; existing triggers in clientSchema.ts and plugins fire on plaintext writes as today — with one change: the row_events invalidation path is retired in favor of direct observer invalidation (see the callout below).

The observer is universal, not E2EE-only. If it only handled E2EE rows, plaintext-workspace sync updates would be stuck in blocks_synced and never reach the app's blocks table — breaking the requirement that plaintext workspaces stay unchanged. The cost: every downloaded row pays one extra local write (staging → blocks), not just E2EE rows. For typical workloads this is small; for a large workspace's initial sync it's N extra writes. The dual-target alternative (Postgres view partitioning E2EE workspaces, two sync rules, one mapping to blocks and one to blocks_synced) avoids the cost on plaintext users at the price of more Postgres-side complexity; not pursued in v1 unless profiling shows the universal-observer cost matters.

The observer is a reconciler, not just a copier. Once PowerSync writes only to blocks_synced, the app-visible blocks table is no longer PowerSync-managed — the observer is now responsible for the local/remote merge that PowerSync's CRUD machinery normally handles for raw tables. If a user has a pending local edit in blocks (already written by TxImpl, queued in ps_crud awaiting upload) and an older server snapshot arrives in staging, a blind copy would overwrite the local edit. The observer needs to consult ps_crud before each staging-row apply: if there's a pending CRUD entry for that block_id (or the local row's updated_at is newer than staging's), skip the copy and let the upload echo reconcile when it returns. This is the load-bearing complexity of Layout B that the original "just decrypt and copy" framing understated.

Rows the observer can't yet materialize stay in staging — but "no WK" is not the test. Because the blocks stream is retargeted globally, the observer drains every workspace's rows out of blocks_synced: plaintext workspaces by copy-through (no key needed), E2EE workspaces by decrypt. The only rows it must leave staged are the ones it can't turn into plaintext yet — a workspace we treat as E2EE but whose WK isn't loaded (a "locked" pin, §6, or the never-pinned key-required state where the server says e2ee, rule 3a), or an encryption-uncertain workspace (rule 3b). A plaintext-pinned workspace has no WK by design, so a naive "skip any row whose workspace has no WK loaded" would strand its rows in staging forever and the app-visible table would never hydrate. Concrete behavior: skip a staging row only when its workspace is E2EE-without-WK (locked or key-required) or encryption-uncertain; otherwise apply it immediately (decrypt for E2EE-with-WK, copy-through for plaintext). A drain pass re-processes a workspace's staged rows whenever it becomes materializable — on WK paste (E2EE) or on plaintext confirmation (a never-pinned workspace the user resolves as plaintext).

The observer subsumes rowEventsTail — it invalidates directly, so the row_events invalidation round-trip is removed (the audit table itself is RETAINED as the local change-history log; see the removals list below). Today PowerSync writes synced rows straight into blocks, bypassing JS, so the app reconstructs each change after the fact: the row_events triggers (clientSchema.ts) serialize before/after JSON into an audit table, and rowEventsTail.ts reads it back, re-parses, LWW-gates, and emits the cache update + handleStore.invalidate notification. Under Layout B the observer is the JS seam that applies every sync write, and it already holds both sides in memory: the after (the decrypted staging row) and the before (the current blocks row it reads for the LWW check anyway). So it can invalidate directly — reusing snapshotsToChangeNotification, the same helper the local repo.tx fast path uses (handleStore.ts) — instead of round-tripping through an audit table. That makes the whole row_events path redundant: drop the three row_events triggers, the row_events table + indexes, and rowEventsTail.ts (including its watermark / throttle-init-race / drain-serialization machinery, which an onDiff push stream obviates). The membership/order, §4.7 cycle-scan, plugin collectFromRowEvent, and LWW logic relocate into the observer — they are not lost.

One LWW gate, not two. The reconciler above already consults ps_crud / updated_at to decide whether a staging row may overwrite blocks. Folding invalidation into the observer means that same decision also gates the cache write and the handle notification, so the current two-gate shape (the observer's write gate plus rowEventsTail's applyIfNewer(_, 'sync')) collapses to one — removing a class of bug where the two gates disagree mid-edit.

What this does to the guarded UPSERT. BLOCKS_RAW_TABLE.put's INSERT ... ON CONFLICT(id) DO UPDATE ... WHERE <diff> shape (blockSchema.ts 116-126) was justified entirely by row_events: preserving the UPDATE trigger shape so before_json stayed populated, and the WHERE-diff guard suppressing no-op re-deliveries from firing UPDATE triggers. With row_events gone, so is that justification. PowerSync now writes opaque ciphertext to blocks_synced, which carries no triggers, so its raw-table put can be a plain INSERT OR REPLACE; the observer's own write into blocks is driven by onDiff (changed rows only) and skips no-ops in JS, so the WHERE-diff guard isn't needed there either. The one residual reason to keep ON CONFLICT DO UPDATE over INSERT OR REPLACE on blocks is FTS-rowid stability in the surviving derived-index triggers (a DELETE+INSERT reallocates the blocks_fts_rowids mapping) — a small, separable call, no longer entangled with invalidation correctness.

Observer writes must leave tx_context.source NULL. The upload-routing triggers in clientSchema.ts enqueue any blocks write whose tx_context.source is non-NULL into ps_crud for upload. If the observer wrote through the normal repo/tx path, every downloaded row would be queued back as a local edit — echo-upload loop. PowerSync's existing apply path already handles this for the tables it directly manages by leaving source = NULL (the upload triggers' WHEN clause checks for non-NULL source). The Layout B observer must follow the same convention: bracket its writes to blocks in a tx_context state where source is NULL, so the upload triggers skip them while the side-index triggers (aliases, types, FTS) still fire normally. (row_events is no longer among them — the observer invalidates directly, above.)

Rows that leave the synced stream split into two scopes: logical removal stays at parity with today (v1), and only deep/secure teardown is deferred. PowerSync's onDiff surfaces a removed set when rows leave the user's buckets — which here is predominantly membership revoke or workspace delete, since ordinary block deletes are soft (deleted = 1; the row stays in the stream and so arrives as an update, copied through normally). A removed entry instead means the row left the synced set entirely, so the observer hard-deletes it: DELETE FROM blocks WHERE id IN (…) under the same source = NULL convention as its other writes. That single delete is at parity with what PowerSync does today when it hard-deletes a block on stream-exit: the source-gated upload triggers skip it (no echo back up), and the ungated derived-index DELETE triggers (blocks_alias_delete, blocks_type_delete, blocks_fts_delete) fire and purge block_aliases / block_types / blocks_fts automatically. PowerSync removes the ciphertext from blocks_synced itself. So both primary copies and the trigger-maintained indexes are cleaned in v1 — this is roughly the same code as the insert/update copy-through branch, not new machinery, and keeping it preserves a capability we have today rather than regressing it. (One assumption to verify against the PowerSync API, not assert: that onDiff().removed reliably reflects a true stream-exit and isn't spuriously emitted during transient checkpoint churn — i.e. that it is a content-diff of blocks_synced between ticks. If so, a removal is authoritative.)

What v1 defers is only the hardening beyond today's posture — the "scrub every possible unencrypted artifact" complexity we don't want to invent just for revoke: the workspace's plaintext ps_crud / ps_crud_rejected queue entries (today also not purged on revoke, so this is pre-existing, not a new gap); physical/secure erase of the freed OPFS pages (SQLite doesn't zero freed pages, so today's deletes already leave recoverable remnants — true erase needs VACUUM / secure_delete / DB rekey-rebuild); and any plaintext that escapes the trigger cascade (in-memory caches, rendered or exported content). These go stale and are cleared by the next explicit Lock & wipe (§6). Deferring them is safe: a revoked member already held the WK and the data they had synced, so revoke can't retroactively protect any of it — this cleanup is local hygiene, not a confidentiality boundary. The one functional wrinkle — a member who is later re-invited could see a pre-revoke queued edit replay when its upload retries — stems from the deferred queue cleanup, not the row removal, and is a known, low-severity cost: it's the user's own edit landing in a workspace they again belong to, not a leak. Two rules hold regardless: revoke stays per-workspace and must never escalate to §6's coarse all-workspaces wipe; and if a partial-workspace wipe is ever added for another reason, the queue-entry teardown above is the natural place to fold it in.

Upload is symmetric. App writes plaintext to blocks as today; the existing upload-routing trigger captures plaintext into ps_crud; PowerSync's uploadData connector hook intercepts the CRUD batch and encrypts the content columns before sending to the server. "The server only ever sees ciphertext" is enforced server-side, not left to client convention: the blocks_require_ciphertext_for_e2ee trigger (§7) rejects any write to an e2ee workspace's blocks whose content columns aren't enc:v1:, so a stale client, a missed upload hook, or a direct PostgREST write can't slip plaintext past the writer-membership RLS policy and store it before newer clients would reject it at the wire boundary.

Always-emit is a few bytes of wire overhead per PATCH; the alternative (have the hook read the local row before encrypting) adds a DB round-trip per upload.
  • What this removes from the existing data layer (the subtractive half — because blocks stops being PowerSync-managed once the stream targets blocks_synced):
    • blocks leaves withRawTables (repoProvider.ts, and the test harness in createTestDb.ts). No stream targets it anymore, so PowerSync no longer manages it and the registration is vestigial — blocks becomes a plain local table written only by TxImpl (local edits) and the observer (sync).
    • The guarded-UPSERT apparatus in blockSchema.ts goes: BLOCKS_RAW_TABLE.put's ON CONFLICT … WHERE <diff> and the builders that exist only to feed it — BLOCK_SYNC_ASSIGNMENTS, BLOCK_SYNC_DIFF_PREDICATES, formatSqlOrList. The new blocks_synced raw table carries no triggers, so its put is a plain INSERT OR REPLACE. Retained: BLOCK_STORAGE_COLUMNS / CREATE_BLOCKS_TABLE_SQL (reused for the blocks_synced column shape) and the read helpers SELECT_BLOCK_COLUMNS_SQL / buildQualifiedBlockColumnsSql (they read plaintext blocks, unaffected). Two already-dead exports here — UPSERT_BLOCK_SQL, buildBlockCrudJsonSql — can go independently of this work.
    • The row_events invalidation round-trip — rowEventsTail.ts and the collectFromRowEvent rule path (per the callout above). Correction (post-implementation): the row_events table + its 3 INSERT/UPDATE/DELETE triggers + indexes (and the triggerTxIdSql helper they use) are RETAINED, not removed. The original list folded them in because their only consumer was the tail; that missed their second job. They are the local change-history / time-travel log and the only durable record of an INCOMING sync change (command_events records local repo.tx operations only, mutator-grain, never sync apply). The observer owning invalidation makes the reader redundant; it does not make the history substrate redundant. Nothing reads row_events at runtime now — it is a write-only history log. (Triggers fire on the observer's materialize into blocks, tagging incoming changes source='sync'.) ApplyIfNewerSource = 'sync' | 'hydrate' in blockCache.ts is not removed: the observer inherits the 'sync' caller role. A few stale comments referencing rowEventsTail (block.ts, repo.ts) get updated.
    • Not removed — but the two trigger families are kept for different reasons, and one of them deliberately does not run on observer writes:
      • The derived-index maintenance triggers (alias/type/FTS) are not source-gated, so they fire on the observer's plaintext writes to blocks exactly as on local TxImpl writes. This is the family that genuinely runs on both paths, and the observer relies on it to keep the indexes current; pulling it into JS would duplicate maintenance across TxImpl and the observer, re-introducing the dual-path problem the trigger approach removed.
      • The validation triggers (blocks_parent_workspace_check_*, blocks_parent_not_deleted_check_*, block_aliases_workspace_alias_unique) are gated on source IS NOT NULL — local writes only — so they do not fire on observer writes (the observer writes with source = NULL, identical to PowerSync's CRUD-apply today). That is the intended contract, not a gap. Parent/workspace integrity is still backed by the server's composite FK on the plaintext parent_id/workspace_id columns (unencrypted even for E2EE); parent-not-deleted is deliberately permissive on sync (the documented strict-local / permissive-sync asymmetry — the server accepts a child that lands before its parent's tombstone during cross-client ordering). Re-running these in the observer would regress that asymmetry, not close a hole. Alias-uniqueness has no server-side counterpart (block_aliases is local-only), so cross-client alias collisions stay the documented V1-latent limitation, unchanged by the observer.
  • Plus: queries are direct SQL on plaintext — no per-row decrypt cost at query time, no UDF latency.
  • Minus: two writes per every block download (staging, then main) — the universal-observer pattern means plaintext-workspace users pay this cost too, not just E2EE users. Brief window between sync apply and observer pickup where the row only exists in staging (mitigated by app code never querying blocks_synced).
  • Risk: PowerSync's reconciliation semantics. Since blocks_synced is the only table PowerSync directly manages, this is contained — PowerSync never sees the plaintext blocks table. Verify that PowerSync doesn't expect its target table to be authoritative for app queries (it shouldn't — raw tables are a documented pattern).
Why a mirror + observer, and not a simpler download transform. The obvious question is why decryption needs a staging table and a JS observer at all — why not just decrypt each row as it lands locally, so the local DB is plaintext and nothing else changes. That per-row download transform is exactly what doesn't exist. Direct investigation of @powersync/web@1.38.1 / @powersync/common@1.53.1 and the PowerSync team's own statements:
  • No download-side hook exists, natively or as a supported extension point. PowerSyncBackendConnector has only fetchCredentials + uploadData — no download counterpart. Sync-apply happens in the Rust/SQLite core (ps_oplog/ps_data → views); there's no JS callback in that path.
  • Raw tables are static SQL only (put/delete statements with a fixed parameter vocabulary), not a JS transform — and the JS sync client ignores raw tables anyway (powersync-js#672), they need the Rust sync client.
  • The PowerSync maintainer confirmed the gap. Discussion powersync-js#864 is this exact question; simolus3's response: a sync-internals middleware approach is "hacky," "not an official feature," "probably a direction worth exploring" but unroadmapped. Recommended instead: SQL transforms over JSON views (a registered decrypt() SQL function inside json_replace()), or local tables kept in sync via triggers — i.e. the mirror pattern.
  • At-rest DB encryption exists (WebEncryptionOptions.encryptionKey, wa-sqlite Multiple Ciphers) but it's the wrong threat model — it protects local disk, it does not let the server hold ciphertext while the client holds plaintext.
Verdict: a true download transform is not obtainable without depending on unsupported sync internals. Layout B's mirror-table + observer is the sanctioned shape, so it's the design we build on rather than a fallback we settle for.

JSONB migration is no longer load-bearing for E2EE (it was only relevant for the rejected Layout A's no-op trick). It remains an independently motivated perf improvement (docs/follow-ups.md); E2EE doesn't gate on it and it doesn't gate on E2EE.

11. Architectural prep that pays off regardless

These changes have value independently of when (or whether) E2EE actually ships. They make the eventual encryption work small and reversible-shaped.

11.1 Establish the sync boundary module (do now)

Introduce src/sync/transform.ts as identity encodeForWire/decodeFromWire, with the getCek(workspaceId) pluggable callback signature in place from day one. Under Layout B the seam hosts the JS observer that watches PowerSync changes and decrypts staging-table rows into blocks, plus the upload-side encrypt in the uploadData connector hook. The boundary is one named module rather than behavior scattered across repo.ts and services/powersync.ts. Pure no-op refactor today; the eventual E2EE PR is a focused crypto change rather than a cross-cutting refactor.

11.2 Drift guard: server-side content features must filter on mode (codify now)

A short repo-level note: any new server-side feature that reads blocks.content, properties_json, or references_json must include workspaces.encryption_mode = 'none' in its filter, or explicitly document a decision to skip E2EE workspaces. The repo already has a drift-guard pattern from prior design docs; this slots in.

11.3 JSONB migration is independent of E2EE now

The follow-up "JSONB across the data layer" has independent motivation (perf wins across ~29 json_extract/json_each callsites). An earlier draft of this sketch treated JSONB as a prerequisite for E2EE under Layout A; with Layout A rejected and both B and C operating on plaintext at trigger time (see §9.2), that coupling is gone. Pursue JSONB on its own schedule for perf reasons; E2EE doesn't gate on it.

11.4 Discipline rules surfaced from review iteration

Two patterns are worth carrying into future cross-cutting designs — encryption-specific only incidentally.

  • Column-add propagation. Adding a column to a workspace-related table today means touching the migration, workspaceSchema.ts, the regenerated sync-config.yaml, every RPC's RETURNS TABLE projection, every RPC's parameter list (when callers need to supply the new field), TS row interfaces and parse functions, and any clientSchema.ts triggers that reference the column. A separate task is open ("Investigate schema-generator extensions") to extend the existing generator coverage so this becomes structural rather than discipline.
  • Lifecycle moments are first-class and not bundleable. Even in the simple model: "user creates first encrypted workspace" and "user pastes WK on a new device" are distinct moments with distinct preconditions. Name each, its preconditions, and the state that comes into existence at it. Do not write "at signup" or "at first launch" without specifying which event you actually mean.

12. Considered & rejected

A one-line index of what was weighed and dropped. Full rationale for each — along with the rejected storage layouts A and C (§9.2) — lives in e2ee-rejected-alternatives.html.

  • Storage layouts A and C (ciphertext at rest, either letting indexes lapse or UDF-decorating triggers). Rejected in favor of Layout B; see §9.2 and the companion doc.
  • Single opaque payload column. Kills PowerSync's per-column PATCH granularity and forecloses server-side features for plaintext workspaces.
  • Passphrase-derived key hierarchy for v1. Over-budgeted for a personal-leaning tool; its headline sharing benefit needs TOFU/key-transparency it doesn't ship; adds a class of bugs the simple model avoids. Strictly upgradable later (§13).
  • Application-layer compression at the sync boundary. HTTP + TOAST already compress; per-row gzip on sub-1KB blocks usually grows them. The seam keeps the option open but we don't exercise it.
  • Replacing server-side blocks_history with a client-uploaded edit log. Would degrade plaintext workspaces to slightly simplify the E2EE path; keep the trigger (it records ciphertext diffs for E2EE workspaces — still PITR-useful).
  • Full at-rest encryption of the local OPFS SQLite, for v1. v1's lock is a coarse wipe (§6), so at-rest encryption isn't required to land E2EE. But it's the mechanism for a non-destructive lock and for device-theft protection — tracked as a post-v1 option in §14, not flatly rejected.

13. Adding the hierarchy later (if needed)

The simple model and the hierarchy share the entire encryption layer. The hierarchy is strictly an additive layer for storing WKs behind a passphrase and (optionally) delivering them via asymmetric crypto. Nothing about the simple model forecloses the upgrade.

Concrete migration shape, when and if usage justifies it:

  1. Add the wrapping schema. ALTER TABLE workspace_members ADD COLUMN wrapped_cek BYTEA; CREATE TABLE user_vault (...) with kdf_params, wrapped_uvk, public_key, wrapped_private_key. Both nullable / empty until opt-in.
  2. Per-user opt-in flow. User enters a new passphrase → derive MK via Argon2id → generate UVK → generate X25519 keypair → wrap UVK with MK → wrap private key with UVK → insert their user_vault row.
  3. Wrap existing WKs. Wrap each WK with the new UVK and write to workspace_members.wrapped_cek. Note the non-extractable IndexedDB copy (§5) can't be exported, so the bytes come from the user re-pasting each WK from their backup at opt-in time; the local paste flow remains as a backup unlock path regardless.
  4. Future workspaces. The create flow now also wraps the new WK with UVK and stores it. Sharing can use the pubkey-mediated handoff (with whatever TOFU / key-transparency story the hierarchy migration brings).
  5. New device flow. Sign in → enter passphrase → unwrap UVK → unwrap all wrapped_cek values → no per-workspace paste needed. Falls back to per-workspace paste for workspaces where the user opted out (or for workspaces predating their hierarchy adoption).

What's a one-way door: almost nothing. The encryption format is preserved (the AAD includes schema_version for future format revs); the WK is structurally identical to the hierarchy's CEK; collaborators who already received WKs out-of-band continue to hold them — that knowledge persists, but it's the same threat shape as any shared key.

Design seams to preserve in v1 for cheap upgrade:

  • schema_version in the AAD (already in §6).
  • The sync boundary's getCek(workspaceId) callback signature (§9). Simple model implements it as an IndexedDB CryptoKey lookup; hierarchy implements it as an in-memory unwrapped-CEK-map read. Same call site, two implementations.
  • No "user has a passphrase" assumption baked anywhere upstream of the sync boundary. The simple model has no user-level secrets; the hierarchy adds them. Keep the upstream layer ignorant.
  • Don't repurpose workspace_members columns for crypto things. The wrapped_cek column will land cleanly when needed.

14. Open questions

  1. WK rotation. If a workspace key is compromised (a collaborator exfiltrated it, or it leaked via a backup), the owner needs to rotate. Mechanically: client generates a new WK, decrypts every block with old WK and re-encrypts with new WK, also regenerates workspaces.wk_canary under the new WK (otherwise §8.2 validation would fail for anyone unlocking after rotation — the canary would still be encrypted under the old WK), uploads the lot; distributes the new WK to current collaborators out-of-band. The "re-encrypt all blocks" cost is real for large workspaces; worth thinking through the UX (progress bar, atomic rotation token to detect mid-rotation states). The canary is immutable to client writes (§7 enforces this with the workspaces_prevent_e2ee_field_change BEFORE UPDATE trigger), so the rotation path is the dedicated rotate_workspace_wk RPC — SECURITY DEFINER, owned by and running as the table owner (postgres). The RPC updates the canary in the same transaction as the re-encrypt rotation token. Because it is SECURITY DEFINER it bypasses RLS, and it needs a NON-forgeable way past the immutability trigger: the trigger permits a wk_canary change only when current_user is the table owner — which a client on the authenticated role can never become — rather than a forgeable session flag. (Switching to the column-privilege model at that point is an equally valid alternative.) Either way the RPC must self-authorize: its body checks that auth.uid() is the workspace owner before touching the canary, otherwise any authenticated caller could rotate or brick another workspace's canary. The whole immutability design hinges on this RPC being the sole canary writer and self-guarding.
  2. Backfill behavior on first WK paste. The first time a device gets a WK for a workspace, the local plaintext-derived indexes need populating. Under Layout B this means draining that workspace's rows from the blocks_synced staging table through the observer. The backfill is N-block work; UX needs a progress indicator for large workspaces.
  3. Are settings or user preferences in scope? Survey didn't surface a settings table. If preferences are stored server-side and contain user content, they may also need encryption — or be explicitly carved out as metadata.
  4. Could the encryption_mode enum be dropped entirely? Since a workspace is E2EE iff wk_canary IS NOT NULL (§6), the column is a denormalized projection of canary presence — strictly redundant for correctness. Keeping it buys readable server-side feature-gating filters (encryption_mode = 'none' reads better than wk_canary IS NULL) and headroom for a future third mode; dropping it removes a column, its CHECK, its immutability trigger, and a create_workspace param. The current design keeps it as an explicit, server-maintained column because the duality thesis (§10) has server code gating on mode — but nothing in the security model depends on it. Revisit if a third mode never materializes.
  5. Non-destructive lock via at-rest encryption (post-v1). v1's "Lock & wipe local data" (§6) is destructive on purpose — it wipes the local store and re-syncs on unlock — which keeps the implementation a single coarse operation. The nicer property ("drop the key, the data stays but is unreadable") needs the local store encrypted at rest, which PowerSync's web SDK supports (wa-sqlite Multiple Ciphers / WebEncryptionOptions.encryptionKey, per §9.2's transport box). Two decisions gate it: (a) granularity — whole-DB at-rest is one key, so it gives an app-level lock, not per-workspace; a per-workspace non-destructive lock keyed by the WK would need one ATTACH'd encrypted DB file per workspace, which fights PowerSync's single-DB model, caps at SQLITE_MAX_ATTACHED (~10), and makes cross-workspace references harder; (b) key source — for the lock to be meaningful the at-rest key must not be silently re-derivable (an app passphrase gives a real lock but reintroduces passphrase management; a device-stored key only defends against offline device theft). It is additive to, not a replacement for, Layout B's server-side per-column encryption.

15. Tradeoffs we're accepting

  • Metadata leak. Block tree shape (parent_id), order (order_key), edit timing, authorship, deletion flag, and per-column "which column changed" are all visible to the server. Acceptable for the stated threat model.
  • No server-side features for E2EE workspaces, ever. Search, AI, summaries, server-rendered share links. This is a real product cost; the opt-in toggle is the user surfacing it explicitly.
  • Lost WK = lost data. No recovery in v1. The UI surfaces this loudly at create time; users who need recovery can store the WK in multiple places themselves.
  • Sharing requires an out-of-band channel. This is a deliberate non-goal, not a defect — an in-app share button would re-introduce the MITM concern. Users pick Signal, password-manager share, paper, whatever.
  • Pasting WK on each new device. One extra step per workspace per device. Tolerable for small N; if N grows past ~5 per user, that's the signal to add the hierarchy (§13).
  • Ciphertext-diff history for E2EE workspaces.
  • Local index side-tables contain plaintext-derived data. block_aliases.alias, FTS trigrams, etc. are plaintext on local disk — as is the app-visible blocks table itself under Layout B (only PowerSync's blocks_synced staging holds ciphertext locally). Irrelevant for threat model A (server can't read content); relevant only if the threat model is later broadened to local-disk attackers, in which case OS disk encryption or the app-level at-rest encryption tracked in §14 is the answer (OS encryption alone doesn't give the "drop the key to lock" property, since its key is always present while the user is logged in).
  • Client-trust required — the residual the web-delivery model can't fully close. The code that holds the keys is re-fetched from the server on every load, and the root HTML that would anchor any integrity check is itself server-served — so there is no in-band trust root, and a backdoored build can exfiltrate keys or plaintext. This is inherent to all browser-delivered E2EE, not specific to this design. It splits into three cases whose mitigations are graded — none a full solution:
    • Passive / data-at-rest (DB dump, backup leak, curious operator, subpoena): fully defeated — the server only ever holds ciphertext plus the canary. This is the guarantee the whole design buys, and it holds regardless of the build problem.
    • Active, broad (backdoor everyone): detectable, not prevented — via open source + reproducible builds + published build hashes / signed provenance (e.g. a Sigstore/Rekor attestation emitted from CI). A bad build becomes observable to anyone auditing the delivered artifact.
    • Active, targeted (clean code to auditors, backdoored code to one user): the hard residual — only an out-of-band verifier that checks the actually-delivered bytes (a "Code Verify"–style browser extension, or a native client with a signed, reviewed update channel) meaningfully addresses it. Documented as future, not solved.
    What the design already leans on: keys are non-extractable CryptoKeys (§5) — a backdoored load can still use the handle to decrypt live, but can't dump raw key bytes, narrowing the attack to a noisy live session; and the code host is separate from the data host — the adversary that is in scope (§2: full read access to the Supabase/PowerSync backend) does not control code delivery (static hosting), so it structurally can't mount this attack at all. Cheap bar-raisers we can add: a strict CSP with a locked connect-src (constrains exfiltration to the first party — weak against a malicious first party, since the server sets it, but cuts off third-party supply-chain paths), and publishing build hashes / provenance from CI (tracked separately). Net: strong against the passive threat the design targets, detection against broad tampering, and an honest residual against a targeted backdoor.

This is a sketch, not a commitment. The architectural prep work in §11 (sync boundary, drift guard, JSONB) is the recommended starting point regardless of whether we proceed with §5–§9 on a specific timeline.
