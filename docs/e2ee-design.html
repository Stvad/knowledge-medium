End-to-End Encryption Design

Status: sketch / not yet committed to. Discussion artifact.

Per-workspace opt-in E2EE so that Supabase and the network see only opaque ciphertext for content in encrypted workspaces, while plaintext workspaces remain fully server-readable and can keep all current and future server-side features.

1. Goals & non-goals

Goals

  • Workspace owner can mark a workspace as end-to-end encrypted at creation. Mode is immutable thereafter.
  • For E2EE workspaces, Supabase (and anyone with DB access) sees only ciphertext for blocks.content, blocks.properties_json, blocks.references_json.
  • Plaintext workspaces remain bit-identical to today — same schema, same RLS, same access to server-side features (current and future).
  • Multi-device: the same user can decrypt their E2EE workspaces on any device they sign in on, given their passphrase.
  • Workspace sharing: an inviter can grant a new member access to an E2EE workspace without ever revealing the workspace key to the server.

Non-goals (for v1)

  • Hiding metadata (workspace structure, block tree shape, edit timing, who-edited-what).
  • Defense against a compromised local device. OS-level disk encryption is the right tool; OPFS already gives per-origin isolation.
  • Cryptographic verification of authorship (signed writes). Useful later, not in scope now.
  • Account recovery without the user's passphrase. Lost passphrase = lost E2EE data unless we ship recovery codes (see open questions).
  • Server-side features over encrypted workspaces (search, AI, embeddings). They simply don't apply.

2. Threat model

The threat we're addressing is the most common one users actually ask about: "can the hosting provider read my notes?" Concretely, in v1:

  • In scope: passive attacker with full read access to the Supabase Postgres, the PowerSync replica, network traffic to/from Supabase, and Supabase logs/backups. They must not learn the contents of E2EE-workspace blocks.
  • Out of scope: attacker who controls the user's browser at runtime (they can read decrypted data and exfiltrate the key); attacker who can compromise the client build pipeline (they can ship a backdoored client); attacker who can perform traffic analysis on edit patterns (still possible).
  • Out of scope, deliberately, in v1: active attacker who can substitute server responses. The invite-time public-key delivery (§8.4) is a trust-on-server channel: a malicious server can return its own X25519 key in place of Bob's and read everything in the shared workspace. Closing this gap requires out-of-band verification — TOFU + safety-number compare, key transparency log, or similar — which is real engineering and UX. Flagged as a v2 requirement in §13 and §14; v1 honestly degrades to passive-only protection on first-share, full passive protection thereafter (once the CEK is on Bob's device, the server can't read past traffic).

3. Why the current architecture is friendly to this

The codebase is already structured in a way that makes E2EE tractable rather than invasive:

  • Server is blind storage today. RLS policies filter by workspace membership, not content. No tsvector, no pg_trgm, no AI/embeddings, no webhooks read block content.
  • Workspace is already the RLS boundary. Per-workspace key material maps cleanly onto a unit that already exists.
  • Per-user OPFS SQLite via kmp-v6-<user_id>.db gives physical isolation between users on a shared device.
  • PowerSync upload compaction already coalesces edits per block, which keeps wire overhead low independent of encoding.
  • Client-side edit log already exists (tx_context, row_events, command_events) — useful if we ever need to move history off the server for encrypted workspaces.

4. Design shape

Two properties drive the entire design:

  1. Per-workspace opt-in. One user can have plaintext and E2EE workspaces side by side on the same device, in the same local DB.
  2. Per-column encryption with the existing schema. We encrypt the three content columns in place rather than consolidating them into one opaque payload. This preserves PowerSync's column-level PATCH granularity, keeps plaintext workspaces bit-identical, and avoids a schema migration that would touch every workspace.
Decision. Threat model A (Supabase/network can't read content). Per-workspace opt-in. Per-column AEAD with self-identifying enc:v1: prefix in the existing content/properties_json/references_json columns.

5. Key hierarchy

  • Master Key (MK) — derived from passphrase via Argon2id. Lives only in memory. Never persisted, never sent anywhere.
  • User Vault Key (UVK) — random 256-bit. Wrapped by MK and stored server-side (user_vault.wrapped_uvk). Multi-device unlock works because any device that can derive MK can unwrap the same UVK.
  • Content Encryption Key (CEK) — random per workspace. Each workspace member holds the CEK wrapped by their UVK (workspace_members.wrapped_cek). One workspace = one CEK shared by all members.
  • Asymmetric keypair — per user. Public key published in user_vault.public_key; private key stored wrapped by UVK. Used at invite time to hand a CEK to a new member without an interactive session.

6. Cryptographic choices

ConcernChoiceWhy
KDF Argon2id (params: m=64MiB, t=3, p=1) Modern memory-hard KDF. If a libsodium dep is unwanted, PBKDF2-SHA256 (600k iters) via WebCrypto is the fallback.
Symmetric AEAD AES-256-GCM Native in WebCrypto, no dependency. XChaCha20-Poly1305 only meaningfully wins on platforms without AES hardware.
Nonce 96-bit random per ciphertext Per-column random nonce. Reuse probability negligible at our write volumes; keep a guard rail counter.
AAD (associated data) block_id ‖ workspace_id ‖ column_name ‖ schema_version Prevents the server from swapping ciphertexts between blocks, between workspaces, or between columns of the same block.
Asymmetric primitive X25519 for key wrapping, optional Ed25519 for future signing Tiny, well-understood. libsodium covers both cleanly; WebCrypto's ECDH P-256 is an acceptable fallback if we stay dependency-light.
Ciphertext format "enc:v1:" + base64url(nonce ‖ ciphertext) Self-identifying prefix lets the decode layer detect E2EE rows without a workspace lookup. The v1 reserves a clean upgrade path.
Downgrade-attack guard. The enc:v1: prefix is convenient but not authoritative on its own — a hostile server could strip it and serve attacker-chosen plaintext. The decode layer must cross-check the row's workspaces.encryption_mode: if the workspace is E2EE and the value lacks the prefix, refuse to render (and surface a tamper warning).

7. Schema additions

One new migration. No changes to blocks columns themselves.

-- Encryption mode is an immutable property of a workspace.
ALTER TABLE workspaces ADD COLUMN encryption_mode TEXT
  NOT NULL DEFAULT 'none'
  CHECK (encryption_mode IN ('none', 'e2ee'));

-- Trigger: encryption_mode is set at create time and never changes.
CREATE TRIGGER workspaces_lock_encryption_mode
  BEFORE UPDATE ON workspaces
  FOR EACH ROW
  WHEN (OLD.encryption_mode IS DISTINCT FROM NEW.encryption_mode)
  EXECUTE FUNCTION raise_immutable_column();

-- At-rest CEK envelope: each member of an E2EE workspace holds the CEK
-- symmetrically wrapped by their own UVK.
ALTER TABLE workspace_members ADD COLUMN wrapped_cek BYTEA;
-- Constraint enforced via trigger (cross-table check): required iff workspace is E2EE.

-- In-transit CEK envelope: when Alice invites Bob, she wraps the CEK with
-- Bob's *public key* (X25519) and parks it here. On accept_invitation, Bob
-- unwraps with his private key, re-wraps with his UVK, and the result lands
-- in workspace_members.wrapped_cek (the at-rest envelope above). The two
-- envelopes are deliberately different shapes so that the unlock path only
-- ever needs the UVK, while invitation delivery doesn't require Bob to be
-- online or unlocked.
ALTER TABLE workspace_invitations
  ADD COLUMN wrapped_cek_for_invitee BYTEA;

-- Per-user vault: enables multi-device unlock and invite-time CEK wrapping.
CREATE TABLE user_vault (
  user_id              UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  kdf_params           JSONB  NOT NULL,  -- algo, salt (base64), m, t, p
  wrapped_uvk          BYTEA  NOT NULL,  -- UVK encrypted by MK
  public_key           BYTEA  NOT NULL,  -- X25519 public, published openly
  wrapped_private_key  BYTEA  NOT NULL,  -- X25519 private, wrapped by UVK
  created_at           TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at           TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- RLS on user_vault: each user can read/write their own row. Public keys
-- can't sit behind a "shares-a-workspace-with-owner" predicate (the very
-- first invite, when Alice and Bob share nothing, would deadlock on it),
-- but they also can't be openly looked up by email: invite_member_by_email
-- today never reaches into auth.users and so doesn't leak which emails are
-- registered. A bare get_user_public_key(email) would be a new account-
-- existence oracle. Instead, gate the lookup on an existing invitation
-- the caller created (see §8.4): invite_member_by_email itself returns the
-- invitee's public_key when known. Probing reduces to "create a fake
-- invitation against this workspace and read the result" — auditable,
-- rate-limitable, and bound to a workspace the caller owns, rather than
-- a free oracle for any authenticated user.

Sync stream needs to learn the new columns. Both powersync/sync-config.yaml and src/data/workspaceSchema.ts enumerate the columns clients hydrate. The yaml is generated by scripts/gen-sync-config.ts from the schema file, so the actual edit is in workspaceSchema.ts:

  • Add encryption_mode to WORKSPACE_COLUMNS — clients need it on the read path to gate triggers and to do the downgrade check from §6.
  • Add wrapped_cek to WORKSPACE_MEMBER_COLUMNS — the §8.2 unlock flow can't unwrap CEKs that never reach the client.
  • Run yarn gen:sync-config to refresh the yaml; yarn check:sync-config in CI catches anyone who forgets.

Notably not changed: blocks columns, sync rule predicates, the streams themselves. The new columns ride existing streams.

Naming convention for envelopes. Wrapped-key columns are named after the wrapping key, not the wrapped contents, so the unwrap path is obvious from the column name: wrapped_cek (UVK-wrapped, at rest) versus wrapped_cek_for_invitee (pubkey-wrapped, in transit). At the TS layer the two should be branded types — a WrappedForUvk can't accidentally flow into a function expecting a WrappedForPubkey envelope. One round of review already tripped on a column called just wrapped_cek being asked to do double duty; the convention is the cheap fix.

8. User-visible flows

8.1 First-time vault setup

Triggered the first time a user either creates an E2EE workspace or is invited to one. Plaintext-only users never see this flow.

  1. Prompt for a passphrase (and confirm).
  2. Generate random salt; run Argon2id → MK.
  3. Generate random UVK (256-bit). Generate X25519 keypair.
  4. Wrap UVK with MK; wrap private key with UVK.
  5. POST user_vault row with kdf_params, wrapped UVK, public key, wrapped private key.
  6. Cache MK and UVK in memory (and unwrapped private key, lazily).

8.2 Sign-in / vault unlock

  1. Supabase signInWithPassword as today.
  2. If the user has a user_vault row, prompt for passphrase. (If they don't, skip — they have no E2EE workspaces yet.)
  3. Re-derive MK using stored kdf_params. Unwrap UVK. Unwrap private key.
  4. Fetch all workspace_members rows for this user with wrapped_cek; unwrap into an in-memory map workspace_id → CEK.
  5. Mark vault as "unlocked." App content code now resolves CEKs synchronously.
Locked-but-syncing state. The session can be valid before the vault is unlocked (or if the user dismisses the passphrase prompt). PowerSync will happily download E2EE blocks during that window. The data layer must surface a "locked" placeholder for those rows rather than throwing on decrypt — otherwise a user who closes the prompt sees a cascade of errors.

8.3 Create an encrypted workspace

  1. If no user_vault, run §8.1 inline.
  2. Generate a random CEK.
  3. Wrap CEK with the creator's UVK.
  4. Call create_workspace(name, encryption_mode='e2ee', wrapped_cek=…). The RPC inserts the workspace, the owner's membership row, and the wrapped CEK atomically.

8.4 Invite a user to an encrypted workspace

The CEK reaches a new member through two different envelopes back-to-back. The in-transit envelope (asymmetric, wrapped with the invitee's public key) lets the inviter deliver the CEK without the invitee being online. The invitee rewraps it into the at-rest envelope (symmetric, wrapped with their own UVK) and passes the result as a parameter to accept_invitation. Without the rewrap a second device would have no way to recover the CEK; without doing it before the RPC call the cross-table trigger that requires workspace_members.wrapped_cek for E2EE memberships would reject the insert, since the membership row is created atomically inside accept_invitation.

Three RPC signatures change. invite_member_by_email grows a return shape so the inviter learns the invitee's public key without a separate lookup (which would otherwise be an email-enumeration oracle — see the RLS note in §7). list_my_pending_invitations surfaces the transit envelope to the invitee. accept_invitation accepts the re-wrapped at-rest envelope atomically with the membership insert. 

-- Inviter learns the invitee's public key (if vaulted) in the same call
-- that creates the invitation. NULL pubkey ⇒ invitee has no vault; the
-- client surfaces "they need to set up a vault first" and either blocks
-- (v1) or queues a defer-wrap (later).
CREATE OR REPLACE FUNCTION invite_member_by_email(
    p_workspace_id text,
    p_email        text,
    p_role         text
  ) RETURNS TABLE(
    invitation_id        text,
    invitee_public_key   bytea   -- new; NULL when no vault for that email
  ) ...;

-- After invite_member_by_email returns a non-NULL public_key, the inviter
-- wraps the CEK with it and attaches the transit envelope to the same
-- invitation row. Caller must be the inviter on that invitation.
CREATE OR REPLACE FUNCTION attach_invite_envelope(
    p_invitation_id           text,
    p_wrapped_cek_for_invitee bytea
  ) RETURNS void ...;

-- Surface the transit envelope so the invitee's client can read it.
CREATE OR REPLACE FUNCTION list_my_pending_invitations()
  RETURNS TABLE(
    id text,
    workspace_id text,
    workspace_name text,
    email text,
    role text,
    invited_by_user_id text,
    create_time bigint,
    wrapped_cek_for_invitee bytea   -- new; NULL for plaintext-workspace invites
  ) ...;

-- Accept the (re-wrapped) at-rest envelope atomically with membership insert.
CREATE OR REPLACE FUNCTION accept_invitation(
    p_invitation_id text,
    p_wrapped_cek   bytea DEFAULT NULL   -- required iff target workspace is E2EE
  ) RETURNS workspace_members ...;

p_wrapped_cek defaults to NULL so plaintext-workspace invites still call the RPC with one argument. The cross-table trigger on workspace_members enforces the "required iff E2EE" half — a NULL parameter on an E2EE invitation aborts the insert with a clear error rather than silently breaking the unlock path later. Splitting invite_member_by_email (creates row + reveals pubkey) from attach_invite_envelope (attaches the wrap) keeps the second step optional — a plaintext-workspace invite just doesn't call it.

Trust-on-server caveat (v1). Alice receives Bob's public key from the server in invite_member_by_email's response, and wraps the CEK with whatever she gets back. A malicious server can substitute its own X25519 public key here and read every block in the resulting shared workspace. v1 lives with this against the (re-scoped) passive threat model in §2; v2 closes it with TOFU + safety-number verification, or a key-transparency log. Flagged in §13 and §14.

The awkward case is inviting someone who hasn't signed up yet (no public key on file). Two defensible options:

  • Block: refuse E2EE-workspace invites to users without a vault. Surface "they need to sign up and set a vault first" UX. Simplest, slightly worse UX.
  • Defer-wrap: store the invite pending; when the invitee signs up and publishes a key, queue a notification to the inviter; next time the inviter is online, wrap the CEK and complete the invitation. Better UX, more moving parts, requires a "complete pending invites" client step.

We can ship "block" in v1 and "defer-wrap" later without schema changes.

9. Sync boundary & per-column encryption

The encrypt/decrypt seam lives in a new module — call it src/sync/transform.ts — that sits between the app domain and the PowerSync CRUD edge. It is the single named place where "what crosses the wire" gets transformed.

The seam owns two pure functions: encodeForWire(block, mode, cek) and decodeFromWire(row, mode, cek). They start as identity functions and grow transforms over time. Today: per-workspace per-column AES-GCM. Later: schema-version migration of payload shapes, redaction for share-link previews, optional compression if data sizes ever justify it.

What "per-column" actually means on the wire

Each of the three content columns is sealed independently with its own random nonce. The AAD carries the column name so the server can't silently move ciphertext from one column to another and have it still decrypt as valid.

9.2 Local SQLite triggers and the JSONB connection

Storing ciphertext in the same columns the app already uses sounds clean until you remember that the local SQLite schema (src/data/internals/clientSchema.ts) runs JSON functions over those columns on every write. Concretely:

  • blocks_row_event_insert/_update call json(NEW.properties_json) and json(NEW.references_json) to snapshot the row into the local edit log.
  • blocks_alias_insert/_update walk json_each(NEW.properties_json, '$.alias') to maintain the block_aliases index.
  • blocks_type_insert/_update walk json_each(NEW.properties_json, '$.types') for block_types.
  • The startup backfills (BACKFILL_BLOCK_ALIASES_SQL, BACKFILL_BLOCK_TYPES_SQL) do the same against historical rows.
  • blocks_fts_insert/_update reads NEW.content directly — not JSON, but still content-bearing.

If we drop a raw enc:v1:… string into properties_json today, json() and json_each() abort with "malformed JSON" and the write fails before it ever reaches the upload queue. The naive per-column scheme as written can't land without addressing this.

There are three paths. They aren't mutually exclusive; #1 and #2 compose.

  1. Adopt JSONB first (recommended). The follow-up "JSONB across the data layer" (docs/follow-ups.md) migrates properties_json and references_json from text to jsonb end-to-end. With JSONB columns, the natural shape for ciphertext is a JSON string scalar — "enc:v1:…" — which is structurally valid JSONB. json() returns the string, json_each(value, '$.alias') gracefully returns zero rows on a non-object value, and the alias/types/row-event triggers become a no-op on encrypted rows without any explicit branching. This is the single change that removes the most friction.
  2. Mode-gate the content-bearing triggers that don't get JSONB's free pass. The FTS trigger reads content (plain text, not JSON), so JSONB doesn't help it; without a gate it would index ciphertext trigrams that can never match user queries (useless, mildly bloated). Add a WHEN clause that consults workspaces.encryption_mode (or equivalently, sniffs the enc:v1: prefix) and skips the index maintenance for encrypted rows. Same gate applies to the alias/types triggers if we don't adopt #1 — they'd still need an explicit branch to avoid touching json_each on a string.
  3. Move encryption to a download-decrypt boundary instead, keeping local plaintext. This is the "Layout B" alternative. The local DB stores plaintext for all workspaces; encryption happens only at PowerSync's upload/download seam. Local triggers and indexes all work normally, including for E2EE workspaces. The cost is that PowerSync doesn't expose a per-row download transform hook, so this requires either a custom client connector or a buffer-table indirection — meaningfully more invasive than #1 + #2. Threat-model-A doesn't require it (the OPFS database is already per-user, and local at-rest protection is the OS disk encryption's job), so we'd only take this path if losing the derived local indexes for E2EE workspaces turns out to be unacceptable for the product.
Working decision. Path #1 + #2 (Layout A). JSONB migration is a strong prerequisite — not strictly required, but doing E2EE without it means hand-gating ~7 triggers and accepting that any future trigger over those columns is one author away from a footgun. The FTS gate remains needed either way.

Consequence for E2EE workspaces under Layout A: the trigger-maintained local indexes (block_aliases, block_types, blocks_fts) silently don't get populated for encrypted rows. Features that rely on them — in-workspace alias autocomplete, type-filtered queries, local FTS — therefore don't work for E2EE workspaces unless we layer on a separate "decrypt-then-populate" pass at vault unlock time. That pass is straightforward to add later (read encrypted rows, decrypt in memory, write into the side tables, never touch blocks itself), but it's not free and it's the reason to flag the feature loss explicitly in §10 and §14.

11. Architectural prep that pays off regardless

These changes have value independently of when (or whether) E2EE actually ships. They make the eventual encryption work small and reversible-shaped.

11.1 Establish the sync boundary module (do now)

Introduce src/sync/transform.ts as identity encode/decode. Route all PowerSync upload mutations and all reads from local SQLite through it. Pure no-op refactor. ~50 LOC of plumbing. After this lands, the E2EE PR is a focused crypto change, not a cross-cutting refactor.

11.2 Decouple auth session from vault unlock (do now)

Introduce useAuthSession() and useUnlockedVault() hooks even before E2EE. Vault is always trivially "unlocked" today. App code that touches content depends on the unlocked-vault hook. Makes the future passphrase prompt, recovery code, or WebAuthn-PRF unlock all drop-in replacements rather than refactors.

11.3 Pre-provision the vault schema (do now)

Add the user_vault table now. Rows aren't created at signup — the schema's non-null kdf_params/wrapped_uvk/wrapped_private_key all depend on the user's passphrase, which we deliberately don't ask for from plaintext-only users. The row gets inserted during §8.1's first-time vault setup, the first time a user either creates an E2EE workspace or accepts an invite to one. Pre-provisioning just the table (empty until the first opt-in) means the eventual E2EE work doesn't have to coordinate a schema migration with the crypto change.

Earlier draft generated a keypair at signup so signed-write-style features could be added later without a backfill problem. That doesn't actually work against the proposed schema — the private key needs a wrapping key, which needs a passphrase, which we don't have at signup. A future signed-write feature would need its own (lighter) keypair table that doesn't require passphrase wrapping, designed when there's a concrete user.

11.4 Drift guard: server-side content features must filter on mode (codify now)

A short repo-level note: any new server-side feature that reads blocks.content, properties_json, or references_json must include workspaces.encryption_mode = 'none' in its filter, or explicitly document a decision to skip E2EE workspaces. The repo already has a drift-guard pattern from prior design docs; this slots in.

11.5 Land the JSONB migration before the encryption work

The follow-up "JSONB across the data layer" already has independent motivation (perf wins across ~29 json_extract/json_each callsites, including the hot alias/types triggers). Doing it before the encryption work is what makes the per-column ciphertext shape in §9 actually tractable: a ciphertext-as-JSON-string-scalar value is a valid JSONB value, so the existing triggers naturally no-op on encrypted rows instead of aborting. Without this prep, the E2EE work has to hand-gate every JSON-bearing trigger and live with the fragility that any future trigger adds another footgun. See §9.2.

11.6 Discipline rules surfaced from review iteration

The review cycle on this sketch produced repeated findings that fell into a small number of shapes. Each pattern below has a one-liner rule worth carrying into future cross-cutting designs — encryption-specific only incidentally. They're cheap to follow if named, easy to miss otherwise.

  • Column-add propagation. Adding a column to a workspace-related table today means touching the migration, workspaceSchema.ts, the regenerated sync-config.yaml, every RPC's RETURNS TABLE projection, every RPC's parameter list (when callers need to supply the new field), TS row interfaces and parse functions, and any clientSchema.ts triggers that reference the column. Three review findings came from missing one of these layers. A separate task is open ("Investigate schema-generator extensions") to extend the existing generator coverage so this becomes structural rather than discipline.
  • Cross-table NOT NULL invariants force atomic RPC parameters. When a constraint like "workspace_members.wrapped_cek is required iff the workspace is E2EE" is enforced by a trigger (because Postgres can't express it as a single-table CHECK), the RPC that creates the dependent row must accept the required value as a parameter and write it atomically. A "create now, fill in later" two-step flow looks reasonable in the design but trips the trigger at INSERT time. Rule: every cross-table NOT NULL becomes a required RPC parameter, not a follow-up write.
  • Lifecycle moments are first-class and not bundleable. "Signup" and "first vault setup" and "first E2EE workspace" are different events with different preconditions. Bundling them — e.g. "generate a keypair at signup" when the schema requires a passphrase that doesn't exist yet — produces impossible flows. Name each moment, its preconditions, and the table rows that come into existence at it. Do not write "at signup" or "at first launch" without specifying which user lifecycle event you actually mean.
  • RPC-as-oracle check. Any new RPC that takes an external identifier (email, phone, username) and returns user-specific data is a potential account-existence oracle. Before adding one, ask: if I pass in an identifier that does not belong to me, what does my ability to call this tell me about that identifier? If the answer is non-trivial, gate the call behind an existing relationship (membership, invitation the caller created, etc.) rather than letting any authenticated user probe. Folds naturally into §11.4's drift-guard pattern.
  • Server-mediated key delivery is trust-on-server unless verified. Anything where the client receives identity material from the server and uses it to encrypt for that identity (the §8.4 invite flow) inherits the server's trustworthiness for that exchange. v1 lives with this against a passive threat model; v2 needs TOFU + safety numbers, a key-transparency log, or out-of-band verification. Carry the assumption explicitly — don't claim active-attacker protection that isn't there.

12. Considered & rejected

Rejected: consolidate content/properties_json/references_json into a single opaque payload column.

Initially attractive because it makes the encryption layer trivial: one sealed blob. Rejected for two compounding reasons:

  • Sync granularity. PowerSync's CRUD upload PATCHes individual columns. Collapsing to one column makes every edit rewrite the entire payload. For blocks with large properties_json and small text edits to content, this is a real regression.
  • Server-side features for plaintext workspaces. The whole point of opt-in is that unencrypted workspaces stay first-class. Making content opaque for everyone forecloses future server-side search, AI, indexed properties, etc. — even for users who never opted into E2EE.
Rejected: application-layer compression at the sync boundary.

HTTP transport already does gzip/br for any response over ~1KB. Postgres TOAST already compresses large TEXT/JSONB on disk. Per-row gzip on typical block sizes (often well under 1KB) frequently grows the payload due to framing overhead. Application-layer compression only starts to win with shared dictionaries trained on the corpus, which is real engineering effort for marginal gains. The sync boundary keeps the option open; we just don't exercise it.

Rejected: replace server-side blocks_history with a client-uploaded edit log.

Tempting because the client-side edit log already exists. But for plaintext workspaces, server-side history earns its keep — server-computed diffs, queryable, no client round-trip. Ripping it out to make the E2EE path slightly cleaner would degrade the plaintext workspace experience. Keep the trigger; accept that for E2EE workspaces it records ciphertext diffs (still useful for PITR, audit-of-when).

Rejected: full at-rest encryption of the local OPFS SQLite.

Out of scope (not part of threat model A). OPFS already gives per-origin and per-user isolation via kmp-v6-<user_id>.db. Device-level protection is OS disk encryption's job. We can revisit if a device-theft threat model is later added.

13. Open questions

  1. Recovery codes in v1? Forgotten passphrase = lost E2EE data unless we ship a recovery code (a high-entropy string, printed at vault setup, that wraps a copy of the UVK). Recovery codes weaken security mildly (the user must store them somewhere) and introduce real UX complexity. Decision deferred; default is "no recovery in v1, document the consequence loudly at setup."
  2. Invite-before-vault policy. Block vs. defer-wrap (§8.4). I'd ship block and revisit if it bites real users.
  3. Argon2id vs. PBKDF2. Argon2id is the right primitive but needs a dependency (e.g. argon2-browser or libsodium.js, ~150KB gzipped). PBKDF2-SHA256 at 600k iterations is in WebCrypto natively, no dep, but weaker against GPU/ASIC attackers. We're protecting one-user-at-a-time data, not a credential dump, so either is defensible.
  4. UVK rotation. Should users be able to change their passphrase without re-encrypting everything? Yes — that's exactly what the UVK-wrapping layer is for: rotate MK by re-wrapping UVK. No content re-encryption needed. Worth confirming this is in scope for v1.
  5. Membership removal. When a member is removed from an E2EE workspace, the CEK is no longer secret from them (they cached it). Strict guarantee requires CEK rotation, which means re-encrypting every block. Mark as a known limitation for v1; add CEK rotation as a v2 capability with a "rekey workspace" client-side job.
  6. Are settings or user preferences in scope? Survey didn't surface a settings table. If preferences are stored server-side and contain user content, they may also need encryption — or be explicitly carved out as metadata.
  7. Layout A vs Layout B for local storage. Path A (§9.2) stores ciphertext in local SQLite and accepts that the derived local indexes don't apply to E2EE workspaces until we add a decrypt-then-populate pass. Path B keeps local plaintext and intercepts at the PowerSync download seam — functionally nicer but architecturally heavier. Current working assumption is A; revisit if product testing shows the lost local features are unacceptable for E2EE workspaces.
  8. JSONB migration as gate vs. parallel work. §11.5 calls for landing the JSONB migration before the encryption work to get the trigger no-op behavior for free. Worth confirming whether that ordering is acceptable (it bundles two independently motivated changes) or whether E2EE should ship with explicit trigger gates and JSONB follows separately.
  9. When to add invite-time public-key verification. v1 takes the §2 scope-down: trust the server's pubkey response during invite. v2 needs some form of out-of-band verification — TOFU + safety-number compare (cheapest UX, Signal-style), a key-transparency log (best technical answer, real engineering), or in-product fingerprint exchange. The decision is more product than crypto: how much UX friction is acceptable for invite flow, and how loud should the "first share with new contact" warning be?

14. Tradeoffs we're accepting

  • Metadata leak. Block tree shape (parent_id), order (order_key), edit timing, authorship, deletion flag, and per-column "which column changed" are all visible to the server. Acceptable under threat model A.
  • No server-side features for E2EE workspaces, ever. Search, AI, summaries, server-rendered share links. This is a real product cost; the opt-in toggle is the user surfacing it explicitly.
  • Lost passphrase = lost data (modulo recovery codes if we ship them).
  • Ciphertext-diff history for E2EE workspaces.
  • Membership removal is best-effort until v2 brings CEK rotation.
  • Local derived indexes inactive in E2EE workspaces under Layout A (§9.2). Alias autocomplete, type-filtered queries, and local FTS don't work for encrypted workspaces until a decrypt-then-populate pass is added.
  • Invite-time pubkey trust-on-server. The server can substitute its own X25519 key in invite_member_by_email's response and read everything in the resulting shared workspace. v1 scopes the threat model to passive attackers (§2); active substitution at invite time becomes a real concern only when v2 adds verification.
  • Client-trust required. A backdoored build can exfiltrate keys. Subresource integrity, reproducible builds, and audit are mitigations, not solutions.

This is a sketch, not a commitment. The architectural prep work in §11 (sync boundary, auth/vault split, public-key identity, drift guard) is the recommended starting point regardless of whether we proceed with §5–§9 on a specific timeline.
