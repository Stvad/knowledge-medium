End-to-End Encryption Design

Status: sketch / not yet committed to. Discussion artifact.

Per-workspace opt-in E2EE with user-managed workspace keys. Supabase and the network see only ciphertext for content in encrypted workspaces; plaintext workspaces remain fully server-readable and keep all current and future server-side features. The user holds the key, types/pastes it on each device, and shares it with collaborators over whatever channel they trust.

1. Goals & non-goals

Goals

  • Workspace owner can mark a workspace as end-to-end encrypted at creation. Mode is immutable thereafter.
  • For E2EE workspaces, Supabase (and anyone with DB access) sees only ciphertext for blocks.content, blocks.properties_json, blocks.references_json.
  • Plaintext workspaces remain bit-identical to today — same schema, same RLS, same access to server-side features (current and future).
  • Multi-device: a user can decrypt their E2EE workspaces on any device they sign in on by pasting the workspace key once per device.
  • Sharing: an owner can give another user access by sending them the workspace key over whatever channel they trust (Signal, in person, encrypted email, etc.).

Non-goals (for v1)

  • Hiding metadata (workspace structure, block tree shape, edit timing, who-edited-what).
  • Defense against a compromised local device. OS-level disk encryption is the right tool; OPFS already gives per-origin isolation.
  • Cryptographic verification of authorship (signed writes). Useful later, not in scope now.
  • Server-mediated key delivery. The app deliberately does not transmit workspace keys through its own server channel — the user picks the channel. This eliminates an entire class of MITM concerns at the cost of one extra step at share time.
  • Account recovery without the user's workspace key. Lost key = lost data; the UI surfaces this clearly at create time.
  • Server-side features over encrypted workspaces (search, AI, embeddings). They simply don't apply.

2. Threat model

The threat we're addressing is the most common one users actually ask about: "can the hosting provider read my notes?" Concretely, in v1:

  • In scope: passive or active attacker with full read access to the Supabase Postgres, the PowerSync replica, network traffic to/from Supabase, and Supabase logs/backups. They must not learn the contents of E2EE-workspace blocks.
  • Out of scope: attacker who controls the user's browser at runtime (they can read decrypted data and exfiltrate the key); attacker who can compromise the client build pipeline (they can ship a backdoored client); attacker who can perform traffic analysis on edit patterns; attacker who can intercept the user's chosen out-of-band channel for sharing the workspace key (that's a property of the channel, not of this system).

Notably, the user-managed key model means active attackers on the server gain very little power over confidentiality: the server never sees a workspace key in any form, so there's nothing to substitute, MITM, or extract. The attacker can serve ciphertext rows or refuse to serve them; they can't read them.

3. Why the current architecture is friendly to this

The codebase is already structured in a way that makes E2EE tractable rather than invasive:

  • Server is blind storage today. RLS policies filter by workspace membership, not content. No tsvector, no pg_trgm, no AI/embeddings, no webhooks read block content.
  • Workspace is already the RLS boundary. Per-workspace key material maps cleanly onto a unit that already exists.
  • Per-user OPFS SQLite via kmp-v6-<user_id>.db gives physical isolation between users on a shared device.
  • PowerSync upload compaction already coalesces edits per block, which keeps wire overhead low independent of encoding.
  • Client-side edit log already exists (tx_context, row_events, command_events) — useful if we ever need to move history off the server for encrypted workspaces.

4. Design shape

Three properties drive the design:

  1. Per-workspace opt-in. One user can have plaintext and E2EE workspaces side by side on the same device, in the same local DB.
  2. Per-column encryption with the existing schema. Encrypt the three content columns in place rather than consolidating them into one opaque payload. Preserves PowerSync column-level PATCH granularity, keeps plaintext workspaces bit-identical, and avoids a schema migration that would touch every workspace.
  3. User-managed workspace key. Each E2EE workspace has a single symmetric key. The user is responsible for storing it and transmitting it to collaborators. The server has no role in key management — no key store, no key wrapping, no per-user identity material.
Decision. v1 ships the simple model: one symmetric workspace key (WK) per encrypted workspace, generated client-side, surfaced to the user once at create, stored by the user (password manager, paper, etc.), and pasted on additional devices. Sharing is out-of-band. No server-side key store, no asymmetric crypto, no KDF, no recovery codes.

5. Workspace key: shape, lifecycle, and storage

The workspace key (WK) is the only secret in the entire design.

PropertyValue
Material256 random bits, generated client-side via crypto.getRandomValues.
LifetimeOne per workspace, generated at workspace create. Stable for the life of the workspace unless explicitly rotated.
User-facing formatkmp-wk-1:<base32(32 bytes)> — 61-character paste-friendly string (52 unpadded base32 chars + 9-char prefix). The 1 reserves a clean upgrade path.
Where it lives(a) the user's password manager / backup of their choice; (b) localStorage on each device the user has pasted it on, keyed by (user_id, workspace_id) (e.g. wk:<user_id>:<workspace_id>). localStorage rather than the local SQLite DB so the WK store has its own lifecycle — a future "lock the app" or "clear my keys" action can drop WKs without touching the PowerSync-managed sync state. Reuses the codebase's existing localStorage adapter rather than introducing IndexedDB as a new storage surface. Raw 32 bytes encode to ~44 chars base64; the cross-tab storage event is a useful side effect (paste-once propagates to other tabs). Never the server, in any form.
Validation on pasteDecrypt workspaces.wk_canary and verify the plaintext equals the workspace id. Works on empty workspaces (no blocks needed). AEAD failure or plaintext mismatch → reject with "that key doesn't decrypt this workspace's data." See §7 (canary column) and §8.2 (paste flow).

The UI shows the WK exactly once, at create time, with prominent "save this somewhere safe — there is no recovery" copy and a recommendation to paste it into a password manager. A confirmation step ("type the last 6 characters") ensures the user actually saw it rather than dismissing the dialog.

6. Cryptographic choices

ConcernChoiceWhy
Symmetric AEAD AES-256-GCM Native in WebCrypto, no dependency. XChaCha20-Poly1305 only meaningfully wins on platforms without AES hardware.
Nonce 96-bit random per ciphertext Per-column random nonce. Reuse probability negligible at our write volumes.
AAD (associated data) block_id ‖ workspace_id ‖ column_name ‖ schema_version Prevents the server from swapping ciphertexts between blocks, between workspaces, or between columns of the same block. schema_version reserves room for future format upgrades (e.g. a hierarchy migration).
Ciphertext format "enc:v1:" + base64url(nonce ‖ ciphertext) Self-identifying prefix lets the decode layer detect E2EE rows without a workspace lookup. The v1 reserves a clean upgrade path.
WK display/paste format kmp-wk-1:<base32> Base32 (Crockford or RFC 4648 without padding) is paste-friendly across messaging apps and looks reasonable on paper. The kmp-wk-1: prefix makes mis-pasted keys identifiable.
KDF None — the WK is the key. No passphrase in v1, so no derivation. If the hierarchy is added later (§13), Argon2id over the user's vault passphrase is the right primitive.
Downgrade-attack guard. The enc:v1: prefix is convenient but not authoritative on its own — a hostile server could strip it and serve attacker-chosen plaintext. The decode layer must cross-check the row's encryption mode and refuse to render (with a tamper warning) if an E2EE-claimed row lacks the prefix.
Pin encryption_mode locally; don't trust server metadata. workspaces.encryption_mode is itself server-supplied state arriving over the same sync path the threat model treats as hostile. A malicious server could serve an existing E2EE workspace's row with encryption_mode = 'none' and a NULL wk_canary; if the client trusted that flip, the UDF would pass attacker plaintext through (no enc:v1: prefix expected for plaintext workspaces), and TxImpl would stop encrypting future edits — leaking new content to the server.

Fix: maintain a client-side known-E2EE workspaces set in localStorage alongside the WKs. The set is populated at two points: (a) when a user creates an E2EE workspace on this device — the creator generated the WK locally, has the canary, and trivially knows the workspace is E2EE; and (b) when a user pastes a WK on another device that decrypts the workspace's wk_canary (the canary is authentically encrypted under the user's WK, so it can't be forged). Once a workspace is in the set, the client treats it as E2EE permanently on this device regardless of any later encryption_mode value from the server. Mode flip from E2EE to plaintext requires explicit local user action (e.g. "remove this workspace's key"), never server initiative. This is TOFU for the mode signal, mirroring how §5's WK store works.

Workspaces not in the local set fall back to trusting encryption_mode on first encounter — otherwise there's no way to know to prompt for a WK. That first encounter is the only window where a malicious server could lie ("this workspace is plaintext, don't ask for a key"); subsequent loads are pinned.

7. Schema additions

One small migration. No changes to blocks columns; no new tables.

-- Encryption mode is an immutable property of a workspace.
ALTER TABLE workspaces ADD COLUMN encryption_mode TEXT
  NOT NULL DEFAULT 'none'
  CHECK (encryption_mode IN ('none', 'e2ee'));

-- Per-column immutability trigger, matching the existing
-- blocks_prevent_workspace_change() pattern in the consolidated
-- migration (concrete function per invariant, not a generic helper).
CREATE OR REPLACE FUNCTION workspaces_prevent_encryption_mode_change()
  RETURNS trigger
  LANGUAGE plpgsql AS $$
begin
  if OLD.encryption_mode is distinct from NEW.encryption_mode then
    raise exception 'workspaces.encryption_mode is immutable (% -> %)',
      OLD.encryption_mode, NEW.encryption_mode;
  end if;
  return NEW;
end;
$$;

CREATE TRIGGER workspaces_prevent_encryption_mode_change_trg
  BEFORE UPDATE ON workspaces
  FOR EACH ROW
  EXECUTE FUNCTION workspaces_prevent_encryption_mode_change();

-- create_workspace accepts encryption_mode and a key-check canary
-- so we can validate pasted WKs even on workspaces that have no
-- block rows yet (freshly invited or just-created E2EE workspaces).
-- The canary is an AEAD-encrypted known plaintext (the workspace id);
-- decrypting it with the candidate WK both authenticates the key
-- (AEAD failure = wrong key) and confirms it's bound to this workspace
-- (right key on the wrong workspace's canary fails the plaintext check).
-- wk_canary format: enc:v1:base64url(nonce || ciphertext) of workspace_id
-- as the plaintext, with AAD = workspace_id || "canary" || schema_version.
ALTER TABLE workspaces ADD COLUMN wk_canary TEXT;

-- NULL for plaintext workspaces; required for E2EE so new devices can
-- validate a pasted WK before any blocks exist. Cross-column CHECK.
ALTER TABLE workspaces
  ADD CONSTRAINT workspaces_wk_canary_required_for_e2ee
  CHECK (encryption_mode = 'none' OR wk_canary IS NOT NULL);

-- Immutable after create, parallel to encryption_mode: an editor with
-- write access to the workspaces row could otherwise replace the canary
-- and brick WK validation for other members.
CREATE OR REPLACE FUNCTION workspaces_prevent_wk_canary_change()
  RETURNS trigger
  LANGUAGE plpgsql AS $$
begin
  if OLD.wk_canary is distinct from NEW.wk_canary then
    raise exception 'workspaces.wk_canary is immutable';
  end if;
  return NEW;
end;
$$;

CREATE TRIGGER workspaces_prevent_wk_canary_change_trg
  BEFORE UPDATE ON workspaces
  FOR EACH ROW
  EXECUTE FUNCTION workspaces_prevent_wk_canary_change();
-- One-shot exception for the rotate_workspace_wk RPC (§14) which
-- updates the canary alongside re-encrypted blocks.

CREATE OR REPLACE FUNCTION create_workspace(
    p_name             text,
    p_encryption_mode  text DEFAULT 'none',  -- 'none' | 'e2ee'
    p_workspace_id     text DEFAULT NULL,    -- client-chosen for E2EE so the
                                             -- canary's AAD/plaintext line up;
                                             -- NULL falls back to server-generated
                                             -- UUID for plaintext (current behavior)
    p_wk_canary        text DEFAULT NULL     -- required iff p_encryption_mode = 'e2ee'
  ) RETURNS jsonb ...;

That's the entire server-side schema change. Notably not added (saved for the eventual hierarchy migration, §13): user_vault, public keys, workspace_members.wrapped_cek, workspace_invitations.wrapped_cek_for_invitee. The invitation RPCs (invite_member_by_email, list_my_pending_invitations, accept_invitation) keep their existing signatures — permissions flow through them, key material does not.

Sync stream learns one new column. Both powersync/sync-config.yaml and src/data/workspaceSchema.ts enumerate columns explicitly. The yaml is generated by scripts/gen-sync-config.ts from the schema file, so the actual edit is to WORKSPACE_COLUMNS in workspaceSchema.ts:

  • Add encryption_mode to WORKSPACE_COLUMNS.
  • Run yarn gen:sync-config to refresh the yaml; yarn check:sync-config in CI catches anyone who forgets.

8. User-visible flows

8.1 Create an encrypted workspace

  1. User clicks "Create encrypted workspace".
  2. Client generates a 32-byte WK via crypto.getRandomValues; encodes it as kmp-wk-1:<base32>.
  3. Client picks the workspace id (UUID v4), then encrypts the id with the WK (AAD = workspace_id || "canary" || schema_version) to produce the wk_canary. The canary is a known-plaintext AEAD blob that any future device can decrypt to validate a pasted WK, even on a workspace that has no blocks yet.
  4. Client stores the WK in localStorage at (user_id, workspace_id) → raw bytes, and pins the workspace into the local known-E2EE set (per §6's downgrade-pin callout) so future loads on this device don't trust a server-supplied encryption_mode = 'none' for this workspace.
  5. Client calls create_workspace(p_name, p_encryption_mode='e2ee', p_workspace_id, p_wk_canary) with the client-chosen id so the canary's plaintext and AAD line up with the server's stored row. Server inserts workspaces with the canary and the owner's workspace_members row, exactly as the plaintext flow does. No WK bytes are sent to the server — only the canary (which is opaque ciphertext).
  6. UI: show the WK once with strong "save this securely — no recovery is possible" copy. Recommend pasting into a password manager. Confirm-by-retype on the last 6 characters before dismissing.

8.2 Open an E2EE workspace on a new device

  1. User signs in on device 2. PowerSync syncs encrypted block rows for workspaces they're a member of.
  2. Client checks localStorage at (user_id, workspace_id) — no WK present.
  3. UI surfaces a "this workspace needs its key" prompt instead of rendering ciphertext.
  4. User pastes their stored WK.
  5. Client validates by decrypting workspaces.wk_canary with the candidate WK and checking the result equals the workspace id. AEAD failure (wrong key) and plaintext mismatch (right key, wrong workspace) both reject. Works for freshly invited / empty workspaces because the canary is created at workspace-create time, not derived from blocks.
  6. On success: store in localStorage, pin the workspace into the local known-E2EE set (§6), render normally. On failure: surface "that key doesn't decrypt this workspace's data," let the user retry.

8.3 Share an E2EE workspace with another user

  1. Owner invites the collaborator via the existing invite_member_by_email → accept_invitation flow. The RPC chain doesn't change; the membership row is plaintext-only metadata.
  2. Owner sends the WK to the collaborator over a channel the user picks themselves — Signal, in person, sealed envelope, encrypted email, password manager share, whatever they trust for this contact. The app does not transmit the WK through its own server channel.
  3. Collaborator accepts the invitation server-side as today. In their UI, the workspace appears with the same "needs key" prompt as §8.2. They paste the WK they received out-of-band.
Sharing is the user's responsibility. The app does not provide an in-app "send key to Bob" button on purpose. Any in-app key delivery routes through the server, which would re-introduce the MITM concern that motivated this design. The "annoying" extra step (using Signal / a password manager / a piece of paper) is the security property.

9. Sync boundary & per-column encryption

The encrypt/decrypt seam lives in a new module — call it src/sync/transform.ts — that sits between the app domain and the PowerSync CRUD edge. It is the single named place where "what crosses the wire" gets transformed.

The seam owns two pure functions: encodeForWire(block, mode, getCek) and decodeFromWire(row, mode, getCek). They start as identity functions and grow transforms over time. Today: per-workspace per-column AES-GCM. Later: schema-version migration of payload shapes, redaction for share-link previews, optional compression if data sizes ever justify it.

Pluggable key lookup. The seam takes getCek(workspaceId) as a callback rather than reading any global. In the simple model, the implementation is "localStorage lookup at (user_id, workspace_id)." If the hierarchy is added later (§13), the implementation becomes "look up in the in-memory unwrapped-CEK map populated at vault unlock." Same call site, two implementations. This is the single most important seam to get right for keeping the hierarchy migration cheap.

9.1 What "per-column" actually means on the wire

Each of the three content columns is sealed independently with its own random nonce. The AAD carries the column name so the server can't silently move ciphertext from one column to another and have it still decrypt as valid.

9.2 Where local plaintext lives: Layouts B and C

The codebase leans heavily on SQL-side processing of content. Triggers in src/data/internals/clientSchema.ts and src/plugins/references/localSchema.ts maintain plaintext-derived indexes (block_aliases, block_types, blocks_fts, references-graph tables) on every write. Queries against those indexes drive core UX: parseReferences, alias autocomplete, type filters, local FTS.

An earlier draft considered Layout A: store ciphertext locally, let the indexes silently not populate for E2EE rows, add a "decrypt-then-populate" pass at WK paste. Not viable. Alias search is a core Roam-style UX; degrading it in E2EE workspaces is a non-starter. Layout A is documented here only to explain why it was rejected.

Two real candidates remain. Both produce server-side ciphertext and a fully-functional local query surface; they differ on where the plaintext lives in the local DB and how it gets there.

Layout B — local plaintext via staging table + JS observer

The local blocks table holds plaintext for everything (encrypted and plaintext workspaces alike). PowerSync's sync apply lands all blocks rows — for both plaintext and E2EE workspaces — in a parallel blocks_synced staging table (the row shape that matches the server). PowerSync's sync rule projects from public.blocks, so the FROM-clause-as-row_type behavior would route everything to the client blocks table by default; the gen-sync-config change re-targets the stream's row_type to blocks_synced, which means every downloaded row (plaintext or encrypted) lands in staging, not just E2EE rows.

A JS observer on PowerSync's change events processes every staging insert/update. For rows in plaintext workspaces it copies through unchanged. For rows in E2EE workspaces it decrypts the content columns with the WK and writes plaintext to blocks. Both cases end up populating blocks as the app-visible table; existing triggers in clientSchema.ts and plugins fire on plaintext writes as today.

The observer is universal, not E2EE-only. If it only handled E2EE rows, plaintext-workspace sync updates would be stuck in blocks_synced and never reach the app's blocks table — breaking the requirement that plaintext workspaces stay unchanged. The cost: every downloaded row pays one extra local write (staging → blocks), not just E2EE rows. For typical workloads this is small; for a large workspace's initial sync it's N extra writes. The dual-target alternative (Postgres view partitioning E2EE workspaces, two sync rules, one mapping to blocks and one to blocks_synced) avoids the cost on plaintext users at the price of more Postgres-side complexity; not pursued in v1 unless profiling shows the universal-observer cost matters.

The observer is a reconciler, not just a copier. Once PowerSync writes only to blocks_synced, the app-visible blocks table is no longer PowerSync-managed — the observer is now responsible for the local/remote merge that PowerSync's CRUD machinery normally handles for raw tables. If a user has a pending local edit in blocks (already written by TxImpl, queued in ps_crud awaiting upload) and an older server snapshot arrives in staging, a blind copy would overwrite the local edit. The observer needs to consult ps_crud before each staging-row apply: if there's a pending CRUD entry for that block_id (or the local row's updated_at is newer than staging's), skip the copy and let the upload echo reconcile when it returns. This is the load-bearing complexity of Layout B that the original "just decrypt and copy" framing understated.

Upload is symmetric. App writes plaintext to blocks as today; the existing upload-routing trigger captures plaintext into ps_crud; PowerSync's uploadData connector hook intercepts the CRUD batch and encrypts the content columns before sending to the server. The server only ever sees ciphertext.

  • What changes in code: add blocks_synced raw table to blockSchema.ts (PowerSync target); update powersync/sync-config.yaml (via scripts/gen-sync-config.ts) so the row_type for the blocks stream maps to blocks_synced rather than blocks — the sync rule's FROM-clause name is the row_type that maps to the client raw table (called out as load-bearing in sync-config.yaml's own header comment), so leaving it pointing at public.blocks means ciphertext lands in the plaintext blocks table and the observer never sees the rows it's meant to decrypt. Add JS observer wiring at the sync boundary (universal, with ps_crud / updated_at reconciliation per the callout above); encrypt-on-upload in the connector hook. Existing trigger code in clientSchema.ts and plugins is untouched (they fire on plaintext writes to blocks as today). Update blockUploadPatchJsonSql in clientSchema.ts to unconditionally include workspace_id in every PATCH envelope — the current trigger emits only changed columns, so a content-only edit reaches uploadData without workspace_id, and the encryption hook needs that field to look up the WK and construct AAD. Always-emit is a few bytes of wire overhead per PATCH; the alternative (have the hook read the local row before encrypting) adds a DB round-trip per upload.
  • Plus: queries are direct SQL on plaintext — no per-row decrypt cost at query time, no UDF latency.
  • Minus: two writes per every block download (staging, then main) — the universal-observer pattern means plaintext-workspace users pay this cost too, not just E2EE users. Brief window between sync apply and observer pickup where the row only exists in staging (mitigated by app code never querying blocks_synced).
  • Risk: PowerSync's reconciliation semantics. Since blocks_synced is the only table PowerSync directly manages, this is contained — PowerSync never sees the plaintext blocks table. Verify that PowerSync doesn't expect its target table to be authoritative for app queries (it shouldn't — raw tables are a documented pattern).

Layout C — ciphertext at rest, UDF-decorated triggers

The local blocks table holds ciphertext at rest, exactly matching the server. A new async user-defined function — e2ee_decrypt_json(workspace_id, value) → text — is registered against the local SQLite via sqlite3.create_function. The UDF needs to know the workspace's encryption_mode to dispatch correctly; cache it JS-side at session start (the mode is immutable per §7) so each call is a Map lookup, not a SQL subquery. Four behaviors based on (mode, input):

  • Plaintext workspace, any input: pass-through unchanged. The UDF only ever runs on the three content columns; for plaintext workspaces those are already plaintext.
  • E2EE workspace, encrypted input (enc:v1: prefix), WK available: look up the WK, call crypto.subtle.decrypt, return plaintext JSON text.
  • E2EE workspace, encrypted input, WK absent (locked workspace): return NULL. Sync apply can land encrypted rows before the user has pasted the WK; the UDF returning NULL makes json_each(NULL, '$.alias') yield zero rows, json(NULL) yield NULL, and the indexes/audit log skip the row. Sync apply succeeds. Per-workspace backfill on WK paste repopulates.
  • E2EE workspace, unprefixed input: return NULL and record a tamper signal. This is the downgrade-attack defense at the UDF layer. A hostile server could strip the enc:v1: prefix and serve attacker-controlled plaintext for an E2EE row; the §6 downgrade guard catches this at parseBlockRow for app reads, but trigger-driven side tables and the row-event/BlockCache path fire before that guard. Without this UDF-level check, attacker plaintext would seed FTS, aliases, references, and the row-event cache as trusted content. Treat unprefixed-in-E2EE the same as locked: trigger maintenance no-ops, surface a tamper warning in the UI.

The row_event snapshot triggers in clientSchema.ts need the UDF wrap too — otherwise after_json would contain enc:v1: ciphertext, which rowEventsTail.ts would pass to BlockCache.applyIfNewer as if it were plaintext, and the cache would emit ciphertext to anyone listening. UDF-wrapping ensures the cache only sees plaintext for E2EE rows (or NULL placeholders when locked).

Triggers wrap column references in the UDF before passing into json_each, json_extract, etc: 

-- Before:
INSERT OR IGNORE INTO block_aliases (block_id, workspace_id, alias, alias_lower)
SELECT NEW.id, NEW.workspace_id, je.value, LOWER(je.value)
FROM json_each(NEW.properties_json, '$.alias') AS je
WHERE typeof(je.value) = 'text';

-- After:
INSERT OR IGNORE INTO block_aliases (block_id, workspace_id, alias, alias_lower)
SELECT NEW.id, NEW.workspace_id, je.value, LOWER(je.value)
FROM json_each(e2ee_decrypt_json(NEW.workspace_id, NEW.properties_json), '$.alias') AS je
WHERE typeof(je.value) = 'text';
Critical: under Layout C, encrypt happens at write time, not upload time. Local writes from TxImpl.create / TxImpl.update must hit the sync boundary's encodeForWire step before the INSERT/UPDATE reaches the blocks table; otherwise the row sits as plaintext locally until a sync echo overwrites it — breaking the ciphertext-at-rest invariant and tripping the §6 downgrade guard on the user's own freshly-written content. The upload-routing trigger then captures whatever's already in the row (ciphertext for E2EE workspaces) and forwards it unchanged. This is the asymmetry with Layout B: B encrypts at upload and decrypts at download; C encrypts at write and decrypts at read (via UDF for triggers, via parseBlockRow for app reads). The transform module's encodeForWire seam is load-bearing under both layouts; the difference is where in the lifecycle it fires.
Codebase survey: where Layout C touches

A focused survey of every place that reads blocks.content/properties_json/references_json came back with a more contained surface than I expected, with one specific schema refactor load-bearing for correctness.

What's already structured for Layout C:

  • Most reads are centralized. The dominant read path routes through txEngine.ts's SELECT ${COLUMN_LIST} FROM blocks → parseBlockRow in repo.ts → getBlockDataById. One decrypt call in parseBlockRow covers it. Caveat: the survey didn't catch every raw-SQL site — see "What the survey did surface" below for the two that bypass parseBlockRow.
  • No WHERE content LIKE '%foo%' or WHERE json_extract(properties_json, ...) = ? in app code. Property-shaped filters ride block_aliases/block_types side tables maintained by triggers. But two raw queries do operate on these columns directly and need Layout C handling — covered below.
  • No generated columns or CHECK constraints over these columns.
  • PowerSync is content-blind internally. Conflict resolution is metadata- driven; the apply path runs the same triggers as local writes.

What the survey did surface:

  • Trigger surface is wider than just clientSchema.ts. Confirmed UDF-wrap targets: row-event, alias, type, and FTS triggers in src/data/internals/clientSchema.ts; the three triggers in src/plugins/references/localSchema.ts (references-graph maintenance); the daily-notes trigger in src/plugins/daily-notes/localSchema.ts reading properties_json. The pattern propagates — any future plugin that maintains a derived index over these columns needs the same UDF wrap.

    Explicit exception: the upload-routing triggers in clientSchema.ts (blocks_upload_insert / blocks_upload_update) must NOT be UDF-wrapped. Their job is to capture NEW.content/NEW.properties_json/NEW.references_json into ps_crud for PowerSync to upload to Supabase. Under Layout C those columns already hold ciphertext (per the encrypt-at-write callout above), and the upload must forward ciphertext unchanged. Wrapping these in the decrypt UDF would queue plaintext for upload — the exact opposite of what we want.
  • Raw-SQL queries that bypass parseBlockRow and operate on the content columns directly. The initial survey claim ("reads are centralized") was wrong. Subsequent review rounds have found progressively more sites; the list below is what's confirmed but should not be treated as exhaustive:
    • src/data/repo.ts:1553-1559 — property-reprojection scan, FROM blocks b, json_each(b.properties_json) prop WHERE prop.key IN (?). UDF-wrap the json_each argument, or skip E2EE rows and rely on the per-workspace backfill on WK paste.
    • src/data/repo.ts:1338-1343 — property-deletion warning count, json_extract(b.properties_json, ?) IS NOT NULL. UDF-wrap or count via the trigger-maintained side tables.
    • src/extensions/extensionLookup.ts:40-49 — reads b.properties_json raw and JSON.parses it client-side to find extension labels. Decrypt before parse; or join through block_aliases (which would be plaintext via the UDF-wrapped alias trigger).
    • src/data/targets.ts:234-251 — alias-seat reader, SELECT deleted, properties_json, content FROM blocks WHERE id = ?, client-side JSON.parse. Decrypt before parse, or route this through the centralized read path.
    • src/plugins/references/renameProcessor.ts:60-66 — backlink rename reads source.content directly. Decrypt at the call site or route through parseBlockRow.
    • src/plugins/find-replace/dataExtension.ts:80-89 — instr(content, ?) literal-substring search. Route through FTS for E2EE workspaces (blocks_fts.content already holds plaintext via the UDF-wrapped FTS trigger).
    • src/data/internals/typedBlockQuery.ts lines 198 and 472-475 — the typed-query compiler builds FROM blocks b and emits every where predicate as json_extract(b.properties_json, ?). This backs repo.queryBlocks / subscribeBlocks, which are centralized query APIs used throughout the data layer. Under Layout C the json_extract runs against ciphertext and fails or skips. Fix: parameterize the compiler so the JSON-text expression is pluggable, and emit e2ee_decrypt_json(b.workspace_id, b.properties_json) for E2EE-scoped queries (or unconditionally, accepting the UDF passthrough cost for plaintext workspaces). The compiler change is small but the consumer surface is large — anywhere repo.queryBlocks runs across an E2EE workspace.
    Each is mechanical once identified, but the list keeps growing under review and almost certainly isn't complete. This is a real cost of Layout C: it scatters encryption-awareness across the codebase wherever raw SQL touches a content column. Worth weighting in the B-vs-C tradeoff (see working decision below).
  • Backfill SQL statements also scan encrypted columns directly. Marker-gated one-shot scans run on schema bootstrap: BACKFILL_BLOCK_ALIASES_SQL, BACKFILL_BLOCK_TYPES_SQL, BACKFILL_BLOCKS_FTS_SQL in src/data/internals/clientSchema.ts, and BACKFILL_BLOCK_REFERENCES_SQL in src/plugins/references/localSchema.ts. They use json_each(b.properties_json, ...), raw b.content, or json_each(b.references_json) directly. If any of these runs after encrypted rows exist (new device's first sync of an existing E2EE workspace), they either throw on enc:v1: text or build indexes from ciphertext. Fix: UDF-wrap each backfill, or mode-skip E2EE rows and rely on the per-workspace WK-paste backfill to cover them.
  • One expression index needs converting to a side table. src/plugins/daily-notes/localSchema.ts:105 has CREATE INDEX idx_blocks_daily_note_date ON blocks (json_extract(properties_json, '$.dailyNoteDate')) — an expression index directly over the encrypted column. We can't use the UDF inside the expression because the UDF's output depends on external state (the WK store); marking it SQLITE_DETERMINISTIC would lie, and the index would contain pre-WK NULLs that don't get rebuilt when the WK lands. The fix is to convert this to a trigger-maintained block_daily_notes side table following the same pattern as block_aliases/block_types. One file's worth of change, follows an existing pattern, but it has to land alongside the encryption work or calendar date-range queries break for E2EE workspaces.
  • Backfill on WK paste. When a workspace's WK becomes available, the trigger-maintained side tables for blocks that arrived encrypted need re-populating. Either re-fire the maintenance triggers via a no-op write (UPDATE blocks SET id = id WHERE workspace_id = ? AND deleted = 0), or run a per-workspace variant of BACKFILL_BLOCK_ALIASES_SQL/_TYPES_SQL/ _FTS scoped by workspace_id. Marker rows in client_schema_state already track first-time backfills; add per-(workspace, index) markers so the rebuild is idempotent.
  • UDF determinism flag. Not marked SQLITE_DETERMINISTIC — the WK store is external state, and lying about determinism causes the expression-index problem above. SQLite won't memoize calls within a query, so the same encrypted value read by N triggers gets decrypted N times. A per-row JS-side decrypt cache keyed by (workspace_id, ciphertext) can claw that back if profiling shows it matters; not load-bearing for v1.
  • What changes in code: register the async UDF at DB open (repoProvider.ts); decrypt in parseBlockRow; UDF-wrap the non-upload triggers in clientSchema.ts (row-events, alias, type, FTS) plus the triggers in plugins/references/localSchema.ts and plugins/daily-notes/localSchema.ts — the blocks_upload_insert/_update triggers stay un-wrapped per the exception noted above; UDF-wrap or mode-skip the four backfill statements (BACKFILL_BLOCK_ALIASES_SQL, BACKFILL_BLOCK_TYPES_SQL, BACKFILL_BLOCKS_FTS_SQL in clientSchema.ts, BACKFILL_BLOCK_REFERENCES_SQL in plugins/references/localSchema.ts); fix the seven known raw-SQL readers (repo.ts:1553-1559 reprojection scan; repo.ts:1338-1343 property-count; extensions/extensionLookup.ts:40-49; data/targets.ts:234-251; plugins/references/renameProcessor.ts:60-66; plugins/find-replace/dataExtension.ts:80-89 via FTS reroute; data/internals/typedBlockQuery.ts:198,472-475 typed-query compiler — parameterize its JSON-text expression so E2EE-scoped queries emit the UDF-wrapped form); convert the daily-notes expression index to a side table; wire encodeForWire into TxImpl.create / TxImpl.update so local writes hit the encrypt step before INSERT/UPDATE reaches blocks (not at upload — see the encrypt-at-write callout above); per-workspace backfill on WK paste. No new top-level tables for the encryption itself (the daily-notes side table is a local schema change, not an E2EE-specific one). Note: the list of raw-SQL readers above keeps growing under review; treat this as a non-exhaustive checklist rather than a complete one.
  • Plus: local DB mirrors the server (mental-model parity); reads keep going through the existing centralized path; existing index machinery stays in SQL.
  • Minus: every trigger fire on an encrypted row pays a WebCrypto decrypt cost. Bulk operations (per-workspace backfill on first WK paste, large imports) pay it N times.
  • Risk — needs empirical verification before committing: wa-sqlite supports async UDFs at the API level (sqlite3.create_function; see the "should return asynchronously" test in node_modules/@journeyapps/wa-sqlite/test/callbacks.test.js), and PowerSync uses the Asyncify build that enables them (mc-wa-sqlite-async.mjs). What I have not verified is whether PowerSync's full query/trigger execution pipeline correctly awaits async UDFs through every code path. PoC scope: register a trivial async UDF, invoke from a trigger via sync apply, confirm the await happens. An afternoon's work; gates the choice between B and C.
  • Note: local index side-tables (block_aliases, etc.) end up containing plaintext-derived data even though blocks.* columns are ciphertext. Irrelevant for threat model A; flagged for any future broadening to local-disk attackers.
Working decision. Layout B and Layout C are both viable; Layout A is rejected. The choice hinges on:
  1. Whether the PoC confirms PowerSync awaits async UDFs end-to-end (if not, C is off the table).
  2. How big the Layout C touch surface actually is. Each review round has found more raw SQL sites that need awareness of the encryption boundary (currently 7 in app code including the typed-query compiler, plus 4 backfill statements, almost certainly more). Under Layout B this surface is empty — local writes are plaintext, all existing readers Just Work.
  3. Which set of risks is easier to live with — B's two-write download path and JS observer wiring (a single concentrated mechanism), versus C's encryption-awareness scattered across the codebase wherever raw SQL touches a content column (mechanical per-site, but a discipline rule the codebase has to maintain forever).
The earlier draft leaned toward C on grounds of "less new infrastructure." That lean was based on a survey claim that turned out incomplete in three successive review rounds. The Layout C touch list is now meaningfully wider, and the long-tail concern is real: any future plugin or feature that adds raw SQL against the content columns is one author away from breaking E2EE silently. Re-lean: I'd default toward Layout B now unless the PoC for C is particularly clean. B's complexity is concentrated at one named seam; C's is structural and ongoing.

JSONB migration is no longer load-bearing for E2EE under either layout (it was only relevant for Layout A's no-op trick). It remains an independently motivated perf improvement (docs/follow-ups.md); E2EE doesn't gate on it and it doesn't gate on E2EE.

11. Architectural prep that pays off regardless

These changes have value independently of when (or whether) E2EE actually ships. They make the eventual encryption work small and reversible-shaped.

11.1 Establish the sync boundary module (do now)

Introduce src/sync/transform.ts as identity encodeForWire/decodeFromWire, with the getCek(workspaceId) pluggable callback signature in place from day one. Under Layout B the seam also hosts the JS observer that watches PowerSync changes and decrypts staging-table rows into blocks; under Layout C it just hosts the upload-side encrypt (download decrypt happens in-SQL via the UDF). Either way, the boundary is one named module rather than behavior scattered across repo.ts and services/powersync.ts. Pure no-op refactor today; the eventual E2EE PR is a focused crypto change rather than a cross-cutting refactor.

11.2 Drift guard: server-side content features must filter on mode (codify now)

A short repo-level note: any new server-side feature that reads blocks.content, properties_json, or references_json must include workspaces.encryption_mode = 'none' in its filter, or explicitly document a decision to skip E2EE workspaces. The repo already has a drift-guard pattern from prior design docs; this slots in.

11.3 JSONB migration is independent of E2EE now

The follow-up "JSONB across the data layer" has independent motivation (perf wins across ~29 json_extract/json_each callsites). An earlier draft of this sketch treated JSONB as a prerequisite for E2EE under Layout A; with Layout A rejected and both B and C operating on plaintext at trigger time (see §9.2), that coupling is gone. Pursue JSONB on its own schedule for perf reasons; E2EE doesn't gate on it.

11.4 Discipline rules surfaced from review iteration

Two patterns are worth carrying into future cross-cutting designs — encryption-specific only incidentally.

  • Column-add propagation. Adding a column to a workspace-related table today means touching the migration, workspaceSchema.ts, the regenerated sync-config.yaml, every RPC's RETURNS TABLE projection, every RPC's parameter list (when callers need to supply the new field), TS row interfaces and parse functions, and any clientSchema.ts triggers that reference the column. A separate task is open ("Investigate schema-generator extensions") to extend the existing generator coverage so this becomes structural rather than discipline.
  • Lifecycle moments are first-class and not bundleable. Even in the simple model: "user creates first encrypted workspace" and "user pastes WK on a new device" are distinct moments with distinct preconditions. Name each, its preconditions, and the state that comes into existence at it. Do not write "at signup" or "at first launch" without specifying which event you actually mean.

12. Considered & rejected

Rejected: consolidate content/properties_json/references_json into a single opaque payload column.

Initially attractive because it makes the encryption layer trivial: one sealed blob. Rejected for two compounding reasons:

  • Sync granularity. PowerSync's CRUD upload PATCHes individual columns. Collapsing to one column makes every edit rewrite the entire payload.
  • Server-side features for plaintext workspaces. The whole point of opt-in is that unencrypted workspaces stay first-class. Making content opaque for everyone forecloses future server-side search, AI, indexed properties, etc. — even for users who never opted into E2EE.
Rejected: passphrase-derived key hierarchy for v1.

A passphrase → MK → UVK → per-workspace CEK hierarchy (with per-user X25519 keypair for server-mediated invite delivery) is what most multi-user E2EE products ship. The earlier iteration of this sketch went deep on that design. It buys: one passphrase for N workspaces, multi-device unlock without re-pasting per workspace, server-brokered key delivery during invite, passphrase rotation without re-encrypting blocks, and CEK rotation for member removal.

Not the right v1 for this product:

  • Over-budgeted. For a personal-leaning knowledge tool, N is usually 1–5, not 50. A password manager covers the multi-key UX entirely.
  • The headline sharing benefit isn't actually delivered in v1. Server-brokered key delivery requires the server to be trusted for that exchange. Closing the active-attacker gap (a malicious server substituting its own pubkey) needs TOFU + safety numbers or a key-transparency log — real engineering and UX. Without that, the hierarchy's "no out-of-band" advantage is illusory; the simple model's deliberate out-of-band step is more honest.
  • Adds a class of bugs the simple model doesn't have. Iterating on the hierarchy sketch surfaced findings across pubkey distribution, email enumeration, envelope confusion, cross-table NOT NULL invariants, RPC signature widening, and lifecycle bundling. The simple model has none of these because there is no per-user identity material on the server.
  • Strictly upgradable later (see §13). The encryption shape is identical; the hierarchy just adds a wrapping layer on top of the WK. Nothing about the simple model forecloses it.
Rejected: application-layer compression at the sync boundary.

HTTP transport already does gzip/br for any response over ~1KB. Postgres TOAST already compresses large TEXT/JSONB on disk. Per-row gzip on typical block sizes (often well under 1KB) frequently grows the payload due to framing overhead. Application-layer compression only starts to win with shared dictionaries trained on the corpus, which is real engineering effort for marginal gains. The sync boundary keeps the option open; we just don't exercise it.

Rejected: replace server-side blocks_history with a client-uploaded edit log.

Tempting because the client-side edit log already exists. But for plaintext workspaces, server-side history earns its keep — server-computed diffs, queryable, no client round-trip. Ripping it out to make the E2EE path slightly cleaner would degrade the plaintext workspace experience. Keep the trigger; accept that for E2EE workspaces it records ciphertext diffs (still useful for PITR, audit-of-when).

Rejected: full at-rest encryption of the local OPFS SQLite.

Out of scope. OPFS already gives per-origin and per-user isolation via kmp-v6-<user_id>.db. Device-level protection is OS disk encryption's job. We can revisit if a device-theft threat model is later added.

13. Adding the hierarchy later (if needed)

The simple model and the hierarchy share the entire encryption layer. The hierarchy is strictly an additive layer for storing WKs behind a passphrase and (optionally) delivering them via asymmetric crypto. Nothing about the simple model forecloses the upgrade.

Concrete migration shape, when and if usage justifies it:

  1. Add the wrapping schema. ALTER TABLE workspace_members ADD COLUMN wrapped_cek BYTEA; CREATE TABLE user_vault (...) with kdf_params, wrapped_uvk, public_key, wrapped_private_key. Both nullable / empty until opt-in.
  2. Per-user opt-in flow. User enters a new passphrase → derive MK via Argon2id → generate UVK → generate X25519 keypair → wrap UVK with MK → wrap private key with UVK → insert their user_vault row.
  3. Wrap existing WKs. For every WK already in the user's device localStorage, wrap it with the new UVK and write to workspace_members.wrapped_cek. Local paste flow remains as a backup unlock path.
  4. Future workspaces. The create flow now also wraps the new WK with UVK and stores it. Sharing can use the pubkey-mediated handoff (with whatever TOFU / key-transparency story the hierarchy migration brings).
  5. New device flow. Sign in → enter passphrase → unwrap UVK → unwrap all wrapped_cek values → no per-workspace paste needed. Falls back to per-workspace paste for workspaces where the user opted out (or for workspaces predating their hierarchy adoption).

What's a one-way door: almost nothing. The encryption format is preserved (the AAD includes schema_version for future format revs); the WK is structurally identical to the hierarchy's CEK; collaborators who already received WKs out-of-band continue to hold them — that knowledge persists, but it's the same threat shape as any shared key.

Design seams to preserve in v1 for cheap upgrade:

  • schema_version in the AAD (already in §6).
  • The sync boundary's getCek(workspaceId) callback signature (§9). Simple model implements it as an localStorage lookup; hierarchy implements it as an in-memory unwrapped-CEK-map read. Same call site, two implementations.
  • No "user has a passphrase" assumption baked anywhere upstream of the sync boundary. The simple model has no user-level secrets; the hierarchy adds them. Keep the upstream layer ignorant.
  • Don't repurpose workspace_members columns for crypto things. The wrapped_cek column will land cleanly when needed.

14. Open questions

  1. WK rotation. If a workspace key is compromised (a collaborator exfiltrated it, or it leaked via a backup), the owner needs to rotate. Mechanically: client generates a new WK, decrypts every block with old WK and re-encrypts with new WK, also regenerates workspaces.wk_canary under the new WK (otherwise §8.2 validation would fail for anyone unlocking after rotation — the canary would still be encrypted under the old WK), uploads the lot; distributes the new WK to current collaborators out-of-band. The "re-encrypt all blocks" cost is real for large workspaces; worth thinking through the UX (progress bar, atomic rotation token to detect mid-rotation states). Note that wk_canary immutability needs a one-shot exception for the rotation path — either a dedicated rotate_workspace_wk RPC that updates the canary in the same transaction as the rotation token, or a temporary suspension of the immutability trigger inside the RPC. Cleanest is the dedicated RPC.
  2. Layout B vs Layout C for local plaintext. §9.2 lays out both. The choice hinges on (a) a small PoC verifying that PowerSync's pipeline correctly awaits async wa-sqlite UDFs through every code path (if not, C is off the table), and (b) which set of risks is easier to live with — B's universal-observer two-writes-per-download cost concentrated at one named seam, versus C's encryption-awareness scattered across 7+ raw-SQL sites and 4 backfill statements (and counting — the survey list has grown each review round). Default lean now toward B, reversed from the earlier draft: B's complexity is contained at the sync boundary while C's is structural and ongoing, with future plugins one author away from breaking E2EE silently. Re-evaluate if the PoC for C is particularly clean. See §9.2's working decision for the longer argument.
  3. Backfill behavior on first WK paste. The first time a device gets a WK for a workspace, the local plaintext-derived indexes need populating. Under Layout B this means draining the staging table for that workspace through the observer. Under Layout C this means triggering a re-evaluation of the maintenance triggers (e.g. UPDATE blocks SET id = id WHERE workspace_id = ? as a no-op-write trick, or a dedicated rebuild SQL). Either way, the backfill is N-block work; UX needs a progress indicator for large workspaces.
  4. UDF decrypt-cache strategy under Layout C. The UDF can't be marked SQLITE_DETERMINISTIC (WK availability is external state), so SQLite won't memoize calls and the same encrypted value gets decrypted multiple times per row when multiple triggers touch it. A per-row JS-side decrypt cache keyed by (workspace_id, ciphertext) claws this back at the cost of a Map in the UDF closure. Probably not load-bearing for v1; worth profiling once the PoC runs.
  5. Are settings or user preferences in scope? Survey didn't surface a settings table. If preferences are stored server-side and contain user content, they may also need encryption — or be explicitly carved out as metadata.

15. Tradeoffs we're accepting

  • Metadata leak. Block tree shape (parent_id), order (order_key), edit timing, authorship, deletion flag, and per-column "which column changed" are all visible to the server. Acceptable for the stated threat model.
  • No server-side features for E2EE workspaces, ever. Search, AI, summaries, server-rendered share links. This is a real product cost; the opt-in toggle is the user surfacing it explicitly.
  • Lost WK = lost data. No recovery in v1. The UI surfaces this loudly at create time; users who need recovery can store the WK in multiple places themselves.
  • Sharing requires an out-of-band channel. This is a deliberate non-goal, not a defect — an in-app share button would re-introduce the MITM concern. Users pick Signal, password-manager share, paper, whatever.
  • Pasting WK on each new device. One extra step per workspace per device. Tolerable for small N; if N grows past ~5 per user, that's the signal to add the hierarchy (§13).
  • Ciphertext-diff history for E2EE workspaces.
  • Local index side-tables contain plaintext-derived data under either Layout B or C. block_aliases.alias, FTS trigrams, etc. are plaintext on local disk even though blocks.* may be ciphertext (Layout C) or PowerSync's staging is (Layout B). Irrelevant for threat model A (server can't read content); relevant only if the threat model is later broadened to local-disk attackers, in which case OS disk encryption is the right answer.
  • Client-trust required. A backdoored build can exfiltrate keys. Subresource integrity, reproducible builds, and audit are mitigations, not solutions.

This is a sketch, not a commitment. The architectural prep work in §11 (sync boundary, drift guard, JSONB) is the recommended starting point regardless of whether we proceed with §5–§9 on a specific timeline.
