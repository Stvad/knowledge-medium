Media attachments

Design for image (and, by extension, arbitrary file) attachments in a system that now has per-workspace opt-in end-to-end encryption. 2026-06-21.

TL;DR. An attachment is a block carrying the registered media type (in the types array, via typesProp), embedded into prose with the existing !((id)) mechanism — no new reference syntax, and metadata rides the existing encrypted block-content columns for free. The bytes live in an object store (Supabase Storage v1) behind a BlobStore interface, content-addressed by a per-workspace keyed hash so uploads are idempotent and object dedup is an invariant (§10). The reshaping constraint is E2EE: for an encrypted workspace the server holds only ciphertext and never the key, so bytes are sealed client-side with the workspace key and access becomes entirely client-side. We do not write two implementations — there is one byte transform parameterized by mode, where plaintext is the identity case, exactly as src/sync/transform.ts already does for text. Decryption happens in-thread (in the app / its data worker), keeping the keys, session, mode pin, and the already-decrypted block all in one context; the renderer resolves both modes through one fetch-with-session → object-URL path — the bucket is private/RLS-gated (§10), so a bare <img src> can't authenticate — with E2EE adding the decrypt step that plaintext skips (both still fetch with the session and both still produce an object URL, §7.3 / §8). A service-worker variant that would make <img src="${BASE_URL}asset/…"> uniform across modes was evaluated and rejected — its one benefit doesn't earn the app↔SW coordination tax (§7 / §7.3 / §18).

1. What we're adding, and scope

The app cannot currently insert images. We want paste / drop / file-pick of an image to render inline, work offline, and sync across a user's devices — and we want the same machinery to extend to PDFs, audio, video, and generic files later without a content migration.

In scope (v1): images, via paste / drop / file picker; inline render with a lightbox; offline-first local caching; cross-device sync; correctness under E2EE workspaces.

Designed-for but not built (vNext): non-image file types (the model is type-dispatched, so a new mime is a new renderer branch, not a schema change); thumbnails / transforms; resumable uploads for large media; public sharing.

Explicitly out of scope: server-side media features for E2EE workspaces (OCR, thumbnails, transcode) — the server has no key; this is the existing E2EE tradeoff (e2ee-design.html §15), inherited, not new. E2EE subtree sharing of media (needs the deferred key hierarchy — see §12).

3. The core model: an attachment is a block

An attachment is not a row in a side table. It is a blocks row carrying the media type and binary-pointer metadata in its properties:

// asset block (illustrative). The type lives in the `types` ARRAY (typesProp),
// NOT a `properties.type` scalar — that's what the type registry, type queries,
// and the property panel read. Set via block.addType('media') / typesProp.
properties: {
  types:       ['media'],       // typesProp — registered media type id
  mime:        'image/png',
  size:        184213,          // plaintext byte length
  hash:        'sha256:…',      // of plaintext bytes; client-side dedup key
  filename:    'screenshot.png',
  width:       1280,            // optional, for layout reservation
  height:      720
  // NOTE: no upload `status` here — that's per-device, tracked in the local
  // upload queue (§9), NOT in synced (encrypted) properties_json. See §8.
}
content: 'image: screenshot.png'   // human-readable summary

The media type is registered as a TypeContribution — defineBlockType({id: 'media', label, properties}) contributed via typesFacet.of(mediaType, {source: 'attachments'}) (the pattern daily-notes / geo / block-tagging use), so type-filtered queries, the property panel, and renderer discovery all see media blocks. This is not the same as video-player's setup, which contributes property schemas + a renderer but no type — and a bare property schema is insufficient: the capture path's repo.addTypeInTx(tx, id, 'media', …) looks the id up in the type registry (types.get(typeId)) and throws unless a TypeContribution was registered through typesFacet.

Why a block, not an attachments table:

  • Inspectable / user-accessible — the project's standing principle. An attachment shows up in the outline, the properties panel, queries, and the agent runtime like any other block.
  • Embeds, backlinks, annotations come for free. Inline placement uses the existing block embed !((id)) (src/components/references/BlockEmbed.tsx); captions / alt-text / OCR / transcripts are child blocks; "what references this image?" is the existing backlink machinery.
  • Type-dispatched rendering already exists. A MediaBlockRenderer registers via blockRenderersFacet with a canRender gate on block.hasType('media') — the same registration pattern as VideoPlayerRenderer. New file types are new mime branches, never a schema change.
  • Dedup is natural and workspace-wide: paste the same screenshot ten times anywhere in a workspace and you get ten !((id)) embeds of one asset block — one byte copy, one upload (see §11).

4. The constraint that reshapes everything: E2EE

For an E2EE workspace the server holds only ciphertext and never the key. The workspace key (WK) lives as a non-extractable WebCrypto CryptoKey in IndexedDB (km-e2ee-keys, keyed by (user_id, workspace_id) — src/sync/keys/keyStore.ts). The server-side trigger blocks_require_ciphertext_for_e2ee actively rejects any plaintext write to an encrypted workspace's content columns.

The E2EE design doc encrypts exactly three text columns and says nothing about binary data — confirmed by inspection. So attachments are an unsolved problem that must be solved consistently with E2EE, not bolted on beside it. Two pillars of any naïve "URL in markdown" approach are dead on arrival:

A signed Storage URL handed straight to <img src> — dead for E2EE. The object is ciphertext, so the browser can't render it directly; the bytes must be fetched and decrypted client-side with the WK first, then handed to <img> as an object URL (the chosen in-thread path; resolution detailed in §7). Note what this does not kill: signed URLs as a fetch mechanism are fine — what's dead is using one as the render URL.
Public-bucket / public-share-URL — incompatible with E2EE. You cannot serve a publicly-readable image out of an encrypted workspace via RLS + signed URLs. RLS gates access to ciphertext; it cannot grant decryption. (This is just e2ee §15 #2 — "no server-side features for E2EE workspaces" — applied to bytes.) The escape hatch survives in a different form; see §13.

What the encryption model gives us in return is also worth stating: because metadata rides the encrypted properties_json column, an attachment's mime, filename, and hash are confidential automatically on an E2EE workspace. The block-row routing identifiers (id, workspace_id) are in clear — exactly the columns the sync transform already leaves in clear.

But the Storage object leaks more than the block row does — call it out, don't fold it into the inherited block-metadata leak. The encrypted blob's byte length (≈ plaintext size + 28 B) is visible to the server via the stored object size and the Content-Length of an RLS-gated GET, and the object's existence and per-workspace count are visible. Exact file size is a much stronger fingerprint than note-text length — it can confirm a specific known file (a leaked document, a known image) by size, and combined with the asset_block_id in the path + timing, correlate which block embeds which known-size asset. This is a new leak outside the enumerated e2ee §15 set (tree shape / order / timing / authorship). Document it as an accepted tradeoff (§17), or pad ciphertext to size buckets if the threat model cares.

5. One byte transform; plaintext is the identity case

E2EE is opt-in, so most workspaces are plaintext. The temptation is two code paths. The codebase already shows the better answer: src/sync/transform.ts is one function parameterized by mode, where mode === 'none' is the identity transform.

// the text precedent, today
encodeForWire(row, 'none', getCek)  // → row unchanged
encodeForWire(row, 'e2ee', getCek)  // → seal content/properties_json/references_json

We mirror it for bytes — a new module in the same src/sync/ seam:

encodeBytes(bytes: Uint8Array, mode: SyncMode, getCek: GetCek, ref: AssetByteRef): Promise<Uint8Array>
decodeBytes(blob:  Uint8Array, mode: SyncMode, getCek: GetCek, ref: AssetByteRef): Promise<Uint8Array>
//   ref = { contentHash, workspaceId } — binds the bytes' AAD (§5.1); decode
//         authenticates the envelope only, the resolver verifies the hash separately
//   mode === 'none'  → identity (return input)
//   mode === 'e2ee'  → AES-256-GCM seal/open with the WK
Decision. There is one attachment implementation, not two. Everything expensive — the asset-as-block model, the BlobStore, the cache, the upload queue, dedup, the renderer — is mode-agnostic and shared. Only the ~15-line encode/decode seam is mode-aware, and plaintext is its degenerate (identity) branch, exactly like encodeForWire.

5.1 Reusing the crypto primitives

The WK CryptoKey is AES-256-GCM and works for crypto.subtle.encrypt over raw bytes, not just the string seal/open in src/sync/crypto/aead.ts. So byte encryption reuses the same getCek(workspaceId) seam the sync transform uses, with two deltas:

  • AAD binding extends the existing pattern. Where content columns bind contentAad(id, workspace_id, column), asset bytes bind (content_hash, workspace_id, "asset-bytes", schema_version) — the content hash, not a block id, because the object is content-addressed and shared by every block carrying those bytes (§10); binding one block id would stop a co-referencing block from opening it. workspace_id + the content hash give cross-workspace / cross-content domain separation, but the real defense against an untrusted server returning arbitrary or stale bytes is the read-side hash check (warn box below): the AAD's content_hash is the same sha256(plaintext) the read path verifies against block.properties.hash, so it is redundant-but-cheap defense-in-depth (it fails at GCM-open, a hair earlier, and survives a hypothetical read-check regression), not the load-bearing check. (schema_version stays in the AAD to cryptographically bind the format version — the encb:v1: magic is unauthenticated framing, so the AAD term is what would make a future cross-version swap fail the tag; same reason the text path binds it.)
  • A binary envelope, not the base64url text envelope. The enc:v1: envelope is a string (base64url), and the text seal/open seam is typed string→string over TextEncoder/TextDecoder — which corrupts raw bytes (invalid UTF-8 is replaced). So bytes need their own framing regardless of the ≈33% base64 bloat: a sibling encb:v1: binary framing (magic ‖ nonce ‖ ciphertext‖tag as raw bytes) sharing the nonce/tag layout. Small new module next to aead.ts.
The AAD does not bind freshness — verify the content hash on read. Every AAD term (the content_hash, workspace_id, the literal, schema version) and the content-addressed storage path are fixed by the bytes themselves. AAD stops cross-content / cross-workspace swaps, but it does not stop an untrusted server (in scope per e2ee §2 — it may serve whichever bytes it likes) from returning arbitrary or stale bytes for a requested content path that happen to decrypt cleanly and pass the GCM tag. The bytes lane has no updated_at/monotonicity channel the text path gets from PowerSync. The fix is already in hand: the asset block stores hash = sha256(plaintext) in its (encrypted) properties_json; the read path must verify the decrypted bytes against that hash before caching/serving, and reject on mismatch. In the chosen in-thread path (§7.3) the resolver already holds the decrypted block, so the expected hash is just block.properties.hash — no extra plumbing. (The rejected SW path couldn't read the encrypted block state, so it would have needed an SW-readable metadata mirror (rejected-sw doc) — not a request param: an <img src> can't carry custom headers, and a ?h= query param would leak the plaintext SHA-256.) Either way it's a positive check that can only fail closed, never downgrade.
Content-addressing makes the write lane intrinsically immutable — no monotonicity machinery. Because the path is derived from the bytes (<ws>/HMAC(K_id, sha256) for E2EE, §10), different content lands at a different path and identical content at the same one — so the "two uploads race at one path / a stale upload overwrites a newer object" hazard simply does not exist, and there is no "older version" for the server to replay. An edit is new bytes → a new content path → a new asset block; the old object becomes unreferenced and is GC'd (§16). First-write-wins at a path is the entire rule (§10.1); the idempotency-token / write-once-rule / duplicate-object-GC machinery a block-id path would have needed is simply gone (the content-addressed path gets it for free — §10). (Dedup reuses an existing asset block by hash — a read, not a rewrite.)

The encb:v1: magic is a format tag, not an authorization signal. Whether to decrypt is decided by the workspace's mode pin (§7), exactly as the text path decides via the pin rather than the enc:v1: prefix (envelope.ts / e2ee §6 rule 2). An E2EE-pinned workspace requires the magic and treats its absence as tampering; only a plaintext-pinned workspace passes bytes through. A workspace with no pin (never validated, or evicted) is not plaintext — it is the resolver's defer/encryption-uncertain state (src/sync/keys/resolver.ts), so the read path fails closed (placeholder), never passthrough — otherwise an encrypted workspace with a missing local pin would serve attacker-controlled plaintext. The decode decision is therefore three-valued — decrypt (E2EE + WK) / passthrough (plaintext-pinned) / fail-closed (E2EE without WK, or no/uncertain pin) — and the in-thread resolver must drive it from that three-valued materializability (getMaterializability), not the two-valued SyncMode/getMode the upload direction uses (§7.3).

6. The byte path, end to end

Bytes do not ride PowerSync. The asset block (metadata) syncs through the normal Layout B path (encrypted content columns, observer-reconciled). The bytes take a parallel path to the object store. The two arrive independently on other devices.

7. The decryption boundary (chosen: in-thread)

Decision: decrypt in-thread (§7.3), not in the service worker. The byte cache is OPFS + object URLs; the renderer carries a small mode branch. The service-worker variant — which would make <img src> uniform across modes at the cost of an app→SW token handshake, an SW-readable mode-pin + per-asset hash mirror, per-clientId sessions, and controller gating — was analyzed and rejected; the full analysis is preserved as the decision record in rejected-sw-decryption-boundary.html (summarized in §7.1). §7.2 (destroying local data) and §8–§10 are boundary-agnostic; §7.3 is the chosen path and the full tally (see also §18).

7.1 The rejected alternative: a service-worker decryption boundary

The other boundary considered was decrypting in a service worker, so that <img src="${BASE_URL}asset/…"> would be uniform across plaintext and E2EE. It is feasible (the SW scope has crypto.subtle, IndexedDB, the Cache API) but was rejected: it concentrates ~5 boundary-artifact coordination mechanisms — an app→SW token handshake, an SW-readable mode-pin + per-asset hash mirror (the SW can't read the DB), per-clientId sessions + per-user cache keying (one SW serves every tab/account), and a controller-readiness / route-ordering / clientId-less-navigation lifecycle — to buy one benefit that <img> doesn't need, since it takes an object URL. The in-thread path (§7.3) pays none of that. The full coordination-tax analysis — the four request mechanics, the two coordination problems, the SW-readable-facts requirement, and the Cache-API authorization boxes — is preserved as the decision record in rejected-sw-decryption-boundary.html. The chosen path follows in §7.3.

7.2 Destroying local data is coarse and system-level — not an in-app wipe

The byte stores add no selective wipe machinery. Destroying local data is a coarse, all-or-nothing action delegated to the platform: the browser's own "clear site data" wipes the whole origin — SQLite DB, E2EE keys, and every byte store (OPFS, IndexedDB, Cache API) — in one shot. This shipped on master (2026-06-23): the selective per-user in-app Lock & Wipe flow (a PENDING_WIPE marker, a boot-time consumePendingWipe purge, an app→SW purge handshake) was removed, and lock_and_wipe_local_data is now a thin guidance dialog (WipeLocalDataDialog) that hands off to the platform control and warns on unsynced changes. The team deliberately did not hand-roll a coarse destroyAllLocalData() either — the platform wipe runs outside the page/SW context, so it sidesteps the cross-tab OPFS race, the surviving service worker, and the pre-render boot gate a hand-rolled wipe would each have to solve (and a SW can't even synthesize a Clear-Site-Data header — Chromium ignores it; a programmatic trigger is deferred until the app leaves GitHub Pages, issue #245). Consequences for this design:
  • No wipe participants. The byte store and upload queue are ordinary local storage; the coarse clear removes them with the DB and keys — nothing selective to drain, mark, or purge, and no app→SW wipe coordination to get right.
  • No "plaintext survives a lock" hazard. Decrypted bytes in OPFS are destroyed by the same clear that removes everything else, so storing them locally is consistent with the accepted posture (e2ee §15 #7) without a per-store wipe rule to enforce.
  • Re-securing a device is the coarse clear — it logs out every account / workspace in the profile, not one. Accepted: a paranoid user would clear site data anyway, and re-affirming workspace status (re-paste the WK) afterward is fine.
The runtime cache-hit authorization checks (§7.3, §8 — account switch, membership revoke, WK absence) are independent of destruction and stay — they gate serving stale bytes, which no amount of wiping addresses.

7.3 The chosen approach: in-thread decryption

In-thread (app, or its dedicated data worker) decryption: the renderer fetches the ciphertext, decrypts via getCek directly (zero duplication), caches plaintext bytes, and hands <img> an object URL. Because it runs in the context that already owns the session, the keys, the mode pin, and the decrypted block, it pays none of the SW's coordination tax. What it avoids:

  • No mode-pin IDB store / no per-asset hash mirror. The app reads the mode pin synchronously (WK-presence then resolves on the already-async render path, via getMaterializability) and already holds the decrypted block (so the hash is just block.properties.hash) — the §7.1 mirror machinery exists only because the SW can't reach app state.
  • Plaintext bytes in OPFS (not SQLite, not Cache API). The app-thread path stores decrypted bytes as plain OPFS files (the single byte store, §8; assets/<user>/<ws>/<content-key> — local assets tree, distinct from the remote attachments bucket) — the store the bytes always wanted (§8/§17 keep large blobs out of SQLite; an earlier draft wrongly floated DB-blob storage purely for a "free wipe," which both bloats the hot query DB and contradicts that principle). Destruction is not a differentiator either way now — the coarse clear (§7.2) removes the OPFS tree and the Cache API alike. The real app-thread win here is simplicity of resolution: the app owns the OPFS handle and the session, so fetch→decrypt→write-OPFS→object-URL is one local flow with no SW cache/token/mirror lifecycle.
  • No token handshake / no per-clientId session — the app already has the Supabase session.

What it costs: a renderer that resolves both modes to an object URL — the private bucket (§10) rules out a bare authenticated <img src>, so both fetch with the app session; E2EE additionally decrypts, plaintext serves as-is (fetching on a local miss) — plus manual object-URL lifecycle (createObjectURL/revokeObjectURL) — two local, well-understood concerns. Note OPFS is not URL-addressable, so <img> needs an object URL either way; a path-like stable URL is the only thing the SW route would have added.

Three properties the in-thread path must carry explicitly. They held in the SW analysis above, but the chosen path is where they ship, so they are restated here rather than left in the rejected framing:

  • Fail-closed decode, not two-way. When decoding a freshly-fetched object before it enters the store, the resolver drives the decision off the three-valued getMaterializability — its values decrypt / copy / defer (decrypt with the WK / pass plaintext through / fail-closed placeholder) — never the two-valued getMode — that downgrade (an evicted-pin E2EE workspace served attacker plaintext) is the §5.1 / e2ee §6 rule 2 hazard, argued there. In-thread specifics: the pin read is synchronous and WK-presence resolves on the already-async render path.
  • Hash mismatch is hard-fail. On a mismatch against block.properties.hash (§5.1) the resolver discards the bytes — never writes them to the OPFS byte store — and renders the broken-asset placeholder; unverified bytes are never served or stored.
  • Stored bytes are not an independent authority — revocation rides the block claw-back. The OPFS byte store is namespaced by user_id (account isolation); on workspace leave/revoke PowerSync drops the workspace's block rows from the local DB, and the out-of-band OPFS bytes need their own claw-back to follow. That claw-back is DEFERRED to a future reference-GC cleanup job, not an eager per-membership-change deleter (§16). The job is the natural home: it deletes byte-store entries whose content hash is no longer referenced by any local media block, after a grace period — which subsumes revoke/leave (the workspace's blocks are clawed back → its bytes go unreferenced), media replacement / deletion (the old object becomes unreferenced), and quota eviction, under one reference-counted sweep run when the lanes are quiescent. (History: an eager reactive purger — a workspace_members-removal subscriber calling byteStore.purgeWorkspace — was prototyped on the down-lane PR and removed. It deleted user media on a transient membership signal that can glitch, and to be correct it needed retry-on-failure plus cancel/epoch coordination with in-flight down-lane puts — too much machinery, and too eager a deletion, for what this very bullet calls an accepted exposure. purgeWorkspace remains in byteStore as the primitive the cleanup job will call.) So media revocation is handled like note text with a lag: a revoked member's local blocks are dropped and the bytes are reclaimed on the next GC sweep, and an already-synced member who keeps a copy in the meantime is the same accepted exposure for bytes as for text (§8) — RLS still blocks any new fetch. WK-absence gates the decode of new downloads (fail-closed, above), not the serving of already-stored, already-verified bytes — lock doesn't scrub the store, exactly as it doesn't scrub the decrypted blocks table (§8).
Why in-thread wins the tally. The SW would concentrate ~5 coordination mechanisms (tallied in §7.1 / the sibling doc) to buy one benefit — a uniform mode-agnostic <img src> — that <img> doesn't need, since it takes an object URL; in-thread pays two small local concerns and avoids all of it. The one portable lesson: a service worker earns its tax only when the consumer needs a real URL, not a blob. <img> takes a blob → in-thread. A consumer that consumes URLs — e.g. the JS module loader for user plugins (import() of a /module/<blockId>) — is where a SW-served URL genuinely pays off, and even there the clean shape keeps the privileged work in-thread and lets the SW serve a pre-populated cache (separate investigation).

8. Local replication: one decrypted byte store

PowerSync already gives every device a full local replica of the block rows, so "it's all local" already holds for metadata. Media bytes are the one thing that breaks it — they live out-of-band in Storage, not in the synced DB — so for genuine offline operation they get their own replication, with the same target steady state as the DB: every asset for a workspace you use is on disk. We reach it by eager intent, lazy timing — a background, idle-priority, proximity-first drip that converges on full replication, not a boot-time thundering herd.

There is one local byte store, holding the decrypted bytes (raw bytes for a plaintext workspace; bytes decrypted with the WK for E2EE), keyed by the content-addressed Storage path (<ws>/<content-key>, §10). It is both the offline replica and the render source — there is no second tier. Decryption happens once, when bytes enter the store (a download decrypts + hash-verifies before writing, §9; a paste already holds plaintext), not on every render — so the render path is a plain local read → object URL, and plaintext and E2EE workspaces are identical locally, differing only in the at-rest format on the server.

Why decrypted-at-rest, not a ciphertext replica. An earlier draft kept two tiers — a durable ciphertext replica (byte-identical to the server object) plus a throwaway decrypted display cache. The only thing the ciphertext tier bought over storing decrypted bytes was replicate-while-locked (a device pre-fetching encrypted bytes before its workspace is unlocked). We don't want that, so it's gone — and with it the duplication. Bytes are now stored exactly like the decrypted blocks table already is: plaintext-derived data on local OPFS, the posture e2ee §15 #7 already accepts (and an attacker with OPFS read already holds all your note text, so bytes-as-ciphertext bought no real confidentiality). Integrity at rest is the sha256 hash-verify (§5.1), exactly as the wire path checks it; re-upload (after GC, or from the up-lane) re-encrypts on the fly — fresh nonce, same content path, first-write-wins idempotent (§10.1). Net: roughly half the E2EE on-disk footprint, no decrypt-on-render, and one store instead of two.
  • Backing store: OPFS (following the in-thread decryption choice, §7.3) — read the file → object URL → <img>. User-scoping (assets/<user_id>) is the access-isolation boundary (§7); workspace-scoping makes leave / revoke purge only the affected bytes. Because the object is content-addressed (§10) it is immutable by construction — an edit is new content at a new path, so a stored entry's bytes never change; a freshly-downloaded byte is accepted only if it matches the block's expected hash (read directly from the decrypted block in-thread, §5.1); never serve a content key's bytes for a different hash.
  • Demand beats prefetch. The render path and the background replicator call the same fetch primitive (fetch ciphertext → decrypt → verify hash → store decrypted bytes) over one in-flight map, so a demand fetch and a queued backlog item for the same asset coalesce into a single download — opening a queued image satisfies its backlog entry instead of racing it. Two priority lanes ride that primitive: a high-priority demand lane (the user is looking at it now) that jumps ahead, and a throttled, idle, low-concurrency backlog lane (the drip) deliberately slow enough never to saturate the link or starve a demand fetch. So opening not-yet-replicated media triggers an immediate high-priority fetch (a brief spinner if online); the only transient, self-resolving hard failure on the happy path is offline AND not-yet-replicated. The persistent (non-transient) failures — origin gone before upload (§9, the one truly unrecoverable case), a poisoned content path (§17, broken until the path frees), a hash mismatch (§5.1) — are separate, and each is called out where it arises.
  • Eager replication is budgeted and scoped, or it backfires. Metadata is cheap to fully replicate; media can be gigabytes, and unbounded eager replication would blow the origin quota and trigger the very eviction it is meant to prevent. So the down-lane (§9) is bounded: (a) scoped to active / opened workspaces (don't pull cold workspaces' bytes — the same reasoning as the "don't touch unopened workspaces" sync-flood lesson); (b) budget-capped — replicate eagerly up to a budget, beyond which the long tail stays lazy-re-fetchable rather than quota-thrashing; (c) polite — idle / low priority, viewport-proximity first, and metered / mobile aware (wifi-only or a user toggle); (d) single-owner across tabs — one tab is elected via a Web Lock (navigator.locks.request) to run the background replicator, so N open tabs of the same workspace don't multiply the egress these controls bound (the lock auto-hands-off when that tab closes; no SharedWorker needed). The viewport-driven demand lane stays per-tab — each tab fetches what its own user is looking at. Reads are direct RLS-gated Storage GETs (writes are direct too — only uploads are writer-gated rather than member-gated, §10.1), so the down-lane is plain egress.
  • Eviction is for quota pressure, not a fixed cap. The store is navigator.storage.persist()-backed (the lever that exempts it from best-effort eviction — spun out as its own investigation) and preserved by default for full offline operation. The only reason to evict is genuine origin-quota / disk pressure, and under it the priority is DB > replicated bytes — the DB always wins a real contest. So the bound is a quota budget (a share of available quota, or a user cap), not an arbitrary fixed size: a fixed ≈500 MB cap would directly contradict the "every asset on disk" goal for any multi-GB workspace. An evicted entry is graceful degradation, never data loss — next view re-fetches + re-decrypts from the server (online). Un-uploaded bytes are exempt — they are the only copy anywhere (§9). Eviction and the exemption are evaluated per content key, not per block: one object can back several blocks (§11), so deleting one block must not evict the shared entry while another local block still carries that hash — local eviction rides the same content-refcount as object GC (§16).
  • Destroying local data: the byte store is ordinary local storage, removed wholesale by the coarse clear-all (§7.2) — no per-store wipe participation. Authorization is separate: on per-workspace revoke/leave, purge that workspace's prefix (a serving-access control, not destruction), fired off the same membership change that makes PowerSync drop the workspace's block rows from the local DB — concretely, a new subscriber the feature wires to the user's own workspace_members row leaving the local DB (there is no existing membership-change callback to reuse). This is a deliberate one-shot purge, not a per-render TTL revalidation: media is clawed back exactly like note text. The block rows fall out of the local DB on revoke (offline, on the next sync after reconnect); the bytes live out-of-band in OPFS so they don't ride that row claw-back and need their own prefix-purge fired off the same change (offline, it fires on reconnect — the same window the rows themselves have). Ordering between the row-drop and the purge is immaterial: with the block gone there is no hash and nothing renders the bytes anyway; and in the reverse window (purge lands first, rows still present) a render misses OPFS and the demand-fetch fails closed against the now-revoked Storage RLS (reads re-evaluate membership per GET, §10) — so neither order ever serves stale bytes. (The purge is race-safe, not race-free: a demand-fetch that finishes its OPFS write just after the purge leaves an inert orphan file — no surviving block renders it, so it is never served — reclaimed by the next re-join purge or the coarse clear.) We deliberately do not re-validate every render against a freshness TTL: that would gate the image on a per-render membership recheck that the far-more-sensitive note text never gets, for data the ex-member already downloaded — a revoked-but-not-yet-synced member can read all the surrounding blocks too, so a media-only recheck is not a coherent boundary, just render-path cost. (Lock alone does not scrub the store, exactly as it doesn't scrub the decrypted blocks table — that's the coarse wipe's job; membership revoke, not WK-absence, is the access gate.)
  • Optimistic local-first write: on paste, the plaintext bytes are written straight to the byte store (durable; the §9 up-lane drains from there). The image renders immediately from local bytes; the upload drains in the background. Whether the bytes have uploaded yet is a per-device fact, read from the local up-lane — see the next point. Offline it simply stays pending and still renders on this device.
Upload status is per-device; do not put it in synced block properties. An earlier draft stored status: 'uploading' | 'synced' on the asset block. But block properties serialize to the single synced (and, for E2EE, encrypted) properties_json column — so a flip to 'synced' would replicate to every device and clobber a peer for which the bytes are not local. "Have the bytes uploaded?" and "are the bytes cached here?" are both per-device questions answered by the local upload queue (§9) / local byte store, keyed by asset_block_id — never by a synced flag. The synced block's mere existence is the only cross-device signal needed.
Un-uploaded bytes are the only copy — keep them durable and exempt from eviction. If a paste happens offline, or the intake ceiling (below) is hit before the up-lane drains, evicting a not-yet-uploaded entry would leave a synced asset block stuck pending forever with no bytes to send. So un-uploaded bytes are exempt from eviction, and the durable write is on the capture path itself. Two corollaries: (1) the durable write at capture must actually be durable — if it fails, or persist() is denied so the write lands in best-effort storage, capture fails with a clear error ("can't attach — local storage is full or not durable") rather than committing metadata over evictable bytes; the §11 staged-before-tx ordering catches a thrown stage, not a silently non-durable one. (2) Because un-uploaded bytes can't be evicted, there are really two distinct bounds, not one number: a retention budget over replicated (uploaded, evictable) bytes, and a separate pending-intake ceiling over un-uploaded (exempt) bytes. A new paste first evicts replicated bytes under the retention budget (DB > replicated bytes, §8) to make room — so a workspace already full of replicated bytes never blocks a new capture. Only when the un-evictable pending bytes alone would breach the intake ceiling does paste apply backpressure ("uploads pending — reconnect to drain them, or discard a pending attachment to reclaim its space") rather than accepting bytes you might have to evict. Ordinary eviction can't relieve this state (un-uploaded bytes are exempt), so the only offline release valve is the user deliberately abandoning a pending attachment — which requires the per-device pending up-lane records (§9) to be surfaced as individually discardable items, a required capture-surface affordance (§11), not just a "bytes pending here" indicator. A discard is content-refcount-gated like every other deleter (§8 eviction / §9 reap / §16 GC) — the record as well as the bytes: it removes both only when no live-or-soft-deleted block still carries the hash (the last reference). While another live block (e.g. a block copy) still carries it, discard reclaims nothing — bytes and the upload obligation both stay — so it can't strip a co-referencing block's bytes or strand its route to the server.

9. The byte-replication subsystem (up-lane + down-lane)

Block metadata rides PowerSync's ps_crud upload queue. Bytes go through the direct, RLS-gated upload path (§10.1), which has no such queue, so they need their own durable one. The queue is a small status record over the §8 byte store, not a second copy of the bytes: an IndexedDB record of { user_id, asset_block_id, workspace_id, status }. The source bytes live once in the byte store (decrypted, §8); while a pending record exists for them they are exempt from eviction, so a pending upload can never lose its source to eviction or an offline window (the §8 hazard). The encb:v1: encoding is done at drain — read the plaintext from the byte store → encode (identity for plaintext workspaces) → PUT — so the queue keeps no ciphertext copy. The user_id field is load-bearing: it namespaces the record (and every cancel / drain key), because the store is shared across accounts in the profile but draining is per-user — each upload rides the active user's session/token, so without the namespace a drain could retry account A's entry under B's session, uploading A's bytes with B's credential or against the wrong workspace membership. The upload target is the content-addressed path (<ws>/HMAC(K_id, hash), §10) computed from the block's hash — so a retry, even after a reload, re-PUTs to the same path and is first-write-wins idempotent (§10.1); no random idempotency token is needed (that was the block-id path's workaround). The record carries a small persisted status closing the metadata-commit / byte-stage ordering gap (§11). The persisted values are staged / pending / failed: staged (durable but not drainable) → pending (drainable) → then either it clears — the record is deleted on a confirmed upload (the bytes stay in the byte store, §8, now eviction-eligible like any replicated bytes; "eviction-eligible" is simply the absence of an up-lane record, not a retained cleared value) — or it moves to failed on a permanent rejection (§10.1: an auth/size/path rejection the upload can't satisfy — a non-writer, an over-limit body, a malformed path — or a content path an active poisoner keeps re-occupying so writer-delete + re-upload can't terminate). A failed record stops the tight correction loop (so a poisoned or rejected upload never burns egress in a hot retry) but keeps its eviction exemption — its bytes are un-uploaded, i.e. still the only copy and the self-heal source (the §9 server-side backstop / §16), so evicting them would be irreversible data loss. The only byte release is the user explicitly discarding the item from the §11 pending-attachments affordance — and that discard is itself content-refcount-gated (§8/§9/§16), the record as well as the bytes: it removes the record and frees the bytes only when no live-or-soft-deleted block still carries the hash (the last reference). If another live block still references the content, there is nothing to reclaim — the bytes are still needed and the upload obligation is still owed — so a discard of a still-shared attachment is a no-op for reclamation (the content stays pending until its last carrier is gone or it uploads), which is also what stops it stranding a co-referencing block's route to the server. In the common capture-dedup path the asset is a single block with a single record (§11), so this gating is moot; the shared case arises only from a block copy / import of a not-yet-uploaded asset. And failed is recoverable, not a dead end — but recovery is bounded and cheap, never the hot loop the state exists to stop: a failed record is re-attempted on app start, on reconnect, by an explicit user retry, and on a slow periodic sweep (a bodiless GET every few hours — so a long-lived online tab still heals once the path frees, rather than waiting on a restart it may never get). Even against a persistent poisoner the cost is one sparse, bodiless probe per such event — never the tight correction loop, and no new lane to walk it: the down-lane fetches bytes that are absent locally, whereas a failed record's bytes are present, so the down-lane would never visit it; this re-attempt sweep is the up-lane's own. Each re-attempt first does a cheap direct RLS-gated GET of the content path (reads are direct, §8 — it streams no body) and branches on what's there: object absent → the path is free, drain normally → clears; present and hash-verifies → another device already materialized it → delete the record, no upload → clears (the bytes are now eviction-eligible like any replicated bytes — note this drops the durability floor for this content from "a guaranteed un-evictable copy" to the same best-effort every replicated byte has, since no device is now obligated to keep an exempt copy; should the server object later be removed — an active poisoner / writer-delete — this degrades to the §9 "synced block outlives its bytes" backstop, a shown-as-unavailable broken image recoverable only while some replica survives, never a silent loss); present but hash-mismatches → still poisoned → stays failed. This covers both causes without conflating them — a poisoned path heals when the probe finds it free or materialized; a shape-rejected body (a stale / buggy client) heals only once the client is fixed, whose post-upgrade re-attempt re-encodes correctly, so until then it stays surfaced-for-discard rather than re-PUTting the same bad bytes. Only permanent rejections land in failed; transient failures stay pending and retry with backoff (below). "Upload in progress" is an in-memory single-flight guard, not a persisted state — a crash mid-PUT leaves the record pending and the single-owner drain (§8) re-PUTs (first-write-wins idempotent, §10.1), so no persisted uploading state is needed. Capture writes the record as staged before the metadata repo.tx, so a committed asset block always has its source bytes already on disk; the post-commit step flips it to pending and arms the drain. The drain acts only on pending+ records, so the "block still exists?" pre-PUT drop applies only after the commit (a post-commit deletion / undo) and never to the legitimate pre-commit window. A flip that never lands (crash or throw between commit and flip) is recovered by a boot-time reconciler that promotes any staged record whose asset block has materialized in blocks (recovering the post-commit flip that never landed) and leaves every other staged record untouched. It deliberately does not reap absent ones. A staged record is non-drainable, so an orphan never uploads on its own; leaving it costs only its bytes, which the §16 content-refcount GC reclaims (no live-or-soft-deleted block carries their hash) — the same single rule that keeps §8 eviction / §16 GC from dropping a shared object out from under a live block. Reaping here would instead demand proving a block never committed — distinguishing a true orphan from a committed-but-unmaterialized block (a locked e2ee workspace withholding it from blocks, the throttled observer still lagging the download, an undecryptable/quarantined row, or another profile account's un-hydrated DB) — and any wrong guess deletes the only un-uploaded copy of a live block's bytes. Trading a near-zero, GC-reclaimable byte leak for "can never reap a live block's bytes" is the right call; the promote reads plain blocks because a miss is non-committal (a later boot retries once it materializes — and it couldn't drain while the workspace is locked anyway), never destructive. A caught tx rollback still deletes its orphan record + bytes inline (it ran in-process and knows the block didn't commit); the reconciler only leaves the residue where a crash lands between the throw and that cleanup, which §16 GC sweeps. A profile account that is never re-opened simply leaves its records inert, costing bounded storage, never loss. Entries drain (direct to Storage, §10.1) when connectivity returns, retrying with backoff; on a confirmed upload the queue entry is deleted, but the bytes remain in the byte store (§8, budget-bounded) — full replication keeps them on disk for offline; they simply lose the un-uploaded eviction-exemption. This is new infra the pre-E2EE proposal under-specified; E2EE doesn't change it, but the design surfaces it clearly. The queue's local status is also the per-device source of truth for the "bytes pending here?" UI (§8) — there is no synced status flag. Cross-device arrival is naturally eventually-consistent: a peer that has the block but not yet the bytes shows the "metadata present, bytes pending" state until the upload from the origin device completes.

The down-lane: background replication. The queue above is the up-lane — local bytes the server doesn't have yet. Its mirror is the down-lane: a background reconciler that walks the media blocks in the local DB for active workspaces and, for any whose bytes are absent from the store, enqueues a fetch → decrypt → hash-verify → store (the §8 budget / scope / politeness rules apply). The two lanes are halves of one byte-replication subsystem over the single §8 byte store, paralleling what PowerSync does for rows: up-lane drains local→server, down-lane reconciles server→local, and the demand lane (§8) lets a render jump the backlog. So the §9 up-lane and the §8 byte store are one store + its per-device upload-status records; the bytes exist once, not a second copy.

Cross-tab execution: the background lanes are single-owner; capture is per-tab. The background work — the down-lane replicator, the up-lane drain (and its periodic re-attempt sweep for failed records, §9), and the boot reconciler — runs under one tab elected by a Web Lock (navigator.locks.request, §8), so multiple open tabs don't multiply egress or redundantly re-PUT (the lock hands off automatically when that tab closes; no SharedWorker required). Capture stays per-tab: any tab can paste, staging its byte record and committing its repo.tx locally. That split is exactly why the up-lane keeps its session/generation stamp — single-owning the reconciler does not remove the reconciler-vs-capture race, because a capture in another tab can be mid-tx (staged record written, block not yet committed) while the owner's reconciler runs; the stamp is what stops it reaping a live capture's record. (A dedicated worker for the lane is a later perf option, not needed for this.)
Undo / delete need no byte-lane compensation — the lane self-heals. §11 puts the asset-block create + the !((id)) insert in one repo.tx (the DB half is atomic). The bytes aren't in that tx and don't need to be: block delete is a restorable soft-delete (§9/§16) and the Storage object is immutable + content-addressed (§10), so undo is a pure DB soft-delete and the byte lane reconciles around it.
  • Undo soft-deletes the block (shared repo.tx) and best-effort cancels the queue entry; it does not touch the Storage object.
  • The drain already re-checks "block exists and is not soft-deleted" before PUT (needed for any post-commit delete, not just undo): so an undo before upload never PUTs, and a PUT racing the undo just confirms idempotently to its immutable path (§10) — a harmless retained object.
  • Redo un-soft-deletes the block (pure DB). If the upload already happened the immutable object is still there — nothing to re-upload; if not, the retained byte-store bytes (never dropped on undo) resume draining — and if the record had reached failed, redo treats it as a fresh trigger to re-attempt (the §9 opportunistic re-attempt), so a restored block never sits in failed without a path forward.
  • The object from an undone-then-not-redone paste is not an orphan — a soft-deleted block still references its hash, so §16's reference-counted sweep retains it like any other soft-deleted block's object (reference-permanent in v1), reclaiming it only when no live-or-soft-deleted block carries the hash.
This is also what keeps it correct under dedup: one content object can back several blocks (§11), so deleting the object on undo of one of them would break the others' live images — the byte lane must never delete an object on undo; the refcounted §16 sweep is the sole deleter.
A synced asset block can outlive its only bytes — the server-side backstop. A device can lose its un-uploaded bytes — a coarse wipe (§7.2), a lost / reinstalled device, or a crash — while the asset block's metadata already synced (PowerSync was up; the byte upload wasn't), leaving the server with a block whose content-addressed object can never materialize, a broken embed on peers that did nothing wrong. There is no reliable client-side fix at that moment (the origin device is gone or offline) — but the handling must not be a destructive server-side reap of the block. Block deletion is a subtree-aware soft-delete (src/data/mutators.ts), so reaping the asset block would cascade-delete the user-authored caption / alt-text / transcript children — turning a missing image into lost text — and elapsed time alone can't distinguish "author offline, still intends to upload" from "author gone forever". So instead — with no new server write-path, because §16 never timed-reaps the block (its mere existence is the cross-device "intended here" signal, §8): (a) a peer that has the block but not the object runs a bounded down-lane retry, then surfaces a settled "image not yet uploaded" state (not an indefinite spinner, and it stops re-hitting Storage with 404s) — but that settled entry is re-attempted on the next down-lane reconcile pass (which re-walks the workspace's media blocks; the peer's bytes are absent locally, so the down-lane legitimately visits it), so a transient 404 (e.g. the §10.1 writer-delete → re-PUT correction window) self-heals rather than sticking; (b) the embed self-heals if the origin ever uploads — the origin's own durable local up-lane record (staged/pending, or a failed record, which keeps its exempt bytes and re-attempts opportunistically once a poisoned path frees, §9) carries the intent and drains on reconnect. Automatic destruction is reserved for orphan objects (bytes with no block — safe to delete); an orphan block is never auto-deleted (its children are real content), so the only unrecoverable case — origin gone before upload — is honestly shown as unavailable, never papered over by deleting user content.

10. Storage layout & RLS

One private bucket, attachments. Object path <workspace_id>/<content-key> — content-addressed: content-key = HMAC(K_id, sha256(plaintext)) for an E2EE workspace, plain sha256(bytes) for a plaintext one. For E2EE the object is the encb:v1: ciphertext; for plaintext, the raw bytes (identity case, §5). Content-addressing makes uploads idempotent and object dedup an invariant; the per-workspace HMAC key is what keeps it from becoming a content oracle (below).

Why a keyed content hash (and what it buys). Content-addressing is the design we want — it makes uploads idempotent (same bytes → same path → a re-PUT is a no-op) and dedup an invariant rather than a race. The only thing that rules out a raw sha256(plaintext) path under E2EE is the content oracle: <ws>/<sha256-of-plaintext> would let the untrusted server confirm "does this workspace contain this known file?" by hashing a candidate and probing for the object. The fix is to key the hash: <ws>/HMAC(K_id, sha256(plaintext)) is deterministic within a workspace (idempotent + dedup), unguessable to the server (no oracle), and distinct across workspaces (no cross-workspace correlation). A plaintext workspace has no secret to protect, so it uses the raw sha256(bytes) — the server already holds those bytes.
  • K_id is an HMAC subkey derived from the WK material via HKDF at import time — not the stored WK handle, which is a non-extractable AES-GCM key WebCrypto won't sign with. Concretely K_id = HKDF-SHA256(ikm = raw WK bytes, salt = "", info = "km/asset-content-key/v1", L = 32), imported non-extractable as {name:'HMAC', hash:'SHA-256'} and used at full length (path-encoded, never truncated — truncation only invites two plaintexts to collide on one path; harmless to confidentiality since the read check holds the full sha256, but a needless self-inflicted DoS). The empty salt is safe (the WK is already 32 uniform bytes); the info label is not optional — it domain-separates this derivation from any other future WK-derived key (the deferred §12 / e2ee §13 key hierarchy, the §13 per-asset share key), exactly as aad.ts domain-separates the canary from content columns. The raw key bytes are briefly in scope at unlock/create (parseWorkspaceKey / importWorkspaceKey) before the non-extractable import; that is the only window to derive K_id — and the WK bytes are zeroed right after import (createEncryptedWorkspace.ts), so K_id can't be re-derived on a later reload. (The unlock path unlockWorkspaceWithKey.ts has no such derive window today — it doesn't even zero the parsed bytes, which drop to GC — so adding derive-then-zero on unlock is a prerequisite of the storage-path phase (§15 phase 2, where K_id is first needed), and incidentally fixes that standalone GC-leak.) It must therefore be persisted as its own non-extractable HMAC CryptoKey in the keyStore, derived once at import/unlock. That adds a second stored handle but no WK-exposure widening — it can't recover the WK, and lives in the same IndexedDB the model already trusts. Co-locate it with the WK in one keyStore record (the store today holds a single CryptoKey per (user_id, workspace_id), so this widens the record shape): IndexedDB is best-effort evictable, and losing only K_id while the WK handle survives would silently fail-close media mid-session (text still working) — so they must evict together, and any K_id-absence routes through the same re-paste affordance as the migration below.
  • Migration: existing E2EE devices have no K_id yet. A workspace unlocked before this feature ships holds a non-extractable WK handle but never derived K_id (the raw bytes are long since zeroed, and a non-extractable AES-GCM key can't be HKDF'd or exported). Such a device must re-paste the WK once (or re-unlock) to populate K_id. Implemented choice (the simpler one): a K_id-less record is treated as LOCKED — the workspace gate prompts a re-paste and the whole workspace (text included) is read-only until it's supplied, rather than a media-only fail-closed path that keeps text live. We took the uniform gate (one "no usable key ⇒ locked" rule) over the narrower media-only carve-out; the stored WK handle is preserved (never discarded), so the re-paste fully restores access and the only cost is a one-time re-paste — acceptable for the small coordinated fleet. A named step, not a silent break. (src/sync/keys/resolveWorkspaceEntry.ts gates on contentKeyHmac !== null.)
  • What this removes (vs a block-id path): the idempotency token (§9/§10.1), the write-once monotonicity rule (§5.1), and duplicate-object GC (§11) — the path is the idempotency key and the immutability guarantee. Fewer moving parts, not just better scaling.
  • Its one cost — the path no longer carries the block id, so a future share-RLS join (object → block) needs a separate content_key → block mapping (§10.1). Zero today: subtree sharing is unshipped and not a design driver (§12).

Every operation is a direct, RLS-gated client call — there is no mediating upload service. Reads are member-gated; inserts are writer-gated; updates have no policy (denied), which is exactly what makes a content-addressed object write-once; deletes are writer-gated:

create policy "attachments read by workspace member"
on storage.objects for select to authenticated
using (
  bucket_id = 'attachments'
  and private.is_workspace_member(
        (storage.foldername(name))[1],   -- <workspace_id>
        (auth.uid())::text)
);

-- INSERT is a DIRECT writer-gated upload. The WITH CHECK keys on the path's
-- first segment, so a writer can only write into a workspace they write — there
-- is no separate workspace param to spoof. The array_length check enforces the
-- FLAT layout (exactly <ws>/<content-key>, one folder level): without it a
-- writer could upload <ws>/sub/plaintext — RLS-allowed (first segment is their
-- workspace) but hidden from the audit's top-level scan, so plaintext could
-- evade the §10.1 tripwire under an E2EE subfolder. There is intentionally NO
-- body inspection: a storage.objects policy can't see the body bytes (§10.1).
create policy "attachments insert by workspace writer"
on storage.objects for insert to authenticated
with check (
  bucket_id = 'attachments'
  and array_length(storage.foldername(name), 1) = 1   -- flat: <ws>/<key> only
  and right(name, 1) <> '/'                          -- non-empty content-key
  and private.is_workspace_writer(
        (storage.foldername(name))[1],   -- <workspace_id>
        (auth.uid())::text)
);

-- NO update policy → an object is write-once (immutability). The path IS the
-- content key (§10), so a second upload to an existing path is a Storage 409 the
-- client reads as first-write-wins idempotent dedup, never an overwrite. Storage
-- needs UPDATE for every other mutation verb too — upsert, move (rename), and
-- copy-onto-existing all require UPDATE — so with no UPDATE policy NONE of them
-- can overwrite an existing object. A writer's only mutation is delete (then a
-- fresh insert) — the accepted §17 poison, not a silent overwrite.

So the mutation split is: reads member-gated, inserts and deletes writer-gated, updates denied (immutability) — all direct RLS, no mediating service. The private.* grant is not free — verify it, don't assume parity with blocks. private.is_workspace_member / is_workspace_writer exist and are SECURITY DEFINER, but the migrations contain no explicit GRANT USAGE ON SCHEMA private / GRANT EXECUTE to the app roles — the blocks RLS works via defaults and the fact that the policy and the table live in public. A storage.objects policy runs in a different ownership/role context (the table is owned by supabase_storage_admin, in the storage schema), so "the same functions blocks uses" is not automatically callable there. Treat the schema-USAGE + EXECUTE grants (or a thin public wrapper) as a first-class, tested step of the bucket migration — a wrong assumption yields a policy that errors at query time or silently denies all reads.

For an E2EE workspace the blob is already ciphertext, so RLS here is defense-in-depth — it stops a non-member from hoarding ciphertext against a future key compromise, and avoids leaking object existence. For a plaintext workspace it is the actual access boundary.

10.1 No server-side write guard — direct writes, with a read-side backstop

RLS is not a ciphertext guarantee, and — unlike blocks — there is no server-side one either. For blocks, a server trigger rejects a plaintext write to an E2EE workspace (is_enc_v1_envelope shape-check), because Postgres has the row body inline. The naïve analog for bytes — "a storage.objects trigger that checks the body" — cannot exist: Supabase Storage object policies/triggers see object metadata (path, owner, size), never the body bytes — the bytes live in S3 and storage.objects is just a metadata row. A check that reads bytes could only sit in code on the write path, and would only enforce anything if that code were the sole writer (a mediating Edge Function). We deliberately do not pay that.

Decision (reversal, 2026-06-25): direct writer-gated uploads; no body-inspecting upload service. An earlier draft made a service-role Edge Function the sole writer so the server could enforce the encb:v1: byte shape, matching the block trigger. We reversed it: that guard is not worth a function on every upload, because the byte-confidentiality outcome is already closed elsewhere.
  • The guard was shape-only, and a malicious writer forges it. The check is an 8-byte magic compare (the server has no key); a writer who wants the server to see plaintext sends encb:v1:‖zeros‖plaintext‖zeros and passes it — or just exfiltrates with the workspace key they already hold. Against the threat that matters (a malicious writer / the untrusted server) it buys nothing.
  • The read side fail-closes — and that is now the sole byte-confidentiality control, so it was a hard Phase-3 acceptance gate. The resolver (§7.3, built — src/attachments/resolver.ts) hash-verifies every fetched object against the block's synced hash and AEAD-opens it (§5.1/§7.3); anything that isn't the genuine plaintext (a fetch failure, an AEAD-open failure, or a hash mismatch — including a poisoned object that AEAD-opens under a known-hash AAD) renders a broken-image placeholder and is never stored or served as content. Phase 1 shipped the primitives (decodeBytes, verifyContentHash); Phase 3 wired them into the resolver — with the tamper/swap and hash-mismatch tests §15 lists — which is what makes this reversal safe; nothing read these objects until it landed. (This is the principled asymmetry with blocks: bytes are content-addressed and self-verifying on read; block text is not — so byte-confidentiality can rest on a read-side check where block-confidentiality can't.)
  • It is a no-op for plaintext workspaces — no body to protect; the server holds those bytes by design.
  • The one residual it uniquely caught is an honest client with an encryption regression accidentally uploading bare plaintext into an E2EE prefix — harmless to readers (fail-closed) but visible to the untrusted server at rest until GC. We cover that with an off-path periodic audit (scripts/attachments-ciphertext-audit.ts) that scans E2EE prefixes and alerts on any object that is not a plausible encb:v1: envelope — missing the magic, or carrying the magic but shorter than the envelope minimum (magic + nonce + tag), so a truncated runt can't pass on the prefix alone — turning a client bug loud without putting a function on the write path. It alerts, it does not delete (deleting can't un-see what the server already holds, and a one-test client invariant catches the bug class more cheaply).
So the write path is a direct, RLS-gated insert (§10): the WITH CHECK keys on the path's first workspace segment, re-imposing exactly the writer authorization the removed function did (and closing the same cross-workspace path-smuggling — the policy evaluates the real first segment of the object being written, not a spoofable param). What the function did that native RLS now does: authorization (writer INSERT policy), write-once immutability (no UPDATE policy → a re-upload to an existing content path is a Storage 409 the client reads as first-write-wins idempotent dedup, never an overwrite), and the size cap (the bucket's file_size_limit). The encb:v1: shape is now the client's pre-upload invariant (§9), backstopped read-side and by the audit. (Storage finalizes an object only on a complete upload, so a truncated request can't win a path with a partial object — the atomicity the function had to hand-roll is now the platform's.)

Existing-path dedup still hash-verifies (in the §9 up-lane drain — Phase 5, not yet wired). On a 409 the uploading client must not blindly clear its §9 queue entry: it fetches + hash-verifies the stored object against its own hash — on a match it clears (genuine idempotent dedup); on a mismatch (a poisoned or buggy prior write) it keeps its bytes and runs writer-delete + re-upload rather than dropping the only good copy. First-write-wins makes a bad first write sticky until the path is freed — an accidentally corrupt path is correctable by a writer delete + re-upload (§10 writer-delete RLS), but an active member-poisoner can DoS a known asset (delete-then-re-PUT doesn't terminate against them), so after a bounded number of correction cycles the client moves its §9 record to failed (§9 — keeps its exempt bytes, surfaces for discard, re-attempts opportunistically once the path frees): a named tradeoff in §17.

Per the project's Supabase rule, the bucket + its RLS land in one migration with storage.objects policies enabled; no public table is created without RLS.

11. The renderer & capture surfaces

Packaged as one attachments plugin, mirroring the video-player plugin's facet wiring:

  • MediaBlockRenderer via blockRenderersFacet, with canRender = ({block}) => block.hasType('media') and a priority above the default. Branches on mime: image → the existing MarkdownImage lightbox fed the object URL the in-thread resolver produces (§7.3); later, PDF / audio / video / generic file-chip.
  • No second image representation. An attachment is always a media block rendered through its !((id)) embed (above) — there is no ![](asset-ref) markdown form to special-case (that would side-step the block model for no gain). Plain external ![](url) images keep today's MarkdownImage lightbox unchanged; ingesting one into our storage means converting it to a media block + embed, not rewriting its src to an internal sentinel.
  • Capture routes through the existing paste verb. Image paste / drop is a branch of the established pasteDecisionVerb (src/paste/decision.ts — already the seam both CodeMirrorContentRenderer and DefaultBlockRenderer route paste through), by extending its input to carry clipboardData.files / dataTransfer.files so a file payload decides "create a media block" — not a parallel domEventHandlers interceptor. The /image command + file-picker is the explicit-invocation path through the existing action / shortcut system.
  • Pending-attachments affordance. The per-device up-lane records (§9) — both pending and failed — surface as a small "pending uploads" list whose entries are individually discardable — this is the only offline release valve when intake backpressure hits (§8), so it's a required capture surface, not merely a "bytes pending here" indicator. Discard is content-refcount-gated (§8/§9/§16) — record and bytes both go only as the last reference — so discarding one media block's record can't strip a co-referencing block's bytes or strand its upload. When a discard reclaims nothing because the content is still shared (the rare copy/import case), the affordance must say so and point at the co-carrier — a silent no-op would leave a user under intake backpressure with no model for why the space didn't free. (This differs from the §9 boot-reconciler reap, which drops a record whose own block is absent-past-checkpoint while keeping shared bytes — different antecedent, same content-refcount gate.)
  • On capture: client-side mime/size guard → hash → create the asset block at a deterministic, content-derived id — uuidv5(<workspaceId>:<content-key>), where the content-key is the keyed hash for E2EE, raw sha256 for plaintext (workspace-scoped, like the storage path §10). It MUST be a UUID: the embed is a block-ref (!((id))) and the block-ref grammar only matches UUID-shaped targets, so a literal media:<content-key> id would render as plain !((media:…)) text, never a media embed. (The media:… form earlier in this doc is shorthand for "deterministic, content-derived", not the literal id string.) Mint it through the codebase's established deterministic-id helper (createOrRestoreTargetBlock — the path daily-notes / roam-import / alias-seats already use), not a bare createOrGet: that helper (a) systemMints the row at the pristine updated_at = 0 sentinel, so two devices' first pastes reconcile to one row instead of racing wall-clocks, and (b) catches DeletedConflictError and restores — so re-pasting an image you just undid (its block is soft-deleted, §9) restores the same block rather than throwing — and re-paste re-stages the byte record from the bytes the paste holds in hand (§8), so it doesn't depend on the undone asset's byte-store entry having survived eviction/GC. (On the restore branch systemMint is inert — a restored block lands a real updated_at, not 0, which is harmless: a real version only wins or ties, never spuriously yields. No seat-probe is needed either, unlike alias seats: the id is the content key, so the only id-collider is identical content — exactly the dedup we want.) Repeat / concurrent pastes therefore dedup to one block id + one object, no lookup (and the keyed id avoids leaking a raw-hash content oracle in the clear blocks.id column — it does hand the server a free object↔block equality join, the same content-key labelling both, but that correlation is timing-available anyway and stays workspace-local since K_id differs per workspace). Placement is not under the containing block: because dedup makes the asset block shared workspace-wide (it may be embedded from many notes), parenting it in one note's subtree would let that note's soft-delete — which cascades to the subtree (§9) — tombstone the shared asset and break every other embed. So media blocks are anchored in a dedicated workspace-level assets container (flat — not a disposable subtree) and embedded everywhere via !((id)). The type is stamped via repo.addTypeInTx(tx, id, 'media', …) (the in-tx variant — the standalone block.addType wouldn't compose into the single tx; and the media TypeContribution must be registered via typesFacet first, §3, or this throws). The asset-block create + the !((id)) insert go in one repo.tx({scope: ChangeScope.BlockDefault}) so they share an undo bucket. The durable byte record is staged (non-drainable) BEFORE that tx, and the drain is armed only AFTER it resolves — not inside it (see the box below).
Stage the byte record durably before the metadata tx; gate its drain on the commit. repo.tx only makes the SQLite/PowerSync writes atomic — an IDB/OPFS byte-queue write is an external side effect that neither joins that tx nor rolls back with it. Two failure windows have to close at once. (a) If the byte write happened after the commit and then failed (quota, private-mode, a store error), the metadata is already committed and syncs to peers with no durable source bytes, no per-device pending status, and no queue entry to drain — a permanently un-uploadable asset on every peer. (b) If the enqueue instead kicked off a drain before the commit is visible, the §9 "block still exists?" pre-PUT re-check would drop the bytes as a missing block. The resolution closes both: write the durable byte record first in a non-drainable staged state (§9), then run repo.tx, then flip staged → pending and arm the drain. Because the bytes are staged before the commit, a committed asset block always has its source bytes already on disk — window (a) is gone (a failed stage aborts capture before any metadata commits, attaching nothing); because staged is not drainable, the drain can't race ahead of the commit — window (b) is gone. The post-commit flip can itself fail, but that is recoverable, not lossy: the durable staged record survives, and a boot-time reconciler (§9) promotes any staged record whose asset block did commit. Undo needs no special byte-lane step (§9): the shared repo.tx soft-deletes the block and best-effort cancels the queue entry; any already-uploaded object is left in place (immutable, content-addressed) and owned by §16's refcounted sweep — never deleted on undo.
Object dedup is an invariant; block identity dedups too on the capture path. The object path is content-addressed (HMAC(K_id, sha256), §10), so concurrent pastes of the same image — two devices, two tabs, offline-then-reconnect — all compute the same path and converge on one Storage object (first-write-wins idempotent, §10.1; keyed off the plaintext hash, so randomized GCM doesn't matter). No duplicate objects, ever. And because capture mints the block at a deterministic content-derived id (above), those same pastes resolve to one block id too — no duplicate block, no lookup. One honest caveat: the cross-device merge of that shared block is whole-row last-writer-wins, not a field merge, so a non-hash property that differs between two pastes (e.g. filename) LWW-resolves — cosmetic, and the render-critical fields (hash, size) are identical by construction; the user-authored caption / alt-text are child blocks, untouched by the parent upsert. The system still tolerates multiple blocks per object from other paths (a block copy, an import) — harmless: they share the one verified object and both render. Crucially, every capture — including a dedup hit on an already-existing block — stages its own in-hand bytes and uploads idempotently (§11 staging box): so reusing an asset whose object hasn't uploaded yet (e.g. the originating device is still offline) does not drop this capture's bytes — whoever is online first materializes the content path. So object GC stays content-reference-counted: GC the object only when no live-or-soft-deleted block carries the hash, behind §16's grace-windowed deferred sweep — never on a live zero-crossing (moving an embed, or a not-yet-synced cross-device add, transiently hits zero), preserving child content. Dedup is workspace-wide, deliberately not subtree-scoped (the model isn't tree-shaped and subtree sharing is unshipped — §12).

12. Subtree sharing (deferred — not a design driver)

Subtree sharing (shares-and-api-tokens.html) is not shipped, and it doesn't map cleanly onto the current data model — the app isn't strictly tree-shaped anymore, so "share this subtree" is itself an unsolved design a future effort would have to work out. So this design does not bend around it: dedup is workspace-wide (§11), asset placement isn't share-driven, and we add no share machinery now. We also don't foreclose it — the content-addressed storage path keeps a future "membership or a share covers this block" RLS join possible (via a §10.1 content_key → block mapping), and retargeting a workspace-wide asset into a shared scope (e.g. copy under the shared root at share time) would be that feature's job, not ours.

E2EE + subtree sharing is doubly deferred. Even once sharing exists, an E2EE subtree-share recipient needs a key — but there is one WK per whole workspace and no per-subtree key; that needs the key hierarchy the e2ee doc defers (e2ee §13). So encrypted subtree sharing is a non-goal for the foreseeable future and media just inherits that boundary (consistent with e2ee §15 #2; E2EE sharing stays out-of-band whole-workspace WK transfer). Don't special-case it.

13. The public-sharing escape hatch

We still reserve properties.publicUrl on asset blocks and a render-audience flag on the block context ('workspace' member / 'shared' subtree recipient / 'public' logged-out). But the implementation for a public, logged-out viewer differs by mode:

ModePublic-share implementation
Plaintext workspace A public bucket copy (or long-lived signed URL) written to properties.publicUrl; the renderer prefers it when audience is 'public'. The original signed-URL story works here.
E2EE workspace Either export-to-plaintext (decrypt + re-upload to a public bucket — the asset deliberately leaves E2EE, an explicit, consensual exit, never silent) or key-in-fragment (app/share#<per-asset-key>; the client decrypts from the URL fragment, which the browser never sends to the server). "Anyone with the link" is exactly key-in-link semantics.
If built, key-in-fragment requires a fresh per-asset key — NEVER the WK. The §5.1 crypto seals with the WK directly (no per-asset key in v1), so a naïve "put the decrypt key in the fragment" would put the WK in a URL and expose the whole workspace. The mechanism (vNext): generate a one-off key, hash-verify the source object against block.properties.hash first (a public viewer can't — §5.1's read-side check lives in the authed app — so a poisoned object §10.1/§17 isn't laundered into a public artifact), re-encrypt under the one-off key, upload to a public bucket, and put only that key in the fragment; the WK never leaves the device (e2ee §5). Two tradeoffs to state when it ships: the fragment is a softer target than the CSP-hardened app (it persists in history, is readable as location.hash, can ride a Referer), and a key-in-link share is irrevocable — once the link leaks, the asset leaks, permanently. Both escape hatches are vNext; the v1 commitment is only the reserved property + audience flag.

"Reserve" here is light, and worth being honest about: block properties are a schemaless map, so publicUrl can simply be added when the feature ships — that's not a migration — and the render-audience branch is renderer code, also addable later. So v1 pre-provisions nothing mechanical; it just names the plug-in points (a publicUrl property + an audience-aware URL resolver branching on workspace mode × render audience) so the design doesn't accidentally foreclose the vNext escape hatch. Practically: don't repurpose the publicUrl key for anything else.

14. Vendor portability (the BlobStore seam)

All object-store access goes behind a ~50-line interface:

interface BlobStore {
  // The object is addressed by a per-workspace CONTENT KEY, not the block id:
  //   contentKey = HMAC(K_id, sha256(plaintext))  (E2EE) | sha256(bytes) (plaintext), §10.
  // The resolver computes it (it holds K_id + the plaintext hash from the decrypted
  // block); BlobStore just routes bytes to /. No idempotency token —
  // the content path IS the idempotency key: a re-PUT to an existing path is
  // first-write-wins success (§10.1). The plaintext sha256 never reaches the origin
  // as such: for E2EE the wire path is the HMAC, unguessable without K_id (no oracle, §10).
  put(workspaceId, contentKey, bytes): Promise<void>     // direct, RLS-gated writer upload (§10.1)
  get(workspaceId, contentKey): Promise<Uint8Array>       // bytes, not a url: the private
  //   bucket (§10) rules out a bare <img src>; the resolver fetches → decrypts →
  //   hash-verifies (§5.1/§7.3), so the seam returns BYTES the resolver turns into an object URL.
  delete(workspaceId, contentKey): Promise<void>
}

v1 backs it with Supabase Storage. E2EE actually increases portability: because the server is a dumb encrypted-blob store and never mediates bytes, swapping to R2 / S3 / B2 later is a bytes-copy plus re-pointing one module — no auth or schema rework. We do not pre-migrate; we keep the seam clean and pull the trigger if egress cost or another driver appears. (A Worker-fronted R2 from day one is defensible only if a Worker is being introduced for other reasons; otherwise it is permanent operational surface for no present benefit.)

16. Open questions

  • Multi-account profile (decided shape, needs care): in-thread, each tab already owns its own Supabase session, so there is no shared-SW session slot to mis-key. The residual work is scoping the OPFS byte store by user_id and preserving the keyStore's (user_id, workspace_id) keying under an account switch, so account A's cache / keys can never serve account B (§7.3).
  • Max inline size + large-media story: a v1 cap (e.g. 10 MB) with a clear error, deferring resumable/tus upload to vNext. Where exactly is the cap enforced?
  • Asset block deletion → byte GC (must be conservative under content-addressing): because one content-addressed object may back several blocks (§11), deleting a block does not delete the object. Object GC is content-reference-counted, but the refcount can only be evaluated client-side against one device's local view (the hash is in encrypted properties_json — the server can't read it). A naive "no live block carries hash H → delete" loses live data three ways: a not-yet-synced block on another device still needs H; a soft-deleted block (delete is a tombstone — mutators.ts — that still carries H) can be undone/restored; and blocks_history (the device-loss-resistant PITR substrate) still references H. So object GC must (a) run only on a settled-synced workspace (the §9 hydration gate), (b) sit behind a grace window like block GC — never a live zero-crossing, (c) count every live or soft-deleted block and every blocks_history reference to the hash — and note the code has no soft-delete retention/purge or blocks_history horizon today (soft-deletes are kept indefinitely, history is append-only), so the refcount is effectively reference-permanent in v1 and object GC essentially never fires for a hash any block or history row ever carried; that is the intended err-toward-retention posture, not a bug — and (d) be recoverable by re-upload from any device still holding the bytes, since the cross-device-unsynced race can't be fully closed from one view. Open: if a soft-delete retention horizon or history pruning is ever added, object GC must gate on it — and on whether a history/PITR restore is meant to recover media bytes (else media is an explicit PITR carve-out); decide that when the horizon is introduced. Today there is no horizon to gate on, and v1 errs toward retention (storage is cheap; a stranded live image is not). The accepted cost is monotonic Storage growth: an object stays referenced (so un-reclaimed) as long as any soft-deleted block or blocks_history row carries its hash — which today is forever — so per-workspace object Storage only grows over v1's life. This is also an adversarial lever, not just an honest-use cost: a member can pin unbounded unreclaimable storage by uploading distinct-content garbage objects (each a new path, first-write-wins, never overwritten), and even deleting the block doesn't reclaim them (the soft-deleted block + its history rows keep the hash referenced) — an accepted residual at this fleet size that the deferred history horizon would bound. Dangling embeds render the existing "unresolved embed" state. Beyond the undo case (which needs no byte-lane action — it's a soft-delete; the object is retained, §9), a vNext orphan sweep covers both directions, but they are not symmetric: orphan objects (a Storage object with no block — e.g. an interrupted delete) are auto-deletable; orphan blocks (an asset block whose object never materialized — e.g. an upload that never completed before the origin device was cleared / lost / reinstalled) are never auto-deleted — their caption/transcript children are real content and a block delete cascades (§9), and elapsed time can't tell an offline-but-intending author from a gone one. The block direction is handled by the peer's bounded "not yet uploaded" settled state + self-heal (§9), with removal left to the user — not a timed server reap (which is why no server-side intent marker is needed).

Block dedup is also an invariant on the capture path (deterministic content-derived id, §11) — other paths (copy, import) may still make multiple blocks per object, harmlessly. The content-path oracle is closed only under K_id secrecy — K_id is static and never rotated, so its compromise reopens the known-file oracle retroactively for that workspace. Because K_id is a non-extractable handle its raw bytes never leak, so the residual is narrow: an attacker who can execute script inside the authenticated origin (XSS or a malicious extension) could use the handle to compute content paths and probe for known files, but cannot exfiltrate K_id itself or widen WK exposure. An accepted residual.
First-write-wins is poisonable by a member Any member can compute HMAC(K_id, sha256(x)) and pre-occupy a content path with garbage; first-write-wins then silently no-ops the honest upload of that exact content → a workspace-wide broken image. The predictable uuidv5(workspace:content-key) block id (§11) is a second pre-occupiable target, but the consequence is identical and bounded (a member-authored, soft-deletable / restorable block whose hash-mismatch renders fail-closed) — a widened DoS surface, not a new spoof. A content-targeted DoS primitive the block-id path lacked. Accepted & named: read-side hash-verify (§5.1) makes it fail-closed (broken image, never spoofed content); the server (no K_id) can't do it; a writer delete + re-upload corrects an accidentally corrupt path but does not terminate against an active poisoner (who re-occupies the freed path), so a member can permanently DoS a known asset — and workspace-wide dedup makes that break every embed of it at once (§10.1). Members are inside the content-correctness trust boundary, and the server-side "verify body matches path" defense is foreclosed (the guard has no key). It stays a broken image, never data loss: the honest uploader's bytes are un-uploaded, so their up-lane record lands in failed which keeps the exempt bytes (§9) — and re-attempts opportunistically (§9) to finish the upload once the poisoned path is freed or the object materializes elsewhere, so the asset self-heals rather than being lost.
Concurrent offline replace of one embed Two devices replacing the same image offline each mint a new immutable asset block; on reconnect the embed pointer (plain block content) resolves last-writer-wins, so one replacement is silently dropped (its orphaned asset is GC'd). Accepted & named (not hidden): no byte corruption — immutability holds; only the pointer is LWW. A merge-surfacing / both-render variant is a vNext product call.
No server-side media features for E2EE No server thumbnails / OCR / transcode for encrypted workspaces. Inherited from e2ee §15 #2, not new. Client-side transforms remain possible.
E2EE subtree-share of media unsupported Encrypted workspaces can't subtree-share images until the key hierarchy lands. Needs e2ee §13. Plaintext workspaces are unaffected; don't special-case.
Supabase Storage in v1 Egress cost at scale. Behind BlobStore; E2EE makes the backend a dumb blob store, so the swap is a bytes-copy + one module.

18. Design notes — where the complexity lives, and the seams that contain it

This design went through several adversarial review passes; the findings clustered in a few places, and the pattern is worth recording so the implementation leans on the seams that keep it simple.

  • Reuse the proven block lane; don't re-derive a parallel one. "An attachment is a block" (§3) is load-bearing: metadata sync, encryption (transform.ts), undo, and permissions all come for free by riding the existing block path. The fragile parts are exactly where the byte lane diverges from how blocks already work — blocks have ps_crud (→ we need a §9 queue), a server ciphertext trigger (→ bytes can't have one, §10.1, so confidentiality leans on the §5.1 read check + an off-path audit), and PowerSync monotonicity (→ a §5.1 hash check). Each divergence had to re-solve a problem blocks already solved. Prefer mirroring the block mechanism over inventing a byte-specific one wherever the choice exists.
  • Funnel URL + mode + hash + cache-key through one asset-resolver seam. Many review findings were not "the design is wrong" but "a fact didn't propagate" — the base-path prefix, the expected-hash source (SW mirror), the mode decision, and the cache key are all facets of one operation ("given an asset block + render context, produce displayable bytes"). Implement that as a single named function so those parameters can't drift apart; scattering them across call sites is what manufactured the drift.
  • Content-addressing is the storage path, keyed for E2EE (§10): the keyed-hash path makes uploads idempotent and object dedup an invariant, and removes the idempotency token, the write-once monotonicity rule, and duplicate-object GC a block-id path would have needed — the simpler-correct choice, not merely the better-scaling one. The one cost (an object→block mapping if subtree-share RLS is ever built) is zero today (§12).
  • Decided: in-thread decryption (§7.3). The biggest genuine-complexity call was not putting decryption in a service worker — a context that can't read app state and is shared/respawned, so it would need ~5 coordination mechanisms to buy a uniform <img src> that <img> doesn't need (it takes an object URL). The full tally is §7.1 / the sibling doc; the portable lesson — a SW earns its tax only when the consumer needs a real URL, not a blob (cf. the user-plugin /module/<id> case) — is the keeper.
