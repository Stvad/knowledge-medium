Action system — architectural critique

Fresh-eyes review of src/shortcuts/ + the action facets — 2026-05-30

Written alongside docs/shortcut-context-precedence.html. That doc fixes one bug (double-fire) and adds one knob (precedence). This doc steps back and asks the broader question the precedence work surfaced: is the action system's shape right, or are we patching a model that wants restructuring? It is a critique + target shape, not an implementation plan. No code changes proposed here.

Bias check up front. This system is not in bad shape. The facet contribution model is genuinely good and most of the friction is concentrated in two places (resolution and the transform pipeline). The honest recommendation (bottom) is "two surgical refactors, leave the rest alone" — not a rewrite. Flagging this so the list of weaknesses below isn't read as a teardown.

The model in one screen

  • Actions (ActionConfig) are contributed via actionsFacet (core.ts:158). Each carries id, context, handler, optional defaultBinding, canRun, icon. ~23 files contribute.
  • Contexts (ActionContextConfig) via actionContextsFacet (core.ts:225); 6 call sites. A context is both an activation scope and a typed dependency bundle (via validateDependencies).
  • Activation is React-lifecycle-driven: blocks call useActionContextActivations (useActionContext.ts:17), which activate()s on mount/effect and deactivate()s on cleanup. The active set is an insertion-ordered Map (ActiveContexts.tsx). 27 surface-activation references.
  • Transform pipeline: actionOverridesFacet → actionDecoratorsFacet → keybindingOverridesFacet, composed in getEffectiveActions (effectiveActions.ts:66).
  • Dispatch happens three ways: keyboard (per-action window listeners in HotkeyReconciler), runActionById (module singleton), and useRunAction (React hook). The latter two share getActiveActionById; the keyboard path does not.

What's good — keep it

  • Facet-based contribution. Actions, contexts, overrides, decorators all flow through facets — plugins compose without editing core, no circular deps. This is the load-bearing strength; nothing below proposes touching it.
  • Decorators that wrap rather than fork. Extending an action's canRun/handler without owning the definition (e.g. srs-rescheduling/rescheduleDecorator.ts) is the right shape for cross-plugin extension.
  • Per-action install/uninstall. Each action owning its subscription makes activation changes cheap (no global teardown). Worth preserving even if the listener model changes.
  • Typed dependency map. ShortcutDependenciesMap keyed by context type is a clean way to give handlers the right inputs.

Weaknesses — grouped by theme

1. Resolution is implemented three times and the keyboard one disagrees

"Given an active set, which action handles X?" is answered in three places:

  • runActionById → the module dispatcher installed by HotkeyReconciler (HotkeyReconciler.tsx:174) → getActiveActionById (last-activated-context wins).
  • useRunAction (runAction.ts:54) → also getActiveActionById, but re-implements the deps-lookup/throw around it.
  • The keyboard path → no resolution at all; every active action installs its own listener, so collisions double-fire (the bug the precedence doc fixes).

There is no single "resolve event → action" function. The keyboard path and the dispatch path have different semantics (double-fire vs single-winner) for what is conceptually one question. Precedence (priority/modal) then has to be taught to each path separately. This is the deepest structural issue and the one the precedence work half-forces us to confront.

2. Three overlapping transform mechanisms

ActionOverride, ActionDecorator, and keybindingOverridesFacet all rewrite the action list, with subtle, undocumented differences:

  • Override vs decorator asymmetry is arbitrary. Both are "(action) → action" transforms; only one may return null (types.ts:164 vs 170). A plugin that wants to conditionally remove an action must use override; one that wants to extend it uses decorator — but nothing names that distinction, so the choice is folklore.
  • Two ways to suppress, neither obvious. The user's "skip decorators" (no-op the handler) and a hypothetical override → null (drop the action) both "disable" an action, with totally different consequences (handler-still-listed-but-inert vs gone-from-palette). No guidance on which to reach for.
  • The '*' wildcard (effectiveActions.ts:27) is a back-door for set-wise transforms bolted onto a per-action API — used once, for the keybinding bridge, and easy to misuse.

3. canRun looks authoritative but is presentational

canRun reads like a guard, but it gates only UI surfaces (palette, swipe menu); the keyboard and runActionById paths never call it (types.ts:148–156 spells this out, but the name fights the doc).

  • The user's skip-decorators exist because there's no dispatch-level "should this run?" — they fake one by mutating the handler. A real predicate-at-dispatch (the precedence doc's Option D) is the missing concept canRun appears to promise but doesn't deliver.
  • Name implies authority it doesn't have. isApplicableForDisplay would be honest; or make it actually gate dispatch and the skip-decorator pattern evaporates.

4. Precedence is incidental because activation order is React-lifecycle order

The active set's order is whatever sequence useEffect activations fired in (ActiveContexts.tsx:86–97). "Latest-activated wins" therefore inherits mount/effect ordering as its precedence signal — which is why the new-surface case resolves backwards (surface mounts early, normal-mode activates on click). The precedence doc adds an explicit priority to escape this, which is the right patch — but the underlying smell is using a lifecycle artifact as a semantic ordering.

5. The runActionById module singleton

runAction.ts holds a module-level dispatcher var set by HotkeyReconciler on mount and nulled on unmount. It works, but: it's global mutable state, it throws if called before mount, and it couples "can I dispatch an action" to "is this one React component mounted." For a system whose whole point is decoupled contribution, the dispatch entry point being a singleton side-channel is a wart.

6. Context conflates scope and dependency-bundle

An ActionContextConfig is simultaneously (a) an activation scope that gates install, (b) a modal/precedence tier, and (c) a typed dependency carrier (validateDependencies + ShortcutDependenciesMap). These mostly co-vary, but the conflation shows up as type friction: as never / as ActionConfig casts in the override/decorator pipeline (effectiveActions.ts:52,57) and the bivarianceHack in ActionHandler (types.ts:126). The generics are being fought rather than expressing the model.

The unifying idea: one resolution core

Most of the weaknesses above (1, 3, 4, partly 5) are the same shape: there is no single place that owns "active set + event → which action runs." Resolution is smeared across per-action listeners, two dispatch helpers, a presentational predicate, and an implicit lifecycle ordering. The high-leverage refactor is to introduce one resolver and route every path through it: 
resolve(activeSet, event | actionId): ActionConfig | null
  // single comparator: (priority desc, activation-recency desc)
  // single canonicalization of chords
  // single place to add modal / declinable / future tiers
Keyboard dispatch, runActionById, and useRunAction all call it. Precedence, canonicalization, modal, and (if chosen) declinable fall-through become properties of the resolver, defined once, instead of being re-taught to each path. This is also exactly what the precedence doc's "shared comparator, used by both paths" bullet is reaching for — generalized.

What's worth doing — prioritized

PriorityChangeWhy / payoff
P1 Single resolution core; route keyboard + both dispatch helpers through it (subsumes the precedence doc's comparator + the double-fire fix) Kills the keyboard/dispatch divergence (weakness 1), gives precedence/canonicalization/modal one home, makes Option D a localized change later instead of a rewrite. Highest leverage.
P1 Decide canRun's identity: either rename to a display-only name, or split into isVisible (UI) + canDispatch (gate) Removes the reason skip-decorators exist (weakness 3). Cheap, clarifies a contract people already misuse.
P2 Unify the transform pipeline: one (action) → action | null transform type with a documented phase, fold decorator/override into it; keep keybinding-override as the one cross-action pass Ends the override/decorator asymmetry and the "which do I use?" folklore (weakness 2). Medium risk — touches every contributor, so worth a codemod.
P3 Replace the runActionById singleton with a runtime-scoped service obtained from context Removes global mutable state + mount-coupling (weakness 5). Mostly mechanical; low urgency.
P3 Tidy context generics (drop the as never/bivarianceHack by modeling scope vs deps separately, or accept and document them) Cosmetic/type-hygiene (weakness 6). Only worth it if touching the area anyway.

Explicitly NOT worth doing

  • Don't replace facets with a registry/event-bus. The contribution model is the part that works; churning it buys nothing.
  • Don't collapse contexts into a flat keymap. The scope concept earns its keep (modal, dependency typing, per-surface activation). The problem is resolution across contexts, not contexts themselves.
  • Don't chase the type-generics cleanup on its own. Real but low-value; bundle it into P1/P2 or skip.
Bottom line. The action system is fundamentally sound; the friction is concentrated in resolution (smeared across three paths with inconsistent semantics) and the transform pipeline (three overlapping mechanisms with arbitrary differences). The single highest-value move is P1: a single resolution core — and the precedence work in PR #57 is the natural first slice of it, if we build that change as "introduce the resolver" rather than "add a dedup to the loop." If we only ever do one thing here, do that. The rest (canRun identity, pipeline unification) are worthwhile follow-ups, not prerequisites. No rewrite warranted.

Relationship to PR #57 (precedence)

PR #57 stays docs-only and ships the narrow precedence model. This critique argues its "shared comparator used by both paths" recommendation is the seed of P1 above — so when precedence is implemented, it's worth spending the marginal effort to land it as the resolution core (one resolve() the keyboard and dispatch paths share) rather than as a dedup bolted onto the install loop. Same user-visible outcome; pays down weakness 1 instead of deepening it. That's an implementation-time decision, recorded here so it isn't lost.
