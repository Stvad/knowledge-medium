Action system — architectural critique

Fresh-eyes review of src/shortcuts/ + the action facets — 2026-05-30

Written alongside docs/shortcut-context-precedence.html. That doc fixes one bug (double-fire) and adds one knob (precedence). This doc steps back and asks the broader question the precedence work surfaced: is the action system's shape right, or are we patching a model that wants restructuring? It is a critique + target shape, not an implementation plan. No code changes proposed here.

Bias check up front. This system is not in bad shape. The facet contribution model is genuinely good and most of the friction is concentrated in two places (resolution and the transform pipeline). The honest recommendation (bottom) is "two surgical refactors, leave the rest alone" — not a rewrite. Flagging this so the list of weaknesses below isn't read as a teardown.

The model in one screen

  • Actions (ActionConfig) are contributed via actionsFacet (core.ts:158). Each carries id, context, handler, optional defaultBinding, canRun, icon. ~23 files contribute.
  • Contexts (ActionContextConfig) via actionContextsFacet (core.ts:225); 6 call sites. A context is both an activation scope and a typed dependency bundle (via validateDependencies).
  • Activation is React-lifecycle-driven: blocks call useActionContextActivations (useActionContext.ts:17), which activate()s on mount/effect and deactivate()s on cleanup. The active set is an insertion-ordered Map (ActiveContexts.tsx). 27 surface-activation references.
  • Transform pipeline: actionOverridesFacet → actionDecoratorsFacet → keybindingOverridesFacet, composed in getEffectiveActions (effectiveActions.ts:66).
  • Dispatch happens three ways: keyboard (per-action window listeners in HotkeyReconciler), runActionById (module singleton), and useRunAction (React hook). The latter two share getActiveActionById; the keyboard path does not.

What's good — keep it

  • Facet-based contribution. Actions, contexts, overrides, decorators all flow through facets — plugins compose without editing core, no circular deps. This is the load-bearing strength; nothing below proposes touching it.
  • Decorators that wrap rather than fork. Extending an action's canRun/handler without owning the definition (e.g. srs-rescheduling/rescheduleDecorator.ts) is the right shape for cross-plugin extension.
  • Per-action install/uninstall. Each action owning its subscription makes activation changes cheap (no global teardown). Worth preserving even if the listener model changes.
  • Typed dependency map. ShortcutDependenciesMap keyed by context type is a clean way to give handlers the right inputs.

Weaknesses — grouped by theme

1. Resolution is implemented three four times and the keyboard one disagrees

"Given an active set, which action handles X?" is answered in four places:

  • runActionById → the module dispatcher installed by HotkeyReconciler (HotkeyReconciler.tsx:174) → getActiveActionById (last-activated-context wins).
  • useRunAction (runAction.ts:54) → also getActiveActionById, but re-implements the deps-lookup/throw around it.
  • The keyboard path → no resolution at all; every active action installs its own listener, so collisions double-fire (the bug the precedence doc fixes).
  • A plugin-level fork: swipe-quick-actions' runBlockAction (SwipeActionMenu.tsx:340) bypasses all of the above. It reaches into getEffectiveActions(runtime), hand-assembles deps ({block, uiStateBlock, …renderScopeId}), gates on canRun, and calls action.handler(deps, trigger) directly — because the dispatcher requires the action's context (e.g. NORMAL_MODE) to be active, and a swipe gesture is its own activation with no such context up. This is the worst of the four: a plugin re-implementing core dispatch by reaching into registry internals, and it's a preview of the input-unification problem (a non-keyboard trigger dispatching a block action with caller-held deps). See weakness 6 — the push-only supply model is why it had to fork.

There is no single "resolve event → action" function. The four paths have different semantics (double-fire vs single-winner vs hand-rolled) for what is conceptually one question. Precedence (priority/modal) then has to be taught to each path separately. This is the deepest structural issue and the one the precedence work half-forces us to confront.

2. Three overlapping transform mechanisms

ActionOverride, ActionDecorator, and keybindingOverridesFacet all rewrite the action list, with subtle, undocumented differences:

MechanismCan drop an action?Cross-action visibility?Ordering
ActionOverride.applyyes (→ null)no1st

keybindingOverridesFacetstrips chord, not actionyes (collision rule)3rd (final)
  • Override vs decorator asymmetry is arbitrary. Both are "(action) → action" transforms; only one may return null (types.ts:164 vs 170). A plugin that wants to conditionally remove an action must use override; one that wants to extend it uses decorator — but nothing names that distinction, so the choice is folklore.
  • Two ways to suppress, neither obvious. The user's "skip decorators" (no-op the handler) and a hypothetical override → null (drop the action) both "disable" an action, with totally different consequences (handler-still-listed-but-inert vs gone-from-palette). No guidance on which to reach for.
  • The '*' wildcard (effectiveActions.ts:27) is a back-door for set-wise transforms bolted onto a per-action API — used once, for the keybinding bridge, and easy to misuse.

3. canRun looks authoritative but is presentational

canRun reads like a guard, and types.ts:148–156 documents it as presentational-only — the keyboard and runActionById paths never call it. But the name fights the doc, and the contract is already violated in practice.

  • The "presentational-only" invariant is already false. swipe's runBlockAction (SwipeActionMenu.tsx:351) does if (action.canRun && !action.canRun(deps)) return false — using canRun as a real dispatch gate, not a display filter. So a plugin already needed a dispatch-level "should this run?" and the only thing on hand was the presentational predicate. The precedence doc leans on "canRun never gates dispatch"; that's a documented contract a shipping plugin breaks. Direct evidence the canDispatch concept is missing and being faked.
  • The user's skip-decorators exist for the same reason — no dispatch-level gate, so they fake one by mutating the handler. A real predicate-at-dispatch (the precedence doc's Option D / canDispatch) is the missing concept canRun appears to promise but doesn't deliver.
  • Fix: split into isVisible (UI, presentational) + canDispatch (real dispatch gate, declinable). Then swipe's canRun-as-gate, the skip-decorators, and the scattered guards all collapse onto the one honest mechanism.

4. Precedence is incidental because activation order is React-lifecycle order

The active set's order is whatever sequence useEffect activations fired in (ActiveContexts.tsx:86–97). "Latest-activated wins" therefore inherits mount/effect ordering as its precedence signal — which is why the new-surface case resolves backwards (surface mounts early, normal-mode activates on click). The precedence doc adds an explicit priority to escape this, which is the right patch — but the underlying smell is using a lifecycle artifact as a semantic ordering.

5. The runActionById module singleton

runAction.ts holds a module-level dispatcher var set by HotkeyReconciler on mount and nulled on unmount. It works, but: it's global mutable state, it throws if called before mount, and it couples "can I dispatch an action" to "is this one React component mounted." For a system whose whole point is decoupled contribution, the dispatch entry point being a singleton side-channel is a wart.

6. Context conflates scope and dependency-bundle — and dependency supply is the deeper issue

An ActionContextConfig (and the context identifier like 'normal-mode') does three jobs with different lifecycles: (a) an activation scope that gates install, (b) a precedence tier (modal, future priority), and (c) a typed dependency bundle (validateDependencies + ShortcutDependenciesMap). (a) and (b) legitimately co-vary. The smell is fusing (c) — per-activation runtime instance data — onto the same identifier as the static config.

This surfaces as type friction (the symptom) and a supply limitation (the real issue).

Symptom: the variance casts

  • ActiveContextsMap stores the widened base (BaseShortcutDependencies), because a heterogeneous "each key a different value type" map isn't expressible. active.get('normal-mode') returns the base; the narrow type is recovered only by trusting validateDependencies ran.
  • override.apply(action as never) (effectiveActions.ts:52,57) — a per-T transform applied over a flattened heterogeneous ActionConfig[]; casts to never to silence variance.
  • bivarianceHack on ActionHandler (types.ts:126) — exists purely to store ActionHandler<NormalMode> in a collection typed for the union.
  • Latent ceiling: deps in a type-keyed slot ⇒ you can't have two simultaneously-active instances of the same context with different deps (two panels both in normal-mode, different focused blocks). Re-activation overwrites. Harmless today; a real wall later.

The real issue: two axes — typing vs supply

"Handler-declared deps" (deps type rides with the handler, context goes back to pure scope+precedence) fixes typing. But supply is a separate axis, and it's where coherence lives:

  • Typing — who owns the dep type? Context (today) vs handler.
  • Supply — who produces the value, and when? Pushed at activation (today: closure into the active map) vs pulled/provided at dispatch.

Today is context-keyed + pushed. Because deps live in the active map, to dispatch an action you must be its active context — even if you already hold every dep it needs. That single constraint is what forces swipe's fork (weakness 1): the swipe handler has block and uiStateBlock in hand but no way to hand them to the dispatcher, so it bypasses dispatch entirely and calls the handler raw. swipe is hand-rolling the supply model the system doesn't offer.

The coherent target: scopes provide typed capabilities

Active scopes become providers of typed tokens; actions declare what they consume; at dispatch the resolver walks active scopes in precedence order to fill each needed token:

activate({ scope: NORMAL_MODE, provides: { [BlockToken]: () => block } })
{ needs: [BlockToken], handler: (deps /* {block} */, trigger) => … }
// resolve(): pick winner → fill needs from {active scopes} ∪ {caller/trigger-supplied}
//            a token with no provider ⇒ not dispatchable  ⇒ this IS canDispatch

This is, structurally, React-context-for-actions: a scoped DI container where the active scope stack is the provider chain and the comparator picks the winning provider. Payoffs beyond the casts:

  • Staleness + ref machinery go away. Getters read fresh at dispatch, so a scope activates once instead of re-activating on every dep-identity change — retiring much of HotkeyReconciler's activeRef/useLayoutEffect dance and the staleness window.
  • canDispatch unifies with availability — "can it run?" = "do its needed tokens resolve right now?". One mechanism replaces canRun-as-gate + skip-decorators + guards.
  • Caller/trigger-supplied tokens dissolve the swipe fork — swipe supplies BlockToken and dispatches through the one path; same seam input-unification needs for mouse/touch.
  • On the path to focus-tree, not a divergence. Providers-as-getters → providers-as-React-context; "walk active scopes for a token" → "read up the tree". The focus-tree model is this model where the provider chain is the component tree.

What helps vs. what's overkill

  • Cheap, do now (Phase 1): make resolve()'s deps-narrowing one explicit validated seam, and source deps from {active scopes} ∪ {caller-supplied} — not active-map-only. This alone lets swipe's fork collapse back into resolve() and gives input-unification its shape, without the full token registry.
  • Cheap, do at P2: design the unified transform type erased ((action) → action | null, no T) so the cast moves from the pipeline to the contributor's definition site (where T is known). Removes the as never.
  • Full provider/capability DI (the token registry + getters everywhere): the real fix for typing and supply, but a rework of all ~27 activation sites. Justified only once dep shapes grow faster than contexts — swipe + input are the first two data points; adopt when there's a third, or when the focus-tree prototype lands (they're the same model).

So #6 is mis-rated as pure cosmetics: the casts are cosmetic, but the supply model is the coherence question, and it already costs us one forked dispatch path. The P3 "model scope vs deps separately" is the endgame; the Phase-1 seam is the down payment that prevents building it twice.

The unifying idea: one resolution core

Most of the weaknesses above (1, 3, 4, partly 5) are the same shape: there is no single place that owns "active set + event → which action runs." Resolution is smeared across per-action listeners, two dispatch helpers, a presentational predicate, and an implicit lifecycle ordering. The high-leverage refactor is to introduce one resolver and route every path through it: 
resolve(activeSet, trigger): ActionConfig | null
  // trigger = actionId | normalized input descriptor
  //   descriptor: {kind:'key'|'mouse'|'touch'|…, modifiers, phase, role, …}
  //   NOT a raw KeyboardEvent — canonicalization (chord/descriptor) sits
  //   behind the shared canonicalizer so non-key triggers reuse the matcher
  // single comparator: (priority desc, activation-recency desc)
  // single place to add modal / declinable / future tiers
Keyboard dispatch, runActionById, and useRunAction all call it. Precedence, canonicalization, modal, and (if chosen) declinable fall-through become properties of the resolver, defined once, instead of being re-taught to each path. This is also exactly what the precedence doc's "shared comparator, used by both paths" bullet is reaching for — generalized.
Downstream consumer (2026-05-30): input unification. Spatial-navigation/selection work on another branch is hitting the input-side mirror of weakness 1 — mouse/click/gesture handling is smeared across ~5 bespoke facets (blockClickHandlersFacet, blockContentSurfacePropsFacet, blockGestureConflictsFacet, blockContentDecoratorsFacet, and a new blockSelectionClickDecoratorsFacet). The intent downstream is to route all input through this same resolve() — which is why the trigger arg is a normalized descriptor, not a KeyboardEvent, and why phase/role must be first-class binding fields (a double-click gesture binds at mousedown on the content node to beat native text selection). This turns two of the prioritized items below from "nice cleanup" into "unblocks active work": the resolver itself (a click handler needs declinable "not mine — fall through" dispatch, which is the same mechanism as a shortcut yielding a key), and the canDispatch split (the declarative gate that replaces both the imperative skip-decorators and the scattered if (isSelectionClick(event)) return guards). This composes with — does not compete with — the focus-derived activation-model exploration: resolve() consumes an active set; the activation model produces one. Either source of activeSet feeds the same resolver, so pursuing the resolver now doesn't foreclose that rework.

What's worth doing — prioritized

PriorityChangeWhy / payoff
P1 Single resolution core; route keyboard + both dispatch helpers through it (subsumes the precedence doc's comparator + the double-fire fix). Build it as the fire-time, declinable coordinator (precedence-doc Option D), with the trigger as a normalized descriptor — not a dedup bolted onto the install loop. Kills the keyboard/dispatch divergence (weakness 1), gives precedence/canonicalization/modal one home. The dedup-in-loop variant would leave the keyboard path as N independent window listeners that input-unification must then tear out and rebuild — so building it that way is negative work. Highest leverage.
P1 Decide canRun's identity: either rename to a display-only name, or split into isVisible (UI) + canDispatch (gate) Removes the reason skip-decorators exist (weakness 3). canDispatch is the declarative "not me — fall through" gate the input-unification work needs to replace its click guards; land it with the resolver. Cheap, clarifies a contract people already misuse.
P2 Unify the transform pipeline: one (action) → action | null transform type with a documented phase, fold decorator/override into it; keep keybinding-override as the one cross-action pass. Sequence close behind the resolver, not stacked into one PR. Ends the override/decorator asymmetry and the "which do I use?" folklore (weakness 2). Also the type that vim's skip-decorators and the downstream blockSelectionClickDecoratorsFacet both collapse into — so it directly retires input-side duplication too. Medium risk — touches every contributor, so worth a codemod.
P3 Replace the runActionById singleton with a runtime-scoped service obtained from context Removes global mutable state + mount-coupling (weakness 5). Mostly mechanical; low urgency.
P3 Tidy context generics (drop the as never/bivarianceHack by modeling scope vs deps separately, or accept and document them) Cosmetic/type-hygiene (weakness 6). Only worth it if touching the area anyway.

Explicitly NOT worth doing

  • Don't replace facets with a registry/event-bus. The contribution model is the part that works; churning it buys nothing.
  • Don't collapse contexts into a flat keymap. The scope concept earns its keep (modal, dependency typing, per-surface activation). The problem is resolution across contexts, not contexts themselves.
  • Don't chase the type-generics cleanup on its own. Real but low-value; bundle it into P1/P2 or skip.
Bottom line. The action system is fundamentally sound; the friction is concentrated in resolution (smeared across three paths with inconsistent semantics) and the transform pipeline (three overlapping mechanisms with arbitrary differences). The single highest-value move is P1: a single resolution core — and the precedence work in PR #57 is the natural first slice of it, if we build that change as "introduce the resolver" rather than "add a dedup to the loop." If we only ever do one thing here, do that. The rest (canRun identity, pipeline unification) are worthwhile follow-ups, not prerequisites. No rewrite warranted.

Relationship to PR #57 (precedence)

PR #57 stays docs-only and ships the narrow precedence model. This critique argues its "shared comparator used by both paths" recommendation is the seed of P1 above — so when precedence is implemented, it's worth spending the marginal effort to land it as the resolution core (one resolve() the keyboard and dispatch paths share) rather than as a dedup bolted onto the install loop. Same user-visible outcome; pays down weakness 1 instead of deepening it. That's an implementation-time decision, recorded here so it isn't lost.
