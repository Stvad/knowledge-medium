Runtime plugin enable/disable

Built-in extensions live in a hardcoded array. To toggle one you change code. This doc lays out a design for flipping them at runtime, modeled on how CodeMirror handles the same problem — compose by value, identify only what users toggle, dedup by reference, and stay out of the dependency-graph business.

Canonical companion: design.ts in this directory — types and signatures, typechecked against the real codebase. The prose describes; the .ts specifies. Verified along with every other docs/**/*.ts sketch by pnpm tsc --noEmit --project docs/tsconfig.json.

Design at a glance

  • Identity via a Compartment-style handle. Two constrained factories — systemToggle({id, name, ...}) for code-defined plugins, userExtensionToggle(block) for blocks — both return a Togglable whose .of(ext) wraps a subtree with a boundary marker. Extensions stay plain values; the handle is the identity.
  • Dependencies are inclusion + dedup. Plugin A needs B → A's extension array contains B's contributions. Reference-equality dedup makes overlap free. Plugin authors export both a bare contributions const (hard-dep inclusion) and a wrapped togglable (user-facing toggle).
  • Ordering already exists as precedence?: number on FacetContribution. Keep it.
  • Implication via enables. A facet contribution can drag siblings along — they exist iff it does.
  • Disable is a boundary-scoped filter. Skip the subtree at any boundary whose handle resolves to off via the overrides map + manifest defaults. One central store, system + user extensions both. The existing app-runtime-update handles rebuild + effect teardown.
  • Settings UI is a tree derived by walking the unfiltered extension tree — nested togglables render as nested checkboxes.
  • Recovery folds into existing ?safeMode — same URL flag also skips non-essential built-in extensions.

Where things stand today

  • Built-in extensions: each system extension wraps its own contributions with systemToggle(...).of(...). The central src/extensions/staticAppExtensions.ts file is just the ordered import list.
  • User extensions: blocks with a system:disabled property, filtered once at load in src/extensions/dynamicExtensions.ts:71. Toggling the property doesn't auto-rebuild — user must invoke "Reload extensions".
  • Hot rebuild exists: refreshAppRuntime() dispatches app-runtime-update; AppRuntimeProvider re-resolves and runs accumulated effect cleanups in reverse order (AppRuntimeProvider.tsx:145–176).
  • Safe mode exists: ?safeMode URL parameter skips all dynamic extensions (dynamicExtensions.ts:56–59). Doesn't yet skip built-in extensions.
  • Per-plugin prefs blocks exist via pluginPrefsExtension().

The CodeMirror model we're borrowing

The system is already CM-shaped (facets, contributions, extension values). Worth borrowing the rest:

ConcernCM answer
Plugin identityNone. Extensions are values, compared by reference.
Toggleable unitsCompartment — wraps a swappable subtree. The handle is the identity.
DependenciesNone declared. Include the dep in your own extension array. Dedup handles overlap.
OrderingPrec.highest/high/default/low/lowest. No DAG. (We use a numeric equivalent — same shape, no migration.)
Implication{value, enables: Extension} — a facet input can drag extensions along.

The five design pieces

1. Identity — Togglable handle, two constrained factories

A Compartment-style handle. The handle is what carries identity; calling .of(ext) returns a plain AppExtension tagged with the handle. The extension stays a value — no manifest object, no public type bifurcation. 

interface Togglable {
  readonly id: string
  readonly name: string
  readonly description?: string
  readonly essential?: boolean
  readonly defaultEnabled?: boolean   // undefined = true
  of(ext: AppExtension): AppExtension
}

const BOUNDARY = Symbol('togglable.boundary')

function markBoundary(handle: Togglable, ext: AppExtension): AppExtension {
  const wrapped: AppExtension = [ext]
  Object.defineProperty(wrapped, BOUNDARY, { value: handle, enumerable: false })
  return wrapped
}

function getBoundary(node: unknown): Togglable | undefined {
  if (!node || typeof node !== 'object') return undefined
  return (node as any)[BOUNDARY]
}

Crucially, handles are produced through two constrained factories, not one generic togglable(). Different contexts can express different fields; pushing the asymmetry into the type system means the prose doesn't have to keep explaining which field is honoured where. 

// Built-in extensions live in code and can express any field.
function systemToggle(opts: {
  id: string
  name: string
  description?: string
  essential?: boolean
  defaultEnabled?: boolean
}): Togglable { /* ... */ }

// User extensions live in blocks. id is forced to block.id (so the
// pre-compile skip can check by id); defaultEnabled is forced to false.
// Name + description come from block properties.
function userExtensionToggle(block: BlockData): Togglable { /* ... */ }

Usage inside an extension module, with staticAppExtensions.ts only listing exports:

export const dailyNotesPlugin = ({repo}) => systemToggle({
  id: 'system:daily-notes',
  name: 'Daily notes',
  description: 'Auto-create dated pages and the today shortcut.',
}).of([
  dailyNotesDataExtension,
  actionsFacet.of(openTodayAction({repo}), {source: 'daily-notes'}),
])

export const commandPalettePlugin = systemToggle({
  id: 'system:command-palette',
  name: 'Command palette',
  essential: true,
}).of([
  appMountsFacet.of(commandPaletteMount, {source: 'command-palette'}),
])

export const staticAppExtensions = ({repo}) => [
  kernelDataExtension,
  dailyNotesPlugin({repo}),
  commandPalettePlugin,
  /* ... */
]
Why two factories instead of one with prose-enforced rules? The earlier draft of this design had a single togglable({id, name, description, essential, defaultEnabled}) and a paragraph explaining that for user extensions id and defaultEnabled are silently overridden. Three review rounds caught issues stemming from that asymmetry leaking — author-chosen id bypassing the pre-compile skip, defaultEnabled: false being unhonorable without compiling, shell rows trying to read author metadata that wasn't there. Two factories make the constraints unrepresentable instead of relying on prose rules.

For user extensions, the dynamic loader wraps each block, but the wrapping can't happen after resolvedExt exists — that would mean the disabled block's module has already been imported and its top-level code has already run. Instead, the loader consults the overrides map before compileExtensionModule (the existing per-block skip point at dynamicExtensions.ts:72 generalizes): 

for (const block of extensionBlocks) {
  if (overrides.get(block.id) === false) {
    // Pre-compile skip. Disabled-shell rows know only block-level info;
    // userExtensionToggle with no authoredHints falls back to alias /
    // block-id link automatically.
    collected.push(userExtensionToggle(block).of([]))
    continue
  }
  // Compile + hint unwrap + validate are all per-block-fallible. Any
  // failure must still emit a shell so the row appears in settings
  // and the user can disable the broken extension. Errors continue to
  // flow through ExtensionLoadErrorStore for status-icon rendering.
  try {
    const {module} = await compileExtensionModule(block.content, block.id, cache)
    const exported = module.default as AppExtension
    const handle = userExtensionToggle(block)
    const wrapped = handle.of(exported)
    const validated = validateAndPrefix(wrapped, block.id)
    if (validated) collected.push(validated)
    else collected.push(userExtensionShellToggle(block).of([]))
  } catch (error) {
    errorReporter(block.id, error)
    collected.push(userExtensionShellToggle(block).of([]))
  }
}

The disabled-shell line, userExtensionToggle(block).of([]), is just the same block-local metadata factory wrapping an empty extension — a boundary that carries identity and no contributions. Discovery sees it and the settings UI renders the row; flipping it on triggers refreshAppRuntime(), which re-runs the loader and this time takes the compile branch.

Display name for user extensions. Extension blocks carry content (the TS/JS module source) and id, not a title field. Metadata lives in block properties:
  • Name comes from extension:name, then alias, then "Extension {id-link}".
  • Description comes from extension:description. Both compiled and shell rows use the same path, because neither needs module execution.
How authors provide name + description: block properties. Authors set extension:name and extension:description on the extension block. This keeps the settings surface informative even when the extension starts disabled and its source is not compiled.

Author usage: 
// Authors import each symbol directly from its real module; discover
// what lives where via describe-runtime's apiSurface / the curated
// catalog in src/extensions/apiCatalog.ts.
import { actionsFacet } from '@/extensions/core.js'

export default [
  actionsFacet.of(myAction, { source: 'my-ext' }),
]
Two toggle primitives remain — systemToggle and userExtensionToggle — each carrying exactly the fields its context can express.
Why user metadata lives outside executable code. The type-system constraint is backed by real reasons:
  • Pre-compile skip can only check by block.id — an author-chosen id wouldn't be visible until the module ran, defeating the skip.
  • User extensions now default disabled, so compiling just to read metadata would execute code before opt-in.
If author-controlled id / defaultEnabled ever becomes necessary, the design path is a designated pre-compile block property (e.g. extension:manifest). Not implemented now because no concrete demand.

Net result: system and user toggles remain indistinguishable to the resolver, and extension authors who care about UX can opt into nice metadata by wrapping their own export.

Why a handle instead of a manifest object ({id, name, extension})? The handle composes anywhere — wrap a single facet contribution, a sub-tree, or a whole plugin with equal ease. It keeps the public type as AppExtension, so nothing else in the resolver grows a new branch. And the same primitive serves both built-in extensions and user extensions, instead of bifurcating into two parallel mechanisms.

2. Dependencies — inclusion + dedup, bare vs. wrapped

A plugin that structurally needs another plugin's contributions includes them by value. The question is which form to include — and that choice is load-bearing. Each plugin exports two things: 

// inside src/plugins/references/index.ts
export const referencesContributions: AppExtension = [
  /* bare facet contributions */
]

export const referencesToggle = systemToggle({
  id: 'references',
  name: 'References',
})

export const references = referencesToggle.of(referencesContributions)

The two forms behave differently under disable:

Include thisWhen disabledUse when
references (wrapped) Subtree skipped at the boundary — contributions disappear. Top-level registration in staticAppExtensions. Or when another plugin wants to respect the user's toggle.
referencesContributions (bare) No boundary to skip — contributions always run. When you structurally cannot function without it. The disable toggle only removes the top-level binding; your bare inclusion still provides the contributions.

Backlinks's structural dependency on references:

// backlinks needs references regardless of the user's toggle state
export const backlinksContributions: AppExtension = [
  referencesContributions,    // bare — always on while backlinks is on
  backlinksViewContribution,
  backlinksActionsContribution,
]

export const backlinks = backlinksToggle.of(backlinksContributions)

The resolver dedups by reference equality during flatten. If referencesContributions is the same const reached via the top-level wrapped references path and via Backlinks's bare inclusion, the underlying facet contributions only land in the runtime once.

Important: the new collection logic (dedup, boundary-aware filter, enables recursion) must live in shared walk logic used by both resolveFacetRuntimeSync (the sync initial render at AppRuntimeProvider.tsx:52) and the async resolveFacetRuntime path that merges in dynamic extensions (AppRuntimeProvider.tsx:92). Putting dedup only in the sync path means behavior diverges the moment dynamic extensions load — which is exactly when the user-extension case kicks in.

Practically: mirror the existing sync/async pair in facet.ts:259/287. Both collectors apply isEnabled(handle, overrides) at each boundary (reading handle off the boundary symbol — no separate essential-id registry needed), recurse into enables, and Set-dedup. They differ only in how they handle function-valued AppExtensions: collectContributionsSync throws (initial render can't await, but the static tree has no functions); collectContributions awaits the function and walks its result (needed for dynamicExtensionsExtension, which is itself a function passed into the async path). See design.ts §4 for both signatures.

┌──────────────────────────┐ │ staticAppExtensions │ │ references │ ← wrapped (toggleable) │ backlinks │ ← wrapped (toggleable) └─────┬────────────────┬───┘ │ │ ▼ ▼ references backlinks boundary boundary └─ referencesContribs └─ referencesContribs ← bare, same ref └─ backlinksView └─ backlinksActions User disables "References": filter walks tree, hits references boundary → skip its subtree walks backlinks boundary (not disabled) → recurse sees referencesContribs by bare ref → include net: references contributions still active (via backlinks)
Why not a "Required by: Backlinks" UI lock that infers the relationship from the tree? Because the bare inclusion has no boundary or provenance to walk — it's just a value, identical to its source const. The honest answer is to let the plugin author's choice (bare vs wrapped) be the relationship, and trust the user to understand that disabling References while Backlinks runs is a no-op for those contributions. The settings UI surfaces what was toggled, not the resolved runtime state. Required-by belongs in dev tooling at most.
Convention to enforce: plugin entry points must be referentially stable. Top-level consts are fine. Factories called inline (dailyNotesPlugin({repo})) return fresh objects on each call and break dedup — hoist the result into a const built once at app setup, or memoize the factory on its args. Worth a startup assertion or lint.

3. Ordering — keep existing numeric precedence

This already exists. FacetContribution carries precedence?: number (src/extensions/facet.ts:13) and the runtime sorts by it (facet.ts:138): 

const values = contributions
  .toSorted((a, b) => (a.precedence ?? 0) - (b.precedence ?? 0))
  .map((contribution) => contribution.value as Input)

Several plugins already rely on this. CM uses named buckets (highest/high/default/low/lowest) instead, but there's no concrete reason to migrate — both work, and the numeric version is already in production. If arms-race / "what does 73 mean" pain ever shows up, a buckets layer can be added on top (a small map from name → number), but not now.

Net change in this design: nothing — just be aware precedence exists and use it when a contribution needs deterministic position relative to siblings (e.g. vim shortcuts intercepting before global ones).

4. Implication — enables

A facet contribution can carry an enables: AppExtension field. When the contribution survives resolution (and passes facet-level validation), its enables enters the tree too (and dedup handles overlap). It expresses "this contribution structurally implies these others, lifecycle-bound to it."

Requires a one-field extension to FacetContributionOptions (src/extensions/facet.ts:13): add enables?: AppExtension alongside the existing precedence? and source?. The collectors then recurse into enables after pushValidatedContribution accepts the parent — if the parent fails its facet's validate hook, its enabled extensions don't enter the runtime.

Example: video-player today vs. with enables

The video-player plugin (src/plugins/video-player/index.ts) is seven sibling contributions:

// today
export const videoPlayerPlugin: AppExtension = [
  propertySchemasFacet.of(videoPlayerViewProp,        {source: 'video-player'}),
  propertySchemasFacet.of(videoNotesPaneRatioProp,    {source: 'video-player'}),
  blockRenderersFacet.of({id: 'videoPlayer', renderer: VideoPlayerRenderer},
                                                      {source: 'video-player'}),
  blockLayoutFacet.of(videoPlayerLayoutContribution,  {source: 'video-player'}),
  markdownExtensionsFacet.of(videoPlayerMarkdownExtension, {source: 'video-player'}),
  shortcutSurfaceActivationsFacet.of(videoPlayerShortcutActivation,
                                                      {source: 'video-player'}),
  videoPlayerActionsExtension,   // adds actionContextsFacet + 2× actionsFacet
]

The implicit dependency chain isn't in the type system:

renderer mounts → creates block context with videoPlayerBlockId → shortcut activation fires when that context is present → VIDEO_PLAYER_CONTEXT becomes active → its action context + actions become reachable

With enables, the chain becomes the structure:

blockRenderersFacet.of({id: 'videoPlayer', renderer: VideoPlayerRenderer}, {
  source: 'video-player',
  enables: [
    blockLayoutFacet.of(videoPlayerLayoutContribution, {source: 'video-player'}),
    shortcutSurfaceActivationsFacet.of(videoPlayerShortcutActivation, {
      source: 'video-player',
      enables: [
        actionContextsFacet.of(videoPlayerActionContext, {source: 'video-player'}),
        ...videoPlayerActions.map(a => actionsFacet.of(a, {source: 'video-player'})),
      ],
    }),
  ],
})

propertySchemasFacet and markdownExtensionsFacet stay top-level — schemas register globally (useful for typing existing data even without rendering), and markdown parsing is independently meaningful.

For video-player specifically, enables is a clarity refactor, not a bug fix — disabling the whole plugin already drops everything atomically. The payoffs are (a) the file declares the dependency chain instead of leaving it implicit; (b) finer-grained toggles become possible (e.g. "video keyboard shortcuts" as a nested togglable that drops actions + context without touching the renderer); (c) third parties can attach actions to the same activation without checking whether video-player is loaded.

Where enables is not the right tool: two unconditionally-bundled things in the same plugin — just put them in the same array. enables is for fine-grained conditional bundling.

User-extension loader needs to (a) preserve the BOUNDARY symbol and (b) recurse into enables. validateAndPrefix in src/extensions/dynamicExtensions.ts:103 currently descends into arrays, functions, and FacetContribution leaves. Two changes needed: (a) the array branch at dynamicExtensions.ts:111 uses extension.map(...), which produces a fresh array — the non-enumerable BOUNDARY symbol on the togglable wrapper is silently dropped. Restore it after mapping: 
if (Array.isArray(extension)) {
  const mapped = extension.map((child) => validateAndPrefix(child, blockId))
  const boundary = (extension as any)[BOUNDARY]
  if (boundary) Object.defineProperty(mapped, BOUNDARY, { value: boundary })
  return mapped
}
Without this, dynamic extension toggles disappear after normalization — the resolver sees a plain array, no boundary, treats the contributions as bare. (b) the leaf branch (line 125) must also recurse into contribution.enables and source-prefix its nested contributions. Otherwise user-supplied enables contributions skip validation and lose block-source attribution.

5. Disable — boundary-scoped filter

The filter is a top-down walk fused with the flatten step. At each BOUNDARY in the tree, evaluate isEnabled(handle, overrides) using the handle now stored on the boundary itself: if it returns false, skip the subtree entirely. Otherwise recurse. Bare contributions (no boundary) are never skipped — they flow into the runtime.

┌──────────────────────────┐ │ Resolved extension tree │ └────────────┬─────────────┘ │ walk top-down │ at each BOUNDARY: if !isEnabled(handle, overrides), skip subtree │ collect contributions (Set-dedup by reference) ▼ ┌──────────────────────────┐ │ Sort by precedence │ └────────────┬─────────────┘ ▼ ┌──────────────────────────┐ │ FacetRuntime → React │ └──────────────────────────┘

Boundary-scoped means: disable only affects the specific wrapped binding it walks through. If references appears wrapped at top level (toggle on/off meaningful) and referencesContributions appears bare inside Backlinks (hard dep), disabling References at top level skips the wrapped subtree; the bare inclusion still flows through. The plugin author chose that asymmetry by picking bare vs wrapped at the inclusion site.

When the disable set changes, dispatch refreshAppRuntime(). The effect-cleanup chain at AppRuntimeProvider.tsx:145–176 already runs all accumulated teardowns in reverse order when runtime changes — no new teardown protocol needed for synchronous cleanups.

Async cleanup is fire-and-forget. AppRuntimeProvider.tsx:134–139 calls cleanup synchronously; if it returns a Promise, errors are caught but the Promise is not awaited before React starts the next effect pass. A plugin whose cleanup is async will see its next start() begin while the previous cleanup is still in flight. Plugin effects with exclusive resources (network sockets, file handles, singleton listeners) must either tolerate that overlap idempotently or coordinate via their own handshake — the framework does not gate restarts on cleanup completion.

Storage — central overrides map, both kinds of plugin

One block under the Preferences tree, owned by an extensions-settings meta-plugin (itself essential):

Preferences └── Extensions ← single block properties: overrides: { "vim-normal-mode": false, ← user turned off (default was on) "experimental-graph": true, ← user turned on (default was off) "<block-uuid>": false, ← user extension toggled off }

An overrides map, not a flat list of disabled ids. Each entry is an explicit user choice; absence means "use the manifest default." This lets the manifest carry defaultEnabled: false for experimental or opt-in features without needing a one-time migration to inject ids into anyone's storage, and lets a future shipped-default change apply cleanly to users who never had an opinion.

The effective enabled state is computed as:

function isEnabled(handle: Togglable, overrides: Map<string, boolean>): boolean {
  if (handle.essential) return true
  if (overrides.has(handle.id)) return overrides.get(handle.id)!
  return handle.defaultEnabled ?? true
}

Worked cases:

HandleOverrideEffective
defaultEnabled: true (omitted)—on
defaultEnabled: truefalseoff
defaultEnabled: false—off
defaultEnabled: falsetrueon
essential: trueanyon (override ignored)

UI toggle convention. The checkbox shows effective state. When the user clicks, compare the new state to handle.defaultEnabled ?? true: if the new state differs from that, write an override; if it matches, delete the override. The ?? true matters because defaultEnabled is typically omitted on default-on plugins (the most common case), so a raw === defaultEnabled comparison would write spurious {id: true} entries instead of clearing the override. Overrides only hold actual deviations from the manifest, so resetting a user to defaults is just overrides = {}.

One mechanism, one settings panel. The per-block system:disabled property on user extensions goes away — alpha, no back-compat shims; one-shot migration walks extension blocks, folds system:disabled === true into the new overrides map as {[block.id]: false}, drops the property.

When a user extension block is deleted, a small subscription prunes its id from the overrides map so we don't accumulate orphans.

First paint & the localStorage cache

The honest first-paint problem: staticAppExtensions resolves synchronously before PowerSync hydrates, so every system plugin's appEffectsFacet would start and every appMountsFacet would mount. ~One round-trip later we'd tear them down. Wasteful for tame plugins and actively wrong for plugins that touch network or write state on boot.

Solution: a narrowly-scoped ClientLocalSettings key per workspace holding the overrides map. Written whenever the synced block changes; read synchronously at boot. First paint reflects the user's most recent intent. Two-phase resolution then applies the canonical block once PowerSync is up and dispatches refreshAppRuntime() if it diverges from the cache.

The codebase convention is "read from DB; tolerate hydration flash; rebuild on arrival." It's correct to not cache DB content in localStorage as a rule. The narrow exception is justified here because this isn't DB content — it's intent about what runs before DB exists. The cost of being wrong (effects start then tear down) is real and observable, unlike most flash-of-empty-state cases.

Settings UI — tree from discovery

The togglable boundary already lives in the resolved extension tree (the BOUNDARY symbol on the wrapping array). Walk that tree before filtering to get the hierarchy. Walks are cheap and decoupled from filtering — discovery sees the whole tree including currently-off branches.

"Currently-off branches" needs a precise meaning for user extensions, since their modules can't be safely compiled just to populate the settings UI. The convention: a disabled user extension contributes a metadata-only userExtensionToggle(block).of([]) — the same factory used for enabled extensions, just with block-property metadata and an empty subtree. The shell appears in the discovered tree with the right handle, but no inner contributions to walk into. For nested togglables that live inside a disabled parent's real subtree (an enabled child of a disabled system plugin), they still appear via the BOUNDARY symbols in the parent's already-resolved tree — those didn't require module compilation.

Discovery must use the async variant. dynamicExtensionsExtension (dynamicExtensions.ts:51) is a top-level function on the async resolver path (AppRuntimeProvider.tsx:92). A sync walk that bails on functions (the natural first sketch) never sees user-extension boundaries. So the design ships a pair: discoverToggleTreeSync for callers that already hold a materialized tree, and discoverToggleTree (async) that awaits and walks function-valued extensions. AppRuntimeProvider calls the async variant once dynamic extensions finish resolving and caches the result alongside the runtime; the settings UI reads from the cached tree. 
type ToggleNode = { handle: Togglable; children: ToggleNode[] }

// Async variant — required when the tree contains
// dynamicExtensionsExtension (a function-valued AppExtension).
async function discoverToggleTree(
  tree: AppExtension,
  context: FacetResolveContext,
): Promise<ToggleNode[]> {
  const out: ToggleNode[] = []
  await walk(tree, out)
  return out

  async function walk(node: AppExtension, sink: ToggleNode[]) {
    if (!node) return
    if (typeof node === 'function') {
      try { await walk(await node(context), sink) }
      catch (e) { console.error('discoverToggleTree: failed to resolve', e) }
      return
    }
    const handle = getBoundary(node)
    if (handle) {
      const child: ToggleNode = { handle, children: [] }
      sink.push(child)
      for (const inner of node as readonly AppExtension[]) await walk(inner, child.children)
      return
    }
    if (Array.isArray(node)) {
      for (const inner of node) await walk(inner, sink)
      return
    }
    // FacetContribution: nothing to discover at the value itself, but if it
    // carries an `enables` payload, nested togglables can live inside it.
    if (isFacetContribution(node) && node.enables) await walk(node.enables, sink)
  }
}

The enables branch matters because a togglable can be introduced through an enables field, not just as a sibling in an array. For example, the nested "Video player keyboard shortcuts" toggle in the §4 refactor sits inside a shortcutSurfaceActivationsFacet contribution's enables. Without this branch, that nested handle never shows up in the settings tree.

UI shape:

┌──────────────────────────────────────────────────────────┐ │ Plugins │ ├──────────────────────────────────────────────────────────┤ │ [x] Video player │ │ [x] Video player keyboard shortcuts │ │ [x] Daily notes │ │ [ ] Vim normal mode │ │ [x] References │ │ [x] Backlinks │ │ [x] Command palette (essential) │ │ │ │ User extensions ─────────────────────────────────────── │ │ [x] My custom slash command │ │ [ ] Experimental block view │ └──────────────────────────────────────────────────────────┘

The tree only shows wrapped togglables — nested ones indent under their parent. Hard-dep inclusions of bare contributions don't appear in the tree because they don't carry boundaries; that's by design (the plugin author opted out of user-visible toggling for those).

Parent toggled off: render children greyed but preserve their checkbox state (matches typical nested-settings UIs; toggling parent off-then-on doesn't lose remembered child state). The overrides map holds explicit user choices keyed by id; child ids stay in the map regardless of parent state.

Recovery — extend ?safeMode

The app already has a safe-mode escape hatch: ?safeMode URL flag (src/App.tsx:319) routes through FacetResolveContext and causes dynamicExtensionsExtension to return an empty list (dynamicExtensions.ts:56–59). Extend it to also skip non-essential built-in extensions. One escape hatch, not two.

What happens when a user disables "Daily Notes"

  1. UI writes {"daily-notes": false} into the Extensions block's overrides map (since the default for daily-notes was on, the new effective state of off deviates from the default and is recorded). Also updates the ClientLocalSettings cache.
  2. The meta-plugin subscribes to the block. On change, dispatches refreshAppRuntime().
  3. AppRuntimeProvider's generation bumps → re-resolution runs.
  4. Tree walk: at the daily-notes boundary, isEnabled(handle, overrides) returns false → skip subtree.
  5. Flatten + dedup over what remains; sort by precedence; produce new runtime.
  6. Effect cleanup chain runs in reverse for the old runtime (AppRuntimeProvider.tsx:171–175); new effects start for the new runtime. React unmounts daily-notes components.

Cascade case: References disabled while Backlinks still on

The filter walks the tree and at the standalone references boundary skips the subtree — that wrapped binding's contributions don't enter the runtime. The walk then descends into backlinks (still enabled), which includes referencesContributions bare. Those contributions still flow into the runtime via this path.

Net runtime effect: references-related behavior continues to work, because Backlinks structurally requires it. The user's toggle removed the standalone binding (any contributions that references added beyond what Backlinks pulls in) but didn't reach the bare hard-dep inclusion. This asymmetry is the author's deliberate choice via bare vs wrapped at the include site; the UI surfaces what was toggled, not the resolved runtime state.

If a plugin author wants the disable toggle to also tear down dependents, they include the wrapped form (references) in dependents instead of the bare form. Then disabling References at top level skips both the top-level and the nested wrapped path.

Sharp edges

  • Reference stability. Factory-called plugins must be hoisted or memoized, otherwise dedup silently fails. Startup assertion would catch this.
  • Coarse effect rebuild + async cleanup overlap. Every effect restarts on every runtime change, and async cleanups are fired-and-forgotten (AppRuntimeProvider.tsx:134–139) so a new start() can begin while the previous cleanup is still in flight. Effects must be both idempotent and tolerate cleanup-in-progress when their start runs. Effects that hold exclusive resources need to coordinate via their own handshake.
  • Bare-vs-wrapped is load-bearing. Plugin authors who include another plugin must consciously choose: bare contributions ignore the user's toggle; wrapped togglables respect it. A lint that flags non-essential plugins importing other plugins' bare contributions (forcing an explicit comment or opt-in marker) would help avoid accidental hard-deps.
  • validateAndPrefix needs two fixes for this design. At src/extensions/dynamicExtensions.ts:103: (a) the array branch .map()s into a fresh array, which strips the non-enumerable BOUNDARY symbol — copy it onto the mapped array so dynamic extension toggles survive normalization; (b) the leaf branch must recurse into contribution.enables and source-prefix nested contributions, otherwise user-supplied enables contributions slip past validation and lose block-source attribution.
  • Bootstrap ordering. The meta-plugin owning the overrides map must be essential and must read the block from PowerSync after the kernel is up — same async second-pass pattern dynamicExtensionsExtension already uses.
  • Manifest id rename leaves stored prefs orphaned. Renaming a plugin's manifest id invalidates any override entry keyed by the old id. The startup warning (above) flags overrides whose id is missing from the discovered tree — non-fatal, so users keep working, but visible enough that maintainers can write a migration. Strict source/id equality is the wrong invariant: bare cross-plugin inclusions (per §2) and the block:${id} prefix that dynamicExtensions.ts:140 applies both legitimately produce sources that don't match the wrapping handle.
  • Conditional togglables. A togglable behind a (ctx) => ... appears in the discovered tree only when that condition holds. Probably fine — conditional togglables are conditional in the UI too — but worth knowing.
