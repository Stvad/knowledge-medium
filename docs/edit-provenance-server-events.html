Edit Provenance on the Server

Status: proposal / discussion artifact — not yet committed to. No code in this PR.

Edit provenance — why a block changed, via what command — lives only on the client today. This proposes shipping it to the server as an append-only, per-commit event log (the existing command_events + row_events tables), additive to the current block-state upload, so that other devices and (for plaintext workspaces) the server can read the granular history of how a graph came to be — without weakening the upload path that the 300K-row Roam import was tuned for. The log is a best-effort overlay, not a complete record: it covers only txs originated, post-feature, by still-live and un-trimmed devices (§8 spells out the holes). The complete, server-guaranteed history stays blocks_history (§7).

1. The problem

The motivating note: "I need to promote the edit provenance command events to the server somehow. We only know things on the client, but they do not know things on the server."

Concretely: when a block changes, the client knows the command behind it — a roam import, a migrate roam:location ref machine op, a genuine content edit, a move, an undo:*. The server knows none of this. It sees only the resulting block row. The Roam timestamp backfill made the gap concrete: to tell a genuine user edit from a machine op it had to run a row_events ⋈ command_events join against a client replica, because the server simply had no provenance to query (see scripts/roam-ts-backfill/README.md).

We want fairly granular provenance available off-device: durable against device loss, visible to other devices, and (where the workspace is not encrypted) queryable server-side.

2. What exists today

The client already has an event-sourcing substrate. Two local-only SQLite tables (src/data/internals/clientSchema.ts):

  • command_events — one row per repo.tx (the "command"): tx_id, scope, description, mutator_calls, user_id, workspace_id, source (uniformly 'user' for commands), created_at. Written at commit time in commitPipeline.ts. Covers local repo.tx only — sync-applied writes never go through repo.tx.
  • row_events — N rows per command, one per block change, each with full before_json / after_json and the same tx_id. Fires for both local and sync-applied writes; the source column ('user' vs 'sync') distinguishes them.

command_events (1) ⋈ row_events (N) on tx_id is the "one command responsible for a series of block edits, each edit with its own diff" model. It already exists — it just never leaves the device.

On the server side there is public.blocks_history (migration 20260522062437): a Postgres AFTER-trigger log of every block write that lands, at sync-checkpoint resolution, with semantic_op + changed_columns + actor + before/after diffs — but no command-level provenance (no description, scope, or mutators). §7 covers how the two relate.

3. Options considered

The discussion walked through three families before landing on the event log:

  • A — Stamp a coarse classification on the block row (e.g. last_edit_kind enum: user / import / migration / processor). Cheap, E2EE-safe, rides the existing blocks upload, lands in blocks_history for free. But lossy: latest-per-row only, no description, no per-edit detail. Retires the backfill's classifier debt but throws away the granularity we said we wanted.
  • B — Stamp a minimal command-event blob on the block row (encrypted JSON: reason, scope, tx id). Richer than A and still rides the blocks upload, but it forces an awkward fit: one command maps to many block edits, so attaching a single diff to a command — or duplicating the same command blob across every block it touched — is the wrong shape. Still latest-per-row.
  • C — Upload the event log (this proposal). Ship command_events and row_events as append-only logs, linked by tx_id. Native 1-command→N-edits structure; each edit keeps its own diff on its own row_event; nothing is collapsed.

4. Batching is not collapsing

A key clarification that unlocks the proposal. The current upload path (compactBlockCrudEntries + apply_block_patches in src/services/powersync.ts) does two independent things that are easy to conflate:

StepWhat it doesProperty
Batching Many operations in one network round trip (the RPC takes an array) Pure performance win — lossless
Collapsing Merge 50 edits to one row into one PATCH Discards intermediate edits — lossy

Collapsing was built for upload throughput (the 300K-row Roam import was slow), not to suppress server-side write/history amplification. Crucially, batching does not depend on collapsing: apply_block_patches already loops an array server-side, so 50 distinct edits can ship in roughly the same number of round trips as one collapsed edit.

What un-collapsing the blocks upload would cost is not round trips, but server writes:

  • ~50× the payload bytes (one round trip, bigger body),
  • 50 UPDATEs on the hot blocks row instead of 1 — each rewriting the row, churning indexes, and firing the blocks_history trigger,
  • under E2EE, 50 separate encryptions.
Insight. Replaying 50 UPDATEs on the live row is the wasteful part — the live row only ever needs its final state. The performant way to keep all 50 edits is appends, not updates: collapse the block state to one converged UPDATE, and append the 50 edits as immutable rows in one bulk INSERT. Appends don't rewrite a hot row or churn its indexes, and batch into a single statement. The event log is "performant without collapsing."

5. Proposal: upload the event log

Upload command_events and the locally-originated row_events as encrypted, append-only logs, linked by tx_id, additive to the existing block-state upload. Each stream then does the job it is cheap at:

  • Block state — keep collapsing + down-sync exactly as today. One converged PATCH per row; blocks stays the fast, LWW state-of-the-world. Collapse goes back to doing only the throughput job it was built for.
  • Event log — uncollapsed, append-batched. N event rows in one multi-row INSERT; append-only means no per-id merge, no ordering hazard, idempotent retry on the primary key. Granular by construction.

This is event sourcing in the honest sense: events are the record; blocks is a materialized projection of them (which it already is on the client — the sync observer materializes blocks from the stream).

Two rules that make it correct

  1. Upload only locally-originated row_events (source ≠ 'sync' / tx_id IS NOT NULL). row_events logs both local and sync-applied writes; without this filter every client would re-upload every edit it received via sync, and the server would double-count. The filter partitions ownership: each client uploads exactly its own commands + edits. The union across devices is not a complete history, though — only sync-applied rows are excluded, and a tx's events exist on exactly one device (the originator). Members who joined after the fact, pre-feature history, and trimmed/wiped originators all leave permanent holes; see §8.
  2. Encrypt the content-bearing fields; keep identifiers clear. before_json / after_json and description / mutator_calls carry workspace content (page names, property values), so on E2EE workspaces they ship as enc:v1: ciphertext. But the routing/RLS identifiers — client_id, tx_id, block_id, workspace_id, kind, timestamps — must stay plaintext: RLS filters on a clear workspace_id (mirroring blocks_history), so sealing the whole row would make the read policy match zero rows. Note workspace_id then lives both in the clear top-level column and inside the (encrypted) *_json blob, which embeds the full row — acceptable (it's an identifier, not content), but it must be called out. Consequence: on E2EE workspaces the server stores the content opaquely — only devices can read it (device-loss recovery, cross-device history). On plaintext workspaces the server can read and query it directly (e.g. re-run the backfill classifier server-side, never against a client replica again).
    Open: the existing enc:v1: primitive is per-column AEAD with AAD bound to (blockId, workspaceId, columnName) — it has no shape for a single serialized event blob (and a hard-delete event references a block_id whose block is gone). The blob AAD must be defined fresh; it cannot reuse the block-column scheme.

6. Shape sketch

Two server tables mirroring the client tables, keyed for idempotent cross-client append. Not synced down via PowerSync (like blocks_history, read via RPC) — syncing every event to every device would balloon buckets for no benefit. Illustrative, not final: 

-- one row per command
command_events_log (
  client_id     text,        -- durable per-device id — MUST be minted/persisted (see warn)
  tx_id         text,        -- command id (text form, as on the client)
  workspace_id  text,        -- nullable: zero-write/no-op txs pin no workspace (see note)
  user_id       text not null,
  scope         text not null,
  description    text,        -- enc:v1: on e2ee workspaces
  mutator_calls  text,        -- enc:v1: on e2ee workspaces
  created_at    bigint not null,
  primary key (client_id, tx_id)
)

-- N rows per command, one per block edit
row_events_log (
  client_id     text,
  install_id    text not null,  -- per-install nonce; with seq forms the reset-proof event id
  seq           bigint not null,-- the client row_events.id: per-install append order (sortable)
  tx_id         text not null,   -- links to command_events_log
  block_id      text not null,
  workspace_id  text not null,
  kind          text not null,   -- create | update | soft-delete | delete
  before_json   text,            -- enc:v1: on e2ee workspaces
  after_json    text,            -- enc:v1: on e2ee workspaces
  created_at    bigint not null,
  primary key (client_id, install_id, seq)
)

-- BOTH tables (public schema): default-deny RLS + revoke, per the repo rule.
-- A CREATE TABLE in public is world-CRUD via the anon key until locked down.
alter table … enable row level security;       -- no policy = default-deny writes
revoke all on … from anon, authenticated;
grant select on … to authenticated;            -- reads still gated by the policy below
create policy …_read on … for select
  using (private.is_workspace_member(workspace_id, (auth.uid())::text));

Re-upload after a retry is a no-op (INSERT … ON CONFLICT DO NOTHING) — events are immutable, so there is never an update to reconcile. RLS mirrors blocks_history: read iff current workspace member (a removed member loses read access to history they authored — same as blocks_history; provenance is gated on membership, not authorship); writes only via the upload path.

The append RPC needs an explicit definer-with-checks contract. The grants above give no INSERT to authenticated and there is no write policy (by design). So a SECURITY INVOKER append function would be unable to insert at all — every event upload rejected. The fix is SECURITY DEFINER (as create_workspace and the blocks_record_history trigger already are), but DEFINER bypasses RLS — so the function must itself verify, for every appended row, that auth.uid() is a writer of that row's workspace_id. Use private.is_workspace_writer (owner/editor), the same helper the blocks_write policy uses — not private.is_workspace_member, which also matches read-only viewers and would let a viewer append forged provenance for writes they could never make. Reject the whole batch otherwise. Without that per-row writer check, DEFINER opens cross-workspace / viewer log writes; with INVOKER, nothing inserts. Spell this out in the implementation, plus set search_path = '' and schema-qualified names, matching the existing DEFINER functions.
Keying & ordering primitives the sketch needs but the codebase does not have yet.
  • client_id does not exist as a sync concept. There is no durable per-device/per-install id in the sync layer. The tempting in-scope values are both wrong: PowerSync's CrudEntry.clientId is the local ps_crud row number (per-entry, resets on reinstall), and auth.uid() is per-user (two devices of one user would collide). A durable client_id (a UUID persisted in client_schema_state / IndexedDB) must be minted before this is buildable. This is a prerequisite, not an open question.
  • row_events.id alone can't be the event key — on two counts. (1) Not reset-proof: it is INTEGER PRIMARY KEY AUTOINCREMENT; a local DB wipe / reinstall / OPFS reset / lockAndWipe recreates the table and restarts the counter at 1 (resetTestDb even clears sqlite_sequence), so a wiped-then-resynced device re-mints 1,2,3… that collide with already-uploaded ids and ON CONFLICT DO NOTHING silently drops genuine new events. (2) Content hashes are insufficient too: a tempting hash(client_id, tx_id, block_id, kind) collides on a real case — a single repo.tx can write the same block more than once (e.g. setProperty then move), and blocks_row_event_update (clientSchema.ts:436) appends a separate row_event on every UPDATE, so those events share all four fields and the later ones get dropped. The key needs a genuine per-event discriminator.
    Workable shape: identify an event by (install_id, seq) where install_id is a per-install UUID nonce persisted in client_schema_state and regenerated on every wipe, and seq is the client row_events.id. That makes the per-install autoincrement globally unique and reset-proof (post-wipe events carry a fresh install_id, so they can never collide with pre-wipe ids — no silent drop), while seq being distinct per event handles repeated writes to one block in one tx. (install_id is a per-install nonce, distinct from the durable per-device client_id above, which is for "originating device" grouping; different ids, different jobs.)
  • Replay order: within an install, exact; across installs, only approximate. seq totally orders one install's events (it's the local append order, ground truth for that device's contribution), so per-install replay — including repeated writes to one block in one tx — is exact. But seq must not be the leading sort key across installs: with install_id first, two installs that edited a block in interleaved order (A1, B1, A2) would replay grouped (A1, A2, B1) — wrong, and a tiebreak applied after install_id can't repair it. The honest position is that this design has no reliable global edit order for a block touched by multiple installs: wall-clock created_at is subject to cross-device skew, and there is no server sequencer or vector clock here (same fundamental limit as blocks' LWW). So the rule is: order primarily by created_at (best-effort global time), then by (install_id, seq) only to break ties and to order within one install. Exact cross-device causal replay would require a server-assigned append order or a per-event vector clock — out of scope, and called out here so consumers don't assume the chain is authoritative across installs.

workspace_id is nullable on command_events_log, matching the client: commitPipeline.ts writes a command_events row for every repo.tx, with workspace_id left NULL for zero-write / no-op transactions (no workspace gets pinned). A NOT NULL column would reject those on upload. They carry no provenance worth shipping anyway — no associated row_events (a no-op-write tx may still have non-empty mutator_calls, so the filter keys on row_events, not on mutator_calls) — so the upload filter also drops command events that produced no uploaded row_events, which incidentally excludes the null-workspace no-ops. (The RLS workspace-membership read policy then only has to reason about rows that do carry a workspace.)

Upload mechanics reuse the existing machinery: an upload-routing trigger emits an envelope per event row (a new type beyond 'blocks'), the connector ships them via a batched append RPC, and the encrypt-on-upload hook covers the content-bearing columns. The block-state upload semantics are preserved — but "additive" is true at the design level, not the code level: the shared connector (uploadTransactionsWithFallback, compactBlockCrudEntries, the batch-encrypt guard, and the rejection-recording path) is restructured to demux by type (see warn below), so a bug there can regress the very block path §4 protects. The blast radius is the connector, not just new tables.

Connector must demux by type first. The current upload loop feeds every ps_crud entry into compactBlockCrudEntries, which throws on any entry.table !== 'blocks' — before the per-tx fallback can apply anything. So a transaction that mixes a block edit with its provenance events would jam the queue and retry forever. The connector has to partition crud entries by type up front and route command/row-event entries down their own append + encrypt path, around block compaction. This is a precondition of the new type, not an afterthought.

7. Relationship to blocks_history

This complements blocks_history; it does not replace it. They look redundant but answer different questions under different trust models:

blocks_historyuploaded event log
Who writes itServer trigger, from the write that actually landsClient uploads it as a parallel stream
ResolutionSync-checkpoint (post-collapse)Per-commit, uncollapsed
ProvenanceNone (semantic_op + changed_columns)Rich (command description, scope, mutators)
ReadabilityServer-readable for plaintext workspacesEncrypted/opaque by design on E2EE
CompletenessAnything that lands in blocks after the trigger shipped, independent of clientsOnly as complete as clients upload + retention keeps

The load-bearing row is the last one. blocks_history is written server-side from the authoritative landed state, so for any write after the trigger landed it can't be missing history, and it is the declared substrate for server-side point-in-time recovery. The event log is a client assertion on a separate upload path: a quarantined event upload, a device that dies before flushing, or a retention cap can each leave gaps for blocks that are definitely on the server. So blocks_history stays as the server-trustworthy backstop; the event log layers rich, commit-grain provenance on top.

blocks_history is not retroactive. Migration 20260522062437 creates the table + trigger but does not backfill existing blocks rows, and maintenance jobs can disable blocks_record_history_trg for bulk work. So blocks created before the trigger (or written while it was disabled) have no history — "complete backstop" holds only for writes recorded after the trigger. Don't let the event-log design treat a pre-trigger block's absence of history as "safe"; closing that gap needs a one-time backfill/anchor (e.g. seed a synthetic create from current row state) that is out of scope here but must not be assumed away.
Even on E2EE, don't trim blocks_history. An earlier draft argued that since blocks_history stores enc:v1: ciphertext on E2EE workspaces it loses its "server-readable" edge and the event log "strictly dominates" it — so it could be skipped/trimmed for E2EE. That's wrong, and it contradicts this doc's own thesis. Readability is only one of blocks_history's advantages; the load-bearing ones survive encryption: it is server-triggered, atomic with the landed write, client-independent, and complete for post-trigger writes. The event log has none of those — it's client-uploaded and best-effort. So on exactly the failure modes this doc keeps flagging (origin device dies / is wiped / its event upload is quarantined), blocks_history is the only surviving record that the write happened, even if the diff is opaque. The event log adds richer (encrypted) provenance on E2EE; it does not replace the backstop. No skip/trim — encrypted or not.

8. Costs & open questions

  • Volume / retention is the real price — and bigger than §4's "appends are cheap" lets on. Appends are cheap on write (no hot-row rewrite); they are not cheap on storage. Concretely (per docs/follow-ups.md): the 300K Roam import wrote ~304K row_events ≈ 262 MB locally. Shipping the locally-originated log means hundreds of MB to the server per importing device, and on E2EE workspaces the blobs are opaque ciphertext that can never be GC'd by content. So the §4 performance framing and this cost must be read together: cheap to append, expensive to keep. A retention stance is a prerequisite, not a someday — the real discussion lives in docs/follow-ups.md ("Periodic row_events trim"; there is no docs/row-events-retention.md despite dangling references to it). Open: keep-forever vs. cap/TTL (and, if capped, keep command_events_log longer than row_events_log, since the per-edit diffs dominate volume?).
  • Conflict with the planned import path. docs/follow-ups.md also proposes an 'import' source that skips both row_events and ps_crud precisely to avoid that ~262 MB bloat. If that lands, imported blocks emit no events to upload — so this design either forces import to keep emitting events (re-bloating the path the codebase was tuned to de-bloat) or accepts that imports have zero server provenance (compounding the completeness holes below). Pick one explicitly.
  • "Complete history" has structural holes, not just transient ones. The union-of-devices is best-effort: (a) a member who joins an existing workspace syncs every historical block as source='sync' → filtered → never uploaded, and the originators may have left/wiped/been-capped, so that history is permanently absent; (b) pre-feature blocks have no upload-routed events and nothing back-fills them; (c) a device's local row_events are subject to the existing trim, so once trimmed the only copy of that provenance is gone. blocks_history (§7) is the complete record precisely because it is server-written from landed state; this log is an overlay. Every "complete" claim elsewhere is downgraded to "complete for post-feature txs from still-live, un-trimmed originators".
  • Server stays blind on E2EE. The server can't classify or query encrypted events; only devices benefit. Accepted — it matches the E2EE design's "server sees only ciphertext" posture. Plaintext workspaces get full server-side queryability.
  • Skipped deterministic creates desync the log. A local create row_event is recorded before upload, but applyBlockCreates uses insert-or-skip (ignoreDuplicates: true) for deterministic-id collisions — the fresh-client user-prefs / user-page / ui-state bootstrap. There the block create is skipped server-side (the row already exists) while the append log would still claim a kind='create' that never landed. So "the server log is the complete history of how the server graph came to be" needs a qualifier: these no-op creates must be filtered (e.g. don't ship create events for deterministic-id rows, or reconcile against the upload result), or the log records creates the server never performed.
  • Partial failure splits a command from its edits — and orphans are unavoidable. The command_event and its row_events are separate queue entries, so a quarantine can land one without the other; there is no server FK to prevent it. Worse, the two filters above (drop no-row-events commands; drop skipped-deterministic creates) mean a command with a mix of a filtered create + a real update ships the update but not the genesis create: the per-block log then starts mid-stream with an update whose before_json references a state no prior event explains, and "each edit keeps its own diff" (§3) can no longer be replayed from the log. Consumers must tolerate orphans on both sides, and the doc must define replay semantics when the genesis event is missing (never filter creates? anchor on the surviving event's before_json? reconcile against the block upload result?).
  • Append events only after the block write succeeds. The event rows ship via a separate append RPC from the block apply path, and the two are not atomic. If the block write is permanently rejected/quarantined (FK violation, missing server row) but the event append already succeeded, the server keeps — idempotently — provenance for a block change that never landed: a false record, worse than a gap. So the ordering must be specified: gate the event append on the corresponding block upload succeeding for that tx (block-success-before-event-append), or use a combined rollback boundary. That makes the only possible divergence "block landed, events missing" — a tolerable gap that blocks_history (§7) still covers — never "events claim a change that didn't happen". Note this composes with the replay-ordering caveat above: gating on block success doesn't give a global order, it only prevents false provenance.
  • Key/clock primitives are missing (see §6 warn). A durable per-device client_id does not exist today, and row_events.id (AUTOINCREMENT) is not reset-proof — so the idempotency key needs both invented before this is buildable. Treat as prerequisites, not open questions.
  • Three history-ish things coexist (client row_events, server blocks_history, server event log). That's the honest cost of keeping a server-guaranteed backstop separate from an encrypted commit-grain provenance log — two contradictory requirements that shouldn't be merged into one table.

9. Rejected / deferred alternatives

Un-collapse the blocks upload itself. Replays 50 UPDATEs on the hot row to manufacture history — maximal write/index/trigger amplification for a row that ends in one state. The append-only event log gets the same granularity far cheaper (§4).
Sync the event log down to every device via PowerSync. Balloons bucket sizes for data almost no device queries. Read via RPC instead, like blocks_history.
Replace blocks_history with the event log. Loses the server-guaranteed, client-independent backstop and the plaintext server-side PITR substrate (§7).
Deferred: full event-sourcing (events as the sole source of truth, server materializes blocks from them). A much larger rewrite (server-side materialization, blocks as projection-only). This proposal is deliberately additive: blocks stays the convergent state path; the event log layers on top.

10. What this PR is (and isn't)

This PR adds only this design document. No schema, migration, connector, or encryption changes. It captures the proposal and its trade-offs for review before any code is written. What gates an implementation:

  1. Decisions: dual-path confirmed? (blocks stays + event log is additive — the recommendation); retention stance (keep-forever vs. cap/TTL); and whether imports emit provenance or are accepted as having none (§8).
  2. Prerequisites that don't exist yet (§6 warn): a durable per-device client_id, and a reset-proof per-event key (not row_events.id). Neither is a "later" — the keying is unbuildable without them.
  3. Specs to write: the connector demux + its effect on the block path (§6); the AAD for an opaque event blob, which can't reuse the per-column scheme (§5); and which columns stay plaintext for RLS (§5).

The honest summary: the direction (append-only event log, additive to collapsed block state, complementing blocks_history) holds up; the completeness claim and the keying were over-stated in earlier drafts and are corrected above. This is a sound overlay for post-feature provenance, not a complete server-side history.
