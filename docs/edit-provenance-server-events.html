Edit Provenance on the Server

Status: proposal / discussion artifact — not yet committed to. No code in this PR.

Edit provenance — why a block changed, via what command — lives only on the client today. The question is how to get it off-device. This doc frames that as one decision (does the server sequence the log, or not?) with three coherent answers, and recommends server-as-log: ship every event through a cheap append path, let the server assign the order, and keep the collapsed block-state upload exactly as is. That reconciles canonical event sourcing with the upload performance the 300K-row Roam import was tuned for — and dissolves the cluster of problems an earlier "client uploads its own ordered log" sketch kept generating.

1. The problem

The motivating note: "I need to promote the edit provenance command events to the server somehow. We only know things on the client, but they do not know things on the server."

When a block changes, the client knows the command behind it — a roam import, a migrate roam:location ref machine op, a genuine content edit, a move, an undo:*. The server knows none of it; it sees only the resulting block row. The Roam timestamp backfill made the gap concrete: to tell a genuine user edit from a machine op it had to join row_events ⋈ command_events against a client replica, because the server had no provenance to query (scripts/roam-ts-backfill/README.md).

Goal: fairly granular provenance available off-device — durable against device loss, visible to other devices, and (where the workspace is not encrypted) queryable server-side. We do not want to lose machine-edit fidelity wholesale; machine edits are edits.

2. What exists today

  • command_events (client-only) — one row per repo.tx: tx_id, scope, description, mutator_calls, user_id, workspace_id (nullable — NULL on zero-write txs), source (uniformly 'user'), created_at. Local repo.tx only — sync-applied writes never go through repo.tx.
  • row_events (client-only) — N rows per command, one per block change, with full before_json / after_json and the same tx_id. Fires for both local and sync-applied writes; source ('user' vs 'sync') distinguishes them, and sync-applied rows carry tx_id = NULL.
  • blocks_history (server, migration 20260522062437) — a Postgres AFTER-trigger log of every block write that lands, at sync-checkpoint resolution, with semantic_op + changed_columns + actor + before/after diffs, but no command-level provenance.

command_events (1) ⋈ row_events (N) on tx_id is already the "one command, a series of block edits, each edit with its own diff" model — an event-sourcing substrate that simply never leaves the device.

3. The crux: a sequencer orders only what it receives

Canonical event sourcing leans on a sequencer — a single authority that assigns each event a global order and identity (Kafka offsets, EventStoreDB's $all, Replicache's server-assigned version). That is the property that makes ordering, identity, and idempotency easy.

A first sketch of this doc had the client own the order: upload command_events + row_events keyed by a client-assigned (install_id, seq), with the server as dumb storage. That shape generated a long tail of problems — colliding ids across resets, no sortable order, wrong cross-install replay, clock-skew inversions, append/authz/atomicity gaps. Every one traces to a single mistake: asking N peer clients to jointly own a global log without a sequencer. That's the "incoherent middle" — it takes on the peer model's hard problems (identity, ordering, completeness) and the server model's (atomicity, authz) at once, and solves neither the canonical way.

The honest objection to "just make the server the sequencer" is: the server can only order what it receives, and we deliberately collapse writes before upload (built for the 300K-import performance). If the client merges 50 edits into one block PATCH, the server never sees the other 49 to sequence them. §4 shows why that objection is narrower than it looks.

4. Batching ≠ collapsing — the transport-cost split

Two independent things the current upload path (compactBlockCrudEntries + apply_block_patches, src/services/powersync.ts) does together:

StepWhat it doesProperty
BatchingMany ops in one round trip (the RPC takes an array)Performance — lossless
CollapsingMerge 50 edits to one row into one PATCHDiscards intermediate edits — lossy

Collapsing was built for upload throughput (the import was slow), and its cost target is block-table op-count and per-row UPDATE amplification. Critically, "we don't ship every change" is a property of the block-state transport, not a law about events:

  • Block state is expensive un-collapsed — each version = a server UPDATE (row rewrite, index churn, history-trigger fire). Keep collapsing it; the live row only needs its final value.
  • Events are cheap in full — each = an immutable INSERT. A bulk append of N rows is one statement: no row rewrite, no hot-row contention, batchable into a few round trips. The amplification collapse protects against isn't incurred by appends.
So you can ship every change — just not as block UPDATEs. Send the full event stream via the cheap append path and let the server assign the order on receipt. Now the server genuinely is the sequencer for the fine-grained stream (canonical ordering/identity/atomicity return), and the import stays fast because block state is still collapsed. The two streams diverge exactly where their costs diverge. The only residual cost is storage/volume, not latency — addressed in §7.

5. Three coherent models

The design space has three internally-consistent points. The decision reduces to: do you want fine-grained provenance server-side, and will you pay storage for it?

ModelWho sequencesSolves cheaplyGives up
A. Server-as-log (recommended) Server, on receipt Ordering, identity, idempotency, atomicity, authz Storage cost; needs the append/demux path built
B. Server-captured checkpoints Server, but only sees collapsed state Everything in A plus zero new transport — stamp a descriptor on the block PATCH; blocks_history records it Per-commit fidelity (checkpoint-grain only — the lossy option)
C. Peer CRDT No one — logical clocks (HLC/Lamport) at edit time Offline-first, no central authority Server-queryability + complete server history; needs full CRDT machinery (a big philosophical shift from today's LWW)

The earlier sketch was none of these — it was A's transport with the client (not server) sequencing. This doc recommends A: our stack already runs server-authoritative Postgres (PowerSync) + a server-trigger log (blocks_history), so A is a small step; C is a large one. B is the fallback if "just enough to classify edits" turns out to be the real requirement.

6. Recommended: server-as-log, full human fidelity

Ship every event through a cheap, append-only path, additive to the existing collapsed block-state upload. Each stream does the job it is cheap at: block state collapses for fast convergent state; events append for lossless granular history. The defining properties:

  1. The server sequences — but the cursor must be commit-ordered, not raw BIGSERIAL. On receipt the server assigns the authoritative global order for replay/pagination. The naive choice — a BIGSERIAL read order — is unsafe as a tail cursor: serial values are allocated before the inserting transaction commits, so under concurrent appends a higher seq can become visible before a lower one commits, and a reader tailing server_seq > cursor advances past the gap and never sees the late-committing lower seq (the classic CDC visibility gap). So the frontier must be commit-ordered: a commit timestamp (track_commit_timestamp), a logical-replication LSN, or appends serialized through a sequence assigned at commit — or a watermark cursor that refuses to advance past a non-contiguous (uncommitted) gap. The client's (install_id, seq) is carried only for idempotent dedup (INSERT … ON CONFLICT DO NOTHING), never as the global order. This is the line between "client log dumped into a table" and server-as-log.
  2. Ordering primitive: monotonic, not wall-clock — and not user_updated_at. seq (the client row_events.id) totally orders one install's events — so per-install replay is exact, including repeated writes to one block in a tx. The global / cross-install order is the server's commit-ordered frontier (above). Do not reach for the updated_at/user_updated_at pair as an ordering key: only updated_at is monotonic (and it's a per-row-version value, not per-event), while user_updated_at is display/provenance time that is intentionally non-monotonic — sourceTimestamps imports historical Roam edit times and {skipMetadata} leaves it untouched, so import/restore/bookkeeping events would sort wrong (§9). If a client-side causal hint is ever wanted beyond seq + server commit-order, it must be a dedicated monotonic event stamp (a logical clock), not a reused row column. Genuinely concurrent cross-device edits remain concurrent — no key invents a true order for them.
  3. Upload only locally-originated events (source ≠ 'sync' / tx_id IS NOT NULL). Otherwise every client re-uploads edits it merely received via sync and the server double-counts. Each client owns exactly its own events; the union is the server log.
  4. Atomicity: append after block success. Events ride a separate route from the block apply, so gate the event append on the corresponding block upload succeeding for that tx (or a combined boundary). Then the only possible divergence is "block landed, events missing" — a gap blocks_history still covers — never a false record of a change that never landed.
  5. Authorization at the append endpoint. The append RPC runs SECURITY DEFINER (the table is default-deny, like blocks_history) and must verify, per appended row, that auth.uid() is a writer of that row's workspace_id via private.is_workspace_writer (owner/editor) — not is_workspace_member, which would let a read-only viewer append forged provenance. set search_path = '' + schema-qualified names, matching the existing definer functions.
  6. Encrypt content, keep identifiers clear. before_json / after_json / description / mutator_calls carry workspace content, so on E2EE workspaces they ship as enc:v1:. Routing/RLS identifiers (install_id, seq, tx_id, block_id, workspace_id, kind, stamps) stay plaintext — RLS filters on a clear workspace_id. Note the existing enc:v1: primitive is per-column AEAD with AAD bound to (blockId, workspaceId, columnName); an opaque event blob needs its AAD defined fresh (it cannot reuse that binding, and a hard-delete event references a block_id whose block is gone).

Shape sketch (illustrative)

Two server tables, not synced down (read via RPC like blocks_history — syncing every event to every device would balloon buckets). command_events_log carries one row per command; row_events_log one per edit, linked by tx_id. The server adds its own server_seq on insert. 

command_events_log (
  install_id   text not null,  -- per-install nonce, regenerated on wipe (NOT a durable device id)
  tx_id        text not null,
  workspace_id text,           -- nullable: zero-write/no-op txs pin no workspace
  user_id      text not null,
  scope        text not null,
  machine_origin text,         -- explicit processor/import marker (NOT source/scope); gates compaction (§7)
  description   text,          -- enc:v1: on e2ee workspaces
  mutator_calls text,          -- enc:v1: on e2ee workspaces
  commit_seq   bigint,         -- server commit-ordered frontier for cursors (LSN / commit-ts; NOT bigserial)
  primary key (install_id, tx_id)
)

row_events_log (
  install_id   text not null,  -- with seq, the reset-proof idempotency key
  seq          bigint not null,-- client row_events.id: exact per-install append order (sortable)
  tx_id        text not null,  -- links to command_events_log
  block_id     text not null,
  workspace_id text not null,
  kind         text not null,  -- create | update | soft-delete | delete
  before_json  text,           -- enc:v1: on e2ee workspaces
  after_json   text,           -- enc:v1: on e2ee workspaces
  commit_seq   bigint,         -- server commit-ordered frontier for cursors (see §6/§8; NOT bigserial)
  primary key (install_id, seq)
)

-- public schema: default-deny RLS + revoke (repo rule); writes only via the
-- SECURITY DEFINER append RPC, which checks private.is_workspace_writer per row.
alter table … enable row level security;
revoke all on … from anon, authenticated;
grant select on … to authenticated;
create policy …_read on … for select
  using (private.is_workspace_member(workspace_id, (auth.uid())::text));

Identity is reset-proof by construction: install_id is a per-install UUID nonce regenerated on every wipe/reinstall (a fresh DB restarts the row_events.id autoincrement at 1, so the id alone is not stable). Post-wipe events carry a new install_id, so they can never collide with already-uploaded ids — no silent ON CONFLICT drop. install_id is a per-install dedup nonce; a durable per-device identity, if ever wanted for "originating device" grouping, is a separate concern and is not required here.

7. Fidelity vs. volume: run-compaction by class

Volume is the one real cost of shipping every event: the 300K import wrote ~304K row_events ≈ 262 MB locally (docs/follow-ups.md), and on E2EE the blobs can't be GC'd by content. But "machine edits are edits" — we don't want to drop them by actor. The resolution is compaction by class via snapshots, not deletion:

Eligibility needs an explicit marker — source/scope can't carry it. command_events.source is uniformly 'user' for every repo.tx (§2), and machine migrations such as scripts/migrate-roam-location-refs.mjs run under ChangeScope.BlockDefault — the same scope as ordinary edits. So a policy keyed on those fields would either compact human edits or miss noisy machine runs. Compaction must be gated on a dedicated, explicit machine/processor marker recorded at commit time (the machine_origin field in the sketch — e.g. processor id, agent-runtime, or import), added as part of this work. Until a command carries that marker, its events are never compacted.
  • Run-compaction (lossless on the facts). Collapse a contiguous run of events on the same block that share a machine_origin marker into one span-event: before = the run's first before-state, after = the run's last after-state, plus the list/count of contributing commands. You keep that it happened, what it net-did, and which commands — you lose only the intermediate snapshots. This is exactly the snapshot+truncate move from event sourcing, and what compactBlockCrudEntries already does for block state, applied per-marker to the log.
  • Per-operation summary (lossy — reserved for uniform bulk). A 300K-block roam import is uniform; collapse it to one record ("imported N blocks under marker X") with no per-block events. This is the only place per-block detail is dropped, and it aligns with the follow-ups.md plan to skip events for import.
  • Human edits are never eligible — they simply carry no machine_origin marker, so they are untouched by construction.

Compaction is a policy on a derived view: keep the fine-grained events locally (in row_events) as long as you like, and compact only what you ship/retain server-side. Default conservative — keep everything, compact only known-noisy machine runs. Retention beyond that (a TTL/cap below a snapshot) is the standard log-compaction story; the open knob is whether command_events_log outlives row_events_log.

8. How the field solves these

Every problem here is canonical. The point of this section is that the recommended design picks the field's standard answer for each rather than improvising.

ProblemStandard solutionHere
Ordering Commit-ordered sequencer (Kafka offset, Postgres LSN/commit-ts — not a pre-commit serial); logical clocks (Lamport/HLC) for causality; per-aggregate version usually suffices Server commit-ordered commit_seq (LSN/commit-ts) for the cursor + per-install seq; per-block is the aggregate
Identity / idempotency (replicaId, counter) with a fresh replicaId on reset (Yjs, Automerge, Replicache (clientID, mutationID)) (install_id, seq), install_id regenerated on wipe
Atomicity (state vs events) Transactional outbox, or single log + projections (no dual-write) Append-after-block-success; block state stays a projection of what landed
Completeness / catch-up Snapshot + log: read snapshot, tail log; full history needs a central retained log Snapshot = the blocks table; central log = these server tables
Command → N events Correlation / causation ids on self-describing events tx_id as correlation id; events carry their own diff
Retention Log compaction + snapshots; accept a history horizon Run-compaction by class (§7)

The honest limit no one escapes: a sequencer's order is receipt order, not causal edit order (an offline device's week-old edits sequence "now"), and genuinely concurrent cross-device edits to one block have no ground-truth order. Logical clocks capture causality where it exists; concurrency is stated as concurrency, not forced into a false line.

9. Lessons from the stale-gate saga

The hydration/stale-gate work (which landed the always-increasing updated_at and the monotonic user_updated_at clamp) is the same family of problem — convergence ordering under LWW with untrustworthy clocks — and it carries three lessons straight into this design:

  • The clock-goes-backward bug is already solved — but borrow the principle, not the wrong column. The saga replaced unreliable wall-clock comparison with a monotonic, always-increasing version. Ordering events by raw created_at reintroduces exactly that inversion. The lesson is "monotonic, not wall-clock" — but be precise about which stamp: updated_at is the monotonic row-version (per row-version, not per-event), whereas user_updated_at is display/provenance time and is deliberately non-monotonic (sourceTimestamps imports historical Roam times; {skipMetadata} leaves it untouched). So user_updated_at is not an ordering key, and the pair must not be treated as one. Event order comes from per-install seq + the server's commit-ordered frontier (§6); if a cross-install causal hint is ever needed, it's a dedicated monotonic event stamp / logical clock, reusing the saga's principle without overloading a row column whose semantics don't fit.
  • Provenance must never feed convergence. The gate deliberately stopped reading author/provenance and became a plain version comparison, because entangling "who/why" with "which state wins" was complex and buggy. Hard constraint for us: the event log is strictly additive and read-only w.r.t. convergence — it records what happened, it never gates which row wins. (So this design has no impact on the stale gate, by intent.)
  • The updated_at / user_updated_at split is the seam we widen. One is the convergence version, the other the human/provenance timestamp. The "state vs provenance" separation we keep rediscovering is already in the schema; the event log generalizes user_updated_at's role while updated_at stays the convergence version, untouched.

The tempting direction the saga rules in only eventually: a complete, server-sequenced log could resolve conflicts by replay/merge instead of LWW-on-updated_at — a more principled convergence. That is "events as the source of truth," the big rewrite deferred below.

10. Relationship to blocks_history

The event log complements blocks_history; it does not replace it. Different trust models:

blocks_historyevent log
Who writes itServer trigger, from the write that landsClient-originated, server-sequenced on receipt
ResolutionSync-checkpoint (post-collapse)Per-commit (run-compacted per §7)
ProvenanceNone (semantic_op + changed_columns)Rich (command, scope, mutators)
CompletenessAnything landing in blocks after the trigger shipped, client-independentAs complete as clients upload + retention keeps

The load-bearing row is the last. blocks_history is server-triggered from authoritative landed state — atomic with the write, client-independent, and the declared PITR substrate — so it is the backstop on exactly the failure modes the log is best-effort about (origin device dies / is wiped / its upload is quarantined). Even on E2EE, do not trim it: readability is only one of its advantages; server-triggered + atomic + client-independent survive encryption, so on an encrypted workspace it remains the only record that a write happened even when its diff is opaque. The event log adds richer (encrypted) provenance; it never substitutes for the backstop.

blocks_history is not retroactive. Migration 20260522062437 creates the table/trigger but does not backfill existing blocks, and maintenance jobs can disable the trigger for bulk work. So blocks written before the trigger (or while it was disabled) have no history — the "complete backstop" guarantee holds only for writes recorded after the trigger. Closing that gap needs a one-time backfill/anchor (seed a synthetic create from current row state); out of scope, but not to be assumed away.

11. Residual costs & open questions

Under server-as-log, the identity/atomicity/authz problems are answered. What genuinely remains:

  • The cursor needs a commit-order frontier. A pre-commit BIGSERIAL is not a safe tail cursor (§6): a serial allocated by an earlier-but-later-committing tx can be skipped by a reader that already advanced. The frontier must be commit-ordered (commit timestamp / LSN / commit-time-assigned sequence), or the cursor must be a watermark that won't advance past an uncommitted gap. This is a real mechanism to choose, not a free property.
  • Compaction needs an explicit machine marker. source/scope don't distinguish machine from human (§7), so a machine_origin field must be added and populated at commit before any run-compaction is safe.
  • Volume / retention. Mitigated by run-compaction (§7) but not eliminated. Open: keep-forever vs. cap/TTL; whether command_events_log outlives row_events_log.
  • Server stays blind on E2EE. Encrypted events are opaque server-side; only devices benefit. Plaintext workspaces get full server-side queryability. Accepted — matches the E2EE posture.
  • Completeness has structural holes. A member who joins later gets current state (the snapshot) but not pre-join event detail unless an originator's log still covers it; pre-feature blocks and trimmed/compacted runs are gaps. The log is an overlay; blocks_history + the blocks snapshot are the complete-state floor.
  • Connector demux. The new event type must be partitioned out before compactBlockCrudEntries (which throws on non-blocks tables) and routed down the append+encrypt path. "Additive" is true at the design level; the shared connector is genuinely restructured, so a bug there can regress the block path.
  • Skipped deterministic creates. applyBlockCreates is insert-or-skip (ignoreDuplicates) for deterministic-id bootstrap; a skipped create must not leave a kind='create' event claiming a row the server already had. The append-after-block- success rule (§6) plus filtering creates whose block upload was a no-op handles this.

12. Rejected / deferred alternatives

Client-authored global order (the incoherent middle). Upload a client-sequenced (install_id, seq) log to a dumb server table. This was the original sketch; it is the source of the ordering/identity/atomicity/authz problem cascade (§3). Superseded by server sequencing.
Un-collapse the blocks upload itself. Replays 50 UPDATEs on the hot row to manufacture history — maximal write/index/trigger amplification for a row that ends in one state. Appends get the same granularity far cheaper (§4).
Sync the event log down to every device. Balloons bucket sizes for data almost no device queries. Read via RPC, like blocks_history.
Replace blocks_history with the event log. Loses the server-guaranteed, client-independent backstop and the plaintext PITR substrate (§10).
Deferred: events as the sole source of truth (server materializes blocks from events; convergence by replay/merge). The principled end state (§9), but a much larger rewrite. This proposal is deliberately additive: blocks stays the convergent state path; the log layers on top.

13. What this PR is (and isn't)

This PR adds only this design document. No schema, migration, connector, or encryption changes. What gates an implementation:

  1. Pick the model (§5). Server-as-log (recommended) vs. server-captured checkpoints (if "enough to classify edits" is the real bar) vs. peer CRDT (if no-central-authority ever becomes a goal).
  2. Compaction + retention policy (§7). An explicit machine_origin marker (source/scope don't distinguish machine from human); which markers are run-compactable; bulk-import summary; keep-forever vs. cap.
  3. Prerequisites (§6). A per-install install_id nonce (mint + persist in client_schema_state, regenerate on wipe); a commit-ordered cursor frontier (LSN / commit-ts / watermark — not a pre-commit BIGSERIAL); per-install seq for exact intra-install order (and a dedicated logical clock only if a cross-install causal hint is needed — not user_updated_at); the append RPC's definer + is_workspace_writer contract; the AAD for an opaque event blob.

The direction (append-only log, server-sequenced, additive to collapsed state, complementing blocks_history, run-compacted for machine bulk) is what the design discussion converged on. The earlier client-sequenced sketch is retained above only as the rejected baseline that motivates it.
