User-defined types

Users author types as blocks — same shape as user-defined property schemas. Code-defined types stop being special. Per-type behavior moves from a setup callback to same-tx processors.

Companion file: design.ts specifies the type surfaces this doc describes by importing real codebase types. Drift between the prose and the codebase fails yarn tsc --noEmit --project docs/tsconfig.json. Pattern follows plugin-runtime-toggle.

The premise has shifted

When docs/type-system.md was written, user-authored types were §9-deferred to a dynamicExtensions-style resolver pattern. The reasoning was strong: FacetRuntime is immutable after construction for real reasons (atomic switchover, upfront validation, deterministic combine, order-independent visibility), and a mutable-contribution sink would compromise those. The follow-up at docs/follow-ups.md:147 still reflects that position.

The user-defined properties work invalidated that argument. It shipped exactly the mutable contribution path the type-system v1 ruled out — repo.setRuntimeContributions(facet, sourceId, contributions) with per-facet rebuild steps (src/data/repo.ts:1340) and an end-to-end service (src/data/userSchemasService.ts) that subscribes to property-schema blocks, builds AnyPropertySchema entries, and publishes them into the 'user-data' source bucket on propertySchemasFacet. Atomic switchover is preserved because each rebuild step is atomic and only the dependent steps re-run.

With that primitive paid for, the question for user-defined types isn't "resolver vs sink" anymore. It's: do user-defined types mirror UserSchemasService, or stay deferred until something forces the issue? This doc takes the mirror path.

Storage model: types are blocks

Symmetric to property-schema blocks. A new kernel type block-type holds three fields: 

// src/data/properties.ts (additions)
export const blockTypeLabelProp       = defineProperty<string>('block-type:label',       {...})
export const blockTypeDescriptionProp = defineProperty<string>('block-type:description', {...})
export const blockTypePropertiesProp  = defineProperty<readonly string[]>('block-type:properties', {
  codec: codecs.refList({targetTypes: ['property-schema']}),
  defaultValue: [],
  changeScope: ChangeScope.BlockDefault,
})

And the type contribution itself, registered in src/data/blockTypes.ts: 

export const BLOCK_TYPE_TYPE = 'block-type'

defineBlockType({
  id: BLOCK_TYPE_TYPE,
  label: 'Type',
  properties: [
    blockTypeLabelProp,
    blockTypeDescriptionProp,
    blockTypePropertiesProp,
  ],
})

Per-workspace Types page (deterministic uuid v5 from workspaceId, parallel to src/data/propertiesPage.ts) hosts these blocks as children. Users navigate to it, see the list of types, click into one to edit name / description / property set.

Type id = block id, not slug

The persisted block:types string array (per docs/type-system.md §2) now mixes semantic kernel ids and block-id uuids: 

// A daily note that's also tagged as a user-defined "meeting" type
block:types = ['daily-note', '7d4e2b1a-...']
                ^kernel       ^user-defined: id of the meeting block-type block

Block ids are the right answer because they survive renaming the type's display label, they don't need uniqueness enforcement, and — load-bearing for Roam adoption — roam:isa = [[Person]] already resolves to the Person page's block id via refList. Promotion becomes mechanical (see §Roam isa adoption). The picker UI shows blockTypeLabelProp, never the id.

Alternative rejected: slug ids in a block-type:id field

Readable in logs, but every alternative has at least one of: brittle on rename, requires global uniqueness enforcement, makes adoption from roam:isa non-mechanical (refs are by block id, slugs aren't), and forces a second registry of "id → block." Block ids win on every axis except eyeballing block:types in raw JSON, and the picker UI takes care of that.

Asymmetry with property schemas: persisted key matches its audience

Property schemas use name as the persisted identifier (the key in properties_json, e.g. status); types use block id (the entry in block:types). The asymmetry is deliberate: property names are read by humans — in DB dumps, in debug tools, in raw JSON inspection — so a human-readable persisted key earns its keep. Type ids are only read by the runtime (the picker UI shows the label); opaque is fine. Rule of thumb for future meta-types: human-readable identifier when a human reads the persisted form, block id otherwise.

UserTypesService

Near-clone of UserSchemasService (src/data/userSchemasService.ts), adapted for the type-contribution shape: 

// src/data/userTypesService.ts (new)
export class UserTypesService {
  private contributions: readonly TypeContribution[] = []
  private nameToBlockId = new Map<string, string>()
  private subscriptionDisposer: Unsubscribe | null = null
  private schemasListenerDisposer: (() => void) | null = null

  constructor(private readonly repo: Repo) {}

  start(): () => void {
    const rebuildFromBlocks = (blocks: readonly Block[]) => {
      const schemas = this.repo.propertySchemas
      const next: TypeContribution[] = []
      for (const block of blocks) {
        const built = this.tryBuildType(block, schemas)
        if (built) next.push(built)
      }
      this.contributions = next
      this.repo.setRuntimeContributions(typesFacet, 'user-data', this.contributions)
    }

    this.subscriptionDisposer = this.repo.subscribeBlocks(
      {types: [BLOCK_TYPE_TYPE]},
      blocks => rebuildFromBlocks(blocks.map(b => this.repo.block(b.id))),
    )
    // Re-resolve when the merged schema map changes — a newly-loaded
    // property schema makes previously-unresolvable refs in
    // block-type:properties suddenly resolvable. Mirrors
    // onValuePresetsChange in UserSchemasService.
    this.schemasListenerDisposer = this.repo.onPropertySchemasChange(() => rebuild())

    return () => this.dispose()
  }

  private tryBuildType(
    block: Block,
    schemas: ReadonlyMap<string, AnyPropertySchema>,
  ): TypeContribution | null {
    const label = block.peekProperty(blockTypeLabelProp) ?? ''
    const refIds = block.peekProperty(blockTypePropertiesProp) ?? []
    const properties: AnyPropertySchema[] = []
    for (const refId of refIds) {
      const schemaBlock = this.repo.peek(refId)
      const name = schemaBlock?.properties[propertyNameProp.name]
      if (typeof name !== 'string') continue            // schema block not yet synced
      const schema = schemas.get(name)
      if (schema) properties.push(schema)
    }
    if (!label) {
      console.warn(`[UserTypesService] block ${block.id} has empty label; skipping`)
      return null
    }
    return {
      id: block.id,
      label,
      description: block.peekProperty(blockTypeDescriptionProp),
      properties,
    }
  }

  // Look up the user-data type contribution's source block id, if any.
  // Used by UI surfaces that want to navigate from a `block:types` entry
  // back to the type-definition block (e.g. a "edit this type" action
  // on a panel section header).
  getTypeBlockId(typeId: string): string | undefined {
    return this.nameToBlockId.get(typeId)
  }
}

No synchronous-append / rollback primitive in v1

Earlier drafts of this design included appendUserType / removeUserType / peekContribution — a synchronous-write path symmetric to UserSchemasService.appendUserSchema, intended to register a user-defined type before the in-tx retag step of Roam-isa adoption. PR #50 pursued the same primitive on the schemas service and found it compounds quickly under concurrent overlap review (per-call CAS tokens, committed-only baselines, restore-vs-remove ordering) for a use case the codebase doesn't actually produce. The two-tx Roam-isa adoption flow (§Roam isa adoption) removes the need entirely — the subscription bridges Phase A and Phase B. If a future caller can prove single-caller-per-name semantics AND genuinely needs single-tx atomicity, a narrow primitive can be added then with a documented contract.

The bootstrap path (repo construction in src/data/repo.ts) starts the service after the workspace's Types page is ensured — same wiring as UserSchemasService.

Replace setup with same-tx processors

setup (docs/type-system.md:60-89) was designed before same-tx processors existed (src/data/api/sameTxProcessor.ts). With same-tx processors now a real primitive, setup is redundant.

What setup provides today:

  • Fires inside repo.addType after initialValues are written, in the same tx.
  • Use cases: child-block templates (meeting → Attendees / Agenda / Action items), computed initial values (due = now() + 7 days), cross-block wiring at add time.

What a same-tx processor watching the properties field provides:

  • Sees before and after BlockData per touched row.
  • Writes via ctx.tx.update/setProperty, amending the same tx; one undo entry covers everything.
  • Rejects via throw new ProcessorRejection(code, meta), rolling the user's tx back atomically.
  • Already has access to merged propertySchemas on ctx.

Lifecycle: addType before vs after

Sketch of a per-type processor

// src/plugins/meeting/typeOnAdd.ts
import {createChild} from '@/data/internals/kernelMutators'
import {addedTypes} from '@/data/properties'

export const meetingOnAdd = defineSameTxProcessor({
  name: 'meeting.onAdd',
  watches: {kind: 'field', table: 'blocks', fields: ['properties']},
  apply: async (event, ctx) => {
    for (const row of event.changedRows) {
      if (!addedTypes(row).includes('meeting')) continue

      // init-if-missing for type-defined computed initial values
      if (row.after.properties[meetingDateProp.name] === undefined) {
        await ctx.tx.setProperty(row.after.id, meetingDateProp, today())
      }

      // child-block template
      for (const label of ['Attendees:', 'Agenda:', 'Action items:']) {
        const childId = await ctx.tx.run(createChild, {
          parentId: row.after.id,
          position: {kind: 'last'},
        })
        await ctx.tx.update(childId, {content: label})
      }
    }
  },
})

Sync-apply gap

Local repo.tx writes (orchestrated, raw, plan) all flow through the commit pipeline at src/data/internals/commitPipeline.ts:263, which is where same-tx processors fire. PowerSync-applied writes from another device take a different path entirely: they land in SQLite via the CRUD-apply mechanism, never set tx_context.source, and surface to the rest of the app via the row_events tail (src/data/internals/rowEventsTail.ts:1-32) which updates the cache and fires invalidations but does not run same-tx or post-commit processors.

So replacing setup with a same-tx processor doesn't introduce a sync-apply gap — setup has exactly the same gap (it only runs inside repo.addType, which is local). But it doesn't close it either, which is what the earlier "every write uniformly" framing implied.

The honest comparison:

  • Local raw tx.update / addBlockTypeToProperties / plan code: setup misses these; same-tx processor catches them. Strictly better.
  • Sync-applied writes from another device: both miss. Equal.

For behaviors that must run on cross-device type changes (e.g., maintaining a derived side-index for typed queries when another client tags a block), the right tool is a separate processor wired to the row_events tail — the same mechanism parseReferences already uses for sync-applied content changes (src/plugins/references/referencesProcessor.ts:230). That's out of scope for replacing setup, but worth flagging: "type-add behavior" splits into two kinds.

  • Atomic with the originating write (child templates, computed initial values, rejecting writes that violate type-level constraints) → same-tx processor. Local-only by construction.
  • Eventually-consistent (side-index maintenance, downstream notifications, cleanup) → post-commit processor on the row_events path. Catches sync apply too.

Most current setup use cases (templates, computed initials) are the first kind — users adding types from another device shouldn't get child templates materialized retroactively without their consent. The sync-apply gap is the right behavior for that class. The second kind didn't have an analog in setup anyway.

What needs to be added

One small ergonomic piece — a delta helper so every processor isn't reimplementing the diff:

// src/data/properties.ts (additions)
// ChangedRow.before is null for inserts; ChangedRow.after is null
// for hard-deletes (`src/data/api/processor.ts:70-73`). Both helpers
// must accept either null without throwing, otherwise any processor
// using them aborts insert / hard-delete txs.
export const addedTypes = (row: ChangedRow): readonly string[] => {
  const before = row.before ? getBlockTypes(row.before) : []
  const after  = row.after  ? getBlockTypes(row.after)  : []
  return after.filter(t => !before.includes(t))
}

export const removedTypes = (row: ChangedRow): readonly string[] => {
  const before = row.before ? getBlockTypes(row.before) : []
  const after  = row.after  ? getBlockTypes(row.after)  : []
  return before.filter(t => !after.includes(t))
}

With the helpers null-safe, the example processor above is also safe on hard-delete events: a deleted block returns addedTypes(row) === [] for any type id, the loop body skips, and row.after is never dereferenced. Processors that genuinely care about removal switch on removedTypes(row) instead.

What repo.addType becomes

Without setup in the picture, the orchestration shrinks to: encode initialValues through their schemas' codecs, append the type id to typesProp. That's mutator-shaped work; it might even fold into a regular registered mutator. The _types retention on Repo stays useful for the property-panel section grouping (§3c in type-system.md) and the "is this id registered?" guard, but setFacetRuntime's _types snapshot is no longer load-bearing for behavior — behavior is in the processor stream now.

Migration moment

Kill setup now, before existing kernel use cases migrate into it. Today the only real user is the deferred daily-note template (type-system.md:678), and it's still being designed. Once daily-note's setup is written, the rewrite has cost; right now it doesn't. Same reasoning that landed user-defined properties before too many ad-hoc adhocSchema call sites were live.

Roam isa adoption

The import already does the survey work. src/plugins/roam-import/typeCandidates.ts walks roam:isa values per imported block, groups by alias, ranks by occurrence count and common-property frequency, and emits a report: 

[[Person]] -> type "person" (53 nodes); common props: dob 41/53 (77%), email 38/53 (72%)
[[Project]] -> type "project" (12 nodes); common props: status 10/12 (83%), owner 9/12 (75%)

What's missing is promotion — the action that turns a Roam page (which is currently the target of roam:isa refs) into a real block-type block and re-tags the referencing blocks.

The promotion primitive

One action, two call sites:

async function promoteToType(args: {
  targetBlockId: string                  // the Roam page that's the isa target
  label: string                          // pre-filled from the page's alias
  propertySchemaIds: readonly string[]   // chosen from the candidate report's common-prop list
  rewriteIsaReferences?: boolean         // default false — leave roam:isa intact for review
}): Promise<void>

The flow uses two transactions bridged by UserTypesService's subscription, rather than a single tx with a synchronously-registered provisional type. The single-tx approach was explored on PR #50 (the parallel withProvisionalSchema for the property-schemas service); it compounds quickly under review — per-call CAS tokens, committed-only mirrors, capture-and-restore on failure — for a use case (concurrent overlapping calls for the same name) the codebase doesn't actually produce. Two txs trades one undo entry for a trivial concurrency model and a cleaner failure surface. See §Lessons from withProvisionalSchema.

Phase A: commit the type-definition block

  1. Read the target block before opening the tx to capture its workspaceId. Background flows / import runs / UI workspace switches must not silently re-scope to activeWorkspaceId — per the merged PR #48 repo.queryBlocks now requires workspaceId at the type level, so any code path that omits it fails to compile rather than quietly mis-scoping.
  2. Open tx A. repo.addTypeInTx(tx, targetBlockId, BLOCK_TYPE_TYPE, {label, properties: propertySchemaIds}) materializes block-type:label and block-type:properties from the action args, plus appends 'block-type' to the target's typesProp. Per merged PR #49 addTypeInTx is strict by default — throws BlockNotFoundForTypeError if the target was deleted concurrently between the pre-tx load() and tx-open, no silent no-op possible. (Sync-apply / processor paths that legitimately want to no-op on tombstone use the new addTypeInTxLenient; promotion uses strict.)
  3. Also tag the target with PAGE_TYPE in the same tx if it isn't already — the "type flow" pattern landed for the Properties page, Readwise root, plugin state, and alias user pages (src/data/propertiesPage.ts). The target stays navigable as a normal page; the new block-type tag is additive, not exclusive.

Subscription tick (between txs)

Tx A commits. The row_events machinery fires; UserTypesService.subscribeBlocks({types: [BLOCK_TYPE_TYPE]}) sees the new row, rebuilds its contribution list, and publishes via repo.setRuntimeContributions(typesFacet, 'user-data', ...). repo.types now contains targetBlockId as a registered type id. No synchronous-append dance; the subscription bridges the txs.

Phase B: retag isa instances

  1. Prefilter candidate instances OUTSIDE the tx via repo.queryBlocks({workspaceId, referencedBy: {id: targetBlockId, sourceField: ROAM_ISA_PROP}}). The block_references edge index from type-system.md §6b answers this in one EXISTS. Outside-the-tx because queryBlocks reads via ctx.db.getAll on the read DB handle; running it inside repo.tx reintroduces the PowerSync queue deadlock documented in tasks/processor-tx-deadlock.md.
  2. Open tx B. For each prefiltered candidate, tx.get(candidateId) re-checks the row (closes the TOCTOU window — the candidate may have been deleted or had its roam:isa rewritten between the pre-tx query and tx-open). If the row still references the target, repo.addTypeInTx(tx, candidateId, targetBlockId, {}) appends the new type id to block:types. The id resolves because the subscription registered it in the interval.
  3. If rewriteIsaReferences: rewrite the roam:isa refList on each instance, dropping the promoted alias. Default off — users typically want both for a while as the promotion settles. 
const target = await repo.load(targetBlockId)
if (!target) throw new Error(`promoteToType: target ${targetBlockId} not found`)
const workspaceId = target.workspaceId

// Phase A: commit the type-definition block. addTypeInTx is strict-by-default
// per PR #49 — throws if targetBlockId is missing/tombstoned at tx-start.
const snapshotA = repo.snapshotTypeRegistries()
await repo.tx(async tx => {
  await repo.addTypeInTx(tx, targetBlockId, BLOCK_TYPE_TYPE, {
    [blockTypeLabelProp.name]: label,
    [blockTypePropertiesProp.name]: propertySchemaIds,
  }, snapshotA)
  await repo.addTypeInTx(tx, targetBlockId, PAGE_TYPE, {}, snapshotA)
}, {scope: ChangeScope.BlockDefault, description: `promoteToType:create ${label}`})

// Bridge tx A → tx B: wait for the subscription rebuild to publish
// targetBlockId into the user-data bucket on typesFacet, then proceed.
// Event-driven via repo.onTypesChange (added in Phase 1, symmetric to
// onPropertySchemasChange). Bounded with a default 10s timeout so a
// stuck registration (e.g. tryBuildType rejecting the block-type block
// because a referenced property-schema doesn't resolve) surfaces a clear
// PromotionRegistrationTimeout instead of hanging indefinitely. The
// caller can pass its own AbortSignal for user-driven cancellation.
await waitForTypeRegistrationBounded(repo, targetBlockId, label, signal, 10_000)

// Phase B: retag instances. queryBlocks must run OUTSIDE the tx — bare DB reads
// inside repo.tx reintroduce the PowerSync queue deadlock (follow-ups.md:19-21).
const candidates = await repo.queryBlocks({
  workspaceId,
  referencedBy: {id: targetBlockId, sourceField: ROAM_ISA_PROP},
})

const snapshotB = repo.snapshotTypeRegistries()
await repo.tx(async tx => {
  for (const candidate of candidates) {
    const row = await tx.get(candidate.id)
    if (!row) continue                          // deleted between prefilter and tx
    const isaRefs = row.properties[ROAM_ISA_PROP] ?? []
    if (!resolveRoamIsaTargets(isaRefs).includes(targetBlockId)) continue
    await repo.addTypeInTx(tx, candidate.id, targetBlockId, {}, snapshotB)
  }
  if (rewriteIsaReferences) /* ... */
}, {scope: ChangeScope.BlockDefault, description: `promoteToType:retag ${label}`})

Why two txs, not one

The single-tx alternative needs to register the new user-defined type before addTypeInTx sees it. That's a setRuntimeContributions call BEFORE opening the tx, plus a rollback on tx failure. PR #50 explored this shape on the parallel UserSchemasService; the rollback semantics compound under review (CAS tokens, committed-only baselines, retry safety) every round, for a concurrent-overlap scenario the codebase has zero callers for.

Two txs sidestep the entire mechanism. The cost is one extra undo entry (create-type, then retag) instead of one combined entry. Acceptable: users will reasonably think of them as two operations anyway, and an undo of just-the-retag leaves the type definition intact, which is the right shape if they want to redo the retag with a different property set.

In-tx existence is enforced by strict addTypeInTx

With PR #49 landed, addTypeInTx throws BlockNotFoundForTypeError when the target row is missing or tombstoned. No explicit tx.get + deleted guard is needed before the addType call — the API surface encodes the invariant.

Partial-promotion recovery: tx A committed, registration timed out

The Phase A→B handoff has a bounded wait (default 10s) on the subscription registering the new type id. If the wait times out (real cause: the type-definition block committed in tx A, but UserTypesService.tryBuildType is rejecting it — e.g. a referenced property-schema id doesn't resolve against the current registry), waitForTypeRegistrationBounded throws PromotionRegistrationTimeout before Phase B runs.

Recovery is well-defined. Tx A's block-type block is on disk. The caller can re-run promoteToType with the same args; the second run's tx A is idempotent (already-typed targets re-addType to a no-op), the registration is by then live (the subscription has had time to either fire or the underlying parse issue has been fixed), and Phase B runs against the now-registered type.

Caller-side cancellation works the same way: the caller passes signal on PromoteToTypeArgs; aborting between Phase A and Phase B leaves the type-definition block committed and instances un-retagged. The error surface tells the user "the type was created; some instances may not have been re-tagged. Re-run to complete." — matches how partial bulk operations surface elsewhere in the app.

Why the query lives outside the tx

repo.queryBlocks and other typed-query reads route through ctx.db.getAll on the read DB handle, not through tx's write-transaction lock. Under PowerSync's serialized single-connection config, holding a writer slot while issuing a bare DB read deadlocks the queue — the read can't progress until the writer releases, and the writer is blocked on the read. The kernel's parseReferences and cleanupOrphanAliases processors paid the same cost and moved their broad reads outside the write tx (docs/follow-ups.md:19); the promotion flow follows the same shape.

The TOCTOU window opened by reading outside the tx is narrow and the in-tx re-read closes it: instances may be deleted or have their roam:isa rewritten between the prefilter and the tx, which the in-tx tx.get + property check catches. Conversely, instances that gained an isa ref during the window are silently missed by this promotion — conservative miss is fine; the user can re-run promotion.

Two surfaces use the primitive:

  • Auto-suggest at import time. The existing type-candidate report (src/plugins/roam-import/report.ts:492) gets an "Adopt as type" button per candidate. Property checkboxes default to the common-prop list. Same code path.
  • After-the-fact. "Promote this page to a type" action on any page that's a roam:isa target. Surfaced through the same action infrastructure that handles other context-actions on pages.

Phasing

Each phase ships independently.

Prerequisites already landed on master:

  • PR #48 — workspaceId required on repo.queryBlocks / subscribeBlocks (active-workspace callers use queryActiveWorkspace). Phase 4 below depends on the explicit-workspace contract.
  • PR #49 — addTypeInTx strict-by-default; throws BlockNotFoundForTypeError on missing/tombstoned blocks. The lenient variant (addTypeInTxLenient) is opt-in for sync-apply paths only. Phase 4 below relies on strict semantics for the in-tx existence guard.

  • Standalone action on any page registered as a roam:isa target.
  • Optional rewriteIsaReferences flag; default off.

Acceptance: on a Roam import with 53 blocks tagged is a [[Person]], promoting Person turns the page into a block-type and re-tags all 53 instances; the Person panel section in the property panel reflects the chosen schemas. Locally completing a block's local edits and reattempting promotion leaves the local state untouched on the retag pass (since strict addTypeInTx on already-typed blocks is a no-op).

Phase 4 — block-type:template (optional, deferred until first consumer)

  • Add blockTypeTemplateProp as a refList on the block-type contribution.
  • typeTemplateMaterializer same-tx processor in src/plugins/user-types/ (or a dedicated user-types plugin) per design.ts. Watches properties; on each addedTypes(row).includes(typeId) match, walks the type's block-type:template refList and clones each root's descendants under the instance via tx.run(createChild, ...). Idempotent per row (init-if-missing) so a re-add doesn't duplicate.
  • templateCycleGuard same-tx processor watches properties + parentId and throws ProcessorRejection('userTypes.templateCycle') when a write would create a self-referential cycle (e.g., a block typed X moved into the subtree of X's template root).
  • Driver / consumer: the recurring-tasks design (parallel work) builds on this for its template-spawn flow.

Acceptance: a block-type with block-type:template = [[Meeting template]] materializes Attendees / Agenda / Action items children under a new instance on type-add. Moving an instance into the meeting-template subtree throws the cycle-guard rejection; the move is rolled back.

Type templates — block-type:template

Some types want to materialize a structured subtree under every new instance. The Roam-imported “meeting” type wants Attendees / Agenda / Action items child blocks; a “weekly review” type wants its standard rubric. The replaced-setup design (§Replace setup with same-tx processors) handles the materialization mechanism; what's left to specify is the data shape of the template itself.

A new property on the kernel block-type type: block-type:template, a refList of root blocks whose subtrees act as templates. When the type is first added to an instance, a same-tx processor materializes those subtrees under the instance.

The type-definition block is the user's block

A user-defined type is just a block. Marking a block as a type (via the type-system UI, or via the recurring-tasks template flow that builds on this) turns that same block into the type definition. It remains a navigable, editable page. Its subtree, if any, is what becomes the template. One block, dual role.

This is worth saying out loud rather than leaving implicit. The alternative shape — a hidden type-definition block plus a refList back to user-visible content — is a worse mental model and forces every consumer to thread the indirection. Naming the dual-role intent here prevents a future maintainer from "tidying up" the redundancy and re-introducing the indirection.

Descendants-only materialization

When the processor materializes a template, it clones the children of each referenced root, not the root itself. The new instance gets those children attached directly underneath; the template root is never duplicated.

  • Include-root would surprise. A “meeting” instance with a duplicate [[Meeting]]-typed root block as its first child reads as a bug, not a feature.
  • Descendants-only composes with every existing child-add path. createChild / tx.run(createChild, ...) already appends children under a parent; the materializer is the same shape repeated for each template root's children.

A future use case for include-root materialization (where you really do want the root cloned along with its subtree) gets a separately-named slot (block-type:template-root-included or similar) rather than flipping the semantics of the existing one. Changing this after consumers exist is painful.

Cycle detection at tx-time

Easy to construct accidentally: user drags an instance back into a template subtree, or makes a typed block itself a template root for its own type. Materialization then recurses indefinitely.

Reject at the moment the cycle is created, not at materialization time. A same-tx processor watches typesProp changes and parentId changes. For each affected row it walks the parent chain looking for an ancestor B such that any of the row's current types appears in B's block-type:template refList; if found, throw new ProcessorRejection('userTypes.templateCycle', ...) with the offending pair so the toast surface can format a clear “this block can't be moved into its own type's template” message.

Tx-time rejection is preferable to a depth-guard at materialization for the same reason most validation should live at the boundary: the error surface is the action the user took, not a downstream symptom. A depth-guard catches recursion ("hit recursion limit during template materialization") but leaves the underlying cycle intact. Rejecting at the move/tag-add prevents the cycle from being persisted at all.

See templateCycleGuard in design.ts for the watch surface; implementation goes in src/plugins/user-types/ (or wherever the user-types plugin lands).

Lessons from withProvisionalSchema

PR #50 built a capture-and-restore primitive on UserSchemasService for exactly the pattern this design's earlier draft applied to the Roam-isa adoption flow: register a contribution synchronously before an in-tx write that depends on it, roll back the registration if the tx fails. The PR closed without merging; the doc was updated in this round to drop the analogous primitive. The reasons are worth pinning down because they apply to any future "register + in-tx-dependent" pattern in this codebase.

What review surfaced, in order

  1. Unconditional remove on rollback wipes legit prior registrations. A retry on an already- registered name finds the prior entry, and unconditional removal erases it. Fix: capture prior, restore-or-remove based on what was there.
  2. The captured prior can be stale if a concurrent caller has already overwritten the slot. Fix: compare-and-swap on the live entry before rolling back.
  3. CAS by blockId isn't unique enough. Two concurrent calls for the same (name, blockId) (retries / dupe submits) share the field. Fix: per-call CAS token, stored in a parallel map; every mutation generates a fresh symbol.
  4. The captured prior can be another in-flight provisional, not a committed entry. If A appends, B starts and captures A as prior, then both fail: A skips rollback (token mismatch), B "restores" A — leaving a failed provisional registered with no successful tx behind it. Fix: track a committed-only mirror updated by successful operations only; rollback uses the committed baseline, not the live peek.
  5. Success-path token check skips committed promotion under overlap. If A's tx succeeds while B's provisional has overwritten A's live token, the success guard fails the CAS and never promotes A to committed. A subsequent B-failure rollback then wipes A's never-committed-but-real registration. Fix: commit-on-success regardless of token state — which then has ordering issues with intervening external commits, which the next round would have found.

The shape of the failure mode

Each fix above is correct. Each one made the primitive heavier, and each round of review found a deeper edge case — entirely because the primitive had taken on responsibility for arbitrary concurrent-overlap correctness on a slot that the codebase doesn't actually contend on. The class of bug is unbounded: the API surface keeps growing to handle interleavings that don't happen.

What this design does instead

  • No withProvisionalType, no appendUserType, no peekContribution. UserTypesService exposes only the read-side surface (getTypeBlockId) needed by UI. Block writes go through the kernel orchestration entry points (addTypeInTx etc.) which already handle the strict/lenient distinction and reject on missing/tombstoned blocks (PR #49).
  • Where in-tx dependents exist, use two txs. The Roam-isa adoption flow commits the type-definition block in tx A, lets the subscription rebuild register the type, then opens tx B for the bulk retag. Two undo entries instead of one; trivial concurrency model; no rollback machinery.
  • If a future caller needs single-tx atomicity, design the primitive against that caller's actual interleavings. Single-caller-per-name semantics make this trivial. Arbitrary concurrent overlap is a much deeper invariant; defer until there's a real caller that needs it.

The deeper meta-lesson: design-time spec rigor on concurrency edge cases isn't free. Each "what if two callers…" question that can be sidestepped at the design level by ordering txs differently or narrowing the API surface is worth sidestepping. The places where it truly can't be sidestepped — same-tx processors on user-driven property edits, post-commit processors on sync-applied writes — have their concurrency model documented at the primitive level (commit pipeline order, row_events tail semantics) and consumers compose against that documented model rather than re-deriving invariants per-feature.

Cross-service rule

The "no sync-append-with-rollback primitive" lesson isn't service-specific. It applies symmetrically to the existing UserSchemasService (src/data/userSchemasService.ts) and any future user-data contribution service. Stated as a shared invariant for future maintainers:

Invariant for all user-data contribution services

Do not add a synchronous-append-with-rollback primitive (appendUserX / removeUserX / withProvisionalX) to a user-data contribution service unless every consumer can prove single-caller-per-key semantics AND single-tx atomicity is actually required. Default to two txs bridged by the subscription: commit the contribution block in one tx, let the service's subscription rebuild register it, then open the dependent tx. The cost is one extra undo entry; the savings are the entire class of concurrent-overlap correctness bugs that PR #50 documented.

Action item for the user-defined-properties doc: add a cross-link from docs/user-defined-properties.md §7 (addSchema) noting that the post-tx appendUserSchema ordering is deliberate and not to be inverted into a pre-tx synchronous- append — PR #50's iteration sequence is the cautionary tale.

What's deferred

Kernel-type overrides via block-type:extends

The motivating use case — user wants daily-note blocks to surface weather and mood slots — is plausible but hypothetical. No concrete consumer exists today. The mechanism adds non-trivial complexity (a second contribution shape, a post-combine merge step, a renderer "extends" picker, the merge-rule matrix for user-wins-vs-kernel-wins) that's not worth carrying through v1.

The cut is shaped so the v2 implementation slots in without re-deciding anything. The forward path:

  • Add blockTypeExtendsProp = defineProperty<string>('block-type:extends', ...) to src/data/properties.ts; lift onto the BLOCK_TYPE_TYPE contribution.
  • Extend UserTypesService.tryBuildType to branch on extendsId — if present, emit a TypeOverride record ({kind: 'override', target, label?, description?, properties[]}) instead of a full TypeContribution.
  • Bucket overrides separately from full contributions when publishing into the user-data source on typesFacet. The rebuild step that populates _types on Repo grows a post-combine pass mergeTypeOverrides(base, overrides) that folds each override into its target. Overrides whose target doesn't resolve get a logged diagnostic and are dropped.
  • Merge rules:
    • properties[] — union with the kernel list, dedup by name via the object-identity rule in mergeLiftedSchemas (src/data/repo.ts:178).
    • label / description — user wins if set, else kernel (empty-string ⇒ undefined so kernel's value isn't blanked).
    • Any behavioral field (setup, if it ever returns) — kernel only.
    • block-type:template — lives on the source block; out of contribution merge entirely.
  • Renderer surfaces an "extends" picker on the block-type block; tested against a kernel type (daily-note) plus a user type.

Why not a separate typeOverridesFacet

The "clean Tana" answer is a parallel facet keyed by target type id. But it adds a service, a publish path, and a rebuild step for one anticipated use case. The single-block-type-with-extends shape above is strictly less surface; if overrides grow real complexity later (per-type render overrides, chained setups, etc.) extracting the facet is mechanical. Don't pre-extract.

Trigger to build: a real user wants per-type panel slots / labels on a kernel type, or a follow-up design (e.g. a "themes" or "workspace presets" mechanism) needs to customize kernel-type metadata. Until then, adding the same fields as their own user-defined type is workable — just not the same UX.

Other deferred items

  • Type inheritance for user types. A user type extending another user type rather than a kernel type. Same mechanism as kernel-type overrides above; only the resolver loop guard (cycle detection) is extra.
  • Required-field validation on add. A type-definition block lists which of its properties[] are required; a same-tx processor rejects writes that leave them unset. Easy once Phase 2 lands.
  • Property-schema authoring inline with the type. Today the user creates schemas on the Properties page, then refs them from a type on the Types page. A combined "type editor" surface that creates schemas as side-effects of editing a type is ergonomic but not necessary for v1.
  • Bulk reimport-time auto-promotion. The import currently suggests; this design's promotion stays user-triggered. Heuristic-based auto-promotion at import time is plausible but a separate UX call.
  • Include-root template materialization. §Type templates fixes descendants-only as the default. A separate slot for include-root semantics (when the template root itself should be cloned) is plausible but no current consumer.
  • property-schema:extends for symmetry with the deferred block-type:extends. User-defined property schemas could plausibly override a kernel schema (e.g., bind a different default value to statusProp without redefining the codec). Today this isn't possible — propertySchemasFacet.combine is last-wins-with-warn on duplicate names, not a merge. Worth proposing alongside any future "let users tweak kernel schema defaults" feature, in lockstep with the types-side override mechanism above.
  • Shared user-contribution subscription primitive. After UserTypesService ships, the machinery shared with UserSchemasService (subscribe to {types: [META]}, build contributions, publish to facet's 'user-data' bucket, re-resolve on dependency change, pin workspace at start()) should be lifted into a single composable (makeUserContributionSubscription({metaType, facet, source, tryBuild, dependsOn})). Best done after both services exist concretely — refactor against two real instances rather than designing the abstraction upfront. Lower priority than the page-bootstrap helper because the two services genuinely diverge on their dependency listeners (presets vs schemas) and per-row build logic.

Decisions locked in

  • User-defined types are blocks of kernel type block-type, hosted on a per-workspace Types page. The Types page itself follows the type-flow pattern: tagged with both PAGE_TYPE and TYPES_PAGE_TYPE at bootstrap.
  • Type id is the block id. Display label is block-type:label. The picker shows the label; block:types arrays carry ids.
  • A type-definition block is the user's block, not a hidden shadow. One block, dual role — navigable as a page, its subtree (if any) acts as the type template.
  • Kernel-type overrides via block-type:extends are deferred — no real consumer yet. Design notes preserved under "What's deferred" so the v2 implementation has a starting point.
  • Replace setup with same-tx processors for atomic per-add behavior. Type contributions become pure metadata. Do this before any current kernel use lands. Behaviors that must run on sync-applied type changes from other devices use a separate post-commit processor on the row_events tail (same path parseReferences uses).
  • Roam isa adoption rides one primitive (promoteToType) usable from import time or after-the-fact. Two-tx flow bridged by the subscription rebuild — no withProvisional-style primitive (per §Lessons).
  • Template materialization (Phase 4): refList on block-type:template; descendants-only clone (root not duplicated); cycle detection at tx-time via a ProcessorRejection.
  • Companion design.ts typechecks against real codebase types. Drift between this prose and the codebase fails yarn tsc --noEmit --project docs/tsconfig.json.
