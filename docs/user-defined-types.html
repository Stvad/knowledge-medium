User-defined types

Users author types as blocks — same shape as user-defined property schemas. Code-defined types stop being special. Kernel types get a clean override path. Per-type behavior moves from a setup callback to same-tx processors.

The premise has shifted

When docs/type-system.md was written, user-authored types were §9-deferred to a dynamicExtensions-style resolver pattern. The reasoning was strong: FacetRuntime is immutable after construction for real reasons (atomic switchover, upfront validation, deterministic combine, order-independent visibility), and a mutable-contribution sink would compromise those. The follow-up at docs/follow-ups.md:147 still reflects that position.

The user-defined properties work invalidated that argument. It shipped exactly the mutable contribution path the type-system v1 ruled out — repo.setRuntimeContributions(facet, sourceId, contributions) with per-facet rebuild steps (src/data/repo.ts:1340) and an end-to-end service (src/data/userSchemasService.ts) that subscribes to property-schema blocks, builds AnyPropertySchema entries, and publishes them into the 'user-data' source bucket on propertySchemasFacet. Atomic switchover is preserved because each rebuild step is atomic and only the dependent steps re-run.

With that primitive paid for, the question for user-defined types isn't "resolver vs sink" anymore. It's: do user-defined types mirror UserSchemasService, or stay deferred until something forces the issue? This doc takes the mirror path.

Storage model: types are blocks

Symmetric to property-schema blocks. A new kernel type block-type holds three fields: 

// src/data/properties.ts (additions)
export const blockTypeLabelProp       = defineProperty<string>('block-type:label',       {...})
export const blockTypeDescriptionProp = defineProperty<string>('block-type:description', {...})
export const blockTypePropertiesProp  = defineProperty<readonly string[]>('block-type:properties', {
  codec: codecs.refList({targetTypes: ['property-schema']}),
  defaultValue: [],
  changeScope: ChangeScope.BlockDefault,
})
// Optional — if present, this is an override of an existing type, not a new one.
export const blockTypeExtendsProp     = defineProperty<string>('block-type:extends', {...})

And the type contribution itself, registered in src/data/blockTypes.ts: 

export const BLOCK_TYPE_TYPE = 'block-type'

defineBlockType({
  id: BLOCK_TYPE_TYPE,
  label: 'Type',
  properties: [
    blockTypeLabelProp,
    blockTypeDescriptionProp,
    blockTypePropertiesProp,
    blockTypeExtendsProp,
  ],
})

Per-workspace Types page (deterministic uuid v5 from workspaceId, parallel to src/data/propertiesPage.ts) hosts these blocks as children. Users navigate to it, see the list of types, click into one to edit name / description / property set.

Type id = block id, not slug

The persisted block:types string array (per docs/type-system.md §2) now mixes semantic kernel ids and block-id uuids: 

// A daily note that's also tagged as a user-defined "meeting" type
block:types = ['daily-note', '7d4e2b1a-...']
                ^kernel       ^user-defined: id of the meeting block-type block

Block ids are the right answer because they survive renaming the type's display label, they don't need uniqueness enforcement, and — load-bearing for Roam adoption — roam:isa = [[Person]] already resolves to the Person page's block id via refList. Promotion becomes mechanical (see §Roam isa adoption). The picker UI shows blockTypeLabelProp, never the id.

Alternative rejected: slug ids in a block-type:id field

Readable in logs, but every alternative has at least one of: brittle on rename, requires global uniqueness enforcement, makes adoption from roam:isa non-mechanical (refs are by block id, slugs aren't), and forces a second registry of "id → block." Block ids win on every axis except eyeballing block:types in raw JSON, and the picker UI takes care of that.

UserTypesService

Near-clone of UserSchemasService (src/data/userSchemasService.ts), adapted for the type-contribution shape: 

// src/data/userTypesService.ts (new)
export class UserTypesService {
  private contributions: readonly TypeContribution[] = []
  private nameToBlockId = new Map<string, string>()
  private subscriptionDisposer: Unsubscribe | null = null
  private schemasListenerDisposer: (() => void) | null = null

  constructor(private readonly repo: Repo) {}

  start(): () => void {
    const rebuildFromBlocks = (blocks: readonly Block[]) => {
      const schemas = this.repo.propertySchemas
      const next: TypeContribution[] = []
      for (const block of blocks) {
        const built = this.tryBuildType(block, schemas)
        if (built) next.push(built)
      }
      this.contributions = next
      this.repo.setRuntimeContributions(typesFacet, 'user-data', this.contributions)
    }

    this.subscriptionDisposer = this.repo.subscribeBlocks(
      {types: [BLOCK_TYPE_TYPE]},
      blocks => rebuildFromBlocks(blocks.map(b => this.repo.block(b.id))),
    )
    // Re-resolve when the merged schema map changes — a newly-loaded
    // property schema makes previously-unresolvable refs in
    // block-type:properties suddenly resolvable. Mirrors
    // onValuePresetsChange in UserSchemasService.
    this.schemasListenerDisposer = this.repo.onPropertySchemasChange(() => rebuild())

    return () => this.dispose()
  }

  private tryBuildType(
    block: Block,
    schemas: ReadonlyMap<string, AnyPropertySchema>,
  ): TypeContribution | null {
    const label = block.peekProperty(blockTypeLabelProp) ?? ''
    const extendsId = block.peekProperty(blockTypeExtendsProp)
    const refIds = block.peekProperty(blockTypePropertiesProp) ?? []
    const properties: AnyPropertySchema[] = []
    for (const refId of refIds) {
      const schemaBlock = this.repo.peek(refId)
      const name = schemaBlock?.properties[propertyNameProp.name]
      if (typeof name !== 'string') continue            // schema block not yet synced
      const schema = schemas.get(name)
      if (schema) properties.push(schema)
    }
    if (extendsId) {
      // Overrides may legitimately omit label/description — the merge
      // rule is "user wins if set, else kernel" (see §Overriding kernel
      // types). Pass undefined through so the rebuild step picks the
      // kernel value rather than overwriting with empty string.
      const description = block.peekProperty(blockTypeDescriptionProp)
      return {
        kind: 'override',
        target: extendsId,
        label: label || undefined,
        description: description || undefined,
        properties,
      } as TypeOverride
    }
    if (!label) {
      console.warn(`[UserTypesService] block ${block.id} has empty label; skipping`)
      return null
    }
    return {
      id: block.id,
      label,
      description: block.peekProperty(blockTypeDescriptionProp),
      properties,
    }
  }

  // Synchronous append used by callers that need a freshly-built
  // type to be registered BEFORE their next write (e.g. Roam
  // promotion's bulk re-tag step). Mirrors
  // UserSchemasService.appendUserSchema at userSchemasService.ts:167.
  // The block-write that backs `contribution` is the caller's
  // responsibility — this only updates the runtime bucket so the
  // id resolves in addTypeInTx's registry check.
  // Look up the current user-data contribution for an id (if any).
  // Callers using the synchronous-append pattern peek before
  // appending so they can restore the prior value on tx failure
  // rather than blindly removing it (which would wipe a legitimate
  // pre-existing registration on a retry).
  peekContribution(typeId: string): {contribution: TypeContribution; blockId: string} | undefined {
    const contribution = this.contributions.find(t => t.id === typeId)
    const blockId = this.nameToBlockId.get(typeId)
    return contribution && blockId !== undefined ? {contribution, blockId} : undefined
  }

  appendUserType(contribution: TypeContribution, blockId: string): void {
    this.contributions = [
      ...this.contributions.filter(t => t.id !== contribution.id),
      contribution,
    ]
    this.nameToBlockId.set(contribution.id, blockId)
    this.repo.setRuntimeContributions(typesFacet, 'user-data', this.contributions)
  }

  // Symmetric rollback for the synchronous-append path. Callers that
  // append before opening a tx use this in a catch arm so a tx
  // failure can't leave an orphan in the runtime. The subscription
  // would only reconcile on the next block-write, which never fires
  // if the tx aborted before writing any block.
  removeUserType(typeId: string): void {
    this.contributions = this.contributions.filter(t => t.id !== typeId)
    this.nameToBlockId.delete(typeId)
    this.repo.setRuntimeContributions(typesFacet, 'user-data', this.contributions)
  }
}

The bootstrap path (repo construction in src/data/repo.ts) starts the service after the workspace's Types page is ensured — same wiring as UserSchemasService.

Overriding kernel types via extends

Plausible use case: a user wants daily-note blocks to surface weather and mood slots in the property panel. They author a block-type block with block-type:extends = 'daily-note' and block-type:properties = [weather, mood].

The naive path — let typesFacet.combine's last-wins-on-id resolve it — is destructive: the user's contribution would replace the kernel daily-note entry, wiping label, kernel properties[], and (if it existed) setup. That's not what they meant.

The clean path: distinguish full contributions from override contributions. The typesFacet.of(...) contribution stays as-is for full types; user-defined override contributions arrive through the same 'user-data' source bucket but as a different shape, and the rebuild step in setFacetRuntime (src/data/repo.ts:1783) folds them in after combine:

The merge rules:

  • properties[] — union with the kernel list, dedup by name via the existing object-identity rule in mergeLiftedSchemas (src/data/repo.ts:178).
  • label / description — user wins if set, else kernel.
  • Any behavioral field (today: setup) — kernel only. User-defined types ship no code, so this isn't a regression from the user's side; it's an architectural invariant.

Why not a separate typeOverridesFacet

Tempting — the "clean Tana" answer is a parallel facet keyed by target type id. But it adds a service, a publish path, and a rebuild step for one anticipated use case. The single-block-type-with-extends shape is strictly less surface; if overrides grow real complexity later (per-type render overrides, chained setups, etc.) extracting the facet is mechanical. Don't pre-extract.

Replace setup with same-tx processors

setup (docs/type-system.md:60-89) was designed before same-tx processors existed (src/data/api/sameTxProcessor.ts). With same-tx processors now a real primitive, setup is redundant.

What setup provides today:

  • Fires inside repo.addType after initialValues are written, in the same tx.
  • Use cases: child-block templates (meeting → Attendees / Agenda / Action items), computed initial values (due = now() + 7 days), cross-block wiring at add time.

What a same-tx processor watching the properties field provides:

  • Sees before and after BlockData per touched row.
  • Writes via ctx.tx.update/setProperty, amending the same tx; one undo entry covers everything.
  • Rejects via throw new ProcessorRejection(code, meta), rolling the user's tx back atomically.
  • Already has access to merged propertySchemas on ctx.

Lifecycle: addType before vs after

Sketch of a per-type processor

// src/plugins/meeting/typeOnAdd.ts
import {createChild} from '@/data/internals/kernelMutators'
import {addedTypes} from '@/data/properties'

export const meetingOnAdd = defineSameTxProcessor({
  name: 'meeting.onAdd',
  watches: {kind: 'field', table: 'blocks', fields: ['properties']},
  apply: async (event, ctx) => {
    for (const row of event.changedRows) {
      if (!addedTypes(row).includes('meeting')) continue

      // init-if-missing for type-defined computed initial values
      if (row.after.properties[meetingDateProp.name] === undefined) {
        await ctx.tx.setProperty(row.after.id, meetingDateProp, today())
      }

      // child-block template
      for (const label of ['Attendees:', 'Agenda:', 'Action items:']) {
        const childId = await ctx.tx.run(createChild, {
          parentId: row.after.id,
          position: {kind: 'last'},
        })
        await ctx.tx.update(childId, {content: label})
      }
    }
  },
})

// Registered alongside the type contribution:
sameTxProcessorsFacet.of(meetingOnAdd, {source: 'meeting'})

Sync-apply gap

Local repo.tx writes (orchestrated, raw, plan) all flow through the commit pipeline at src/data/internals/commitPipeline.ts:263, which is where same-tx processors fire. PowerSync-applied writes from another device take a different path entirely: they land in SQLite via the CRUD-apply mechanism, never set tx_context.source, and surface to the rest of the app via the row_events tail (src/data/internals/rowEventsTail.ts:1-32) which updates the cache and fires invalidations but does not run same-tx or post-commit processors.

So replacing setup with a same-tx processor doesn't introduce a sync-apply gap — setup has exactly the same gap (it only runs inside repo.addType, which is local). But it doesn't close it either, which is what the earlier "every write uniformly" framing implied.

The honest comparison:

  • Local raw tx.update / addBlockTypeToProperties / plan code: setup misses these; same-tx processor catches them. Strictly better.
  • Sync-applied writes from another device: both miss. Equal.

For behaviors that must run on cross-device type changes (e.g., maintaining a derived side-index for typed queries when another client tags a block), the right tool is a separate processor wired to the row_events tail — the same mechanism parseReferences already uses for sync-applied content changes (src/plugins/references/referencesProcessor.ts:230). That's out of scope for replacing setup, but worth flagging: "type-add behavior" splits into two kinds.

  • Atomic with the originating write (child templates, computed initial values, rejecting writes that violate type-level constraints) → same-tx processor. Local-only by construction.
  • Eventually-consistent (side-index maintenance, downstream notifications, cleanup) → post-commit processor on the row_events path. Catches sync apply too.

Most current setup use cases (templates, computed initials) are the first kind — users adding types from another device shouldn't get child templates materialized retroactively without their consent. The sync-apply gap is the right behavior for that class. The second kind didn't have an analog in setup anyway.

What needs to be added

One small ergonomic piece — a delta helper so every processor isn't reimplementing the diff:

// src/data/properties.ts (additions)
// ChangedRow.before is null for inserts; ChangedRow.after is null
// for hard-deletes (`src/data/api/processor.ts:70-73`). Both helpers
// must accept either null without throwing, otherwise any processor
// using them aborts insert / hard-delete txs.
export const addedTypes = (row: ChangedRow): readonly string[] => {
  const before = row.before ? getBlockTypes(row.before) : []
  const after  = row.after  ? getBlockTypes(row.after)  : []
  return after.filter(t => !before.includes(t))
}

export const removedTypes = (row: ChangedRow): readonly string[] => {
  const before = row.before ? getBlockTypes(row.before) : []
  const after  = row.after  ? getBlockTypes(row.after)  : []
  return before.filter(t => !after.includes(t))
}

With the helpers null-safe, the example processor above is also safe on hard-delete events: a deleted block returns addedTypes(row) === [] for any type id, the loop body skips, and row.after is never dereferenced. Processors that genuinely care about removal switch on removedTypes(row) instead.

What repo.addType becomes

Without setup in the picture, the orchestration shrinks to: encode initialValues through their schemas' codecs, append the type id to typesProp. That's mutator-shaped work; it might even fold into a regular registered mutator. The _types retention on Repo stays useful for the property-panel section grouping (§3c in type-system.md) and the "is this id registered?" guard, but setFacetRuntime's _types snapshot is no longer load-bearing for behavior — behavior is in the processor stream now.

Migration moment

Kill setup now, before existing kernel use cases migrate into it. Today the only real user is the deferred daily-note template (type-system.md:678), and it's still being designed. Once daily-note's setup is written, the rewrite has cost; right now it doesn't. Same reasoning that landed user-defined properties before too many ad-hoc adhocSchema call sites were live.

Roam isa adoption

The import already does the survey work. src/plugins/roam-import/typeCandidates.ts walks roam:isa values per imported block, groups by alias, ranks by occurrence count and common-property frequency, and emits a report: 

[[Person]] -> type "person" (53 nodes); common props: dob 41/53 (77%), email 38/53 (72%)
[[Project]] -> type "project" (12 nodes); common props: status 10/12 (83%), owner 9/12 (75%)

What's missing is promotion — the action that turns a Roam page (which is currently the target of roam:isa refs) into a real block-type block and re-tags the referencing blocks.

The promotion primitive

One action, two call sites:

async function promoteToType(args: {
  targetBlockId: string                  // the Roam page that's the isa target
  label: string                          // pre-filled from the page's alias
  propertySchemaIds: readonly string[]   // chosen from the candidate report's common-prop list
  rewriteIsaReferences?: boolean         // default false — leave roam:isa intact for review
}): Promise<void>

Order matters here. repo.addTypeInTx verifies the type id is in the merged registry and throws on unknown ids (docs/type-system.md §3a, [addType] type id ... is not registered); the brand-new targetBlockId isn't in the registry yet at the moment we want to retag instances with it, because UserTypesService's subscription publishes the contribution after the tx commits. The service needs the same synchronous-append escape hatch UserSchemasService has (appendUserSchema at src/data/userSchemasService.ts:167) — call it appendUserType(contribution, blockId), same shape, updating the user-data bucket on typesFacet synchronously via setRuntimeContributions.

With that primitive in place, plus a symmetric removeUserType(typeId) for cleanup:

  1. Before opening the tx — build the TypeContribution from the action args (id = targetBlockId, label, properties = resolved schemas for propertySchemaIds), then userTypesService.appendUserType(contribution, targetBlockId). repo.types now includes targetBlockId; the next repo.snapshotTypeRegistries() (taken inside the tx) sees it.
  2. Still before the tx — resolve the target's workspace and prefilter the instance set. Read the target block to capture its workspaceId, then call repo.queryBlocks({workspaceId, referencedBy: {id: targetBlockId, sourceField: ROAM_ISA_PROP}}) to get the candidate instance ids. The block_references edge index from type-system.md §6b answers this in one EXISTS. Two reasons this runs outside the tx (see callout):
    • repo.queryBlocks reads via ctx.db.getAll, not the active write-tx lock context. Running it inside repo.tx reintroduces the PowerSync queue deadlock documented in tasks/processor-tx-deadlock.md and called out at docs/follow-ups.md:19-21.
    • Explicit workspaceId avoids queryBlocks falling back to activeWorkspaceId, which can differ from the target's workspace during background flows / import runs / UI workspace switches and would silently mis-scope the retag set.
  3. Open the tx under a try/catch. repo.addTypeInTx(tx, targetBlockId, BLOCK_TYPE_TYPE, {...}) on the Person page, materializing block-type:label and block-type:properties.
  4. For each prefiltered instance id, re-read in-tx via tx.get(instanceId) (cheap; the writer slot is already held) and skip if the row is gone or no longer carries the roam:isa ref to the target — the TOCTOU narrow guard. For survivors, repo.addTypeInTx(tx, instanceId, targetBlockId, {}) appends the new type id to block:types. The id resolves because step 1 already registered it. Any sameTxProcessor watching the field fires per row.
  5. If rewriteIsaReferences: rewrite the roam:isa refList on each instance, dropping the promoted alias. Default off — users typically want both for a while.
  6. On tx failure: in the catch arm, call userTypesService.removeUserType(targetBlockId) explicitly, then rethrow. Don't rely on the next subscription rebuild to clean up — see callout below. 
const target = await repo.load(targetBlockId)
const workspaceId = target.workspaceId
const candidates = await repo.queryBlocks({
  workspaceId,
  referencedBy: {id: targetBlockId, sourceField: ROAM_ISA_PROP},
})

// Capture the prior user-data contribution for this id (if any).
// Restored on tx failure rather than blindly removed — see
// "Rollback must preserve prior registration" callout.
const prior = userTypesService.peekContribution(targetBlockId)
userTypesService.appendUserType(contribution, targetBlockId)
try {
  await repo.tx(async tx => {
    // In-tx guard: the target may have been deleted concurrently
    // between our pre-tx load() and the tx opening. addTypeInTx is a
    // no-op when tx.get(blockId) returns null (src/data/repo.ts) so
    // without this guard the loop below would happily tag N instances
    // with a type id whose backing block doesn't exist. Throwing here
    // aborts the tx and triggers the rollback path below.
    const targetInTx = await tx.get(targetBlockId)
    if (!targetInTx || targetInTx.deleted) {
      throw new Error(`promoteToType: target ${targetBlockId} no longer exists`)
    }
    await repo.addTypeInTx(tx, targetBlockId, BLOCK_TYPE_TYPE, {...})

    for (const candidate of candidates) {
      const row = await tx.get(candidate.id)
      if (!row) continue                          // deleted between prefilter and tx
      const isaRefs = row.properties[ROAM_ISA_PROP] ?? []
      if (!resolveRoamIsaTargets(isaRefs).includes(targetBlockId)) continue   // no longer references target
      await repo.addTypeInTx(tx, candidate.id, targetBlockId, {})
    }
    if (rewriteIsaReferences) /* ... */
  })
} catch (err) {
  if (prior) {
    userTypesService.appendUserType(prior.contribution, prior.blockId)   // restore
  } else {
    userTypesService.removeUserType(targetBlockId)                       // drop provisional
  }
  throw err
}

In-tx existence guard for the target

repo.addTypeInTx(tx, blockId, ...) silently no-ops when tx.get(blockId) returns null or a tombstone — reasonable behavior on sync-applied deletes where another device removed the row, but a footgun for orchestration code that's about to fan out work tied to the target. Without the explicit tx.get + deleted check, a concurrent delete between the pre-tx load() and the tx opening would let the loop persist targetBlockId on every candidate's block:types while the target itself stays a non-block-type tombstone. The next runtime rebuild then drops the provisional registration, leaving orphan ids on N rows.

Rollback must preserve prior registration

A retry on an already-promoted target (e.g. user re-runs promotion with a different property set) has peekContribution(targetBlockId) returning the existing user-data contribution. An unconditional removeUserType on tx failure would wipe that legitimate registration, breaking every block currently tagged with this id until the next subscription tick reconciles. Capture-and-restore is the right primitive: peek → append → on-failure-restore-or-drop.

Why the query lives outside the tx

repo.queryBlocks and other typed-query reads route through ctx.db.getAll on the read DB handle, not through tx's write-transaction lock. Under PowerSync's serialized single-connection config, holding a writer slot while issuing a bare DB read deadlocks the queue — the read can't progress until the writer releases, and the writer is blocked on the read. The kernel's parseReferences and cleanupOrphanAliases processors paid the same cost and moved their broad reads outside the write tx (docs/follow-ups.md:19); the promotion flow follows the same shape.

The TOCTOU window opened by reading outside the tx is narrow and the in-tx re-read closes it: instances may be deleted or have their roam:isa rewritten between the prefilter and the tx, which the in-tx tx.get + property check catches. Conversely, instances that gained an isa ref during the window are silently missed by this promotion — conservative miss is fine; the user can re-run promotion.

Why explicit rollback — subscription rebuild isn't enough

The subscription that reconciles user-data type contributions from blocks fires only on block writes. A tx that fails before writing any block-type block (validation error, mid-tx exception, abort signal) doesn't trigger a subscription tick, so the provisionally-registered targetBlockId persists in the runtime indefinitely. During that window repo.addTypeInTx(_, _, targetBlockId, {}) from any other caller would succeed, attaching an orphan type id to unrelated blocks.

Explicit removeUserType on the catch arm closes the window. The asymmetry vs. addSchema (which appends after commit and doesn't need rollback) is justified by the in-tx dependency: step 4's addTypeInTx calls need the registration to be visible before the tx commits, so the registration has to precede the writes that depend on it.

Two surfaces use the primitive:

  • Auto-suggest at import time. The existing type-candidate report (src/plugins/roam-import/report.ts:492) gets an "Adopt as type" button per candidate. Property checkboxes default to the common-prop list. Same code path.
  • After-the-fact. "Promote this page to a type" action on any page that's a roam:isa target. Surfaced through the same action infrastructure that handles other context-actions on pages.

Phasing

Each phase ships independently. Earlier phases don't depend on the deferred follow-ups in type-system.md.

Phase 4 — Roam isa adoption

  • promoteToType action in src/plugins/roam-import/.
  • UI: "Adopt as type" button per high-confidence candidate in the import report; column of property checkboxes pre-checked at ≥50% frequency.
  • Standalone action on any page registered as a roam:isa target.
  • Optional rewriteIsaReferences flag; default off.

Acceptance: on a Roam import with 53 blocks tagged is a [[Person]], promoting Person turns the page into a block-type and re-tags all 53 instances; the Person panel section in the property panel reflects the chosen schemas.

What's deferred

  • Type templates as data. A subtree to materialize on instance creation, declared on the type-definition block via a block-type:template refList. Once the same-tx processor pattern is established, a single generic typeTemplateProcessor can dispatch from data. Defer until at least one type wants this in practice.
  • Type inheritance for user types. A user type extending another user type rather than a kernel type. The override merge already handles "child extends parent" semantics; only the resolver loop guard (cycle detection) is missing.
  • Required-field validation on add. A type-definition block lists which of its properties[] are required; a same-tx processor rejects writes that leave them unset. Easy once Phase 3 lands.
  • Property-schema authoring inline with the type. Today the user creates schemas on the Properties page, then refs them from a type on the Types page. A combined "type editor" surface that creates schemas as side-effects of editing a type is ergonomic but not necessary for v1.
  • Bulk reimport-time auto-promotion. The import currently suggests; this design's promotion stays user-triggered. Heuristic-based auto-promotion at import time is plausible but a separate UX call.

Decisions locked in

  • User-defined types are blocks of kernel type block-type, hosted on a per-workspace Types page.
  • Type id is the block id. Display label is block-type:label. The picker shows the label; block:types arrays carry ids.
  • Override existing types via block-type:extends; merge at the rebuild step, never destructive.
  • Behavioral fields (today: setup) cannot come from data. The merge rule is "kernel only" for those.
  • Replace setup with same-tx processors for atomic per-add behavior. Type contributions become pure metadata. Do this before any current kernel use lands. Behaviors that must run on sync-applied type changes from other devices use a separate post-commit processor on the row_events tail (same path parseReferences uses).
  • Roam isa adoption rides one primitive (promoteToType) usable from import time or after-the-fact.
