Video notes view as a navigation-native, pane-scoped view mode

The video player's notes view is a viewport-fixed modal overlay invisible to history and layout serialization — this design turns it into a pane-scoped view mode carried by the URL, so Back/Forward and saved perspectives just work — and playback position survives.

1 Problem

The video player's notes view (video left, editable child notes right — entered via the hover button or $mod+Shift+n) is a modal takeover, not a place you navigate to:

  • It renders as a viewport-fixed overlay (fixed inset-0 z-50, VideoPlayerRenderer.tsx:283) painted over the header, sidebar, and every other pane — regardless of which pane the video block actually lives in.
  • It is driven by video:playerView, a ChangeScope.UiState property on the video block itself (view.ts:10-14). That has three consequences:
    • Invisible to navigation. The hash-route ↔ panel-layout projection (panelLayoutProjection.ts) never sees it. Entering/leaving notes view creates no history entry: browser Back while in notes view doesn't exit the view — it navigates the layout underneath the overlay.
    • Not part of the layout, so layout round-trips destroy it. Anything that reconciles panel rows (switching to another saved layout/perspective and back, following a link and pressing Back, restoring a session) rebuilds panes from LayoutSlots, which encode only which block a pane shows — never how. The overlay disappears or reappears at the wrong time, and because the pane content remounts, the <video> element is torn down: playback stops and the position is lost.
    • Wrong scope. UiState writes sync across devices (changeScope.ts:55-59, see the correction note in navigation-redesign.md), so toggling notes view on one device flips it for the same block everywhere it renders — on every device and in every pane.

What it should be instead: a pane-scoped view mode that participates in navigation. Entering notes view is a navigation step (Back exits, Forward re-enters); the mode is part of the pane's identity in the layout, so any layout snapshot/restore (saved perspectives, session restore, URL sharing) brings it back exactly; and playback survives the round-trip — ideally the video keeps playing while you're away and is still playing when you return.

2 Current implementation (facts)

  • videoPlayerViewProp: 'default' | 'notes' + videoNotesPaneRatioProp — src/plugins/video-player/view.ts.
  • The notes layout is a blockLayoutFacet contribution gated on ctx.block.id === ctx.blockContext.videoPlayerBlockId (VideoPlayerRenderer.tsx:356-364); VideoPlayerLayout early-returns DefaultBlockLayout unless view === 'notes' (:263-264) and otherwise renders the fixed overlay, mounting Shell so the video block keeps its shortcut surface (:343).
  • Enter/exit paths: hover button → enterVideoNotesView (notes.ts:67-74, sets the prop + ensures an editable child); video.toggle_notes_view (actions.ts:136-157) reads/writes the prop; close button calls setView('default') (VideoPlayerRenderer.tsx:333).
  • Player handles are registered per (renderScopeId, blockId) in registry.ts; video.insert_timestamp / video.toggle_focus resolve the player through it. Toggling the view swaps VideoPlayerLayout's root element type (DefaultBlockLayout vs Shell, :264/:343), so React remounts VideoPlayerContentRenderer — the <video> is torn down and playback position is lost even on today's plain enter/exit toggle. The actions keep working only because the new mount re-registers under the same (renderScopeId, blockId) key (:101-111).
  • One pre-existing weakness this design makes first-class: at panel top level, renderScopeId is a pure function of the block id (outlineRenderScopeId(topLevelBlockId), PanelRenderer.tsx:188, renderScope.ts:6-7) — so two panes showing the same video share one registry key and the last mount wins; timestamp insertion and focus toggle resolve to whichever pane registered last. See §4.3.
  • Layout substrate (from the navigation redesign, navigation-redesign.md): panel rows are blocks under ui-state/layout-sessions/<sessionId>, each carrying topLevelBlockIdProp / focusedBlockLocationProp / scrollTopProp (panelLayoutProjection.ts:203-229). PanelLayoutProjection projects rows → hash (pushState) and hash → rows (reconcilePanelRows, an LCS diff) on hashchange/popstate. The hash grammar (routing.ts:1-14) is #<workspaceId>/<slot>/... where a slot is a bare block id or a stack (s:a,b). Query params after ? are carried across rewrites (preserveHashQueryParams) but are not compared by sameLayoutSlots.

3 Design goals

  1. History-native: entering notes view pushes a history entry; browser Back exits to the plain block view; Forward re-enters.
  2. Layout-native: the mode is encoded in the layout (slot grammar + panel row), so any consumer of layout serialization — session restore, saved layouts/perspectives (including downstream extensions that snapshot slots via buildLayoutFromSlots), shared URLs — restores it for free.
  3. Pane-scoped: the mode belongs to one pane, not to the block. Two panes can show the same video, one in notes view; another device is unaffected.
  4. Playback continuity: returning to the notes view (Back/Forward, perspective switch and back) does not lose playback position; stretch goal: the video keeps playing while its pane is away and is still playing on return.
  5. No regression to the in-view features: timestamp insertion, focus toggle, resizable split, note auto-creation, shortcut surface.

Non-goals (this change)

  • True chrome-hiding "theater mode" routes (header/sidebar suppression). The pane-scoped view fills the pane; whole-layout immersion is the companion maximize-pane gesture (§4.4), not a video feature.
  • Durable cross-restart playback position (would mean sync writes per pause; revisit separately).
  • A general per-pane renderer-override system. We add a minimal view mode channel; generalizing to arbitrary renderer selection stays open (§8).

4 Proposed design

4.1 Pane view mode in the layout substrate

New property on panel rows (same family as topLevelBlockIdProp):

// src/data/properties.ts (or panel-scoped module)
export const panelViewModeProp = defineProperty<string | undefined>(
  'panel:viewMode',
  {codec: codecs.optionalString, defaultValue: undefined, changeScope: ChangeScope.UiState},
)

Semantics: an opaque mode id, owned by whichever plugin recognizes it (here: 'video-notes'). Unset = default rendering. It lives on the panel row, so it inherits the panel substrate's properties: per-tab layout session, restored on reload, reconciled by the projection, deleted with the pane.

4.2 Slot grammar: leaf annotations — <blockId>(~<annotation>)*

Extend the leaf token with a ~-separated annotation list: #ws/a/b~video-notes/c (stacks unchanged; a stack child is a leaf and can carry annotations). Two annotation kinds, sharing one channel:

  • a view mode — a plugin-owned opaque id (video-notes), at most one per leaf, projected from panelViewModeProp;
  • max — a core-owned reserved annotation for the maximize-pane gesture (§4.4), projected from panelMaximizedProp.

~ is the separator because it's in RFC 3986's unreserved set — nothing percent-encodes it (encodeURIComponent leaves it intact), it cannot appear in a UUID, and it doesn't collide with the grammar's reserved / ( ) : ,. (! is a sub-delim that defensive encoders sometimes escape; . is the likeliest char to appear in future non-UUID ids.)

Plumbing:

  • parseSlot (routing.ts:83-96): split leaf tokens on ~; LayoutSlot leaf gains optional viewMode?: string and maximized?: boolean.
  • buildLayoutSlot (routing.ts:117-120): append annotations when set (stable order: mode first, then max).
  • layoutSlotsFromRows (panelLayoutProjection.ts:90-112): read panelViewModeProp / panelMaximizedProp off leaf rows into the slot.
  • sameLayoutSlots (panelLayoutProjection.ts:81-88): compare annotations — this is what makes a mode/maximize change count as a layout change, driving both directions of the projection.
  • reconcilePanelRows (panelLayoutProjection.ts:405-…): when a row is reused for a leaf with the same blockId but different annotations, set the properties on the existing row rather than delete/create — this keeps the panel id (and therefore the pane's React subtree, focus, scroll, and panel history) stable across toggles. The plan-matching step needs no change: planReconciliation/lcsMatches (panelLayoutProjection.ts:133-201) already match purely on block ids, so an annotation-only diff maps the same row to the same-id leaf as-is. The whole reconcile delta is the sameLayoutSlots gate, the prop writes on reused rows, and seeding new rows in createPanelRowInTx.

Result: toggling an annotation writes the panel prop → projection pushes a new hash (history entry). Back/Forward → popstate → reconcilePanelRows → prop update → layout re-renders. Goals 1–3 fall out of the existing machinery; no new sync or storage concepts.

Mode lifetime — cleared on in-pane navigation

A view mode belongs to the (pane, block) pair, not the pane alone: when writePanelContent / navigateInPanel (panelHistory.ts:219-270) changes the pane's top-level block, panel:viewMode is cleared in the same tx (otherwise the next block renders under a mode it doesn't own and the hash reads B~video-notes). So that panel-level Back can restore it, VisitState — explicitly designed extensible (panelHistory.ts:35-41) — gains viewMode?: string: the snapshotter PanelRenderer registers captures the pane's current mode, and restore writes it back. Without this, browser-Back (hash-driven) and chevron-back (panelHistory-driven) would diverge — exactly the navigation-split this design exists to remove. Precedence: on a URL-driven reconcile, the slot annotation is authoritative for the mode; VisitState.viewMode is consulted only by panel-level chevron back/forward. (reconcileUrlNavigation matches history entries by blockId alone, panelHistory.ts:127-147 — a multi-step browser jump can land on a hash entry whose matched VisitState carries a mode captured at a different time, and the restore must not overwrite what the hash says.) max is not cleared on in-pane navigation: maximize is pane-arrangement state, independent of which block the pane shows.

4.3 Video plugin changes

  • VideoPlayerLayout gates on the panel's mode instead of the block prop: read blockContext.panelId → usePropertyValue(panelBlock, panelViewModeProp) === 'video-notes', and only when the block is the pane's top-level block (a video embedded mid-outline shouldn't take the pane over; entering notes view on a nested video first navigates the pane to that block and sets the mode).
  • Navigate+mode must be one tx, via an explicit API. navigateInPanel owns its whole tx (panelHistory.ts:244-269) with no seam for extra writes, and a two-tx fallback (navigate, then set mode) is nondeterministic — the projection's subtree subscription may coalesce the two writes into one pushState or deliver two, so Back would sometimes exit to plain-V and sometimes skip it. PR 1 extends the seam: navigateInPanel(panelBlock, blockId, {viewMode}) threading into writePanelContent, whose clearing rule becomes "set panel:viewMode to the incoming option (default: unset)" — clear and set are the same write.
  • Note auto-creation stays gesture-only. With the mode in the layout, it can now arrive with no user gesture — Back/Forward, a shared V~video-notes URL, session restore, a layout-snapshot apply. The render path must never write: ensureEditableVideoNoteChild is a durable, undoable, synced document mutation (repo.mutate.createChild), and navigation must not mutate the document. Only the enter gestures (hover button, video.toggle_notes_view) call it; a notes pane whose video has zero children renders an empty-state affordance (click / Enter creates the first note) instead.
  • Close semantics after a nested-video enter. Enter-on-nested-video navigates the pane P→V; a close that merely clears the mode would strand the pane zoomed into V (today's overlay close reveals P underneath). Close therefore goes back (goBackInPanel) when the pane's previous history entry is the enter transition it created, and just clears the mode when the mode arrived without a navigation (URL, session restore, snapshot apply).
  • Enter/exit (enterVideoNotesView, the toggle action, the close button) write panelViewModeProp on the panel row instead of videoPlayerViewProp. videoPlayerViewProp is deleted; stale video:playerView values on blocks are simply ignored (it was UiState, nothing durable depends on it).
  • Presentation: pane takeover, not viewport takeover. The notes layout drops fixed inset-0 z-50 for absolute inset-0 within the pane (the pane is relative). With a single pane this looks nearly identical (header stays visible); with splits, other panes stay usable — which is what "notes view in this pane" means. This is a deliberate behavior change and called out in the upstream issue.
  • Shortcut surface, videoPlayerBlockId context, timestamp/focus actions, and the pane-ratio pref are untouched: the same VideoPlayerRenderer wraps the block and the layout still mounts Shell.
  • Per-pane player identity (fixes the §2 registry collision). Goal 3 ("two panes, same video, one in notes view") makes the shared-scope collision first-class, so PR 1 threads the panel id into the top-level render scope (outlineRenderScopeId gains a panel discriminator at the PanelRenderer call site) — registry handles, and therefore video.insert_timestamp / video.toggle_focus, resolve per pane. Without this, goal 3 only holds for different videos.

4.4 Companion: generic maximize-pane gesture (core)

Making the notes view pane-scoped removes today's accidental "fullscreen": in a split layout the video now gets only its pane. The immersion story is deliberately not video-specific — it's a generic core gesture, riding the same annotation channel (decision from design discussion; alternatives were native requestFullscreen() — transient, no chrome, not navigation state — and leaving it to downstream window-management extensions).

  • Substrate: panel:maximized boolean prop on the panel row (UiState), projected ↔ the ~max leaf annotation exactly like viewMode (§4.2). Keeping it a row prop (not a layout-session-level pointer like activePanelIdProp) keeps the projection a pure function of rows. Invariant — at most one maximized leaf — has a single writer: the toggle action clears any other row's flag in the same tx. Reconcile writes whatever the hash says (a multi-max hash can only be hand-crafted; outbound rebuilds it stably, no loop), and the renderer tolerates it by rendering the first maximized row.
  • Rendering: LayoutRenderer already has a single-pane path — on mobile it renders only the active (or fallback) panel (LayoutRenderer.tsx:169-175). Maximize is the desktop analog: if a visible row is maximized, render only that pane. Sibling rows are untouched in the substrate, so un-maximize restores the exact arrangement (nothing was closed). On mobile the flag is ignored (already single-pane).
  • The maximized pane must be (or become) the active panel. ~max also arrives inbound with no gesture — Back/Forward, shared URL, snapshot apply — and reconcilePanelRows sets the row flag without touching activePanelIdProp. Today a stale active pane is at worst un-highlighted but visible; maximize is the first configuration where the active panel can be invisible while keyboard dispatch still targets it (keys landing in a pane the user cannot see). The existing repair effect (LayoutRenderer.tsx:179-186) only fires when activePanelId is unset — PR 2 adds the symmetric coercion: when a maximized row renders alone, activePanelIdProp is coerced to it.
  • Gesture: a layout.toggle_maximize_panel action targeting the active panel, bindable + surfaced in panel chrome (binding choice left to upstream).
  • Semantics: maximizing is a slot change → history entry → Back un-maximizes, consistent with the notes-view story. Closing the maximized pane deletes its row, flag included. In-pane navigation keeps maximize (see §4.2 — it's arrangement state, not (pane, block) state). Notes view does not auto-maximize; the two annotations compose (b~video-notes~max) and the user stacks gestures.

Ships as its own PR after the annotation grammar lands (§6 sequencing) — pure core, no video-plugin coupling.

4.5 Playback continuity

Two stages; the first ships with the main change, the second is a follow-up.

Stage A — position restore (cheap, robust). A module-level in-memory map Map<blockId, {currentTime: number, playing: boolean, at: number}> in the video plugin. VideoPlayerContentRenderer records into it on pause/rate/seek events and on unmount (via the existing player.current ref); on mount, if an entry is fresh, seek to currentTime and, when playing was true, resume. In-memory only — no UiState writes (playback position is high-frequency, device-local, and worthless cross-device), no persistence across reload. This makes Back/Forward and perspective round-trips lossless up to a brief remount gap — and it is load-bearing for the plain notes toggle too: entering/leaving notes view swaps the layout root type and remounts the player (§2), so today even the in-place toggle drops position. Two honest limits: blockId keying means the same video live in two panes restores whichever pane unmounted last (accepted — positions are conceptually per-video, and the dual-pane case is rare); and auto-resume is best-effort under autoplay policies — Back/Forward rides recent user activation, but e.g. a keyboard-driven snapshot apply may not, degrading to paused-at-position.

Stage B — keep-alive (video keeps playing while away). No precedent in the codebase today (no <Activity> usage, portals only for floating UI), so this is deliberately separated. Sketch:

  • A parked-player host: a component mounted once at app level (same seam as DialogHost) owns a hidden "parking lot" element and a module map Map<blockId, HTMLDivElement> of stable container divs. It renders each active ReactPlayer via createPortal into its container once; the portal target never changes, so React never remounts the <video>.
  • Panes adopt the container DOM node on mount and return it to the parking lot on unmount. Honest platform limits: element.moveBefore() is state-preserving by spec but Chromium-only today; the appendChild fallback keeps a playing native <video> alive (the spec's remove-pause task is skipped on synchronous reinsertion) but reloads iframe-based sources unconditionally — and ReactPlayer renders YouTube / Vimeo / etc. as iframes. So true keep-alive is scoped to native-<video> sources or Chromium; everywhere else Stage B degrades to Stage A (seek-and-resume), which must therefore ship first and stand alone.
  • Retention policy (must be bounded): a parked container is kept only while its video playing === true, plus a short grace period (~30s) for paused ones; otherwise the player is released and Stage A's position map takes over. At most N (say 2) parked players; oldest parked container evicted (pause → record position → release) — a currently-adopted, visible container is never evicted out from under its pane.
  • Ownership moves. Once the host owns the ReactPlayer, the handle registration, player ref, and isPlaying state move with it — the registry entry is fed by the host, not the pane renderer. Stage B is a re-plumbing of VideoPlayerContentRenderer, not an overlay on it.
  • Multi-pane same video: the registry already anticipates one block live in several render scopes. The adopted live element goes to one pane (last-requested); additional simultaneous panes get ordinary fresh players.

Stage B is the only part with real platform risk (browser quirks around moving media elements), which is why Stage A must be able to stand alone.

5 Alternatives considered

  • Hash query param (#…?video-notes=<panelId>): params already survive rewrites (preserveHashQueryParams), but sameLayoutSlots ignores them, so inbound reconciliation and history comparison wouldn't fire without special cases; panel ids in a shared URL are meaningless on another device. The slot is where per-pane display state belongs. Rejected.
  • Keep the block prop, add a history shim (manual pushState on enter + popstate listener): fights the projection (which owns the hash), invisible to layout serialization (perspectives/session restore still lose it), still block-global and cross-device. Rejected.
  • rendererProp override / a dedicated "notes view" renderer id: rendererProp is ChangeScope.BlockDefault — a synced, undoable document property; flipping it changes the block for everyone, everywhere, and lands in the undo stack. Also renderer selection can't express "only in this pane". Rejected.
  • React <Activity> around panes for keep-alive: perspective/layout switches delete panel rows, so the pane subtree is genuinely gone — you'd have to keep every historical layout mounted hidden. Way out of proportion; the parked-player host scopes retention to exactly the expensive resource (the media element). Rejected for now.

6 Implementation sketch

Sequenced as two PRs: PR 1 = annotation grammar + viewMode + video plugin (+ VisitState); PR 2 = maximize-pane (pure core, builds on the grammar from PR 1).

PR 1 — core (small, generic):

  • routing.ts: LayoutSlot leaf annotation viewMode?, parseSlot/buildLayoutSlot; grammar comment reserves the max token (the maximized? field itself ships with PR 2 — no dead code ahead of its consumer).
  • properties.ts (or alongside panel props): panelViewModeProp.
  • panelLayoutProjection.ts: layoutSlotsFromRows, sameLayoutSlots, reconcilePanelRows learn the annotation (plan-matching untouched, §4.2).
  • panelHistory.ts: navigateInPanel(…, {viewMode}) → writePanelContent sets/clears panel:viewMode in the same tx; VisitState.viewMode capture/restore (slot annotation authoritative on URL-driven reconcile, §4.2).
  • renderScope.ts / PanelRenderer.tsx: panel discriminator in the top-level render scope (§4.3, per-pane player identity).

PR 1 — video plugin:

  • view.ts: drop videoPlayerViewProp (keep videoNotesPaneRatioProp).
  • notes.ts / actions.ts / VideoPlayerRenderer.tsx: enter/exit/gate via the panel prop; gesture-only note creation + empty-state affordance; close-goes-back semantics (§4.3); overlay → pane-scoped absolute layout.
  • New playbackState.ts (Stage A map) wired into VideoPlayerContentRenderer.

PR 2 — maximize:

  • panelMaximizedProp + LayoutSlot.maximized?; ~max projection; single-maximized invariant in the toggle action (sole writer); LayoutRenderer desktop single-pane path + active-panel coercion (§4.4); layout.toggle_maximize_panel action + panel-chrome affordance.

Tests:

  • routing.test: parse/build round-trip of annotated leaves (mode, max, both), incl. inside stacks; ids containing no ~ unaffected.
  • panelLayoutProjection.test: annotation-only change is a slot change (projection pushes); inbound annotation-only diff updates props on the same panel row (id stable); annotations cleared when the hash has none.
  • panelHistory.test: mode cleared on in-pane navigation; chevron-back restores it via VisitState; navigate+mode is a single history entry; slot annotation beats a stale VisitState.viewMode on URL-driven reconcile.
  • Video plugin: toggle action writes panel prop; layout gates on it; Stage A restore (seek + resume) on remount; no document write on a navigation-driven mode arrival (zero-child video, mode set via reconcile → no child created); per-pane handle resolution with the same video in two panes.
  • LayoutRenderer: maximized row renders alone on desktop (first wins when several are flagged); flag ignored on mobile; un-maximize restores siblings; maximized row coerced to active panel.

7 Migration / compat

  • Old URLs (no ~) parse unchanged. On an old client, a new-grammar URL degrades concretely as: parseSlot reads the whole token as a block id, reconcilePanelRows persists a panel row whose topLevelBlockIdProp is the literal V~video-notes, and the pane renders blank until navigated away — no crash, but a persisted dead pane in that layout session. The realistic "old client" is a lagging PWA install (iOS caches aggressively), not a hypothetical.
  • video:playerView values left on blocks are ignored dead data (UiState scope; safe to leave, cheap to sweep later).
  • Downstream code that round-trips layouts as slots or strings (layoutSlotsFromRows → buildLayoutFromSlots → parseLayout → reconcilePanelRows) carries annotations with no rebuild at all — the slot objects/tokens pass through opaquely. Only code that reconstructs bare {kind: 'leaf', blockId} literals from block ids drops annotations and needs touching.
