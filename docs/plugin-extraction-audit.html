Core → Plugin Extraction Audit

Candidates currently sitting in src/extensions/, src/markdown/, src/shortcuts/, and src/components/ that look pluggable. Sorted by expected upside vs. effort.

High confidence — clean seam, real upside Medium — needs a small refactor Possible — but low value alone Don't extract — too foundational

High confidence

1. Wikilinks & block-refs markdown

High Effort: S
Now lives at
src/markdown/wikilinks/, src/markdown/blockrefs/
Does
Parses [[page]] wikilinks and ^(block-id) embeds; renders the React components.
Why
Already half-extracted — the only place these extensions get registered is src/plugins/backlinks/index.ts (imports wikilinkMarkdownExtension / blockrefMarkdownExtension). Core doesn't depend on them; backlinks does. They belong with their consumer.
Coupling
One leak: RefPropertyEditor.tsx imports BlockEmbed & BlockRefAncestorsProvider. Either move RefPropertyEditor alongside, or expose a tiny embed primitive from core and keep the cycle-guard provider in the plugin.

2. Breadcrumb renderer & header

High Effort: S
Now lives at
src/components/Breadcrumbs.tsx, src/components/BreadcrumbList.tsx, src/components/renderer/BreadcrumbRenderer.tsx, registered in src/extensions/defaultRenderers.tsx:23,35-38
Does
Renders parent hierarchy above each top-level block; registers a 'breadcrumb' renderer id.
Why
Pure navigation chrome. Already self-gates on isTopLevel. Contributes via blockHeaderFacet — the exact public seam plugins use. Nothing in core dispatches to the 'breadcrumb' renderer id directly.
Coupling
Easy. Depends on useParents + nav utils, both stable.

3. Roam Research importer

High Effort: S
Now lives at
src/shortcuts/defaultShortcuts.ts:64-68, 401-460 + src/utils/roamImport/
Does
Imports Roam JSON exports; exposes window.__omniliner.roamImport for agent scripts; renders a progress banner.
Why
Niche, one-off feature. Lives uncomfortably inside the "default shortcuts" module. Its presence in core forces every user to ship the importer code & progress UI. Clean action-only seam already exists (actionsFacet).
Coupling
Low. Move src/utils/roamImport/ into the plugin folder; the only back-reference is the agent runtime window hook — make that opt-in via the plugin.

4. SQLite export / import actions

High Effort: S
Now lives at
src/shortcuts/defaultShortcuts.ts:471-519 + src/utils/exportSqliteDb.ts (and an import counterpart)

Dumps / restores the raw PowerSync SQLite blob — power-user backup/restore.

6. GFM markdown extras

High Effort: S
Now lives at
src/markdown/defaultMarkdownExtension.ts, registered in src/extensions/defaultRenderers.tsx:13,29
Does
Adds remark-gfm (tables, strikethrough, task lists) and the external-link target/rel anchor override.
Why
This is exactly what markdownExtensionsFacet was designed for. Other plugins (wikilinks, blockrefs) already use it. GFM is a flavor choice, not a kernel requirement — splitting it makes the markdown layer symmetric: everyone contributes through the facet.
Coupling
Easy. The only thing core actually needs is the base markdown renderer; GFM and the anchor override travel together.

9. topLevel renderer

Medium Effort: M
Now lives at
src/components/renderer/TopLevelRenderer.tsx, registered in src/extensions/defaultRenderers.tsx:19

Wraps the focused block as a "page".

Don't extract

These look feature-y but are foundational once you trace the deps:

  • kernelPropertyUiExtension & kernelValuePresetsExtension — the 7 built-in property type editors. ref/refList are used by backlinks, todo, and SRS plugins; splitting buys nothing.
  • propertyPanelSections.ts — infrastructure for the property panel, not a feature.
  • Renderers default, extension, propertySchema, panel, layout, missingData — dispatched by core code paths (panel layout, missing-data fallback, extension authoring).
  • The base markdown renderer itself, BlockEditor, BlockComponent, the facet runtime, repo & PowerSync wiring.

Suggested order of attack

  1. Wikilinks/blockrefs — clearest signal, already half-done. Resolves a layering inversion (core exports what only a plugin consumes).
  2. Roam import + SQLite export/import + example-extensions seeder — same shape (action + util), can land as one PR each or batched.
  3. GFM markdown — small, symmetric, makes the markdown story cleaner.
  4. Breadcrumb renderer — proves the renderer-as-plugin path end to end, paves the way for the trickier topLevel split.
  5. Daily notes — needs a small API surface first; do it after the action-only extractions to validate the pattern.

Audit covers src/extensions/, src/markdown/, src/shortcuts/defaultShortcuts.ts, src/components/ renderers, and staticAppExtensions.ts / staticDataExtensions.ts. Line numbers are approximate against claude/audit-core-plugins-PHDk3.
