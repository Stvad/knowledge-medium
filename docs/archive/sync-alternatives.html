Sync-layer evaluation: long-term architecture

Choosing the sync stack for a row-shaped block app with heavy edit traffic, multi-user multi-workspace, local SQLite (any WASM build with OPFS) preferred, backend choice open. The PowerSync bucket-wipe bug (powersync-js#954; see powersync-bucket-wipe-bug-report.md) is one input among many — weighed against PowerSync's overall trajectory, the maintainer's response posture, and the cost of any alternative.

Recommendation

Stay on PowerSync; have ElectricSQL documented as the migration plan if 954 stalls or other quality issues emerge. The expected-value math favors staying: bug 954 is a single well-isolated issue actively owned by simolus3 (same-day response, comparable adjacent bug #584 fixed in 6 days), PowerSync's trajectory is accelerating (monthly releases, npm downloads up ~11x in 10 months to ~128k/mo, Rust sync client became default May 2026), and a sync-layer migration is multi-month work. Realistic wait for an upstream fix: 2-6 weeks.

① Stay on PowerSync — recommended
② ElectricSQL + own writes — migration plan if we leave
③ Custom on Postgres logical replication — fallback if Electric doesn't fit
④ Zero (Rocicorp) — off-the-shelf alternative; replaces SQLite

The earlier framing of this doc — "escape PowerSync at all costs" — was wrong. PowerSync is mature, well-maintained, on a strong growth curve, and the 954 fix is plausibly weeks away from a maintainer who has already taken ownership. The question isn't whether we can survive the bug; it's whether the long-term architecture is right. On that question PowerSync still wins on least disruption and best fit for our existing trigger-based pipeline.

If we do migrate, ② ElectricSQL is the clear pick — and this is a genuine correction from the earlier draft. The "PGlite or bust" framing was wrong. Electric Next is a pure HTTP/Shape protocol; @electric-sql/client is ~29 KB gzipped with no SQLite/Postgres runtime deps. Four documented write patterns exist; the simplest is exactly our current shape — "hit your REST API, let Shapes stream the result back" — and the write_id echo-skip recipe is published, not DIY. We keep SQLite, the 15 triggers, the upload connector; we replace the PowerSync sync stream with Electric Shapes.

③ Custom on Postgres logical replication is the fallback if Electric turns out not to fit (e.g. write-path operational complexity, no production references for non-PGlite client stores). Most of the "multi-month custom engineering" in the previous draft is exactly what Electric provides off-the-shelf (initial snapshot, ordered replication, reconnect/resume, write_id round-trip). With Electric in the picture, Custom drops from "Path A" to a defensive option.

④ Zero is the off-the-shelf alternative if we'd be willing to give up local SQLite. Server-authoritative mutator replay structurally precludes the PowerSync class of bug, but the client store is Zero-owned (IDB/SQLite, opaque) and bundle/OPFS specifics are undocumented. Not preferred given our investment in the SQLite + triggers pipeline.

LiveStore deserved evaluation (it uses a wa-sqlite fork, OPFS-only — actually closer to our setup than several other candidates) but is currently blocked on eventlog compaction (livestore#136, open). Per-keystroke-debounced edits would grow the event log linearly forever. Re-evaluate when compaction ships.

Other candidates (Replicache, Triplit, Yjs, Automerge, Loro, Jazz, TinyBase, Liveblocks, Convex, WatermelonDB) each fail on a separate axis — none changes the ranking. Detail cards retained below for reference.

Stack fit
Already integrated. wa-sqlite + 15 triggers + ps_crud + REST upload connector + sync-config.yaml are in production. No migration needed.
Bug 954 ownership
The server omits the data frame for ~1/3 of the uploader-of-record's own writes, leaving the Rust client unable to recompute its bucket checksum and forcing a re-download. Issue thread: simolus3 (Simon Binder, owns most of the Rust client commits) responded same-day, called the behavior "really bad" on 2026-05-13, said "should never happen per protocol," and explicitly took it on with "I'll need to try this out myself." No client-side workaround surface exists per his comment.
Adjacent track record
powersync-service#584 (different but adjacent checksum-mismatch class, server-side) was fixed in ~6 days. Bug 954 looks isolated, not a class pattern — no other open issues describe write-echo / uploader-of-record symptoms.
Trajectory
Healthy and growing. Release cadence: monthly (2026-05-11, 2026-04-14, 2026-04-13). Open issues on powersync-js: 26. Closed since 2025-05: ~68 issues, ~35 sync/bucket/checkpoint-related — high fix throughput, not stagnant. npm downloads: Jun 2025 ~11k → Apr 2026 ~128k (~11×). Major 2026 product investment: Rust sync client became default May 2026 (JS sync client removed); Sync Streams beta replacing Sync Rules; BSON over HTTP; .NET and Tauri SDKs. Release feed.
Risks of staying
(a) Bug fix slips past expected window — mitigated by the OS service repo being OSS; we can contribute the patch (OP has already done the diagnostic work). (b) Different correctness bug emerges that does suggest a class pattern — covered by quarterly review. (c) Periodic full bucket re-download in the meantime — bad UX but not data loss; keep buckets per-workspace so re-download is cheap.
Exit triggers
Revisit / start migration to ② Electric if any of: 954 stalls past ~8 weeks with no in-flight PR; a new sync-correctness bug suggests a class pattern; PowerSync's investment in sync-protocol semantics drops materially.
Cost
Zero — already deployed. Either OSS self-host (powersync-service) or PowerSync Cloud.
③ Custom on Postgres logical replication + outbox Fallback Preserves the 15 triggers + the REST connector. Local store can be wa-sqlite, @sqlite.org/sqlite-wasm, or libSQL — trigger SQL is portable across SQLite impls. Most of what would make this "multi-month" is what ② Electric provides off-the-shelf (initial snapshot, ordered replication, reconnect/resume, write_id round-trip). Defensible only if Electric turns out not to fit operationally.
Stack fit
Native. Supabase Realtime broadcasts row-level INSERT/UPDATE/DELETE over WebSocket from Postgres WAL. Pair with a local outbox table (renamed ps_crud) and our existing upload connector.
Echo-skip
DIY. Add a client_write_id column on the outbox and on every blocks row. The trigger in clientSchema.ts already runs only when tx_context.source = 'user'; stamp the id there. Realtime subscriber drops events whose client_write_id is still pending in the outbox.
Row vs doc
Row, column-level. postgres_changes emits at row+column granularity.
Maturity
Supabase Realtime is GA; tens of thousands of projects using postgres_changes. Caveat: it is single-threaded per DB; Supabase itself recommends Broadcast over postgres_changes for heavy throughput. No public reference impl of wa-sqlite + Supabase Realtime + custom outbox. Closest published wisdom is RxDB's Supabase plugin and Antoine Forand's SQLite sync-engine post.
Migration cost
Low for what survives, but the new code is non-trivial. Survives: wa-sqlite, 15 triggers, upload connector, RLS, block schema, audit log. Rebuilds: write loop (no longer driven by PowerSync CRUD), Realtime subscriber, initial-snapshot REST bootstrap, reconnect/resume by updated_at > checkpoint.
Conflict model
LWW on server-stamped updated_at. Column-level LWW is achievable in Postgres triggers if needed.
Web runtime
wa-sqlite (~1 MB) + Supabase Realtime client (~80 KB). Smallest footprint of any option here.
Pricing
Included in Supabase plan. Heavy keystroke PATCH traffic likely needs a Small/Medium compute add-on to keep postgres_changes responsive.
Risk: PowerSync handles dozens of corner cases here (out-of-order delivery, partial replication of joined writes, exponential backoff, causal ordering across multi-row tx). Plan on a multi-month investment before this is production-grade. Mitigation for multi-row tx atomicity: write a tx_completed fence row and defer apply until the fence arrives. Mitigation for Realtime throughput limits: validate end-to-end with synthetic keystroke load before committing.
② ElectricSQL (Electric Next, electric.ax) Migration plan Major correction from the earlier draft. Electric Next is BYO-local-store, not "PGlite or bust." @electric-sql/client is ~29 KB gzipped with no SQLite/Postgres runtime deps. Four documented write patterns; the simplest is exactly our current shape — "hit your REST API, let Shapes stream the result back" — and the published write_id round-trip recipe handles echo-skip. Production-ready (GA Mar 2025; Trigger.dev runs 20k changes/s).
Stack fit
Native. ShapeStream subscribes to a filtered Postgres row subset and emits row-level ChangeMessage{key, value, op} events. We consume them into our existing wa-sqlite via the BLOCKS_RAW_TABLE.put SQL we already use. The pivot post is explicit: "Client-side data access library — not core. There are many ways of mapping a replication stream to objects, graphs or databases in the client." The current site: "You can sync data into anything you like."
Correction note
The "wa-sqlite dropped" framing applied to legacy Electric (electric-sql/electric-old), where the client was a wa-sqlite-driven CRDT engine. Electric Next is a pure HTTP/Shape protocol — the client store question was deleted from scope, not narrowed to PGlite. PGlite is a sibling product Electric publishes, not a required target.
Echo-skip
Published recipe. Attach a write_id (UUID) to the local optimistic write AND send it as a column in the API request. write_id round-trips through Postgres back into the Shape. Client dedupes with matchBy('write_id', value.write_id) for inserts/updates and matchBy('id', value.id) for deletes. Adding write_id to blocks is a schema migration + 1-line trigger update.
Write patterns
Electric documents four, in increasing complexity: (1) online writes — plain REST; (2) optimistic state — component-scoped optimistic + API; (3) shared persistent optimistic state — shared reactive store + API; (4) through-the-DB — _synced/_local shadow tables. Patterns 1–3 are "hit your existing API, let Shapes stream the result back." Pattern 4 is the only one that requires PGlite.
Row vs doc
Per-row. Shapes are row-shaped Postgres replicas.
Maturity
GA 1.0 March 2025; v1.1 August 2025 (~100× faster writes, new storage engine). Production users: Trigger.dev (20k changes/s, read-path), Otto, IP.world. Unknown: any public production deployment of "Shape → custom SQLite consumer" — absence of marketing evidence, not a technical block.
Migration cost
Moderate. Lose: PowerSync SDK, ps_crud queue, sync-config.yaml, db.connect/disconnect plumbing. Keep: wa-sqlite (or swap), the 15 triggers (SQLite syntax stays), upload connector, blocks schema, row-events audit. New: ShapeStream consumer (~150 LOC), write_id column migration + trigger update, Electric service deployment.
Conflict model
LWW on server timestamps for synced state. Through-the-DB shadow tables available if more sophisticated rebase is ever needed.
Web runtime
@electric-sql/client: 127 KB unminified ESM, ~29 KB gzipped (verified directly from jsdelivr). One runtime dep: @microsoft/fetch-event-source. No WASM, no Postgres driver. No PGlite unless we opt in.
Pricing
OSS self-host (Docker, Railway). Electric Cloud: $1/M writes, free reads, free under 5M writes/mo; Pro $249/mo; Scale $1,999/mo. Per-keystroke PATCH at a few thousand users could exceed Pro tier; self-host is the right call for this workload.
④ Zero (Rocicorp) Off-the-shelf alt Server-authoritative mutator-replay model structurally rules out the PowerSync bug class. Works with any Postgres. Cost: client store is Zero-owned — replaces wa-sqlite entirely. Bundle size + OPFS story not publicly documented.
Stack fit
Backend: reads Supabase Postgres natively (since v0.20; schema-change detection in 1.0). Client store: Zero-owned (IndexedDB/SQLite, not user-controlled). Would replace wa-sqlite, not wrap it.
Echo-skip
Server-authoritative mutator replay. Client rolls back the optimistic local effect on checkpoint and replaces it with the authoritative server row. Identified by mutation id; structurally precludes the PowerSync echo bug.
Row vs doc
Per-mutator JSON ops replayed server-side. Column UPDATEs are fine inside a mutator; you give up the per-CRUD-row abstraction.
Maturity
1.0 declared (a thin bump from 0.26.2). Active Apache-2 releases. No public list of large prod deployments; docs flag no offline write + TypeScript-only.
Migration cost
High. Triggers, ps_crud, sync-config.yaml, raw-table mapping do not survive. Sync rules become row-level auth (Postgres RLS / Zero permissions DSL).
Conflict model
Server-authoritative LWW on rebased state with custom mutator logic.
Web runtime
WebSocket to zero-cache; client IDB/SQLite. Bundle size not published.
Pricing
Apache-2, self-hostable. Community deploys cite $5-10/mo single-node. Managed pricing not public.
⑤ LiveStore Evaluated, blocked Event-sourced local-first framework (Johannes Schickling, ex-Prisma). Uses a fork of wa-sqlite with OPFS — structurally the closest match to our existing local-store setup of any candidate here. Blocked on eventlog compaction (livestore#136) — per-keystroke-debounced edits would grow the log linearly forever in both OPFS and the sync backend. Re-evaluate when compaction ships.
Data model
Events are source of truth; SQLite tables are projections via materializers. Each event has past-tense name + typed args + version (BlockCreated, BlockContentEdited, ...). Two categories: synced and client-only. Schema evolution adds/removes optional fields but cannot delete event definitions post-deployment.
Sync model
Git-style push/pull. Each event has seqNum + parentSeqNum + 6-char client nanoid. Clients pull upstream before push; local pending events rebase on upstream. Backends supply ordered query-from-cursor + reactivity (WebSocket / HTTP / poll). First-party adapters: @livestore/sync-cf (Cloudflare Durable Objects), @livestore/sync-electric, @livestore/sync-s2.
Echo-skip
Structural via seqNum + client nanoid. Client recognises its own already-applied events on pull and skips re-materialising. Design-implicit; not documented as a named feature — would need code review to confirm idempotence under replay.
Local store
Fork of wa-sqlite (@livestore/wa-sqlite) with sync APIs + session extension. OPFS-only persistence; WAL mode. Leader-tab elected across browser tabs/workers. LiveStore owns the SQLite instance — not pluggable into an existing wa-sqlite handle.
Row-shape fit
Good. Define BlockCreated / BlockContentEdited / BlockMoved / BlockPropertiesChanged events; materializers emit UPSERTs against our existing blocks table. The SQLite projection is ours to shape — LiveStore does not impose a schema.
Maturity
Beta. v0.3.1 (June 2025), Apache-2.0, 3.6k stars / 127 forks. Powers Overtone (Schickling's music app, source of the extraction) in production. Docs explicitly say "not yet ready for all production scenarios" with three breaking-change categories warned (API, client storage, backend storage). No 1.0 date.
Migration cost
Substantial but contained. Delete: ps_crud queue, the 15 upload-routing triggers (event log replaces them), row_events table (event log IS the audit log). Rewrite: every tx.X call site becomes a commit(events.Y(...)); materializers replace the trigger SQL. Keep: row schema as the projection target.
Conflict model
LWW ordered by rebased seqNum. Custom merge logic requires the unshipped "facts system."
Web runtime
~180 KB gzip JS + ~300 KB gzip WASM (~480 KB total). OPFS-only (no IndexedDB fallback). Recommends DB ≤ ~1 GB and "10s to low 100s collaborating on the same thing."
Pricing
Apache-2 OSS; you host the sync provider yourself (CF DO, Electric, S2, or custom).
🚫 Blocking concern: eventlog compaction. (livestore#136, open, blocked by #254.) Planned Kafka-style key-based compaction (a new event for the same key drops the prior). Until that ships, every debounced keystroke batch persists forever in both the local OPFS event log AND the sync backend. For our heavy-edit workload this is the single biggest risk. Today's mitigations are weak: coarser debounce, periodic snapshot+GC events, or wait.

Migration sketch — ② ElectricSQL

Sketched for the migration plan: the path we'd take if exit triggers in the ★ Stay card fire. The shape is "Electric handles the read path; we keep the rest of our data layer." Most of what would make ③ Custom "multi-month" — initial snapshot, ordered replication, reconnect/resume — Electric provides as a product. Our part is a small Shape→SQLite consumer plus the write_id column for echo-skip.

Risks that remain

  • Trailblazing risk. No public production deployment of "Electric Shape → custom SQLite consumer." The pattern is documented in writes guide pattern 1–3 and the write_id recipe is published, but we'd be the first visible case study. Mitigation: build a thin proof-of-concept against a single Shape before committing the full migration.
  • Causal ordering across joined writes. Electric Shapes emit per-row events, like Realtime. A multi-row tx (block move + ordering key updates) can be observed mid-state. Mitigation: tx_completed fence row, defer apply until fence arrives. Same as Path ③.
  • Self-host operational complexity. Running the Electric service adds an operational surface we don't have today (it's stateful, talks logical replication to Postgres). Lower than running PowerSync's own service was, but non-zero.
  • Echo-skip correctness under reconnect. If the local outbox is wiped and a pending write's write_id is lost, the round-tripped event will replay as a remote write. Outbox needs durability guarantees similar to PowerSync's ps_crud — already what we have, but verify it survives the rename.

Open questions

  1. Bug 954 timing. Realistic expectation is 2–6 weeks for an upstream fix (based on simolus3's response on the issue and a ~6-day fix for the comparable adjacent powersync-service#584). What's the maximum we'd wait before triggering the ② Electric migration? Proposed: 8 weeks with no in-flight PR. Open for discussion.
  2. Electric: any production Shape → custom SQLite consumer to learn from? Public evidence is thin. Not a technical block (the API is straightforward), but a small proof-of-concept against a single Shape is worth doing before committing to the full migration if/when we trigger it.
  3. Electric self-host operational cost. Running the Electric service is non-zero ops work. Worth pricing out (CPU/RAM footprint per N synced rows, replication-slot impact on Postgres) against what PowerSync Cloud currently costs us.
  4. Could we contribute the 954 server-side fix ourselves? powersync-service is OSS and the OP has already done the diagnostic work isolating the missing data frame. A targeted PR may be the fastest path to "fix lands." Worth assessing the powersync-service codebase complexity.
  5. Zero's client store internals. Still unknown — bundle size, OPFS support, client SQLite/IDB choice not publicly documented. Only matters if we ever end up considering ④ Zero seriously (we don't right now).
  6. LiveStore eventlog compaction timeline. When (if) livestore#136 ships, ⑤ LiveStore becomes a genuine contender (closest local-store match to our setup of any candidate). Worth watching the issue.
  7. Turso / libSQL embedded replicas. Not researched in this pass — a SQLite-native sync model where the server is libSQL and the client is a local libSQL DB that syncs from the server's WAL. If we ever leave PowerSync and Electric somehow fails to fit, this is the next thing to research before going to ③ Custom.
