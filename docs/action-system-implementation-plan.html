Action system — implementation plan & sequencing

Synthesis of the precedence + critique + prior-art + input-unification discussion — 2026-05-30

Consolidates the decisions from docs/shortcut-context-precedence.html (the bug + precedence options) and docs/action-system-critique.html (the architectural critique), plus the prior-art survey, the focus-tree activation exploration, and the downstream input-unification constraint, into one sequenced plan. This doc is the "what to build, in what order" — the other two are the "why."

The spine: one component everything hangs off. Every thread converges on resolve(activeSet, trigger) → action. The precedence bug needs it (single winner); the critique names it (P1); input-unification consumes it (mouse/touch through the same path); the focus-tree model only changes what produces activeSet. So the plan is "build the spine, hang things off it," not a list of independent fixes. Don't build the narrow dedup-in-loop fix — given a committed input-unification consumer, it's work that gets torn out.

Sequencing

Phase 0 — Shared canonicalizer enabling

What: lift normalizeChord (keybindings-settings/keyCapture.ts:205) into src/shortcuts/, add sequence-chord parsing, shape it as chord↔descriptor from day one (not chord-only). Retrofit keybindingConflicts.ts off it.

Why first: Phases 1–3 all assume canonical comparison. Cheap, no behavior change, unblocks everything. Addresses: precedence-doc "canonicalization is not free" review finding.

Done when: alias-equivalent chords (Meta+K / $mod+k) compare equal (already true today, including modifier order — see Steps); sequence chords (d d, g g) parse into an ordered list of presses instead of one atomic key; one shared helper, used by dedup + conflict detection.

Files & shape:

// new: src/shortcuts/canonicalizeChord.ts
export type ChordDescriptor = {
  kind: 'key'                       // Phase 3 adds 'mouse' | 'touch'
  key: string                       // canonical, e.g. 'k'

  phase: 'keydown' | 'keyup' | 'hold'
}
// a chord is a SEQUENCE of presses: 'd d' / 'g g' → two descriptors; ordinary chords are length 1.
export type ChordSequence = readonly ChordDescriptor[]
export const canonicalizeChord = (raw: string, phase?: Phase): string   // stable key; sequence-aware
export const parseChord = (raw: string): ChordSequence                  // for matching; splits on SPACE first

Steps: (1) move normalizeChord body out of keybindings-settings/keyCapture.ts into the new module — it already alias-folds, lowercases, AND sorts modifiers (keyCapture.ts:236, tested at test/keyCapture.test.ts:147), so this is a relocation, not a fix; (2) add the real gap — sequence-chord parsing: today it splits on + only and treats a space-bearing token like d d / g g as a single atomic key, so the canonicalizer must split on space first into an ordered list of presses, then canonicalize each press; (3) re-export from keyCapture.ts so the settings UI is unchanged; (4) point keybindingConflicts.ts' toChordArray bucketing at the canonical key. Keep the descriptor shape open (the kind field) so Phase 3 adds variants, not a rewrite.

Phase 1 — Resolver core + precedence + canDispatch closes the bug lays the spine

What (one or two PRs — see "what's coupled vs. what can split" below):

  • resolve(activeSet, trigger) → action as a fire-time, declinable coordinator (precedence-doc Option D): one keydown/keyup listener, trigger → ordered candidates → run-until-handled, replacing the N per-action window listeners. Route the keyboard path, runActionById, and useRunAction through it.
  • Comparator (priority desc, then activation-recency desc), global as reserved top tier, modal preserved. Shared with getActiveActionById so the dispatch and keyboard paths can't diverge.
  • priority as named tiers 'low' | 'default' | 'high' (see "Precedence tiers" below — not a raw number) on ActionContextConfig → fixes the surface early/late inversion.
  • canRun → isVisible (UI) + canDispatch (dispatch gate). canDispatch ships with the coordinator — it's the skip predicate the single-winner loop iterates on (PR 1); the handler-return sentinel joins it as a second skip-condition in PR 2.
  • Deps seam: resolve deps from {active scopes} ∪ {caller/trigger-supplied}, narrowed at one explicit validated boundary — not active-map-only, not scattered casts.

What's genuinely coupled vs. what can split (and why the split costs no rework): the resolver core — resolve + compareContexts + priority + the Phase-0 canonicalizer + the canDispatch concept + the deps seam — is built once and shared by both PRs; nothing in it is thrown away. The split is purely about the dispatch contract: PR 1 builds the single coordinator (one keydown + one keyup listener) with single-winner selection: normalize event → gather matching candidates → order via compareContexts → iterate and dispatch the first whose canDispatch passes. That loop already exists, so it closes the double-fire and fixes the early/late inversion, and depends on nothing downstream. PR 2 adds only the declinable fall-through: widen the handler return to a not-handled sentinel and add "handler returned false" as one more skip-condition to the loop PR 1 already wrote (Option D). That is pure addition — no listener wiring is rewritten or discarded. Scope that claim precisely: "no rework" is about the listener wiring specifically. PR 2 still widens the shared ActionHandler return type (add the not-handled sentinel). That is back-compat — void = handled, so no handler is forced to change — but it touches a signature that ~176 handler: definitions across ~49 files structurally depend on, so budget PR 2 as a wide-blast-radius type review even though nothing breaks. So the split boundary is "everything the bug fix needs" (PR 1, incl. the coordinator) vs. "the dispatch-contract change the downstream input work needs" (PR 2): PR 1 can merge immediately; PR 2 lands with / just ahead of the spatial-selection branch that is its only fall-through consumer. Option D remains the committed target. Note the trade — and don't mis-scope PR 1 as "small": building the coordinator in PR 1 (rather than keeping per-action listeners and deferring the rewrite) front-loads the one mechanically risky change — the reconciler rewrite, which carries the documented useEffectEvent regression history — into the bug-fix PR. It is not speculative there: the coordinator is the single-winner decision point that PR 2 builds on, so it isn't discarded. (Strictly, a per-listener self-check against the shared comparator could fix only the double-fire at lower mechanical risk — but it gets thrown out once Option D lands, so it's only the lower-risk choice if Option D were ever dropped.) The earlier "negative work" warning applies to that other way to fix the bug (activation-time install-filtering / per-listener self-check), which this plan explicitly does not do precisely because it would be thrown out at PR 2.

Retires: the double-fire (bug); critique weakness 1 (keyboard/dispatch divergence) and the swipe fork's reason-to-exist; weakness 3 (canRun-as-gate); weakness 4 symptom (priority escapes lifecycle ordering).

Tests: higher-priority context wins a same-key collision even when activated earlier; deactivation restores the lower binding; equal priority → last-wins; modal suppresses non-colliding chords; global survives a modal; same key different phase (hold s vs keyup s) does NOT dedup; alias-equivalent chords DO dedup; a declining winner falls through to the next candidate; sequence chords (d d) still fire under the coordinator.

Carries the load-bearing invariant comment: resolution is correct because eventFilter only expands, never declines-and-defers — declinable handling is the supported fall-through, not filter scoping.

Core signature & types:

// new: src/shortcuts/resolve.ts
type Trigger =
  | { kind: 'action'; actionId: string }          // runActionById / useRunAction
  | ChordDescriptor                                // keyboard now; mouse/touch in Phase 3

// pure, unit-testable, no React/DOM:
resolve(
  actions: readonly ActionConfig[],   // effective actions (post-transform)
  active: ActiveContextsMap,
  trigger: Trigger,
): readonly ActionConfig[]            // ORDERED candidates, best-first

// comparator (the single source of precedence, shared with getActiveActionById):
compareContexts(a, b): number  // 1) modal-over-global rule  2) priority desc  3) recency desc

Sequence chords vs. this signature — reconcile explicitly: a single ChordDescriptor cannot represent a mid-sequence keypress (the first d of d d completes no chord). So matching — including tinykeys-style sequence state — lives in the coordinator, per-candidate; resolve() only orders the candidates that have already completed a match this event. Read the ChordDescriptor in the keyboard Trigger as "the chord that just completed" (used for ordering / dedup / logging), not as the matcher input. Do not reduce a keydown to one descriptor and hand it to resolve to match — that path is exactly how g g went dead historically (vim-normal-mode/actions.ts:204–208). And keep resolve's internal Trigger distinct from what the handler receives: handlers keep their existing second arg ActionTrigger = KeyboardEvent | CustomEvent (types.ts:108) — the raw event still flows through to the handler; the ChordDescriptor is internal to resolution/ordering and is never handed to handlers.

Coordinator control flow (the highest-regression-risk seam in PR 1 — spell it out, don't just say "keep a matcher"): keep one tinykeys matcher per candidate binding (today's createKeybindingsHandler, HotkeyReconciler.tsx:257). On each keydown / keyup the coordinator feeds the event to every candidate's matcher; tinykeys fires a binding's callback synchronously when that binding's sequence completes on this event. Rewire each callback to push its candidate into a "completed-this-event" set instead of running the handler directly. After feeding all candidates, the coordinator orders the completed set via compareContexts and runs the loop (canDispatch → resolveDeps → handler; run-until in PR 2). Because tinykeys completion is synchronous within the event dispatch, collect-then-order happens in the same tick — no async-ordering hazard. Multiple candidates can complete on one keypress (two contexts both bind g g) — that is precisely the collision the comparator resolves; single chords are the degenerate length-1 case. This per-candidate-matcher → collect → order → run-loop is the substantive new design in PR 1, not "addition," and it is where g g/d d will regress if guessed wrong — give it dedicated sequence tests.

Precedence tiers (named, not raw ints — CodeMirror Prec precedent): type Priority = 'low' | 'default' | 'high' on ActionContextConfig (default 'default'). global = reserved top; modal orthogonal (own-all-chords). The reporting surface declares 'high' to beat normal-mode. Keep the set tiny; add a tier only when a real case needs one.

The global-vs-modal sub-question (the "Escape case") — DECIDE before coding: global owns Escape (clear focus) as reserved top tier, but a modal mode (e.g. scrub) wants Escape to cancel itself. Recommended & assumed by this plan: an active modal context may override global on a shared chord (consistent with modal already being the tier that suppresses everything else). Encode as: comparator ranks active-modal above global, global above all other tiers. Two alternatives, if you'd rather not bake a permanent modal-over-global inversion into the one comparator everything routes through: (a) keep global absolutely inviolable and make modals bind a non-Escape cancel key; or (b) conditional global — global's reserved chords carry an implicit "no modal active" filter, so scrub's Escape wins simply because global's Escape isn't contending while a modal is up. Option (b) preserves "global is top tier" as a rule while still letting a mode own its cancel key, and is closest to today's behavior (modal shadows non-global; global stays installed). Flag the choice before building.

File map:

  • src/shortcuts/resolve.ts — new; resolve + compareContexts (pure).
  • src/shortcuts/types.ts — add priority: 'low' | 'default' | 'high' (named tier, default 'default') to ActionContextConfig; split canRun → isVisible + canDispatch; widen ActionHandler return to allow a "not handled" sentinel (boolean; void = handled, for back-compat). Async rule: the not-handled signal must be a synchronous boolean — the run-until-handled loop has to choose the next candidate within the same event and cannot await. A handler returning a Promise counts as "handled" the moment it returns (fall-through stops there); only a synchronous false falls through.
  • src/shortcuts/effectiveActions.ts — getActiveActionById calls compareContexts (kills the divergence). This is a behavior change, not a no-op refactor: today getActiveActionById is pure reverse-activation-order with no global carve-out (verified — zero global references in the file). Routing it through the comparator means a global-vs-scoped id collision that resolves by recency today will instead resolve to global as reserved top tier — so runActionById / the command palette need their own regression test, not just the keyboard path.
  • src/shortcuts/HotkeyReconciler.tsx — the big change. Replace the per-action window.addEventListener install loop with one keydown + one keyup coordinator that: normalize event → ChordDescriptor (Phase 0) → gather candidates whose (chord, phase) match and whose context is installable → order via compareContexts → run down the ordered candidates and dispatch the first whose canDispatch passes (PR 1); in PR 2 a handler returning the not-handled sentinel also falls through. Preserve withRecoveredLetterKey, the existing shouldHandleEvent filter cascade, the separate hold-binding observer, eventOptions (preventDefault/stopPropagation) precedence, and tinykeys sequence state (d d, g g) — keep a per-candidate sequence matcher; don't reduce each event to a single chord.
  • src/shortcuts/runAction.ts — runActionById/useRunAction go through resolve(..., {kind:'action', actionId}) + the shared deps seam. Migrate both copies: the module dispatcher (runAction.ts:15) and useRunAction each have their own lookup + "not found / context not active" throw (runAction.ts:51–56) — easy to migrate one and miss the other.
  • Call sites that pass deps: keep working via the seam (see below).

Caution — stopPropagation changes meaning under one coordinator: today each action installs its own window listener, so a binding's stopPropagation suppresses the sibling action-listeners (that sibling-listener model is the root of the double-fire). With one coordinator, losing candidates are loop entries, not separate listeners — there's nothing left among them for stopPropagation to stop. It still matters versus the separate hold observer and any app-level listeners. Heads-up for the audit: no in-repo binding actually sets stopPropagation: true today — every eventOptions/defaultEventOptions only sets preventDefault (the binding default is stopPropagation: false at HotkeyReconciler.tsx:443/470), so the in-repo audit comes back empty. The real consumers are the hold observer, app-level listeners, and plugins. Write the regression test against the mechanism (a binding declaring stopPropagation: true), not against a pre-existing in-tree case — there isn't one. This is a behavior change hiding inside "preserve eventOptions."

Deps seam (the Phase-1 slice of the dependency model): one function resolveDeps(action, active, supplied?) → deps | null that merges the action's active-context deps with optional caller-supplied deps, runs the context's validateDependencies, and is the only place the widened→narrow cast happens. null ⇒ not dispatchable — and in the run-until-handled loop that means skip this candidate and try the next, never abort the loop. The three fall-through triggers — canDispatch false, resolveDeps null, handler returns false — are all "try the next candidate" and the loop must treat them identically. Build it caller-suppliable now so swipe's runBlockAction fork can later pass {block, uiStateBlock} through resolve() instead of bypassing it — even though the swipe migration itself is Phase 3.

Suggested internal order (steps 1–6 = PR 1; step 7 = PR 2): (1) resolve + compareContexts pure, with unit tests — no wiring; (2) priority type + getActiveActionById swap; (3) the canRun→isVisible/canDispatch split (canDispatch as the single-winner loop's skip predicate, no handler-return sentinel yet); (4) rewrite the reconciler install loop onto the single coordinator (single-winner dispatch); (5) deps seam; (6) integration tests → PR 1 merges here, bug fixed. (7) PR 2: widen the handler return to a not-handled sentinel and add it as a skip-condition in the existing loop (Option D), with the spatial branch. Steps 1–3 are independently reviewable even within PR 1.

Dependency thread (why Phase 1's deps seam matters)

The deps work has two axes: typing (who owns the dep type — context vs handler) and supply (who produces the value, when — pushed at activation vs provided at dispatch). "Handler-declared deps" only fixes typing; supply is the coherence issue. Today's push-only model means "to dispatch an action you must be its active context" — which is precisely why swipe's runBlockAction forks (it holds the deps but can't hand them to the dispatcher). Phase 1 builds only the seam (deps from {active scopes} ∪ {caller-supplied}); the full provider/capability DI model (scopes provide typed tokens, actions declare needs, missing token ⇒ not dispatchable ⇒ that is canDispatch) is deferred until a third data point or the focus-tree prototype — they are the same model at different levels of "derive from tree." See critique weakness 6 for the full treatment.

The separate bet — do NOT gate anything on it

Focus-tree activation model — tracked in issue #101. Replace the flat, runtime-maintained ActiveContextsMap with an active set derived from the focus/DOM tree, and split the one activate verb into "establish scope" (containment, structural) vs "override" (precedence, declared). This is the genuine best-in-class move — it retires the timing-as-precedence bug class at the root (weakness 4) by killing the registry that can desync.
  • It does NOT fix the surface bug — the surface wants to override against containment depth (it's the ancestor; the block is deeper), which needs the explicit priority/override signal in any architecture. Structural depth alone gives the same wrong answer as last-wins.
  • It composes with the spine, doesn't compete: it changes what produces activeSet; resolve() consumes one either way. And the provider/capability deps model is literally this model with React-context providers as the chain.
  • Heaviest change discussed (~27 activation sites + typed-deps threading + non-focus-shaped scopes like multi-select). Recommend a throwaway-branch prototype to validate deps-threading and multi-select before it's a real proposal. Phase 1's priority already patches the bug symptom, so there's no urgency.
