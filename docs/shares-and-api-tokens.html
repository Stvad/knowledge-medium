Shares + API tokens — unified access model

Supersedes the “single-root subtree share” framing in subtree-sharing.md. Draft, 2026-05-15.

1. Why a new doc

The existing subtree-sharing.md nails the hard parts: RLS unification via blocks.effective_share_ids, role monotonicity, token isolation, trigger-maintained access materialization, and a single-mode-per-session bootstrap. We're keeping all of that. What's changed since it was written:

  1. API access is now a first-class requirement. Programmatic agents and third-party integrations need stable scoped access without a browser session. Treating an API token as a kind of share grant collapses two designs into one.
  2. A single share should cover multiple subtrees — “share these three projects with my contractor” or “the agent gets read access to this page and that page.”
  3. Workspace-scoped tokens are valuable. Both for “give my agent the whole workspace” and as the natural ceiling of the subtree model.

This doc is the diff against subtree-sharing.md. Sections that aren't re-litigated here (read-only snapshot RPC, presentation root clipping, advisory locks in the recompute, the “workspace member overrides share grant” rule, the test plan, the guardrails) remain authoritative in the original doc.

2. The unifying primitive

A share is the access primitive. Everything else — human invites, link URLs, API tokens — is a way of putting some principal into a share's audience.

Share = {
  workspace_id,                       // a share always lives inside one workspace
  scope:    'workspace' | 'subtree',  // entire workspace, or a set of subtree roots
  roots:    text[]                    // only meaningful for scope='subtree'; one or more block ids
  role:     'viewer' | 'editor',      // default role for new audience members
  purpose:  'human' | 'api',          // who this share is for; drives UI/listing only
}

ShareMember = (share_id, principal_id, role, source)
  // principal_id is auth.users.id, whether human or API service user
  // source is one of: 'invite' | 'link' | 'api_token'

ShareLink = a redeemable URL fragment (one-shot per user) for humans
ApiToken  = a long-lived credential (re-usable, single service principal) for integrations

The two are the same shape: an opaque secret stored in a tightest-RLS sidecar table, validated by a SECURITY DEFINER RPC, which then upserts a row into block_share_members for the redeeming principal. The differences are operational: lifecycle (one-shot human redeem vs. long-lived API exchange), audit fields, and UI surface.

Mental model: a workspace member is just a built-in “workspace-scope share with the user as a member.” A subtree share is the same idea narrowed to N roots. An API token is a share whose audience is a per-token service user. One RLS predicate handles all four cases.

3. Schema deltas vs. the existing doc

Same migration posture as subtree-sharing.md §1: existing tables (blocks, workspaces, workspace_members, workspace_invitations) are not touched destructively. The delta below changes only the new tables introduced by the share design, plus one additional column on blocks.

3.1 block_shares — multi-root + scope kind

ColumnOriginal (subtree-sharing.md §4.2)New
root_block_id scalar text, immutable removed — moved to block_share_roots
scope (implicit: always subtree) new: text not null check (scope in ('workspace','subtree'))
purpose (not present) new: text not null default 'human' check (purpose in ('human','api'))
workspace_id denormalized + stamped at insert, immutable unchanged
default_role, created_by_user_id, create_time, revoked_at as before unchanged
create table public.block_shares (
  id                   text primary key,
  workspace_id         text not null references public.workspaces(id) on delete cascade,
  scope                text not null check (scope in ('workspace','subtree')),
  purpose              text not null default 'human' check (purpose in ('human','api')),
  default_role         text not null check (default_role in ('viewer','editor')),
  created_by_user_id   text not null,
  create_time          bigint not null,
  revoked_at           bigint
);

3.2 block_share_roots — new junction table

One row per (share, root) pair. Only populated for shares with scope='subtree'; enforced by trigger.

create table public.block_share_roots (
  share_id       text not null references public.block_shares(id) on delete cascade,
  root_block_id  text not null references public.blocks(id)        on delete cascade,
  workspace_id   text not null,  -- stamped + checked equal to block_shares.workspace_id and blocks.workspace_id
  create_time    bigint not null,
  primary key (share_id, root_block_id)
);
create index idx_block_share_roots_root_block_id on public.block_share_roots(root_block_id);
Why a junction table instead of text[] on block_shares: roots get added/removed over time, and we want a per-root foreign key with cascade-on-delete so deleting a block that's a share root cleanly removes that scope without leaving a dangling id in an array. The same reasoning that made workspace_members a table rather than an array on workspaces applies here.

3.3 blocks.effective_share_ids — unchanged shape, narrower scope

The column stays as designed in subtree-sharing.md §4.1, but with one rule change: it only carries block_shares.ids for shares with scope='subtree'. Workspace-scope shares are matched by joining workspace_id directly in RLS — cheaper than re-stamping every block in the workspace whenever a workspace-scope share is created or revoked.

This keeps the recompute function (subtree-sharing.md §5) unchanged: it still walks descendants from one root, unioning block_share_roots matches. When a multi-root share is created, we just call recompute_share_ids_subtree once per root. Adding/removing a root is one recompute on that root.

3.4 share_links — add purpose and audit fields

ColumnOriginalNew
purpose n/a new: text not null default 'human_link' check (purpose in ('human_link','api_token'))
name n/a new: text — human label for API tokens (“laptop CLI”, “Readwise sync”)
last_used_at n/a new: bigint — bumped on every exchange (API tokens) or redeem (human links)
last_used_ip n/a new: text — best-effort, may be null
service_user_id n/a new: text — for purpose='api_token', the dedicated auth.users row that this token authenticates as. Null for human links. One-to-one with the link.
expires_at, max_redemptions, redemption_count, revoked_at as before semantics unchanged; for API tokens, redemption_count increments on first exchange only (a single service user redeems once, then re-uses the JWT)

3.5 share_link_secrets — unchanged

Token storage table from subtree-sharing.md §4.2 is reused as-is for both human links and API tokens. Same RLS (creator-only SELECT), same hash-not-plaintext storage, same one-time-reveal-on-creation discipline.

For API tokens, we store the SHA-256 of the token in share_link_secrets.token. Human links can do the same (it was an open choice in the original doc); switching both to hashed storage makes the table genuinely “a secret never escapes Postgres alive”.

4. RLS — the unified blocks predicate

One read policy, one write policy, four cases covered:

create policy blocks_read on public.blocks
for select using (
  -- 1. Workspace member (existing behavior)
  public.is_workspace_member(workspace_id, auth.uid()::text)

  -- 2. Workspace-scope share recipient or API-token principal
  or exists (
    select 1
    from public.block_share_members sm
    join public.block_shares s on s.id = sm.share_id
    where sm.user_id = auth.uid()::text
      and s.revoked_at is null
      and s.scope = 'workspace'
      and s.workspace_id = blocks.workspace_id
  )

  -- 3. Subtree-scope share recipient or API-token principal
  or exists (
    select 1
    from public.block_share_members sm
    join public.block_shares s on s.id = sm.share_id
    where sm.user_id = auth.uid()::text
      and s.revoked_at is null
      and s.scope = 'subtree'
      and s.id = any (blocks.effective_share_ids)
  )
);

The blocks_write policy mirrors this with sm.role = 'editor' in both share branches and a matching with check. (Same shape as subtree-sharing.md §6.1.)

Notice what this buys us. An API token is just an auth.uid() with a block_share_members row. The RLS predicate doesn't know it's an API call. No “is this a token request?” branch in policy, no service-role bypass anywhere in the data path. PostgREST + Supabase auth handle it natively.

5. Recompute — what changes

subtree-sharing.md §5.1's recompute_share_ids_subtree(p_root_block_id) needs one small tweak: it currently joins block_shares on root_block_id = child.id. With the junction table, that becomes:

-- inside the recursive CTE, replacing the existing join:
coalesce((
  select array_agg(s.id)
  from public.block_share_roots r
  join public.block_shares     s on s.id = r.share_id
  where r.root_block_id = child.id
    and s.scope = 'subtree'
    and s.revoked_at is null
), '{}'::text[])

Triggers calling the recompute (subtree-sharing.md §5.2) fan out one additional case:

TriggerArgument(s)
blocks AFTER INSERTparent's effective_share_ids (unchanged)
blocks AFTER UPDATE OF parent_idrecompute(NEW.id) (unchanged)
block_share_roots AFTER INSERT or DELETErecompute(NEW.root_block_id) (or OLD)
block_shares AFTER UPDATE OF revoked_atfor scope='subtree': recompute on each root. For scope='workspace': no recompute needed.

Workspace-scope share creation/revocation does not trigger a recompute. It's RLS-only; membership is checked against blocks.workspace_id directly.

6. PowerSync sync streams

Two changes to subtree-sharing.md §7:

6.1 shared_blocks stream — multi-root, scope-aware

The original stream joins blocks via block_share_members ↔ effective_share_ids. With workspace-scope shares, we union two predicates:

shared_blocks:
  auto_subscribe: true
  query: |
    SELECT blocks.*
    FROM public.blocks
    JOIN public.block_share_members sm ON sm.user_id = auth.user_id()
    JOIN public.block_shares          s ON s.id = sm.share_id AND s.revoked_at IS NULL
    WHERE (
      (s.scope = 'workspace' AND s.workspace_id = blocks.workspace_id)
      OR (s.scope = 'subtree' AND s.id = ANY(blocks.effective_share_ids))
    )

For API tokens this stream isn't reached because API consumers don't run PowerSync — they hit PostgREST or a snapshot RPC. The stream exists for human share recipients. A workspace-scope human share recipient (rare in v1 — see §6.2) gets the entire workspace via this stream without us having to put them in workspace_members.

6.2 V1 simplification: workspace-scope shares are API-only

For v1, only purpose='api' shares may have scope='workspace'. Human shares stay subtree-only. Rationale: the human flow for “share my whole workspace with you” is already covered by invite_member_by_email + workspace_members. Adding a parallel path for humans creates a fork in the share-mode bootstrap (which root do we land on? breadcrumbs to where?) for no real benefit.

Enforced by check constraint:

alter table public.block_shares
  add constraint workspace_scope_is_api_only
  check (scope = 'subtree' or purpose = 'api');

If a real use case for “guest with the whole workspace, not yet a member” appears, drop the constraint and finish the human bootstrap path. The RLS predicate already supports it.

7. API token lifecycle

7.1 Creation (UI flow, in-app)

  1. User opens Settings → API tokens in the app, or invokes “Create API token” from the existing share dialog (when context is a specific block).
  2. User chooses:
    • Scope: this block + descendants / these blocks (multi-select) / entire workspace
    • Role: viewer or editor
    • Name: e.g. “Claude on laptop”
    • Expiry: 30/90/365 days / never
  3. Client calls one RPC: create_api_token(p_scope, p_roots, p_role, p_name, p_expires_at). Server-side it:
    1. Creates a block_shares row with purpose='api' and the chosen scope.
    2. For subtree scope, inserts block_share_roots rows.
    3. Creates a dedicated service user (auth.admin.createUser with a generated email like svc+<random>@knowledge-medium.local and a random password; user is never used for password sign-in — the token is the credential).
    4. Inserts block_share_members(share_id, user_id=service_user_id, role).
    5. Generates the token (encode(gen_random_bytes(32), 'base64url') = 43 chars prefixed with km_).
    6. Inserts share_links(purpose='api_token', service_user_id, ...) + share_link_secrets(token=sha256(plaintext)).
    7. Returns the share + the plaintext token once.
  4. Client surfaces the plaintext token with copy-to-clipboard and a “you won't see this again” warning, then stores nothing.
The service user creation step is the only piece that needs a Supabase Edge Function or a service_role-keyed RPC — auth.admin.createUser isn't callable from client SQL. Everything else stays as regular SECURITY DEFINER RPCs callable by authenticated clients.

7.2 Exchange (every API call)

Third-party clients POST to a small edge function /functions/v1/api-token-exchange:

POST /functions/v1/api-token-exchange
Authorization: Bearer km_<43 chars>

200 OK
{
  "access_token": "<short-lived Supabase JWT, sub=service_user_id>",
  "expires_in": 900,
  "share": { "id": "...", "scope": "subtree", "workspace_id": "..." }
}

The edge function:

  1. Hashes the bearer token, looks it up in share_link_secrets.
  2. Validates: link not revoked, not past expires_at, share not revoked.
  3. Bumps last_used_at, last_used_ip on share_links.
  4. Signs a JWT with the project JWT secret claiming sub = service_user_id, role = 'authenticated', aud = 'authenticated', short TTL (15 min).
  5. Returns the JWT.

The client now uses the JWT against PostgREST exactly like any Supabase client. RLS resolves through the block_share_members row attached to the service user.

7.3 What 3rd parties get for free

  • GET /rest/v1/blocks?workspace_id=eq.<id>&deleted=eq.false
  • GET /rest/v1/blocks?parent_id=eq.<id>&order=order_key.asc
  • POST /rest/v1/blocks (editor role only; workspace_id trigger enforces immutability, RLS enforces scope match)
  • PATCH /rest/v1/blocks?id=eq.<id>

For ergonomic agent surfaces (subtree fetch in one round trip, search across properties_json, batch tree creation), a separate /functions/v1/api/… edge function exposes high-level endpoints over the same JWT. These wrap RPCs and are the layer that shields third parties from schema churn. v1 can ship without this and rely on PostgREST; the high-level surface is a follow-up.

Read-only fast path. For tokens with role='viewer' the existing subtree-sharing.md §8.4 get_shared_subtree(token) RPC is already a perfectly good zero-state read endpoint. It validates the token, returns a flat snapshot, doesn't need the service user round-trip. This is the v1 “public read” endpoint already designed. Extend it to accept multi-root subtree shares and workspace shares (returns paginated blocks for workspace scope).

7.4 Revocation, rotation, list

  • revoke_api_token(p_link_id) → sets share_links.revoked_at and revokes the share (cascades to membership). Service user kept around for audit (so blocks updated by the token still resolve).
  • rotate_api_token(p_link_id) → generates a new token, updates the secret row, returns the new plaintext once. Old token stops working immediately.
  • list_my_api_tokens() → returns rows with everything except the token.

10. Open questions

  1. Service-user audit identity. Should blocks.created_by / updated_by store the service user's id, or the issuing user's id with a side reference to the token? Storing the service-user id is simpler and means the audit trail naturally reflects “this token did this” (linkable via share_links.service_user_id). Recommended.
  2. Token prefix scheme. Proposing km_live_ for production, km_test_ for any future dev/scratch instance. Trivial to extend.
  3. Per-token rate limits. Probably enforce in the exchange edge function (deny if last 100 exchanges were within 60s) rather than at PostgREST. Out of scope for v1 but worth a column on share_links if cheap.
  4. Workspace-scope human shares. Locked off in v1 by the workspace_scope_is_api_only constraint. Re-open if a real use case shows up — the RLS predicate is already ready.
  5. Snapshot RPC for workspace scope. get_shared_subtree works naturally for subtree shares; the workspace-scope equivalent returns potentially the whole workspace. Add pagination (p_after_id, p_limit) before exposing it.
  6. Token revocation lag. Issued JWTs live 15 min; an attacker with a stolen JWT keeps access for that window even after token revocation. Acceptable for v1; if not, the exchange function can mint shorter JWTs or we can move to a check-on-every-request model with a custom PostgREST GUC. Default: live with 15 min.

11. What this doc is not changing

All of the following from subtree-sharing.md remain authoritative and untouched by this proposal:

  • §3.5 Role monotonicity — same rule, applied identically to API token grants (you can't lower a service user's role by re-issuing a lower-role token; you have to revoke).
  • §3.7 Token isolation — same share_link_secrets table, same RLS, same hash storage. Tokens never appear in any sync stream.
  • §3.8 Presentation root clipping — generalized to a set instead of a single id, but the principle is unchanged.
  • §3.9 Single write chokepoint — all writes still go through Repo client-side, all server writes still go through SECURITY DEFINER RPCs or RLS-protected PostgREST.
  • §5.2-5.3 Concurrency + cross-row insert order — advisory lock + the “parent's insert fixes orphaned children” trick are unchanged.
  • §11 Test plan — reused wholesale; add four new tests:
    • Multi-root recompute: a share with roots {R1, R2}; descendants under either are reachable; revoking the share clears both subtrees.
    • Workspace-scope share: every block in W is visible to the recipient; blocks in a different workspace are not.
    • API token exchange: token → JWT → PostgREST read/write succeeds within scope, fails outside it.
    • Token rotation: old token stops working immediately; new token works.
  • §13 Guardrails — all 14 still apply; #1 (role monotonicity), #2 (token isolation), #7 (effectiveShareIds immutability), and #12 (idempotent redeem) are load-bearing for API tokens.

12. Summary

The sharing design and the API-access design are the same design. By generalizing block_shares to multi-root + scope-kind, and treating API tokens as share_links bound to per-token service users, we get:
  • One RLS predicate for humans-in-workspace, humans-via-subtree-share, agents-via-workspace-token, and agents-via-subtree-token.
  • One token-storage table with the same isolation invariants.
  • One exchange RPC family — redeem_share_link for humans, api_token_exchange for agents — sharing the same validation logic.
  • One sync stream serving every human share recipient regardless of scope.
  • Multi-root shares for free via the junction table; ergonomically valuable on the human side too.
The cost on top of the existing sharing doc is small: one column (scope), one constraint, one junction table, one new edge function, one new UI surface. The big-ticket items (effective_share_ids materialization, the recompute, RLS, token isolation, presentation clipping) carry over unchanged.
