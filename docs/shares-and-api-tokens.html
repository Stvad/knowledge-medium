Shares + API tokens

Design for scoped programmatic access. 2026-05-15. Standalone — supersedes subtree-sharing.md (historical reference only).
TL;DR. Agents and third parties need to read/write specific notes via an API. The right primitive is a share: a workspace-or-subtree scope plus an audience. An API token is the same shape as a future human invite — it puts a principal (a per-token service user) into a share's audience. One RLS predicate on blocks then covers everyone — workspace members, share recipients, and API tokens alike. PostgREST is the v1 API surface; an edge function swaps a token for a short-lived JWT and clients use Supabase as normal. The non-trivial piece is materializing "which subtree shares reach this block" onto a blocks.effective_share_ids column — workspaces can be 300k+ blocks and we can't walk parent pointers on every read.

Reading guide: §1–3 are the conceptual picture; §3a is a concrete end-to-end walkthrough; §4–7 are the schema/RLS/sync mechanics; §8 is the API-token lifecycle; §9–14 are operational (UI, phasing, tests, tradeoffs). If you only read three things: the TL;DR, §3a, and §6 with its tree picture.

1. What we have today

The current data model (supabase/migrations/20260510222352_consolidated_initial.sql):

  • public.blocks with an immutable workspace_id on every row.
  • public.workspaces + public.workspace_members (owner / editor / viewer).
  • public.workspace_invitations (email-keyed).
  • RLS on blocks keys off membership via private.is_workspace_member (read) and private.is_workspace_writer (write).
  • PowerSync streams scoped to "workspaces the user belongs to" (powersync/sync-config.yaml).
  • Read-only enforcement is a session-level flag on Repo (src/data/repo.ts — isReadOnly at line 459, setReadOnly at line 1014).
  • Local agent access today: a browser-paired relay, README.md §"Agent Runtime Access". Powerful (full JS eval inside the running app) but requires the browser tab.

What's not there: anything between "workspace member" (sees everything) and "stranger" (sees nothing). No scoped credentials, no programmatic API, no subtree-level access.

2. What we're adding, and why

  1. API tokens — long-lived, revocable credentials for agents and third-party integrations to read/write notes without a browser session. Each token is scoped at creation: either an entire workspace or one-or-more subtrees.
  2. Shares as the underlying primitive — the same primitive that will later back human "share this page" UX. Building it as the shared substrate now (even if only API uses it in v1) means we don't fork the auth surface later.
Scale constraint up front. A real workspace today is ~300k blocks (imported Roam graph). The design has to assume large workspaces from day one. That rules out any "walk parents at query time" RLS or sync predicate — access decisions on the read path must be O(1) per block.

3. The unifying primitive

A share is the access primitive. Everything else is a way of putting a principal into a share's audience. The diagram below shows how three different "ways of joining a share" all funnel into the same audience table, which a single RLS predicate then evaluates against.

Share
  workspace_id    text                          // always lives inside one workspace
  scope           'workspace' | 'subtree'       // entire workspace, OR one-or-more subtree roots
  roots           text[]                        // only for scope='subtree'; via junction table
  default_role    'viewer' | 'editor'
  purpose         'human' | 'api'               // drives UI/listing, not policy

Audience member (block_share_members)
  share_id, user_id (auth.users.id), role, source

Credential surface (share_links + share_link_secrets)
  link metadata (role, purpose, expiry, etc.)  // synced, no token
  token         hashed, separate row            // never synced, creator-only SELECT

A human invitation, a redeemable link URL, and an API token are three flavors of the same shape: an opaque secret → SECURITY DEFINER RPC that validates → row in block_share_members for the redeeming principal → RLS does the rest.

The reason this is worth doing as one design rather than two: one RLS predicate covers workspace members, workspace-scope shares, subtree-scope shares, human invites, link redemptions, and API tokens. Adding a "future" share type (e.g. by-tag) doesn't disturb anything.

3a. End-to-end: what happens when you create an API token

The rest of this doc gets into schema and triggers. To make the picture concrete first, here is one full scenario from token creation through a single API call to revocation.

The setup

You want to let Claude (an agent) read and write three project pages in your workspace. You open Settings → API tokens, click "Create token", and fill in: scope = subtree, roots = those three pages, role = editor, name = "Claude on laptop", expiry = 90 days.

What the server does (one transaction)

  1. Inserts a block_shares row: scope='subtree', purpose='api', default_role='editor'.
  2. Inserts three block_share_roots rows, one per chosen page.
  3. An AFTER INSERT trigger on block_share_roots fires recompute_share_ids_subtree for each root — this stamps the share's id onto blocks.effective_share_ids for the root and every descendant. The materialization picture in §6 shows what this looks like.
  4. Creates a dedicated service user (a auth.users row whose only purpose is to authenticate this token). This is the only step that needs service_role; it lives inside the edge function.
  5. Inserts a block_share_members row tying the service user to the share with role='editor', source='api_token'.
  6. Generates a token, km_live_<43 chars>, stores sha256(token) in share_link_secrets, stores share_links metadata pointing at the service user.
  7. Returns the plaintext token to your browser exactly once. You paste it into Claude's config.

What every Claude API call does

  1. Claude POSTs the token in Authorization: Bearer to /functions/v1/api-token-exchange.
  2. The edge function hashes the bearer, looks up the row in share_link_secrets, checks the share/link aren't revoked or expired, mints a 15-minute Supabase JWT signed with sub = service_user_id.
  3. Claude calls /rest/v1/blocks?id=eq.<some_block_id> with that JWT.
  4. Postgres evaluates the RLS predicate as the service user: is there a block_share_members row whose share covers this block?
    • For blocks in the three chosen subtrees and their descendants: yes — matched because the share id appears in that block's effective_share_ids. Row returned.
    • For any other block in the workspace: no — the service user has no workspace_members row and no other share grant. Row denied.

Revoking

You click "Revoke" in the UI. revoke_api_token sets revoked_at on the share and the link. The RLS predicate filters revoked shares out automatically — new requests fail. JWTs that were already issued continue to work until their 15-minute TTL expires.

That's the entire feature in one trace. Everything below is either the precise schema/RLS to make the trace work, or operational concerns (UI, phases, tests, tradeoffs).

4. Schema

All additive. Existing tables and migrations are not touched destructively. The picture below is the at-a-glance shape of the new tables; SQL details follow.

4.1 New column on blocks

alter table public.blocks
  add column effective_share_ids text[] not null default '{}';

create index idx_blocks_effective_share_ids
  on public.blocks using gin (effective_share_ids);

For each block: the set of subtree-scope block_shares.ids whose root is the block or an ancestor. Workspace-scope shares are NOT materialized here — they match by workspace_id directly. Reasoning in §6.

4.2 Shares + scope

create table public.block_shares (
  id                   text primary key,
  workspace_id         text not null references public.workspaces(id) on delete cascade,
  scope                text not null check (scope in ('workspace','subtree')),
  purpose              text not null default 'api' check (purpose in ('human','api')),
  default_role         text not null check (default_role in ('viewer','editor')),
  created_by_user_id   text not null,
  create_time          bigint not null,
  revoked_at           bigint
);
create index idx_block_shares_workspace_id on public.block_shares(workspace_id);

-- One row per (share, root). Only populated for scope='subtree'; enforced by trigger.
create table public.block_share_roots (
  share_id       text not null references public.block_shares(id) on delete cascade,
  root_block_id  text not null references public.blocks(id)        on delete cascade,
  workspace_id   text not null,
  create_time    bigint not null,
  primary key (share_id, root_block_id)
);
create index idx_block_share_roots_root_block_id on public.block_share_roots(root_block_id);

-- Stamped + checked equal to block_shares.workspace_id and the block's workspace_id at insert.
-- Prevents cross-workspace mistakes; share_root.workspace_id is immutable.
v1 constraint: human shares are subtree-only. The natural way to give a human access to an entire workspace is already workspace_members; a parallel "workspace share" path would add ambiguity (which mode are they in?) for no real win. Enforce: 
alter table public.block_shares
  add constraint workspace_scope_is_api_only
  check (scope = 'subtree' or purpose = 'api');
Where this is enforced, in decreasing order of authority:
  1. Database check constraint above — the canonical boundary. Holds regardless of which code path tries to violate it. Any drift in higher layers fails loudly at INSERT.
  2. create_api_token / future create_block_share RPCs pre-validate scope + purpose and return a clean error message before the constraint fires.
  3. Settings UI filters the scope picker to "subtree only" for human shares so the invalid combination isn't offered.
Drop the constraint if a real human use case for workspace-wide guest access appears. RLS already supports it.

4.3 Audience

create table public.block_share_members (
  id                   text primary key,
  share_id             text not null references public.block_shares(id) on delete cascade,
  user_id              text not null,
  role                 text not null check (role in ('viewer','editor')),
  source               text not null check (source in ('invite','link','api_token')),
  source_link_id       text,
  granted_by_user_id   text,
  create_time          bigint not null,
  unique (share_id, user_id)
);
create index idx_block_share_members_user_id  on public.block_share_members(user_id);
create index idx_block_share_members_share_id on public.block_share_members(share_id);

Inserts use GREATEST(existing.role, new.role) semantics on the unique constraint — granting the same share at a lower role does not silently demote. The one place that may lower a role is the explicit update_share_member_role RPC.

4.4 Credential surface

-- Metadata. Sees the light of day; tokens are stored in share_link_secrets.
create table public.share_links (
  id                  text primary key,
  share_id            text not null references public.block_shares(id) on delete cascade,
  role                text not null check (role in ('viewer','editor')),
  purpose             text not null default 'api_token'
                        check (purpose in ('human_link','api_token')),
  name                text,                       -- "Claude on laptop", "Readwise sync", etc.
  service_user_id     text,                       -- for purpose='api_token'; null for human links
  created_by_user_id  text not null,
  create_time         bigint not null,
  expires_at          bigint,
  max_redemptions     integer,
  redemption_count    integer not null default 0,
  last_used_at        bigint,
  last_used_ip        text,
  revoked_at          bigint
);
create index idx_share_links_share_id        on public.share_links(share_id);
create index idx_share_links_service_user_id on public.share_links(service_user_id);

-- Token storage isolated from share_links so RLS on share_links can stay relaxed
-- (workspace writers can list their own links' metadata) without leaking tokens.
-- Stored hashed: token plaintext leaves Postgres exactly once, at creation.
create table public.share_link_secrets (
  link_id             text primary key references public.share_links(id) on delete cascade,
  token_hash          bytea not null unique,       -- sha256(plaintext_token)
  created_by_user_id  text not null
);
create index idx_share_link_secrets_token_hash on public.share_link_secrets(token_hash);

4.5 PowerSync publication

  public.block_shares,

  public.block_share_members,

share_link_secrets is intentionally NOT published. Tokens never appear in any sync stream.

5. RLS — one read, one write

The single block-read policy covers all four cases (workspace member, workspace-scope share recipient, subtree-scope share recipient, API service principal). Existing helpers private.is_workspace_member / private.is_workspace_writer stay as-is.

drop policy if exists blocks_read  on public.blocks;
drop policy if exists blocks_write on public.blocks;

create policy blocks_read on public.blocks
for select using (
  -- 1. Workspace member (existing path)
  private.is_workspace_member(workspace_id, (auth.uid())::text)

  -- 2. Workspace-scope share / API token
  or exists (
    select 1
    from public.block_share_members sm
    join public.block_shares s on s.id = sm.share_id
    where sm.user_id = (auth.uid())::text
      and s.revoked_at is null
      and s.scope = 'workspace'
      and s.workspace_id = blocks.workspace_id
  )

  -- 3. Subtree-scope share / API token, via materialized array
  or exists (
    select 1
    from public.block_share_members sm
    join public.block_shares s on s.id = sm.share_id
    where sm.user_id = (auth.uid())::text
      and s.revoked_at is null
      and s.scope = 'subtree'
      and s.id = any (blocks.effective_share_ids)
  )
);

create policy blocks_write on public.blocks
for all
  using       (/* same shape; sm.role = 'editor' in both share branches; is_workspace_writer for #1 */)
  with check  (/* same as using */);

RLS on the new tables themselves:

  • block_shares, share_links: readable by creator and workspace writers (so the in-app UI can list). All mutations via security-definer RPCs.
  • block_share_members: readable by the member themselves, the share creator, and workspace writers (not plain members — workspace viewers should not enumerate API tokens).
  • block_share_roots: readable by anyone who can read the parent share row.
  • share_link_secrets: readable only by the link creator. The exchange RPC reads via security definer.

6. Materialization — effective_share_ids

Why this exists. With 300k blocks per workspace, RLS cannot afford to walk parent_id at query time. Sync rules especially are evaluated per row by PowerSync; a recursive predicate there is a non-starter. We pre-compute "which subtree-scope shares cover this block" and stamp it on the row. Workspace-scope shares deliberately don't materialize — they'd require a one-shot 300k-row update on every share create/revoke. The RLS branch on workspace_id is O(1) per block, already indexed.

The picture is the easiest way to grasp this. When share s1 is created with one of its roots set to block A, a trigger walks the descendants of A and stamps s1 onto every block's effective_share_ids array. RLS later just asks "is any share id in this block's array one I'm a member of?"

A second share s2 nested at, say, block D would result in D and G carrying [s1, s2]. Different audiences can be granted to s1 and s2 — effective access becomes the union, role becomes the max. The array shape (rather than a scalar nearest-ancestor) is what makes nesting work without losing the outer audience.

6.1 Recompute function

One function, one recursive CTE. Walks descendants from a root, unioning each block's existing effective_share_ids contribution from the parent with any new share ids anchored at the block itself. The result is an O(subtree-size) UPDATE on public.blocks.

create or replace function public.recompute_share_ids_subtree(p_root_block_id text)
returns void
language plpgsql
volatile
as $$
declare
  v_lock_key bigint := hashtext('subtree:' || p_root_block_id);
begin
  -- Serialize concurrent recomputes touching this root. Released on commit/rollback.
  perform pg_advisory_xact_lock(v_lock_key);

  with recursive
    parent_share_ids as (
      select coalesce(parent.effective_share_ids, '{}'::text[]) as ids
      from public.blocks self
      left join public.blocks parent on parent.id = self.parent_id
      where self.id = p_root_block_id
    ),
    walk(block_id, share_ids) as (
      select
        p_root_block_id,
        (select ids from parent_share_ids)
        || coalesce((
          select array_agg(s.id)
          from public.block_share_roots r
          join public.block_shares     s on s.id = r.share_id
          where r.root_block_id = p_root_block_id
            and s.scope = 'subtree'
            and s.revoked_at is null
        ), '{}'::text[])
      union all
      select
        child.id,
        walk.share_ids
        || coalesce((
          select array_agg(s.id)
          from public.block_share_roots r
          join public.block_shares     s on s.id = r.share_id
          where r.root_block_id = child.id
            and s.scope = 'subtree'
            and s.revoked_at is null
        ), '{}'::text[])
      from walk
      join public.blocks child on child.parent_id = walk.block_id
    )
  update public.blocks
  set effective_share_ids = (
    select array(select distinct unnest(walk.share_ids) order by 1)
    from walk
    where walk.block_id = blocks.id
  )
  from walk
  where blocks.id = walk.block_id;
end $$;

6.2 Triggers calling it

TriggerBehavior
blocks AFTER INSERT Set NEW.effective_share_ids to parent.effective_share_ids (or '{}' if no parent). One-row update, no recursion. Also fire a fix-up for pre-existing children of NEW whose array is empty — handles PowerSync uploading child rows before their parent.
blocks AFTER UPDATE OF parent_id Call recompute_share_ids_subtree(NEW.id). The moved subtree gets reconsidered against its new ancestor chain.
block_share_roots AFTER INSERT Call recompute_share_ids_subtree(NEW.root_block_id). Stamps the share id onto the root and all descendants.
block_share_roots AFTER DELETE Call recompute_share_ids_subtree(OLD.root_block_id). Same walk; the deleted share no longer appears in the array.
block_shares AFTER UPDATE OF revoked_at For scope='subtree': recompute on each root in block_share_roots. For scope='workspace': no recompute needed.

6.3 Cost in practice

  • Subtree share over a 5k-block region: a single UPDATE on ~5k rows. Postgres handles this in low single-digit seconds even on Supabase's smaller tiers.
  • Workspace-scope share: zero block writes (matches via workspace_id at RLS time).
  • Re-parenting a 1k-block subtree: one recompute of 1k rows.
  • Day-to-day block edits inside a non-shared subtree: zero recompute work.
Concurrency, deliberately deferred. The advisory lock above prevents simultaneous recomputes on the same root. It does NOT serialize cross-root interactions (e.g. moving a subtree out of share A while simultaneously creating share B at an ancestor). If we hit racy outcomes in practice, the fix is either widening the lock key (e.g. lock on workspace_id) or driving recompute through a per-workspace job queue. Neither is needed in v1; ship the simple version and add the property test from §11 to catch divergence.

7. PowerSync sync rules

One additional stream covers all share recipients regardless of scope. Add to powersync/sync-config.yaml via scripts/gen-sync-config.ts:

shared_blocks:
  auto_subscribe: true
  with:
    user_share_grants: |
      SELECT
        block_share_members.share_id,
        block_shares.workspace_id,
        block_shares.scope
      FROM public.block_share_members
      JOIN public.block_shares
        ON block_shares.id = block_share_members.share_id
       AND block_shares.revoked_at IS NULL
      WHERE block_share_members.user_id = auth.user_id()
  queries:
    - |
      SELECT blocks.id, blocks.workspace_id, blocks.parent_id,
             blocks.order_key, blocks.content,
             blocks.properties_json, blocks.references_json,
             blocks.created_at, blocks.updated_at,
             blocks.created_by, blocks.updated_by,
             blocks.deleted, blocks.effective_share_ids
      FROM public.blocks
      JOIN user_share_grants g
        ON (g.scope = 'workspace' AND g.workspace_id = blocks.workspace_id)
        OR (g.scope = 'subtree'   AND g.share_id   = ANY(blocks.effective_share_ids))

And add effective_share_ids to the existing workspace_data stream's blocks SELECT, so members see it for share-badge UI.

Verify ANY(array) support in PowerSync stream queries before merging. If unsupported, fall back to a junction table block_share_coverage(block_id, share_id) maintained by the same triggers; only the storage shape changes, all RLS / RPC logic carries over unchanged.

8. API tokens

An API call has two hops: first the client trades its long-lived bearer token for a short-lived Supabase JWT, then it uses that JWT against PostgREST exactly like the in-app client would. The diagram below shows the full round trip; subsections below give the details.

8.1 Why service-users-per-token

An API token authenticates as a dedicated auth.users row (a "service user") whose only membership grant is the token's share. This makes RLS uniform — the policy in §5 doesn't need to know anything about tokens. It also gives clean audit: blocks.updated_by is the service user's id, joinable to share_links.service_user_id to identify the responsible token.

Considered: minting a JWT with the issuing user's sub plus a custom api_token_scope claim, and reading the claim inside RLS via current_setting('request.jwt.claims'). This avoids creating perpetual auth.users rows, but every RLS branch grows a claim check, and the audit story is worse (writes attributed to the issuing user with a side-channel for "which token"). The service-user approach is what we ship.

8.2 Creation

One server-side RPC, called from the in-app settings UI:

create_api_token(
  p_scope         text,           -- 'workspace' | 'subtree'
  p_workspace_id  text,           -- required
  p_roots         text[],         -- required for subtree; empty for workspace
  p_role          text,           -- 'viewer' | 'editor'
  p_name          text,           -- human label
  p_expires_at    bigint          -- ms epoch, null = never
) returns table (
  share        public.block_shares,
  link         public.share_links,
  token_plain  text                -- shown exactly once
)
language plpgsql security definer set search_path = public

Inside the RPC:

  1. Verify caller is a workspace writer for p_workspace_id.
  2. For subtree scope, verify caller can write each root block.
  3. Insert block_shares + block_share_roots rows. Recompute trigger stamps effective_share_ids for subtree case.
  4. Service user creation is the one step that needs service_role: call out to POST /auth/v1/admin/users via a Supabase Edge Function wrapping this RPC, or register an extension. The service user gets an opaque generated email (svc-<random>@knowledge-medium.local) and a random unrecoverable password.
  5. Insert block_share_members(share_id, user_id = service_user_id, role, source='api_token').
  6. Generate the token in the edge function (not in SQL — Postgres' encode() only supports 'base64' | 'hex' | 'escape', not 'base64url', and keeping plaintext out of the database avoids it appearing in pg_stat_statements or replication logs): 
    const plaintext = 'km_live_' + crypto.randomBytes(32).toString('base64url')
const hash      = crypto.createHash('sha256').update(plaintext).digest()
  7. Pass only hash into the SQL RPC; insert share_links + share_link_secrets(token_hash = hash).
  8. Return plaintext from the edge function to the caller, once. Postgres never sees the plaintext.
Practical packaging. Because step 4 needs service_role, the cleanest shape is a single Edge Function supabase/functions/create-api-token that the in-app client calls. The function authenticates the caller (their Supabase JWT), validates with RLS-style SECURITY DEFINER helpers, creates the service user, and then runs the rest of the RPC. Keeping service_role inside the function and never on the client is the security boundary.

8.3 Exchange (every API call)

Third parties hit an edge function supabase/functions/api-token-exchange with the token in Authorization: Bearer. It:

  1. Hashes the bearer, looks up share_link_secrets by hash.
  2. Validates: link not revoked, not past expires_at, share not revoked.
  3. Bumps last_used_at / last_used_ip.
  4. Mints a Supabase JWT signed with the project secret. Claims: sub = service_user_id, role = 'authenticated', aud = 'authenticated', TTL set by env (see below).
  5. Returns the JWT plus a small share stanza so the client knows scope/workspace.

TTL is a deployment-level setting, read from API_TOKEN_JWT_TTL_SECONDS in the edge function's environment. Default: 900 (15 minutes). Higher-security deployments can lower it (e.g. 60s — effectively per-request); the cost is one extra exchange round-trip per API call. v1 keeps it global; per-token override is a future addition (column on share_links) if the use case appears.

8.4 What 3rd parties use the JWT for

Direct PostgREST, talking to the same Supabase URL the app uses. Examples:

GET  /rest/v1/blocks?workspace_id=eq.<ws>&deleted=eq.false&select=id,content,properties_json
GET  /rest/v1/blocks?parent_id=eq.<id>&order=order_key.asc
POST /rest/v1/blocks                              -- editor only
PATCH /rest/v1/blocks?id=eq.<id>             -- editor only
POST /rest/v1/rpc/<some_rpc>

The same JWT works against any RPC the service user is allowed to call. RLS in §5 enforces scope automatically.

v1 stance: raw PostgREST is the API. We accept the coupling to column names (including the JSON-encoded-string properties_json / references_json shape, which is mid-evolution — see properties-model-evolution.md). If the schema needs to break compatibly later, we add a /api/v1/… Edge Function layer at that point. Don't build it speculatively.

8.5 Revocation, rotation, list

revoke_api_token(p_link_id text) returns void;
  -- Sets share_links.revoked_at and block_shares.revoked_at.
  -- Service user is NOT deleted -- preserves audit trail for past writes.

rotate_api_token(p_link_id text) returns text;
  -- New plaintext token; updates share_link_secrets.token_hash; old token stops working.

list_my_api_tokens() returns table (... no token, no hash ...);
  -- For settings UI.
Revocation lag. Issued JWTs live for the configured TTL (default 15 minutes — see §8.3). Revoking a token does not invalidate an already-issued JWT until it expires. Acceptable for v1; lower API_TOKEN_JWT_TTL_SECONDS for strict-revocation deployments, or add a per-request check against share_links.revoked_at in an RLS-callable function for near-zero-lag revocation at the cost of one extra row read per request.

9. UI surface

One new screen in v1: Settings → API tokens.

  • List existing tokens with name, scope summary ("entire workspace W" / "subtree of B + 2 more"), role, last used, expiry.
  • "Create token" dialog: scope picker (workspace toggle, or a block multi-picker), role, name, expiry. Returns the plaintext token in a copy-once panel.
  • Per-token: rotate, revoke.
  • Subtree picker is the same component that will later back the share dialog — build it once.
  • Expiration reminders. Rows for tokens within 14 days of expiry show a warning badge in the list and surface an in-app notification on app load. Rows past expiry show a distinct "expired" badge and stop being usable (the exchange function already refuses them; the UI just makes the state visible so a token doesn't silently lock out an integration). Email reminders are a follow-up — in-app is enough for v1.
  • Last-used telemetry from share_links.last_used_at / last_used_ip is shown per row, so the user can spot unused tokens to revoke and notice unexpected activity.

Human subtree-sharing UX (share dialog on a block bullet, invite by email, redeemable link URLs, share-mode bootstrap, presentation root clipping) is the follow-up. Schema and RLS already support it; only the client routing, dialog, and snapshot RPC for read-only links remain.

11. Test plan

The materialization is the load-bearing piece — if effective_share_ids is wrong, RLS and sync silently leak or hide blocks. Build the property test first:

  1. Recompute property test. Random sequence of (create_share | revoke_share | add_root | remove_root | move_block | insert_block | delete_block). After each op, every block's effective_share_ids must equal an oracle computed inline (recursive CTE walking parent_id upward, unioning block_share_roots matches for non-revoked subtree shares). ≥1000 random sequences.
  2. RLS matrix. Two workspace users + two service-user tokens (one workspace, one subtree). Assert SELECT/INSERT/UPDATE/DELETE on blocks match the expected success/fail matrix.
  3. Token isolation. Querying share_link_secrets as a non-creator returns zero rows. No row of share_links, block_shares, or block_share_members ever carries a token in any column or sync stream.
  4. Exchange round-trip. Create a token; exchange; use the JWT against /rest/v1/blocks and confirm scope is enforced; revoke; subsequent exchanges fail; the previously-issued JWT continues working until TTL expiry (documented behavior).
  5. Rotation. Rotate a token; old hash gone; old plaintext rejected; new plaintext succeeds.
  6. Cross-row insert order (PowerSync replay). Insert a child before its parent in a transaction; verify the after-insert trigger fixup catches the child once the parent lands (§6.2 row 1).
  7. Concurrent move + share-mutation. One transaction creates a share at root R; another moves a subtree of R out from under it. Both commit. Final effective_share_ids matches the oracle. (This is the case the advisory lock handles for overlapping subtrees; widen the lock or fix via queue if oracle disagrees.)

12. Tradeoffs we're accepting (and where they lead)

ChoiceCostMitigation / exit
Raw PostgREST as the v1 API surface Column names (properties_json, references_json) become a quasi-public contract; schema evolution constrained. Add a thin /api/v1/ Edge Function layer when forced; PostgREST remains for in-app use. Don't pre-build.
Materialized effective_share_ids Trigger complexity; subtree-share creation cost is O(subtree size); concurrent moves + share-mutations need property test coverage. Property test in CI. Lock-widening or job-queue if races appear.
Service users per API token Perpetual auth.users rows; service_role required (in an Edge Function) to create them. Acceptable. Audit trail is the win.
15-minute JWT TTL after exchange Revoking a token doesn't kill in-flight JWTs for up to 15 min. Shorten TTL or add an explicit revoke check in RLS if needed.
Workspace-scope shares are API-only in v1 No "guest with whole-workspace read access" for humans. Drop the check constraint; finish the human bootstrap path. RLS already supports it.
Single advisory lock per root in recompute Cross-root races (move out of A while creating B at an ancestor) not serialized. Property test will surface; widen lock to workspace_id or use a per-workspace job queue.

13. Explicitly out of scope

  • Per-block or per-property granularity (read this block but not this property).
  • Append-only / comments-only roles.
  • Tag-based shares ("everything tagged #public").
  • Mixed-mode permission resolution for users with both workspace membership and overlapping share grants. When human sharing lands, a workspace member who is also a share recipient uses their workspace role.
  • Anonymous-user redemption (for human "open a link without an account" UX). Deferred to the human-sharing follow-up phase.
  • Service-user garbage collection. Revoked tokens leave their service user around for audit.
  • Cross-workspace block moves — still rejected by the existing immutability rule.
  • Audit log of share grants/revocations beyond the timestamps already on the row.
  • Realtime updates to read-only snapshot consumers; v1 read-only snapshots (if/when added) require re-fetch.

14. Summary

The unification is the design: API tokens are shares whose audience is a per-token service user. One RLS predicate, one materialization column, one credential surface. v1 ships only the API-token flavor; human subtree sharing drops onto the same schema as a follow-up with no rework.

The 300k-block workspace constraint is what forces materialization despite the trigger complexity — recursive RLS isn't viable at that scale. Workspace-scope shares dodge the cost entirely by matching on workspace_id directly. PostgREST stays the API surface in v1; we accept the column-name coupling and reserve /api/v1/ for the day schema churn demands it.
