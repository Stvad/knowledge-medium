Media attachments

Design for image (and, by extension, arbitrary file) attachments in a system that now has per-workspace opt-in end-to-end encryption. 2026-06-21.

TL;DR. An attachment is a block carrying the registered media type (in the types array, via typesProp), embedded into prose with the existing !((id)) mechanism — no new reference syntax, and metadata rides the existing encrypted block-content columns for free. The bytes live in an object store (Supabase Storage v1), behind a BlobStore interface. The reshaping constraint is E2EE: for an encrypted workspace the server holds only ciphertext and never the key, so bytes are sealed client-side with the workspace key and access becomes entirely client-side. We do not write two implementations — there is one byte transform parameterized by mode, where plaintext is the identity case, exactly as src/sync/transform.ts already does for text. Decryption happens in the service worker behind a self-describing binary envelope, so the renderer stays mode-agnostic (<img src="${BASE_URL}asset/<ws>/<id>"> "just works" for both modes — base-prefixed so it stays inside the SW scope, §7). The load-bearing risk is making the SW a participant in the E2EE lock/wipe lifecycle.

1. What we're adding, and scope

The app cannot currently insert images. We want paste / drop / file-pick of an image to render inline, work offline, and sync across a user's devices — and we want the same machinery to extend to PDFs, audio, video, and generic files later without a content migration.

In scope (v1): images, via paste / drop / file picker; inline render with a lightbox; offline-first local caching; cross-device sync; correctness under E2EE workspaces.

Designed-for but not built (vNext): non-image file types (the model is type-dispatched, so a new mime is a new renderer branch, not a schema change); thumbnails / transforms; resumable uploads for large media; public sharing.

Explicitly out of scope: server-side media features for E2EE workspaces (OCR, thumbnails, transcode) — the server has no key; this is the existing E2EE tradeoff (e2ee-design.html §15), inherited, not new. E2EE subtree sharing of media (needs the deferred key hierarchy — see §12).

3. The core model: an attachment is a block

An attachment is not a row in a side table. It is a blocks row carrying the media type and binary-pointer metadata in its properties:

// asset block (illustrative). The type lives in the `types` ARRAY (typesProp),
// NOT a `properties.type` scalar — that's what the type registry, type queries,
// and the property panel read. Set via block.addType('media') / typesProp.
properties: {
  types:       ['media'],       // typesProp — registered media type id
  mime:        'image/png',
  size:        184213,          // plaintext byte length
  hash:        'sha256:…',      // of plaintext bytes; client-side dedup key
  filename:    'screenshot.png',
  width:       1280,            // optional, for layout reservation
  height:      720
  // NOTE: no upload `status` here — that's per-device, tracked in the local
  // upload queue (§9), NOT in synced (encrypted) properties_json. See §8.
}
content: 'image: screenshot.png'   // human-readable summary

The media type is registered as a TypeContribution — defineBlockType({id: 'media', label, properties}) contributed via typesFacet.of(mediaType, {source: 'attachments'}) (the pattern daily-notes / geo / block-tagging use), so type-filtered queries, the property panel, and renderer discovery all see media blocks. This is not the same as video-player's setup, which contributes property schemas + a renderer but no type — and a bare property schema is insufficient: the capture path's repo.addTypeInTx(tx, id, 'media', …) looks the id up in the type registry (types.get(typeId)) and throws unless a TypeContribution was registered through typesFacet. (Stamping a bare properties.type scalar instead would also make blocks invisible to every types-array consumer — getBlockTypes / hasBlockType / block.hasType.)

Why a block, not an attachments table:

  • Inspectable / user-accessible — the project's standing principle. An attachment shows up in the outline, the properties panel, queries, and the agent runtime like any other block.
  • Embeds, backlinks, annotations come for free. Inline placement uses the existing block embed !((id)) (src/components/references/BlockEmbed.tsx); captions / alt-text / OCR / transcripts are child blocks; "what references this image?" is the existing backlink machinery.
  • Type-dispatched rendering already exists. A MediaBlockRenderer registers via blockRenderersFacet with a canRender gate on block.hasType('media') — the same registration pattern as VideoPlayerRenderer. New file types are new mime branches, never a schema change.
  • Dedup within a subtree is natural: paste the same screenshot ten times into a page and you get ten !((id)) embeds of one asset block, one byte copy. (Reuse is scoped to the containing subtree, not workspace-wide, to stay compatible with the §12 share model — see §11.)

4. The constraint that reshapes everything: E2EE

For an E2EE workspace the server holds only ciphertext and never the key. The workspace key (WK) lives as a non-extractable WebCrypto CryptoKey in IndexedDB (km-e2ee-keys, keyed by (user_id, workspace_id) — src/sync/keys/keyStore.ts). The server-side trigger blocks_require_ciphertext_for_e2ee actively rejects any plaintext write to an encrypted workspace's content columns.

The E2EE design doc encrypts exactly three text columns and says nothing about binary data — confirmed by inspection. So attachments are an unsolved problem that must be solved consistently with E2EE, not bolted on beside it. Two pillars of any naïve "URL in markdown" approach are dead on arrival:

Signed-URL-on-render serving displayable bytes — dead for E2EE. A signed Storage URL returns ciphertext; the browser cannot <img src> that. Bytes must be fetched, then decrypted client-side with the WK before display.
Public-bucket / public-share-URL — incompatible with E2EE. You cannot serve a publicly-readable image out of an encrypted workspace via RLS + signed URLs. RLS gates access to ciphertext; it cannot grant decryption. (This is just e2ee §15 #2 — "no server-side features for E2EE workspaces" — applied to bytes.) The escape hatch survives in a different form; see §13.

What the encryption model gives us in return is also worth stating: because metadata rides the encrypted properties_json column, an attachment's mime, filename, and hash are confidential automatically on an E2EE workspace. The block-row routing identifiers (id, workspace_id) are in clear — exactly the columns the sync transform already leaves in clear.

But the Storage object leaks more than the block row does — call it out, don't fold it into the inherited block-metadata leak. The encrypted blob's byte length (≈ plaintext size + 28 B) is visible to the server via the stored object size and the Content-Length of an RLS-gated GET, and the object's existence and per-workspace count are visible. Exact file size is a much stronger fingerprint than note-text length — it can confirm a specific known file (a leaked document, a known image) by size, and combined with the asset_block_id in the path + timing, correlate which block embeds which known-size asset. This is a new leak outside the enumerated e2ee §15 set (tree shape / order / timing / authorship). Document it as an accepted tradeoff (§17), or pad ciphertext to size buckets if the threat model cares.

5. One byte transform; plaintext is the identity case

E2EE is opt-in, so most workspaces are plaintext. The temptation is two code paths. The codebase already shows the better answer: src/sync/transform.ts is one function parameterized by mode, where mode === 'none' is the identity transform.

// the text precedent, today
encodeForWire(row, 'none', getCek)  // → row unchanged
encodeForWire(row, 'e2ee', getCek)  // → seal content/properties_json/references_json

We mirror it for bytes — a new module in the same src/sync/ seam:

encodeBytes(bytes: Uint8Array, mode: SyncMode, getCek: GetCek): Promise<Uint8Array>
decodeBytes(blob:  Uint8Array, mode: SyncMode, getCek: GetCek): Promise<Uint8Array>
//   mode === 'none'  → identity (return input)
//   mode === 'e2ee'  → AES-256-GCM seal/open with the WK
Decision. There is one attachment implementation, not two. Everything expensive — the asset-as-block model, the BlobStore, the cache, the upload queue, dedup, the renderer, Lock-and-Wipe participation — is mode-agnostic and shared. Only the ~15-line encode/decode seam is mode-aware, and plaintext is its degenerate (identity) branch, exactly like encodeForWire.

5.1 Reusing the crypto primitives

The WK CryptoKey is AES-256-GCM and works for crypto.subtle.encrypt over raw bytes, not just the string seal/open in src/sync/crypto/aead.ts. So byte encryption reuses the same getCek(workspaceId) seam the sync transform uses, with two deltas:

  • AAD binding extends the existing pattern. Where content columns bind contentAad(id, workspace_id, column), asset bytes bind (asset_block_id, workspace_id, "asset-bytes", schema_version) so a ciphertext blob cannot be swapped between blocks or workspaces. (Schema version stays in the AAD for the same cheap-hierarchy-upgrade reason §13 of the e2ee doc keeps it there.)
  • A binary envelope, not the base64url text envelope. The enc:v1: envelope is base64url (≈33% bloat) — wrong for bytes. Add a sibling encb:v1: binary framing (magic ‖ nonce ‖ ciphertext‖tag as raw bytes) sharing the nonce/tag layout. Small new module next to aead.ts.
The AAD does not bind freshness — verify the content hash on read. Every AAD term (asset_block_id, workspace_id, the literal, schema version) and the storage path are stable across an asset's whole life. AAD stops cross-block / cross-workspace swaps, but it does not stop a rollback/replay of an older ciphertext for the same asset_block_id: an untrusted server (in scope per e2ee §2 — it may serve whichever bytes it likes) can return a stale blob that decrypts cleanly and the GCM tag verifies. The bytes lane has no updated_at/monotonicity channel the text path gets from PowerSync. The fix is already in hand: the asset block stores hash = sha256(plaintext) in its (encrypted) properties_json; the read path must verify the decrypted bytes against that hash before caching/serving, and reject on mismatch. Because the SW can't read the encrypted block state, it gets the expected hash from the SW-readable metadata mirror (§7.1.1) — not from the request (an <img src> can't carry custom headers, and a ?h= query param would leak the plaintext SHA-256, §7.1.1). It's a positive check that can only fail closed, never downgrade.
Read-side fail-closed is not enough — the write lane must be monotonic too. A new hash makes reads reject stale bytes, but two uploads targeting the same <ws>/<asset_block_id> path can still race: if an asset is edited and a stale queued/in-flight upload lands after the newer one, the old body overwrites the object after the block's hash already advanced — then every reader rejects on mismatch and the asset is permanently broken (the newer queue entry was already confirmed and dropped). The fix is immutable, write-once asset objects: an edit/replacement creates a new asset block (new id → new path) and re-points the embed, rather than re-uploading under the existing id. Bytes under a given asset_block_id never change, so there is no same-path collision and no "older version" for the server to replay; the §10.1 upload guard enforces it with a no-overwrite precondition (reject if the object already exists). The old object orphans and is GC'd (§16). (Dedup still reuses an existing immutable asset block by hash — that's a read, not a rewrite.)

The encb:v1: magic is a format tag, not an authorization signal. Whether to decrypt is decided by the workspace's mode pin (§7), exactly as the text path decides via the pin rather than the enc:v1: prefix (envelope.ts / e2ee §6 rule 2). An E2EE-pinned workspace requires the magic and treats its absence as tampering; only a plaintext-pinned workspace passes bytes through. A workspace with no pin (never validated, or evicted) is not plaintext — it is the resolver's defer/encryption-uncertain state (src/sync/keys/resolver.ts), so the SW fails closed (placeholder), never passthrough — otherwise an encrypted workspace with a missing local pin would serve attacker-controlled plaintext.

6. The byte path, end to end

Bytes do not ride PowerSync. The asset block (metadata) syncs through the normal Layout B path (encrypted content columns, observer-reconciled). The bytes take a parallel path to the object store. The two arrive independently on other devices.

7. Service-worker decryption (the unification)

The cleanest way to keep the renderer mode-agnostic is to decrypt in the service worker. This is viable — the SW global scope has crypto.subtle, IndexedDB, and the Cache API — and it collapses the entire plaintext/E2EE branch into one place the renderer never sees.

  • The renderer emits a stable virtual URL — BASE_URL-prefixed. <img src="${BASE_URL}asset/<workspace_id>/<asset_block_id>">, not a root-absolute /asset/… (shorthand below). The SW registers with scope: import.meta.env.BASE_URL (src/registerServiceWorker.ts), so under a non-root APP_BASE_PATH (e.g. /app/) a root-absolute URL escapes the SW scope entirely — the worker never sees the request, the auth/decrypt path is bypassed, and the origin returns 404. Build the URL relative to BASE_URL and match the ${BASE_URL}asset/… route in the worker. No mode awareness, no createObjectURL lifecycle for inline subresources (<img>, <video>) from a controlled tab. A raw <a download href="${BASE_URL}asset/…"> is not covered — a download/navigation can lack a clientId to authorize (§7.1), so "download original" is an app-mediated action (the app resolves the bytes and saves from an in-memory blob), not a raw link to the asset URL. (Range/streaming is also an exception under E2EE — see the note below; images and moderate files are unaffected.)
  • The SW resolves it, auth-first: establish the active user_id (for the requesting client — §7.1) and the workspace's mode pin + WK availability before consulting the cache → serve from cache only on a namespace match, a still-valid session, and a matching content hash (from the SW metadata mirror; the cache key includes it, §8) → on miss, fetch the stored blob → decrypt/passthrough per the mode pin (below) → verify against the mirror's expected hash (§5.1) → put the displayable bytes in the user-scoped, hash-keyed cache → respond.
  • The mode pin decides decrypt-vs-passthrough, NOT the envelope magic. This is a deliberate carry-over of the rule the text path already enforces (src/sync/crypto/envelope.ts / e2ee §6 rule 2: "the prefix is a FORMAT tag, not a decrypt trigger — the durable mode pin decides"). The SW reads the workspace's mode pin from a SW-readable source (§7.1.1): if the workspace is E2EE-pinned, the SW requires a well-formed encb:v1: envelope and decrypts — a blob that arrives without the magic is treated as tampering and rejected, never served as plaintext. Only a plaintext-pinned workspace passes bytes through; an unpinned / no-pin workspace is the resolver's defer state and fails closed (not plaintext). Deciding from the self-describing magic alone would re-introduce exactly the downgrade vector (hostile server strips the magic → attacker plaintext reaches <img>) that the text wire-boundary was hardened against.
  • Render gates on SW readiness (cold start). An <img src="${BASE_URL}asset/…"> can fire before the app has handed the SW a token (§7.1) or before the SW has re-read its state on a cold respawn. The SW must not "fail open": a request with no established active user / no fresh unlock signal respondWiths a deterministic placeholder (or holds briefly on a readiness promise, short timeout → placeholder), never a stale cache hit and never an unauthorized fetch. But the SW can only do this if it is controlling the page — and on a first production visit (or where SW registration is unavailable) it is not: public/sw.js deliberately omits clients.claim(), so a fresh install controls only the next load. An ${BASE_URL}asset/… <img> emitted while uncontrolled goes straight to the origin (→ 404, bypassing all of the above). So the renderer must gate URL emission on navigator.serviceWorker.controller: only emit the virtual URL when a controller exists; otherwise render a placeholder (and retry once controlled) or fall back to the app-thread resolve (§7.3). Don't rely on the SW catching an already-dispatched request — an uncontrolled page never reaches it.

7.1.1 The SW needs three things the renderer/app hold — make them SW-readable

The URL carries only workspace_id and asset_block_id. The decrypt path needs two more facts that today live where the SW cannot reach them — this is a real data-availability requirement, not an aside:

  • Mode pin → a SW-readable durable store (IndexedDB). Pins live in window.localStorage (src/sync/keys/modePin.ts), which a service worker cannot read. The SW needs a durable IndexedDB source, sharing the pin's "survives Lock & Wipe" rule (the mode must stay known after a lock so the key can be re-prompted — e2ee §6). The mode is authoritative, so it must come from this durable pin, never from a per-request claim (an app-claimed "plaintext" would be the downgrade vector again). If no pin exists for the workspace yet (never validated), the SW fails closed — never a magic-fallback.
    Decision: move the pin to IndexedDB as the single source of truth (not a localStorage mirror) — one store, no dual-write bookkeeping. The modePin module's backing changes from localStorage to IDB; both the app and the SW read the same store.
    • Async is a non-issue because the app keeps a boot-hydrated in-memory cache: at data-layer init (where the keyStore / DB already open) load all pins — a handful, one per workspace — into an in-memory map, and the app's synchronous getModePin reads off that map. Writes are write-through (sync map + async IDB) and are rare (only on mode establishment / key validation), so async IDB never touches the hot read path. The SW reads IDB directly, but its fetch handler is already async (and the §7.1.1 in-SW cache below amortizes it).
    • Durability downgrade is acceptable and fails closed. IDB is evictable under storage pressure (localStorage less so) — but a missing pin for an encrypted workspace resolves to defer (don't materialize, re-prompt), never "plaintext". So eviction → user re-affirms workspace status → pin re-established; it can never silently downgrade. Preserve the existing pin semantics through the move: survives Lock & Wipe (not purged) and keyed per (user_id, workspace_id) — the IDB store is shared across accounts in the profile but each record is per-user, exactly as modePin.ts keys today; a workspace-only record would let account B consume account A's pin for the same workspace, skipping B's own first-encounter validation and breaking the per-user WK/wipe model. And the in-memory fallback stays plaintext-only: modePin.ts today falls back in-session only for a confirmed-plaintext pin and throws on durable-pin failure, precisely because E2EE needs a wipe-surviving authority. Keep that — if the durable IDB pin can't be written, an E2EE create/unlock must fail (don't pin in memory), since a reload or lock would lose the authority and strand the workspace in first-encounter defer. A memory-only E2EE pin is never acceptable.
  • Expected hash → SW-readable metadata mirror, NOT a URL param. The replay/rollback defense (§5.1) verifies decrypted bytes against the block's hash, but that hash is in the block's encrypted properties_json — which the SW can't read. An earlier draft put it in the URL (?h=<sha256>); that is wrong — it leaks the plaintext SHA-256 to the origin on any SW-interception miss (and misses are real: public/sw.js deliberately omits clients.claim(), so a fresh install controls only the next load, and registration can be unavailable), which is exactly the known-file content oracle §10 rejects and §4 promises against. Instead, the app writes per-asset metadata to the same SW-readable IDB mirror as the mode pin (§7.1.1, extended to {asset_block_id → expected hash}) when it materializes the block; the SW reads the expected hash from there to (a) build the hash-keyed cache key (§8) and (b) verify decrypted bytes before caching/serving. The URL stays clean (${BASE_URL}asset/<ws>/<id>) — no plaintext hash ever on the wire. A missing mirror entry for an E2EE asset fails closed (placeholder), never serves unverified bytes — closing the same-asset replay window without the leak. The mirror holds a plaintext-derived hash, so it is a Lock & Wipe participant alongside the cache (§7.2).
  • Active user + token — already covered by the §7.1 per-clientId handshake.
Cache the SW's IDB reads on the hot path — keyed by (user_id, workspace_id). Both the mode-pin store and the WK store (km-e2ee-keys) are now SW-readable IDB, so a naïve handler does two async IDB opens per <img> fetch. Keep a small in-SW in-memory cache of { (user_id, workspace_id) → mode pin, WK handle, map of asset_block_id → expected hash }. The expected hash is per-asset, not per-workspace — a workspace has many attachments, so a single expected hash on the workspace entry would verify (or key the cache for) asset B with asset A's hash → false mismatches; hold a per-asset_block_id map inside the workspace entry (or a separate (user_id, asset_block_id) → hash map). The mode pin / WK stay at (user_id, workspace_id). The user_id in the key is load-bearing, not an optimization detail: the keyStore deliberately keys WKs by (user_id, workspace_id) and wipes per-user, so a workspace-only hot cache would let account A's populated WK/pin serve account B's request for the same encrypted workspace — B could decrypt while B is locked or has never loaded its own WK, bypassing the per-user isolation. Resolve the requesting client's user_id (§7.1) and key both the cache and its invalidation by (user_id, workspace_id). Populated lazily and invalidated on the wipe and auth/token messages (the same signals that drive §7.1/§7.2) — and on the per-workspace revoke/leave purge. The cache is per-SW-lifetime, so a respawn re-reads IDB once; correctness still comes from IDB + the invalidation signals, the cache only avoids per-request opens.
Whole-blob AEAD ≠ native range/streaming for E2EE. The encb:v1: envelope is a single AES-GCM seal over the whole byte array, and a GCM ciphertext byte range cannot be decrypted independently. So an E2EE <video> (or any Range request) cannot stream a slice: the SW would have to fetch+decrypt the entire object and synthesize a 206 from the plaintext — fine for images and moderate files, but it defeats true streaming for large video. Two honest paths, both vNext: the SW fully materializes + serves synthesized ranges (simple, no streaming), or large media uses a chunked envelope (per-chunk nonce/tag, range maps to chunk boundaries) designed before relying on streaming <video>. v1 images don't hit this; the doc just must not promise native ranges under E2EE. (Plaintext workspaces keep native ranges — the object is the raw bytes.)
Cache hits must be authorized, and the cache key must be user-scoped. Decrypted bytes in the Cache API are not themselves access-controlled, so a cache key of /asset/<ws>/<id> alone would let a later request return previously-decrypted bytes without re-checking the active session or key — bypassing Storage RLS and key availability across an account switch, a workspace revoke/leave, or a purge that missed an entry. Three defenses, all required: (1) namespace the cache by user_id (a per-user bucket assets:<user_id>); (2) resolve the active user per requesting client, not globally — a single SW serves every tab, and a browser profile can hold two accounts at once (the keyStore is shared across accounts), so the SW keys its session map by event.clientId → that tab's user, never a single global "active user" slot that the last onAuthStateChange wins; and (3) validate before every hit against a push-driven locked state — lock is signalled to the SW (§7.2), and the SW treats "no fresh unlock signal for this workspace" as locked, evicting rather than serving. Pulling "is the WK present in IndexedDB?" is not sufficient on its own: that read can be stale between a lock and the SW's next read, and it doesn't gate plaintext-workspace cached bytes at all. Together these make the §7.2 purge a backstop, not the only line of defense — but note the cross-account case is the subtle one (§16): the cache namespace isolates reads, while the per-client session keying is what isolates the fetch token.
Membership revocation needs the same backstop as lock — especially for plaintext. The locked-state check above is the E2EE arm; the symmetric hazard is a user removed from a workspace server-side (by another admin), with no local signal. For a plaintext workspace — where §10 says RLS is the access boundary — a stale cache hit in assets:<user_id> would keep serving bytes after removal if the revoke-purge is delayed or misses an entry, because a cache hit never goes back through Storage RLS. So a cache hit must never be treated as an independent authority: (a) purge the workspace's cache prefix on leave/revoke (primary, but it can race / miss); and (b) bound cache authority — give cached entries a short freshness TTL after which the SW re-validates access (cheapest: re-issue the RLS-gated Storage GET, which 403s for a removed member, before serving a stale-past-TTL hit). The membership signal is not locally authoritative, so the TTL-bounded revalidation — not just the purge — is what closes the plaintext-removal window. (E2EE leans on the same shape, with key-absence as the faster signal.)

The thing that actually constrains the SW approach is not decryption — it's two coordination problems, and they are the real work of this design:

7.1 Coordination problem A — fetching private bytes needs the session

Storage objects are RLS-protected; reads need a bearer token, which lives in the app's supabase-js session in localStorage — which the SW cannot read. Resolution: each app client postMessages { userId, accessToken } to the SW on onAuthStateChange / token refresh, the SW records it keyed by the sender's client.id (not a single global slot — see the cross-account defense above), and on a fetch the SW looks up the session for that request's event.clientId and issues GET /storage/v1/object/attachments/<ws>/<id> with Authorization: Bearer. This also eliminates per-object signed URLs for the in-app case (signed URLs are only needed for the logged-out public case, §13). The token expires hourly; the app re-posts on refresh.

This auth scheme covers controlled-tab subresource fetches only — not raw navigations. A <img>/<video> subresource carries the originating tab's event.clientId, so the per-user token lookup works. A navigation to an asset URL (new-tab open, or a download flow that navigates) may have no usable clientId (a fresh top-level navigation has only a not-yet-messaged resultingClientId), so there's no per-user token to attribute — and falling back to a global token would break the multi-account isolation this whole section preserves. So direct asset navigations are not a supported auth path: "download" / "open original" is an app-mediated action (the app resolves the bytes through the same path and triggers the download from an in-memory blob), never a raw navigation to ${BASE_URL}asset/…. The route still sits ahead of the navigation branch (§11) so such a request can't return the SPA shell — but for a clientId-less navigation it returns a placeholder/redirect into the app, not bytes and not the shell. (If first-class direct navigations are ever wanted, they need an opaque app-minted capability token in the URL — deferred; the public-share path §13 is the related case.)

Persistence across SW respawn is deliberately minimal and wipe-gated: the token store, if persisted at all, lives under the same per-user PENDING_WIPE marker the DB wipe uses (§7.2), so a boot-time consumer drops it deterministically. A bearer token must never outlive a lock just because a runtime postMessage was missed; gating it on the marker (rather than only on a best-effort message) is what guarantees that.

7.2 Coordination problem B — Lock & Wipe must reach the SW

This is the load-bearing risk — and a fire-and-forget purge does NOT close it. If the SW caches decrypted bytes in the Cache API, those are plaintext-at-rest in a context that src/sync/keys/flows/lockAndWipe.ts does not touch today. Critically, the real lock sequence does not delete the local store inline: lockAndWipe drops the WKs (clearForUser) and arms a per-user PENDING_WIPE marker; the caller (src/shortcuts/defaultShortcuts.ts) then immediately calls window.location.reload(), and the OPFS DB file is deleted on the next boot by consumePendingWipe (before the DB is opened) — precisely because an inline delete isn't safe. A postMessage(SW, 'purge') issued just before that synchronous reload() is fire-and-forget: it races the navigation, the SW need not dispatch it before the page is gone, and nothing keeps an idle SW alive to finish caches.delete. So the obvious mechanism leaves decrypted bytes (and the bearer token) alive past a lock.
Fix: gate the SW purge on the existing wipe marker, not on a message. Mirror the OPFS pattern that already works. Arm the asset-cache + token purge under the same per-user PENDING_WIPE marker, and have consumePendingWipe (which already runs at next boot, before the DB opens) perform caches.delete('assets:<userId>') and clear the SW token store. The runtime postMessage, if kept, is only a fast path; the marker is the guarantee. This must be wired in from day one and covered by the same flavor of tests that pin lockAndWipe.

And it must block marker-clear — it is NOT best-effort like the compiled-extension clear. The compiled-extension cache is non-secret, so its clear can fail-and-forget; the asset cache is the primary plaintext store for attachments (and the token store holds a live bearer token). So consumePendingWipe must clear PENDING_WIPE only after both purges succeed — if caches.delete or the token-store clear throws/partial-fails, keep the marker armed (retry next boot) or fail the boot, never proceed with the DB open. Otherwise a transient Cache/IDB failure on the wipe boot leaves decrypted bytes or a token alive while the marker that would retry the wipe is gone.

Storing decrypted bytes in the Cache API is consistent with the accepted posture (e2ee §15 #7 already accepts plaintext-derived data on local disk) — but only because, and only as long as, it is wiped on lock like everything else.

Working choice (the boundary itself is the one open fork — §7.3, §18). Decrypt in the service worker behind the self-describing encb:v1: envelope; the renderer stays mode-agnostic, and the Cache API doubles as the byte cache. If kept, the mandatory companions are the app→SW token handshake (§7.1) and the marker-gated lock→SW wipe (§7.2). §7.3 lays out the app-thread alternative honestly; everything below in §8–§10 is boundary-agnostic except where it names the SW.

7.3 The open alternative: app-thread decryption

The alternative is app-thread (or dedicated-Worker) decryption: the renderer fetches, decrypts via getCek directly (zero duplication), caches plaintext, and sets an object URL. §7–§10 are written for the SW path, but this fork is deliberately left open (§18) — and the tally is more even than an earlier draft implied. What the app-thread path avoids, because it runs in the context that owns app state:

  • No mode-pin IDB store / no per-asset hash mirror. The app reads the mode pin synchronously and already holds the decrypted block (so the hash is just block.properties.hash) — the §7.1.1 mirror machinery exists only because the SW can't reach app state.
  • Plaintext bytes in OPFS (not SQLite, not Cache API). The app-thread path caches decrypted bytes as plain OPFS files (attachments/<user>/<ws>/<id>) — the store the bytes always wanted (§6/§17 keep large blobs out of SQLite; an earlier draft wrongly floated DB-blob storage purely for a "free wipe," which both bloats the hot query DB and contradicts that principle). Wipe is not a differentiator: under the coarse destroyAllLocalData() lock, the whole OPFS tree is cleared anyway, so the attachments dir goes with it for free; even under a surgical wipe it's one removeEntry in consumePendingWipe. The real app-thread win here is simplicity of resolution: the app owns the OPFS handle and the session, so fetch→decrypt→write-OPFS→object-URL is one local flow with no SW cache/token/mirror lifecycle. (The SW path, by contrast, uses the Cache API as its SW-native byte store — a separate lock-wipe participant.)
  • No token handshake / no per-clientId session — the app already has the Supabase session.

What it costs: a mode-aware renderer and manual object-URL lifecycle (two local, well-understood concerns). What it does not cost, contrary to the earlier framing: native ranges — whole-blob AEAD defeats Range/streaming for large E2EE media regardless of boundary (both paths must fully materialize the plaintext; the fix is chunked encryption, §7 range note), so "native streaming" is not a real SW advantage. The SW path's genuine win is a single mode-agnostic <img src> for plaintext and E2EE alike; the app-thread path's win is not making the service worker an E2EE-lifecycle participant. Decide on that, not on the illusory range benefit.

8. Local cache & Lock-and-Wipe participation

  • Backing store: with SW decryption, the Cache API is the natural byte cache (SW-served Responses, native streaming). If we ever move decryption to the app thread, OPFS becomes the cache instead — the choice follows the decrypt location. The cache is keyed by user_id, then workspace_id, then the content hash (assets:<user_id> → <ws>/<id>/<hash>): user-scoping is the access-isolation boundary (§7), workspace-scoping makes leave / lock evict only the affected bytes, and including the hash is what makes a re-upload self-invalidating — an edited asset gets a new hash (§5.1) in the SW metadata mirror (§7.1.1), so the new cache key misses and triggers a fresh fetch+decrypt+verify rather than serving the stale plaintext (the old entry ages out by LRU). The cache hit must match the mirror's current expected hash; never serve <ws>/<id> bytes for a different hash.
  • Eviction: soft LRU cap (≈500 MB default). An evicted entry simply round-trips to the object store on next view.
  • Lock & Wipe: the cache, the SW token store, and the §9 byte-upload queue store are all purged via the PENDING_WIPE marker consumed at next boot (§7.2/§9), not a runtime message. On per-workspace revoke/leave: purge that workspace's prefix.
  • Optimistic local-first write: on paste, the encoded bytes are written to the durable upload queue's own store (§9) and the display cache is warmed. The image renders immediately from local bytes; the upload drains in the background. Whether the bytes have uploaded yet is a per-device fact, read from the local queue — see the next point. Offline it simply stays pending and still renders on this device.
Upload status is per-device; do not put it in synced block properties. An earlier draft stored status: 'uploading' | 'synced' on the asset block. But block properties serialize to the single synced (and, for E2EE, encrypted) properties_json column — so a flip to 'synced' would replicate to every device and clobber a peer for which the bytes are not local. "Have the bytes uploaded?" and "are the bytes cached here?" are both per-device questions answered by the local upload queue (§9) / local cache, keyed by asset_block_id — never by a synced flag. The synced block's mere existence is the only cross-device signal needed.
The LRU display cache must not be the only home for un-uploaded bytes. The cache is evictable (and capped); if a paste happens offline, or the cap is hit before the queue drains, an evicted entry would leave a synced asset block stuck in uploading forever with no bytes to send. So pending bytes are owned by the upload queue's durable store (§9), which is not subject to the display cache's LRU, until the upload succeeds. Only then are they purely a display-cache concern.

9. Offline: the byte upload queue

Block metadata rides PowerSync's ps_crud upload queue. Bytes go through the guarded upload function (§10.1) on a different path with no such queue, so they need their own durable one. Crucially the queue owns the encoded bytes, not just a pointer: an IndexedDB/OPFS record of { user_id, asset_block_id, workspace_id, status, idempotencyToken, encodedBytes } (the encb:v1: payload for E2EE, raw bytes otherwise), held in a store that is not the evictable display cache, so a pending upload can never lose its source bytes to LRU eviction or an offline window (the §8 hazard). Two record fields are load-bearing and were under-specified earlier: (a) the idempotencyToken (§10.1) must be persisted in the record and threaded through BlobStore.put, so a retry after a reload re-presents the original token — otherwise the already-written object looks like a no-token/different overwrite and becomes a permanent failure; (b) the user_id namespaces the record (and every cancel / drain / purge key), because the store is shared across accounts in the profile but Lock & Wipe and draining are per-user — without it, account A's lock can't tell A's pending bytes from B's (purging B's, or surviving A's), and a drain could retry A's entry under B's session. The status field has a lifecycle that closes the metadata-commit / byte-stage ordering gap (§11): staged (durable but not drainable) → pending (drainable) → uploading → deleted-on-confirm. Capture writes the record as staged before the metadata repo.tx, so a committed asset block always has its source bytes already on disk; the post-commit step flips it to pending and arms the drain. The drain acts only on pending+ records, so the "block still exists?" pre-PUT drop applies only after the commit (a post-commit deletion / undo) and never to the legitimate pre-commit window. A flip that never lands (crash or throw between commit and flip) is recovered by a boot-time reconciler that promotes any staged record whose asset block is present and committed, and reaps any staged record whose block never committed. Entries drain (to the §10.1 function) when connectivity returns, retrying with backoff; the durable copy is deleted only on a confirmed upload, after which the bytes are purely a display-cache concern. This is new infra the pre-E2EE proposal under-specified; E2EE doesn't change it, but the design surfaces it clearly. The queue's local status is also the per-device source of truth for the "bytes pending here?" UI (§8) — there is no synced status flag. Cross-device arrival is naturally eventually-consistent: a peer that has the block but not yet the bytes shows the "metadata present, bytes pending" state until the upload from the origin device completes.

Undo / delete must compensate the byte lane. §11 puts the asset-block create + the !((id)) insert in one repo.tx so they share an undo bucket — but that only makes the DB half atomic. The bytes (queue entry + any already-PUT object) are not transactional. So: (1) the drain step re-checks the block still exists (and is not deleted) before PUT, and drops the entry otherwise; (2) undo of an asset-block insert cancels the queue entry (keyed by (user_id, asset_block_id)) and, if the object already uploaded, enqueues a delete. Without this, undo leaves an orphaned Storage object and a queue entry that resurrects bytes for a block that no longer exists. (This is the undo case the §16 orphan sweep does not cover.)
The queue store is a Lock & Wipe participant too — it holds un-uploaded bytes. This store sits outside both the display cache and PowerSync's ps_crud (so the existing flushUploadQueue in lockAndWipe.ts, which drains only ps_crud, does not touch it). Left alone it would (a) keep raw bytes for plaintext workspaces on disk past a wipe, and (b) lose bytes for a synced-metadata block, stranding it as uploading forever. So the queue joins the lock flow: at lock time a best-effort drain (upload + confirm) like flushUploadQueue does for ps_crud, then the store is purged under the same per-user PENDING_WIPE marker consumed at next boot (§7.2). Bytes that couldn't upload before the wipe are lost — consistent with the destructive-by-design lock — and that loss is surfaced, not silent.

10. Storage layout & RLS

One private bucket, attachments. Object path <workspace_id>/<asset_block_id> — block-id-addressed, NOT content-hash-addressed. For an E2EE workspace the object is the encb:v1: ciphertext; for a plaintext workspace it is the raw bytes (identity case, §5).

Why not content-addressed paths (and what we keep instead). An earlier (pre-E2EE) sketch keyed everything off sha256(bytes) — path, cache, dedup — which is clean because content-addressing makes uploads idempotent and dedup free. E2EE removes the hash from the two server-visible identifiers, for two reasons:
  • Content oracle. A path <ws>/<sha256-of-plaintext> lets the untrusted server confirm "does this workspace contain this known file?" by hashing a candidate and probing for the object — a confidentiality break for known files.
  • Share-RLS join. The §12 share path keys off the block id (object → block → effective_share_ids); a hash path breaks that join.
Hashing itself is not lost — it stays everywhere it's client-local: the cache key (<ws>/<id>/<hash>, §8), within-subtree dedup (§11), and the integrity/replay check (§5.1); and the hash lives in the encrypted properties_json so the server never sees it. Only the server-visible path and the block id drop it. The direct cost is that idempotent-by-content uploads and free dedup are gone — which is the root of the dedup/duplicate-object and "a fresh block must re-upload" wrinkles (§11).

The alternative that fixes that: a keyed hash. Path <ws>/HMAC(K_id, sha256-of-plaintext) restores content-addressing — deterministic within a workspace (dedup + idempotent PUT return), unguessable to the server (no oracle), and distinct across workspaces (no correlation). (K_id is a separate HMAC-capable subkey derived from the WK material via HKDF at import time — not the stored WK handle, which is a non-extractable AES-GCM key WebCrypto won't let you sign with; deriving the subkey while the raw bytes are briefly available at paste is the only window.) It is genuinely the better-scaling design: it eliminates the dedup races and duplicate-object / re-upload wrinkles entirely, and those races get more likely as device count and concurrency grow, not less. The real cost is a design tradeoff, not a size question: the path no longer carries the block id, so the share-RLS join needs a separate cleartext object_key → block_id mapping (which leaks the same object↔block association the block-id path already does — no worse on that axis). So it's a true fork — block-id paths buy share-join simplicity; keyed-hash paths buy idempotency + dedup that scale. v1 may ship block-id paths to move faster, but that is a deliberate design call to revisit on merits, explicitly not settled by current fleet size — adopting keyed-hash later is a storage re-key + mapping backfill, and that migration is the only place the small fleet legitimately makes it cheap. Recorded so it isn't re-discovered.

Storage reads are RLS-gated by workspace membership on the path's first segment. Storage writes are NOT direct — they go through the guarded upload path (§10.1); the bucket grants no client insert/update:

create policy "attachments read by workspace member"
on storage.objects for select to authenticated
using (
  bucket_id = 'attachments'
  and private.is_workspace_member(
        (storage.foldername(name))[1],   -- <workspace_id>
        (auth.uid())::text)
);
-- NO insert/update policy for `authenticated` on the attachments bucket:
-- direct client writes are denied so they can't bypass the §10.1 byte guard.
-- The service-role Edge Function (which receives + inspects the body) is the
-- ONLY writer. Not a signed upload URL: uploadToSignedUrl writes direct to
-- Storage without the function seeing bytes, defeating the encb:v1: check (§10.1).

-- DELETE is allowed directly for workspace writers — delete removes an object,
-- there is no body to inspect, so the §10.1 byte guard doesn't apply and a
-- writer-only RLS policy is sufficient (and is REQUIRED: without it BlobStore.delete()
-- is default-denied, so §9 undo / §16 GC can't reap already-uploaded objects).
create policy "attachments delete by workspace writer"
on storage.objects for delete to authenticated
using (
  bucket_id = 'attachments'
  and private.is_workspace_writer(
        (storage.foldername(name))[1],   -- <workspace_id>
        (auth.uid())::text)
);

So the mutation split is: writes go through the body-inspecting Edge Function (§10.1); deletes are a direct, writer-only RLS delete (no byte to inspect); reads are direct, member-gated RLS. BlobStore.delete() (used by §9 undo and §16 GC) uses that delete policy — without it, already-uploaded objects would survive undo/delete as orphans.

The private.* grant is not free — verify it, don't assume parity with blocks. private.is_workspace_member / is_workspace_writer exist and are SECURITY DEFINER, but the migrations contain no explicit GRANT USAGE ON SCHEMA private / GRANT EXECUTE to the app roles — the blocks RLS works via defaults and the fact that the policy and the table live in public. A storage.objects policy runs in a different ownership/role context (the table is owned by supabase_storage_admin, in the storage schema), so "the same functions blocks uses" is not automatically callable there. Treat the schema-USAGE + EXECUTE grants (or a thin public wrapper) as a first-class, tested step of the bucket migration — a wrong assumption yields a policy that errors at query time or silently denies all reads.

For an E2EE workspace the blob is already ciphertext, so RLS here is defense-in-depth — it stops a non-member from hoarding ciphertext against a future key compromise, and avoids leaking object existence. For a plaintext workspace it is the actual access boundary.

10.1 Server-side ciphertext enforcement for E2EE bytes

RLS is not the ciphertext guarantee. For blocks, a server trigger rejects a plaintext write to an E2EE workspace (is_enc_v1_envelope shape-check), so "the server only ever sees ciphertext" does not rest solely on the client upload hook. The naïve analog — "storage.objects trigger that checks the bytes" — does not work: Supabase Storage object policies/triggers see object metadata (path, owner, size), not the body bytes, so they cannot inspect the envelope. A check that can read bytes has to sit in code on the write path — and that code only enforces anything if it is the only write path.

Decision: a guarded upload service is the sole writer; direct client writes are denied. Attachments must match the block guarantee — the server enforces ciphertext, not by client convention. So: (1) the attachments bucket grants no insert/update to authenticated (§10); and (2) all uploads go through an upload-mediating Edge Function that receives and inspects the body bytes itself and then writes them with service-role. Per request it:
  1. Authenticates the caller's JWT and authorizes the write — checks private.is_workspace_writer(<workspace_id>, caller) and that the path's workspace/asset is the intended target — before using the bypass-RLS credential. The service-role write must re-impose the authorization that direct RLS would have enforced; otherwise a read-only or recently-removed member who can invoke the function could create/replace <ws>/<asset> bytes that blocks_write/storage policies deny.
  2. Inspects the bytes — for an object under an E2EE-workspace prefix, rejects bodies not beginning with the encb:v1: magic + the minimum nonce(12)+tag(16) length (the binary analog of is_enc_v1_envelope), with the same permanent-failure discipline so a stale client quarantines rather than retries forever. Shape-only (the server has no key), sufficient to keep bare plaintext out.
  3. Enforces write-once, idempotently. Asset objects are immutable; an edit/replacement is a new asset block at a fresh path (§5.1), so a stale queued upload can never overwrite a newer object and strand the asset on a hash mismatch. The guard rejects an overwrite by different content, but must treat a re-PUT of the same content as success, not failure: if the function wrote the object but the response was lost before §9 deleted the durable queue entry, the retry would otherwise turn a completed upload into a permanent queue error. So the client sends an idempotency token, the function stores it as object metadata, and on a PUT to an existing path: same token → 200 (already done); different token → reject. The token is a per-asset random value (a UUID minted at capture, carried by the queue entry so retries present the same one) — not HMAC(WK, …): the WK is a non-extractable AES-GCM CryptoKey (workspaceKey.ts / keyStore.ts) that WebCrypto allows only to encrypt/decrypt, never to sign/HMAC or export, so a keyed digest can't be computed from the stored handle. A random token needs no key and is opaque to the server anyway. (Under immutable ids only the same-token retry is legitimate; a different token to an existing path signals a bug/attack.)
Not a signed-upload token. Minting a one-shot signed upload URL would let the client uploadToSignedUrl the body directly to Storage, which Supabase accepts without the function seeing the bytes — defeating the magic check. The body must flow through the function. The §9 byte-upload queue therefore drains to this function, not to Storage directly. Cost: every upload is one function invocation that streams the body (latency + per-call cost); acceptable for the guarantee, and reads stay direct/RLS-gated (the hot path). Lands in phase 2 (§15) with the bucket + RLS migration.
Share-ready by construction. Keeping asset_block_id in the path means storage RLS can later be widened from "membership" to "membership or a share covers this block" by joining storage.objects → blocks → effective_share_ids when subtree sharing ships (§12). We do not build that branch now (shares aren't shipped) but the path scheme does not foreclose it. This is the cheap-option-value posture, not speculative building.

Per the project's Supabase rule, the bucket + its RLS land in one migration with storage.objects policies enabled; no public table is created without RLS.

11. The renderer & capture surfaces

Packaged as one attachments plugin, mirroring the video-player plugin's facet wiring:

  • MediaBlockRenderer via blockRenderersFacet, with canRender = ({block}) => block.hasType('media') and a priority above the default. Branches on mime: image → the existing MarkdownImage lightbox fed the virtual URL; later, PDF / audio / video / generic file-chip.
  • Markdown img override via markdownExtensionsFacet (alongside defaultMarkdownExtension.ts) for any inline image whose src is a virtual asset URL — resolve and delegate to the lightbox.
  • Capture surfaces: a CodeMirror domEventHandlers extension via codeMirrorExtensionsFacet intercepting paste / drop with clipboardData.files / dataTransfer.files; a block-shell paste/drop arm parallel to today's text/plain handlers in CodeMirrorContentRenderer.tsx / DefaultBlockRenderer.tsx; and a /image command + file-picker action through the existing action/shortcut system.
  • On capture: client-side mime/size guard → hash → subtree-scoped best-effort dedup lookup (see the caveat below) → otherwise create the asset block as a child of the containing block (see §12), stamping the type with repo.addTypeInTx(tx, id, 'media', …) (the in-tx variant — the standalone block.addType wouldn't compose into the single tx; and the media TypeContribution must be registered via typesFacet first, §3, or this throws). The asset-block create + the !((id)) insert go in one repo.tx({scope: ChangeScope.BlockDefault}) so they share an undo bucket. The durable byte record is staged (non-drainable) BEFORE that tx, and the drain is armed only AFTER it resolves — not inside it (see the box below).
Stage the byte record durably before the metadata tx; gate its drain on the commit. repo.tx only makes the SQLite/PowerSync writes atomic — an IDB/OPFS byte-queue write is an external side effect that neither joins that tx nor rolls back with it. Two failure windows have to close at once. (a) If the byte write happened after the commit and then failed (quota, private-mode, a store error), the metadata is already committed and syncs to peers with no durable source bytes, no per-device pending status, and nothing for the §9 compensator to cancel — a permanently un-uploadable asset on every peer. (b) If the enqueue instead kicked off a drain before the commit is visible, the §9 "block still exists?" pre-PUT re-check would drop the bytes as a missing block. The resolution closes both: write the durable byte record first in a non-drainable staged state (§9), then run repo.tx, then flip staged → pending and arm the drain. Because the bytes are staged before the commit, a committed asset block always has its source bytes already on disk — window (a) is gone (a failed stage aborts capture before any metadata commits, attaching nothing); because staged is not drainable, the drain can't race ahead of the commit — window (b) is gone. The post-commit flip can itself fail, but that is recoverable, not lossy: the durable staged record survives, and a boot-time reconciler (§9) promotes any staged record whose asset block did commit. Undo still works via the §9 compensator — undo of the (committed) asset block cancels the queue entry and deletes any already-uploaded object.
Dedup must respect the share-coverage boundary (§12). Naïve "reuse any same-hash asset block in the workspace" breaks the share model: §12.1 grants byte access by walking parent_id, so an embed of an asset whose parent lives under a different page is not covered when you share this page's subtree — the recipient gets a broken image, and the §10 storage-RLS share branch never stamps that asset. So dedup is scoped to assets already under the containing subtree: reuse only an existing asset block whose parent_id chain is within (or an ancestor shared with) the current containing block; otherwise create a fresh asset block as a child here. A fresh block still uploads its own bytes. Object paths are <ws>/<asset_block_id> and storage RLS/share lookup key off that block id, so a new id is a new, currently-absent object — "reuse the hash" only saves re-reading/re-encoding the local source bytes; it must still PUT them (via §10.1) to the new block's path, or other devices / shared viewers 404. (A hash-addressed storage path would avoid the copy, but it's ruled out: under E2EE a plaintext-hash path is a content oracle, and it breaks the block-id share-RLS join — §11 dedup note, §10.) Cross-page reuse that deliberately crosses subtrees is then the same "cross-subtree embed" case §12 already calls out — resolved at share time by copying the referenced-but-outside asset into the shared subtree, not by a single workspace-wide asset block.
Dedup is best-effort, not an invariant. The lookup reads local synced state, so two devices (or two tabs, or an offline-then-reconnect) pasting the same image concurrently both miss, both create an asset block, and — because the object path is <ws>/<asset_block_id> — produce two Storage objects for identical bytes. The server can't break the tie (randomized GCM ⇒ no ciphertext dedup; hash is inside encrypted properties_json). So: state dedup as best-effort and make GC tolerate multiple blocks per hash. The tempting "make the id a deterministic function of (workspace_id, hash) so pastes converge" is unsafe as stated under E2EE — an id derived from the plaintext hash is a content oracle (the server could confirm a known image's presence by id, or correlate the same image across workspaces). If determinism is wanted, derive the id with a workspace HMAC subkey (HMAC(K_id, hash), K_id derived from WK material at import — see §10; the stored AES-GCM WK handle itself can't sign) so it's workspace-local and reveals nothing to the server — otherwise keep random ids + tolerate duplicates.
SW route ordering — the asset route must be FIRST. The existing public/sw.js fetch handler matches in order: (1) navigations (request.mode === 'navigate' or Accept: text/html) → shellNetworkFirst (returns the SPA shell), then (2) isCacheableAsset (an <img> has destination 'image') → assetCacheFirst. The new ${BASE_URL}asset/<ws>/<id> route (base-prefixed, §7) must be matched ahead of BOTH — not just the image-cache branch. An inline <img> would otherwise hit the image-cache branch; a navigation / Accept: text/html request (new-tab open) would otherwise hit the navigation branch and get the SPA shell. Put the /asset/ match before the navigation check at the top of the fetch listener. What it returns differs by request: a subresource fetch with a usable clientId → the (auth'd, decrypted) bytes; a clientId-less navigation → a placeholder / redirect into the app (NOT bytes — it can't be authorized — and NOT the SPA shell). Direct asset navigations and downloads are app-mediated, not raw navigations to the asset URL (§7.1).

12. Interaction with subtree sharing

Subtree sharing (shares-and-api-tokens.html) is not shipped but is on the roadmap. It informs two decisions now and exposes one genuine gap.

12.1 Reinforces: asset block = child of the containing block

Subtree sharing walks parent_id and stamps effective_share_ids. If the asset block is a child of the page that uses it, sharing that page's subtree automatically covers the asset (its effective_share_ids gets stamped, and the share-aware storage RLS branch grants byte access). A flat per-workspace asset container would be invisible to subtree shares. So asset-as-child is not just tidy — it is what makes a shared page carry its images.

12.2 Reinforces: keep asset_block_id in the storage path

A subtree-share recipient is not a workspace member, so granting them byte access requires storage RLS to join through to the block's share coverage (§10). The block-id-in-path keeps that join possible. Design the policy as "membership today, with an explicit seam to add OR effective_share_ids covers this block."

12.3 The gap: E2EE + subtree sharing is unsolved

A subtree-share recipient of an E2EE workspace needs a key to read anything, including asset bytes — but the model has exactly one WK per whole workspace. There is no per-subtree key. Granting subtree access to encrypted content requires the per-subtree key hierarchy the e2ee doc explicitly defers (e2ee §13). So for the foreseeable future subtree sharing is a plaintext-workspace feature, and media inherits that boundary. This is consistent with the existing tradeoff stack (e2ee §15 #2; §8.3 — E2EE sharing is out-of-band whole-workspace WK transfer). Do not special-case it.

13. The public-sharing escape hatch

We still reserve properties.publicUrl on asset blocks and a render-audience flag on the block context ('workspace' member / 'shared' subtree recipient / 'public' logged-out). But the implementation for a public, logged-out viewer differs by mode:

ModePublic-share implementation
Plaintext workspace A public bucket copy (or long-lived signed URL) written to properties.publicUrl; the renderer prefers it when audience is 'public'. The original signed-URL story works here.
E2EE workspace Either export-to-plaintext (decrypt + re-upload to a public bucket — the asset deliberately leaves E2EE, an explicit, consensual exit, never silent) or key-in-fragment (app/share#<per-asset-key>; the client decrypts from the URL fragment, which the browser never sends to the server). "Anyone with the link" is exactly key-in-link semantics.
The fragment key MUST be a fresh per-asset key, NEVER the workspace key. The §5.1 crypto seals bytes with the WK directly — there is no per-asset key in the v1 design. So a naïve "put the decrypt key in the fragment" would put the WK in a URL, exposing the entire workspace, not one image. Key-in-fragment therefore requires its own step: generate a one-off key, re-encrypt the bytes under it, upload that to a public bucket, and put only that per-asset key in the fragment. The WK never leaves the device (preserving the e2ee §5 invariant).
Document the fragment's leak surface and irrevocability. "Not sent in the HTTP request" is necessary but not sufficient: a fragment persists in history, is readable as location.hash by any script on the share page, and can ride a Referer or a third-party resource off-page. A public share page that loads a key into the DOM is a far softer target than the authenticated app (which is CSP-hardened). And it is irrevocable — once the link leaks, the asset leaks, permanently, short of re-encrypting under a new key. Both escape hatches are vNext; neither ships without these stated as accepted, explicit tradeoffs.

Reserving the property + the audience flag costs almost nothing now and avoids a content migration later. The renderer's URL resolver branches on both workspace mode and render audience.

14. Vendor portability (the BlobStore seam)

All object-store access goes behind a ~50-line interface:

interface BlobStore {
  // No `hash` parameter: this seam drains to the server (the §10.1 Edge Function), and the
  // plaintext sha256 must never reach the origin — it is the §10 content oracle. The hash lives
  // ONLY in client-local / SW-readable paths (cache key §8, replay check §5.1, dedup §11),
  // sourced from the encrypted properties_json + the SW metadata mirror (§7.1.1), never the wire.
  // idempotencyToken (§9/§10.1) IS passed: it is persisted in the queue record so a post-reload
  // retry re-presents it; same token on an existing object → success, not failure.
  put(workspaceId, assetBlockId, idempotencyToken, bytes): Promise<void>
  // The virtual URL is CLEAN — ${BASE_URL}asset//, no hash on the wire.
  // The expected hash (cache key §8 + replay check §5.1) comes from the SW-readable
  // metadata mirror (§7.1.1), never a URL param — a ?h= would leak the plaintext
  // sha256 to the origin on a SW-interception miss (the §10 content oracle).
  url(workspaceId, assetBlockId): string
  delete(workspaceId, assetBlockId): Promise<void>
}

v1 backs it with Supabase Storage. E2EE actually increases portability: because the server is a dumb encrypted-blob store and never mediates bytes, swapping to R2 / S3 / B2 later is a bytes-copy plus re-pointing one module — no auth or schema rework. We do not pre-migrate; we keep the seam clean and pull the trigger if egress cost or another driver appears. (A Worker-fronted R2 from day one is defensible only if a Worker is being introduced for other reasons; otherwise it is permanent operational surface for no present benefit.)

16. Open questions

  • Cache backing if SW decryption is adopted: Cache API only, or Cache API + OPFS for an explicit LRU index? Cache API alone is simpler; OPFS gives finer eviction control.
  • Multi-account profile (decided shape, needs care): the SW session map is keyed by event.clientId → that tab's user, and the cache by user_id (§7), so a concurrent second account neither reads the wrong cache nor borrows the wrong bearer token. The residual work is making this correct under switch and under the per-user wipe — mirror the care keyStore.clearForUser already takes (it is per-user precisely so account A's wipe doesn't disturb B).
  • Max inline size + large-media story: a v1 cap (e.g. 10 MB) with a clear error, deferring resumable/tus upload to vNext. Where exactly is the cap enforced?
  • Asset block deletion → byte GC: delete the asset block ⇒ delete the object (storage cascade or an explicit step). Embeds that dangle render the existing "unresolved embed" state. The undo case is handled by the §9 byte-lane compensator; the broader orphan sweep is a vNext follow-up.
  • Cache backing: Cache API only, or Cache API + an IndexedDB LRU index for finer eviction? Cache API alone is simpler; the durable upload-queue store (§9) is separate from either.

17. Tradeoffs we're accepting

ChoiceCostWhy / exit
SW is an E2EE-lifecycle participant A per-client token handshake; decrypted bytes + token in a new (Cache API / SW) location. Buys a mode-agnostic renderer and native media semantics. Wipe is marker-gated at next boot (§7.2), not a racy runtime message; mandatory and tested (incl. a "plaintext survives lock" regression). Consistent with e2ee §15 #7.
Storage byte-length is visible to the server Exact encrypted-object size (≈ plaintext + 28 B) + existence/count — a stronger known-file fingerprint than the accepted block-metadata leak (§4). New accepted leak, outside the e2ee §15 set. Size-bucket padding if the threat model later cares.
Byte AAD doesn't bind freshness An untrusted server can replay an older ciphertext for the same asset (no PowerSync monotonicity on the byte lane). Closed by verifying decrypted bytes against the block's synced hash on read (§5.1) — not left as a residual.
Server byte-ciphertext enforcement is a deliberate add A body-inspecting mediating Edge Function as the sole write path (a storage.objects trigger can't see body bytes, §10.1). Decided: ship it (§10.1), so the byte ciphertext guarantee matches the block trigger rather than resting on client correctness.
Bytes off the sync path A second durable upload queue; eventual consistency between metadata and bytes; undo needs a byte-lane compensator (§9). Keeps large blobs out of SQLite/PowerSync entirely; the queue is small and self-draining.
Dedup is best-effort Concurrent same-hash pastes can produce duplicate objects (§11). GC tolerates multiple blocks per hash; WK-keyed deterministic ids are the only safe convergence option (a plaintext-hash id would be a content oracle).
No server-side media features for E2EE No server thumbnails / OCR / transcode for encrypted workspaces. Inherited from e2ee §15 #2, not new. Client-side transforms remain possible.
E2EE subtree-share of media unsupported Encrypted workspaces can't subtree-share images until the key hierarchy lands. Needs e2ee §13. Plaintext workspaces are unaffected; don't special-case.
Supabase Storage in v1 Egress cost at scale. Behind BlobStore; E2EE makes the backend a dumb blob store, so the swap is a bytes-copy + one module.

18. Design notes — where the complexity lives, and the seams that contain it

This design went through several adversarial review passes; the findings clustered in a few places, and the pattern is worth recording so the implementation leans on the seams that keep it simple.

  • Reuse the proven block lane; don't re-derive a parallel one. "An attachment is a block" (§3) is load-bearing: metadata sync, encryption (transform.ts), undo, and permissions all come for free by riding the existing block path. The fragile parts are exactly where the byte lane diverges from how blocks already work — blocks have ps_crud (→ we need a §9 queue), a server ciphertext trigger (→ a §10.1 guard), and PowerSync monotonicity (→ a §5.1 hash check). Each divergence had to re-solve a problem blocks already solved. Prefer mirroring the block mechanism over inventing a byte-specific one wherever the choice exists.
  • Funnel URL + mode + hash + cache-key through one asset-resolver seam. Many review findings were not "the design is wrong" but "a fact didn't propagate" — the base-path prefix, the expected-hash source (SW mirror), the mode decision, and the cache key are all facets of one operation ("given an asset block + render context, produce displayable bytes"). Implement that as a single named function so those parameters can't drift apart; scattering them across call sites is what manufactured the drift.
  • Content-addressing is confined to client-local use under E2EE (§10): cache, dedup, and integrity keep the hash; the server-visible storage path and block id do not (oracle + share-join). The dedup/duplicate-object friction is the price of block-id paths; a keyed hash (HMAC(K_id, …) under a WK-derived subkey, §10) is the better-scaling alternative (idempotent uploads, no dedup races) at the cost of an object→block mapping for shares. It's a real design fork to decide on merits — not on current fleet size; fleet size bears only on how cheap the migration to it is (§10).
  • Open: the decryption boundary. The single biggest source of genuine (not cosmetic) complexity is putting decryption in the service worker — a context that can't read app state (→ IDB mode-pin + per-asset hash mirror, since a ?h= URL would leak the plaintext hash), is shared across tabs/accounts and respawned arbitrarily (→ per-clientId sessions, readiness gating), and caches plaintext (→ a second store in the lock-wipe lifecycle). The app-thread-decrypt alternative (§7.3) trades all of that for a mode-aware renderer + manual object-URL lifecycle. This choice is deliberately left open pending a focused re-evaluation; §7–§10 are written for the SW path but most of the data model is boundary-agnostic.
