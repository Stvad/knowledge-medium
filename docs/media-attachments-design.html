Media attachments

Design for image (and, by extension, arbitrary file) attachments in a system that now has per-workspace opt-in end-to-end encryption. 2026-06-21.

TL;DR. An attachment is a block carrying the registered media type (in the types array, via typesProp), embedded into prose with the existing !((id)) mechanism — no new reference syntax, and metadata rides the existing encrypted block-content columns for free. The bytes live in an object store (Supabase Storage v1), behind a BlobStore interface. The reshaping constraint is E2EE: for an encrypted workspace the server holds only ciphertext and never the key, so bytes are sealed client-side with the workspace key and access becomes entirely client-side. We do not write two implementations — there is one byte transform parameterized by mode, where plaintext is the identity case, exactly as src/sync/transform.ts already does for text. Decryption happens in the service worker behind a self-describing binary envelope, so the renderer stays mode-agnostic (<img src="/asset/<ws>/<id>"> "just works" for both modes). The load-bearing risk is making the SW a participant in the E2EE lock/wipe lifecycle.

1. What we're adding, and scope

The app cannot currently insert images. We want paste / drop / file-pick of an image to render inline, work offline, and sync across a user's devices — and we want the same machinery to extend to PDFs, audio, video, and generic files later without a content migration.

In scope (v1): images, via paste / drop / file picker; inline render with a lightbox; offline-first local caching; cross-device sync; correctness under E2EE workspaces.

Designed-for but not built (vNext): non-image file types (the model is type-dispatched, so a new mime is a new renderer branch, not a schema change); thumbnails / transforms; resumable uploads for large media; public sharing.

Explicitly out of scope: server-side media features for E2EE workspaces (OCR, thumbnails, transcode) — the server has no key; this is the existing E2EE tradeoff (e2ee-design.html §15), inherited, not new. E2EE subtree sharing of media (needs the deferred key hierarchy — see §12).

3. The core model: an attachment is a block

An attachment is not a row in a side table. It is a blocks row carrying the media type and binary-pointer metadata in its properties:

// asset block (illustrative). The type lives in the `types` ARRAY (typesProp),
// NOT a `properties.type` scalar — that's what the type registry, type queries,
// and the property panel read. Set via block.addType('media') / typesProp.
properties: {
  types:       ['media'],       // typesProp — registered media type id
  mime:        'image/png',
  size:        184213,          // plaintext byte length
  hash:        'sha256:…',      // of plaintext bytes; client-side dedup key
  filename:    'screenshot.png',
  width:       1280,            // optional, for layout reservation
  height:      720
  // NOTE: no upload `status` here — that's per-device, tracked in the local
  // upload queue (§9), NOT in synced (encrypted) properties_json. See §8.
}
content: 'image: screenshot.png'   // human-readable summary

The media type is registered with the type registry (a property schema, as video-player does for its own props) so type-filtered queries, the property panel, and renderer discovery all see media blocks — stamping a bare properties.type would make them invisible to every current types-array consumer (getBlockTypes / hasBlockType / block.hasType).

Why a block, not an attachments table:

  • Inspectable / user-accessible — the project's standing principle. An attachment shows up in the outline, the properties panel, queries, and the agent runtime like any other block.
  • Embeds, backlinks, annotations come for free. Inline placement uses the existing block embed !((id)) (src/components/references/BlockEmbed.tsx); captions / alt-text / OCR / transcripts are child blocks; "what references this image?" is the existing backlink machinery.
  • Type-dispatched rendering already exists. A MediaBlockRenderer registers via blockRenderersFacet with a canRender gate on block.hasType('media') — the same registration pattern as VideoPlayerRenderer. New file types are new mime branches, never a schema change.
  • Dedup within a workspace is natural: paste the same screenshot ten times into a page and you get ten !((id)) embeds of one asset block, one byte copy.

4. The constraint that reshapes everything: E2EE

For an E2EE workspace the server holds only ciphertext and never the key. The workspace key (WK) lives as a non-extractable WebCrypto CryptoKey in IndexedDB (km-e2ee-keys, keyed by (user_id, workspace_id) — src/sync/keys/keyStore.ts). The server-side trigger blocks_require_ciphertext_for_e2ee actively rejects any plaintext write to an encrypted workspace's content columns.

The E2EE design doc encrypts exactly three text columns and says nothing about binary data — confirmed by inspection. So attachments are an unsolved problem that must be solved consistently with E2EE, not bolted on beside it. Two pillars of any naïve "URL in markdown" approach are dead on arrival:

Signed-URL-on-render serving displayable bytes — dead for E2EE. A signed Storage URL returns ciphertext; the browser cannot <img src> that. Bytes must be fetched, then decrypted client-side with the WK before display.
Public-bucket / public-share-URL — incompatible with E2EE. You cannot serve a publicly-readable image out of an encrypted workspace via RLS + signed URLs. RLS gates access to ciphertext; it cannot grant decryption. (This is just e2ee §15 #2 — "no server-side features for E2EE workspaces" — applied to bytes.) The escape hatch survives in a different form; see §13.

What the encryption model gives us in return is also worth stating: because metadata rides the encrypted properties_json column, an attachment's mime, filename, and hash are confidential automatically on an E2EE workspace. The block-row routing identifiers (id, workspace_id) are in clear — exactly the columns the sync transform already leaves in clear.

But the Storage object leaks more than the block row does — call it out, don't fold it into the inherited block-metadata leak. The encrypted blob's byte length (≈ plaintext size + 28 B) is visible to the server via the stored object size and the Content-Length of an RLS-gated GET, and the object's existence and per-workspace count are visible. Exact file size is a much stronger fingerprint than note-text length — it can confirm a specific known file (a leaked document, a known image) by size, and combined with the asset_block_id in the path + timing, correlate which block embeds which known-size asset. This is a new leak outside the enumerated e2ee §15 set (tree shape / order / timing / authorship). Document it as an accepted tradeoff (§17), or pad ciphertext to size buckets if the threat model cares.

5. One byte transform; plaintext is the identity case

E2EE is opt-in, so most workspaces are plaintext. The temptation is two code paths. The codebase already shows the better answer: src/sync/transform.ts is one function parameterized by mode, where mode === 'none' is the identity transform.

// the text precedent, today
encodeForWire(row, 'none', getCek)  // → row unchanged
encodeForWire(row, 'e2ee', getCek)  // → seal content/properties_json/references_json

We mirror it for bytes — a new module in the same src/sync/ seam:

encodeBytes(bytes: Uint8Array, mode: SyncMode, getCek: GetCek): Promise<Uint8Array>
decodeBytes(blob:  Uint8Array, mode: SyncMode, getCek: GetCek): Promise<Uint8Array>
//   mode === 'none'  → identity (return input)
//   mode === 'e2ee'  → AES-256-GCM seal/open with the WK
Decision. There is one attachment implementation, not two. Everything expensive — the asset-as-block model, the BlobStore, the cache, the upload queue, dedup, the renderer, Lock-and-Wipe participation — is mode-agnostic and shared. Only the ~15-line encode/decode seam is mode-aware, and plaintext is its degenerate (identity) branch, exactly like encodeForWire.

5.1 Reusing the crypto primitives

The WK CryptoKey is AES-256-GCM and works for crypto.subtle.encrypt over raw bytes, not just the string seal/open in src/sync/crypto/aead.ts. So byte encryption reuses the same getCek(workspaceId) seam the sync transform uses, with two deltas:

  • AAD binding extends the existing pattern. Where content columns bind contentAad(id, workspace_id, column), asset bytes bind (asset_block_id, workspace_id, "asset-bytes", schema_version) so a ciphertext blob cannot be swapped between blocks or workspaces. (Schema version stays in the AAD for the same cheap-hierarchy-upgrade reason §13 of the e2ee doc keeps it there.)
  • A binary envelope, not the base64url text envelope. The enc:v1: envelope is base64url (≈33% bloat) — wrong for bytes. Add a sibling encb:v1: binary framing (magic ‖ nonce ‖ ciphertext‖tag as raw bytes) sharing the nonce/tag layout. Small new module next to aead.ts.
The AAD does not bind freshness — verify the content hash on read. Every AAD term (asset_block_id, workspace_id, the literal, schema version) and the storage path are stable across an asset's whole life. AAD stops cross-block / cross-workspace swaps, but it does not stop a rollback/replay of an older ciphertext for the same asset_block_id: an untrusted server (in scope per e2ee §2 — it may serve whichever bytes it likes) can return a stale blob that decrypts cleanly and the GCM tag verifies. The bytes lane has no updated_at/monotonicity channel the text path gets from PowerSync. The fix is already in hand: the asset block stores hash = sha256(plaintext) in its (encrypted) properties_json; the read path must verify the decrypted bytes against that hash before caching/serving, and reject on mismatch. This also closes the re-upload-version question for free (a new version writes a new hash on the block, which syncs with its own monotonicity).

The encb:v1: magic is a format tag, not an authorization signal. Whether to decrypt is decided by the workspace's mode pin (§7), exactly as the text path decides via the pin rather than the enc:v1: prefix (envelope.ts / e2ee §6 rule 2). An E2EE-pinned workspace requires the magic and treats its absence as tampering; only an unpinned workspace passes bytes through.

6. The byte path, end to end

Bytes do not ride PowerSync. The asset block (metadata) syncs through the normal Layout B path (encrypted content columns, observer-reconciled). The bytes take a parallel path to the object store. The two arrive independently on other devices.

7. Service-worker decryption (the unification)

The cleanest way to keep the renderer mode-agnostic is to decrypt in the service worker. This is viable — the SW global scope has crypto.subtle, IndexedDB, and the Cache API — and it collapses the entire plaintext/E2EE branch into one place the renderer never sees.

  • The renderer emits a stable virtual URL: <img src="/asset/<workspace_id>/<asset_block_id>">. No mode awareness, no createObjectURL lifecycle, native <img> / <video> / range-requests / <a download> all work uniformly.
  • The SW resolves it, auth-first: establish the active user_id (for the requesting client — §7.1) and the workspace's mode pin + WK availability before consulting the cache → serve from cache only on a namespace match and a still-valid session → on miss, fetch the stored blob → decrypt/passthrough per the mode pin (below) → put the displayable bytes in the user-scoped cache → respond.
  • The mode pin decides decrypt-vs-passthrough, NOT the envelope magic. This is a deliberate carry-over of the rule the text path already enforces (src/sync/crypto/envelope.ts / e2ee §6 rule 2: "the prefix is a FORMAT tag, not a decrypt trigger — the durable mode pin decides"). The SW reads the workspace's local mode pin (it has workspace_id from the URL): if the workspace is E2EE-pinned, the SW requires a well-formed encb:v1: envelope and decrypts — a blob that arrives without the magic is treated as tampering and rejected, never served as plaintext. Only an unpinned/plaintext workspace passes bytes through. Deciding from the self-describing magic alone would re-introduce exactly the downgrade vector (hostile server strips the magic → attacker plaintext reaches <img>) that the text wire-boundary was hardened against.
  • Render gates on SW readiness (cold start). An <img src="/asset/…"> can fire before the app has handed the SW a token (§7.1) or before the SW has re-read its state on a cold respawn. The SW must not "fail open": a request with no established active user / no fresh unlock signal respondWiths a deterministic placeholder (or holds briefly on a readiness promise, short timeout → placeholder), never a stale cache hit and never an unauthorized fetch.
Cache hits must be authorized, and the cache key must be user-scoped. Decrypted bytes in the Cache API are not themselves access-controlled, so a cache key of /asset/<ws>/<id> alone would let a later request return previously-decrypted bytes without re-checking the active session or key — bypassing Storage RLS and key availability across an account switch, a workspace revoke/leave, or a purge that missed an entry. Three defenses, all required: (1) namespace the cache by user_id (a per-user bucket assets:<user_id>); (2) resolve the active user per requesting client, not globally — a single SW serves every tab, and a browser profile can hold two accounts at once (the keyStore is shared across accounts), so the SW keys its session map by event.clientId → that tab's user, never a single global "active user" slot that the last onAuthStateChange wins; and (3) validate before every hit against a push-driven locked state — lock is signalled to the SW (§7.2), and the SW treats "no fresh unlock signal for this workspace" as locked, evicting rather than serving. Pulling "is the WK present in IndexedDB?" is not sufficient on its own: that read can be stale between a lock and the SW's next read, and it doesn't gate plaintext-workspace cached bytes at all. Together these make the §7.2 purge a backstop, not the only line of defense — but note the cross-account case is the subtle one (§16): the cache namespace isolates reads, while the per-client session keying is what isolates the fetch token.

The thing that actually constrains the SW approach is not decryption — it's two coordination problems, and they are the real work of this design:

7.1 Coordination problem A — fetching private bytes needs the session

Storage objects are RLS-protected; reads need a bearer token, which lives in the app's supabase-js session in localStorage — which the SW cannot read. Resolution: each app client postMessages { userId, accessToken } to the SW on onAuthStateChange / token refresh, the SW records it keyed by the sender's client.id (not a single global slot — see the cross-account defense above), and on a fetch the SW looks up the session for that request's event.clientId and issues GET /storage/v1/object/attachments/<ws>/<id> with Authorization: Bearer. This also eliminates per-object signed URLs for the in-app case (signed URLs are only needed for the logged-out public case, §13). The token expires hourly; the app re-posts on refresh.

Persistence across SW respawn is deliberately minimal and wipe-gated: the token store, if persisted at all, lives under the same per-user PENDING_WIPE marker the DB wipe uses (§7.2), so a boot-time consumer drops it deterministically. A bearer token must never outlive a lock just because a runtime postMessage was missed; gating it on the marker (rather than only on a best-effort message) is what guarantees that.

7.2 Coordination problem B — Lock & Wipe must reach the SW

This is the load-bearing risk — and a fire-and-forget purge does NOT close it. If the SW caches decrypted bytes in the Cache API, those are plaintext-at-rest in a context that src/sync/keys/flows/lockAndWipe.ts does not touch today. Critically, the real lock sequence does not delete the local store inline: lockAndWipe drops the WKs (clearForUser) and arms a per-user PENDING_WIPE marker; the caller (src/shortcuts/defaultShortcuts.ts) then immediately calls window.location.reload(), and the OPFS DB file is deleted on the next boot by consumePendingWipe (before the DB is opened) — precisely because an inline delete isn't safe. A postMessage(SW, 'purge') issued just before that synchronous reload() is fire-and-forget: it races the navigation, the SW need not dispatch it before the page is gone, and nothing keeps an idle SW alive to finish caches.delete. So the obvious mechanism leaves decrypted bytes (and the bearer token) alive past a lock.
Fix: gate the SW purge on the existing wipe marker, not on a message. Mirror the OPFS pattern that already works. Arm the asset-cache + token purge under the same per-user PENDING_WIPE marker, and have consumePendingWipe (which already runs at next boot, before the DB opens) perform caches.delete('assets:<userId>') and clear the SW token store as injected best-effort steps — alongside the compiled-extension cache clear it already does. The runtime postMessage, if kept, is only a fast path; the marker is the guarantee. This must be wired in from day one and covered by the same flavor of tests that pin lockAndWipe.

Storing decrypted bytes in the Cache API is consistent with the accepted posture (e2ee §15 #7 already accepts plaintext-derived data on local disk) — but only because, and only as long as, it is wiped on lock like everything else.

Decision. Decrypt in the service worker behind the self-describing encb:v1: envelope; the renderer stays mode-agnostic. The Cache API doubles as the byte cache. Mandatory companions: the app→SW token handshake (§7.1) and the lock→SW wipe handshake (§7.2).

7.3 The alternative, and why it loses

The alternative is app-thread (or Worker) decryption: the renderer fetches, decrypts via getCek directly (zero duplication), caches plaintext in OPFS (already in lockAndWipe), and sets an object URL. It avoids both handshakes — but it makes the renderer mode-aware, forces manual object-URL lifecycle, and means native <img> / <video> / range / download each get re-implemented per consumer. It only wins if we decide we do not want the SW to be an E2EE-lifecycle participant. We accept that participation for the unification it buys.

8. Local cache & Lock-and-Wipe participation

  • Backing store: with SW decryption, the Cache API is the natural byte cache (SW-served Responses, native streaming). If we ever move decryption to the app thread, OPFS becomes the cache instead — the choice follows the decrypt location. The cache is keyed by user_id then workspace_id (assets:<user_id> → <ws>/<id>): user-scoping is the access-isolation boundary (§7), workspace-scoping makes leave / lock evict only the affected bytes.
  • Eviction: soft LRU cap (≈500 MB default). An evicted entry simply round-trips to the object store on next view.
  • Lock & Wipe: the cache + SW token store are purged via the PENDING_WIPE marker consumed at next boot (§7.2), not a runtime message. On per-workspace revoke/leave: purge that workspace's prefix.
  • Optimistic local-first write: on paste, the encoded bytes are written to the durable upload queue's own store (§9) and the display cache is warmed. The image renders immediately from local bytes; the upload drains in the background. Whether the bytes have uploaded yet is a per-device fact, read from the local queue — see the next point. Offline it simply stays pending and still renders on this device.
Upload status is per-device; do not put it in synced block properties. An earlier draft stored status: 'uploading' | 'synced' on the asset block. But block properties serialize to the single synced (and, for E2EE, encrypted) properties_json column — so a flip to 'synced' would replicate to every device and clobber a peer for which the bytes are not local. "Have the bytes uploaded?" and "are the bytes cached here?" are both per-device questions answered by the local upload queue (§9) / local cache, keyed by asset_block_id — never by a synced flag. The synced block's mere existence is the only cross-device signal needed.
The LRU display cache must not be the only home for un-uploaded bytes. The cache is evictable (and capped); if a paste happens offline, or the cap is hit before the queue drains, an evicted entry would leave a synced asset block stuck in uploading forever with no bytes to send. So pending bytes are owned by the upload queue's durable store (§9), which is not subject to the display cache's LRU, until the upload succeeds. Only then are they purely a display-cache concern.

9. Offline: the byte upload queue

Block metadata rides PowerSync's ps_crud upload queue. Bytes go to the object store on a different path with no such queue, so they need their own durable one. Crucially the queue owns the encoded bytes, not just a pointer: an IndexedDB/OPFS record of { asset_block_id, workspace_id, status, encodedBytes } (the encb:v1: payload for E2EE, raw bytes otherwise), held in a store that is not the evictable display cache, so a pending upload can never lose its source bytes to LRU eviction or an offline window (the §8 hazard). Entries drain when connectivity returns, retrying with backoff; the durable copy is deleted only on a confirmed upload, after which the bytes are purely a display-cache concern. This is new infra the pre-E2EE proposal under-specified; E2EE doesn't change it, but the design surfaces it clearly. The queue's local status is also the per-device source of truth for the "bytes pending here?" UI (§8) — there is no synced status flag. Cross-device arrival is naturally eventually-consistent: a peer that has the block but not yet the bytes shows the "metadata present, bytes pending" state until the upload from the origin device completes.

Undo / delete must compensate the byte lane. §11 puts the asset-block create + the !((id)) insert in one repo.tx so they share an undo bucket — but that only makes the DB half atomic. The bytes (queue entry + any already-PUT object) are not transactional. So: (1) the drain step re-checks the block still exists (and is not deleted) before PUT, and drops the entry otherwise; (2) undo of an asset-block insert cancels the queue entry (keyed by asset_block_id) and, if the object already uploaded, enqueues a delete. Without this, undo leaves an orphaned Storage object and a queue entry that resurrects bytes for a block that no longer exists. (This is the undo case the §16 orphan sweep does not cover.)

10. Storage layout & RLS

One private bucket, attachments. Object path <workspace_id>/<asset_block_id>. For an E2EE workspace the object is the encb:v1: ciphertext; for a plaintext workspace it is the raw bytes (identity case, §5).

Storage RLS in v1 is a workspace-membership check on the path's first segment:

create policy "attachments read by workspace member"
on storage.objects for select to authenticated
using (
  bucket_id = 'attachments'
  and private.is_workspace_member(
        (storage.foldername(name))[1],   -- <workspace_id>
        (auth.uid())::text)
);
-- writes: analogous, private.is_workspace_writer.
The private.* grant is not free — verify it, don't assume parity with blocks. private.is_workspace_member / is_workspace_writer exist and are SECURITY DEFINER, but the migrations contain no explicit GRANT USAGE ON SCHEMA private / GRANT EXECUTE to the app roles — the blocks RLS works via defaults and the fact that the policy and the table live in public. A storage.objects policy runs in a different ownership/role context (the table is owned by supabase_storage_admin, in the storage schema), so "the same functions blocks uses" is not automatically callable there. Treat the schema-USAGE + EXECUTE grants (or a thin public wrapper) as a first-class, tested step of the bucket migration — a wrong assumption yields a policy that errors at query time or silently denies all reads.

For an E2EE workspace the blob is already ciphertext, so RLS here is defense-in-depth — it stops a non-member from hoarding ciphertext against a future key compromise, and avoids leaking object existence. For a plaintext workspace it is the actual access boundary.

10.1 Server-side ciphertext enforcement for E2EE bytes

RLS is not the ciphertext guarantee — and there is no storage analog to blocks_require_ciphertext_for_e2ee for free. For blocks, a server trigger rejects a plaintext write to an E2EE workspace (is_enc_v1_envelope shape-check), so "the server only ever sees ciphertext" does not rest solely on the client upload hook. Storage has no such trigger by default, so for attachment bytes that invariant would rest entirely on client correctness — a stale client, a bug in encodeBytes, a missed mode check, or a direct authenticated write could land plaintext image bytes in an E2EE workspace's bucket. Two honest options, pick one explicitly: (a) add a server-side upload guard (a Storage trigger / Edge Function that, for objects under an E2EE workspace's prefix, rejects bytes that don't begin with the encb:v1: magic + minimum nonce/tag length — the binary analog of is_enc_v1_envelope); or (b) accept and document that, for bytes, the ciphertext guarantee is client-enforced only — a strictly weaker property than the text path, with the recovery path named. Recommended: (a), so attachments match the block guarantee.
Share-ready by construction. Keeping asset_block_id in the path means storage RLS can later be widened from "membership" to "membership or a share covers this block" by joining storage.objects → blocks → effective_share_ids when subtree sharing ships (§12). We do not build that branch now (shares aren't shipped) but the path scheme does not foreclose it. This is the cheap-option-value posture, not speculative building.

Per the project's Supabase rule, the bucket + its RLS land in one migration with storage.objects policies enabled; no public table is created without RLS.

11. The renderer & capture surfaces

Packaged as one attachments plugin, mirroring the video-player plugin's facet wiring:

  • MediaBlockRenderer via blockRenderersFacet, with canRender = ({block}) => block.hasType('media') and a priority above the default. Branches on mime: image → the existing MarkdownImage lightbox fed the virtual URL; later, PDF / audio / video / generic file-chip.
  • Markdown img override via markdownExtensionsFacet (alongside defaultMarkdownExtension.ts) for any inline image whose src is a virtual asset URL — resolve and delegate to the lightbox.
  • Capture surfaces: a CodeMirror domEventHandlers extension via codeMirrorExtensionsFacet intercepting paste / drop with clipboardData.files / dataTransfer.files; a block-shell paste/drop arm parallel to today's text/plain handlers in CodeMirrorContentRenderer.tsx / DefaultBlockRenderer.tsx; and a /image command + file-picker action through the existing action/shortcut system.
  • On capture: client-side mime/size guard → hash → best-effort dedup lookup (same-hash-in-workspace → new !((existing-id)) embed) → otherwise create the asset block as a child of the containing block (see §12), registering the type with repo.addTypeInTx(tx, id, 'media', …) (the in-tx variant — the standalone block.addType wouldn't compose into the single tx) + write bytes to the durable queue store + enqueue upload, all inside one repo.tx({scope: ChangeScope.BlockDefault}) so the URI-insert and the asset block land in the same undo bucket (byte-lane compensation per §9).
Dedup is best-effort, not an invariant. The lookup reads local synced state, so two devices (or two tabs, or an offline-then-reconnect) pasting the same image concurrently both miss, both create an asset block, and — because the object path is <ws>/<asset_block_id> — produce two Storage objects for identical bytes. The server can't break the tie (randomized GCM ⇒ no ciphertext dedup; hash is inside encrypted properties_json). So: state dedup as best-effort and make GC tolerate multiple blocks per hash. The tempting "make the id a deterministic function of (workspace_id, hash) so pastes converge" is unsafe as stated under E2EE — an id derived from the plaintext hash is a content oracle (the server could confirm a known image's presence by id, or correlate the same image across workspaces). If determinism is wanted, derive the id under the WK (HMAC(WK, hash)) so it's workspace-local and reveals nothing to the server — otherwise keep random ids + tolerate duplicates.
SW route ordering. The existing public/sw.js fetch handler routes by request.destination, and an <img> has destination 'image' — which the current handler sends to the generation assetCacheFirst branch. The new /asset/<ws>/<id> route must be matched before that branch, or the auth-first resolution in §7 is shadowed by the existing image-cache path.

12. Interaction with subtree sharing

Subtree sharing (shares-and-api-tokens.html) is not shipped but is on the roadmap. It informs two decisions now and exposes one genuine gap.

12.1 Reinforces: asset block = child of the containing block

Subtree sharing walks parent_id and stamps effective_share_ids. If the asset block is a child of the page that uses it, sharing that page's subtree automatically covers the asset (its effective_share_ids gets stamped, and the share-aware storage RLS branch grants byte access). A flat per-workspace asset container would be invisible to subtree shares. So asset-as-child is not just tidy — it is what makes a shared page carry its images.

12.2 Reinforces: keep asset_block_id in the storage path

A subtree-share recipient is not a workspace member, so granting them byte access requires storage RLS to join through to the block's share coverage (§10). The block-id-in-path keeps that join possible. Design the policy as "membership today, with an explicit seam to add OR effective_share_ids covers this block."

12.3 The gap: E2EE + subtree sharing is unsolved

A subtree-share recipient of an E2EE workspace needs a key to read anything, including asset bytes — but the model has exactly one WK per whole workspace. There is no per-subtree key. Granting subtree access to encrypted content requires the per-subtree key hierarchy the e2ee doc explicitly defers (e2ee §13). So for the foreseeable future subtree sharing is a plaintext-workspace feature, and media inherits that boundary. This is consistent with the existing tradeoff stack (e2ee §15 #2; §8.3 — E2EE sharing is out-of-band whole-workspace WK transfer). Do not special-case it.

13. The public-sharing escape hatch

We still reserve properties.publicUrl on asset blocks and a render-audience flag on the block context ('workspace' member / 'shared' subtree recipient / 'public' logged-out). But the implementation for a public, logged-out viewer differs by mode:

ModePublic-share implementation
Plaintext workspace A public bucket copy (or long-lived signed URL) written to properties.publicUrl; the renderer prefers it when audience is 'public'. The original signed-URL story works here.
E2EE workspace Either export-to-plaintext (decrypt + re-upload to a public bucket — the asset deliberately leaves E2EE, an explicit, consensual exit, never silent) or key-in-fragment (app/share#<per-asset-key>; the client decrypts from the URL fragment, which the browser never sends to the server). "Anyone with the link" is exactly key-in-link semantics.
The fragment key MUST be a fresh per-asset key, NEVER the workspace key. The §5.1 crypto seals bytes with the WK directly — there is no per-asset key in the v1 design. So a naïve "put the decrypt key in the fragment" would put the WK in a URL, exposing the entire workspace, not one image. Key-in-fragment therefore requires its own step: generate a one-off key, re-encrypt the bytes under it, upload that to a public bucket, and put only that per-asset key in the fragment. The WK never leaves the device (preserving the e2ee §5 invariant).
Document the fragment's leak surface and irrevocability. "Not sent in the HTTP request" is necessary but not sufficient: a fragment persists in history, is readable as location.hash by any script on the share page, and can ride a Referer or a third-party resource off-page. A public share page that loads a key into the DOM is a far softer target than the authenticated app (which is CSP-hardened). And it is irrevocable — once the link leaks, the asset leaks, permanently, short of re-encrypting under a new key. Both escape hatches are vNext; neither ships without these stated as accepted, explicit tradeoffs.

Reserving the property + the audience flag costs almost nothing now and avoids a content migration later. The renderer's URL resolver branches on both workspace mode and render audience.

14. Vendor portability (the BlobStore seam)

All object-store access goes behind a ~50-line interface:

interface BlobStore {
  put(workspaceId, assetBlockId, bytes): Promise<void>
  url(workspaceId, assetBlockId): string        // the virtual /asset/… URL
  delete(workspaceId, assetBlockId): Promise<void>
}

v1 backs it with Supabase Storage. E2EE actually increases portability: because the server is a dumb encrypted-blob store and never mediates bytes, swapping to R2 / S3 / B2 later is a bytes-copy plus re-pointing one module — no auth or schema rework. We do not pre-migrate; we keep the seam clean and pull the trigger if egress cost or another driver appears. (A Worker-fronted R2 from day one is defensible only if a Worker is being introduced for other reasons; otherwise it is permanent operational surface for no present benefit.)

16. Open questions

  • Cache backing if SW decryption is adopted: Cache API only, or Cache API + OPFS for an explicit LRU index? Cache API alone is simpler; OPFS gives finer eviction control.
  • Multi-account profile (decided shape, needs care): the SW session map is keyed by event.clientId → that tab's user, and the cache by user_id (§7), so a concurrent second account neither reads the wrong cache nor borrows the wrong bearer token. The residual work is making this correct under switch and under the per-user wipe — mirror the care keyStore.clearForUser already takes (it is per-user precisely so account A's wipe doesn't disturb B).
  • Max inline size + large-media story: a v1 cap (e.g. 10 MB) with a clear error, deferring resumable/tus upload to vNext. Where exactly is the cap enforced?
  • Asset block deletion → byte GC: delete the asset block ⇒ delete the object (storage cascade or an explicit step). Embeds that dangle render the existing "unresolved embed" state. The undo case is handled by the §9 byte-lane compensator; the broader orphan sweep is a vNext follow-up.
  • Cache backing: Cache API only, or Cache API + an IndexedDB LRU index for finer eviction? Cache API alone is simpler; the durable upload-queue store (§9) is separate from either.

17. Tradeoffs we're accepting

ChoiceCostWhy / exit
SW is an E2EE-lifecycle participant A per-client token handshake; decrypted bytes + token in a new (Cache API / SW) location. Buys a mode-agnostic renderer and native media semantics. Wipe is marker-gated at next boot (§7.2), not a racy runtime message; mandatory and tested (incl. a "plaintext survives lock" regression). Consistent with e2ee §15 #7.
Storage byte-length is visible to the server Exact encrypted-object size (≈ plaintext + 28 B) + existence/count — a stronger known-file fingerprint than the accepted block-metadata leak (§4). New accepted leak, outside the e2ee §15 set. Size-bucket padding if the threat model later cares.
Byte AAD doesn't bind freshness An untrusted server can replay an older ciphertext for the same asset (no PowerSync monotonicity on the byte lane). Closed by verifying decrypted bytes against the block's synced hash on read (§5.1) — not left as a residual.
Server byte-ciphertext enforcement is a deliberate add Without §10.1's upload guard, "server sees only ciphertext" rests on client correctness for bytes — weaker than the block trigger. Recommended: ship the storage upload guard (§10.1) so bytes match the block guarantee; otherwise document the weaker property explicitly.
Bytes off the sync path A second durable upload queue; eventual consistency between metadata and bytes; undo needs a byte-lane compensator (§9). Keeps large blobs out of SQLite/PowerSync entirely; the queue is small and self-draining.
Dedup is best-effort Concurrent same-hash pastes can produce duplicate objects (§11). GC tolerates multiple blocks per hash; WK-keyed deterministic ids are the only safe convergence option (a plaintext-hash id would be a content oracle).
No server-side media features for E2EE No server thumbnails / OCR / transcode for encrypted workspaces. Inherited from e2ee §15 #2, not new. Client-side transforms remain possible.
E2EE subtree-share of media unsupported Encrypted workspaces can't subtree-share images until the key hierarchy lands. Needs e2ee §13. Plaintext workspaces are unaffected; don't special-case.
Supabase Storage in v1 Egress cost at scale. Behind BlobStore; E2EE makes the backend a dumb blob store, so the swap is a bytes-copy + one module.
