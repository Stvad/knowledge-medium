Media attachments

Design for image (and, by extension, arbitrary file) attachments in a system that now has per-workspace opt-in end-to-end encryption. 2026-06-21.

Status: design — not yet implemented (no src/ code yet); last verified against code 2026-06-23 (a grounding pass found zero wrong code claims). Treat as current intent. The least-settled part is the byte-lane lifecycle / failure-recovery machinery (§8–§9, §16): the happy path and the crypto / threat model are solid; the recovery edges are where the spec is still hardening. The block-id vs keyed-hash storage path (§10) is an open design fork.
TL;DR. An attachment is a block carrying the registered media type (in the types array, via typesProp), embedded into prose with the existing !((id)) mechanism — no new reference syntax, and metadata rides the existing encrypted block-content columns for free. The bytes live in an object store (Supabase Storage v1), behind a BlobStore interface. The reshaping constraint is E2EE: for an encrypted workspace the server holds only ciphertext and never the key, so bytes are sealed client-side with the workspace key and access becomes entirely client-side. We do not write two implementations — there is one byte transform parameterized by mode, where plaintext is the identity case, exactly as src/sync/transform.ts already does for text. Decryption happens in-thread (in the app / its data worker), keeping the keys, session, mode pin, and the already-decrypted block all in one context; the renderer is mode-aware (plaintext → a direct <img src>; E2EE → fetch + decrypt + an object URL). A service-worker variant that would make <img src="${BASE_URL}asset/…"> uniform across modes was evaluated and rejected — its one benefit doesn't earn the app↔SW coordination tax (§7 / §7.3 / §18).

1. What we're adding, and scope

The app cannot currently insert images. We want paste / drop / file-pick of an image to render inline, work offline, and sync across a user's devices — and we want the same machinery to extend to PDFs, audio, video, and generic files later without a content migration.

In scope (v1): images, via paste / drop / file picker; inline render with a lightbox; offline-first local caching; cross-device sync; correctness under E2EE workspaces.

Designed-for but not built (vNext): non-image file types (the model is type-dispatched, so a new mime is a new renderer branch, not a schema change); thumbnails / transforms; resumable uploads for large media; public sharing.

Explicitly out of scope: server-side media features for E2EE workspaces (OCR, thumbnails, transcode) — the server has no key; this is the existing E2EE tradeoff (e2ee-design.html §15), inherited, not new. E2EE subtree sharing of media (needs the deferred key hierarchy — see §12).

3. The core model: an attachment is a block

An attachment is not a row in a side table. It is a blocks row carrying the media type and binary-pointer metadata in its properties:

// asset block (illustrative). The type lives in the `types` ARRAY (typesProp),
// NOT a `properties.type` scalar — that's what the type registry, type queries,
// and the property panel read. Set via block.addType('media') / typesProp.
properties: {
  types:       ['media'],       // typesProp — registered media type id
  mime:        'image/png',
  size:        184213,          // plaintext byte length
  hash:        'sha256:…',      // of plaintext bytes; client-side dedup key
  filename:    'screenshot.png',
  width:       1280,            // optional, for layout reservation
  height:      720
  // NOTE: no upload `status` here — that's per-device, tracked in the local
  // upload queue (§9), NOT in synced (encrypted) properties_json. See §8.
}
content: 'image: screenshot.png'   // human-readable summary

The media type is registered as a TypeContribution — defineBlockType({id: 'media', label, properties}) contributed via typesFacet.of(mediaType, {source: 'attachments'}) (the pattern daily-notes / geo / block-tagging use), so type-filtered queries, the property panel, and renderer discovery all see media blocks. This is not the same as video-player's setup, which contributes property schemas + a renderer but no type — and a bare property schema is insufficient: the capture path's repo.addTypeInTx(tx, id, 'media', …) looks the id up in the type registry (types.get(typeId)) and throws unless a TypeContribution was registered through typesFacet. (Stamping a bare properties.type scalar instead would also make blocks invisible to every types-array consumer — getBlockTypes / hasBlockType / block.hasType.)

Why a block, not an attachments table:

  • Inspectable / user-accessible — the project's standing principle. An attachment shows up in the outline, the properties panel, queries, and the agent runtime like any other block.
  • Embeds, backlinks, annotations come for free. Inline placement uses the existing block embed !((id)) (src/components/references/BlockEmbed.tsx); captions / alt-text / OCR / transcripts are child blocks; "what references this image?" is the existing backlink machinery.
  • Type-dispatched rendering already exists. A MediaBlockRenderer registers via blockRenderersFacet with a canRender gate on block.hasType('media') — the same registration pattern as VideoPlayerRenderer. New file types are new mime branches, never a schema change.
  • Dedup is natural and workspace-wide: paste the same screenshot ten times anywhere in a workspace and you get ten !((id)) embeds of one asset block — one byte copy, one upload (see §11).

4. The constraint that reshapes everything: E2EE

For an E2EE workspace the server holds only ciphertext and never the key. The workspace key (WK) lives as a non-extractable WebCrypto CryptoKey in IndexedDB (km-e2ee-keys, keyed by (user_id, workspace_id) — src/sync/keys/keyStore.ts). The server-side trigger blocks_require_ciphertext_for_e2ee actively rejects any plaintext write to an encrypted workspace's content columns.

The E2EE design doc encrypts exactly three text columns and says nothing about binary data — confirmed by inspection. So attachments are an unsolved problem that must be solved consistently with E2EE, not bolted on beside it. Two pillars of any naïve "URL in markdown" approach are dead on arrival:

A signed Storage URL handed straight to <img src> — dead for E2EE. The object is ciphertext, so the browser can't render it directly; the bytes have to be fetched and decrypted client-side with the WK first. Note what this does not kill: signed URLs as a fetch mechanism are fine — what's dead is using one as the render URL. The fix is to fetch + decrypt client-side and hand <img> the result: in-thread (the chosen path, §7.3 — fetch → decrypt → a blob: object URL), or via a service-worker proxy behind a virtual ${BASE_URL}asset/… URL (§7, the rejected alternative). Either way the byte fetch can use a signed URL or a bearer-token GET (the in-app path uses the bearer token, §7.1). So <img src> works again — pointed at an object URL (in-thread) or the virtual URL (SW), never the raw ciphertext object.
Public-bucket / public-share-URL — incompatible with E2EE. You cannot serve a publicly-readable image out of an encrypted workspace via RLS + signed URLs. RLS gates access to ciphertext; it cannot grant decryption. (This is just e2ee §15 #2 — "no server-side features for E2EE workspaces" — applied to bytes.) The escape hatch survives in a different form; see §13.

What the encryption model gives us in return is also worth stating: because metadata rides the encrypted properties_json column, an attachment's mime, filename, and hash are confidential automatically on an E2EE workspace. The block-row routing identifiers (id, workspace_id) are in clear — exactly the columns the sync transform already leaves in clear.

But the Storage object leaks more than the block row does — call it out, don't fold it into the inherited block-metadata leak. The encrypted blob's byte length (≈ plaintext size + 28 B) is visible to the server via the stored object size and the Content-Length of an RLS-gated GET, and the object's existence and per-workspace count are visible. Exact file size is a much stronger fingerprint than note-text length — it can confirm a specific known file (a leaked document, a known image) by size, and combined with the asset_block_id in the path + timing, correlate which block embeds which known-size asset. This is a new leak outside the enumerated e2ee §15 set (tree shape / order / timing / authorship). Document it as an accepted tradeoff (§17), or pad ciphertext to size buckets if the threat model cares.

5. One byte transform; plaintext is the identity case

E2EE is opt-in, so most workspaces are plaintext. The temptation is two code paths. The codebase already shows the better answer: src/sync/transform.ts is one function parameterized by mode, where mode === 'none' is the identity transform.

// the text precedent, today
encodeForWire(row, 'none', getCek)  // → row unchanged
encodeForWire(row, 'e2ee', getCek)  // → seal content/properties_json/references_json

We mirror it for bytes — a new module in the same src/sync/ seam:

encodeBytes(bytes: Uint8Array, mode: SyncMode, getCek: GetCek): Promise<Uint8Array>
decodeBytes(blob:  Uint8Array, mode: SyncMode, getCek: GetCek): Promise<Uint8Array>
//   mode === 'none'  → identity (return input)
//   mode === 'e2ee'  → AES-256-GCM seal/open with the WK
Decision. There is one attachment implementation, not two. Everything expensive — the asset-as-block model, the BlobStore, the cache, the upload queue, dedup, the renderer — is mode-agnostic and shared. Only the ~15-line encode/decode seam is mode-aware, and plaintext is its degenerate (identity) branch, exactly like encodeForWire.

5.1 Reusing the crypto primitives

The WK CryptoKey is AES-256-GCM and works for crypto.subtle.encrypt over raw bytes, not just the string seal/open in src/sync/crypto/aead.ts. So byte encryption reuses the same getCek(workspaceId) seam the sync transform uses, with two deltas:

  • AAD binding extends the existing pattern. Where content columns bind contentAad(id, workspace_id, column), asset bytes bind (asset_block_id, workspace_id, "asset-bytes", schema_version) so a ciphertext blob cannot be swapped between blocks or workspaces. (Schema version stays in the AAD for the same cheap-hierarchy-upgrade reason §13 of the e2ee doc keeps it there.)
  • A binary envelope, not the base64url text envelope. The enc:v1: envelope is base64url (≈33% bloat) — wrong for bytes. Add a sibling encb:v1: binary framing (magic ‖ nonce ‖ ciphertext‖tag as raw bytes) sharing the nonce/tag layout. Small new module next to aead.ts.
The AAD does not bind freshness — verify the content hash on read. Every AAD term (asset_block_id, workspace_id, the literal, schema version) and the storage path are stable across an asset's whole life. AAD stops cross-block / cross-workspace swaps, but it does not stop a rollback/replay of an older ciphertext for the same asset_block_id: an untrusted server (in scope per e2ee §2 — it may serve whichever bytes it likes) can return a stale blob that decrypts cleanly and the GCM tag verifies. The bytes lane has no updated_at/monotonicity channel the text path gets from PowerSync. The fix is already in hand: the asset block stores hash = sha256(plaintext) in its (encrypted) properties_json; the read path must verify the decrypted bytes against that hash before caching/serving, and reject on mismatch. In the chosen in-thread path (§7.3) the resolver already holds the decrypted block, so the expected hash is just block.properties.hash — no extra plumbing. (The rejected SW path couldn't read the encrypted block state, so it would have needed an SW-readable metadata mirror, §7.1.1 — not a request param: an <img src> can't carry custom headers, and a ?h= query param would leak the plaintext SHA-256.) Either way it's a positive check that can only fail closed, never downgrade.
Read-side fail-closed is not enough — the write lane must be monotonic too. A new hash makes reads reject stale bytes, but two uploads targeting the same <ws>/<asset_block_id> path can still race: if an asset is edited and a stale queued/in-flight upload lands after the newer one, the old body overwrites the object after the block's hash already advanced — then every reader rejects on mismatch and the asset is permanently broken (the newer queue entry was already confirmed and dropped). The fix is immutable, write-once asset objects: an edit/replacement creates a new asset block (new id → new path) and re-points the embed, rather than re-uploading under the existing id. Bytes under a given asset_block_id never change, so there is no same-path collision and no "older version" for the server to replay; the §10.1 upload guard enforces it with a no-overwrite precondition (reject if the object already exists). The old object orphans and is GC'd (§16). (Dedup still reuses an existing immutable asset block by hash — that's a read, not a rewrite.)

The encb:v1: magic is a format tag, not an authorization signal. Whether to decrypt is decided by the workspace's mode pin (§7), exactly as the text path decides via the pin rather than the enc:v1: prefix (envelope.ts / e2ee §6 rule 2). An E2EE-pinned workspace requires the magic and treats its absence as tampering; only a plaintext-pinned workspace passes bytes through. A workspace with no pin (never validated, or evicted) is not plaintext — it is the resolver's defer/encryption-uncertain state (src/sync/keys/resolver.ts), so the read path fails closed (placeholder), never passthrough — otherwise an encrypted workspace with a missing local pin would serve attacker-controlled plaintext. The decode decision is therefore three-valued — decrypt (E2EE + WK) / passthrough (plaintext-pinned) / fail-closed (E2EE without WK, or no/uncertain pin) — and the in-thread resolver must drive it from that three-valued materializability (getMaterializability), not the two-valued SyncMode/getMode the upload direction uses (§7.3).

6. The byte path, end to end

Bytes do not ride PowerSync. The asset block (metadata) syncs through the normal Layout B path (encrypted content columns, observer-reconciled). The bytes take a parallel path to the object store. The two arrive independently on other devices.

7. The decryption boundary (chosen: in-thread)

Decision: decrypt in-thread (§7.3), not in the service worker. The byte cache is OPFS + object URLs; the renderer carries a small mode branch. The SW machinery documented in §7.1–§7.2 below — the token handshake, the SW-readable mode-pin + per-asset hash mirror, per-clientId sessions, controller-readiness gating — is not built; it is retained as the analyzed-and-rejected alternative, because it is the cost side of the decision. §7.3 is the chosen path and the full tally; §8–§10 are boundary-agnostic except where they name the SW. (Earlier drafts led with the SW path; this section keeps that analysis, but the verdict is in-thread — see §7.3 / §18.)

The case for the SW was that it keeps the renderer mode-agnostic: decrypt in the worker and <img src="${BASE_URL}asset/…"> "just works" for plaintext and E2EE alike. It is viable — the SW global scope has crypto.subtle, IndexedDB, and the Cache API. What sinks it is not feasibility but the coordination tax of putting the decryptor in a context that can't read app state and is shared across tabs:

  • The renderer emits a stable virtual URL — BASE_URL-prefixed. <img src="${BASE_URL}asset/<workspace_id>/<asset_block_id>">, not a root-absolute /asset/… (shorthand below). The SW registers with scope: import.meta.env.BASE_URL (src/registerServiceWorker.ts), so under a non-root APP_BASE_PATH (e.g. /app/) a root-absolute URL escapes the SW scope entirely — the worker never sees the request, the auth/decrypt path is bypassed, and the origin returns 404. Build the URL relative to BASE_URL and match the ${BASE_URL}asset/… route in the worker. No mode awareness, no createObjectURL lifecycle for inline subresources (<img>, <video>) from a controlled tab. A raw <a download href="${BASE_URL}asset/…"> is not covered — a download/navigation can lack a clientId to authorize (§7.1), so "download original" is an app-mediated action (the app resolves the bytes and saves from an in-memory blob), not a raw link to the asset URL. (Range/streaming is also an exception under E2EE — see the note below; images and moderate files are unaffected.)
  • The SW resolves it, auth-first: establish the active user_id (for the requesting client — §7.1) and the workspace's mode pin + WK availability before consulting the cache → serve from cache only on a namespace match, a still-valid session, and a matching content hash (from the SW metadata mirror; the cache key includes it, §8) → on miss, fetch the stored blob → decrypt/passthrough per the mode pin (below) → verify against the mirror's expected hash (§5.1) → put the displayable bytes in the user-scoped, hash-keyed cache → respond.
  • The mode pin decides decrypt-vs-passthrough, NOT the envelope magic. This is a deliberate carry-over of the rule the text path already enforces (src/sync/crypto/envelope.ts / e2ee §6 rule 2: "the prefix is a FORMAT tag, not a decrypt trigger — the durable mode pin decides"). The SW reads the workspace's mode pin from a SW-readable source (§7.1.1): if the workspace is E2EE-pinned, the SW requires a well-formed encb:v1: envelope and decrypts — a blob that arrives without the magic is treated as tampering and rejected, never served as plaintext. Only a plaintext-pinned workspace passes bytes through; an unpinned / no-pin workspace is the resolver's defer state and fails closed (not plaintext). Deciding from the self-describing magic alone would re-introduce exactly the downgrade vector (hostile server strips the magic → attacker plaintext reaches <img>) that the text wire-boundary was hardened against.
  • Render gates on SW readiness (cold start). An <img src="${BASE_URL}asset/…"> can fire before the app has handed the SW a token (§7.1) or before the SW has re-read its state on a cold respawn. The SW must not "fail open": a request with no established active user / no fresh unlock signal respondWiths a deterministic placeholder (or holds briefly on a readiness promise, short timeout → placeholder), never a stale cache hit and never an unauthorized fetch. But the SW can only do this if it is controlling the page — and on a first production visit (or where SW registration is unavailable) it is not: public/sw.js deliberately omits clients.claim(), so a fresh install controls only the next load. An ${BASE_URL}asset/… <img> emitted while uncontrolled goes straight to the origin (→ 404, bypassing all of the above). So the renderer must gate URL emission on navigator.serviceWorker.controller: only emit the virtual URL when a controller exists; otherwise render a placeholder (and retry once controlled) or fall back to the app-thread resolve (§7.3). Don't rely on the SW catching an already-dispatched request — an uncontrolled page never reaches it.

The thing that actually constrains the SW approach is not decryption — it's two coordination problems, and they are the real work of this design:

7.1 Coordination problem A — fetching private bytes needs the session

Storage objects are RLS-protected; reads need a bearer token, which lives in the app's supabase-js session in localStorage — which the SW cannot read. Resolution: each app client postMessages { userId, accessToken } to the SW on onAuthStateChange / token refresh, the SW records it keyed by the sender's client.id (not a single global slot — see the cross-account defense above), and on a fetch the SW looks up the session for that request's event.clientId and issues GET /storage/v1/object/attachments/<ws>/<id> with Authorization: Bearer. This also eliminates per-object signed URLs for the in-app case (signed URLs are only needed for the logged-out public case, §13). Signed URLs would not be cheaper anyway: minting one needs the user's JWT (which is this handshake), or the app must push per-object expiring URLs to the SW — heavier than one hourly-refreshed bearer token. The token expires hourly; the app re-posts on refresh.

This auth scheme covers controlled-tab subresource fetches only — not raw navigations. A <img>/<video> subresource carries the originating tab's event.clientId, so the per-user token lookup works. A navigation to an asset URL (new-tab open, or a download flow that navigates) may have no usable clientId (a fresh top-level navigation has only a not-yet-messaged resultingClientId), so there's no per-user token to attribute — and falling back to a global token would break the multi-account isolation this whole section preserves. So direct asset navigations are not a supported auth path: "download" / "open original" is an app-mediated action (the app resolves the bytes through the same path and triggers the download from an in-memory blob), never a raw navigation to ${BASE_URL}asset/…. The route still sits ahead of the navigation branch (§11) so such a request can't return the SPA shell — but for a clientId-less navigation it returns a placeholder/redirect into the app, not bytes and not the shell. (If first-class direct navigations are ever wanted, they need an opaque app-minted capability token in the URL — deferred; the public-share path §13 is the related case.)

Persistence across SW respawn is deliberately minimal: the per-clientId token store is in-memory (SW-lifetime), so a respawn simply waits for the app to re-post on the next onAuthStateChange / refresh. A bearer token is never written to durable storage — so there is nothing for the coarse clear (§7.2) to miss — and the SW drops a client's entry when that client goes away.

7.1.1 The SW needs three things the renderer/app hold — make them SW-readable

The URL carries only workspace_id and asset_block_id. The decrypt path needs two more facts that today live where the SW cannot reach them — this is a real data-availability requirement, not an aside:

  • Mode pin → a SW-readable durable store (IndexedDB). Pins live in window.localStorage (src/sync/keys/modePin.ts), which a service worker cannot read. The SW needs a durable IndexedDB source (the mode must stay known across reloads so an encrypted workspace re-prompts for its key rather than being misread as plaintext — e2ee §6). The mode is authoritative, so it must come from this durable pin, never from a per-request claim (an app-claimed "plaintext" would be the downgrade vector again). If no pin exists for the workspace yet (never validated), the SW fails closed — never a magic-fallback.
    (Rejected-path detail.) The SW path would have required moving the pin to IndexedDB as the single source of truth (not a localStorage mirror) so the SW could read it. The chosen in-thread path needs none of this — it reads the existing localStorage pin synchronously (§7.3). Kept only as rejected-alternative analysis: the modePin backing would change from localStorage to IDB, both app and SW reading the same store.
    • Async is a non-issue because the app keeps a boot-hydrated in-memory cache: at data-layer init (where the keyStore / DB already open) load all pins — a handful, one per workspace — into an in-memory map, and the app's synchronous getModePin reads off that map. Writes are write-through (sync map + async IDB) and are rare (only on mode establishment / key validation), so async IDB never touches the hot read path. The SW reads IDB directly, but its fetch handler is already async (and the §7.1.1 in-SW cache below amortizes it).
    • Durability downgrade is acceptable and fails closed. IDB is evictable under storage pressure (localStorage less so) — but a missing pin for an encrypted workspace resolves to defer (don't materialize, re-prompt), never "plaintext". So eviction → user re-affirms workspace status → pin re-established; it can never silently downgrade. Preserve the existing pin semantics through the move: keyed per (user_id, workspace_id) — the IDB store is shared across accounts in the profile but each record is per-user, exactly as modePin.ts keys today; a workspace-only record would let account B consume account A's pin for the same workspace, skipping B's own first-encounter validation and breaking the per-user WK model. And the in-memory fallback stays plaintext-only: modePin.ts today falls back in-session only for a confirmed-plaintext pin and throws on durable-pin failure, precisely because E2EE needs a reload-surviving authority. Keep that — if the durable IDB pin can't be written, an E2EE create/unlock must fail (don't pin in memory), since a reload would lose the authority and strand the workspace in first-encounter defer. A memory-only E2EE pin is never acceptable.
  • Expected hash → SW-readable metadata mirror, NOT a URL param. The replay/rollback defense (§5.1) verifies decrypted bytes against the block's hash, but that hash is in the block's encrypted properties_json — which the SW can't read. An earlier draft put it in the URL (?h=<sha256>); that is wrong — it leaks the plaintext SHA-256 to the origin on any SW-interception miss (and misses are real: public/sw.js deliberately omits clients.claim(), so a fresh install controls only the next load, and registration can be unavailable), which is exactly the known-file content oracle §10 rejects and §4 promises against. Instead, the app writes per-asset metadata to the same SW-readable IDB mirror as the mode pin (§7.1.1, extended to {asset_block_id → expected hash}) when it materializes the block; the SW reads the expected hash from there to (a) build the hash-keyed cache key (§8) and (b) verify decrypted bytes before caching/serving. The URL stays clean (${BASE_URL}asset/<ws>/<id>) — no plaintext hash ever on the wire. A missing mirror entry for an E2EE asset fails closed (placeholder), never serves unverified bytes — closing the same-asset replay window without the leak. (The mirror is ordinary SW-readable IDB, removed by the coarse clear-all like every other local store, §7.2.)
  • Active user + token — already covered by the §7.1 per-clientId handshake.
Cache the SW's IDB reads on the hot path — keyed by (user_id, workspace_id). Both the mode-pin store and the WK store (km-e2ee-keys) are now SW-readable IDB, so a naïve handler does two async IDB opens per <img> fetch. Keep a small in-SW in-memory cache of { (user_id, workspace_id) → mode pin, WK handle, map of asset_block_id → expected hash }. The expected hash is per-asset, not per-workspace — a workspace has many attachments, so a single expected hash on the workspace entry would verify (or key the cache for) asset B with asset A's hash → false mismatches; hold a per-asset_block_id map inside the workspace entry (or a separate (user_id, asset_block_id) → hash map). The mode pin / WK stay at (user_id, workspace_id). The user_id in the key is load-bearing, not an optimization detail: the keyStore deliberately keys WKs by (user_id, workspace_id) (shared across accounts in the profile), so a workspace-only hot cache would let account A's populated WK/pin serve account B's request for the same encrypted workspace — B could decrypt while B is locked or has never loaded its own WK, bypassing the per-user isolation. Resolve the requesting client's user_id (§7.1) and key both the cache and its invalidation by (user_id, workspace_id). Populated lazily and invalidated on auth/token messages (§7.1) and on the per-workspace revoke/leave purge. The cache is per-SW-lifetime, so a respawn re-reads IDB once; correctness still comes from IDB + the invalidation signals, the cache only avoids per-request opens.
Whole-blob AEAD ≠ native range/streaming for E2EE. The encb:v1: envelope is a single AES-GCM seal over the whole byte array, and a GCM ciphertext byte range cannot be decrypted independently. So an E2EE <video> (or any Range request) cannot stream a slice: the SW would have to fetch+decrypt the entire object and synthesize a 206 from the plaintext — fine for images and moderate files, but it defeats true streaming for large video. Two honest paths, both vNext: the SW fully materializes + serves synthesized ranges (simple, no streaming), or large media uses a chunked envelope (per-chunk nonce/tag, range maps to chunk boundaries) designed before relying on streaming <video>. v1 images don't hit this; the doc just must not promise native ranges under E2EE. (Plaintext workspaces keep native ranges — the object is the raw bytes.)
Cache hits must be authorized, and the cache key must be user-scoped. Decrypted bytes in the Cache API are not themselves access-controlled, so a cache key of /asset/<ws>/<id> alone would let a later request return previously-decrypted bytes without re-checking the active session or key — bypassing Storage RLS and key availability across an account switch, a workspace revoke/leave, or a purge that missed an entry. Three defenses, all required: (1) namespace the cache by user_id (a per-user bucket assets:<user_id>); (2) resolve the active user per requesting client, not globally — a single SW serves every tab, and a browser profile can hold two accounts at once (the keyStore is shared across accounts), so the SW keys its session map by event.clientId → that tab's user, never a single global "active user" slot that the last onAuthStateChange wins; and (3) validate before every hit against key/session state — for E2EE, treat the workspace's WK being absent from IndexedDB as locked (evict rather than serve). There is no in-app lock action racing the SW (destruction is the coarse clear, §7.2), so WK-absence is an eventually-consistent signal, made robust by the TTL revalidation below; it also doesn't gate plaintext-workspace cached bytes, so neither check stands alone — the revoke/leave purge + the TTL revalidation are the real defense. Note the cross-account case is the subtle one (§16): the cache namespace isolates reads, while the per-client session keying is what isolates the fetch token.
Membership revocation needs the same backstop as lock — especially for plaintext. The locked-state check above is the E2EE arm; the symmetric hazard is a user removed from a workspace server-side (by another admin), with no local signal. For a plaintext workspace — where §10 says RLS is the access boundary — a stale cache hit in assets:<user_id> would keep serving bytes after removal if the revoke-purge is delayed or misses an entry, because a cache hit never goes back through Storage RLS. So a cache hit must never be treated as an independent authority: (a) purge the workspace's cache prefix on leave/revoke (primary, but it can race / miss); and (b) bound cache authority — give cached entries a short freshness TTL after which the SW re-validates access (cheapest: re-issue the RLS-gated Storage GET, which 403s for a removed member, before serving a stale-past-TTL hit). The membership signal is not locally authoritative, so the TTL-bounded revalidation — not just the purge — is what closes the plaintext-removal window. (E2EE leans on the same shape, with key-absence as the faster signal.)

7.2 Destroying local data is coarse and system-level — not an in-app wipe

The byte stores add no selective wipe machinery. Destroying local data is a coarse, all-or-nothing action delegated to the platform: the browser's own "clear site data" wipes the whole origin — SQLite DB, E2EE keys, and every byte store (OPFS, IndexedDB, Cache API) — in one shot. This shipped on master (2026-06-23): the selective per-user in-app Lock & Wipe flow (a PENDING_WIPE marker, a boot-time consumePendingWipe purge, an app→SW purge handshake) was removed, and lock_and_wipe_local_data is now a thin guidance dialog (WipeLocalDataDialog) that hands off to the platform control and warns on unsynced changes. The team deliberately did not hand-roll a coarse destroyAllLocalData() either — the platform wipe runs outside the page/SW context, so it sidesteps the cross-tab OPFS race, the surviving service worker, and the pre-render boot gate a hand-rolled wipe would each have to solve (and a SW can't even synthesize a Clear-Site-Data header — Chromium ignores it; a programmatic trigger is deferred until the app leaves GitHub Pages, issue #245). Consequences for this design:
  • No wipe participants. The replica, display cache, upload queue, SW token store and hash mirror are ordinary local storage; the coarse clear removes them with the DB and keys — nothing selective to drain, mark, or purge, and no app→SW wipe coordination to get right.
  • No "plaintext survives a lock" hazard. Decrypted bytes in the Cache API (SW path) or OPFS (app-thread path) are destroyed by the same clear that removes everything else, so storing them locally is consistent with the accepted posture (e2ee §15 #7) without a per-store wipe rule to enforce.
  • Re-securing a device is the coarse clear — it logs out every account / workspace in the profile, not one. Accepted: a paranoid user would clear site data anyway, and re-affirming workspace status (re-paste the WK) afterward is fine.
The runtime cache-hit authorization checks above (account switch, membership revoke, WK absence) are independent of destruction and stay — they gate serving stale bytes, which no amount of wiping addresses.
That tax is why the SW path is rejected. Everything in §7.1–§7.2 — the token handshake, the SW-readable mode/hash mirror, per-clientId sessions, controller gating — is coordination the SW incurs only to buy one thing: a uniform mode-agnostic <img src>. The in-thread path (§7.3) gives that up for a small renderer mode-branch and pays none of the tax. See §7.3 for the full tally and §18 for the decision record. §8–§10 below are boundary-agnostic except where they name the SW.

7.3 The chosen approach: in-thread decryption

In-thread (app, or its dedicated data worker) decryption: the renderer fetches the ciphertext, decrypts via getCek directly (zero duplication), caches plaintext bytes, and hands <img> an object URL. Because it runs in the context that already owns the session, the keys, the mode pin, and the decrypted block, it pays none of the SW's coordination tax. What it avoids:

  • No mode-pin IDB store / no per-asset hash mirror. The app reads the mode pin synchronously and already holds the decrypted block (so the hash is just block.properties.hash) — the §7.1.1 mirror machinery exists only because the SW can't reach app state.
  • Plaintext bytes in OPFS (not SQLite, not Cache API). The app-thread path caches decrypted bytes as plain OPFS files (attachments/<user>/<ws>/<id>) — the store the bytes always wanted (§6/§17 keep large blobs out of SQLite; an earlier draft wrongly floated DB-blob storage purely for a "free wipe," which both bloats the hot query DB and contradicts that principle). Destruction is not a differentiator either way now — the coarse clear (§7.2) removes the OPFS tree and the Cache API alike. The real app-thread win here is simplicity of resolution: the app owns the OPFS handle and the session, so fetch→decrypt→write-OPFS→object-URL is one local flow with no SW cache/token/mirror lifecycle.
  • No token handshake / no per-clientId session — the app already has the Supabase session.

What it costs: a mode-aware renderer (plaintext sets <img src> directly; E2EE fetches + decrypts + sets an object URL) and manual object-URL lifecycle (createObjectURL/revokeObjectURL) — two local, well-understood concerns. Note OPFS is not URL-addressable, so <img> needs an object URL either way; a path-like stable URL is the only thing the SW route would have added.

Three properties the in-thread path must carry explicitly. They held in the SW analysis above, but the chosen path is where they ship, so they are restated here rather than left in the rejected framing:

  • Fail-closed decode, not two-way. The resolver decides per a three-valued materializability — decrypt (E2EE + WK present) / passthrough (plaintext-pinned) / fail-closed placeholder (E2EE without WK, or an unpinned / defer workspace) — read from getMaterializability (src/sync/keys/resolver.ts), never the two-valued SyncMode/getMode the upload path uses. Driving the read direction off getMode (which collapses unpinned → none) would pass attacker-controlled bytes through to <img> for an E2EE workspace whose pin was evicted — the exact downgrade §5.1 / e2ee §6 rule 2 forbid.
  • Hash mismatch is hard-fail. On a mismatch against block.properties.hash (§5.1) the resolver discards the bytes — never writes them to the OPFS display cache — and renders the broken-asset placeholder; unverified bytes are never served or cached.
  • Cached bytes are not an independent authority. The OPFS display cache is namespaced by user_id (account isolation); on workspace leave/revoke its prefix is purged (primary, but can race / miss); and — the backstop that actually closes the plaintext-removal window — a cached entry past a short freshness TTL is re-validated by re-issuing the RLS-gated Storage GET (403 for a removed member) before serving — that revalidation GET must hit the origin (auth-bearing, non-cacheable, and the private bucket sets no public cacheControl) so a CDN 200 can't mask the 403. For E2EE, WK-absence is treated as locked (evict rather than serve). These are the same guarantees the rejected SW path stated for the Cache API (the §7.1.1 cache-auth boxes), now for the in-thread OPFS cache.
Why in-thread wins the tally. The SW concentrates ~5 distinct coordination mechanisms to buy exactly one benefit; in-thread pays two small local concerns and avoids all of them.
  • SW-only costs, each a boundary artifact: the app→SW token handshake (§7.1); the SW-readable mode-pin + per-asset hash mirror, because the SW can't read the DB (§7.1.1); per-clientId sessions + per-user cache keying, because one SW is shared across all tabs/accounts of the origin; and the controller-readiness / route-ordering / clientId-less-navigation lifecycle (§7, §11). The in-thread context has none of these — it reads the mode pin synchronously (resolving WK-presence on the already-async render path) and already holds the decrypted block (the hash is block.properties.hash), so the mirror simply doesn't exist.
  • The SW's one benefit is weak here: a uniform <img src> across modes. But <img> takes an object URL just fine, so in-thread reproduces the result with a one-line mode branch; the SW's interception isn't even fully uniform (clientId-less navigations already route through the app, §7.1). It would only pay off if some media render site emitted a raw <img src="…asset…"> the app can't funnel through its resolver — and in a controlled markdown pipeline there is none.
  • Signed URLs don't rescue the SW: minting one needs the user's JWT (which is the handshake), or the app must feed per-object expiring URLs to the SW (heavier than one bearer token). A viable fetch mechanism (§4), not a cheaper one.
  • The DB isn't reachable from the SW even though a DB worker exists: PowerSync runs a SharedWorker in multi-tab mode (repoProvider.ts), but it is internal and a ServiceWorker can't connect to it; reading a row would mean a BroadcastChannel round-trip through a live tab — and if a tab is alive to answer, decryption belongs in that tab. So the hash mirror (or in-thread) is the only practical answer; in-thread makes it vanish.
  • Not differentiators: native Range/streaming — whole-blob AEAD defeats it on both paths (fix is chunked encryption); and wipe participation — destruction is the coarse platform clear on both (§7.2).
The general rule, useful beyond media: a service worker earns its tax only when the consumer needs a real URL, not a blob. <img> takes a blob → in-thread. A consumer that consumes URLs — e.g. the JS module loader for user plugins (import() of a /module/<blockId>) — is where a SW-served URL genuinely pays off, and even there the clean shape keeps the privileged work in-thread and lets the SW serve a pre-populated cache (separate investigation).

8. Local replication: byte replica & display cache

PowerSync already gives every device a full local replica of the block rows, so "it's all local" already holds for metadata. Media bytes are the one thing that breaks it — they live out-of-band in Storage, not in the synced DB — so for genuine offline operation they get their own replication, with the same target steady state as the DB: every asset for a workspace you use is on disk. We reach it by eager intent, lazy timing — a background, idle-priority, proximity-first drip that converges on full replication, not a boot-time thundering herd. That turns local bytes from a mere cache (lazy, fetch-on-render) into a replica (the same relationship the DB has to the server), in two tiers:

TierContents & keyDurability
Byte replica Ciphertext at rest — encb:v1: for E2EE, raw bytes for plaintext (identity, §5); mirrors exactly what the server holds. Keyed by block id (<ws>/<id>, the Storage path). The offline guarantee: eagerly replicated, persist()-backed, evicted only under genuine quota pressure and only after the DB. A locked E2EE device can still replicate it (it stores ciphertext it can't yet read) — offline-prep before unlock, and plaintext never touches durable storage.
Display cache Decrypted bytes ready for <img>, derived from the replica at render time. Keyed <ws>/<id>/<hash> (the hash gives integrity / self-invalidation, §5.1). Genuinely throwaway — re-derive from the replica with no network. Only E2EE needs it as a separate tier; for a plaintext workspace the replica bytes are the display bytes, so it collapses to one store.

This dissolves the durability question: don't try to make the cache durable — put a durable replica underneath it. Priority under storage pressure is DB replica > byte replica > display cache. The lever that exempts the durable tiers from best-effort eviction is navigator.storage.persist(), not the store choice: the Cache API is not the HTTP cache — it sits under the same Storage-Standard per-origin quota as IndexedDB and OPFS and is evicted on identical terms, so Cache-vs-OPFS buys no durability, only persist() does (spun out as its own investigation). The display cache is preserved by default for full offline operation; eviction is graceful degradation (a re-fetch + decrypt round-trip, never data loss), and a real storage contest is always won by the DB.

  • Backing store: OPFS (following the in-thread decryption choice, §7.3). The store was always secondary to the decryption boundary, not dictated by it; with in-thread decryption OPFS files are the natural fit (read the file → object URL → <img>). (The rejected SW path would have used the Cache API as its SW-native store instead — same bytes, different shell.) User-scoping (assets:<user_id>) is the access-isolation boundary (§7); workspace-scoping makes leave / lock evict only the affected bytes. Because assets are immutable write-once — an edit is a new block id at a fresh path (§5.1/§10.1) — a given block id's replica bytes never change, so the display cache carries the hash in its key (self-invalidating if a buggy path ever produced a mismatch), and a fetched byte is accepted only if it matches the block's expected hash (read directly from the decrypted block in-thread); never serve <ws>/<id> bytes for a different hash.
  • Demand beats prefetch. The render path and the background replicator call the same fetch primitive (fetch ciphertext → verify hash → store in replica) over one in-flight map, so a demand fetch and a queued backlog item for the same asset coalesce into a single download — opening a queued image satisfies its backlog entry instead of racing it. Two priority lanes ride that primitive: a high-priority demand lane (the user is looking at it now) that jumps ahead, and a throttled, idle, low-concurrency backlog lane (the drip) deliberately slow enough never to saturate the link or starve a demand fetch. So opening not-yet-replicated media triggers an immediate high-priority fetch (a brief spinner if online); the only hard failure is offline AND not-yet-replicated.
  • Eager replication is budgeted and scoped, or it backfires. Metadata is cheap to fully replicate; media can be gigabytes, and unbounded eager replication would blow the origin quota and trigger the very eviction it is meant to prevent. So the down-lane (§9) is bounded: (a) scoped to active / opened workspaces (don't pull cold workspaces' bytes — the same reasoning as the "don't touch unopened workspaces" sync-flood lesson); (b) budget-capped — replicate eagerly up to a budget, beyond which the long tail stays lazy-re-fetchable rather than quota-thrashing; (c) polite — idle / low priority, viewport-proximity first, and metered / mobile aware (wifi-only or a user toggle). Reads are direct RLS-gated Storage GETs (only uploads go through the §10.1 function), so the down-lane is plain egress.
  • Eviction: the display cache and any over-budget replica bytes carry a soft LRU cap (≈500 MB default); an evicted entry round-trips on next view (replica → re-fetch ciphertext; display → re-derive locally from the replica). Un-uploaded replica bytes are exempt — they are the only copy of those bytes anywhere (§9).
  • Destroying local data: the replica, display cache, and §9 byte-replication store are ordinary local storage, removed wholesale by the coarse clear-all (§7.2) — no per-store wipe participation. Authorization is separate: on per-workspace revoke/leave, purge that workspace's cache prefix (a serving-access control, not destruction) — and because a purge can race or miss, bound cached-entry authority with a short freshness TTL after which the in-thread resolver re-validates via the RLS-gated Storage GET (§7.3); for E2EE, WK-absence counts as locked. The TTL revalidation, not the purge, is what closes the plaintext-removal window.
  • Optimistic local-first write: on paste, the encoded bytes are written straight to the durable replica (the §9 up-lane's source, never the evictable display cache) and the display cache is warmed. The image renders immediately from local bytes; the upload drains in the background. Whether the bytes have uploaded yet is a per-device fact, read from the local up-lane — see the next point. Offline it simply stays pending and still renders on this device.
Upload status is per-device; do not put it in synced block properties. An earlier draft stored status: 'uploading' | 'synced' on the asset block. But block properties serialize to the single synced (and, for E2EE, encrypted) properties_json column — so a flip to 'synced' would replicate to every device and clobber a peer for which the bytes are not local. "Have the bytes uploaded?" and "are the bytes cached here?" are both per-device questions answered by the local upload queue (§9) / local cache, keyed by asset_block_id — never by a synced flag. The synced block's mere existence is the only cross-device signal needed.
The LRU display cache must not be the only home for un-uploaded bytes. The cache is evictable (and capped); if a paste happens offline, or the cap is hit before the queue drains, an evicted entry would leave a synced asset block stuck in uploading forever with no bytes to send. So pending bytes are owned by the durable replica store (§8/§9), which is not subject to the display cache's LRU. After upload they stay there as the replica (§8) — the display cache is only the decrypted layer derived on top, never the sole home for the bytes. Two corollaries: (1) the durable-replica write at capture must actually be durable — if it fails, or persist() is denied so the write lands in best-effort storage, capture fails with a clear error ("can't attach — local storage is full or not durable") rather than committing metadata over evictable bytes; the §11 staged-before-tx ordering catches a thrown stage, not a silently non-durable one. (2) Because un-uploaded bytes are the only copy (and thus exempt from eviction), the budget must constrain intake, not retention — when pending bytes would exceed a safety budget, apply backpressure at paste ("uploads pending — reconnect or free space") rather than accepting bytes you might have to evict.

9. The byte-replication subsystem (up-lane + down-lane)

Block metadata rides PowerSync's ps_crud upload queue. Bytes go through the guarded upload function (§10.1) on a different path with no such queue, so they need their own durable one. Crucially the queue owns the encoded bytes, not just a pointer: an IndexedDB/OPFS record of { user_id, asset_block_id, workspace_id, status, idempotencyToken, encodedBytes } (the encb:v1: payload for E2EE, raw bytes otherwise), held in a store that is not the evictable display cache, so a pending upload can never lose its source bytes to LRU eviction or an offline window (the §8 hazard). Two record fields are load-bearing and were under-specified earlier: (a) the idempotencyToken (§10.1) must be persisted in the record and threaded through BlobStore.put, so a retry after a reload re-presents the original token — otherwise the already-written object looks like a no-token/different overwrite and becomes a permanent failure; (b) the user_id namespaces the record (and every cancel / drain key), because the store is shared across accounts in the profile but draining is per-user — each upload rides the active user's session/token, so without the namespace a drain could retry account A's entry under B's session, uploading A's bytes with B's credential or against the wrong workspace membership. The status field has a lifecycle that closes the metadata-commit / byte-stage ordering gap (§11): staged (durable but not drainable) → pending (drainable) → uploading → uploaded (entry cleared; bytes retained in the replica, §8). Capture writes the record as staged before the metadata repo.tx, so a committed asset block always has its source bytes already on disk; the post-commit step flips it to pending and arms the drain. The drain acts only on pending+ records, so the "block still exists?" pre-PUT drop applies only after the commit (a post-commit deletion / undo) and never to the legitimate pre-commit window. A flip that never lands (crash or throw between commit and flip) is recovered by a boot-time reconciler — but its commit-vs-rollback test must not misread a not-yet-hydrated DB: it runs after PowerSync reports initial sync settled, so an absent block means truly-absent (not merely not-yet-loaded — otherwise it would reap the only durable copy of a committed-and-synced block's un-uploaded bytes). It then promotes any staged record whose asset block is present, and reaps one whose block is absent past that checkpoint. To shrink the inferred set to the genuine crash window, a caught tx rollback also stamps the record staged → reapable directly, and records carry a session/generation stamp so a reaper never races an in-flight capture. Entries drain (to the §10.1 function) when connectivity returns, retrying with backoff; on a confirmed upload the queue entry is deleted, but the bytes remain in the durable replica (§8, budget-bounded) — full replication keeps them on disk for offline rather than downgrading to an evictable display copy. This is new infra the pre-E2EE proposal under-specified; E2EE doesn't change it, but the design surfaces it clearly. The queue's local status is also the per-device source of truth for the "bytes pending here?" UI (§8) — there is no synced status flag. Cross-device arrival is naturally eventually-consistent: a peer that has the block but not yet the bytes shows the "metadata present, bytes pending" state until the upload from the origin device completes.

The down-lane: background replication. The queue above is the up-lane — local bytes the server doesn't have yet. Its mirror is the down-lane: a background reconciler that walks the media blocks in the local DB for active workspaces and, for any whose replica bytes are absent, enqueues a fetch → hash-verify → store (the §8 budget / scope / politeness rules apply). The two lanes are halves of one byte-replication subsystem over a single durable byte store — the §8 replica — paralleling what PowerSync does for rows: up-lane drains local→server, down-lane reconciles server→local, and the demand lane (§8) lets a render jump the backlog. So the §9 "durable store" and the §8 "replica" are the same store, seen from the two lanes; the up-lane status record is the per-device upload-tracking over that store, not a second copy of the bytes.

Undo / delete must compensate the byte lane. §11 puts the asset-block create + the !((id)) insert in one repo.tx so they share an undo bucket — but that only makes the DB half atomic. The bytes (queue entry + any already-PUT object) are not transactional. So: (1) the drain step re-checks the block still exists (and is not deleted) before PUT, and drops the entry otherwise; (2) undo tombstones the queue entry (undone, keyed by (user_id, asset_block_id)) rather than hard-cancelling. A snapshot-then-act compensator has a TOCTOU hole — the in-flight PUT can confirm between the undo's status read and the cancel, so it sees uploading, skips the delete, and orphans the just-uploaded object. With a tombstone, both sides consult it and whichever of {upload-confirm, undo} lands second owns cleanup: a confirm that finds the entry undone enqueues the object delete itself. Redo re-stages from the retained replica bytes (the replica is not dropped on undo), re-uploading if the object was already deleted. Without this, undo leaves an orphaned Storage object and a queue entry that resurrects bytes for a deleted block. (The §16 orphan-object sweep covers it only as a slow backstop.)
A synced asset block can outlive its only bytes — the server-side backstop. The byte stores are ordinary local storage, so the coarse clear-all (§7.2) removes the upload queue with everything else — there is no per-store wipe participation to coordinate, and no raw-bytes-survive-a-wipe hazard (plaintext bytes go with the DB). But that destruction model, like a lost / reinstalled device or a crash, means a device can lose its un-uploaded bytes while the asset block's metadata already synced (PowerSync was up; the byte upload wasn't) — leaving the server with a block whose <ws>/<asset_block_id> object can never materialize, a broken embed on peers that did nothing wrong. There is no reliable client-side fix at that moment (the origin device is gone or offline) — but the handling must not be a destructive server-side reap of the block. Block deletion is a subtree-aware soft-delete (src/data/mutators.ts), so reaping the asset block would cascade-delete the user-authored caption / alt-text / transcript children — turning a missing image into lost text — and elapsed time alone can't distinguish "author offline, still intends to upload" from "author gone forever". So instead: (a) while a staged /pending up-lane entry exists, the origin refreshes a tiny server-side upload-intent marker (a non-byte write), so a device returning from offline re-asserts intent before any window matters; (b) a peer that has the block but not the object runs a bounded down-lane retry, then surfaces a terminal "image not yet uploaded by its author" state (not an indefinite spinner, and it stops re-hitting Storage with 404s); (c) the embed self-heals if the origin ever uploads. Automatic destruction is reserved for orphan objects (bytes with no block — safe to delete); an orphan block is never auto-deleted (its children are real content), so the only unrecoverable case — origin gone before upload — is honestly shown as unavailable, never papered over by deleting user content.

10. Storage layout & RLS

One private bucket, attachments. Object path <workspace_id>/<asset_block_id> — block-id-addressed, NOT content-hash-addressed. For an E2EE workspace the object is the encb:v1: ciphertext; for a plaintext workspace it is the raw bytes (identity case, §5).

Why not content-addressed paths (and what we keep instead). An earlier (pre-E2EE) sketch keyed everything off sha256(bytes) — path, cache, dedup — which is clean because content-addressing makes uploads idempotent and dedup free. E2EE removes the hash from the two server-visible path, primarily for one reason:
  • Content oracle. A path <ws>/<sha256-of-plaintext> lets the untrusted server confirm "does this workspace contain this known file?" by hashing a candidate and probing for the object — a confidentiality break for known files.
  • (Secondary) share-join shape. A block-id path keeps a future share-RLS join (object → block) trivially possible; a raw-hash path would need a separate mapping. Minor — subtree sharing is unshipped and not a design driver (§12) — but noted.
Hashing itself is not lost — it stays everywhere it's client-local: the cache key (<ws>/<id>/<hash>, §8), workspace-wide dedup (§11), and the integrity/replay check (§5.1); and the hash lives in the encrypted properties_json so the server never sees it. Only the server-visible path and the block id drop it. The direct cost is that idempotent-by-content uploads are gone (the server can't return "already have these bytes" for a re-PUT under a new id) — but workspace-wide dedup reuses the existing block + object, so the common "same image again" case still uploads exactly once (§11).

The alternative that fixes that: a keyed hash. Path <ws>/HMAC(K_id, sha256-of-plaintext) restores content-addressing — deterministic within a workspace (dedup + idempotent PUT return), unguessable to the server (no oracle), and distinct across workspaces (no correlation). (K_id is a separate HMAC-capable subkey derived from the WK material via HKDF at import time — not the stored WK handle, which is a non-extractable AES-GCM key WebCrypto won't let you sign with; deriving the subkey while the raw bytes are briefly available at paste is the only window.) It is genuinely the better-scaling design: it eliminates the dedup races and duplicate-object / re-upload wrinkles entirely, and those races get more likely as device count and concurrency grow, not less. Its one cost — the path no longer carries the block id, so a future share-RLS join would need a separate object_key → block_id mapping — is now minor, since subtree sharing is unshipped and no longer a design driver (§12). So the fork is mostly block-id paths (simplest to ship now) vs keyed-hash paths (idempotent uploads + dedup convergence that scale). v1 may ship block-id paths to move faster, but that is a deliberate design call to revisit on merits, explicitly not settled by current fleet size — adopting keyed-hash later is a storage re-key + mapping backfill, and that migration is the only place the small fleet legitimately makes it cheap. Recorded so it isn't re-discovered.

Storage reads are RLS-gated by workspace membership on the path's first segment. Storage writes are NOT direct — they go through the guarded upload path (§10.1); the bucket grants no client insert/update:

using (
  bucket_id = 'attachments'
  and private.is_workspace_member(
        (storage.foldername(name))[1],   -- <workspace_id>

);
-- NO insert/update policy for `authenticated` on the attachments bucket:
-- direct client writes are denied so they can't bypass the §10.1 byte guard.
-- The service-role Edge Function (which receives + inspects the body) is the
-- ONLY writer. Not a signed upload URL: uploadToSignedUrl writes direct to
-- Storage without the function seeing bytes, defeating the encb:v1: check (§10.1).

-- DELETE is allowed directly for workspace writers — delete removes an object,
-- there is no body to inspect, so the §10.1 byte guard doesn't apply and a
-- writer-only RLS policy is sufficient (and is REQUIRED: without it BlobStore.delete()
-- is default-denied, so §9 undo / §16 GC can't reap already-uploaded objects).
create policy "attachments delete by workspace writer"
on storage.objects for delete to authenticated
using (
  bucket_id = 'attachments'
  and private.is_workspace_writer(
        (storage.foldername(name))[1],   -- <workspace_id>
        (auth.uid())::text)
);

So the mutation split is: writes go through the body-inspecting Edge Function (§10.1); deletes are a direct, writer-only RLS delete (no byte to inspect); reads are direct, member-gated RLS. BlobStore.delete() (used by §9 undo and §16 GC) uses that delete policy — without it, already-uploaded objects would survive undo/delete as orphans.

The private.* grant is not free — verify it, don't assume parity with blocks. private.is_workspace_member / is_workspace_writer exist and are SECURITY DEFINER, but the migrations contain no explicit GRANT USAGE ON SCHEMA private / GRANT EXECUTE to the app roles — the blocks RLS works via defaults and the fact that the policy and the table live in public. A storage.objects policy runs in a different ownership/role context (the table is owned by supabase_storage_admin, in the storage schema), so "the same functions blocks uses" is not automatically callable there. Treat the schema-USAGE + EXECUTE grants (or a thin public wrapper) as a first-class, tested step of the bucket migration — a wrong assumption yields a policy that errors at query time or silently denies all reads.

For an E2EE workspace the blob is already ciphertext, so RLS here is defense-in-depth — it stops a non-member from hoarding ciphertext against a future key compromise, and avoids leaking object existence. For a plaintext workspace it is the actual access boundary.

10.1 Server-side ciphertext enforcement for E2EE bytes

RLS is not the ciphertext guarantee. For blocks, a server trigger rejects a plaintext write to an E2EE workspace (is_enc_v1_envelope shape-check), so "the server only ever sees ciphertext" does not rest solely on the client upload hook. The naïve analog — "storage.objects trigger that checks the bytes" — does not work: Supabase Storage object policies/triggers see object metadata (path, owner, size), not the body bytes, so they cannot inspect the envelope. A check that can read bytes has to sit in code on the write path — and that code only enforces anything if it is the only write path.

Decision: a guarded upload service is the sole writer; direct client writes are denied. Attachments must match the block guarantee — the server enforces ciphertext, not by client convention. So: (1) the attachments bucket grants no insert/update to authenticated (§10); and (2) all uploads go through an upload-mediating Edge Function that receives and inspects the body bytes itself and then writes them with service-role. Per request it:
  1. Authenticates the caller's JWT and authorizes the write — checks private.is_workspace_writer(<workspace_id>, caller) and that the path's workspace/asset is the intended target — before using the bypass-RLS credential. The service-role write must re-impose the authorization that direct RLS would have enforced; otherwise a read-only or recently-removed member who can invoke the function could create/replace <ws>/<asset> bytes that blocks_write/storage policies deny.
  2. Inspects the bytes — for an object under an E2EE-workspace prefix, rejects bodies not beginning with the encb:v1: magic + the minimum nonce(12)+tag(16) length (the binary analog of is_enc_v1_envelope), with the same permanent-failure discipline so a stale client quarantines rather than retries forever. Shape-only (the server has no key), sufficient to keep bare plaintext out.
  3. Enforces write-once, idempotently. Asset objects are immutable; an edit/replacement is a new asset block at a fresh path (§5.1), so a stale queued upload can never overwrite a newer object and strand the asset on a hash mismatch. The guard rejects an overwrite by different content, but must treat a re-PUT of the same content as success, not failure: if the function wrote the object but the response was lost before §9 deleted the durable queue entry, the retry would otherwise turn a completed upload into a permanent queue error. So the client sends an idempotency token, the function stores it as object metadata, and on a PUT to an existing path: same token → 200 (already done); different token → reject. The token is a per-asset random value (a UUID minted at capture, carried by the queue entry so retries present the same one) — not HMAC(WK, …): the WK is a non-extractable AES-GCM CryptoKey (workspaceKey.ts / keyStore.ts) that WebCrypto allows only to encrypt/decrypt, never to sign/HMAC or export, so a keyed digest can't be computed from the stored handle. A random token needs no key and is opaque to the server anyway. (Under immutable ids only the same-token retry is legitimate; a different token to an existing path signals a bug/attack.)
Not a signed-upload token. Minting a one-shot signed upload URL would let the client uploadToSignedUrl the body directly to Storage, which Supabase accepts without the function seeing the bytes — defeating the magic check. The body must flow through the function. The §9 byte-upload queue therefore drains to this function, not to Storage directly. Cost: every upload is one function invocation that streams the body (latency + per-call cost); acceptable for the guarantee, and reads stay direct/RLS-gated (the hot path). Lands in phase 2 (§15) with the bucket + RLS migration.
Doesn't foreclose a future share-join. Because the path carries asset_block_id, storage RLS could later widen from "membership" to "membership or a share covers this block" (join storage.objects → blocks) if subtree sharing is ever built (§12). We neither build nor design around that now — it is incidental option value from the natural block-id addressing, not a requirement.

Per the project's Supabase rule, the bucket + its RLS land in one migration with storage.objects policies enabled; no public table is created without RLS.

11. The renderer & capture surfaces

Packaged as one attachments plugin, mirroring the video-player plugin's facet wiring:

  • MediaBlockRenderer via blockRenderersFacet, with canRender = ({block}) => block.hasType('media') and a priority above the default. Branches on mime: image → the existing MarkdownImage lightbox fed the object URL the in-thread resolver produces (§7.3); later, PDF / audio / video / generic file-chip.
  • Markdown img override via markdownExtensionsFacet (alongside defaultMarkdownExtension.ts) for any inline image whose src is an asset reference (an app-internal sentinel, not an SW-intercepted URL) — route it through the in-thread resolver (§7.3) to an object URL and delegate to the lightbox.
  • Capture surfaces: a CodeMirror domEventHandlers extension via codeMirrorExtensionsFacet intercepting paste / drop with clipboardData.files / dataTransfer.files; a block-shell paste/drop arm parallel to today's text/plain handlers in CodeMirrorContentRenderer.tsx / DefaultBlockRenderer.tsx; and a /image command + file-picker action through the existing action/shortcut system.
  • On capture: client-side mime/size guard → hash → workspace-wide best-effort dedup lookup (see the caveat below) → otherwise create the asset block (placement isn't load-bearing — a child of the containing block is the simple default), stamping the type with repo.addTypeInTx(tx, id, 'media', …) (the in-tx variant — the standalone block.addType wouldn't compose into the single tx; and the media TypeContribution must be registered via typesFacet first, §3, or this throws). The asset-block create + the !((id)) insert go in one repo.tx({scope: ChangeScope.BlockDefault}) so they share an undo bucket. The durable byte record is staged (non-drainable) BEFORE that tx, and the drain is armed only AFTER it resolves — not inside it (see the box below).
Stage the byte record durably before the metadata tx; gate its drain on the commit. repo.tx only makes the SQLite/PowerSync writes atomic — an IDB/OPFS byte-queue write is an external side effect that neither joins that tx nor rolls back with it. Two failure windows have to close at once. (a) If the byte write happened after the commit and then failed (quota, private-mode, a store error), the metadata is already committed and syncs to peers with no durable source bytes, no per-device pending status, and nothing for the §9 compensator to cancel — a permanently un-uploadable asset on every peer. (b) If the enqueue instead kicked off a drain before the commit is visible, the §9 "block still exists?" pre-PUT re-check would drop the bytes as a missing block. The resolution closes both: write the durable byte record first in a non-drainable staged state (§9), then run repo.tx, then flip staged → pending and arm the drain. Because the bytes are staged before the commit, a committed asset block always has its source bytes already on disk — window (a) is gone (a failed stage aborts capture before any metadata commits, attaching nothing); because staged is not drainable, the drain can't race ahead of the commit — window (b) is gone. The post-commit flip can itself fail, but that is recoverable, not lossy: the durable staged record survives, and a boot-time reconciler (§9) promotes any staged record whose asset block did commit. Undo still works via the §9 compensator — undo of the (committed) asset block cancels the queue entry and deletes any already-uploaded object.
Dedup is workspace-wide. Reuse any same-hash asset block in the workspace — the embed is just a !((id)) to it, wherever that asset block lives. This is the natural model and restores the content-addressing win the block-id path otherwise loses: a reused asset keeps the same block id and the same <ws>/<asset_block_id> object, so "paste the same image again" creates no new block and no new upload — only a fresh paste with no workspace match uploads. A deduped asset therefore has multiple embed referrers, so its lifetime rides the existing block-reference machinery (§3): GC the asset block + its object when the referrer count reaches zero, not when any single embed is removed. But never reap on the live zero-crossing — moving an embed (cut then paste), or a cross-device add that hasn't synced yet, transiently drops the count to zero and would reap an asset about to be re-referenced. Reap only assets that stay referrer-less past a grace window, behind the same deferred sweep as orphan objects (§16), preserving any child content. No new refcount system — just no synchronous reap. We deliberately do not scope dedup to a "containing subtree": the data model isn't strictly tree-shaped, and subtree sharing is unshipped (§12), so scoping dedup to it would pay for a feature that doesn't exist and doesn't map cleanly onto today's blocks. If subtree sharing is ever built, retargeting a workspace-wide asset into a shared scope (e.g. copy under the shared root at share time) is that feature's job; the block-id path leaves it possible.
Dedup is best-effort, not an invariant. The lookup reads local synced state, so two devices (or two tabs, or an offline-then-reconnect) pasting the same image concurrently both miss, both create an asset block, and — because the object path is <ws>/<asset_block_id> — produce two Storage objects for identical bytes. The server can't break the tie (randomized GCM ⇒ no ciphertext dedup; hash is inside encrypted properties_json). So: state dedup as best-effort and make GC tolerate multiple blocks per hash. The tempting "make the id a deterministic function of (workspace_id, hash) so pastes converge" is unsafe as stated under E2EE — an id derived from the plaintext hash is a content oracle (the server could confirm a known image's presence by id, or correlate the same image across workspaces). If determinism is wanted, derive the id with a workspace HMAC subkey (HMAC(K_id, hash), K_id derived from WK material at import — see §10; the stored AES-GCM WK handle itself can't sign) so it's workspace-local and reveals nothing to the server — otherwise keep random ids + tolerate duplicates.

12. Subtree sharing (deferred — not a design driver)

Subtree sharing (shares-and-api-tokens.html) is not shipped, and it doesn't map cleanly onto the current data model — the app isn't strictly tree-shaped anymore, so "share this subtree" is itself an unsolved design a future effort would have to work out. So this design does not bend around it: dedup is workspace-wide (§11), asset placement isn't share-driven, and we add no share machinery now. We also don't foreclose it — the block-id storage path keeps a future "membership or a share covers this block" RLS join possible (§10), and retargeting a workspace-wide asset into a shared scope (e.g. copy under the shared root at share time) would be that feature's job, not ours.

E2EE + subtree sharing is doubly deferred. Even once sharing exists, an E2EE subtree-share recipient needs a key — but there is one WK per whole workspace and no per-subtree key; that needs the key hierarchy the e2ee doc defers (e2ee §13). So encrypted subtree sharing is a non-goal for the foreseeable future and media just inherits that boundary (consistent with e2ee §15 #2; E2EE sharing stays out-of-band whole-workspace WK transfer). Don't special-case it.

13. The public-sharing escape hatch

We still reserve properties.publicUrl on asset blocks and a render-audience flag on the block context ('workspace' member / 'shared' subtree recipient / 'public' logged-out). But the implementation for a public, logged-out viewer differs by mode:

ModePublic-share implementation
Plaintext workspace A public bucket copy (or long-lived signed URL) written to properties.publicUrl; the renderer prefers it when audience is 'public'. The original signed-URL story works here.
E2EE workspace Either export-to-plaintext (decrypt + re-upload to a public bucket — the asset deliberately leaves E2EE, an explicit, consensual exit, never silent) or key-in-fragment (app/share#<per-asset-key>; the client decrypts from the URL fragment, which the browser never sends to the server). "Anyone with the link" is exactly key-in-link semantics.
The fragment key MUST be a fresh per-asset key, NEVER the workspace key. The §5.1 crypto seals bytes with the WK directly — there is no per-asset key in the v1 design. So a naïve "put the decrypt key in the fragment" would put the WK in a URL, exposing the entire workspace, not one image. Key-in-fragment therefore requires its own step: generate a one-off key, re-encrypt the bytes under it, upload that to a public bucket, and put only that per-asset key in the fragment. The WK never leaves the device (preserving the e2ee §5 invariant).
Document the fragment's leak surface and irrevocability. "Not sent in the HTTP request" is necessary but not sufficient: a fragment persists in history, is readable as location.hash by any script on the share page, and can ride a Referer or a third-party resource off-page. A public share page that loads a key into the DOM is a far softer target than the authenticated app (which is CSP-hardened). And it is irrevocable — once the link leaks, the asset leaks, permanently, short of re-encrypting under a new key. Both escape hatches are vNext; neither ships without these stated as accepted, explicit tradeoffs.

Reserving the property + the audience flag costs almost nothing now and avoids a content migration later. The renderer's URL resolver branches on both workspace mode and render audience.

14. Vendor portability (the BlobStore seam)

All object-store access goes behind a ~50-line interface:

interface BlobStore {
  // No `hash` parameter: this seam drains to the server (the §10.1 Edge Function), and the
  // plaintext sha256 must never reach the origin — it is the §10 content oracle. The hash lives
  // ONLY in client-local paths (cache key §8, replay check §5.1, dedup §11),
  // sourced from the encrypted properties_json (in-thread, the resolver holds the decrypted
  // block), never the wire. idempotencyToken (§9/§10.1) IS passed: it is persisted in the queue
  // record so a post-reload retry re-presents it; same token on an existing object → success.
  put(workspaceId, assetBlockId, idempotencyToken, bytes): Promise<void>
  // The fetch URL (RLS-gated Storage GET target) is CLEAN — no hash on the wire.
  // The expected hash (cache key §8 + replay check §5.1) comes from block.properties.hash
  // (the in-thread resolver holds the decrypted block), never a URL param — a ?h= would
  // leak the plaintext sha256 to the origin (the §10 content oracle).
  url(workspaceId, assetBlockId): string
  delete(workspaceId, assetBlockId): Promise<void>
}

v1 backs it with Supabase Storage. E2EE actually increases portability: because the server is a dumb encrypted-blob store and never mediates bytes, swapping to R2 / S3 / B2 later is a bytes-copy plus re-pointing one module — no auth or schema rework. We do not pre-migrate; we keep the seam clean and pull the trigger if egress cost or another driver appears. (A Worker-fronted R2 from day one is defensible only if a Worker is being introduced for other reasons; otherwise it is permanent operational surface for no present benefit.)

16. Open questions

  • Multi-account profile (decided shape, needs care): in-thread, each tab already owns its own Supabase session, so there is no shared-SW session slot to mis-key. The residual work is scoping the OPFS display cache by user_id and preserving the keyStore's (user_id, workspace_id) keying under an account switch, so account A's cache / keys can never serve account B (§7.3).
  • Max inline size + large-media story: a v1 cap (e.g. 10 MB) with a clear error, deferring resumable/tus upload to vNext. Where exactly is the cap enforced?
  • Asset block deletion → byte GC, and the two orphan directions: deleting an asset block (once its last !((id)) referrer is gone — workspace-wide dedup means one asset can have many, §11) ⇒ delete the object (storage cascade or an explicit step); dangling embeds render the existing "unresolved embed" state. Beyond the undo case (handled by the §9 byte-lane compensator), a vNext orphan sweep covers both directions, but they are not symmetric: orphan objects (a Storage object with no block — e.g. an interrupted delete) are auto-deletable; orphan blocks (an asset block whose object never materialized — e.g. an upload that never completed before the origin device was cleared / lost / reinstalled) are never auto-deleted — their caption/transcript children are real content and a block delete cascades (§9), and elapsed time can't tell an offline-but-intending author from a gone one. The block direction is handled by the §9 upload-intent marker + the peer's bounded "not yet uploaded by its author" terminal state, with removal left to the user — not a timed server reap.

18. Design notes — where the complexity lives, and the seams that contain it

This design went through several adversarial review passes; the findings clustered in a few places, and the pattern is worth recording so the implementation leans on the seams that keep it simple.

  • Reuse the proven block lane; don't re-derive a parallel one. "An attachment is a block" (§3) is load-bearing: metadata sync, encryption (transform.ts), undo, and permissions all come for free by riding the existing block path. The fragile parts are exactly where the byte lane diverges from how blocks already work — blocks have ps_crud (→ we need a §9 queue), a server ciphertext trigger (→ a §10.1 guard), and PowerSync monotonicity (→ a §5.1 hash check). Each divergence had to re-solve a problem blocks already solved. Prefer mirroring the block mechanism over inventing a byte-specific one wherever the choice exists.
  • Funnel URL + mode + hash + cache-key through one asset-resolver seam. Many review findings were not "the design is wrong" but "a fact didn't propagate" — the base-path prefix, the expected-hash source (SW mirror), the mode decision, and the cache key are all facets of one operation ("given an asset block + render context, produce displayable bytes"). Implement that as a single named function so those parameters can't drift apart; scattering them across call sites is what manufactured the drift.
  • Content-addressing is confined to client-local use under E2EE (§10): cache, dedup, and integrity keep the hash; the server-visible storage path and block id do not (oracle + share-join). The dedup/duplicate-object friction is the price of block-id paths; a keyed hash (HMAC(K_id, …) under a WK-derived subkey, §10) is the better-scaling alternative (idempotent uploads, no dedup races) at the cost of an object→block mapping for shares. It's a real design fork to decide on merits — not on current fleet size; fleet size bears only on how cheap the migration to it is (§10).
  • Decided: in-thread decryption (§7.3). The single biggest source of genuine (not cosmetic) complexity was putting decryption in the service worker — a context that can't read app state (→ IDB mode-pin + per-asset hash mirror) and is shared across tabs/accounts and respawned arbitrarily (→ per-clientId sessions, controller-readiness gating). Those are ~5 coordination mechanisms bought for one benefit — a uniform mode-agnostic <img src> — that <img> doesn't actually need, since it takes an object URL. In-thread trades all of it for a mode-aware renderer + object-URL lifecycle, so that is the adopted approach. §7.1–§7.2 are retained as the rejected-alternative analysis; §8–§10 are boundary-agnostic. (General lesson: a SW earns its tax only when the consumer needs a real URL, not a blob — see the user-plugin /module/<id> case.)
