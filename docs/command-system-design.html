Commands as a core foundation

Upgrading the action system into a typed, observable command substrate — so behaviour like navigation can be intercepted, replaced, and composed through one generic seam instead of a new bespoke one each time.

1The thesis

The navigation work proved a pattern we keep repeating: to make one behaviour extensible (observe it, wrap it, veto it, replace it), we hand-build a bespoke seam for it. Navigation got navigationVerb; paste got pasteDecisionVerb. Each is correct in isolation, but each is a new vocabulary the rest of the system doesn't share.

The higher-leverage move is to make two things core foundations that any behaviour can ride on, instead of re-deriving the seam per feature:

  • Commands — the imperative channel: "do X, give me a typed result," with one dispatch pipeline that every caller and every interceptor shares.
  • Events — the reactive channel: "X happened," that observers subscribe to without being able to veto. (Scoped separately; see §7 and the event-bus investigation task.)
Core claim Actions are already most of a command system — they have identity, binding, context, and a dispatch path. What's missing (a typed result channel, typed arguments, invocation-time middleware, one lifecycle) is exactly what we hand-rolled into the verb facet. Closing that gap turns "actions" into "commands," and — once typed handles and the middleware slots exist — lets the bespoke verb seams retire into the generic bus. The claim is that this is higher-leverage than building the next bespoke seam, because it's paid once for all behaviours; §1.1 weighs it against the cheaper alternatives.

1.1  Alternatives & the decision

The verb facet's own docstring argues most seams shouldn't pay a four-slot structure — "reach for this only when there's one output function." So full convergence is not self-evidently right; the realistic alternatives:

  • (A) G1-only + keep bespoke verbs. Give actions a typed result (G1) so imperative/surface callers stop fire-and-forgetting, and let any new interceptable behaviour keep using defineVerbFacet as-is. Cheapest; preserves the verb facet's selectivity. Cost: the "double thing" persists — every new wrappable behaviour is still a hand-wired seam, and there's no uniform place to observe/veto across behaviours.
  • (B) A typed command layer on the async face. G1 + defineCommand typed handles (G2) + handle-registered middleware (G3) — the verb facet's slots gaining identity, binding metadata, and deps resolution, without touching the keyboard coordinator. This already retires both bespoke verbs: navigate.apply is irreducibly async and paste awaits its decision today (§8), so all their middleware lives on the async face — and keyboard-triggered navigations are covered because the bound handlers call navigate() internally (the zoom actions do exactly this, with a comment saying so). What (B) does not buy: middleware on the chord face itself, a wildcard that sees keyboard dispatches, and the lifecycle/canDispatch unification (G4).
  • (C) Full convergence. (B) plus G4: one lifecycle whose sync face the coordinator dispatches through (§6), wildcard middleware on every dispatch. Costs (B) doesn't pay: coordinator surgery across five trigger kinds and their deliberate per-kind policies (§9 step 5); a second, permanently-degraded sync middleware executor (§5); and the extension-trust question (§10) becomes load-bearing — if wildcard middleware ends up first-party-only, the extensibility bet that justifies (C) shrinks with it.
The decision Recommendation: stage it. Land G1, then (B) — typed handles before middleware, so the middleware layer is born typed (§5) — and fold both verbs. Treat G4 and the sync-face middleware as a separately gated track: build them when a concrete lifecycle defect bites (the palette isVisible/canDispatch mismatch in G4 is the standing candidate) or chord-face middleware is actually demanded — not as a default continuation. What to decide, and when: G1 needs no decision; "(B) or stop at G1?" is owed before the typed-handles step; the G4 gate is its own later decision. Settle alongside (B): the DECLINE/VETOED outcome channels (§5) and the extension-trust stance (§10).

2Where we are today

We have two partial command systems that don't know about each other.

The action system — identity, binding, context; no result, no invocation middleware

An Action (src/shortcuts/types.ts) carries an id, a context, a handler, optional keyboard/pointer/gesture bindings, and the gating predicates isVisible / canDispatch. It is dispatched through four entry paths that have quietly diverged — plus a fifth that bypasses dispatch entirely:

EntryDeps fromContext must be active?canDispatch?Returns

dispatchActionWithDepscaller-suppliednogatedboolean (handled?)

The last row is worth dwelling on: the agent runtime bridge needed a result channel so badly that it skips dispatch altogether — it resolves the effective action, builds deps by hand, invokes the handler directly (no canDispatch, no deps validation, no sentinel coercion), and ships whatever came back over the wire untyped. The code itself flags the split too: runAction.ts notes that unifying the canDispatch gates "is tracked with the broader dispatch-lifecycle work." And the handler return type is the giveaway:

export type ActionHandlerResult = void | false | Promise<void>

There is no typed-payload return channel. The returns that do exist — the handled-boolean from the supplied-deps and pointer/gesture paths — carry only "did a candidate handle this?", the same axis as the false sentinel: a synchronous "not handled — try the next candidate" signal for the single-winner loop, not a value. So a caller that needs data back from a dispatch can't have it. Concretely, the PWA/share-target dispatcher (plugins/app-intents) has to fire-and-forget: 

const result = runActionById(actionId, new CustomEvent('app-intent'))
void Promise.resolve(result).catch(...)   // result is void; nothing to read

Rewriting action definitions is possible — actionTransformsFacet folds transforms over the action list at resolution time (effectiveActions.ts). Transforms can replace any field: vim-normal-mode and spatial-navigation wrap the handler (spatial's vertical-move transform also replaces description, a definition field — which matters for migration, §9); srs-rescheduling wraps the isVisible predicate and contributes three handler wrappers of its own (swipe-right, todo-cycle ×2); returning null unbinds. (Key rebinding is a separate pass — applyKeybindingOverrides reading keybindingOverridesFacet — not these plugins.) But every transform rewrites the definition, before any invocation. There is no way to intercept, observe, or veto a specific dispatch uniformly across the entry paths.

The verb facet — typed signature and invocation middleware; no identity, no binding

defineVerbFacet (src/facets/verbFacet.ts) is, in effect, a single-command bus. For one typed (input) => output it bundles four facets with the right combine algebra: impl (Replace, last-wins), decorators (Wrap, middleware), and before/after (Sum, observers) — plus an onError: 'rethrow' | 'fallback' policy that distinguishes effectful from pure verbs. run() executes before → decorators(impl)(input) → after.

This is precisely the invocation-time middleware the action system lacks — but it has no id you can dispatch by name, no binding, no context/deps resolution. You wire each verb by hand at its one call site.

The smell Navigation now maintains navigationVerb in parallel to the action system, purely to get a typed result and middleware that actions can't give it. That's the "double thing": a bespoke per-feature bus shadowing the generic dispatch path. Every future "make behaviour X extensible" request will reproduce it unless we close the gap once.

3The four gaps

What separates today's Action from a real command, ordered by importance of the gap (build order is §9's critical path):

G1 Typed result channel structural blocker

Dispatch returns void (or a boolean "handled"). A command must return a typed Result to its caller. This is the only structural blocker — it's what forced navigation into a bespoke verb and the agent bridge into direct handler invocation. Scope note: G1 alone buys "a dispatch returns a typed value" (already a real win — see surfaces); retiring navigationVerb also needs G2+G3 (typed handles, and somewhere for its decorators/observers to live). Caveat that the model must resolve: once dispatch returns a real Result, the false not-handled signal can no longer be the return value (a toggle→boolean command's legitimate false would be read as "not handled"). The decline signal becomes out-of-band — a distinct DECLINE sentinel, representable on both dispatch faces so it never gets coerced away (§5, §6).

G2 Typed arguments, distinct from deps

Handlers receive deps (injected services, context- or caller-resolved) and a raw trigger event — but no typed argument payload. To pass data you smuggle it through deps or parse it off the event. A command is Command<Args, Result>: Args (what the caller asks for) is separate from deps (what the runtime injects). Typing flows through module-level handles, not through string ids (§5).

G3 Invocation-time middleware spawned: S1

actionTransformsFacet rewrites definitions; the verb facet wraps invocations but only per-bespoke-verb. We want generic, dispatch-time before/decorator/after registered through a command's typed handle (plus an explicitly-untyped wildcard), applied uniformly to every dispatch. Handler-wrapping transform consumers migrate onto this; key-rebinding/unbinding and predicate/definition-field rewrites stay definition transforms.

G4 One dispatch lifecycle gated

Four coordinated entries (plus the bridge bypass) with different canDispatch behaviour and return shapes. Collapse to one skeleton — resolve → gate → middleware+handler — with per-trigger-kind policy preserved (the entries differ partly on purpose: modal shadowing, activation requirements, and error behaviour deliberately vary by caller type) and two execution faces at the tail (async dispatch / sync dispatchSync, §6). The motivating latent defect: the command palette filters by isVisible only, then dispatches through the path that ignores canDispatch — an action with a canDispatch guard but no isVisible is shown, then runs ungated. Gated in §9: built when that class of defect bites or chord-face middleware is demanded.

4The shape of a command (the equation)

The cleanest way to see the design: a command is the union of what each existing system already gives us.

Action gives

  • identity (id)
  • binding (key/pointer/gesture)
  • context + deps resolution
  • gating (canDispatch, isVisible)
  • registration via a facet

Command =

  • dispatch (cmd, args) → Result
  • typed Args + Result
  • handle-keyed middleware
  • one lifecycle
  • observable

Verb facet gives

  • typed (input) → output
  • impl Replace
  • decorator Wrap
  • before/after Sum
  • onError policy

Command ≈ Action's identity / binding / context  ∪  Verb's typed signature / middleware

Where the union frays Two seams the equation has to reconcile, not just add:
  • One command can carry one onError; navigation needs two. Navigation is not one verb — it's a pipeline of navigationIntentVerb (pure, onError:'fallback', gesture → NavigateInput) → navigationVerb (effectful, onError:'rethrow', request → NavigationResult). A single command with a single failure policy can't host both, so "the navigate command" is really two commands (navigate.resolve pure, navigate.apply effectful) or one command with an explicit two-phase internal structure. §8 resolves which.
  • The two halves resolve winners differently. The action coordinator picks one of several candidates per chord via a synchronous "run until not-false" loop; the verb facet has a single last-wins impl and an async run. The union must keep the sync multi-candidate resolution (§6) while gaining the verb's typed/middleware shape — they don't simply add.

5Proposed model

A sketch (illustrative, not final API) of where the pieces land:

// A command: a verb with identity, binding, and context/deps resolution.
// (In also threads today's ActionDispatch (activate/deactivate) — elided here.)
type In<Args, Ctx> = { args: Args; deps: DepsFor<Ctx>; trigger?: ActionTrigger }

// EXACTLY ONE of run / runSync — a discriminated pair. Both may DECLINE (the
// out-of-band "not handled" that frees `false` to be a real Result). runSync's
// extra contract: the DECLINE decision is returned in-tick — Promise<DECLINE>
// is forbidden; the Result may still be a promise the coordinator never awaits
// (§6). This mirrors how types.ts forbids Promise<false> but allows Promise<void>.
type Command<Args, Result, Ctx extends ActionContextType> = Base<Args, Result, Ctx> & (
  | { run:     (i: In<Args, Ctx>) => Promise<typeof DECLINE | Result> }       // async: dispatch() only
  | { runSync: (i: In<Args, Ctx>) => typeof DECLINE | MaybePromise<Result>    // chord/pointer-bindable
      defaultBinding?: ...; pointerBinding?: ...; gestureBinding?: ... }
)
interface Base<Args, Result, Ctx> {
  id: string
  context: Ctx
  // Deps-only, NOT (args) — deliberately. These run at chord-resolution time,
  // before any args exist. An arg-dependent guard returns DECLINE from
  // run/runSync instead — which the invoking surface can observe and turn into
  // a disabled state or a toast, not a silent no-op.
  canDispatch?: (deps: DepsFor<Ctx>) => boolean
  isVisible?: (deps: DepsFor<Ctx>) => boolean
}
// The type forbids binding a chord to a `run` (async-decision) command — the
// coordinator must learn DECLINE-or-handled in-tick to fall through / preventDefault.

// TWO dispatch entries, not one — §6 for why the sync one is mandatory. Typed
// via the handle; DECLINE stays representable end-to-end on BOTH faces, so no
// false→undefined-style coercion ever reappears.
dispatch(cmd: CommandHandle<Args, Result>, args: Args, opts?): Promise<typeof DECLINE | Result>
dispatchSync(cmd: CommandHandle<Args, Result>, args: Args, opts?): typeof DECLINE | MaybePromise<Result>  // decision in-tick; result not awaited here

Typed handles first; strings only at the edge

A by-string dispatch<Args, Result>(id: string, ...) — caller-asserted generics over an erased registry — is VS Code's executeCommand<T>, and it is unsound: the assertion is checked nowhere. The verb facet's core virtue is the opposite: contributions are typed through the module-level handle (navigationVerb.decorator(...) type-checks its input and result today). Commands keep that. defineCommand<Args, Result> returns a typed handle; dispatch and middleware registration go through the handle, typed end-to-end. A by-string entry (dispatchById(id, args?): Promise<unknown>) remains — cross-plugin callers dispatch by id, never by module import (the toggle-store / audit-B3 convention), and the palette, agent bridge, and app-intents are genuinely dynamic — but it is explicitly untyped at that edge, never a fake generic. This also fixes the build order: handles (G2) land before middleware (G3), so the middleware layer is born typed rather than string-keyed first and retyped later (§9).

Prior art cuts both ways and half of it lives in this repo: VS Code (string command registry, unsound executeCommand<T>, static args payloads in keybindings) and CodeMirror (sync (view) => boolean commands, fall-through by keymap precedence, typing by closure — exactly how src/utils/codemirror.ts uses StateCommand today). Both corroborate the two load-bearing choices here: an in-tick handled/declined decision on the bound path, and typing by handle/closure rather than by asserted string dispatch.

Middleware = the verb facet's slots, lifted to dispatch

We don't invent a new middleware mechanism — the verb facet's three slots, registered through a command's typed handle (or the untyped wildcard):

  • Replace — swap a command's run entirely (what navigationVerb.impl(...) does).
  • Wrap (decorator) — intercept a dispatch: observe args, transform them, call next or short-circuit.
  • Sum (before/after) — pure observers; after sees the outcome and cannot change it.

On the async dispatch face this is literally defineVerbFacet.run, onError policy included (effectful commands rethrow — no double-execution; pure ones may fall back). The sync face cannot reuse it: run is async-only and awaits the impl to build the outcome it hands to after, which the no-await dispatchSync contract forbids. So if/when the sync face is built (§9's gated step 5), it gets a separate, lighter fold with a degraded contract: sync-deciding middleware only — a decorator may have an async body, exactly like runSync itself, but its handled/declined/veto decision must not require an await — and after observes invocation, not settled outcome. A wildcard (mirroring WILDCARD_ACTION_ID) lets one contribution wrap all dispatches — a telemetry tap, a command-log — and necessarily sees args: unknown: one more reason the wildcard is observe/transform-shaped, not result-producing.

Veto needs its own outcome. The navigation verbs veto by returning null — legitimate only because NavigationResult | null has a none-arm by design. A generic decorator (and any wildcard, whose Result it cannot know) can't fabricate a Result — and returning DECLINE is the wrong veto on the sync path, because DECLINE means fall through: the next candidate runs, or no candidate handles and the chord is released to the browser un-preventDefaulted. A "veto" that un-suppresses the native default is worse than no veto. So the model needs a third out-of-band outcome — VETOED: consume the event, run nothing, surface per §10 — distinct from both DECLINE and any Result; or wildcard middleware is restricted to observe/transform and veto-by-return stays limited to commands whose Result carries a none-arm. Part of the decision owed alongside (B) (§1.1).

Decide before building: trust, ordering, cost Generalizing a wildcard wrap from a handful of trusted verbs to every command changes the threat and performance model — this app runs untrusted user-installed extension code that can import internals and contribute to these facets:
  • Trust. A wildcard decorator can veto every command by simply not calling next (a veto is legitimate, so it's indistinguishable from intended behaviour) and can read the args of every dispatch (block ids, content — a new data-egress surface). Decide: can extensions register wildcard/command middleware at all, or first-party only? How is a non-next veto contained vs. a throw (which the verb facet already isolates)?
  • Ordering. Cross-plugin middleware precedence must be deterministic — presumably the facet's precedence-asc-then-registration order — and the compose order between a surviving definition transform and a new dispatch middleware on the same command must be specified (today both could wrap the same action).
  • Cost. Every dispatch now pays two facet reads + a per-call decorator-fold allocation + (async face) a microtask. On the keyboard path this fires per keystroke; bound the wildcard count and keep middleware off the sync path unless it's sync-deciding.

Resolution is its own stage; the coordinator stays synchronous

"Which command wins this chord?" is not hidden inside dispatch — it is stage zero of the lifecycle. The coordinator matches a chord, gets an ordered candidate list from resolve(), and tries each in-tick: a DECLINE falls through to the next candidate; the first non-DECLINE is the winner, and preventDefault applies to it alone — today's run-until-not-false loop with the sentinel renamed. Middleware wraps the winning dispatch, not every attempt: a wildcard command-log sees one entry per chord and the per-dispatch cost doesn't multiply by candidate count; the trade-off is that dispatch middleware can't influence which candidate wins — that stays the resolver's job (definition transforms still can). Two prerequisites this stage surfaces: command ids must become globally unique by invariant — today uniqueness is convention, the real runtime key is context:id, and the edit.cm.* id prefixes exist precisely to dodge collisions, so id-keyed anything silently misses a same-name command in another context; and bound or palette-run commands take no caller args — they need Args ≅ void or a statically-bound args payload (VS Code keybindings carry static args JSON), which constrains the G2 adapter.

Surfaces become first-class

Several places already treat actions as commands, each working around the missing channel differently. app-intents and the header/toast callers fire by id and discard the result; the swipe menu and mobile bottom-nav dispatch with supplied deps and get only a handled-boolean; the agent runtime bridge went furthest — it bypasses dispatch entirely, invokes action.handler(...) directly with hand-built deps, and ships an untyped {returned} wire value, because it needs exactly the result channel actions don't have. On a real command foundation each becomes a plain dispatch with typed args, a real (or explicitly-unknown) result, and observability — and the bridge's bespoke invocation path retires into dispatch.

6Synchronous dispatch is a hard requirement

The load-bearing constraint The keyboard / pointer / click surfaces — the majority of actions — are synchronous by contract. If "actions become commands" routes them through an async Promise<Result> pipeline, three things silently break. This is not an optimization to defer; it gates whether chord-face convergence is admissible at all.

What the current code depends on, in the same tick the handler runs:

  • preventDefault / stopPropagation. The coordinator calls the handler synchronously, reads its return synchronously, then applies event options — all before the UA evaluates the native default. Cross a microtask and the browser has already saved/scrolled/typed: a bound chord silently stops suppressing native behaviour.
  • The not-handled fall-through. Picking the winning candidate for a chord reads the synchronous decline signal to try the next one — "the loop can't await to find out." An async pipeline can't decide the winner within the event.
  • DOM capture. The pointer path reads currentTarget / targetElement synchronously, because React nulls them once the handler returns.

This is not hypothetical — the navigation work, this proposal's founding success story, already hit this wall. useBlockOpener decides native-passthrough and calls preventDefault synchronously, before the async intent verb runs, with a comment that the carve-out is "intentionally NOT plugin-overridable" precisely because the new-tab default must not be prevented. The async verb facet (run is async-only — "a sync verb still pays one microtask") forced a sync escape hatch around itself. Generalizing the async verb to all actions reproduces that problem everywhere, not just at one call site.

What the design must commit to
  1. A first-class synchronous dispatch (dispatchSync) the coordinator uses, with the handler invoked, the DECLINE-or-handled decision read, and preventDefault all in one tick. Only the decision is synchronous — the command's effect may still resolve async (a keyboard delete's repo.tx), unawaited, just as today. The async dispatch is for non-DOM callers that want the result.
  2. Sync-deciding middleware only on the sync path (§5): a guard whose decision requires an await cannot run before a sync preventDefault. Async-deciding guards are async-face-only — see below.
  3. The narrowed thesis. "Imperative / surface callers become commands; the keyboard/pointer coordinator keeps its sync path and merely resolves a command it dispatches synchronously" — carried through to the dispatch signature, not just asserted in one line.
Async guards do not run on the sync path A sync-deciding wildcard (telemetry, logging, a sync veto) can run on both faces. An async-deciding guard — the canonical "confirm destructive command" that awaits a dialog (openDialog is promise-based) — can only run on the async dispatch face. So a keyboard-triggered delete (sync) would bypass an async confirm. This is the one place the two-tier risk is real, and the design must pick: (a) confirm-style guards are an async-face capability, advertised as such; or (b) a sync guard vetoes (consuming the event, §5) and kicks off an async confirm that re-dispatches on approval — a heavier, explicit protocol. Don't advertise "one guard for every command" without choosing one.

Scope note: the companion "sync resolution path for pure verbs" task gates only folding the keyboard-chord pure verbs (the in-tick fall-through decision) — paste folds onto the async path unchanged (§8).

7Commands and events

The two foundations are complementary, and must not be conflated:

CommandsEvents
Directionimperative — caller asksreactive — broadcaster announces
Returnstyped Resultnothing
Can veto / transform?yes (decorator)no
In the call path?yesno — fire-after
Todayactions + verb facetspost-commit processors (data only)

They compose, but the boundary is sharp and easy to get wrong. The verb after slot runs inline and awaited, inside the dispatch — it is in the call path (per the table above). So it is the right home only for in-path lifecycle hooks the caller awaits (a command-log entry that must land before the result returns, a timing pair with before) — and only on the async face: on dispatchSync the outcome isn't awaited (§5), so a sync-path after sees invocation, not success/failure. A keyboard-command telemetry/command-log built on after records "fired", not "succeeded"; per-outcome surfacing on the unawaited path is the §10 open question. It is the wrong tool for fire-and-forget domain reactions ("a block was created → reindex"), which belong off the call path on the event bus. Don't reach for after to avoid building the event bus, and don't route a typed domain event through a decorator. The event bus is the broader reactive surface (domain + lifecycle events), tracked separately by the typed domain-event/lifecycle bus investigation task — and it must stay typed (the audit-B3 constraint): post-commit processors are already a typed data-event bus, and the investigation generalizes that; no relapse to the string-keyed window.CustomEvent bus that was deliberately removed.

8The "double thing": verbs fold into commands

Once commands carry a typed result and the handle-keyed middleware slots exist (G1 with G2+G3 — not G1 alone, since the verbs' decorators/observers need a typed home), the bespoke verbs have nothing left that's special. The end state:

  • It's two of navigation's five seams, not "navigation". The fold covers exactly the two verbs: navigate.resolve (pure intent, fallback) and navigate.apply (effectful execution, rethrow); each verb's impl/decorators/before/after become generic middleware on that command's handle, and the two parallel verbs retire. But back/forward and URL-restoration are deliberately not routed through the verbs ([navigation.ts:25-34], navigation-redesign.md) — they flow through writePanelContent, a separate un-folded seam, alongside the still-TODO urlSerializerFacet/panelHistoryFacet. Consequence: command-level navigation middleware sees gesture/command navigations, not browser-back or deep-link restoration. Don't read "the verbs retire" as "navigation is a command."
  • What the fold buys navigation is the G1/G2/G3 wins — a typed Result to the caller, handle-keyed + wildcard middleware, one registry — not new coverage of the click path. A navigation click already goes through the bus today (useBlockOpener → navigateFromGesture → navigationVerb.run); the click's only synchronous obligation is preventDefault + the native-passthrough decision, which §6 keeps as a non-overridable carve-out outside the bus; and navigate.apply is irreducibly async (repo.tx), so it dispatches on the async face regardless. This is why the fold does not wait for G4 (§1.1).
  • pasteDecisionVerb is pure and is the easy case. It folds onto the async dispatch face (the onError: 'fallback' policy maps directly) because its DOM handlers already preventDefault eagerly and await the decision today (CodeMirror deliberately re-reads live editor state after the await). Making the decision synchronous would be an optional latency win, nothing more.
  • Two consumers the fold must not strand. extensions/api.ts exports both verbs on the public user-extension barrel — extensions are DB-installed and un-greppable from the repo, so retiring the verbs needs a deprecation shim there, not deletion. And resolveGlobalCommandTarget runs the intent verb as an effect-free read probe (resolving daily-note prev/next anchors and the shared destination-resolution read path), not a dispatch — the command shape needs an explicit probe entry that runs the resolution without firing before/after observers and never chains into navigate.apply.

The convergence tax doesn't vanish under "commands as foundation" — making a behaviour interceptable still means routing its callers through one choke. What changes is that the choke is generic and shared, so we pay it once for all behaviours instead of re-paying per feature.

9Migration path

Strictly incremental; each step is independently valuable and existing callers keep working. The dispatch surface to keep source-compatible is real but bounded: ≈17 production call sites across the by-id and supplied-deps entries (≈30 counting tests), plus the pointer/gesture dispatchers, the coordinator itself, and the agent bridge's direct-invocation path.

Critical path: G1 → G2 → G3 → verb folds. G1 unblocks "a dispatch returns a typed result"; retiring the verbs additionally needs G2 and G3 — typed handles first, precisely so the middleware layer is born typed (§5). G4 is deliberately off the critical path: everything above runs on the async face (§1.1), so the coordinator/sync-face work is a separately gated track.

  1. Return channel (G1) do first — widen the handler/dispatch result to carry a typed value, with DECLINE as the out-of-band not-handled signal (rationale in G1). Two consumers change mechanically: dispatchActionWithDeps's boolean widens to DECLINE | Result (its two in-tick readers — the swipe menu's preventDefault decision and the interaction-fallback check — adjust their test), and the agent bridge's {returned} wire payload starts carrying real values (today a declining handler surfaces there as returned: false). Valuable on its own: app-intents, the bridge, and a future command palette stop fire-and-forgetting.
  2. Typed command handles (G2) — defineCommand<Args, Result> returning a typed handle (§5). Handles are an intra-plugin convenience: the cross-plugin convention stays dispatch-by-id, never module imports (audit B3), and by-id keeps today's throw-on-missing semantics that app-intents relies on for disabled plugins. The adapter from existing actions must thread the third handler argument (ActionDispatch — activate/deactivate) that hold-style handlers use.
  3. Invocation middleware (G3) spawned: S1 — the verb facet's slots registered through handles, plus the untyped wildcard (§5). Two specification duties: (a) coverage — definition transforms reach every handler invocation today, including the three sites that bypass the shared candidate loop (hold-binding fire, gesture progress, and the agent bridge, which runs effective actions); dispatch middleware must reach all of them, or the gap is a silent behaviour regression for migrated plugins. (b) compose order between a surviving definition transform and dispatch middleware on the same command. Migration inventory, from the code: vim-normal-mode's and spatial-navigation's handler wraps and srs-rescheduling's three handler wraps (swipe-right, todo-cycle ×2) become decorators — noting spatial's vertical transform also replaces description, a definition field, so it splits into decorator + residual definition transform; srs's isVisible wrap and key-rebind/unbind stay definition transforms. Spatial's decorators are async-bodied but sync-deciding, which is exactly what the §5 sync contract admits.
  4. Verb folds — re-express navigate as the resolve/apply command pair and fold pasteDecisionVerb; all on the async face, so not gated on the sync-path task (§6, §8). Honour the two §8 caveats: a deprecation shim for the verbs on the extension barrel, and an effect-free probe entry for the intent-resolution read path.
  5. Lifecycle unification (G4) gated — collapse the entries onto one skeleton with per-trigger-kind policy, not a naive two-face merge: modal shadowing applies to the keyboard only (the by-id path deliberately skips it; pointer/gesture bypass it with an open FIXME for opt-in), hold bindings invert the lifecycle (preventDefault at arm time, gate re-checked at timer fire), the event-filter cascade gates keyboard dispatch only, and the entries disagree on throw-vs-silent-false for a missing action. Build when the gate opens — the palette isVisible/canDispatch mismatch (G4) bites for real, or chord-face middleware is demanded — and build dispatchSync's degraded middleware fold (§5) with it.

10Open questions

  • Outcome channels: the exact DECLINE shape lands with G1 (step 1); the VETOED outcome and the sync-deciding middleware contract (§5, §6) must be settled before the middleware step. The async-guard protocol — (a) async-face capability vs (b) veto-then-re-dispatch (§6) — is part of the same decision.
  • Extension trust: may user extensions register command/wildcard middleware, or first-party only? How is a veto contained, and the per-command args egress surface bounded? (§5.) This is load-bearing for the staged bet: if the answer is first-party-only, the uniform seam pays out less to the extension ecosystem the §1.1 bet counts on.
  • Vocabulary: do "actions" simply become commands (binding is a thin layer that dispatches a command), or do commands sit beside actions as a superset actions adapt into? Strong lean: one vocabulary — actions are commands with bindings. Requires the id-uniqueness invariant (§5).
  • Args vs deps boundary: is trigger an arg or ambient context? (Lean: ambient/optional.) How do bindings and palette rows supply args — Args ≅ void only, or a statically-bound args payload (§5)?
  • Imperative canDispatch: gate by default for consistency (which also closes the palette shown-then-ungated mismatch, G4), or preserve runActionById's "caller owns the precondition" escape hatch as an explicit opt-out?
  • Cancellation: does run take an AbortSignal for long/superseded commands (rapid navigation, in-flight search)?
  • Failure surfacing & tests: how does a vetoed/declined/failed command reach the user — today verbs swallow to console.error, navigate swallows to null, and a DECLINE must be observable to the invoking surface (disable/toast) rather than a silent no-op. A toast analogue to processorRejectionToast, or silence? And the test strategy for the sync vs async paths (AGENTS.md forbids timeout-based async tests).

11What this is not

  • Not a replacement for the facet system. Commands are built on facets — registration is a facet, middleware is facets (exactly as the verb facet already is).
  • Not a generic global pub/sub. Commands are typed and identity-bearing; the event side is typed too (audit B3). No string-keyed window bus.
  • Not a rendering/palette rebuild. Quick-find, the eventual command palette, app-intents, the agent bridge are command surfaces — they consume the foundation, they aren't part of it.
  • Not "build it all now." G1 is independently valuable and safe under every §1.1 option; the staged recommendation, its gates, and the decisions owed at each stage live in the §1.1 decision callout. The live design questions before the middleware step are the outcome channels (§5) and extension trust (§10).
