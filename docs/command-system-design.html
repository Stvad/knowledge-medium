Commands as a core foundation

Upgrading the action system into a typed, observable command substrate — so behaviour like navigation can be intercepted, replaced, and composed through one generic seam instead of a new bespoke one each time.

1The thesis

The navigation work proved a pattern we keep repeating: to make one behaviour extensible (observe it, wrap it, veto it, replace it), we hand-build a bespoke seam for it. Navigation got navigationVerb; paste got pasteDecisionVerb. Each is correct in isolation, but each is a new vocabulary the rest of the system doesn't share.

The higher-leverage move is to make two things core foundations that any behaviour can ride on, instead of re-deriving the seam per feature:

  • Commands — the imperative channel: "do X, give me a typed result," with one dispatch pipeline that every caller and every interceptor shares.
  • Events — the reactive channel: "X happened," that observers subscribe to without being able to veto. (Scoped separately; see §7 and the event-bus investigation task.)
Core claim Actions are already most of a command system — they have identity, binding, context, and a dispatch path. What's missing (a typed result channel, typed arguments, invocation-time middleware, one lifecycle) is exactly what we hand-rolled into the verb facet. Closing that gap turns "actions" into "commands," and — once the middleware slots exist — lets the bespoke verb seams retire into the generic bus. The claim is that this is higher-leverage than building the next bespoke seam, because it's paid once for all behaviours; §1.1 costs that against the cheaper alternatives rather than asserting it.

1.1  Alternatives considered

The verb facet's own docstring argues most seams shouldn't pay a four-slot structure — "reach for this only when there's one output function." So full convergence is not self-evidently right; the realistic alternatives:

  • (A) G1-only + keep bespoke verbs. Give actions a typed result (G1) so imperative/surface callers stop fire-and-forgetting, and let any new interceptable behaviour keep using defineVerbFacet as-is. Cheapest; preserves the verb facet's selectivity. Cost: the "double thing" persists — every new wrappable behaviour is still a hand-wired seam, and there's no uniform place to observe/veto across behaviours (no app-wide command log; no single sync precondition across commands). Note the delta vs (C) is smaller than it looks: an async confirm-guard is async-path-only even under (C) (§5), so that particular win isn't exclusive to convergence.
  • (B) A lighter shared helper. A thin dispatchById → typed result plus an opt-in per-command decorator list, without unifying the three lifecycles or the keyboard coordinator. Less risk than full convergence; cost: the canDispatch divergence and the sync/async split (§6) stay unsolved, so it doesn't actually let navigate-as-command happen.
  • (C) Full convergence (this proposal). Worth it only if we expect several more "make behaviour X interceptable" requests and want cross-behaviour middleware (telemetry, command palette with real results, agent-bridge dispatch with typed returns). The bet is that we do. If we don't, (A) is the right call.

Stance: this proposal recommends (C), with (A) as the fallback if the "several more requests" bet doesn't materialize. G1 is valuable under all three and commits to nothing, so it's safe to start there regardless — but building past G1 (the §9 steps, the spawned S1 + sync-path tasks) is (C)-specific work and constitutes choosing (C). The decision to make is therefore "(C) or stop at G1?", and it's owed before step 2, not deferred indefinitely.

2Where we are today

We have two partial command systems that don't know about each other.

The action system — identity, binding, context; no result, no invocation middleware

An Action (src/shortcuts/types.ts) carries an id, a context, a handler, optional keyboard/pointer/gesture bindings, and the gating predicates isVisible / canDispatch. It is dispatched through three entry points that have quietly diverged:

EntryDeps fromContext must be active?canDispatch?Returns
runActionById / useRunActionactive contextyesignoredvoid | Promise<void>
dispatchActionWithDepscaller-suppliednogatedboolean (handled?)
keyboard / pointer / gesture coordinatoractive contextyesgated— (single-winner)

The code itself flags the split: runAction.ts notes that unifying the canDispatch gates "is tracked with the broader dispatch-lifecycle work." And the handler return type is the giveaway:

export type ActionHandlerResult = void | false | Promise<void>

There is no typed-payload return channel. The one return that exists — dispatchActionWithDeps's boolean — carries only "did a candidate handle this?", and it's the same axis as the false sentinel: the synchronous false is a "not handled — try the next candidate" signal for the single-winner coordinator, not a value. So a caller that needs data back from a dispatch can't have it. Concretely, the PWA/share-target dispatcher (plugins/app-intents) has to fire-and-forget: 

const result = runActionById(actionId, new CustomEvent('app-intent'))
void Promise.resolve(result).catch(...)   // result is void; nothing to read

Rewriting action definitions is possible — actionTransformsFacet folds transforms over the action list at resolution time (effectiveActions.ts). Transforms can replace any field: vim-normal-mode and spatial-navigation wrap the handler; srs-rescheduling's flagship transform wraps the isVisible predicate; returning null unbinds. (Key rebinding is a separate pass — applyKeybindingOverrides reading keybindingOverridesFacet — not these plugins.) But every transform rewrites the definition, before any invocation. There is no way to intercept, observe, or veto a specific dispatch uniformly across the three entry points.

The verb facet — typed signature and invocation middleware; no identity, no binding

defineVerbFacet (src/facets/verbFacet.ts) is, in effect, a single-command bus. For one typed (input) => output it bundles four facets with the right combine algebra: impl (Replace, last-wins), decorators (Wrap, middleware), and before/after (Sum, observers) — plus an onError: 'rethrow' | 'fallback' policy that distinguishes effectful from pure verbs. run() executes before → decorators(impl)(input) → after.

This is precisely the invocation-time middleware the action system lacks — but it has no id you can dispatch by name, no binding, no context/deps resolution. You wire each verb by hand at its one call site.

The smell Navigation now maintains navigationVerb in parallel to the action system, purely to get a typed result and middleware that actions can't give it. That's the "double thing": a bespoke per-feature bus shadowing the generic dispatch path. Every future "make behaviour X extensible" request will reproduce it unless we close the gap once.

3The four gaps

What separates today's Action from a real command, ordered by importance of the gap (not build order — the dependency-ordered critical path is G1→G3→G4→G2, see §9):

G1 Typed result channel structural blocker

Dispatch returns void (or a boolean "handled"). A command must return a typed Result to its caller. This is the only structural blocker — it's what forced navigation into a bespoke verb. Scope note: G1 alone buys "a dispatch returns a typed value" (already a real win — see surfaces); it does not on its own let navigationVerb retire, which also needs G3 (somewhere for its decorators/observers to live). Caveat that G4 must resolve: once dispatch returns a real Result, the false not-handled signal can no longer be the return value (a toggle→boolean command's legitimate false would be read as "not handled"). The decline signal must become out-of-band (a distinct DECLINE sentinel, or candidacy decided before invocation) — see §6.

G2 Typed arguments, distinct from deps

Handlers receive deps (injected services, context- or caller-resolved) and a raw trigger event — but no typed argument payload. To pass data you smuggle it through deps or parse it off the event. A command is Command<Args, Result>: Args (what the caller asks for) is separate from deps (what the runtime injects).

G3 Invocation-time middleware spawned: S1

actionTransformsFacet rewrites definitions; the verb facet wraps invocations but only per-bespoke-verb. We want generic, dispatch-time before/decorator/after keyed by command id (with a wildcard), applied uniformly to every dispatch. Handler-wrapping transform consumers migrate onto this; key-rebinding/unbinding stays a definition transform.

G4 One dispatch lifecycle

Three entries with different canDispatch behaviour and return shapes. Collapse to one pipeline: resolve command → resolve deps (active or supplied) → gate → run middleware+handler → return typed result. Removes the divergence the code already flags as a TODO.

4The shape of a command (the equation)

The cleanest way to see the design: a command is the union of what each existing system already gives us.

Action gives

  • identity (id)
  • binding (key/pointer/gesture)
  • context + deps resolution
  • gating (canDispatch, isVisible)
  • registration via a facet

Command =

  • dispatch (id, args) → Result
  • typed Args + Result
  • id-keyed middleware
  • one lifecycle
  • observable

Verb facet gives

  • typed (input) → output
  • impl Replace
  • decorator Wrap
  • before/after Sum
  • onError policy

Command ≈ Action's identity / binding / context  ∪  Verb's typed signature / middleware

Where the union frays Two seams the equation has to reconcile, not just add:
  • One command can carry one onError; navigation needs two. Navigation is not one verb — it's a pipeline of navigationIntentVerb (pure, onError:'fallback', gesture → NavigateInput) → navigationVerb (effectful, onError:'rethrow', request → NavigationResult). A single command with a single failure policy can't host both, so "the navigate command" is really two commands (navigate.resolve pure, navigate.apply effectful) or one command with an explicit two-phase internal structure. §8 resolves which.
  • The two halves resolve winners differently. The action coordinator picks one of several candidates per chord via a synchronous "run until not-false" loop; the verb facet has a single last-wins impl and an async run. The union must keep the sync multi-candidate resolution (§6) while gaining the verb's typed/middleware shape — they don't simply add.

5Proposed model

A sketch (illustrative, not final API) of where the pieces land:

// A command: a verb with identity, binding, and context/deps resolution.
type In<Args, Ctx> = { args: Args; deps: DepsFor<Ctx>; trigger?: ActionTrigger }

// A command exposes EXACTLY ONE of run / runSync — a discriminated pair, not
// an async fn that's "sometimes sync". The invariant §6 needs is NOT "the
// effect is sync" (delete does an async repo.tx and is keyboard-bound — fine).
// It is: the DECLINE *decision* is synchronous. The result may still be a
// promise; the coordinator treats any non-DECLINE return as handled and does
// NOT await it — exactly as today a Promise-returning handler is "handled"
// while only a sync `false` falls through (types.ts forbids Promise<false>,
// not Promise<void>). So runSync forbids Promise<DECLINE>, not Promise<Result>.
type Command<Args, Result, Ctx extends ActionContextType> = Base<Args, Result, Ctx> & (
  | { run:     (i: In<Args, Ctx>) => MaybePromise<Result> }                 // async: dispatch() only
  | { runSync: (i: In<Args, Ctx>) => typeof DECLINE | MaybePromise<Result>  // chord/pointer-bindable;
      defaultBinding?: ...; pointerBinding?: ...; gestureBinding?: ... }      // DECLINE returned in-tick, result may be async
)
interface Base<Args, Result, Ctx> {
  id: string
  context: Ctx
  canDispatch?: (deps: DepsFor<Ctx>) => boolean
  isVisible?: (deps: DepsFor<Ctx>) => boolean
}
// The type forbids binding a chord to a `run` (async-decision) command — the
// coordinator must learn DECLINE-or-handled in-tick to fall through / preventDefault.

// TWO dispatch entries, not one — see §6 for why the sync one is mandatory.
// Async: imperative / surface callers that want a typed result.
dispatch<Args, Result>(
  id: string,
  args: Args,
  opts?: { deps?: Partial<BaseShortcutDependencies>; trigger?: ActionTrigger },
): Promise<Result>

// Sync: the keyboard/pointer/click coordinator. Runs the handler in the same
// tick so preventDefault / currentTarget / the not-handled fall-through still work.
// Only sync-capable middleware participates; DECLINE is the out-of-band fall-through.
dispatchSync<Args, Result>(
  id: string,
  args: Args,
  opts?: { deps?: ...; trigger?: ActionTrigger },
): typeof DECLINE | MaybePromise<Result>   // === DECLINE checked in-tick; result not awaited here

Middleware = the verb facet, generalized to the dispatch layer

We don't invent a new middleware mechanism — we lift the verb facet's three slots to the dispatch pipeline, keyed by command id:

  • Replace — swap a command's run entirely (what navigationVerb.impl(...) does).
  • Wrap (decorator) — intercept a dispatch: observe args, transform them, call next or short-circuit (veto by returning, not throwing — same contract as verb decorators).
  • Sum (before/after) — pure observers; after sees the VerbOutcome (success/failure) and cannot change it.

A wildcard id (mirroring WILDCARD_ACTION_ID) lets one contribution wrap all dispatches — a telemetry tap, a command-log, a sync precondition. The onError distinction carries over directly: effectful commands rethrow (no double-execution); pure ones may fall back.

Async middleware does not run on the sync path A synchronous wildcard (telemetry, logging, a sync veto returning DECLINE) runs on both faces. An asynchronous guard — the canonical "confirm destructive command" that awaits a dialog (openDialog is promise-based) — can only run on the async dispatch path. So a keyboard-triggered delete (sync) would bypass an async confirm. This is the one place the two-tier risk is real, and the design must pick: (a) confirm-style guards are an async-path capability, advertised as such; or (b) a sync guard returns DECLINE (suppressing the action) and kicks off an async confirm that re-dispatches on approval — a heavier, explicit protocol. Don't advertise "one guard for every command" without choosing one.
Decide before building: trust, ordering, cost Generalizing a wildcard wrap from a handful of trusted verbs to every command changes the threat and performance model — this app runs untrusted user-installed extension code that can import internals and contribute to these facets:
  • Trust. A wildcard decorator can veto every command by simply not calling next (a veto-by-return is legitimate, so it's indistinguishable from intended behaviour) and can read the typed args of every dispatch (block ids, content — a new data-egress surface). Decide: can extensions register wildcard/command middleware at all, or first-party only? How is a non-next veto contained vs. a throw (which the verb facet already isolates)?
  • Ordering. Cross-plugin middleware precedence must be deterministic — presumably the facet's precedence-asc-then-registration order — and the compose order between a surviving definition transform and a new dispatch middleware on the same id must be specified (today both could wrap the same action).
  • Cost. Every dispatch now pays two facet reads + a per-call decorator-fold allocation + (async path) a microtask. On the keyboard path this fires per keystroke; bound the wildcard count and keep middleware off the sync path unless it's sync-capable.

The single-winner coordinator stays, narrowed — and synchronous

Keyboard/pointer/gesture resolution — "which command wins this chord?" — remains its own concern, and it stays on dispatchSync (§6). It resolves a command, then dispatches it in the same tick. The fall-through signal is DECLINE (out-of-band from Result), read synchronously to try the next candidate; it never appears in a command's public Result. The async dispatch is for callers that hold no DOM event and want the typed value.

Surfaces become first-class

Several places already dispatch actions by id as if they were commands — app-intents, the agent runtime bridge, mobile bottom-nav, quick-find. Today they're fire-and-forget by-id calls. On a real command foundation they get typed args, real results, and observability for free, instead of each working around the missing channel.

6Synchronous dispatch is a hard requirement

The load-bearing constraint The keyboard / pointer / click surfaces — the majority of actions — are synchronous by contract. If "actions become commands" routes them through an async Promise<Result> pipeline, three things silently break. This is not an optimization to defer; it gates whether the convergence is admissible at all.

What the current code depends on, in the same tick the handler runs:

  • preventDefault / stopPropagation. The coordinator calls the handler synchronously, reads its return synchronously, then applies event options — all before the UA evaluates the native default. Cross a microtask and the browser has already saved/scrolled/typed: a bound chord silently stops suppressing native behaviour.
  • The not-handled fall-through. Picking the winning candidate for a chord reads the synchronous decline signal to try the next one — "the loop can't await to find out." An async pipeline can't decide the winner within the event.
  • DOM capture. The pointer path reads currentTarget / targetElement synchronously, because React nulls them once the handler returns.

This is not hypothetical — the navigation work, this proposal's founding success story, already hit this wall. useBlockOpener decides native-passthrough and calls preventDefault synchronously, before the async intent verb runs, with a comment that the carve-out is "intentionally NOT plugin-overridable" precisely because the new-tab default must not be prevented. The async verb facet (run is async-only — "a sync verb still pays one microtask") forced a sync escape hatch around itself. Generalizing the async verb to all actions reproduces that problem everywhere, not just at one call site.

What the design must commit to
  1. A first-class synchronous dispatch (dispatchSync) the coordinator uses, with the handler invoked, the DECLINE-or-handled decision read, and preventDefault all in one tick. Only the decision is synchronous — the command's effect may still resolve async (a keyboard delete's repo.tx), unawaited, just as today. The async dispatch is for non-DOM callers that want the result.
  2. Which middleware may run on the sync path. A before/decorator that returns a Promise cannot run before a sync preventDefault; sync commands admit only sync-capable middleware (or middleware runs only on the async path).
  3. The narrowed thesis. "Imperative / surface callers become commands; the keyboard/pointer coordinator keeps its sync path and merely resolves a command it dispatches synchronously" — carried through to the dispatch signature, not just asserted in one line.

The companion "sync resolution path for pure verbs" task addresses the same root for paste/navigation-intent. It is a precondition for folding any synchronous-dispatch caller onto the command pipeline — not an incidental cost.

7Commands and events

The two foundations are complementary, and must not be conflated:

CommandsEvents
Directionimperative — caller asksreactive — broadcaster announces
Returnstyped Resultnothing
Can veto / transform?yes (decorator)no
In the call path?yesno — fire-after
Todayactions + verb facetspost-commit processors (data only)

They compose, but the boundary is sharp and easy to get wrong. The verb after slot runs inline and awaited, inside the dispatch — it is in the call path (per the table above). So it is the right home only for in-path lifecycle hooks the caller awaits (a command-log entry that must land before the result returns, a timing pair with before). It is the wrong tool for fire-and-forget domain reactions ("a block was created → reindex"), which belong off the call path on the event bus. So the two are complementary, not substitutes: don't reach for after to avoid building the event bus, and don't route a typed domain event through a decorator. The event bus is the broader reactive surface (domain + lifecycle events), tracked separately by the typed domain-event/lifecycle bus investigation task.

Hard constraint (audit B3) The event bus must be typed, not a relapse to the untyped window.CustomEvent UI bus that was deliberately removed. Post-commit processors are already a typed data-event bus; the investigation generalizes that, it does not reintroduce a string-keyed global. Veto belongs in a command decorator; reaction belongs in an event observer — keep the two channels distinct.

8The "double thing": verbs fold into commands

Once commands carry a typed result and the id-keyed middleware slots exist (G1 and G3 — not G1 alone, since the verb's decorators/observers need somewhere to live), the bespoke verbs have nothing left that's special. The end state:

  • It's two of navigation's five seams, not "navigation". The fold covers exactly the two verbs: navigate.resolve (pure intent, fallback) and navigate.apply (effectful execution, rethrow); each verb's impl/decorators/before/after become generic middleware keyed by that command's id, and the two parallel verbs retire. But back/forward and URL-restoration are deliberately not routed through the verbs ([navigation.ts:25-34], navigation-redesign.md) — they flow through writePanelContent, a separate un-folded seam, alongside the still-TODO urlSerializerFacet/panelHistoryFacet. Consequence: command-level navigation middleware sees gesture/command navigations, not browser-back or deep-link restoration. Don't read "the verbs retire" as "navigation is a command."
  • The sync win is narrower than "the click now goes through the bus". A navigation click already goes through the bus today (useBlockOpener → navigateFromGesture → navigationVerb.run), so middleware on navigate.apply already applies. The click's only synchronous obligation is preventDefault + the native-passthrough decision — which §6 keeps as a non-overridable carve-out outside the bus. And navigate.apply is irreducibly async (repo.tx), so it runs via dispatch (async), not dispatchSync — relocating, not removing, the async boundary. The real gains over today are the G1/G2/G3 ones (a typed Result to the caller, id-keyed wildcard middleware, one registry), not "uniform middleware on the sync click".
  • pasteDecisionVerb is pure; it either stays a pure verb or becomes a pure command (the onError: 'fallback' policy maps cleanly). Its dispatch must stay synchronous (a runSync command, §5), which is a precondition handled by the sync resolution path for pure verbs task (§6) — the fold is clean conditional on that succeeding, not unconditionally.

The convergence tax doesn't vanish under "commands as foundation" — making a behaviour interceptable still means routing its callers through one choke. What changes is that the choke is generic and shared, so we pay it once for all behaviours instead of re-paying per feature.

9Migration path

Strictly incremental; each step is independently valuable and the existing callers keep working — and there are 60+ of them across the four dispatch entries (runActionById ≈29, useRunAction ≈16, navigateFromGlobalCommand ≈16, dispatchActionWithDeps ≈7), so each step must be source-compatible, not just behaviour-compatible.

Critical path (corrected) G1 → G3 → G4 → G2 → navigate-fold. The earlier framing "G1 is the only blocker for navigate-as-command" was wrong: G1 alone unblocks "navigate dispatch returns a typed result"; retiring the verb (the §8 end-state) also needs G3 (a home for its middleware) and G4 (so the middleware applies on the keyboard path too).
  1. Return channel (G1) do first — widen the handler/dispatch result so imperative callers receive a typed value. Make the not-handled signal out-of-band (a DECLINE sentinel), since false can no longer double as both a value and "not handled" (G1). Valuable on its own: app-intents / agent-bridge / a command palette stop fire-and-forgetting.
  2. Invocation middleware (G3) spawned: S1 — add dispatch-time before/decorator/after keyed by command id + wildcard. Migrate the handler-wrapping actionTransformsFacet consumers (vim-normal-mode, spatial-navigation); leave key-rebind/unbind and predicate-wrapping (srs isVisible) as definition transforms. Specify the compose order between a surviving definition transform and a new dispatch middleware on the same id (today both can wrap one action).
  3. Lifecycle unification (G4) — collapse the entries to one pipeline with a sync (dispatchSync) and async (dispatch) face (§6); decide the imperative canDispatch policy (resolving the flagged TODO) and the DECLINE channel.
  4. Typed command handles (G2) — introduce defineCommand<Args, Result> returning a typed handle (like verb facets do); Args distinct from deps. New commands use it; existing actions interop via an adapter.
  5. Navigate-as-command — re-express navigate as the resolve/apply command pair (§8); retire both navigation verbs. Paste follows, gated on the sync-path task.

10Open questions

  • Sync/async split (blocking — §6): the exact shape of dispatchSync and the DECLINE channel, and which middleware may run synchronously. This is the gating decision, not a detail.
  • Extension trust: may user extensions register command/wildcard middleware, or first-party only? How is a veto-by-not-calling-next contained, and the per-command args egress surface bounded? (§5)
  • Vocabulary: do "actions" simply become commands (binding is a thin layer that dispatches a command), or do commands sit beside actions as a superset actions adapt into? Strong lean: one vocabulary — actions are commands with bindings.
  • Args vs deps boundary: is trigger an arg or ambient context? (Lean: ambient/optional.) How do non-React/imperative callers supply args + deps without re-deriving context?
  • Imperative canDispatch: gate by default for consistency, or preserve runActionById's "caller owns the precondition" escape hatch as an explicit opt-out?
  • Cancellation: does run take an AbortSignal for long/superseded commands (rapid navigation, in-flight search)?
  • Failure surfacing & tests: how does a vetoed/failed command reach the user (today verbs swallow to console.error and navigate swallows to null) — a toast analogue to processorRejectionToast, or silence? And the test strategy for the sync vs async paths (AGENTS.md forbids timeout-based async tests).

11What this is not

  • Not a replacement for the facet system. Commands are built on facets — registration is a facet, middleware is facets (exactly as the verb facet already is).
  • Not a generic global pub/sub. Commands are typed and identity-bearing; the event side is typed too (audit B3). No string-keyed window bus.
  • Not a rendering/palette rebuild. Quick-find, the eventual command palette, app-intents, the agent bridge are command surfaces — they consume the foundation, they aren't part of it.
  • Not "build it all now." G1 is independently valuable and safe to land first under any of the §1.1 options; the full convergence (and navigate-as-command) is a separate commitment gated on the §6 sync decision.
  • A recommendation, not a settled spec. It recommends (C) with (A) as fallback; the dispatchSync/DECLINE shape (§6) and the async-guard protocol (§5) are the live design questions to settle before step 2. Debate the bet and those shapes; G1 needs no decision.
