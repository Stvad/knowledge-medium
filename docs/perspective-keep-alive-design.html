Perspective keep-alive

Sessions-per-perspective + a warm session host — 2026-07-12

Switching perspectives (window-management extension) currently unmounts and remounts every pane's React tree — a playing video restarts, editors lose transient state, heavy views (agenda) pay full re-mount cost. Goal: switching a perspective keeps every other perspective's views fully alive, so coming back finds them exactly as left — playback still running.

Secondary goal: this is a step toward upstreaming the perspectives extension (today it lives in spolakh's fork as agent-extensions/window-management/ — emacs-style panes, numbered + per-item perspectives, a bottom workspace bar). The extension stays an extension; the enabling mechanism lands in core as a generic, self-contained feature ("switchable layout sessions with keep-alive") that others could build tab-like UX on.

Why switching remounts today

A perspective switch never swaps trees — it rewrites the single live layout-session block in place. applyPerspective (agent-extensions/window-management/index.tsx) calls reconcilePanelRows (src/utils/panelLayoutProjection.ts:677), which LCS-diffs panel rows by topLevelBlockId. Across perspectives the block ids are disjoint, so one of two things happens, and both destroy the mounted content:

  1. Row reused, content rewritten (equal pane counts — the common case): the panel row id — the React key on the top-level slot list (src/components/renderer/LayoutRenderer.tsx:180, slotsToRender.map) and on PanelSlotView's context provider (LayoutRenderer.tsx:85) — stays stable. LayoutRenderer mounts the row's <BlockComponent blockId={slot.id}/> (LayoutRenderer.tsx:91), which resolves to PanelRenderer; that reads topLevelBlockIdProp (src/components/renderer/PanelRenderer.tsx:65) and renders <BlockComponent blockId={topLevelBlockId}/> (PanelRenderer.tsx:256, no key). When the perspective switch rewrites topLevelBlockIdProp, that inner element reconciles to a different block and the whole content subtree unmounts.
  2. Row deleted/created (pane counts differ): tx.delete / createPanelRowInTx → new row id → new key → the entire PanelSlotView unmounts.

The perspective snapshot (storage.ts, per-device localStorage) stores only content identity — a serialized layout hash of block ids + tree shape, plus activeBlockId. There is no handle to a live view to come back to. Keep-alive cannot be bolted onto this model; "one session, rewritten in place" is the specific thing defeating it.

The meta-seam: why this needs core at all

Worth stating precisely, because it defines the smallest possible core change. The extension rewrites one session's rows not by choice but because the layout root bypasses both extension mechanisms. Two couplings live in App.tsx, neither routed through a facet:

  1. App.tsx:309 hardcodes <BlockComponent blockId={initial.layoutSessionBlock.id}/> as the layoutBoundary root. Block renderers are a facet, so an extension can override how that block renders — but not which block is the root, and not mount more than one. The root id comes from bootstrapWorkspace → getLayoutSessionId(), fixed at boot.
  2. App.tsx:190 constructs the one new PanelLayoutProjection({layoutSessionBlock, …}) in an effect and runs it — it owns URL↔rows + history. No facet, no per-session instancing; App unconditionally owns it. The class is exported/importable, but a second projection would just fight the first over the hash.

So the missing seam is not "an affordance to replace sessionId" — that is the narrowest possible patch. Two things had to be true for a plugin to build this unaided: (a) the render tree — already reachable in principle via the block-renderer facet, since the root is a BlockComponent; the only gap is that App picks the root block id rather than resolving it through a seam, and mounts exactly one. (b) the URL/history projection — not reachable at all: App always constructs and runs exactly one, bound to that single block.

Minimal meta-seam — a "layout-root provider". A single contribution point that owns (a) what mounts at the layout boundary and (b) the projection/URL binding; App's current hardcoded mount + projection becomes its default implementation. When a contribution is present it suppresses App's default projection and provides the root. Combined with the block-renderer override that already exists on the root, that alone unlocks plugin-side perspectives: the extension supplies the provider, mounts N session blocks with keep-alive under its own renderer, and owns the hash. The session-swap store is then pure pluginland. This is the core ask — smaller and more general than a session API.

Audit: the non-injected globals that block it

The feature is gated by a chain of module-level globals, each read directly with no injection seam, all resolving the same single per-tab session — so overriding any one alone is insufficient. Ranked by how much each blocks multiple-sessions:

What the provider looks like

It needs no new facet — the seam is "App mounts the root through the registry + projection ownership moves into that renderer," which is exactly the "delegate to a blockRenderer ASAP" intent (App.tsx has outgrown its boot role).

  • Thin App. App shrinks to: resolve the root block, mount <BlockComponent blockId={root.id}/> under {layoutBoundary:true}, and keep only the always-on hashchange listener that must survive the no-layout states (locked/waiting — App.tsx:260 — where there's no projection yet to re-resolve the workspace). Its projection effect (App.tsx:188-220) moves out.
  • Default layout-root renderer. A RendererContribution matching the root context that does what App's effect does today: resolve the single session via getLayoutSessionId(), mount it, own one PanelLayoutProjection through a usePanelLayoutProjection(sessionBlock) hook (construct/start/dispose, carrying the StrictMode disposed-guard and workspace-switch teardown). Behavior identical to today — a pure extraction, shippable on its own as the "App.tsx got too big" cleanup, independent of perspectives.
  • The override. An extension registers a higher-priority renderer for the root block (or rendererProp-pins it) → LayoutSessionHost: mounts N session children with keep-alive, owns the active-session store, runs one usePanelLayoutProjection bound to the active session. Two enabling changes, both small because the channels exist:
    • Session id via context, not the global — convert the ~9 getLayoutSessionId() callers to read layoutSessionBlockId from block context (already a field). The host then provides a different id per session subtree; nested consumers follow. The global stays as the default the thin App / default renderer seeds.
    • Projection construction in the renderer — already injectable at the URL layer; just constructed by whoever renders the root, not by App.
The one structural tension. AppRuntimeProvider — where facets/renderers become available — currently sits below where App computes layoutSessionBlock and above the mounted block. Moving projection ownership into a renderer (below the runtime) means it needs workspaceId + the root block (computed above it) and must feed a version-bump back into App's getInitialLayout cache. So the extraction also nudges that boundary: App passes the resolved root + a re-resolve callback down through the runtime rather than owning the projection above it. Not hard, but it's the part that isn't a pure move.

Design: perspective = its own layout-session block

Flip the model. Each perspective owns a distinct layout-session block; switching changes which session is visible. The outgoing perspective's rows, scroll, focus, and React tree are never touched — keep-alive falls out of the data model instead of being retrofitted.

Data model — synced definitions, per-device live sessions

Two layers (upstream review outcome — all state in the DB, perspectives available across devices):

  • Perspective definitions — synced. A perspective registry of per-workspace UiState blocks: name, slot/order, kind (numbered vs per-item), target block for item perspectives. Defined once, available on every device. This replaces the extension's localStorage snapshot store entirely.
  • Live sessions — per-device. Each device materializes its own layout-session block per perspective (uuidv5(deviceBaseId + perspectiveId)). Kept per-device deliberately: per-row view state (scrollTopProp, focusedBlockLocationProp, activePanelIdProp) lives on session/row blocks and syncs — two devices sharing one live session would ping-pong scroll and focus at each other. Sharing the definition but not the live session gives cross-device perspectives without that fight. A fully-shared live session ("my workspace follows me") is a possible later evolution, but requires first splitting per-device view state out of the shared layout structure — separable, bigger lift.

Note layout sessions already live in the database (UiState blocks under user page → ui-state → layout-sessions/<id>); the only genuinely device-local state left is two pointers — the stable base id and the active perspective — and even the active pointer can move into a device-scoped DB block if the all-state-in-DB principle is applied strictly.

Core half — LayoutSessionHost

  • Two ids, kept distinct. getLayoutSessionId() (src/utils/layoutSessionId.ts:54) today yields the one live per-device id (sessionStorage in a tab, localStorage installed). That id becomes the stable per-device base id — it never changes, and its storage key stays where it is. Alongside it, add a runtime-switchable active session id (module store à la createToggleStore, persisted per-device separately), defaulting to the base id — zero behavior change when nothing drives it. Per-perspective session ids are always derived from the base id (uuidv5(baseId + ':' + perspectiveId)), never from the currently active id — deriving from the active id would make session identity drift as you switch.
  • Perspective in the URL. PR #338's matrix-param + comma-stack grammar is now merged (src/utils/routing.ts — LayoutSlot carries viewMode/active/rest): slots take ;key=value context, unknown keys round-trip opaquely through rest[] (free forward-compat), and the projection already syncs the ;active key ↔ panel rows — so the active pane is URL-borne today, which folds neatly into per-session state (each session's hash carries its own active). Perspective is workspace-scoped context, not slot-scoped, so it wants to ride on the workspace segment: #ws;persp=work/a/b,c. One honest gap: matrix context is parsed on slots only; parseLayout still splits the workspace segment as a bare id ([workspaceId, …cols] = splitTopLevel(hash, '/')), so ;persp= there is not parsed yet. Extending parseContextEntries to the ws segment is the same-shaped small addition — and ;active is the proof that a core-owned context key already works end-to-end. With synced definitions the link is portable across devices (the receiver materializes its own live session for that definition), and history entries become perspective-qualified — which settles Back-across-a-switch: Back returns to the prior perspective, because that's what the prior history entry encodes. The apparent duplication (a name standing for a set of panes, plus the panes spelled out) is an identity+value pair: persp= is identity (which live session / history lane), slot tokens are value (the exact panes). Precedence on open: persp selects/creates the local session; slot tokens, when present, reconcile that session's rows; absent slot tokens (short-form link #ws;persp=work, the natural shape for sharing), the session shows whatever state it's in. Persp-only URLs couldn't restore within-perspective layout history; slots-only URLs lose the lane attribution — both parts earn their place.
  • Warm-set host — a block renderer, not app chrome. src/App.tsx:309 mounts exactly one <BlockComponent blockId={layoutSessionBlock.id}/>; the layout-sessions container block (user page → ui-state → layout-sessions, src/data/stateBlocks.ts) is never rendered. Instead, App mounts <BlockComponent blockId={container.id}/> and LayoutSessionHost is the container's registered block renderer — the concrete form of the layout-root provider from the meta-seam above (keeps the block-display hierarchy renderer-overridable, so users can replace the multi-session policy surface like any renderer). It reads the active-session store + warm set and keeps an LRU set of recently-active session blocks mounted (cap ~3, configurable), each under a stable key={sessionBlockId}, inactive ones hidden per the hiding policy below. Set membership stable → no remount → playback continues.
  • URL projection binds to the active session only. PanelLayoutProjection (src/utils/panelLayoutProjection.ts:967; constructed at App.tsx:190) keeps rows ↔ URL hash in sync bidirectionally and pushes a hash entry on every layout change (pushHash, :1105). Owning this is exactly the layout-root provider's job (b): hidden sessions must not push; on switch, rebind (or select among per-session instances) and push the new session's layout hash once. Keeps the maximize/restore history.back() leg working.
  • Input isolation — the subtle correctness area, and it is wider than shortcut surfaces. Each session carries its own activePanelIdProp, so with N sessions mounted, N panels all believe they are active. Everything downstream of that belief must be gated on session visibility:
    • Shortcut surfaces: keybind dispatch and "active panel" resolution go through the active session only (e.g. a sessionHidden context flag checked by dispatch/activations).
    • Action contexts: PanelRenderer registers MULTI_SELECT_MODE for its active panel via useActionContext (src/components/renderer/PanelRenderer.tsx:203, PanelMultiSelectActionContext) — hidden sessions' active panels would each register it too. All useActionContext registrations under a hidden session must be suppressed (the same visibility flag, threaded through block context so it reaches every registration site).
    • DOM selectors: spatial navigation (src/plugins/spatial-navigation/actions.ts:270, crossPanelFocus) and window-management (agent-extensions/window-management/index.tsx:236, panelElements) locate the layout via document.querySelector('[data-layout-session-id]') — first match wins, which becomes ambiguous with several sessions mounted. The host must make the active session unambiguously addressable (e.g. only the visible wrapper carries a data-layout-session-active marker, plus inert on hidden wrappers as focus/click belt-and-braces), and a core helper (activeLayoutSessionElement()) replaces raw first-match queries at every call site.
    • On reveal, re-assert DOM focus from the session's persisted focus state (the focus class survives — it's prop-driven; only DOM focus needs a nudge).

    Under <Activity mode="hidden"> the effect-registered half solves itself (verified): useActionContextActivations registers through useEffect (src/shortcuts/useActionContext.ts:37), so hidden sessions' action contexts and effect-attached listeners deactivate on hide and re-activate on reveal — no sessionHidden gating needed for those. What Activity does not solve: the DOM-selector ambiguity (hidden DOM stays in the document) and the reveal-time focus nudge — those two remain regardless of hiding mode. This is a major argument for the Activity-first policy below.

Extension half — window-management shrinks

  • Slots become registry entries (synced perspective-definition blocks, see the data model above), both numbered ('1'..'8') and item (item:<blockId>). Each maps on a given device to a deterministic session block id (uuidv5 of base id + perspective id, same pattern as src/data/stateBlocks.ts).
  • Switch = one store write (set active session id + repaint the bar), replacing the whole capture → reconcilePanelRows → repair-active-panel dance (performSwitch, index.tsx:553).
  • The serialized-layout snapshot survives only as: (a) migration from the current localStorage format into the registry, (b) seed when a perspective's session block doesn't exist yet on this device, (c) the item-slot expiry sweep — which now must also delete the expired perspective's session block subtrees (sessions are UiState blocks that upload; leaked sessions are synced garbage).
  • The bottom bar moves from imperative document.body DOM to an appMountsFacet contribution (src/extensions/core.ts:218) — React, app scope, outside the layout tree, survives switches; the imperative DOM was expedience, not necessity. If multiple extensions ever compose indicators, a statusBarItemsFacet mirroring headerItemsFacet's region pattern is the natural core addition (upstream review outcome; a LayoutRenderer override is heavier than the bar needs).

Hiding policy: <Activity>

React 19.2.6 (installed) exports stable <Activity> — the official hide-with-state primitive. What it buys over a hand-rolled display:none wrapper: hidden subtrees' re-renders are deferred to background priority (they never block the visible perspective), state and DOM are preserved with official semantics, the effect-registered half of input isolation solves itself (above), and it's where React is investing (mode="hidden" pre-rendering could later give "warm up a perspective before first visit").

The crux is effect semantics: hiding runs effect cleanups, revealing re-runs them — state and DOM are preserved throughout. An earlier revision of this doc assumed that kills playback; verifying against the installed react-player corrected it:

react-player v3 has no destructive effect cleanup (verified against node_modules/react-player/dist): players render as declarative web components (youtube-video-element etc.), the only .pause() in the dist is gated on playing === false (Player.js), and the PiP effect has no cleanup. Under Activity-hidden the media element stays in the document and should keep playing. What degrades while hidden is only the effect-registered integrations: the registerVideoPlayer imperative handle (src/plugins/video-player/VideoPlayerRenderer.tsx:101) unregisters (arguably correct — nothing should remote-control a hidden player; it re-registers on reveal), and the play/pause prop-sync effect sleeps until reveal. CodeMirror editors (created in effects) are destroyed on hide and recreated on reveal — acceptable; edit mode is transient.

Policy (upstream review outcome): wrap every inactive session in <Activity mode="hidden">, full stop — no parallel display:none path built. The deciding question is empirical, not architectural, so rollout step 1 includes a spike (start a video, hide the session under Activity, confirm audio continues). Contingency, not planned work: if that spike shows a specific provider can't survive Activity's effect unmount, the isolated fix is to wrap that one media subtree in a plain <div style="display:none"> (which keeps effects mounted) — same host, one-line escape hatch, revisited only if forced.

Background cost model: per warm session, retained DOM (bounded — see LazyViewportMount below) plus live query subscriptions that re-render the hidden tree on writes. A quiet workspace costs ≈ zero (handle-level structural-diff dedup); the real cost is write bursts (sync storms) re-rendering N hidden trees. Controls, in order: Activity's background-priority deferral of hidden re-renders (the default mode), the small LRU cap (2–3; eviction fully unmounts and releases everything), and item-perspective expiry deleting dead sessions.

LazyViewportMount composes well: it never unmounts once mounted, and while hidden its IntersectionObserver sees nothing — not-yet-seen blocks stay unmounted until the perspective is revealed. Free partial hydration for warm sessions.

Alternatives considered

  • Portal-based content cache — app-level host owns one <BlockComponent> per content block id and portals it into whichever panel shows it; no data-model change. Rejected: portals keep the owner's React context, so panel-scoped context (panelId, renderScopeId, ui-state resolution) must be re-threaded per target, and physically moving a playing <video> between DOM containers pauses/hiccups playback in practice. Fights the framework everywhere the session-swap design goes with the grain.
  • One session, hide rows instead of deleting — rows tagged per-perspective, LayoutRenderer filters. Pollutes the layout data model: URL projection, subtree queries, active-panel repair, and the layout watchdog all must learn to ignore hidden rows. More invasive than it looks.
  • Media-only persistence — global media-element registry, or save-position-and-resume. Far less core change, but only fixes video, not the general "my perspective is exactly as I left it" property. Fallback if the core change stalls upstream.

Open questions (resolve at implementation time)

  • Where exactly keybind dispatch resolves "the active panel" — the hidden-session gating design hangs on it.
  • PanelLayoutProjection: one instance rebound on switch vs. one per mounted session (hidden ones muted)?
  • Mobile: LayoutRenderer already renders only the active panel there; keep-alive is likely desktop-only (warm cap = 1 on mobile).
  • CodeMirror revealed from display:none may need a requestMeasure nudge.
  • Warm-set eviction: is plain LRU enough, or should a session with actively playing media be pinned against eviction?
  • Definition ↔ session reconciliation: when a synced perspective definition changes while a device holds a diverged live session for it, what wins on next switch (reconcile rows to the definition vs. keep the local session and offer a reset)? Related: should local layout edits write back to the definition, and when?

Rollout sketch

  1. Core, stands alone: extract the default layout-root renderer — move App's projection effect + root mount into a RendererContribution + usePanelLayoutProjection hook; App shrinks to resolving + mounting the root. Behavior identical (single session); this is the "App.tsx got too big" cleanup and needs no perspectives to justify it.
  2. Core: session id read from layoutSessionBlockId context instead of the getLayoutSessionId() global at the ~9 call sites (global becomes the default seed); projection construction fully owned by the renderer. This is what makes the root overridable.
  3. Core: active-session store + LayoutSessionHost override behind a default-off switch; single-session behavior unchanged. Includes the Activity spike: start a video, hide its session under <Activity mode="hidden">, confirm playback continues.
  4. Core: input gating + DOM-selector disambiguation; verify with two hand-made sessions in the sandbox (video playing in one, switch away/back).
  5. Extension: synced perspective registry + per-device deterministic session ids, migration from localStorage snapshots into the registry, expiry sweep deletes session subtrees, bar → appMountsFacet.
  6. Sandbox-verify the full loop, then promote + upstream.
