Schema unification: definitions are blocks, behavior is presets, code seeds

Collapse the two property-schema (and block-type) worlds — inline code objects merged by name vs. definition blocks projected by id — into one: every definition is a synced block with a block-id identity; code registers behavior (ValuePresets) and seeds definitions instead of registering them.

TL;DR

Today kernel/plugin schemas are inline PropertySchema objects merged by name (mergeLiftedSchemas, refProjection.ts:29) while user schemas are property-schema blocks projected by block id — two identity spaces, two registration paths, and a set of workarounds the properties-as-blocks migration (PR #288 §6) can no longer absorb: synthetic property:<name> fieldIds leaking into the reference index, a childBacked flag with no durable home for plugins, cross-source name collisions, and a two-phase cold-start bootstrap.

The end state: one source of definitions (blocks), one identity (block id), one registry path (the projector). Kernel and plugin schemas become seeded definition blocks with deterministic per-workspace ids (uuidv5, the stateBlocks.ts pattern); their codecs are rebuilt from code-registered ValuePresets exactly the way user schemas already are (tryBuildSchema, userSchemasService.ts:52). A code-resident synthesis layer makes kernel definitions available synchronously before any block exists, so cold start needs no two-phase bootstrap. Types get the same move (§9). The preset-expressibility gate passes — every registered schema is expressible as (preset, config) (§10).

1Why now — the consumers

Two in-flight designs consume this one:

Properties as blocks (PR #288). That migration makes every property value a child block whose field row carries reference_target_id = fieldId. Its §6 states hard requirements the current kernel/plugin-vs-DB schema split cannot meet, and its §11 rollout bends around them. This project is the fix it names. Specifically, the migration needs:

  1. fieldId = definition block id, for every schema, kernel included. The spike's synthetic ids (fieldIdForPropertyName → property:<name>, src/data/api/propertySchema.ts:90 on origin/props-as-childrent-spike-merge) put non-block ids into the synced reference_target_id column and the backlink index (#288 §6, residual 2). After this project, that helper and its reference-projection special-case never ship.
  2. childBacked (and hidden) as synced fields on the definition, classifiable without the owning plugin installed. #288 §6 calls a plugin flag expressed as a code constant "a data fork waiting to happen" and requires durable persistence — this design is that persistence mechanism.
  3. Kernel definitions available at cold start before first sync. #288 §11 step 5 needs a two-phase first-sync projection because schema discovery reads the cell; deterministic seeded ids dissolve phase 1 into a pure computation (§11.2 below).
  4. Fleet-wide classifiability. #288 §11 step 5 can only mark rows fully-child-backed when every key belongs to a schema every client can classify — which today excludes plugin keys. Definitions-as-blocks makes plugin schemas classifiable everywhere (§11.1).

Plugin materialization (PR #289). Its parallel track (§8 there, "materializing types & properties") is this project sketched in one paragraph: seed block-type/property-schema blocks from system definitions through the existing definition-block projector. This doc designs that stage in full and reuses #289's structure where it fits: the kernel → materialize → assemble boot staging (§4.4), the provenance/base-version thinking for seed upgrades (§6), and the toggle-id-as-stable-API decision for seed keys (§4.2). Where #289 materializes code on demand behind an approval gate, definitions are pure data — no approval, no compile pipeline, seeded eagerly rather than ejected on demand. Per the project brief: we account for future plugin materialization but do not gate on it; nothing here depends on #289 landing.

2.2Block world: user definitions, id-keyed, projected

  • property-schema blocks store {propertyName, presetId, presetConfig} (src/data/properties.ts:170-192); the codec is rebuilt from the code-registered preset: tryBuildSchema validates config through preset.configCodec and calls preset.build(config) (userSchemasService.ts:52-94).
  • ValuePreset (src/data/api/valuePresets.ts:13) is already the behavior-as-code contract: id, build(config) → codec, defaultValue, configCodec, Editor, Glyph, ConfigEditor, hideFromPicker. All 9 existing presets are kernel-owned (kernelValuePresets.ts:72-161), registered under the system:kernel-value-presets toggle — no plugin defines a preset today.
  • Forward-compat is already built: a schema block referencing an unknown presetId is skipped with a warning and re-resolves when the preset's plugin loads (tryBuildSchema skip path + secondarySignal: onValuePresetsChange, userSchemasService.ts:107).
  • User types follow the same pattern: block-type blocks, block id = type id, block-type:properties refList resolved through the schema projector's handle (userTypesService.ts:60-91).
  • The shared lifecycle is ProjectorRuntime (src/data/projectorRuntime.ts): subscribe to {workspaceId, types:[metaType]}, project per row with per-row failure isolation, publish into the facet's durable 'user-data' runtime bucket. Projectors start in an AppRuntimeProvider effect — after first paint (AppRuntimeProvider.tsx:239-243), so user definitions are not available to workspace bootstrap today.
  • Deterministic per-workspace block ids via uuidv5 have precedent in stateBlocks.ts:54-58 (user page, state children) and the kernel pages (propertiesPage.ts:13).

The two worlds already agree on the important shape — definitions reduce to (name, preset, config, scope) and behavior lives in presets. What differs is identity (name vs block id), availability (always vs after projector prime), and who owns registration (merge step vs projector). Unification keeps the block world's shape and gives it the code world's availability. The §10 audit verifies feasibility against this inventory: the gate passes — every schema above is expressible as (preset, config).

3Target model — the definition/behavior split

Split today's PropertySchema into a definition (data, on a property-schema block) and behavior (code, a ValuePreset keyed by presetId). The definition is the user-defined shape that already exists, extended with the fields the migration needs:

The runtime registry entry — what consumers get from repo.propertySchemas — becomes:

interface PropertySchema<T> {
  readonly name: string
  readonly fieldId: string        // the definition block id — THE identity (registry entries only, see below)
  readonly workspaceId: string    // the workspace this entry belongs to (registry entries are per-workspace, §4.4)
  readonly codec: Codec<T>        // preset.build(config)
  readonly defaultValue: T        // decoded property-schema:default, else preset.defaultValue
  readonly changeScope: ChangeScope
  readonly childBacked: boolean   // populated once #288's flag lands; absent → false
  readonly hidden: boolean
  readonly origin: 'kernel' | `plugin:${string}` | 'user'  // derived from seed:key
}
fieldId is workspace-scoped — it never lives on a static handle

Seeded fieldIds are a function of (workspaceId, seedKey) (§4.2), but the exported schema consts are module-level values created before any workspace is pinned and reused across workspace switches. So fieldId exists only on registry entries, which are per-workspace by construction (the synthesis layer and the projector are both workspace-bound). Code handles carry seedKey instead, and anything that needs a concrete fieldId — #288's write path above all — resolves it through the registry snapshot bound to the transaction, keyed by the row's own workspaceId, never the ambient active-workspace pin. This rule is what keeps a write that lands during a workspace switch from stamping the wrong workspace's fieldId into synced data (§4.4).

Two things deliberately do not change:

  • Typed code handles survive. typesProp, isCollapsedProp, srsNextReviewDateProp, … remain exported consts that call sites pass to block.get/tx.setProperty for compile-time typing. What changes is how they're constructed: defineProperty(name, {codec, …}) becomes a seed declaration whose codec comes from preset.build(config) — the same construction path the projector uses, so the code handle and the block-fed registry entry are behaviorally identical by construction (§4.1).
  • ValuePreset's contract is unchanged — it already carries everything behavior needs — but it splits along the same data/presentation line the schema types already draw (propertySchema.ts:5-11): a codec core (id, build, configCodec, defaultValue, defaultConfig) that lives in the data layer, and a presentation half (Editor, Glyph, ConfigEditor, label, hideFromPicker) joined by preset id in the UI layer. Today the whole preset — codec factory included — lives in src/components/propertyEditors/kernelValuePresets.ts behind a UI toggle, which the data layer cannot import; seed declarations (§4.1) and headless surfaces (CLI, agent bridge) need the codec cores below React. The work beyond the split is registering presets for the codecs that don't have one yet (§10).
Decision

Definition fields live as ordinary block properties (not new columns): they ride properties_json, sync per the existing column-narrow pipeline, work in E2EE workspaces, and are editable through the existing property-schema block renderer. The one identity-adjacent thing that stays derived is origin — computed from seed:key, never stored separately.

4Seeding — code declares, the workspace materializes

4.1Seed declarations replace registrations

Kernel and plugin code stops registering schema objects and starts declaring seeds. The authoring surface keeps defineProperty's ergonomics and returns the typed handle call sites already use:

// before (properties.ts today)
export const isCollapsedProp = defineProperty<boolean>('system:collapsed', {
  codec: codecs.boolean, defaultValue: false, changeScope: ChangeScope.BlockDefault,
})

// after — a seed declaration; the handle is still a typed schema handle
export const isCollapsedProp = seedProperty<boolean>({
  seedKey: 'system:kernel-data/property/system:collapsed',
  name: 'system:collapsed',
  preset: 'boolean',            // ValuePreset id; config omitted for void-config presets
  changeScope: ChangeScope.BlockDefault,
  hidden: true,
})

// per-schema default when it differs from the preset's:
export const srsIntervalProp = seedProperty<number>({
  seedKey: 'system:srs-rescheduling/property/interval',
  name: 'interval',
  preset: 'number',
  defaultValue: 2,              // → property-schema:default; absent → preset.defaultValue
  changeScope: ChangeScope.BlockDefault,
})

seedProperty resolves the preset's codec core at module init (a data-layer static import once the §3 preset split lands), calls preset.build(config) for the handle's codec, and records the declaration in the module-level seed table its extension contributes via a new definitionSeedsFacet. For value types the preset can't express in TS (e.g. optionalIdentity<FocusedBlockLocation> building an untyped object codec), the helper takes the type parameter on faith — runtime behavior is identical; the compile-time type is an assertion, exactly as unsafeIdentity<T> is today.

Why one construction path matters

The registry entry is built by the projector from the block (preset.build(config)); the code handle is built by seedProperty from the same (preset, config). Because build must be deterministic in config (already a documented preset contract, valuePresets.ts:19-22), the two produce behaviorally identical codecs even though they are distinct objects. Consumers must not rely on schema-object identity across the two — today they don't: the projector already produces fresh objects per rebuild and every consumer joins by name.

4.2Identity: deterministic per-workspace ids

Decision

fieldId is the definition block id for every schema. Seeded definitions get deterministic per-workspace ids: uuidv5(`${workspaceId}:${seedKey}`, DEFINITION_SEED_NS) — the stateBlocks.ts:54 pattern, so two offline clients (and the local synthesis layer, §5.1) converge on the same row. Ids are per-workspace; cross-workspace copy/paste resolves by name, consistent with #288 §9.

Seed keys are stable strings, never derived from the current display name at runtime: <ownerToggleId>/property/<key> and <ownerToggleId>/type/<key>, e.g. system:kernel-data/property/types, system:srs-rescheduling/property/next-review-date. The owner segment reuses the togglable-boundary ids (src/facets/togglable.ts) that #289 §12 leans toward committing to as a public compatibility surface (its open question, stated leaning "commit — cheap now"); this design takes that leaning as settled — one stable-id vocabulary across both. Dynamic/user extensions use their extension block's toggle id (the block id) as the owner segment. The trailing key is frozen at introduction; renaming a schema's display name later changes property-schema:name, never the seed key or the block id.

4.3The seed pass

A single idempotent pass over the currently-visible seed declarations:

  1. Probe: one batched read of all seed ids (SELECT id, properties_json, deleted FROM blocks WHERE id IN (…) — on the order of 100 rows).
  2. Diff: for each declaration, compare the stored seed:revision and the seed-owned fields against the code seed (§6 rules).
  3. Write: one tx (chunked if large) that creates missing definition blocks (children of the Properties/Types pages, systemMint ids), restores soft-deleted ones, and rewrites stale ones. Steady state is zero writes — the pass is a read-only probe, same cost profile as ensureStateChild's fast path (stateBlocks.ts:100-109).

When it runs — the trigger set matters, because declarations arrive in waves. On cold start, at bootstrapWorkspace time (right after ensureSystemPages, workspaceBootstrap.ts:210) the only installed runtime is the static-data one (repo.tsx:33, resolved from staticDataExtensions' 11 entries with no toggle overrides) — most schema-registering plugins (attachments, quick-find, video-player, srs-review, extensions-settings, …) contribute their declarations only when the full app runtime installs in a post-paint effect (AppRuntimeProvider.tsx:114-119), and dynamic/user extensions later still. A bootstrap-only pass would therefore never seed them. So the pass runs:

  • at workspace bootstrap, over KERNEL-owned declarations only. The static-data runtime is toggle-blind and already carries plugin data extensions (srs, geo, backlinks, todo) — seeding everything visible then would materialize disabled plugins' definitions, contradicting the toggle policy below. Filtering the bootstrap trigger to owner = system:kernel-data keeps it correct by construction; safe mode gets kernel-only seeding for free.
  • again on every setFacetRuntime swap / definitionSeedsFacet change — an idempotent, diff-only re-probe over whatever declarations that toggle-aware runtime carries. This is when plugin seeds actually land (the app-runtime install), and it is also how dynamic/user/agent extensions seed: their declarations join the facet when the extension loads, the pass re-runs, their definitions upsert. No separate mechanism.
  • and when a hold releases. The holds below are awaited, not merely checked: a pass that parks on the drain or membership hold resumes when the signal fires (the onFirstSync hook shape) — otherwise a fresh device whose runtime swaps all complete before sync finishes would silently never seed that session.

Toggle policy: a disabled plugin's extension isn't resolved into the app runtime, so its declarations aren't visible and it neither seeds nor upgrades — its existing definition blocks simply persist (§5.3).

Scheduling: the pass is deep-idle work, not bootstrap-blocking. Registry correctness never waits on it — the synthesis layer serves entries regardless (§5.1); the pass only materializes rows. So every trigger above schedules the pass (the workspace-backfill idle pattern, repo.ts:1861-1899) rather than running it inline; app startup pays only the in-memory synthesis (microseconds) plus, in steady state, one deferred PK-batched SELECT whose diff short-circuits on seed:revision equality. The accepted transient: in a brand-new workspace, a field row (#288) can briefly reference a definition block that hasn't materialized yet — the id is already correct (deterministic), the block appears when the idle pass runs.

  • Hold until first sync drain (existing workspaces): on a device whose initial sync for the workspace hasn't completed, the probe would see seeded rows as locally absent and re-create them — resetting their created_at and destabilizing the §7 collision order fleet-wide. The pass therefore waits for the initial drain on not-freshly-created workspaces (the same rule #288 imposes on its backfill; "freshly created" is definable — resolveWorkspace returns it). Open implementation item: master has no per-workspace drain signal — onFirstSync wraps PowerSync's client-global hasSynced (internals/firstSync.ts:13-31), which a long-synced device joining an additional workspace passes while that workspace's rows are still downloading. The hold needs a per-workspace definition (e.g. global hasSynced AND at least one synced row for the workspace present, or a per-bucket checkpoint if PowerSync exposes one); #288's backfill hold shares this gap. Targeted §6.1 repairs deliberately bypass the hold — they update rows in place and never re-mint.
  • Scope: ChangeScope.Automation — machine-authored, durable, synced, not undoable (changeScope.ts:24). A user must not be able to ctrl-Z the kernel's schema definitions out of existence, and seed writes must not pollute the document undo stack.
  • Read-only workspaces: skipped by an explicit isReadOnly check (the gate scheduleWorkspaceBackfills uses, repo.ts:1862) — Automation scope alone would not stop the writes (readOnly: 'allow', changeScope.ts:65-69). And because a fresh device's role row may not have synced yet (App.tsx:76-84 deliberately defaults to writable), the pass on a not-freshly-created workspace additionally waits for the membership row — otherwise a viewer's first open would enqueue ~100 RLS-rejected creates into the quarantine. The drain-hold above already implies this. The synthesis layer (§5.1) serves the registry regardless.
  • Fresh-workspace cost: ~100 definition blocks created once, uploaded once. Existing-workspace upgrade cost: only the definitions whose seeds changed.

4.4Boot ordering — the #289 staging, instantiated

This is #289's kernel → materialize → assemble ordering applied to declarations:

StageWhat happensRegistry state

Workspace switch is the hard case, not cold start — key the buckets by workspace

The registry buckets are workspace-agnostic today, and the projector teardown that clears them runs in React effect cleanup — after the next workspace's bootstrap has already executed inside the suspended layout promise (App.tsx:59-127 vs AppRuntimeProvider.tsx:239-243). userSchemasService.ts:233-245 documents this exact leak for one write path and pins its workspace by hand — and a mere pin-time clear would not close it: the old workspace's projector is still live through that window (its disposed guard explicitly does not stop a straggler, projectorRuntime.ts:27-32), so a sync-delivered row could re-publish stale entries right after the clear. Once fieldIds are synced data (#288), any lookup served from the old workspace's entries would stamp workspace-A fieldIds into workspace-B's rows: permanent, synced, cross-workspace dangling ids. The structural fix, rather than clearing discipline:

  • Runtime buckets become workspace-keyed. A publisher writes under the workspace it is pinned to (the projector already pins at start(), projectorRuntime.ts:156; synthesis is computed per workspace by definition); the registry read filters to the active workspace's keys. A straggler publish from the outgoing workspace lands under the old key and is invisible to the new workspace — harmless by construction, no clearing choreography, and the hand-pin in addSchema becomes redundant rather than load-bearing. (Minimal encoding: fold the workspace into the runtime-contribution source id, e.g. user-data@<wsId>; a first-class key dimension is the cleaner variant — implementation's choice. Precedence between a workspace's seed-synth and user-data buckets is Map insertion order, facet.ts:246-248 — publish synthesis-then-projector per workspace and update in place, never delete-then-recreate.)
  • Per-row fieldId resolution as defense-in-depth: writes resolve fieldIds from the row's own workspaceId via the tx-bound registry snapshot (§3 callout — entries carry workspaceId so the snapshot can enforce it), so even a code path that somehow crosses workspaces cannot borrow the pin.
  • Handles never cache: code handles carry seedKey, not fieldId (§3), so there is nothing stale to read.

Edge cases the pin step must define: setActiveWorkspaceId(null) (legal today) leaves the read filter matching no workspace-keyed bucket — the registry falls back to stage-0 synthesis, whose entries are degraded by declaration: no fieldId/workspaceId (the §3 interface marks them optional-absent in this state; nothing that consumes them can run without a workspace). Repos constructed without a facet runtime (installKernelRuntime: false, tests/tooling) make the pin's registry step a no-op.

This dissolves #288 §11-12's two-phase kernel-first bootstrap concern: phase 1 ("project code-defined schemas' keys, recognizable by code-known fieldIds") stops needing any DB state at all — kernel/plugin fieldIds are computable from (workspaceId, seedKey) with zero rows synced. If #289's boot restage later moves projector start pre-mount, stage 2 just happens earlier; nothing here depends on it.

5The unified registry

5.1One projector, two layers

The registry keeps its existing plumbing — facet runtime buckets feeding the propertySchemas rebuild step (facetBridge.ts:239) — but the inputs change from three sources (kernel registrations, plugin registrations + type-lift, projector bucket) to two layers:

  • Synthesis layer ('seed-synth' bucket, workspace-keyed per §4.4): entries computed from the in-memory seed declarations — a pure function of (declarations, workspaceId), no DB read, no async. Realized as a FacetBridge rebuild step with definitionSeedsFacet as its input, so it recomputes on every runtime swap and seed-facet change (steps re-run exactly then, facetBridge.ts:150-160) — declarations arriving in later waves (§4.3) enter synthesis the moment they load. The pin is simply the trigger that changes workspaceId. Two ordering requirements the implementation must not miss: (a) the bucket is published durable, and marked before the projector's first publish — swap adoption copies durable buckets in mark order before the step loop runs (facetBridge.ts:147-150, facet.ts:336-351), so a non-durable seed-synth re-created by the step would re-insert after the adopted projector bucket and silently invert this section's precedence on every swap of every session (Map insertion order, facet.ts:293-301); (b) the synthesis step is ordered before the propertySchemas step in the rebuild list — per-facet change listeners are wired only after the step loop (facetBridge.ts:150-167), so step order, not notification, is what makes a swap coherent. This is what makes definitions available "before first sync" (requirement 3): on a fresh device with an empty DB the registry is complete for every loaded declaration wave, no rows required.
  • Block layer ('user-data' bucket, the existing projector): authoritative once primed. The projector generalizes from "user schemas" to "all definitions": it publishes an entry for every property-schema block, seeded or user-authored.

Why consumers never wait on the DB even though blocks are the source of truth: consumers read the registry, never definition blocks directly — and for seeded definitions the synthesis entry and the block-fed entry are the same definition by construction (same deterministic fieldId, same preset.build(config) codec, same fields), so an entry served pre-prime is identical to what the block would give. The DB read is verification-and-override, not a prerequisite. What only the DB knows — a newer seed revision written by a newer client, the archived flag (§7), and user-created definitions, which have no code counterpart — takes effect when the projector primes, which is exactly the timing user schemas already have on master today. The seed pass (writing rows) serves the block affordances — backlink targets, #288 field rows, the Properties-page UI — which nothing resolves schemas through; that is what makes it idle-schedulable (§4.3). One consequence must be stated as a rule: the pre-prime window is read-safe, not identity-write-safe. Because archived flags and user definitions arrive only at prime, a name resolved pre-prime can differ from the §7 winner (e.g. a reused name resolving to an archived seed's fieldId) — harmless for display, wrong to persist. So any write that stamps a name→fieldId resolution into synced data (#288's derive/materialize processors, field-row creation) gates on the definition projector having primed for the workspace — one subscription tick over ~100 rows, and the same schema-readiness gate shape #288's machinery already carries for its backfill.

Two housekeeping notes: workspace-keying does not supersede the existing teardown contract — the projector's dispose keeps clearing its bucket (projectorRuntime.ts:181-195; facet.ts:332-335), which bounds retained buckets and raw-view pollution; and one consumer reads raw buckets beneath the workspace read-filter — the agent bridge's describe-runtime enumerates contributions by bucket id — so it filters by the source id's workspace suffix (or shows all, labeled) rather than pretending the filter applies.

Runtime buckets append after static contributions and keyed-map facets are last-wins (facet.ts:242-249, :102-123 — the shadowing order #289 §2.4 verified), so publishing seed-synth before user-data gives block-fed entries precedence by construction. One guard runs projector-side: a seeded block whose seed:revision is older than the loaded code's revision projects the code synthesis instead of the stale block — and doubles as the repair trigger: observing such a row schedules a targeted seed-pass diff for that id (§6.1) through the same serialized, debounced job queue the workspace backfills use, so a stale bag that syncs in mid-session is repaired without waiting for the next workspace open. Scheduling is suppressed on read-only clients (the pass couldn't write; without suppression the guard would re-schedule a no-op on every rebuild). The registry map gains an id-keyed index alongside the name-keyed one — findSchemaByFieldId being O(schemas) is already flagged as a risk in #288 §12.

5.2What dies

  • dies mergeLiftedSchemas's name-merge and the TypeContribution.properties type-lift (refProjection.ts:29-57): the rebuild step reads one merged map from the buckets. (changedRefSchemaNames and the reprojection scheduling survive unchanged — they compare any two maps.)
  • dies direct propertySchemasFacet.of registrations — all 29 call sites in src/plugins (and, in the types slice, all 12 typesFacet.of sites), plus the ~30 sites in agent-extensions/ and the tutorial extension templates (§2.1): dynamic/user extensions that register schemas through the facet must be rewritten to seed declarations (near-1:1) or to plain data (addSchema → an ordinary user property-schema block — the better home for most extension schemas). The same conversion covers their propertyEditorOverridesFacet.of registrations (~19 in agent-extensions/) — the §8 name join those overrides rely on dies in the same slice, so an unconverted override's settings editor stops rendering. No compat path is kept; affected extension blocks are enumerable beforehand (content LIKE any of propertySchemasFacet.of / typesFacet.of / propertyEditorOverridesFacet.of) and the fleet is small enough to coordinate. The facets themselves survive as the internal transport between buckets and the bridge.
  • dies the "kernel/plugin schemas have no backing block" asymmetry: UserSchemasService.getSchemaBlockId's "returns undefined for kernel/plugin schemas" caveat (userSchemasService.ts:144-147), and eventually the service's "user" prefix.

"Absent from the registry" stops being ambiguous between "toggled off" and "deleted".

6Seed upgrades & write policy

6.1Reconciliation rules

The genuinely new problem class: app v(N+1) changes a kernel schema's seed while the block may have been touched by an older client or a user. Following #289 §9's provenance approach, adapted from hashes to monotonic per-definition revisions (why revisions: §14):

Seed pass observesAction
Block missing / soft-deletedCreate / restore with current seed payload, stamp seed:revision.
seed:revision < code revisionUpgrade: rewrite all seed-owned fields to the current seed, stamp the new revision. One tx, one column-narrow PATCH — per-column LWW takes the whole write from one winner, so mixed-version clients can't interleave field-level.
seed:revision = code revision, fields differDrift (a user edit through the panel, an agent-bridge write, an import): repair seed-owned fields back to the seed. Self-healing, same philosophy as #288 §5's consistency sweep.
seed:revision > code revisionLeave untouched — a newer client owns this definition. The registry's projector-side guard (§5.1) does not apply (the block is newer, not staler); the older app runs against the newer definition where it can (known preset: configCodec validates → newer config in effect), and where it can't — unknown preset, or config that fails the older configCodec — the block-fed entry drops out and the older client's own synthesis entry (its old payload) serves. Behaviorally safe either way; degrade tests should assert the fallback-to-synthesis case, not "runs against the newer definition".

When repair actually happens. The full pass runs on the §4.3 triggers (workspace open + runtime swaps + hold releases), and the projector guard schedules a targeted diff whenever a stale seed-keyed row arrives via sync mid-session (§5.1) — so on any writable client with the owning code loaded, a stale bag is repaired within one observation cycle (read-only clients never repair — they serve synthesis locally and leave the block; a workspace whose only current-code member is a viewer keeps the stale block until a writer opens it). The honest residual: a definition whose owning plugin is loaded nowhere in the fleet is never upgraded or repaired — its blocks simply persist at their last-written payload, which is also the correct behavior (there is no code left to say otherwise). Consequently a childBacked flip (a seed-revision bump, §6.2) propagates with a bounded disagreement window — clients read the flag off the block, and the block converges via sync + repair, not atomically; #288's machinery must tolerate that window exactly as it already tolerates its own flag-flip rollout windows.

The seed source of truth stays in code; blocks are its materialization. A seed change is a code change (revision bump in the declaration) that materializes per-workspace on the §4.3 triggers — there is no fleet data-migration step, and unedited workspaces converge via ordinary sync of the one upgraded row.

6.2Field ownership & write policy by origin

For seeded definitions, every definition field is seed-owned except definition:archived (user-owned, §3/§7): name, preset, config, changeScope, hidden — and childBacked once it lands (#288-owned field, §3). Per-field user ownership beyond that (overriding hidden, recoloring a plugin type) is a deliberate v2: it needs a per-field ownership vocabulary and merge rules — §15. User-authored definitions keep today's full editability.

7Name resolution & collisions

Ids cannot collide; display names can. Names are load-bearing in three places: the properties_json cell key, alias-entry ([[status]] typed by hand — #288 §7's field-row grammar resolves the schema-name map first), and every name-keyed consumer in §2.1. Today mergeLiftedSchemas resolves collisions last-wins-by-load-order with a warning — nondeterministic across clients with different plugin sets, which #288 §6 flags as a hard error under child-backing (the cell key is the name while child identity is the fieldId; a per-client winner strands the loser's field rows).

Decision

Because every definition is now a synced block, collision resolution can finally be deterministic fleet-wide: the name winner is the earliest live, non-archived definition — ordered by (created_at, id), computed over synced block state only. Locally-varying facts (is the plugin installed? enabled? does its preset resolve?) never enter the competition: toggles are per-user and presets per-install, so any owner- or codec-dependent rule would make the winner differ per client — precisely #288's hard error. Local state may gate UX (what the creation dialog offers), never the winner. Losers stay registered by fieldId (child rows keep classifying, backlinks keep resolving) but are excluded from the name map and the cell projection, and surface in the Properties page UI as shadowed — rename to activate. Earliest-wins protects existing user data from newly-arriving plugin seeds: a user's long-standing status schema is not hijacked when a plugin introducing status is installed. (In a fresh workspace seeds usually predate meaningful user data — but not guaranteedly, since the pass is idle-scheduled (§4.3); when a fast first-minute user schema out-dates a seed, earliest-wins hands the name to the user's row, which is the intended bias, not a failure.)

  • Dead plugins release names through definition:archived (§3), not through local dormancy. A dead plugin's definitions would otherwise squat their names forever (creation-prevention + read-only + persist-after-uninstall compose into a permanent reject — worse than today, where toggling off frees the name). Archiving is an explicit, synced user action: the archived definition leaves the name map and pickers on every client identically, and the seed pass never un-archives — concretely: every seed write preserves the tx-read archived value in the bag it writes, since definition fields share one properties_json column and a repair that wrote the bag blind would clobber the flag under column LWW. Residual race, accepted and stated: a seed upgrade on client B concurrent with an archive on client A can still lose the archive (B faithfully preserves its own stale archived: false replica and wins the column) — rare (upgrades happen at deploy cadence), self-evident (the name reappears), re-archivable, and closed for good when definition fields themselves ride #288's per-key child-backing. (If it bites sooner, the escape is moving the flag to its own row — an archive-marker child — for row-level LWW independence.) The creation dialog offers it when it rejects a name whose owner is locally absent. Name reuse after archiving is era-dependent: once #288's field rows exist for the key, the old values keep classifying by fieldId and cleanly leave the cell to the new schema; before that, cell values carry no identity, so a new same-name schema reads the archived schema's stored values as its own — the same semantics any name reuse has today. The creation dialog must say so when reusing an archived name pre-materialization (adopt the values, or pick a fresh name). Consequences to state plainly: a client that still runs the plugin will find its schema nameless — the plugin degrades until re-enabled and un-archived, surfaced in settings; and un-archiving restores competition, where the older definition reclaims the name and shadows the newcomer — under #288 that re-keys the cell to the old definition's field rows, so the shadowed schema's values leave the cell view (visible in the tree, surfaced in UI) until the collision is resolved by rename.
  • The order must be protected from the design's own machinery: created_at is an ordinary synced column, so anything that re-creates a seeded row resets it and can flip an established winner fleet-wide. The §4.3 drain-hold exists for exactly this (a fresh device must not "re-seed" rows it merely hasn't downloaded yet), and restore-vs-create in the seed pass preserves created_at on soft-deleted rows by construction (restore doesn't re-mint the row). Residual, accepted for now: the rule trusts client clocks — a skewed or hand-written created_at can back-date a definition into winning a name. For the current two-user fleet that's a non-threat; revisit (e.g. server-stamped ordering) if the fleet grows.
  • Prevention first: addSchema and the rename flow reject a name already claimed by any live, non-archived definition — same test as the competition, so prevention and the winner can never disagree. Exact-match names; case-insensitivity is an open question (§15). The collision rule is for the unpreventable case: two definitions created concurrently offline, or a plugin seed arriving into a workspace that already used the name.
  • The audit says this is real: srs-rescheduling seeds bare, generic names — interval, factor, grade, archived, next-review-date, review-count, snapshot-history (srs-rescheduling/schema.ts:19-55) — exactly the names users would pick. New plugin schemas should namespace (srs:interval) as most already do; renaming existing plugin names is a data migration (the cell key changes) and is explicitly not attempted here — earliest-wins keeps the status quo working either way.

8Editor overrides & presentation joins

PropertyEditorOverride joins schemas by name today (propertySchema.ts:32; consumed via uis.get(name) in resolvePropertyDisplay, defaults.tsx:440). The inventory: 20 overrides — 13 kernel hidden:true markers, the types autocomplete editor, and 6 plugin config-property editors.

Decision

Two moves. (1) The 13 kernel hidden:true overrides dissolve into the definitions themselves — hidden is now a definition field (§3), synced and classifiable, which #288's hidden-field-row filtering needs anyway (its §10 notes the curated hidden list must grow at the kernel flip; a definition field replaces the curated list wholesale). (2) The remaining editor overrides join by fieldId instead of name: code-declared overrides reference the seed handle (overrideFor(typesProp, {Editor: …})) — i.e. they carry the seedKey, and the join resolves seedKey → definition id lazily at render time through the workspace-scoped registry (the same discipline as §3's fieldId callout: no override map is ever frozen against a workspace, so a switch can't serve stale ids). Immune to display-name changes once renames unlock; the name join dies with the cutover slice.

resolvePropertyDisplay's preset join (presets.get(codec.type)) is untouched — editor resolution stays keyed by the codec's type discriminator, which is orthogonal to definition identity.

9Types get the same move

Everything above applies to block types with one extra wrinkle: type ids are stored data. Every block's types property and the block_types index hold strings like 'page', 'todo', 'daily-note' — for user types the string is the definition block id; for kernel/plugin types it's a legacy constant. Rewriting those strings across every workspace would be a huge, risky data migration for zero user-visible gain, so:

Decision

block-type definition blocks gain block-type:type-id — the string written into typesProp. Absent (every existing user type) → the block id, preserving the block-id = type-id rule. Seeded kernel/plugin types set it to their constant ('page', 'todo', …). This is the permanent identity model, not a compat shim: a stable readable string is the better type identity — it can be a compile-time constant, which per-workspace uuids structurally cannot (a uuid-unification data migration is cheap on data at this fleet size but would force every types: ['daily-note'] query, every cross-plugin type-constant import, and every hasBlockType check through per-workspace runtime resolution, forever; #289's coupling taxonomy counts those constants as "retargets for free" precisely because they're stable strings). There is nothing to migrate or remove later. The type registry stays keyed by type id; getTypeBlockId becomes total (every registered type has a backing block). Whether typesProp values should become refs to definition blocks is out of scope — flagged as future work, unchanged from the project brief.

type-id claims need a rule — it's a user-writable property now

Once block-type:type-id exists, "ids cannot collide" (§7) no longer covers types: any block — a cross-workspace paste, an import, an agent-bridge write — could claim 'page', and the projector would publish both contributions with last-wins resolved by subscription row order (projectorRuntime.ts publishes every built row; keyedMapFacet is last-wins), nondeterministically deciding addTypeInTx validation and every renderer lookup. The rule: a block-type:type-id differing from the block's own id is honored only when the block's identity checks out against the deterministic-id equation — block.id === uuidv5(`${workspaceId}:${seed:key}`, DEFINITION_SEED_NS) — which every client can verify with zero knowledge of the owning plugin (merely writing a seed:key property proves nothing; the pass only probes/repairs ids derived from local declarations, so a fabricated key on a random block is never repaired by anyone). On any block failing the check, the claim is ignored and the type projects under its block id. Residual: a fully self-consistent fake seed (correct uuid, invented toggle id) is repaired by nobody and falls to the §7 clock-trusting tie-break — accepted under the small-fleet stance, stated rather than hidden.

The rest is mechanical application of §3–§6:

  • New definition fields on block-type blocks: block-type:type-id, block-type:hide-from-completion (today code-only, blockType.ts:26), plus seed:key/seed:revision. Existing fields (label, description, color, hide-from-block-display, properties refList) already cover the rest of TypeContribution.
  • TypeContribution.properties → block-type:properties refList of definition-block ids. Seed declarations reference the property seed handles; the seeder resolves them to per-workspace ids. The two lift-only schemas the audit found (status, daily-note:date) get their own property seeds, which their type seeds reference — the lift path they currently ride dies.
  • The prefs/ui-state container types (pluginPrefsExtension/pluginUIStateExtension, 7 call sites) seed the merged shape — the helper-injected hideFromCompletion: true is part of the seed payload, not re-injected at registration.
  • addTypeInTx keeps working unmodified: it validates against types.get(typeId) and encodes caller-supplied initial values by schema name (typeTagger.ts:56, :85) — both maps still exist, just block-fed. snapshotTypeRegistries (repo.ts:1579) is shape-unchanged.
  • Sequencing: types cut over after property schemas (their own slice, §13) — the type projector already depends on the schema projector (userTypesService.ts:132), and seeded type declarations need property seeds to exist first.
  • Alignment with PR #293 (type label = alias): #293 makes a user type claim its label as an alias (createTypeBlock sets alias = [label]), so [[Label]] resolves to the type block and type names become workspace-unique via the alias-uniqueness trigger. Seeded types follow suit — claim the label alias when it is free, skip on collision (an existing workspace may already have a [[Todo]] page; the definition then lives without the alias, navigable from the Types page, surfaced rather than fought over). Kernel infrastructure types (page, panel, …) claim no aliases. Mechanics of "skip": the uniqueness backstop is the client-side SQLite trigger block_aliases_workspace_alias_unique (clientSchema.ts:833-852) — RAISE(ABORT) rolls back the entire local tx synchronously, before anything is queued for upload. So the claim needs a local pre-check and must be a separate write from the definition create/repair, wrapped in a try/catch for the structured alias.collision: a combined tx would abort the seed row's own create, silently and locally. Timing residual, stated: the pass is idle-scheduled, so [[Todo]] typed in the window before it runs mints a rival alias seat (lookup-first resolution — the race #293 closes for user types at creation); the seeded type then stays alias-less with the rival surfaced for manual merge. Two knock-ons: seed repair of a label must go through #293's label→content→alias lockstep (writeBlockTypeLabel + aliasSyncProcessor) — where a re-claim can hit the same collision rejection and follows the same pre-check rule; and the alias-uniqueness trigger becomes a DB-level adjacent guard to §7's registry-level rule for type labels — one mechanism per layer, don't add a third.

9.1Renaming — display edits over stable identity

Identity (seedKey/block id) never moves, so renaming is always a display-field edit — but the two kinds differ in what hangs off the name today:

  • Type labels are pure display — stored membership uses the type id, never the label. User-type renames work today and #293 keeps content + the [[label]] alias in lockstep. Seeded type labels are seed-owned in v1: an upstream rename ships as a seed revision bump whose repair honors the same lockstep.
  • Property names are the cell key until #288's end state, and this project ships no rename machinery at all: the rename-reproject pass is #288's deliverable (its slice B2), and property renames — user and seeded alike — stay blocked outright until it ships (without it a rename is the "mass silent unset" #288 §7 documents). Once B2 exists, seeded property renames additionally ship as seed revision bumps. Child rows and references are untouched either way (fieldId identity).
  • The endgame that makes renames free: once #288's children-authoritative end state makes the cell a derived local projection, re-keying the cell by fieldId instead of name becomes a local reprojection rather than a fleet data migration — renames then need no reproject pass at all, and the §7 cell-key collision problem dissolves (names become pure display + alias-entry sugar). Deliberately not part of this project (value keying stays untouched here — that's what keeps the rollout non-breaking); flagged as the designated successor in §15.

12Edge cases & carve-outs

  • Non-document scopes seed too. UiState / UserPrefs / Automation schemas (9 kernel + 12 plugin) become seeded definitions like everything else — definitions are ordinary BlockDefault data even where the values they describe are per-user; once #288's flag lands (§3), its permanent childBacked: false carve-out gets stamped in their seeds. No circularity: reads and writes of scoped values go through code handles (block.set(schema, …)), which never consult the registry; the seed pass itself writes definition blocks using kernel property handles, not registry lookups.
  • E2EE workspaces: no new constraints. Definition blocks are ordinary blocks; their properties ride the encrypted properties_json envelope; the projector reads locally-materialized plaintext. (Contrast with #288's reference_target_id gate — nothing here adds a synced plaintext column.)
  • Read-only workspaces: seed pass skipped; synthesis layer serves kernel/plugin entries; user definitions arrive as synced blocks. A viewer client never writes definitions.
  • Safe mode / plugin toggled off: the plugin's seeds and presets don't load; its definition blocks persist and project as kernel-preset entries or declarations (§5.3). Nothing is deleted by absence — matching the projector's existing "absence is not deletion" stance.
  • Old clients during rollout: skew is tab-lifetime (web app) and benign — pre-cutover clients keep their code registries and ignore the new definition fields; seeded blocks are inert rows to them; a stray delete is restored by the next current client's seed pass.
  • Cross-workspace copy/paste of a definition block: the pasted block carries seed:key for a seed the destination already has → it's a duplicate name at a wrong id; the collision rule (§7) shadows it and the UI surfaces it. Paste of field rows across workspaces resolves by name per #288 §9. A paste-time strip of seed:* props (plus block-type:type-id and the label alias, whose uniqueness trigger would otherwise reject the paste) keeps the common case clean.
  • Workspace deletion/reset: nothing special — seeds re-materialize on next open, same as kernel pages.

14Rejected alternatives

  • Name-based identity everywhere (keep merging by name, add childBacked to code): fails #288's requirements 1-2 outright — names collide across sources per-client, and code constants fork data across plugin versions. This is the status quo the migration doc already rejected.
  • Global (cross-workspace) definition ids: a block id is unique per row; the same definition in two workspaces would collide in one local DB. Per-workspace ids + name-based cross-workspace resolution match #288 §9 and the existing state-block precedent.
  • Registry entries synthesized from code only, blocks as passive mirrors (seed pass writes blocks but registry never reads them): loses the entire point — user edits of plugin definitions (v2), missing-plugin classification, and #288's durable childBacked all need the block to be authoritative.
  • Columns instead of properties for the new definition fields: rejected in §3's decision aside — properties ride existing machinery.
  • Hash-based seed provenance (#289's base-hash) instead of revisions: hashes detect difference but not direction — an old client couldn't tell "my seed is stale" from "the block is newer than my seed" and would fight newer clients. Revisions give ordering; drift detection falls out of the revision-equal + fields-differ case (§6.1). (#289's hash approach remains right for code, where there is no total order on user edits.)
  • Seeding lazily on first use (definition materialized when a value is first written): leaves the registry incomplete for pickers/panel, makes fieldIds appear at unpredictable times mid-#288-migration, and saves ~100 rows once. Not worth the nondeterminism.
  • Migrating kernel/plugin type-id strings to block ids: a fleet-wide rewrite of every types array and the block_types index for cosmetic uniformity; block-type:type-id costs one property (§9).

15Open questions

  1. Per-field user ownership (v2): which seeded fields become user-editable first (likely hidden, type color/label), and how ownership is recorded (per-field marker vs an "edited fields" set) so the seed pass stops repairing them. #289 §8's "edit this type" affordance lands here. Note the §5.1 code/block equivalence weakens for exactly the overridden fields — a user-edited value exists only in the block, so the pre-prime window serves the seed's value (same class as archived today); v2 either accepts that bounded flash for display-only fields or gates them on projector prime.
  2. Case sensitivity of name collision/prevention — exact match (status quo of the alias map) or case-insensitive? Affects §7's prevention check only; the deterministic rule works either way.
  3. seed:hash alongside seed:revision: field-compare at revision-equal is cheap at this scale; a stored payload hash would make drift detection O(1) and align vocabulary with #289. Decide in slice B implementation.
  4. Picker exposure of declarations: should AddPropertyForm's picker offer schemas whose plugin is disabled (they're fully functional if kernel-preset-built, §5.3)? Leaning yes for kernel-preset ones, no for declaration-only.
  5. New-plugin type ids: should plugins introduced after slice D use block ids as type ids (no readable constant)? Leaning no — §9's argument (compile-time constants, retarget-for-free string references) applies to new types as much as grandfathered ones; readable string ids stay the norm, and "absent → block id" remains a user-created-types convenience.
  6. Bulk-archive UX: per-definition archiving is v1 (§7); a "this plugin is gone — archive all N of its definitions" affordance on the Properties page is the remaining convenience. Nothing breaks meanwhile.
  7. Id-keyed cell (the rename endgame, §9.1): after #288's children-authoritative end state makes the cell a derived local projection, re-key it by fieldId — renames become pure display edits and the cell-key half of §7's collision problem dissolves. Independent follow-up sequenced behind #288; whether the non-document carve-out scopes (which stay cell-primary) migrate too or keep name keys is part of that design.
