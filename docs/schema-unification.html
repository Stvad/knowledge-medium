Design · knowledge-medium

Schema unification: definitions are blocks, behavior is presets, code seeds

Collapse the two property-schema (and block-type) worlds — inline code objects merged by name vs. definition blocks projected by id — into one: every definition is a synced block with a block-id identity; code registers behavior (ValuePresets) and seeds definitions instead of registering them.

TL;DR

Today kernel/plugin schemas are inline PropertySchema objects merged by name (mergeLiftedSchemas, refProjection.ts:29) while user schemas are property-schema blocks projected by block id — two identity spaces, two registration paths, and a set of workarounds the properties-as-blocks migration (PR #288 §6) can no longer absorb: synthetic property:<name> fieldIds leaking into the reference index, a childBacked flag with no durable home for plugins, cross-source name collisions, and a two-phase cold-start bootstrap.

The end state: one durable source of definitions (blocks), one identity (block id), one behavioral contract (ValuePresets). Kernel and plugin schemas become seeded definition blocks with deterministic per-workspace ids (uuidv5, the stateBlocks.ts pattern); every codec — code handle, user schema, missing-plugin fallback — is built through the same preset.build(config) path user schemas already ride (tryBuildSchema, userSchemasService.ts:52). In v1 the blocks are identity/metadata mirrors, not behavioral authority: behavior comes from the code declaration where one is loaded, from the block only as fallback (§5.1). A code-resident synthesis layer makes declaration behavior available synchronously before any block exists; identity-bearing resolution waits for the definition projector's first complete local snapshot so synthesis cannot outrank stored winner metadata. Types get the same move (§9). The preset-expressibility gate passes — every registered schema is expressible as (preset, config) (§10).

The runtime types layer in three — the design's identity rules are encoded in the type system, not in optional fields. The behavioral core is the ambient registry's value type (and all the projector builds for user rows, which carry no seed:key); the static typed handle extends it for code-declared schemas (module-level, workspace-agnostic, no identity); the workspace-resolved entry is what identity consumers get, and it is obtainable only from the §7 resolution primitive:

// The behavioral core — the ambient registry's (repo.propertySchemas) value type.
// Exactly the shape tryBuildSchema builds for user rows today; user-authored
// schemas have no seed:key, so the common shape carries none.
interface PropertySchemaEntry<T> {
  readonly name: string
  readonly codec: Codec<T>        // preset.build(config)
  readonly defaultValue: T        // declaration/row default, else preset.defaultValue
  readonly changeScope: ChangeScope
}

// The static typed handle — what seedProperty returns and call sites pass to
// block.get / tx.setProperty. Code-declared schemas only. Module-level, created
// before any workspace is pinned, reused across switches: it carries NO
// identity, so there is nothing to cache stale.
interface PropertyHandle<T> extends PropertySchemaEntry<T> {
  readonly seedKey: string        // stable identity KEY (§4.2) — resolved per-workspace at the boundary
}

// The workspace-resolved registry entry — what identity consumers (#288 stamping,
// classification, the winner gate) receive. Obtainable ONLY from the tx/workspace-
// bound resolution primitive (§7): branded / internally constructed, so holding one
// PROVES resolution ran against the row's own workspace. Identity fields are
// required — there is no "entry with missing identity" state to represent; the
// no-workspace stage-0 registry (§4.4) serves plain PropertySchemaEntry values and
// simply cannot produce this type.
interface ResolvedPropertySchema<T> {
  readonly name: string
  readonly fieldId: string        // the definition block id — THE identity
  readonly workspaceId: string
  readonly codec: Codec<T>
  readonly defaultValue: T        // decoded property-schema:default, else preset.defaultValue
  readonly changeScope: ChangeScope
  readonly hidden: boolean
  readonly origin: 'kernel' | `plugin:${string}` | 'user'  // derived from seed:key ('user' rows carry none)
}

Two deliberate omissions: childBacked is not on either type — it is #288's field end to end (the reserved row above; sequencing note below), joining ResolvedPropertySchema when that project lands. And the §5.3 definition-metadata index rows are a third, codec-less shape with their own type — keeping codec non-optional on both types above is what lets every repo.propertySchemas consumer dereference it unconditionally (§5.3).

fieldId is workspace-scoped — identity is a capability of the resolution path, not a field you might find

Seeded fieldIds are a function of (workspaceId, seedKey) (§4.2), but the exported schema consts are module-level values created before any workspace is pinned and reused across workspace switches — so handles carry seedKey and structurally cannot carry a fieldId. The ambient name-keyed registry (repo.propertySchemas) serves PropertySchemaEntry values — the behavioral core, which is also all the projector builds for user rows — and never carries identity either, even post-pin: anything that needs a concrete fieldId — #288's write path above all — must go through the §7 resolution primitive, bound to the transaction and keyed by the row's own workspaceId, never the ambient active-workspace pin. Because ResolvedPropertySchema is constructible only there, "resolved against the right workspace" is a compile-time property of holding the type — this is what keeps a write that lands during a workspace switch from stamping the wrong workspace's fieldId into synced data (§4.4), with no optional-field discipline to police.

Per the agreed ordering, this project lands in full before #288's slices begin. childBacked is therefore #288's field end to end: no slice here reads, writes, migrates, or types it. What this design contributes is only its future home (a definition property, reserved above) and its future flip mechanism (a seed-revision bump through the §6 upgrade path). Every mention of childBacked elsewhere in this doc (§5.1, §6.1–§6.2, §11.1, §12) is forward-looking requirements traceability for #288, not scope of this project.

Two things deliberately do not change:

  • Typed code handles survive. typesProp, isCollapsedProp, srsNextReviewDateProp, … remain exported consts that call sites pass to block.get/tx.setProperty for compile-time typing. What changes is how they're constructed: defineProperty(name, {codec, …}) becomes a seed declaration whose codec comes from preset.build(config) — the same construction path the projector uses, so the code handle and the block-fed registry entry are behaviorally identical by construction (§4.1).
  • ValuePreset's behavioral contract is unchanged, but Slice A split it along the same data/presentation line the schema types already draw (propertySchema.ts:5-11): codec cores now live in src/data/kernelValuePresetCores.ts and plugin data extensions, while presentation halves live in UI/plugin trees and join by preset id. Seed declarations (§4.1) and headless surfaces (CLI, agent bridge) can therefore use codec cores without importing React.
Decision

Definition fields live as ordinary block properties (not new columns): they ride properties_json, sync per the existing column-narrow pipeline, work in E2EE workspaces, and are editable through the existing property-schema block renderer. The one identity-adjacent thing that stays derived is origin — computed from seed:key, never stored separately.

4Seeding — code declares, the workspace materializes

4.1Seed declarations replace registrations

Kernel and plugin code stops registering schema objects and starts declaring seeds. The authoring surface keeps defineProperty's ergonomics and returns the typed handle call sites already use:

// before (properties.ts today)
export const isCollapsedProp = defineProperty<boolean>('system:collapsed', {
  codec: codecs.boolean, defaultValue: false, changeScope: ChangeScope.BlockDefault,
})

// after — a seed declaration; the handle is still a typed schema handle
export const isCollapsedProp = seedProperty({
  seedKey: 'system:kernel-data/property/system:collapsed',
  revision: 1,
  name: 'system:collapsed',
  preset: 'boolean',            // ValuePreset id; config omitted for void-config presets
  changeScope: ChangeScope.BlockDefault,
  hidden: true,
})

// per-schema default when it differs from the preset's:
export const srsIntervalProp = seedProperty({
  seedKey: 'system:srs-rescheduling/property/interval',
  revision: 1,
  name: 'interval',
  preset: 'number',
  defaultValue: 2,              // → property-schema:default; absent → preset.defaultValue
  changeScope: ChangeScope.BlockDefault,
})

seedProperty resolves the preset's codec core at module init — kernel cores by id (a literal-keyed data-layer map); plugin-owned cores (§10's six custom codecs) are passed by object, the core object passed inline for construction since the plugin module owns both; the declaration carries the built codec, normalized config, and preset id (the seeded row still stores the preset id string) — so handle codec construction never crosses a runtime boundary. Kernel ids own and infer their value/config types, so a caller cannot claim number while naming the string preset. The helper calls preset.build(config) for the handle's codec, and records the declaration in the module-level seed table its extension contributes via a new definitionSeedsFacet. Deliberately broad json/optional-json cores retain assertion-style type parameters for engine-owned object shapes. The user-facing enum core infers string because its empty unset sentinel is part of the runtime range; code-declared strict string subtypes use strict-enum with typed options and an explicit default. string-list infers readonly string[]; the historical mutable aliasesProp contract is narrowed locally instead of widening the public overload. Runtime behavior is identical; the remaining compile-time assertions follow the same deliberate boundary as unsafeIdentity<T>.

Why one construction path matters

For a locally-declared definition the behavioral registry entry is the code declaration (§5.1) — but the same definition gets built from its bag elsewhere: the §5.1 fallback on a client without the plugin, and user rows through the projector generally. Because build must be deterministic in config (already a documented preset contract, valuePresets.ts:19-22), a declaration-built codec and a bag-built codec are behaviorally identical — clients with and without the plugin agree on semantics even though they construct distinct objects. Consumers must not rely on schema-object identity across builds — today they don't: the projector already produces fresh objects per rebuild and every consumer joins by name.

4.2Identity: deterministic per-workspace ids

Decision

fieldId is the definition block id for every schema. Seeded definitions get deterministic per-workspace ids: uuidv5(`${workspaceId}:${seedKey}`, DEFINITION_SEED_NS) — the stateBlocks.ts:54 pattern, so two offline clients (and the local synthesis layer, §5.1) converge on the same row. Ids are per-workspace; cross-workspace copy/paste resolves by name, consistent with #288 §9. And the equation is a validity check, not just a mint recipe: a row counts as seeded only if its id checks out — block.id === uuidv5(`${workspaceId}:${seed:key}`, DEFINITION_SEED_NS). Every consumer that keys off seededness applies it: handle resolution (seedKey → entry), seeded-origin/read-only attribution, the §5.1 fallback builds, the operator step's repair targeting. A row that merely carries seed:key at any other id (a cross-workspace paste, an import, a fabricated write) demotes to an ordinary user-authored definition — it competes by name (§7) like any other block and is never resolved as the seed, never repaired, never rendered read-only. (§9's block-type:type-id claim rule is this same check, instantiated for the one extra field a fake could hijack.)

Seed keys are stable strings, never derived from the current display name at runtime: <ownerToggleId>/property/<key> and <ownerToggleId>/type/<key>, e.g. system:kernel-data/property/types, system:srs-rescheduling/property/next-review-date. The owner segment reuses the togglable-boundary ids (src/facets/togglable.ts) that #289 §12 leans toward committing to as a public compatibility surface (its open question, stated leaning "commit — cheap now"); this design takes that leaning as settled — one stable-id vocabulary across both. Dynamic/user extensions use their extension block's toggle id (the block id, URI-encoded so it remains one seed-key path segment) as the owner segment. Since source text cannot know its containing block id, authors call extensionPropertySeedKey(key); it produces a reserved @extension/property/<key> placeholder, and the dynamic loader binds that same declaration/handle object to <encodedBlockId>/property/<key> before runtime resolution. Hard-coded dynamic owners are rejected, and the compile cache scopes live module instances by block plus content hash: two extension blocks with identical source therefore keep distinct bound handles instead of sharing and mutating one module. The trailing key is frozen at introduction; renaming a schema's display name later changes property-schema:name, never the seed key or the block id.

4.3The seed pass

A single idempotent pass over the currently-visible seed declarations:

  1. Probe: one batched read of all seed ids (SELECT id, properties_json, deleted FROM blocks WHERE id IN (…) — on the order of 100 rows).
  2. Diff: existence and deleted state decide the create/restore set — that is all the background pass acts on. The stored seed:revision is read for diagnostics only (logged when it trails the code revision); payload diffing belongs to the operator upgrade command (§6.1).
  3. Write: one tx (chunked if large) that creates missing definition blocks (children of the Properties/Types pages, systemMint ids) and restores soft-deleted ones — a deleted-column-only update, deliberately not a bag rewrite: rewriting seed-owned fields on restore would be a PATCH-class write decided from local state (this section's hazard class). A restored row therefore reanimates whatever bag it carried — current in the common case; stale or drifted only if it was tampered with or deleted inside an upgrade window — and catches up at the operator step, whose enumeration includes soft-deleted seeded rows precisely so that window stays short (§6.1). Creates run unconditionally — first materialization included. The first post-cutover client to open a workspace materializes its rows; there is no arming state, no operator materialization step, and deliberately no "zero rows until the fleet reloads" guarantee. The consequence is a brief mixed-version window per cutover slice: rows created by an updated client sync into still-open pre-cutover tabs, whose projectors publish them as user schemas / duplicate user types (userSchemasService.ts:100-107, userTypesService.ts:127-136) until those tabs reload — analyzed and accepted in §12/§13 (benign for property rows; for type rows the one persistent mistake, picking a UUID-backed duplicate from the picker, is handled by runbook coordination). An anchor/arming gate holding creates until an operator step was designed and rejected: at this fleet size it buys minutes of window at the cost of arming state, a fresh-workspace bypass residual, a dormant-workspace materialization gap, and a cleanup slice — and organic materialization is what makes any workspace self-heal on first open with no operator dependency. Nothing else, in v1: revision upgrades and drift repair are explicit post-sync operator steps in the §13 runbook, not background machinery — under the §5.1 mirror model a stale bag is cosmetic wherever code is present. Steady state is zero writes — the pass is a read-only probe, same cost profile as ensureStateChild's fast path (stateBlocks.ts:100-109).

When it runs — the trigger set matters, because declarations arrive in waves. On cold start, at bootstrapWorkspace time (right after ensureSystemPages, workspaceBootstrap.ts:210) the only installed runtime is the static-data one (repo.tsx:33, resolved from staticDataExtensions' 11 entries with no toggle overrides) — most schema-registering plugins (attachments, quick-find, video-player, srs-review, extensions-settings, …) contribute their declarations only when the full app runtime installs in a post-paint effect (AppRuntimeProvider.tsx:114-119), and dynamic/user extensions later still. A bootstrap-only pass would therefore never seed them. So the pass runs:

  • at workspace bootstrap, over every declaration the installed runtime carries — which at bootstrap is the static-data runtime, toggle-blind. A kernel-only owner filter was the earlier design and is wrong, because the static-data runtime's write surfaces are themselves toggle-blind: ensureSystemPages runs every registered system-page ensure off the installed runtime (repo.ts:1840-1842) — including geo's Locations page (systemPagesFacet.of, geo/dataExtension.ts:29), which writes MAP_TYPE (locationsPage.ts:31) whether or not geo is enabled — scheduleWorkspaceBackfills snapshots the same runtime's backfills facet, and the §5.1 landing writes ride it too. A kernel-only trigger would let the system write plugin-typed data into a disabled-plugin workspace while withholding the definitions that classify it — #288 residue minted by our own bootstrap. The rule is write-visibility = seed-visibility: every trigger seeds exactly the declarations of the runtime that can write at that moment — no owner filter at all. Safe mode still gets minimal seeding for free (its runtime carries less); a disabled static-data plugin gets its handful of definition rows materialized regardless — the price of its toggle-blind writers, consistent with definitions-persist-on-disable (§5.3).
  • again on every setFacetRuntime swap / definitionSeedsFacet change — an idempotent, diff-only re-probe over whatever declarations that toggle-aware runtime carries. This is when plugin seeds actually land (the app-runtime install), and it is also how dynamic/user/agent extensions seed: their declarations join the facet when the extension loads, the pass re-runs, their definitions upsert. No separate mechanism.
  • and when the membership gate releases. The access gate below is awaited, not merely checked: a pass parked on the membership row resumes when it syncs in — otherwise a fresh device whose runtime swaps all complete before the row arrives would silently never seed that session.

Toggle policy: a disabled plugin's extension isn't resolved into the toggle-aware app runtime, so its later-wave declarations aren't visible to the background pass — it neither seeds nor restores, and its existing definition blocks simply persist (§5.3). (The static-data set is the deliberate exception: it seeds toggle-blind at bootstrap because its writers run toggle-blind — write-visibility = seed-visibility above.) The §13 operator upgrade command is deliberately toggle-blind over shipped code (§6.1): this policy guards creating a disabled plugin's rows (durable new state); it must not freeze an already-existing row's metadata at an old revision. One transition case has no existing blocks to persist: a plugin disabled before slice B ever ran, which may still have left values or type memberships from when it was enabled — keys #288's fully-child-backed marker would need classifiable (§11.1). This is handled empirically, not with machinery: at slice-B time, one query per workspace (declared names against properties_json keys; declared type ids against the trigger-maintained block_types index — a block can carry residue as nothing but types: ['todo']) says whether the state exists in the fleet at all. It can never arise again — anything disabled after B keeps its already-seeded definitions, and the static-data set (the only plugins whose writers run while disabled) seeds toggle-blind at bootstrap per write-visibility = seed-visibility above — so if the query comes back empty, nothing is built; if residue turns up, the fix is a one-off script or simply enabling the plugin once (which seeds everything). An automated detection sweep was designed and cut as YAGNI for a two-workspace fleet. The general backstop is that #288 needs an orphan-key story regardless: historic keys with no schema of any origin already exist in the data, and its migration must materialize or infer definitions for them wholesale — disabled-plugin residue is that problem's easy sub-case (the declarations exist in code to infer from), and dormant dynamic extensions sit in the same bucket.

Scheduling: the pass is deep-idle work, not bootstrap-blocking. Registry correctness never waits on it — the synthesis layer serves entries regardless (§5.1); the pass only materializes rows. So every trigger above schedules the pass (the workspace-backfill idle pattern, repo.ts:1861-1899) rather than running it inline; app startup pays only the in-memory synthesis (microseconds) plus, in steady state, one deferred PK-batched SELECT whose diff short-circuits on seed:revision equality. The accepted transient: in a brand-new workspace, a field row (#288) can briefly reference a definition block that hasn't materialized yet — the id is already correct (deterministic), the block appears when the idle pass runs.

  • No drain-hold — the create-only pass needs an access gate, not a completeness signal. A device that hasn't finished downloading the workspace may probe seeded rows as locally absent and CREATE them — harmless: creates ride the apply_block_creates insert-or-touch RPC (powersync.ts:368-383), so a deterministic-id collision preserves the server row wholesale (created_at included — the guarantee user-page/ui-state bootstrap already rides) and the no-op touch WAL-echo reconciles the local phantom through normal sync (#336/#244); restore-on-delete touches only the deleted column, same class. The hazardous class is PATCH-class decisions from local state — an upgrade or drift-repair decided against a partially-downloaded replica is a fleet-rollback vector (local code at N, replica showing N−1, server already at a newer client's N+1: "repairing" to N wins the column LWW), and the server can't compare-and-set a revision that lives inside the encrypted properties_json bag. v1 sidesteps the class instead of gating it: the background pass makes no PATCH-class writes at all — revision upgrades are explicit post-sync operator steps in the §13 runbook. The per-workspace drain/checkpoint primitive is therefore not a v1 prerequisite; it stays specified for what genuinely needs it — #288's backfill and childBacked transitions, and any future background PATCH-class seed op. (Specification, for then: a sentinel row is not a signal — a first synced row proves download started; the real signal is checkpoint-based, e.g. "a full checkpoint completed after this workspace's buckets joined the subscription set" via SyncStatus.statusForPriority/priorityStatusEntries/lastSyncedAt, SyncStatus.d.ts:118-136 — onFirstSync's client-global hasSynced is not it, internals/firstSync.ts:13-31.)
  • Scope: ChangeScope.Automation — machine-authored, durable, synced, not undoable (changeScope.ts:24). A user must not be able to ctrl-Z the kernel's schema definitions out of existence, and seed writes must not pollute the document undo stack.
  • Read-only workspaces: skipped by an explicit isReadOnly check (the gate scheduleWorkspaceBackfills uses, repo.ts:1862) — Automation scope alone would not stop the writes (readOnly: 'allow', changeScope.ts:65-69). And because a fresh device's role row may not have synced yet (App.tsx:76-84 deliberately defaults to writable), the pass on a not-freshly-created workspace additionally waits for the membership row — otherwise a viewer's first open would enqueue ~100 RLS-rejected creates into the quarantine. This access gate is the one wait the create-only pass keeps (the bullet above). The synthesis layer (§5.1) serves the registry regardless. One consequence, stated: in a shared workspace whose only post-cutover openers are viewers, no seeded rows exist yet — a viewer's registry is synthesis-only, so fallback builds and classification for plugins that viewer doesn't have loaded (§5.3) are unavailable there until any writable member first opens the workspace (their open is the trigger). That's no worse than today's registry, where an uninstalled plugin's schemas simply don't exist; the block-dependent guarantees (§5.3, §11.1) carry this materialized-by-a-writable-open precondition rather than gaining a dedicated read-only materialization path.
  • Fresh-workspace cost: ~100 definition blocks created once, uploaded once. Existing-workspace upgrade cost: only the definitions whose seeds changed.

If #289's boot restage later moves projector start pre-mount, stage 2 just happens earlier; nothing here depends on it.

5The unified registry

5.1One behavioral source per definition — blocks are metadata mirrors in v1

Decision (v1) — authority splits by field kind, not by source

Installed code is authoritative for behavior (codec, config, default, changeScope) wherever a local declaration exists; the definition block is authoritative for durable identity and migration state (fieldId, name/winner metadata via created_at, hidden, origin — and childBacked once #288, which starts after this project ships, adds it) — always. The block additionally serves as a behavioral fallback where no local declaration exists: an identity-checked (§4.2) seeded row builds through kernel preset cores (preset.build(config) off the bag), so a client without the plugin still runs its kernel-preset schemas fully — only the six custom-codec schemas render raw without their plugin (§5.3). The rule that deletes a whole reconciliation class: exactly one behavioral source is live per definition per client — code when declared, block when not — so the two representations never compete. Legitimate divergence between them cannot exist in v1 anyway: seeded bags are wholly seed-owned (§6.2, no user-editable fields), so a block can only ever mirror some code revision. Block behavior-authority returns in v2 when per-field user editing creates an actual requirement for it (§14, §15).

The registry keeps its existing plumbing — facet runtime buckets feeding the propertySchemas rebuild step (facetBridge.ts:239) — with the inputs changing from three sources (kernel registrations, plugin registrations + type-lift, projector bucket) to two:

  • Synthesis layer ('seed-synth' bucket, workspace-keyed per §4.4): declaration behavior is computed from (declarations, workspaceId) with no DB read and recomputes on every runtime swap, seed-facet change, and workspace pin. It applies no owner filter — matching the seed pass's write-visibility = seed-visibility rule (§4.3) — so the static-data runtime exposes kernel plus plugin data declarations during bootstrap. That behavior is needed by landing paths such as the tutorial, but it is not sufficient identity evidence: tutorial, daily-note, and every other active-workspace transaction queue at Repo._runAndDispatch until the first fresh user-schemas snapshot publishes, fresh workspaces included. A complete empty snapshot releases a new workspace; no freshness exemption exists. The gate waits only on local projection, not a full sync drain, so a definition still undownloaded can later create the same accepted offline-collision case described in §7. Map insertion order is never load-bearing: declarations supply locally-loaded seed behavior, projected rows supply user/fallback behavior and winner metadata, and same-name competition is resolved by the §7 rule.
  • Block layer ('user-data' bucket, the existing projector): the projector generalizes from "user schemas" to "all definitions", but what it publishes splits by kind — behavioral entries for user-authored rows (exactly today's semantics), and for identity-checked seeded rows with no matching local declaration (the fallback build: kernel preset cores off the bag, dropped the moment the declaration's wave loads and code takes over); plus identity/winner metadata for every definition row — the §7 competition inputs and #288's classification facts, exposed as the fieldId- and name-keyed definition-metadata index (§5.3 — the property panel depends on it to keep declaration-only rows from rendering as editable inferred properties). It never publishes a behavioral entry that competes with a live declaration.

Who waits for the DB: consumers read the registry, never definition blocks directly — and for a locally-declared definition the behavioral entry is the code declaration, full stop. That behavior can serve degraded reads before projection. Identity is different: user-created definitions, §7 winner metadata, and fallback builds for undeclared plugins are unknowable until the projector's first fresh, complete tick. The seed pass (writing rows) remains idle-schedulable because registry readiness waits only for the local read projection, not for seed-row materialization (§4.3). The rule is therefore: the pre-prime window is read-safe, not identity-write-safe. A name guessed from declarations before prime can differ from the §7 winner (for example, a stored user schema may shadow the seed or a deterministic seeded row may have been renamed). That is harmless for display and wrong to persist; the hazard includes the plain name-keyed cell write, not just future fieldId stamping. The implemented boundary makes the identity-bearing registry unavailable until user-schemas publishes, and the shared Repo transaction boundary queues every active-workspace transaction before it captures identity. A workspace/generation recheck cancels a queued tx that crossed a switch; a fresh-loader contract prevents cached away-and-back data from releasing it; initial loader errors reject readiness rather than hanging. bootstrapWorkspace explicitly awaits the same promise on fresh and existing workspaces — one local subscription tick over roughly 100 rows, with a complete empty tick sufficient for a fresh workspace.

Two housekeeping notes: workspace-keying does not supersede the existing teardown contract — the projector's dispose keeps clearing its bucket (projectorRuntime.ts:181-195; facet.ts:332-335), which bounds retained buckets and raw-view pollution; and one consumer reads raw buckets beneath the workspace read-filter — the agent bridge's describe-runtime enumerates contributions by bucket id — so it filters by the source id's workspace suffix (or shows all, labeled) rather than pretending the filter applies.

There is no revision guard — under the one-source rule the registry never reads a seeded row's behavioral fields where code is present, so a stale mirror cannot leak stale behavior: current-code clients run their own declarations, plugin-less clients' fallback builds use the newest synced bag (which is exactly what they should want), and the stored bag itself is refreshed by the §13 operator upgrade step, not by background reconciliation. (Both the guard's overlay projection and a guard-triggered targeted repair were designed in earlier revisions and cut — the repair as YAGNI, the guard with the mirror model.) The registry map gains an id-keyed index alongside the name-keyed one — findSchemaByFieldId being O(schemas) is already flagged as a risk in #288 §12.

Under the mirror model this row is automatically inert: behavior on this client comes from its own declaration regardless of what the bag says (never newer semantics through an older handle — no pin rule needed, it falls out of one-source-per-definition), and the bag's newer content matters only to plugin-less clients' fallback builds, which correctly want the newest synced state. Degrade tests assert that a newer bag changes nothing for a client with the declaration loaded.

When upgrades actually happen. In v1: at the §13 operator step — post-sync, on a writable operator client — after any deploy whose seeds bumped revisions. Between deploy and step, staleness is invisible where it could matter (behavior comes from code, §5.1) and cosmetic where it shows (a Properties-page row displaying the old bag). Read-only clients never write. One scope rule the command must implement: enumeration is toggle-blind over shipped code — it aggregates the in-repo seed tables directly (kernel + every in-repo plugin, enabled or disabled; module-level declarations, no runtime-facet resolution), because a per-workspace toggle must not freeze a row's metadata at the revision it was disabled at. Behavior would stay correct on re-enable regardless (code, §5.1) — but the block-read facts (hidden, type labels, childBacked once it lands, and the bag plugin-less clients' fallback builds serve) would silently stay stale across every deploy the toggle outlasts. §4.3's toggle policy is not violated: it guards creating a disabled plugin's rows; upgrading a row that already exists materializes nothing new. Enumeration also includes soft-deleted seeded rows: their bags upgrade and repair in place, without undeleting, so a later background restore (§4.3) reanimates a current payload rather than a stale or drifted one — the residual (a row deleted and restored between operator runs carries its old bag until the next run) joins the same accepted staleness class as the rest of this section, and the tamper-then-delete case surfaces in the drift check when the command next runs. The honest residual otherwise narrows to code that genuinely cannot run anywhere: disabled or uninstalled dynamic/user extensions compile to settings-row shells (dynamicExtensions.ts:205-212), so their declarations are unavailable to the operator's client too — those rows persist at their last-written payload (written by some real code revision, so never garbage) and catch up on the first operator run after the extension is next enabled; the §4.3 diagnostic (revision-trailing rows are logged) is what surfaces them. A childBacked flip (a seed-revision bump, §6.2) propagates with a bounded disagreement window — clients read the flag off the block, and the block converges via sync + the operator step, not atomically; #288's machinery must tolerate that window exactly as it already tolerates its own flag-flip rollout windows (its flips are #288-rollout-coordinated events anyway).

The seed source of truth stays in code; blocks are its materialization. A new seed materializes per-workspace on the §4.3 triggers; a change to an existing seed (a revision bump in the declaration) materializes at the §13 operator step and reaches the rest of the fleet as ordinary sync of the one upgraded row — there is no fleet data-migration step either way.

Operational load — one command per cutover, not a recurring tax

Materialization needs no operator at all — creates, first materialization included, are fully automatic (§4.3). Operator commands are needed only when a deploy edits an existing definition's payload — a revision bump: changing a default, a preset config, a seeded display name, or (with #288) flipping childBacked. Adding schemas, adding plugins, enabling plugins, and new workspaces all ride the automatic pass. Payload edits are rare and always deploy-coupled, so the command should ride the deploy script itself (run from a fully-synced client, e.g. an agent-bridge verb), not human memory; forgetting it is behavior-safe for v1's own fields (staleness is cosmetic under the mirror model — the paragraph below). The one exception is a childBacked flip, which is #288-owned: its data migration rides the seed upgrade (§6.2), so there the command is a hard step of #288's flip rollout — already a coordinated event on that project's runbook, never a fire-and-forget deploy. And the posture is v1-window-only by design: once #288 builds the per-workspace checkpoint-drain primitive for its own backfill (§4.3), upgrades and drift repair are slated to become drain-gated background work like everything else — the operator command then retires to ad-hoc repair. No steady-state design here requires recurring human intervention.

7Name resolution & collisions

Ids cannot collide; display names can. Names are load-bearing in three places: the properties_json cell key, alias-entry ([[status]] typed by hand — #288 §7's field-row grammar resolves the schema-name map first), and every name-keyed consumer in §2.1. Today mergeLiftedSchemas resolves collisions last-wins-by-load-order with a warning — nondeterministic across clients with different plugin sets, which #288 §6 flags as a hard error under child-backing (the cell key is the name while child identity is the fieldId; a per-client winner strands the loser's field rows).

Decision

Because every definition is now a synced block, collision resolution can finally be deterministic fleet-wide: the name winner is the earliest live definition — ordered by (created_at, id), computed over synced block state only. Locally-varying facts (is the plugin installed? enabled? does its preset resolve?) never enter the competition: toggles are per-user and presets per-install, so any owner- or codec-dependent rule would make the winner differ per client — precisely #288's hard error. Local state may gate UX (what the creation dialog offers), never the winner. Losers stay registered by fieldId (child rows keep classifying, backlinks keep resolving) but are excluded from the name map and the cell projection, and surface in the Properties page UI as shadowed — rename to activate. The exclusion must also gate the typed-handle boundary: block.get/tx.setProperty read and write properties[schema.name] directly today, so without a check, an installed plugin whose definition is shadowed would keep decoding and overwriting the winner's cell values. At cutover the boundary call sites — the four block.get/tx.setProperty sites plus addTypeInTx's initial-value encoding, which writes cells by name through TypeTagger without touching tx.setProperty (§9) — resolve the handle through the registry (seedKey → entry) and, when the handle's definition is not the name winner, degrade loudly instead of touching the winner's key: reads return the schema default, writes reject with a structured error. The gate is implemented once, not five times: a single tx/workspace-bound resolution primitive — resolve(handle | name, txContext) → ResolvedPropertySchema or a structured rejection — is the only path from a handle or name to identity, and the only place the winner check lives; all five boundary sites call it, none re-implement it. The §3 two-type split makes this compiler-enforced rather than convention-enforced: ResolvedPropertySchema is constructible only by the primitive, so an identity-consuming code path that skipped the gate would not typecheck. (Post-#288, a shadowed schema's own values stay reachable by fieldId through its child rows; pre-#288 they are simply inaccessible while shadowed — the price of a name collision, surfaced in UI.) Earliest-wins protects existing user data from newly-arriving plugin seeds: a user's long-standing status schema is not hijacked when a plugin introducing status is installed. (In a fresh workspace seeds usually predate meaningful user data — but not guaranteedly, since the pass is idle-scheduled (§4.3). For plugin names, a fast first-minute user schema out-dating the seed hands the name to the user's row — the intended bias, not a failure. For kernel names — types, alias, the definition meta-fields — that outcome would degrade kernel machinery through the boundary gate above, so it must be unreachable through the UI: the prevention check below reserves synthesized names before their rows exist. The residual paths — import, raw agent-bridge write, a concurrent offline create — leave the kernel seed shadowed and its handles degraded, surfaced as shadowed kernel definition — rename the user schema to restore: self-serviceable and never silent.)

  • Dead plugins keep their names reserved in v1 — reclaim is deliberately v2. Creation-prevention + read-only + persist-after-uninstall compose into a permanent reject for a once-seeded name, and v1 ships no release mechanism — stated plainly as an accepted regression versus today, where toggling a plugin off frees its names from the registry. A definition:archived flag was fully designed (archive-aware competition, preservation-on-repair, a projector overlay, an accepted upgrade-vs-archive whole-bag LWW race, un-archive and name-reuse semantics) and cut as YAGNI: at this fleet size squatting requires both abandoning a plugin and wanting one of its exact names, the affected users can ship the remedy when it bites, and deferral lands the feature on better ground — after #288's per-key child-backing a user-owned flag rides its own row, so v2's reclaim needs none of the preservation machinery and the LWW race never ships anywhere (§15). What v1 does instead: preflight — a runbook step at each cutover landing (§13: property names before B's deploy, type labels/ids before C's) checks existing user names against every seeded name, remediated by hand (two workspaces) before the winner gate can start degrading kernel/plugin handles; namespacing — new plugin schemas use prefixed names (srs:interval), so the reserved set stops growing bare words; and the creation dialog's rejection names the owning plugin ("held by srs-rescheduling") so the reservation is legible, not mysterious. The field name definition:archived stays reserved for v2 (§3).
  • The order must be protected from the design's own machinery: created_at is an ordinary synced column, so a re-create of an established row must not reset it. The load-bearing guarantee is the upload connector's insert-or-touch CREATE path (apply_block_creates, powersync.ts:368-383): a deterministic-id collision preserves the server row wholesale, created_at included — a fresh device "re-seeding" rows it merely hasn't downloaded yet cannot flip a winner fleet-wide, only mint a local phantom that the touch-echo reconciles — nothing prevents that transient in v1 (there is no download-drain hold); insert-or-touch is what makes it harmless. Restore-vs-create in the seed pass preserves created_at on soft-deleted rows by construction (restore is an update, not a re-mint). Residual, accepted for now: the rule trusts client clocks — a skewed or hand-written created_at can back-date a definition into winning a name. For the current two-user fleet that's a non-threat; revisit (e.g. server-stamped ordering) if the fleet grows.
  • Prevention first: addSchema and the rename flow reject a name already claimed by any live definition or by any synthesis-layer entry. addSchema first awaits the captured workspace projector generation, then checks stored winners and loaded declarations atomically with its creation path; kernel names are therefore reserved in a fresh workspace after the complete empty projection and before any definition row exists. Kernel declarations load on every client, plugin ones where installed. Prevention may thus be stricter than the competition — local knowledge is allowed to gate creation UX (per the decision aside), never to pick the winner — but never looser: the synced-state test is a subset of what it checks. Exact-match names; case-insensitivity is an open question (§15). The collision rule is for the unpreventable case: two definitions created concurrently offline, a plugin seed arriving into a workspace that already used the name, or a write that bypassed prevention (import, raw bridge write).
  • The audit says this is real: srs-rescheduling seeds bare, generic names — interval, factor, grade, archived, next-review-date, review-count, snapshot-history (srs-rescheduling/schema.ts:19-55) — exactly the names users would pick. New plugin schemas should namespace (srs:interval) as most already do; renaming existing plugin names is a data migration (the cell key changes) and is explicitly not attempted here — earliest-wins keeps the status quo working either way.

9Types get the same move

Everything above applies to block types with one extra wrinkle: type ids are stored data. Every block's types property and the block_types index hold strings like 'page', 'todo', 'daily-note' — for user types the string is the definition block id; for kernel/plugin types it's a legacy constant. Rewriting those strings across every workspace would be a huge, risky data migration for zero user-visible gain, so:

Decision

block-type definition blocks gain block-type:type-id — the string written into typesProp. Absent (every existing user type) → the block id, preserving the block-id = type-id rule. Seeded kernel/plugin types set it to their constant ('page', 'todo', …). This is the permanent identity model, not a compat shim: a stable readable string is the better type identity — it can be a compile-time constant, which per-workspace uuids structurally cannot (a uuid-unification data migration is cheap on data at this fleet size but would force every types: ['daily-note'] query, every cross-plugin type-constant import, and every hasBlockType check through per-workspace runtime resolution, forever; #289's coupling taxonomy counts those constants as "retargets for free" precisely because they're stable strings). There is nothing to migrate or remove later. The type registry stays keyed by type id; getTypeBlockId becomes total (every registered type has a backing block). Whether typesProp values should become refs to definition blocks is out of scope — flagged as future work, unchanged from the project brief.

type-id claims need a rule — it's a user-writable property now

Once block-type:type-id exists, "ids cannot collide" (§7) no longer covers types: any block — a cross-workspace paste, an import, an agent-bridge write — could claim 'page', and the projector would publish both contributions with last-wins resolved by subscription row order (projectorRuntime.ts publishes every built row; keyedMapFacet is last-wins), nondeterministically deciding addTypeInTx validation and every renderer lookup. The rule: a block-type:type-id differing from the block's own id is honored only when the block's identity checks out against the deterministic-id equation — block.id === uuidv5(`${workspaceId}:${seed:key}`, DEFINITION_SEED_NS) — which every client can verify with zero knowledge of the owning plugin (merely writing a seed:key property proves nothing; the pass only probes/repairs ids derived from local declarations, so a fabricated key on a random block is never repaired by anyone). On any block failing the check, the claim is ignored and the type projects under its block id. Residual: a fully self-consistent fake seed (correct uuid, invented toggle id) is repaired by nobody and falls to the §7 clock-trusting tie-break — accepted under the small-fleet stance, stated rather than hidden.

The rest is mechanical application of §3–§6:

  • New definition fields on block-type blocks: block-type:type-id, block-type:hide-from-completion (today code-only, blockType.ts:26), plus seed:key/seed:revision. Existing fields (label, description, color, hide-from-block-display, properties refList) already cover the rest of TypeContribution.
  • TypeContribution.properties → block-type:properties refList of definition-block ids. Seed declarations reference the property seed handles; the seeder resolves them to per-workspace ids. The two lift-only schemas the audit found (status, daily-note:date) get their own property seeds, which their type seeds reference — the lift path they currently ride dies.
  • The prefs/ui-state container types (pluginPrefsExtension/pluginUIStateExtension, 7 call sites) seed the merged shape — the helper-injected hideFromCompletion: true is part of the seed payload, not re-injected at registration.
  • addTypeInTx keeps its signature, not its encoding path: it still validates against types.get(typeId) and accepts name-keyed initial values, but its value encoding is the fifth gated write path — today TypeTagger looks up propertySchemas.get(name) and writes next[name] directly (typeTagger.ts:83-92), bypassing tx.setProperty, so left unmodified it would reintroduce §7's overwrite through the side door (a user-owned status shadowing the todo seed: addTypeInTx(TODO_TYPE, {status: 'open'}) would encode into the user's cell). At cutover each initial-value key resolves through the type's own block-type:properties refs (fieldId → entry) and applies the §7 winner check through the same resolution primitive as the other boundary sites — no hand-rolled check here either: a shadowed referenced definition rejects that value with the same structured error, never writing the winner's key by name. snapshotTypeRegistries (repo.ts:1579) is shape-unchanged.
  • Sequencing: types cut over after property schemas (their own slice, §13) — the type projector already depends on the schema projector (userTypesService.ts:132), and seeded type declarations need property seeds to exist first.
  • Alignment with PR #293 (type label = alias — merged as b468d65c): a user type claims its label as an alias (createTypeBlock sets alias = [label]), so [[Label]] resolves to the type block and type names become workspace-unique via the alias-uniqueness trigger. The merged version goes further than the draft this doc first tracked: a kernel same-tx processor (blockTypeTypeifyProcessor.ts) completes any block gaining block-type in a local repo.tx — tags PAGE_TYPE, adopts content as the label, and ensure-present claims the label alias in the same tx. That flips the seeding mechanics from "choose to claim separately" to "opt out or be claimed for": the seed pass's type-create tx fires this processor, so on a label the workspace already uses (an existing [[Todo]] page) the trigger's RAISE(ABORT) would roll back the seed row's own create, silently and locally. The policy is unchanged — claim the label alias when it is free, skip on collision (the definition then lives without the alias, navigable from the Types page, surfaced rather than fought over) — and the who-claims rule is principled, not a list: a type with hideFromCompletion: true claims no alias; visible content types claim theirs. A type hidden from the #type vocabulary shouldn't enter the [[…]] vocabulary either; this excludes every kernel infrastructure type (whose labels — "Page", "Type", "User", blockTypes.ts:44-102 — are exactly the common words a workspace-unique claim would squat) and gets a plugin's hidden helper types right for free. Slice C's mechanics become: a seed-aware carve-out in the typeify processor (skip the alias-ensure step for rows passing §4.2's seeded-identity check — carrying seed:key alone proves nothing; the PAGE_TYPE/label-adopt steps stay, they're no-ops against what the seeder writes), with the seeder's own claim as a separate write behind a tx-aware pre-check (tx.aliasLookup) and a try/catch for the structured rejection — the trigger's alias_collision RAISE decodes through raiseProtocol.ts into ProcessorRejection('alias.collision'), no string-parsing at call sites. The carve-out is also what implements the hidden-types-claim-nothing rule (seeds for hidden types simply never make the separate claim). And because it applies §4.2's identity check, a fabricated or pasted seed:key at a non-deterministic id doesn't dodge it: the row fails the check, demotes to an ordinary user type, and gets the normal auto-claim path — no cosplaying past the alias-uniqueness machinery. The trigger itself (block_aliases_workspace_alias_unique, clientSchema.ts:834) stays the client-side, synchronous, before-upload backstop — not the mechanism. Timing residual, stated: type rows materialize when the first post-C client opens the workspace (§4.3), so [[Todo]] typed before that moment mints a rival alias seat (lookup-first resolution — the race #293 closes for user types at creation); the seeded type then stays alias-less with the rival surfaced for manual merge. Two knock-ons: seed repair of a label must go through #293's label→content→alias lockstep (writeBlockTypeLabel + aliasSyncProcessor; the merged version also releases the alias when a label is blanked) — where a re-claim can hit the same collision rejection and follows the same pre-check rule; and the alias-uniqueness trigger becomes a DB-level adjacent guard to §7's registry-level rule for type labels — one mechanism per layer, don't add a third.

9.1Renaming — display edits over stable identity

Identity (seedKey/block id) never moves, so renaming is always a display-field edit — but the two kinds differ in what hangs off the name today:

  • Type labels are pure display — stored membership uses the type id, never the label. User-type renames work today and #293 keeps content + the [[label]] alias in lockstep. Seeded type labels are seed-owned in v1: an upstream rename ships as a seed revision bump whose repair honors the same lockstep.
  • Property names are the cell key until #288's end state, and this project ships no rename machinery at all: the rename-reproject pass is #288's deliverable (its slice B2), and property renames — user and seeded alike — stay blocked outright until it ships (without it a rename is the "mass silent unset" #288 §7 documents). Once B2 exists, seeded property renames additionally ship as seed revision bumps. Child rows and references are untouched either way (fieldId identity).
  • The endgame that makes renames free: once #288's children-authoritative end state makes the cell a derived local projection, re-keying the cell by fieldId instead of name becomes a local reprojection rather than a fleet data migration — renames then need no reproject pass at all, and the §7 cell-key collision problem dissolves (names become pure display + alias-entry sugar). Deliberately not part of this project (value keying stays untouched here — that's what keeps the rollout non-breaking); flagged as the designated successor in §15.

Scoped reads and writes ride the same typed-handle boundary as document values — and get §7's winner gate with it: a shadowed scoped schema degrades loudly (default reads, rejected writes) rather than silently writing the winner's name key. Declaration behavior is available before rows sync, but active-workspace scoped and seed-pass transactions still queue at the shared readiness boundary before identity resolution. There is no circularity: the user-schemas projector is read-only and can publish a complete empty snapshot without those writes; seed-row materialization follows afterward.
  • E2EE workspaces: no new constraints. Definition blocks are ordinary blocks; their properties ride the encrypted properties_json envelope; the projector reads locally-materialized plaintext. (Contrast with #288's reference_target_id gate — nothing here adds a synced plaintext column.)
  • Read-only workspaces: seed pass skipped; synthesis layer serves kernel/plugin entries; user definitions arrive as synced blocks. A viewer client never writes definitions.
  • Safe mode / plugin toggled off: the plugin's later-wave seeds and presets don't load — with the §4.3 exception that static-data plugins' declarations still seed toggle-blind at bootstrap (write-visibility = seed-visibility: their system-page/backfill/landing writers run whether or not the toggle is on). Its definition blocks persist and either project as kernel-preset entries or stay registry-skipped with block-readable classification (§5.3). Nothing is deleted by absence — matching the projector's existing "absence is not deletion" stance.
  • Old clients during rollout: pre-cutover clients keep their code registries, but rows materialize organically once the first updated client opens a workspace (§4.3), so a pre-cutover replica will see seeded rows mid-window — a still-open tab for minutes-to-hours (the update path never force-reloads, registerServiceWorker.ts:15-21; the runbook's reload step bounds it), a forgotten offline device for longer. The exposure is the same either way. Property rows: the shadow's codec builds faithfully (slice A ships the presets before B), but pre-B tryBuildSchema hardcodes defaultValue: preset.defaultValue and changeScope: BlockDefault (userSchemasService.ts:88-92 — the §3 fields are post-B vocabulary), and the projected entry wins name-keyed resolution (runtime user-data merges after static registrations, last-wins — facet.ts:110-118, :246-248). Handle-mediated reads and writes are unaffected — pre-B handles never consult the registry — so the divergence is confined to registry-mediated surfaces, chiefly the property panel: the scope-based auto-hide keys off the shadowed BlockDefault entry, so normally-hidden UiState/UserPrefs properties surface there, and a panel edit writes at document scope — a mis-scoped, durable, but visible and hand-repairable stray (plus wrong displayed defaults for schemas with seeded defaults). Type rows publish as duplicate user types — picker noise, plus the one persistent mistake of selecting a UUID-backed duplicate, which mints wrong-id memberships enumerable as types entries equal to seeded block ids. Runbook coordination (prompt reloads; avoid the property panel pre-reload at B and the type picker at C) handles the tab case; coordinate/update forgotten devices, or accept the bounded risk — an old-client-unreadable row shape or a forced-reload gate was considered and rejected as disproportionate at this fleet, same stance as §4.3's organic materialization. A stray delete is restored by the steady-state pass.
  • Cross-workspace copy/paste of a definition block: the pasted block carries seed:key at a wrong id → it fails §4.2's seeded-identity check and demotes to an ordinary user definition (never resolved as the seed, never repaired, not read-only), leaving a duplicate name that the collision rule (§7) shadows and the UI surfaces. Paste of field rows across workspaces resolves by name per #288 §9. A paste-time strip of seed:* props (plus block-type:type-id and the label alias, whose uniqueness trigger would otherwise reject the paste) keeps the common case clean.
  • Workspace deletion/reset: nothing special — seeds re-materialize on next open, same as kernel pages.

14Rejected alternatives

  • Name-based identity everywhere (keep merging by name, add childBacked to code): fails #288's requirements 1-2 outright — names collide across sources per-client, and code constants fork data across plugin versions. This is the status quo the migration doc already rejected.
  • Global (cross-workspace) definition ids: a block id is unique per row; the same definition in two workspaces would collide in one local DB. Per-workspace ids + name-based cross-workspace resolution match #288 §9 and the existing state-block precedent.
  • Blocks as fully passive mirrors (registry never reads them at all): rejected — missing-plugin classification, fallback builds, §7 winner metadata, and #288's durable childBacked all need block-read paths. The adopted v1 model (§5.1) is the field-kind split this rejection pointed toward: blocks authoritative for identity/classification/winner metadata plus fallback behavior where undeclared; code authoritative for behavior where declared; block behavior-authority deferred to v2's editing requirement. (Revs 1-3 of this doc made blocks behavior-authoritative post-prime; the reconciliation machinery that demanded — bucket-precedence choreography, a revision guard, deploy-skew handle pins — is what rev 4 deletes.)
  • Columns instead of properties for the new definition fields: rejected in §3's decision aside — properties ride existing machinery.
  • seed:hash alongside seed:revision: a stored payload hash would make drift detection O(1) — but drift is checked only in the operator command (§6.1), over ~100 wholly seed-owned rows, where field comparison is trivially cheap; revisions already give the direction a hash can't. YAGNI.
  • Hash-based seed provenance (#289's base-hash) instead of revisions: hashes detect difference but not direction — an old client couldn't tell "my seed is stale" from "the block is newer than my seed" and would fight newer clients. Revisions give ordering; drift detection falls out of the revision-equal + fields-differ case (§6.1). (#289's hash approach remains right for code, where there is no total order on user edits.)
  • Seeding lazily on first use (definition materialized when a value is first written): leaves the registry incomplete for pickers/panel, makes fieldIds appear at unpredictable times mid-#288-migration, and saves ~100 rows once. Not worth the nondeterminism.
  • Migrating kernel/plugin type-id strings to block ids: a fleet-wide rewrite of every types array and the block_types index for cosmetic uniformity; block-type:type-id costs one property (§9).

15Open questions

  1. Per-field user ownership (v2): which seeded fields become user-editable first (likely hidden, type color/label), and how ownership is recorded (per-field marker vs an "edited fields" set) so the seed pass stops repairing them. #289 §8's "edit this type" affordance lands here. Note the §5.1 code/block equivalence weakens for exactly the overridden fields — a user-edited value exists only in the block, so the pre-prime window serves the seed's value (a class v1 avoids entirely by having no user-owned fields on seeded rows); v2 either accepts that bounded flash for display-only fields or gates them on projector prime.
  2. Case sensitivity of name collision/prevention — exact match (status quo of the alias map) or case-insensitive? Affects §7's prevention check only; the deterministic rule works either way.
  3. Picker exposure of declarations: should AddPropertyForm's picker offer schemas whose plugin is disabled (they're fully functional if kernel-preset-built, §5.3)? Leaning yes for kernel-preset ones, no for declaration-only.
  4. New-plugin type ids: should plugins introduced after slice C use block ids as type ids (no readable constant)? Leaning no — §9's argument (compile-time constants, retarget-for-free string references) applies to new types as much as grandfathered ones; readable string ids stay the norm, and "absent → block id" remains a user-created-types convenience.
  5. One-touch extension conversion: as sliced, extension authors touch their blocks twice — property registrations at B, type registrations at C. Pulling slice C's type synthesis forward into B would let them convert once. Leaning no — it saves exactly one coordination event with one extension author today; accept two-touch unless the extension count grows before B.
  6. Dead-plugin name reclaim (v2): v1 keeps once-seeded names reserved even after a plugin dies — an accepted regression vs today's toggle-off-frees-the-name (§7). The reclaim/archive affordance is deferred until squatting demonstrably bites, and deliberately lands after #288's per-key child-backing, where a user-owned flag rides its own row and needs none of the whole-bag LWW preservation machinery v1 would have carried (including the upgrade-vs-archive race, which now ships nowhere). definition:archived stays a reserved field name (§3); a bulk "this plugin is gone — reclaim all N names" affordance belongs to the same feature.
  7. Id-keyed cell (the rename endgame, §9.1): after #288's children-authoritative end state makes the cell a derived local projection, re-key it by fieldId — renames become pure display edits and the cell-key half of §7's collision problem dissolves.
