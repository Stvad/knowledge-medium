Schema unification: definitions are blocks, behavior is presets, code seeds

Collapse the two property-schema (and block-type) worlds — inline code objects merged by name vs. definition blocks projected by id — into one: every definition is a synced block with a block-id identity; code registers behavior (ValuePresets) and seeds definitions instead of registering them.

TL;DR

Today kernel/plugin schemas are inline PropertySchema objects merged by name (mergeLiftedSchemas, refProjection.ts:29) while user schemas are property-schema blocks projected by block id — two identity spaces, two registration paths, and a set of workarounds the properties-as-blocks migration (PR #288 §6) can no longer absorb: synthetic property:<name> fieldIds leaking into the reference index, a childBacked flag with no durable home for plugins, cross-source name collisions, and a two-phase cold-start bootstrap.

The end state: one source of definitions (blocks), one identity (block id), one registry path (the projector). Kernel and plugin schemas become seeded definition blocks with deterministic per-workspace ids (uuidv5, the stateBlocks.ts pattern); their codecs are rebuilt from code-registered ValuePresets exactly the way user schemas already are (tryBuildSchema, userSchemasService.ts:52). A code-resident synthesis layer makes kernel definitions available synchronously before any block exists, so cold start needs no two-phase bootstrap. Types get the same move (§9). The audit gate passes: all 9 existing presets are kernel-owned, 36 of 44 plugin schemas already build from kernel codecs, and the remaining 7 custom codecs wrap into hidden presets one-liner-style (§10).

1Why now — the consumers

Two in-flight designs consume this one:

Properties as blocks (PR #288). That migration makes every property value a child block whose field row carries reference_target_id = fieldId. Its §6 states hard requirements the current kernel/plugin-vs-DB schema split cannot meet, and its §11 rollout bends around them. This project is the fix it names. Specifically, the migration needs:

  1. fieldId = definition block id, for every schema, kernel included. The spike's synthetic ids (fieldIdForPropertyName → property:<name>, src/data/api/propertySchema.ts:90 on origin/props-as-childrent-spike-merge) put non-block ids into the synced reference_target_id column and the backlink index (#288 §6, residual 2). After this project, that helper and its reference-projection special-case never ship.
  2. childBacked (and hidden) as synced fields on the definition, classifiable without the owning plugin installed. #288 §6 calls a plugin flag expressed as a code constant "a data fork waiting to happen" and requires durable persistence — this design is that persistence mechanism.
  3. Kernel definitions available at cold start before first sync. #288 §11 step 5 needs a two-phase first-sync projection because schema discovery reads the cell; deterministic seeded ids dissolve phase 1 into a pure computation (§11.2 below).
  4. Fleet-wide classifiability. #288 §11 step 5 can only mark rows fully-child-backed when every key belongs to a schema every client can classify — which today excludes plugin keys. Definitions-as-blocks makes plugin schemas classifiable everywhere (§11.1).

Plugin materialization (PR #289). Its parallel track ("declaration materialization", §8 there) is this project sketched in one paragraph: seed block-type/property-schema blocks from system definitions through the existing definition-block projector. This doc designs that stage in full and reuses #289's structure where it fits: the kernel → materialize → assemble boot staging (§4.4), the provenance/base-version thinking for seed upgrades (§6), and the toggle-id-as-stable-API decision for seed keys (§4.2). Where #289 materializes code on demand behind an approval gate, definitions are pure data — no approval, no compile pipeline, seeded eagerly rather than ejected on demand. Per the project brief: we account for future plugin materialization but do not gate on it; nothing here depends on #289 landing.

2Ground truth — the two worlds today

Verified against master 8f957555, 2026-07-10. Counts exclude tests.

2.1Code world: kernel + plugins, name-keyed, merged

  • 27 kernel schemas in src/data/properties.ts (KERNEL_PROPERTY_SCHEMAS, :453), registered by kernelDataExtension.ts:61; 10 kernel types in src/data/blockTypes.ts:44, registered at :62.
  • 44 plugin schemas across 17 propertySchemasFacet.of sites; 18 plugin types across 11 typesFacet.of sites (six prefs-container types and one ui-state container type register through the pluginPrefsExtension/pluginUIStateExtension helpers, src/data/pluginStateExtensions.ts:58,:70, which inject hideFromCompletion at registration).
  • Two plugin schemas are lift-only — registered nowhere directly, entering the registry solely through TypeContribution.properties: status (todo) and daily-note:date (daily-notes).
  • The merged registry is built by mergeLiftedSchemas (src/data/internals/refProjection.ts:29): type-lifted schemas first, then direct registrations, last-wins by name with a console.warn. Wired as the propertySchemas rebuild step in facetBridge.ts:239, which re-runs on any change to typesFacet or propertySchemasFacet.
  • Repo pre-seeds its registries with the kernel maps at construction (repo.ts:183-184, :309-310) — kernel schemas are available before any setFacetRuntime. Every consumer is name-keyed or whole-map: the typed-query compiler (typedBlockQuery.ts:155), reference projection (plugins/references/referenceProjection.ts:32), the property panel (propertyPanel/model.ts, visibility.ts:19), TypeTagger initial values (typeTagger.ts:85), resolvePropertyDisplay (propertyEditors/defaults.tsx:438). There is no id-keyed lookup anywhere.

2.2Block world: user definitions, id-keyed, projected

  • property-schema blocks store {propertyName, presetId, presetConfig} (src/data/properties.ts:170-192); the codec is rebuilt from the code-registered preset: tryBuildSchema validates config through preset.configCodec and calls preset.build(config) (userSchemasService.ts:52-94).
  • ValuePreset (src/data/api/valuePresets.ts:13) is already the behavior-as-code contract: id, build(config) → codec, defaultValue, configCodec, Editor, Glyph, ConfigEditor, hideFromPicker. All 9 existing presets are kernel-owned (kernelValuePresets.ts:72-161), registered under the system:kernel-value-presets toggle — no plugin defines a preset today.
  • Forward-compat is already built: a schema block referencing an unknown presetId is skipped with a warning and re-resolves when the preset's plugin loads (tryBuildSchema skip path + secondarySignal: onValuePresetsChange, userSchemasService.ts:107).
  • User types follow the same pattern: block-type blocks, block id = type id, block-type:properties refList resolved through the schema projector's handle (userTypesService.ts:60-91).
  • The shared lifecycle is ProjectorRuntime (src/data/projectorRuntime.ts): subscribe to {workspaceId, types:[metaType]}, project per row with per-row failure isolation, publish into the facet's durable 'user-data' runtime bucket. Projectors start in an AppRuntimeProvider effect — after first paint (AppRuntimeProvider.tsx:239-243), so user definitions are not available to workspace bootstrap today.
  • Deterministic per-workspace block ids via uuidv5 have precedent in stateBlocks.ts:54-58 (user page, state children) and the kernel pages (propertiesPage.ts:13).

The two worlds already agree on the important shape — definitions reduce to (name, preset, config, scope) and behavior lives in presets. What differs is identity (name vs block id), availability (always vs after projector prime), and who owns registration (merge step vs projector). Unification keeps the block world's shape and gives it the code world's availability.

3Target model — the definition/behavior split

Split today's PropertySchema into a definition (data, on a property-schema block) and behavior (code, a ValuePreset keyed by presetId). The definition is the user-defined shape that already exists, extended with the fields the migration needs:

Property on the property-schema blockStatusMeaning
property-schema:nameexistsDisplay name and (for now) the cell key. Not identity.
property-schema:presetexistsValuePreset.id the codec is rebuilt from.
property-schema:configexistsPreset-specific config, validated by preset.configCodec.
property-schema:change-scopenewOne of the five ChangeScope values (changeScope.ts:5). Absent → block-default, which keeps every existing user schema block valid unchanged (tryBuildSchema hardcodes BlockDefault today, userSchemasService.ts:92).
property-schema:child-backednewThe #288 §6 flag, durable and synced. Absent → false during rollout; seeds and the migration's flips stamp it explicitly.
property-schema:hiddennewHide from the property panel / outline by default (today only expressible as a code-side PropertyEditorOverride.hidden, propertySchema.ts:41, or the system: name-prefix convention, propertyPanel/visibility.ts:21). Readable without the owning plugin.
seed:keynewSeed provenance (§4.2). Absent → user-authored. Shared vocabulary with block-type seeds and future declaration kinds.
seed:revisionnewMonotonic integer, bumped when the code seed changes (§6).

The runtime registry entry — what consumers get from repo.propertySchemas — becomes:

interface PropertySchema<T> {
  readonly name: string
  readonly fieldId: string        // the definition block id — THE identity
  readonly codec: Codec<T>        // preset.build(config)
  readonly defaultValue: T        // preset.defaultValue
  readonly changeScope: ChangeScope
  readonly childBacked: boolean
  readonly hidden: boolean
  readonly origin: 'kernel' | `plugin:${string}` | 'user'  // derived from seed:key
}

Two things deliberately do not change:

  • Typed code handles survive. typesProp, isCollapsedProp, srsNextReviewDateProp, … remain exported consts that call sites pass to block.get/tx.setProperty for compile-time typing. What changes is how they're constructed: defineProperty(name, {codec, …}) becomes a seed declaration whose codec comes from preset.build(config) — the same construction path the projector uses, so the code handle and the block-fed registry entry are behaviorally identical by construction (§4.1).
  • ValuePreset's contract is unchanged — it already carries everything behavior needs — but it splits along the same data/presentation line the schema types already draw (propertySchema.ts:5-11): a codec core (id, build, configCodec, defaultValue, defaultConfig) that lives in the data layer, and a presentation half (Editor, Glyph, ConfigEditor, label, hideFromPicker) joined by preset id in the UI layer. Today the whole preset — codec factory included — lives in src/components/propertyEditors/kernelValuePresets.ts behind a UI toggle, which the data layer cannot import; seed declarations (§4.1) and headless surfaces (CLI, agent bridge) need the codec cores below React. The work beyond the split is registering presets for the codecs that don't have one yet (§10).
Decision

Definition fields live as ordinary block properties (not new columns): they ride properties_json, sync per the existing column-narrow pipeline, work in E2EE workspaces, and are editable through the existing property-schema block renderer. The one identity-adjacent thing that stays derived is origin — computed from seed:key, never stored separately.

4Seeding — code declares, the workspace materializes

4.1Seed declarations replace registrations

Kernel and plugin code stops registering schema objects and starts declaring seeds. The authoring surface keeps defineProperty's ergonomics and returns the typed handle call sites already use:

// before (properties.ts today)
export const isCollapsedProp = defineProperty<boolean>('system:collapsed', {
  codec: codecs.boolean, defaultValue: false, changeScope: ChangeScope.BlockDefault,
})

// after — a seed declaration; the handle is still a PropertySchema<boolean>
export const isCollapsedProp = seedProperty<boolean>({
  seedKey: 'system:kernel-data/property/system:collapsed',
  name: 'system:collapsed',
  preset: 'boolean',            // ValuePreset id; config omitted for void-config presets
  changeScope: ChangeScope.BlockDefault,
  childBacked: true,            // stamped per #288 §6 defaults at the kernel flip
  hidden: true,
})

seedProperty resolves the preset's codec core at module init (a data-layer static import once the §3 preset split lands), calls preset.build(config) for the handle's codec, and records the declaration in the module-level seed table its extension contributes via a new definitionSeedsFacet. For value types the preset can't express in TS (e.g. optionalIdentity<FocusedBlockLocation> building an untyped object codec), the helper takes the type parameter on faith — runtime behavior is identical; the compile-time type is an assertion, exactly as unsafeIdentity<T> is today.

Why one construction path matters

The registry entry is built by the projector from the block (preset.build(config)); the code handle is built by seedProperty from the same (preset, config). Because build must be deterministic in config (already a documented preset contract, valuePresets.ts:19-22), the two produce behaviorally identical codecs even though they are distinct objects. Consumers must not rely on schema-object identity across the two — today they don't: the projector already produces fresh objects per rebuild and every consumer joins by name.

4.2Identity: deterministic per-workspace ids

Decision

fieldId is the definition block id for every schema. Seeded definitions get deterministic per-workspace ids: uuidv5(`${workspaceId}:${seedKey}`, DEFINITION_SEED_NS) — the stateBlocks.ts:54 pattern, so two offline clients (and the local synthesis layer, §5.1) converge on the same row. Ids are per-workspace; cross-workspace copy/paste resolves by name, consistent with #288 §9.

Seed keys are stable strings, never derived from the current display name at runtime: <ownerToggleId>/property/<key> and <ownerToggleId>/type/<key>, e.g. system:kernel-data/property/types, system:srs-rescheduling/property/next-review-date. The owner segment reuses the togglable-boundary ids (src/facets/togglable.ts) that #289 §12 commits to as a public compatibility surface — one stable-id vocabulary across both designs. The trailing key is frozen at introduction; renaming a schema's display name later changes property-schema:name, never the seed key or the block id.

4.3The seed pass

At workspace open — in bootstrapWorkspace, right after ensureSystemPages (src/bootstrap/workspaceBootstrap.ts:210), which guarantees the Properties/Types pages exist — a single idempotent pass:

  1. Probe: one batched read of all seed ids (SELECT id, properties_json, deleted FROM blocks WHERE id IN (…) — on the order of 100 rows).
  2. Diff: for each declaration, compare the stored seed:revision and the seed-owned fields against the code seed (§6 rules).
  3. Write: one tx (chunked if large) that creates missing definition blocks (children of the Properties/Types pages, systemMint ids), restores soft-deleted ones, and rewrites stale ones. Steady state is zero writes — the pass is a read-only probe, same cost profile as ensureStateChild's fast path (stateBlocks.ts:100-109).
  • Scope: ChangeScope.Automation — machine-authored, durable, synced, not undoable (changeScope.ts:24). A user must not be able to ctrl-Z the kernel's schema definitions out of existence, and seed writes must not pollute the document undo stack.
  • Read-only workspaces: skipped entirely (same gate as scheduleWorkspaceBackfills, repo.ts:1862). The synthesis layer (§5.1) serves the registry regardless.
  • Fresh-workspace cost: ~100 definition blocks created once, uploaded once. Existing-workspace upgrade cost: only the definitions whose seeds changed.
  • Plugin scope: a plugin's seeds run only where its code is loaded (its data extension contributes them). A workspace synced to a client without the plugin keeps the definition blocks — that's the point (§5.3) — it just never seeds or upgrades them there.

5The unified registry

5.1One projector, two layers

The registry keeps its existing plumbing — facet runtime buckets feeding the propertySchemas rebuild step (facetBridge.ts:239) — but the inputs change from three sources (kernel registrations, plugin registrations + type-lift, projector bucket) to two layers:

  • Synthesis layer ('seed-synth' runtime bucket): entries computed directly from the in-memory seed declarations the moment a workspace is pinned. Pure function of (declarations, workspaceId) — no DB read, no async. This is what makes kernel definitions available "before first sync" (requirement 3): on a fresh device with an empty DB the registry is complete for kernel + loaded plugins from stage 1.
  • Block layer ('user-data' bucket, the existing projector): authoritative once primed. The projector generalizes from "user schemas" to "all definitions": it publishes an entry for every property-schema block, seeded or user-authored.

Runtime buckets append after static contributions and keyed-map facets are last-wins (facet.ts:242-249, :102-123 — the shadowing order #289 §2.4 verified), so publishing seed-synth before user-data gives block-fed entries precedence by construction. One guard runs projector-side: a seeded block whose seed:revision is older than the loaded code's revision projects the code synthesis instead of the stale block (the seed pass will upgrade the block shortly; until it syncs, the newer app version's behavior should not regress to the older payload). The registry map gains an id-keyed index alongside the name-keyed one — findSchemaByFieldId being O(schemas) is already flagged as a risk in #288 §12.

5.2What dies

  • dies mergeLiftedSchemas's name-merge and the TypeContribution.properties type-lift (refProjection.ts:29-57): the rebuild step reads one merged map from the buckets. (changedRefSchemaNames and the reprojection scheduling survive unchanged — they compare any two maps.)
  • dies direct propertySchemasFacet.of registrations — all 17 sites (and, in the types slice, all 11 typesFacet.of sites). The facets themselves survive as the internal transport between buckets and the bridge.
  • dies the "kernel/plugin schemas have no backing block" asymmetry: UserSchemasService.getSchemaBlockId's "returns undefined for kernel/plugin schemas" caveat (userSchemasService.ts:144-147), and eventually the service's "user" prefix.

5.3Missing-plugin degradation — the declaration tier

Declaration-only entries live in a separate map (repo.propertyDeclarations, keyed by both name and fieldId), not in repo.propertySchemas — every existing consumer of the schema map dereferences schema.codec unconditionally (typed queries typedBlockQuery.ts:110-124, reference projection, the panel's decode), so a codec-less entry in that map would be a footgun. Consumers that need classification (the #288 machinery, panel row rendering, field-row recognition) read declarations ∪ schemas.

Degraded UX for a declaration-only property: the panel shows the row with its display name and raw value, read-only, attributed via seed:key ("provided by srs-rescheduling — not installed/disabled"). The value is never decoded, never edited, never dropped.

This fixes a standing hazard class

Today, toggling a plugin off makes its schemas vanish from the registry entirely — the silent-deletion vector that once wiped ~10k next-review-date backlinks on SRS toggle-off and forced the defensive latestRefProjectionSchema machinery (refProjection.ts:121-151). After unification the definition persists as a block: for kernel-preset schemas ref-ness stays fully known (the ref/refList preset id is on the block), and even declaration-only entries prove the name is still a schema. "Absent from the registry" stops being ambiguous between "toggled off" and "deleted".

6Seed upgrades & write policy

6.1Reconciliation rules

The genuinely new problem class: app v(N+1) changes a kernel schema's seed while the block may have been touched by an older client or a user. Following #289 §9's provenance approach, adapted from hashes to revisions (declarations are small structured data; an explicit monotonic seed:revision per definition gives ordering, which a hash cannot):

Seed pass observesAction
Block missing / soft-deletedCreate / restore with current seed payload, stamp seed:revision.
seed:revision < code revisionUpgrade: rewrite all seed-owned fields to the current seed, stamp the new revision. One tx, one column-narrow PATCH — per-column LWW takes the whole write from one winner, so mixed-version clients can't interleave field-level.
seed:revision = code revision, fields differDrift (a user edit through the panel, an agent-bridge write, an import): repair seed-owned fields back to the seed. Self-healing, same philosophy as #288 §5's consistency sweep.
seed:revision > code revisionLeave untouched — a newer client owns this definition. The registry's projector-side guard (§5.1) does not apply (the block is newer, not staler); the local, older app runs against the newer definition, which is exactly the forward-compat the preset system already handles (unknown preset → declaration tier; known preset, newer config → configCodec validates or the row degrades).

The seed source of truth stays in code; blocks are its materialization. A seed change is a code change (revision bump in the declaration) that materializes per-workspace on next open — there is no fleet data-migration step, and unedited workspaces converge via ordinary sync of the one upgraded row.

6.2Field ownership & write policy by origin

Decision (v1)

For seeded definitions, every definition field is seed-owned: name, preset, config, changeScope, childBacked, hidden. The UI renders seeded definition blocks read-only (delete/rename/config affordances disabled, attributed to their origin); the seed pass repairs out-of-band edits. Per-field user ownership (e.g. letting a user override hidden, or recolor a plugin type) is a deliberate v2: it needs a per-field ownership vocabulary and merge rules — §15. User-authored definitions keep today's full editability.

7Name resolution & collisions

Ids cannot collide; display names can. Names are load-bearing in three places: the properties_json cell key, alias-entry ([[status]] typed by hand — #288 §7's field-row grammar resolves the schema-name map first), and every name-keyed consumer in §2.1. Today mergeLiftedSchemas resolves collisions last-wins-by-load-order with a warning — nondeterministic across clients with different plugin sets, which #288 §6 flags as a hard error under child-backing (the cell key is the name while child identity is the fieldId; a per-client winner strands the loser's field rows).

Decision

Because every definition is now a synced block, collision resolution can finally be deterministic fleet-wide: the name winner is the earliest definition — ordered by (created_at, id) over definition blocks, an order every client computes identically regardless of installed plugins. Losers stay registered by fieldId (child rows keep classifying, backlinks keep resolving) but are excluded from the name map and the cell projection, and surface in the Properties page UI as shadowed — rename to activate. Earliest-wins protects existing user data from newly-arriving plugin seeds: a user's long-standing status schema is not hijacked when a plugin introducing status is installed; in a fresh workspace, seeds run first and win.

  • Prevention first: addSchema and the rename flow reject a name already present in declarations ∪ schemas (exact match; case-insensitivity is an open question, §15). The collision rule is for the unpreventable case: two definitions created concurrently offline, or a plugin seed arriving into a workspace that already used the name.
  • The audit says this is real: srs-rescheduling seeds bare, generic names — interval, factor, grade, archived, next-review-date, review-count, snapshot-history (srs-rescheduling/schema.ts:19-55) — exactly the names users would pick. New plugin schemas should namespace (srs:interval) as most already do; renaming existing plugin names is a data migration (the cell key changes) and is explicitly not attempted here — earliest-wins keeps the status quo working either way.

8Editor overrides & presentation joins

PropertyEditorOverride joins schemas by name today (propertySchema.ts:32; consumed via uis.get(name) in resolvePropertyDisplay, defaults.tsx:440). The inventory: 20 overrides — 13 kernel hidden:true markers, the types autocomplete editor, and 6 plugin config-property editors.

Decision

Two moves. (1) The 13 kernel hidden:true overrides dissolve into the definitions themselves — hidden is now a definition field (§3), synced and classifiable, which #288's hidden-field-row filtering needs anyway (its §10 notes the curated hidden list must grow at the kernel flip; a definition field replaces the curated list wholesale). (2) The remaining editor overrides join by fieldId instead of name: code-declared overrides reference the seed handle (overrideFor(typesProp, {Editor: …})), which resolves to the per-workspace definition id — immune to display-name changes once renames unlock. The name join dies with the cutover slice.

resolvePropertyDisplay's preset join (presets.get(codec.type)) is untouched — editor resolution stays keyed by the codec's type discriminator, which is orthogonal to definition identity.

9Types get the same move

Everything above applies to block types with one extra wrinkle: type ids are stored data. Every block's types property and the block_types index hold strings like 'page', 'todo', 'daily-note' — for user types the string is the definition block id; for kernel/plugin types it's a legacy constant. Rewriting those strings across every workspace would be a huge, risky data migration for zero user-visible gain, so:

Decision

block-type definition blocks gain block-type:type-id — the string written into typesProp. Absent (every existing user type) → the block id, preserving the block-id = type-id rule. Seeded kernel/plugin types set it to their legacy constant ('page', 'todo', …). The type registry stays keyed by type id; getTypeBlockId becomes total (every registered type has a backing block). Whether typesProp values should become refs to definition blocks is out of scope — flagged as future work, unchanged from the project brief.

The rest is mechanical application of §3–§6:

  • New definition fields on block-type blocks: block-type:type-id, block-type:hide-from-completion (today code-only, blockType.ts:26), plus seed:key/seed:revision. Existing fields (label, description, color, hide-from-block-display, properties refList) already cover the rest of TypeContribution.
  • TypeContribution.properties → block-type:properties refList of definition-block ids. Seed declarations reference the property seed handles; the seeder resolves them to per-workspace ids. The two lift-only schemas the audit found (status, daily-note:date) get their own property seeds, which their type seeds reference — the lift path they currently ride dies.
  • The prefs/ui-state container types (pluginPrefsExtension/pluginUIStateExtension, 7 call sites) seed the merged shape — the helper-injected hideFromCompletion: true is part of the seed payload, not re-injected at registration.
  • addTypeInTx keeps working unmodified: it validates against types.get(typeId) and encodes caller-supplied initial values by schema name (typeTagger.ts:56, :85) — both maps still exist, just block-fed. snapshotTypeRegistries (repo.ts:1579) is shape-unchanged.
  • Sequencing: types cut over after property schemas (their own slice, §13) — the type projector already depends on the schema projector (userTypesService.ts:132), and seeded type declarations need property seeds to exist first.

11What this buys downstream

11.1For the properties-as-blocks migration (#288)

  • Synthetic-id leak: deleted. No property:<name> ids exist; the reference-projection special-case #288 slice B planned for non-block fieldIds is unnecessary. Every reference_target_id is a real block id, and field rows on schema definitions give "where is this field used?" backlinks for kernel/plugin fields too.
  • childBacked durability: solved — it's a synced definition field; a plugin's flip is a seed-revision bump (§6.2), not a code constant.
  • The fully-child-backed eligibility marker widens. #288 §11 step 5 restricts marking to rows whose keys are kernel/user-classifiable, because "a plugin not installed here may be live elsewhere". With definitions as synced blocks, plugin keys are classifiable on every client (§5.3) — rows carrying plugin-backed keys become markable, which step 4-5's own analysis says is "effectively a prerequisite for this step delivering much". (Floor: clients older than the cutover slice can't classify; the marker work is post-cutover anyway.)
  • Cold-start two-phase projection: dissolved into §4.4's staging — phase 1 is a pure computation, not a cell-dependent discovery pass.
  • Name-collision hard error: implemented as §7's deterministic rule rather than a per-client warning.

Follow-up (coordinate, don't edit as part of this work): once this design is accepted, #288's doc trims §6's synthetic-id residuals, the §11 step-5 marker restriction, and the §11/§12 two-phase bootstrap discussion against it.

11.2For plugin materialization (#289)

  • Its "declaration materialization" parallel track is fully designed (this doc); its §8 "edit this type" affordance becomes: flip the seeded definition's fields from seed-owned to user-owned — the v2 per-field ownership work (§15), with the block already in place.
  • Seed keys reuse toggle ids (§4.2), reinforcing its §12 commitment to toggle-id stability.
  • A replaced (ejected) plugin's seeds come from the replacement's code at the same manifest position — declaration seeding composes with ejection without new machinery.

11.3Housekeeping wins

  • The registry-vanishes-on-toggle hazard class closes (§5.3 callout).
  • The 13-entry curated hidden list (typesPropertyUi.ts:31-50) and #288 §10's "extend it to every system schema" deliverable collapse into a definition field.
  • describe-runtime/agent-bridge introspection (plugins/agent-runtime/dataModelGuide.ts:154) reads one coherent surface; the authoring catalog's schema-registration templates update to seed declarations.

12Edge cases & carve-outs

  • Non-document scopes seed too. UiState / UserPrefs / Automation schemas (9 kernel + 12 plugin) become seeded definitions like everything else — definitions are ordinary BlockDefault data even where the values they describe are per-user; #288 §6's permanent childBacked: false carve-out is stamped in their seeds. No circularity: reads and writes of scoped values go through code handles (block.set(schema, …)), which never consult the registry; the seed pass itself writes definition blocks using kernel property handles, not registry lookups.
  • E2EE workspaces: no new constraints. Definition blocks are ordinary blocks; their properties ride the encrypted properties_json envelope; the projector reads locally-materialized plaintext. (Contrast with #288's reference_target_id gate — nothing here adds a synced plaintext column.)
  • Read-only workspaces: seed pass skipped; synthesis layer serves kernel/plugin entries; user definitions arrive as synced blocks. A viewer client never writes definitions.
  • Safe mode / plugin toggled off: the plugin's seeds and presets don't load; its definition blocks persist and project as kernel-preset entries or declarations (§5.3). Nothing is deleted by absence — matching the projector's existing "absence is not deletion" stance.
  • Old clients during rollout: pre-cutover clients ignore the new definition fields (unknown cell keys in the bag are preserved by column-narrow upload) and keep their code registries; seeded blocks are inert extra rows on the Properties/Types pages (visible, harmless — the pages already host user definitions). A pre-cutover client that deletes a seeded block loses nothing: the next current client's seed pass restores it.
  • Cross-workspace copy/paste of a definition block: the pasted block carries seed:key for a seed the destination already has → it's a duplicate name at a wrong id; the collision rule (§7) shadows it and the UI surfaces it. Paste of field rows across workspaces resolves by name per #288 §9. A paste-time strip of seed:* props keeps the common case clean.
  • Workspace deletion/reset: nothing special — seeds re-materialize on next open, same as kernel pages.

14Rejected alternatives

  • Name-based identity everywhere (keep merging by name, add childBacked to code): fails #288's requirements 1-2 outright — names collide across sources per-client, and code constants fork data across plugin versions. This is the status quo the migration doc already rejected.
  • Global (cross-workspace) definition ids: a block id is unique per row; the same definition in two workspaces would collide in one local DB. Per-workspace ids + name-based cross-workspace resolution match #288 §9 and the existing state-block precedent.
  • Registry entries synthesized from code only, blocks as passive mirrors (seed pass writes blocks but registry never reads them): loses the entire point — user edits of plugin definitions (v2), missing-plugin classification, and #288's durable childBacked all need the block to be authoritative.
  • Columns instead of properties for the new definition fields: schema migration + sync-rule deploy + e2ee surface for data that has no query-path need; properties ride existing machinery.
  • Hash-based seed provenance (#289's base-hash) instead of revisions: hashes detect difference but not direction — an old client couldn't tell "my seed is stale" from "the block is newer than my seed" and would fight newer clients. Revisions give ordering; drift detection falls out of the revision-equal + fields-differ case (§6.1). (#289's hash approach remains right for code, where there is no total order on user edits.)
  • Seeding lazily on first use (definition materialized when a value is first written): leaves the registry incomplete for pickers/panel, makes fieldIds appear at unpredictable times mid-#288-migration, and saves ~100 rows once. Not worth the nondeterminism.
  • Migrating kernel/plugin type-id strings to block ids: a fleet-wide rewrite of every types array and the block_types index for cosmetic uniformity; block-type:type-id costs one property (§9).

15Open questions

  1. Per-field user ownership (v2): which seeded fields become user-editable first (likely hidden, type color/label), and how ownership is recorded (per-field marker vs an "edited fields" set) so the seed pass stops repairing them. #289 §8's "edit this type" affordance lands here.
  2. Case sensitivity of name collision/prevention — exact match (status quo of the alias map) or case-insensitive? Affects §7's prevention check only; the deterministic rule works either way.
  3. seed:hash alongside seed:revision: field-compare at revision-equal is cheap at this scale; a stored payload hash would make drift detection O(1) and align vocabulary with #289. Decide in slice B implementation.
  4. Picker exposure of declarations: should AddPropertyForm's picker offer schemas whose plugin is disabled (they're fully functional if kernel-preset-built, §5.3)? Leaning yes for kernel-preset ones, no for declaration-only.
  5. New-plugin type ids: should plugins introduced after slice D use block ids as type ids (no legacy constant)? Leaning yes — block-type:type-id then exists only for grandfathered types.
  6. Definition cleanup affordance: UX for definitions whose plugin is gone for good (bulk archive from the Properties page?). Deliberately post-v1; nothing breaks meanwhile.
