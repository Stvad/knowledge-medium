Share-mode aliases & references — explainer

Companion to shares-and-api-tokens.html. Focuses on what happens when a share recipient types [[Foo]] and Foo already exists in the workspace but they can't see it. 2026-05-15.
TL;DR. Reference creation in a share stays local-first. The guest types [[Foo]], the client creates a local block claiming alias "Foo" (just like today), and syncs. The only new thing is what happens if Postgres rejects the upload because the workspace already has a "Foo": the existing ProcessorRejection('alias.collision') machinery hands us the canonical block id, and we use that to (1) rewrite any references that pointed at the local block to point at canonical instead, and (2) remove the failed alias claim from the local block. The local block stays put with its children intact — it just stops claiming the alias. No pre-flight remote calls. The remote endpoint we do need is a small peek_block(id) for rendering chips when a reference target isn't in local view — same shape as resolving any cross-scope reference.

1. What I got wrong in the design discussion

Earlier I described a flow with a resolve_workspace_alias RPC the client would call before creating a reference. That breaks the local-first model: every keystroke that produces a [[name]] would require a remote round-trip before the local block exists.

It's also unnecessary. The information that RPC would return (canonical id for an alias) is exactly what the server already includes in the alias_collision rejection payload when an optimistic local write collides at sync time. So we can run the happy path entirely locally and only do remote work in the (rare) collision case.

The "annotation block" concept I introduced is also less novel than I made it sound. An annotation is just what a share-local block looks like after its alias claim got rejected on sync. It's a state of a normal block, not a new schema concept.

2. What's in the local DB for a share recipient

Recap the data architecture so the rest of the doc has the right vocabulary:

  • Each device has a local SQLite, populated by PowerSync streams keyed off the user's grants.
  • For a workspace member: the workspace_data stream gives them every non-deleted block in their workspaces.
  • For a share recipient (whether human or API token): the shared_blocks stream gives them only the blocks their share covers — the subtree (or the workspace, if it's a workspace-scope share).
  • The client maintains a derived block_aliases(workspace_id, alias, block_id) table locally, updated by triggers as blocks change. It only contains aliases of blocks present in the local DB.

So a share recipient's local alias index sees only aliases of blocks inside their share. Aliases anywhere else in the workspace are invisible to them.

3. The optimistic flow for [[Foo]]

When the share recipient types [[Foo]] in some block B inside the share, the client does what it does today:

  1. Look up "Foo" in the local block_aliases table.
  2. If found, set B.references_json to include that local block's id. Done.
  3. If not found:
    • Create a new block locally: id L, alias "Foo", parent = wherever new pages go in the share (the share root, by default).
    • Set B.references_json to include L.
  4. Queue both writes for sync.

This is identical to the workspace-member flow. No new code path on the "type and create" side.

4. What can go wrong at sync time

The local block L uploads to Postgres. The Postgres-side block_aliases (workspace_id, alias) UNIQUE constraint (the Phase B prereq from the main doc) checks for a collision in the workspace's namespace. Two outcomes:

OutcomeMeaningServer response
No collision No block in the workspace claims "Foo" anywhere. Upload succeeds. L becomes the canonical "Foo" of the workspace. Whether it's currently inside or outside any share doesn't matter — it's the only claimant. Done.
Collision A block with id C already claims "Foo" somewhere in the workspace, possibly outside the share recipient's view. alias_collision rejection, payload includes C (the canonical id). The client's existing ProcessorRejection('alias.collision', meta) handler in repo.ts:1085 receives this.

The non-collision case is the typical one for fresh page names — nothing more to discuss. Everything below is about handling collisions.

5. Reconciliation on collision (this is Case C)

The rejection tells the client that L can't claim the alias because canonical C already does. The client's job: get to a consistent state where:

  • No local block claims "Foo" as an alias (the canonical does, and we can't duplicate it).
  • Any references that pointed at L instead point at C.
  • The user's typed work (the block L itself, any children they typed under it) is preserved.

The reconciliation has three steps:

  1. Demote L from canonical to annotation. Remove the alias = "Foo" entry from L.properties_json. The block keeps its id, its other content, and its children. It's now a normal share-local block.
  2. Add a reference from L to canonical C. Append C to L.references_json. L is now an "annotation of C" — a share-local outline node whose content is associated with the workspace's canonical Foo.
  3. Rewrite references that pointed at L to point at C. In particular, B.references_json entry for L gets replaced with C. (Other blocks in the share that also reference L get the same rewrite.) The references now resolve to canonical, which is the right answer semantically.

All three steps are local writes against the local SQLite. They're enqueued for sync. None of them collide with the workspace (we removed the alias claim; L is now just a share-local block with some references; B's updated references_json is just an ordinary edit).

7. What does references_json end up pointing to

This was the second part of your question. The answer changes across the timeline:

WhenB.references_jsonL.references_json
T1, just after typing [L] — points at the local block we just made []
T3, after reconcile [C] — rewritten to canonical [C] — L now annotates C

So references in the share converge to canonical block ids, even though the client never had to ask the server "what's the canonical id for this alias" up front. The collision rejection is what conveys the canonical id to the client, lazily.

Why is this important? Because the "append from share into the main workspace" flow requires that references inside the share already resolve to the right workspace blocks. If the owner promotes a guest-authored subtree out of the share, the references inside it work without rewriting.

8. What the user sees

From the guest's perspective:

  1. They type [[Foo]]. L appears locally with its label "Foo." They can outline under it (Tab/Enter). Their children land under L's id, share-local.
  2. Sync runs in the background. If no collision, nothing changes visually.
  3. If collision, the renderer's view of L updates: where before it looked like a normal block named "Foo," now it shows with an indicator that there's a workspace-canonical "Foo" outside their view (e.g., a chip "Foo · referenced from workspace"). Their children are still there, still editable, attached to L.
  4. The [[Foo]] reference in B still renders as "Foo," but clicking it now leads them to L (the annotation in their share) rather than canonical (which they can't reach). The chip can offer "Foo (workspace) · no access" as a secondary affordance.

The key UX property: no work is lost on collision. The local block stays, its children stay, the reference still resolves to a renderable thing.

9. The remote endpoint we still need (and the one we don't)

Needed: peek_block(id) → {exists, accessible, label}

For the renderer to display "Foo · referenced from workspace" on the chip after reconciliation, it needs to know something about canonical C: at minimum that C exists and isn't accessible. Optionally a label.

This is a small SECURITY DEFINER RPC. It bypasses RLS for the existence/accessibility check, but only returns metadata derivable without exposing content (e.g., the alias). It's called lazily by the renderer when it encounters a reference id not present in the local DB. Results can be cached per session.

This is the same RPC needed for cross-workspace references (a separate but related feature). One helper, two use cases.

Not needed: a pre-flight resolve_workspace_alias

I described this earlier. It's redundant. The information it would convey (canonical id for an alias) is already in the alias_collision rejection. Adding a pre-flight RPC would mean every [[Foo]] keystroke costs a remote round-trip on the synchronous write path, which is exactly what the local-first model was designed to avoid.

An optional background-prefetch of "aliases in this workspace I might want to reference" could exist as a perf tweak (warm a local hint table), but it's not part of the correctness model.

10. Putting it all together

The model that keeps the architecture local-first while supporting subtree shares with workspace- canonical aliases:
  1. Type-and-create stays local, with optimistic alias claim.
  2. Server-side block_aliases UNIQUE(workspace_id, alias) is the source of truth for uniqueness. On collision, it rejects loudly.
  3. The rejection's payload carries the canonical id.
  4. The client demotes its local block to an annotation (drops the alias claim, references canonical) and rewrites in-share references to canonical. All locally.
  5. Subsequent rendering uses peek_block to display references to inaccessible canonicals.
The "annotation block" is just the post-collision state of an ordinary block. No new schema concept, no synchronous remote calls in the typing path.

11. Open questions worth being explicit about

  1. Collision-handling latency. Between T1 (type) and T3 (reconcile), the local state has L claiming "Foo" and B referencing L. If the user types more references to "Foo" in this window, those too point at L. The reconciliation step rewrites all of them at once. Manageable, but tests should cover the "five quick refs before sync completes" case.
  2. Concurrent shares both first-creating "Foo". Two share recipients in different shares both type [[Foo]] at the same time, neither sees any canonical yet. Both create local L1 and L2. Both upload. First one wins; second gets alias_collision with the winner's id as canonical. Self-healing via the same reconciliation.
  3. What does peek_block reveal? Returning the alias as label is harmless (the guest already typed it). Returning content snippets is a leak; default to alias-only.
  4. What if the guest deletes their annotation block L? Annotations are normal blocks; delete cascades to children. References to L elsewhere become stale (point to a deleted block). After T3, though, references should point to C — not L — so deletion of L affects only its children, not other refs.
  5. Aliases on non-share-leaf blocks. The flow above describes creating an alias via reference auto-create. The guest could also try to alias an existing block in the share (set its properties_json to include an alias). Same code path: optimistic local claim, server-side uniqueness check, collision handled identically. No new machinery.
