Unified block query / backlinks filter

One query language. Backlinks becomes a thin wrapper. Predicates can target the block itself or any ancestor.

The bug we're fixing

A backlinks panel showed a block even after the user added DONE to "Remove reference". The block had status: done set.

Root cause: the backlink filter matches against the reference graph (block_references). status is stored as a scalar property, not a reference, so there's no [[DONE]] ref to match against. Same problem for srs:archived.

The decision

Don't invent fake refs for property values. Instead, extend the existing TypedBlockQuery to express two new things:

  • Property predicates (already partially there as where) usable as positive or negative filters.
  • Scope: a predicate matches against the block itself, or against the block or any ancestor.

Backlinks then becomes: "queryBlocks for sources that reference target X, plus whatever extra predicates the user added."

The predicate shape

interface BlockPredicate {
  scope?: 'self' | 'ancestor'             // default: 'self'. 'ancestor' = block-or-any-ancestor
  where?: Record<string, unknown>          // scalar equality, codec.where.encode
  referencedBy?: {id: string; sourceField?: string}
}

No operators beyond equality. Negation comes from list membership (exclude vs match), not from the predicate itself.

The TypedBlockQuery extension

interface TypedBlockQuery {
  workspaceId?: string
  types?: readonly string[]
  // Existing fields — shorthand for {scope:'self', ...}. Unchanged.
  where?: Record<string, unknown>
  referencedBy?: {id: string; sourceField?: string}
  // New: ANDed predicates. Each carries its own scope.
  match?: readonly BlockPredicate[]
  // New: NORed predicates. Block matches iff none of these match.
  exclude?: readonly BlockPredicate[]
}

Examples

1. Existing typed query (unchanged)

// All open todos.
repo.queryBlocks({types: ['todo'], where: {status: 'open'}})

2. Backlinks with property + ref filters

"Sources referencing today's daily note. Exclude any whose source-or-ancestor has status=done or references [[archived]]. Require source-or-ancestor to reference [[work]]."

repo.queryBlocks({
  referencedBy: {id: dailyNoteId},                              // self-scope shorthand
  match: [
    {scope: 'ancestor', referencedBy: {id: workId}},
  ],
  exclude: [
    {scope: 'ancestor', where: {status: 'done'}},
    {scope: 'ancestor', referencedBy: {id: archivedId}},
  ],
})

3. Backlinks as a thin wrapper

backlinks(targetId, filter) = repo.queryBlocks({
  referencedBy: {id: targetId},
  match: filter.include,
  exclude: filter.exclude,
}).filter(b => b.id !== targetId)   // drop self-reference

The backlinks-specific SQL goes away. The ancestor-chain CTE moves into the typed query compiler, used only when a predicate has scope: 'ancestor'.

Why this shape (vs. alternatives)

I considered three shapes and recommend the one above (Option A). Open question — see below — is whether to keep the shorthand where/referencedBy at all, or force everything through match/exclude.

Option A: keep where/referencedBy + add match/excludeRecommended

Shorthand for self-scope stays. Scoped predicates go in arrays. ~15 existing call sites untouched.

queryBlocks({types:['todo'], where:{status:'open'}})    // stays
queryBlocks({referencedBy:{id:target}, exclude:[{scope:'ancestor', where:{status:'done'}}]})

Option B: uniform match/exclude only, no shorthand

Every predicate is a BlockPredicate with explicit scope. More uniform mental model. Rewrite every existing call site (hooks, services, ~10 test cases).

queryBlocks({types:['todo'], match:[{scope:'self', where:{status:'open'}}]})

Option C: parallel ancestorWhere / ancestorReferencedBy fields Falls over

Each scope gets its own pair of fields. But: only one referencedBy per scope, and exclude/include need duplicate *Not fields. Six fields and still can't express "exclude (status=done OR refs [[archived]])". Discarded.

SQL implementation sketch

The typed query compiler grows one CTE that runs only when an ancestor-scoped predicate is present:

-- emitted only if any predicate has scope:'ancestor'
WITH RECURSIVE ancestor_chain(block_id, anc_id, anc_props, depth, path) AS (
  SELECT b.id, b.id, b.properties_json, 0, '!' || hex(b.id) || '/'
  FROM blocks b WHERE b.workspace_id = ? AND b.deleted = 0
  UNION ALL
  SELECT ac.block_id, parent.id, parent.properties_json, ac.depth + 1, ac.path || '!' || hex(parent.id) || '/'
  FROM ancestor_chain ac
  JOIN blocks parent ON parent.id = (SELECT parent_id FROM blocks WHERE id = ac.anc_id)
  WHERE parent.deleted = 0 AND ac.depth < 100 AND INSTR(ac.path, '!' || hex(parent.id) || '/') = 0
)

Each ancestor-scoped predicate becomes:

EXISTS (
  SELECT 1 FROM ancestor_chain ac
  WHERE ac.block_id = b.id
    AND <compiled predicate against ac.anc_id / ac.anc_props>
)

Self-scoped predicates compile the same as today (no CTE involvement). Excludes wrap their EXISTS in NOT.

Migration

None. Old stored filters ({includeIds, removeIds}) are discarded; old daily-note defaults reset. Property name can be renamed to make the discard explicit.

Decisions locked in

  • Property predicates are real SQL predicates, not synthetic refs.
  • Per-predicate scope ('self' or 'ancestor'); 'ancestor' includes self.
  • Negation = list membership (exclude vs match), no operator on the predicate.
  • Backlinks wraps queryBlocks. The dedicated backlinks SQL goes away.
  • Chip UI: autocomplete from registered property schemas, value editor from the codec's preset.
  • No data migration — discard old filter values.

Open question

Should we go with Option A (shorthand + new arrays) or Option B (uniform arrays only)?

Option A is backwards-compatible and keeps the common case terse — the existing {where: {status: 'open'}} shape stays. Option B is more uniform but rewrites ~15 call sites for no functional gain.
