Alias sync + rename — case-by-case

Two post-commit processors in two plugins, both watching {content, properties} on blocks:

  • Sync (alias plugin): same-block content↔alias reconciliation. Replace when old value matches the other side; heal additively when the block has drifted.
  • Rename / backlink rewrite (references plugin): cross-block. When aliases are removed or swapped, rewrite source backlinks. Lives next to parseReferences because it needs the block_references projection to find sources.

Sync composes into rename: when a content edit produces an alias-array swap, the rename processor picks it up uniformly via the alias diff. No special path.

Sync (content ↔ aliases, replace)

Case A1 · Content edit on aliased block (clean rename via content) sync → rename
Target — before
content:"Old name"
aliases:["Old name"]
→
Target — after (user edit)
content:"New name"
aliases:["Old name"]
⇣ sync
After sync processor
content:"New name"
aliases:["New name"]
Sync action: diff: content went "Old name" → "New name". Old value was in aliases → replace that entry with the new content (not append). Result is a 1-for-1 alias swap.
Cascade: the alias-array swap re-fires the watcher; rename processor sees {removed: ["Old name"], added: ["New name"]} → rewrites [[Old name]] → [[New name]] in every backlink source. Sync and rename compose — content edits and alias edits flow through the same backlink-rewrite path.
Case A2 · Content edit, new value already an alias sync collapses an alias
Target — before
content:"X"
aliases:["X", "Y"]
→
Target — after (user edit)
content:"Y"
aliases:["X", "Y"]
⇣ sync
After sync
content:"Y"
aliases:["X", "Y"]
Sync action: old content "X" is in aliases — replace it with "Y", which dedupes against the existing "Y" entry. Net effect: X is dropped. The user said "this is now called Y"; keeping X around would contradict that.
Cascade: rename diff = {removed: ["X"], added: []} — pure remove with aliases still present. Hits Case R4 below: backlink sources rewrite [[X]] → [X](((target-id))) to preserve original display text.
Case A3 · Content drifted from aliases (no match) — healing sync, additive
Target — before
content:"already drifted"
aliases:["Original"]
→
Target — after (user edit)
content:"another edit"
aliases:["Original"]
⇣ sync
After sync
content:"another edit"
aliases:["Original", "another edit"]
Sync action: old content "already drifted" is not in aliases → no anchor to replace. Heal by adding the new content as a fresh alias. From this point on the block is in-sync (next content edit will hit A1/A2). No backlink rewrite — rename diff is {removed: [], added: ["another edit"]}, which is Case R3.

Reverse sync (alias edit → content)

Case AR1 · Alias rename, content matches the renamed alias sync
Target — before
content:"Foo"
aliases:["Foo"]
→
Target — after (user edits alias)
content:"Foo"
aliases:["Bar"]
⇣ sync
After sync
content:"Bar"
aliases:["Bar"]
Sync action: alias diff = {removed: ["Foo"], added: ["Bar"]}, and content === "Foo" (the removed value) → rewrite content to "Bar". Symmetric with A1 (content edit propagating to alias).
Cascade: rename processor still sees the same alias diff and rewrites [[Foo]] → [[Bar]] in source backlinks (Case R1). The two processors fire on the same commit; they don't depend on each other's writes.
Case AR2 · Alias rename, content does NOT match no content write
Target — before
content:"some long body text"
aliases:["Foo"]
→
Target — after
content:"some long body text"
aliases:["Bar"]
Sync action: content doesn't match the removed alias → no anchor. Leave content alone. (Block is drifted from the title perspective, but the alias rename is what the user expressed; respecting it.)

Rename (target alias edit → backlink rewrite)

Case R1 · Clean rename (1-for-1 in aliases) rename
Target — before
aliases:["Old"]
→
Target — after (alias edit)
aliases:["New"]
Source block(s) — before
content:"See [[Old]] for context."
⇣
Source block(s) — after rename processor
content:"See [[New]] for context."
Rename action: diff = {removed: ["Old"], added: ["New"]} with equal cardinality 1 → treat as rename. Find sources via block_references where target_id = this.id, rewrite [[Old]] → [[New]] in each source's content. parseReferences re-fires on those edits and refreshes the references column for free.
Case R2 · Rename with other aliases left untouched rename
Target — before
aliases:["Old", "X", "Y"]
→
Target — after
aliases:["New", "X", "Y"]
Rename action: diff = {removed: ["Old"], added: ["New"]}, others unchanged → 1-for-1 rename of Old→New. Same backlink rewrite as R1. X and Y are unaffected.
Case R3 · Pure alias add no-op for rename
Target — before
aliases:["A"]
→
Target — after
aliases:["A", "B"]
Rename action: diff = {removed: [], added: ["B"]} → not a rename. Nothing to rewrite. Future [[B]] typings resolve here via aliasLookup.
Case R4 · Pure alias remove (some aliases remain) preserve display text
Target — before
aliases:["A", "B"]
→
Target — after
aliases:["A", "B"]
Source — before
content:"... [[B]] ..."
⇣
Source — after rename processor
content:"... [B](((target-id))) ..."
Rename action: diff = {removed: ["B"], added: []}. The removed alias still has backlinks → rewrite each [[B]] in source content to [B](((target-id))) (the existing aliased-blockref syntax — see remark-blockrefs.ts). Preserves the original display text the source author wrote while pointing at the same target by stable id — no silent re-targeting, no aliasLookup miss, no orphan stub seat.
Case R5 · Multi-rename / consolidation (shouldn't happen via UI, but) same as R4
Target — before
aliases:["A", "B"]
→
Target — after
aliases:["C", "D"]   or   ["C"]
Source — before
content:"... [[A]] ... [[B]] ..."
⇣
Source — after rename processor
content:"... [A](((target-id))) ... [B](((target-id))) ..."
Rename action: Both the multi-rename and consolidation shapes get the same treatment as R4: for each removed alias α with backlinks, rewrite [[α]] → [α](((target-id))). UI is expected to commit one rename per edit (already the case per current UI), so this branch is mostly a safety net. Special-casing the diff heuristic isn't worth the complexity.
Case R7 · Remove last alias (aliases becomes empty) forced blockref form
Target — before
aliases:["A"]
→
Target — after
aliases:["A"]
Source — before
content:"... [[A]] ..."
⇣
Source — after rename processor
content:"... [A](((target-id))) ..."
Rename action: No aliases remain — main-alias fallback isn't available. Must use the [A](((target-id))) form to keep the link pointing at the target. This is the "block-ref form is the only option" floor of the fallback ladder.

Decision ladder

Sync processor (alias plugin) — fires on {content, properties} change to the same block:

  1. Content changed, old value ∈ aliases (A1, A2) → replace that alias entry with new content. Dedupe.
  2. Content changed, old value ∉ aliases (A3 — drift) → add new content as a fresh alias. Block heals.
  3. Alias diff is 1-for-1, content === removed alias (AR1) → rewrite content to the new value.
  4. Otherwise → no sync write. The rename processor may still act on the alias diff.

Rename processor (references plugin) — fires on alias-property diff. For each α ∈ removed with live backlinks:

  1. 1-for-1 swap (R1, R2, A1-cascade, AR1-cascade): |removed| = |added| = 1 → rewrite [[α]] → [[new]].
  2. Anything else with backlinks (R4, R5, R6, R7, A2-cascade): rewrite [[α]] → [α](((target-id))) (aliased blockref). Preserves display text; doesn't depend on what's left in aliases.
  3. Pure add (R3): no removed aliases → nothing to rewrite.

Interaction with parseReferences (sanity check)

Idempotency on second pass

Sync and rename writes re-fire the field-watcher. Convergence depends on the second pass being a no-op:

  • Sync (A1/A2/A3/AR1): first pass writes the reconciled side. Second pass sees no diff in its trigger field → no work. Cross-processor: sync's alias write re-fires rename, which acts on the (now-final) alias diff.
  • Rename (R1/R2 etc): rewrites source content. Source edits trigger parseReferences; target wasn't touched again, so rename doesn't re-fire.
  • Rename → blockref form (R4/R5/R6/R7): source content now has [label](((id))). parseReferences re-fires on the source, parses the aliased blockref, refreshes the references column to point at the same target id. Convergent.
  • parseReferences: already convergent — see referencesChanged check at referencesProcessor.ts:219.

Seat-id mitigation: indexed-deterministic fallback

ensureAliasTarget's current behavior (targets.ts:128) creates seats at uuidv5(workspaceId:alias, ALIAS_NS). After a rename the id is stale — the seat now claims a different alias. New typings of the old alias silently resolve to the renamed block.

Fix: probe an indexed-deterministic namespace until we find an empty slot or one that claims our alias.

  1. Try id0 = uuidv5("ws:alias:0", ALIAS_NS).
  2. If empty → create here. Done.
  3. If live and claims this alias → reuse (normal convergence path).
  4. If live and claims something else, OR tombstone → probe next: id1, id2, … until step 2 or 3 hits.

Convergence holds: two offline clients in the same world-state probe the same way and land on the same slot. The "claims this alias?" check is what preserves the happy-path convergence at id0 — without it the algorithm would always probe past the live Foo-seat.

Side benefit: tombstone-restore-on-retype goes away too (probe past tombstones). A typed [[Foo]] always means a fresh Foo, not "restore the deleted Foo I had three months ago."

Open questions / known wrinkles

  • Empty-alias-array edge case in AR1. If the rename diff is {removed: ["Foo"], added: []} and content === "Foo", do we rewrite content to "" (matching the now-removed alias)? Probably no — blank-content blocks are weird. Treat aliases.after = [] as "no sync target" and leave content alone.
  • UI guarantee. One rename per commit is already how the alias editor behaves; the rename processor relies on that to keep R5/R6 rare. No new constraint needed.
