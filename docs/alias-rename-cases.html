Alias sync + rename — case-by-case

Two post-commit processors in two plugins, both watching {content, properties} on blocks:

  • Sync (alias plugin): same-block content↔alias reconciliation. Replace when old value matches the other side; heal additively when the block has drifted. Runs same-tx — amends the user's transaction so content and aliases commit atomically; throws ProcessorRejection on alias collision so the whole tx rolls back cleanly.
  • Rename / backlink rewrite (references plugin): cross-block. When aliases are removed or swapped, rewrite source backlinks. Runs follow-up (post-commit) — the cross-block writes can be many and shouldn't block the user's commit. Lives next to parseReferences because it needs the block_references projection to find sources.

Sync composes into rename: when a content edit produces an alias-array swap (committed atomically in the user's tx), the rename processor picks it up uniformly via the alias diff. No special path.

On the same-tx vs follow-up split: same-tx is reserved for cheap, correctness-critical, single-row work (sync, reference-array normalization). Follow-up keeps expensive or cross-row work (parseReferences, rename, cleanup) off the user-commit path. The earlier decision to keep parseReferences follow-up (data-layer redesign §7.1) stands — that was a typing-latency call about that specific processor, not a category-wide stance.

Sync action: diff: content went "Old name" → "New name". Old value was in aliases → replace that entry with the new content (not append). Result is a 1-for-1 alias swap.
Cascade: the alias-array swap re-fires the watcher; rename processor sees {removed: ["Old name"], added: ["New name"]} → rewrites [[Old name]] → [[New name]] in every backlink source. Sync and rename compose — content edits and alias edits flow through the same backlink-rewrite path.
Case A2 · Content edit, new value already an alias sync collapses an alias
Target — before
content:"X"
aliases:["X", "Y"]
→
Target — after (user edit)
content:"Y"
aliases:["X", "Y"]
⇣ sync
After sync
content:"Y"
aliases:["X", "Y"]
Sync action: old content "X" is in aliases — replace it with "Y", which dedupes against the existing "Y" entry. Net effect: X is dropped. The user said "this is now called Y"; keeping X around would contradict that.
Cascade: rename diff = {removed: ["X"], added: []} — pure remove with aliases still present. Hits Case R4 below: backlink sources rewrite [[X]] → [X](((target-id))) to preserve original display text.
Case A3 · Content drifted from aliases (no match) — healing sync, additive
Target — before
content:"already drifted"
aliases:["Original"]
→
Target — after (user edit)
content:"another edit"
aliases:["Original"]
⇣ sync
After sync
content:"another edit"
aliases:["Original", "another edit"]
Sync action: old content "already drifted" is not in aliases → no anchor to replace. Heal by adding the new content as a fresh alias. From this point on the block is in-sync (next content edit will hit A1/A2). No backlink rewrite — rename diff is {removed: [], added: ["another edit"]}, which is Case R3.

Sync action: content doesn't match the removed alias → no anchor. Leave content alone. (Block is drifted from the title perspective, but the alias rename is what the user expressed; respecting it.)

Decision ladder

Sync processor (alias plugin) — fires on {content, properties} change to the same block:

  1. Content changed, old value ∈ aliases (A1, A2) → replace that alias entry with new content. Dedupe.
  2. Content changed, old value ∉ aliases (A3 — drift) → add new content as a fresh alias. Block heals.
  3. Alias diff is 1-for-1, content === removed alias (AR1) → rewrite content to the new value.
  4. Otherwise → no sync write. The rename processor may still act on the alias diff.

Rename processor (references plugin) — fires on alias-property diff. For each α ∈ removed with live backlinks:

  1. 1-for-1 swap (R1, R2, A1-cascade, AR1-cascade): |removed| = |added| = 1 → rewrite [[α]] → [[new]].
  2. Anything else with backlinks (R4, R5, R6, R7, A2-cascade): rewrite [[α]] → [α](((target-id))) (aliased blockref). Preserves display text; doesn't depend on what's left in aliases.
  3. Pure add (R3): no removed aliases → nothing to rewrite.

Interaction with parseReferences (sanity check)

Idempotency on second pass

Sync and rename writes re-fire the field-watcher. Convergence depends on the second pass being a no-op:

  • Sync (A1/A2/A3/AR1): first pass writes the reconciled side. Second pass sees no diff in its trigger field → no work. Cross-processor: sync's alias write re-fires rename, which acts on the (now-final) alias diff.
  • Rename (R1/R2 etc): rewrites source content. Source edits trigger parseReferences; target wasn't touched again, so rename doesn't re-fire.
  • Rename → blockref form (R4/R5/R6/R7): source content now has [label](((id))). parseReferences re-fires on the source, parses the aliased blockref, refreshes the references column to point at the same target id. Convergent.
  • parseReferences: already convergent — see referencesChanged check at referencesProcessor.ts:219.

Seat-id mitigation: indexed-deterministic fallback

Problem (pre-fix): the legacy ensureAliasTarget created seats at uuidv5(workspaceId:alias, ALIAS_NS) — a single id per (alias, ws). After a rename the id is stale — the seat now claims a different alias — and new typings of the old alias silently resolve to the renamed block. Tombstone-restore- on-retype had the same flavour: deleting an alias seat and re- typing the wikilink resurrected the deleted target instead of creating a fresh one.

Implemented in targets.ts (resolveAliasSeatId + computeAliasSeatId(alias, ws, index)): probe an indexed-deterministic namespace until we find an empty slot or one that claims our alias.

  1. Try id0 = uuidv5("ws:alias:0", ALIAS_NS).
  2. If empty → create here. Done.
  3. If live and claims this alias → reuse (normal convergence path).
  4. If live and claims something else, OR tombstone → probe next: id1, id2, … until step 2 or 3 hits.

Convergence holds: two offline clients in the same world-state probe the same way and land on the same slot. The "claims this alias?" check is what preserves the happy-path convergence at id0 — without it the algorithm would always probe past the live Foo-seat.

Side benefit: tombstone-restore-on-retype goes away too (probe past tombstones). A typed [[Foo]] always means a fresh Foo, not "restore the deleted Foo I had three months ago."

Note on collision detection: the seat-id probe answers "what id should a fresh claimant get for this alias?" — the right question for ensureAliasTarget. It's the wrong question for "does any live block already claim this alias?" (which doesn't care about ids at all). Collision detection uses the trigger-maintained block_aliases index via tx.aliasLookup instead — see "Alias collisions" below.

Undo interactions

With sync as a same-tx processor, the user's content and alias writes commit atomically in one tx — one BlockDefault undo entry per user action. Rename remains follow-up (the cross-block writes go in ChangeScope.References, which is recorded for parity but not user-exposed; see undoManager.ts). So a single user action becomes one user-visible undo entry plus a follow-up rename tx that the field-watcher re-fires on cmd-Z. Whether the pipeline self-heals depends on whether the forward edit was reversible.

Self-healing cases: forward edit was a pure derivation (alias swap / content swap) that the same rules produce in reverse. Non-self-healing cases: forward edit was information-destructive on source content ([[α]] → [α](((id))) erases the original wikilink form; re-firing rules can't reconstruct it).

Self-healing on Cmd-Z no extra mechanism needed
  • A1 (content edit, old ∈ aliases). Forward: same-tx sync amends the user's tx so content "Old"→"New" and aliases [Old]→[New] commit atomically. Rename R1 fires follow-up and writes [[Old]]→[[New]] at sources. Cmd-Z reverts the single user tx (content and aliases revert together); rename re-fires on the diff and rewrites [[New]]→[[Old]]. ✓
  • AR1 (alias rename, content matches). Forward: same-tx sync sees the user's aliases [Foo]→[Bar] write, amends the tx to also rewrite content "Foo"→"Bar". One atomic commit. Rename R1 fires follow-up and writes [[Foo]]→[[Bar]]. Cmd-Z reverts the atomic content+alias swap; rename re-fires and writes [[Bar]]→[[Foo]]. ✓
  • R1 / R2 (clean alias rename). Forward: [Old]→[New] → rename rewrites [[Old]]→[[New]]. Undo: [New]→[Old] → rename rewrites [[New]]→[[Old]]. ✓
  • R3 (pure alias add). Forward: no source rewrite (no removed alias). Undo: pure remove of the added alias → R4 with no backlinks (no [[α]] exists) → no source writes. ✓

The same-tx move closes the previous "two-tx undo, two undo buckets" gap for A1, AR1, and A3 — those used to commit the user's edit and sync's follow-up alias write as separate txs. Now the alias write rides on the user tx and reverts atomically.

Non-self-healing on Cmd-Z (the gap) user-visible regression
  • R4 / R5 / R6 / R7 (alias remove with backlinks). Forward: alias B removed → rename writes [[B]] → [B](((target-id))) in every source. Undo of target's BlockDefault tx: aliases get B back → rename sees {removed: [], added: [B]} → R3 (pure add) → no-op. Sources are stuck in blockref form; the original [[B]] text is gone. Re-firing the pipeline can't reconstruct it.
  • A2-cascade (content edit collapses alias → R4). Forward: same-tx sync amends the user's tx so content "X"→"Y" and aliases [X, Y]→[Y] commit atomically (Case A2). Rename fires follow-up, hits R4 for removed X, rewrites source [[X]] → [X](((id))). Undo content+alias atomically (one tx): "Y"→"X", aliases [Y]→[X, Y]. The source stays in blockref form because rename's destructive R4 rewrite is in References scope and isn't reverted by the user undo. Same-tx fixed the alias side; the source-rewrite side is what R4-R7 Options A/D address.
  • A3-residue (drift heal). Note: with same-tx sync, A3 healing happens in the user's own tx, so cmd-Z reverts the heal atomically alongside the content. This residue case no longer exists under same-tx placement — kept here for historical reference until the framework lands.
Common cause: rename's [α](((id))) rewrite is information- destructive on source content. The forward-direction signal (alias remove with backlinks) has no inverse signal on undo (alias add doesn't tell the processor to undo a rewrite — the processor can't even see that the source's current blockref form came from this pipeline vs. the user typing it directly).

Thoughts on R4-R7

Four options, ordered by invasiveness. None is free; the choice depends on how aggressively we want to preserve display text vs. preserve undo.

Option A — Don't rewrite source; surface as broken link simplest; undo-clean

Rename only handles the 1-for-1 swap case (R1, R2). For R4-R7, leave [[α]] in source content. The wikilink no longer resolves; render it with a "broken link" affordance. Restoring the alias on the target (via Cmd-Z or manual re-add) naturally re-resolves the link.

  • Pros: trivially undo-correct. No engine work. Matches Roam's mental model (renaming a page doesn't rewrite backlink text). Restores cleanly on re-add.
  • Cons: link "breaks" when the user removes the last alias (R7) or removes a referenced alias from a multi- alias target (R4). Display text is preserved (it's still [[B]]) but the link is dangling until the user acts.
  • Implementation: drop the R4/R5/R6/R7 branch from the rename decision ladder. parseReferences continues to project the wikilink into block_references; the lookup just misses, and the renderer shows a broken-link state (already a UI affordance for any [[α]] with no matching alias seat).
Option B — Engine-level cause-chain undo grouping cleanest semantics; biggest lift

Every processor tx records the originating user txId on its snapshot (causedByTxId). The undo manager groups consecutive undo entries that share a cause: Cmd-Z pops them all atomically and replays each via applyRaw. One user action ↔ one undo step, regardless of how many processor txs fired downstream.

  • Pros: generalises beyond aliases — any post-commit processor with destructive side effects (today none; tomorrow probably more) gets undo for free. Keeps the current R4 source rewrite (display-text preservation) AND makes it undoable.
  • Cons: engine work — undoManager needs cause tracking, the tx context needs causedByTxId propagation across the scheduler boundary, the replay path needs to apply a group atomically (and probably suppress processors from firing on the replay to avoid re-cascading). Subtle interactions with the References-bucket-not-user-visible rule today.
  • Effort: medium-large — touches undo manager, tx context, processor runner.
Option C — Inverse-rewrite log on the target complex; storage cost

Rename stamps a previousAliases side property (or side table) on the target when it removes an alias. On the "alias added back" signal (pure add of α that's in previousAliases), rename treats that as an undo and walks sources rewriting [α](((id))) back to [[α]].

  • Pros: R4 rewrite stays as-is. No engine work. Reverses on alias re-add regardless of how the re-add happens (Cmd-Z, manual, sync from peer).
  • Cons: sources may have multiple [α](((id))) spans that didn't all come from this pipeline — disambiguating is fuzzy. previousAliases needs TTL / cleanup. Cross-source consistency on partial sync states is hard. Doesn't help A3-residue or A2-cascade (sync's own collapse isn't reversed).
Option D — Hybrid: drop R4-R7, keep R1-R2; revisit later recommended for v1

Ship Option A now: rename only handles the 1-for-1 swap case (R1, R2). For R4-R7, leave the wikilink dangling and rely on the renderer's broken-link UI. Revisit Option B when (a) other processors with destructive side effects appear, or (b) the dangling-link UX proves to be a real pain point. R4 forward behavior in production is rare anyway (the UI commits one alias rename per edit, so R5/R6 are safety nets; R4 / R7 happen only on explicit alias removal).

What this means for the existing implementation: delete the replacementFor "anything else" branch in renameProcessor.ts so non-1-for-1 diffs return early. Update the corresponding tests (R4, R7 cases assert "source untouched" rather than rewrite-to-blockref). A2-cascade test asserts that sync still collapses the alias but the source content stays [[X]].

Alias collisions

A user can request an alias that is already claimed by a different live block — by typing [[Existing]] at the top of their own block (A1, A2) or by adding "Existing" to their block's aliases property directly (AR1). Without intervention the sync processor would either steal the name (if it wrote unconditionally) or silently no-op (leaving content and aliases drifted — the "stuck state"). Neither is acceptable.

V1 policy is refuse the rename atomically: the user's transaction rolls back before commit, leaving content and aliases in their pre-edit state. A toast surfaces the reason. The eventual goal is Roam-style "suggest merge" (offer to merge the two blocks into one), but the merge flow has its own design surface — disallowing for now keeps the door open without committing to the merge UX.

Placement: same-tx alias sync enables atomic rejection

Collision detection isn't a standalone validator — it folds into the alias sync processor, which moves from post-commit to same-tx placement for this work. The same-tx facility is the v4.19 processor mode that v4.20 of the data- layer redesign dropped for being uncalled; it's being reintroduced specifically for cheap, correctness-critical, single-row work like sync (and reference-array normalization, which is currently hardcoded inline in txEngine.ts). The earlier decision to keep parseReferences as follow-up (data-layer redesign §7.1) stands — that decision was about typing-path latency for an expensive processor, not about same-tx as a category.

Concretely: sync registers via sameTxProcessorsFacet (sibling to postCommitProcessorsFacet); its apply receives the user's live Tx; it reads staged state, amends the tx with alias writes when planning succeeds, or throws ProcessorRejection when the collision check fails. The user's whole tx rolls back atomically inside SQLite's writeTransaction.

Detection unified across A1 / A2 / AR1

Sync's plan phase computes the set of aliases the user is claiming for this block, then asks tx.aliasLookup(alias, workspaceId, excludeId = plan.id) — a tx-aware primitive against the trigger-maintained block_aliases index (the same index parseReferences, quick-find, backlinks filters, and core.aliasLookup read). If the lookup returns any non-null block, that's a collision (the excludeId parameter has already filtered out self). Self-reclaim falls out naturally: with self excluded, the only OTHER claimant is what we care about.

Why excludeId is mandatory here: by the time the same-tx processor runs, the user's own setProperty (in AR1 / direct-claim paths) has already fired the block_aliases trigger inside the writeTransaction. So the attempting row appears in the index for its new alias. SELECT_BLOCK_BY_ALIAS_IN_WORKSPACE_SQL returns the OLDEST claimant by created_at with LIMIT 1; without exclusion, an older attempter masks a younger conflicting claimant (the SQL returns the attempter, claimant.id === plan.id wins the self-reclaim check, and the rejection never fires). Excluding self by id resolves the ambiguity at the SQL layer rather than in JS — important because the LIMIT 1 means we can't see past the first row.

  • A1 (content adds [[New]]): newAliases = aliasesFromContent(after) \ before.aliases.
  • A2 (content rewrites [[Old]] → [[New]]): same as A1 — the new name is in the added set.
  • AR1 (user writes aliases directly): newAliases = after.aliases \ before.aliases.
  • Insert (before === null): every non-blank alias on the brand-new row is a fresh claim. The planner emits a collision-only plan for these so tx.create + setProperty can't bypass the rejection policy by starting with the taken name already in place.
  • Restore (before.deleted && !after.deleted): a soft-delete clears the row from block_aliases, freeing its alias for reuse; restoring the row (either via tx.restore or repo.undo of a delete) re-introduces it as a claimant. Treated the same as an insert — every non-blank alias is a fresh claim. The processor's watches list includes deleted so a plain tx.restore(id) with no content/property patch still fires the check.
  • A3 (alias removal): no new claim, no check.

Detection runs inside the user's tx, so tx.aliasLookup sees staged writes from this tx (the block_aliases triggers fire synchronously inside the writeTransaction). Cascades within a single tx — e.g. a mutator that adds two colliding aliases — are caught coherently.

Why not the seat-id probe? An earlier implementation walked computeAliasSeatId(alias, ws, 0..N). That assumed every alias claimant's id is in the seat-id space for the alias it currently claims — which fails in three real cases: (a) a block created via tx.create with an arbitrary id then later given an alias, (b) a block originally seat-id-keyed for alias α then renamed to β (its id is still uuidv5("ws:α:N"), not the seat for β), and (c) a block claiming an alias whose canonical seat is held by a different live block. The block_aliases index covers all three. The seat-id mechanism is kept for ensureAliasTarget (which DOES want a deterministic id for a new claimant — a different question).

Response: throw ProcessorRejection atomic, no stuck state

On collision, the same-tx sync processor throws ProcessorRejection('alias.collision', {alias, conflictingBlockId, conflictingBlockTitle, workspaceId, attemptedOn}). The error bubbles out of the user's writeTransaction, SQLite rolls back atomically, no rows were committed, no snapshots entry was recorded, no undo entry was created. The user is in the exact pre-edit state.

The error surfaces at the repo.tx call site (the editor save / chip edit / programmatic mutator). A single repo.onUserError subscriber wired at Repo bootstrap (context/repo.tsx) routes ProcessorRejections to the sonner-based toast layer. The alias.collision handler renders "Alias 'X' is already used by 'Y'. Your edit was reverted — try a different name." with an "Open" action button that calls navigate(repo, …). conflictingBlockTitle in the rejection meta is snapshotted at detection time (one extra tx.aliasLookup read returns the full BlockData) so the toast doesn't need a follow-up SQL query.

What this replaces: an earlier post-commit-revert design used repo.undo(BlockDefault) after the user's tx had already committed, with a behavioral guard (single-row snapshot + snap.after === current) and a "refuse and leave content" fallback when the guard missed. Same-tx rejection makes all of that unnecessary — there is no committed state to revert, no fallback to manage, no multi-row-snapshot edge case.

Why same-tx over editor-layer validation considered alternatives
  • Editor-layer guard (refuse in CodeMirror): clean UX but only catches collisions originating from the user-typing path. Misses programmatic mutators, the chip editor, import flows, and any future surface that writes blocks. The data layer is the right enforcement boundary.
  • Post-commit revert via repo.undo: doable without engine changes, but creates the stuck-state fallback case (multi-row tx where revert isn't safe) and depends on a behavioral guard that's easy to get subtly wrong. Same-tx is simpler end-to-end.
  • argsSchema validation at the mutator boundary: doesn't see the cross-block state needed to detect collision (a block_aliases SELECT in the active workspace is required). Schema-level checks are for shape, not semantics.
Future: suggest merge deferred

Roam's mental model: a collision means the user probably wants to merge the two blocks (move children, redirect references, delete the duplicate). The toast's action button is the obvious surface ("Merge with 'Y'?"). The merge implementation has its own design — moving children, transferring aliases, handling properties — and isn't blocked by collision detection. Ship detection + refuse first; merge layers on top by replacing the toast action with a confirmation flow that runs a merge mutator.

Open questions / known wrinkles

  • Empty-alias-array edge case in AR1. If the rename diff is {removed: ["Foo"], added: []} and content === "Foo", do we rewrite content to "" (matching the now-removed alias)? Probably no — blank-content blocks are weird. Treat aliases.after = [] as "no sync target" and leave content alone.
  • UI guarantee. One rename per commit is already how the alias editor behaves; the rename processor relies on that to keep R5/R6 rare. No new constraint needed.
