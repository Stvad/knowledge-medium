# Adversarial review of the current session

Fan out several review agents **in parallel** over the work done in **this session**, then synthesize their findings. Most agents are *adversarial* — each told to break the work from a different angle; their job is to find real problems, not to praise. Alongside them a standing **quality** agent always runs, hunting the constructive axis the adversarial pass walks right past: reuse you missed, needless complexity, dead weight, an obvious efficiency win. The two axes are reported separately (see **Synthesize**) so a "could be simpler" never competes with a "this loses data".

## Argument

`$ARGUMENTS` = number of **adversarial** agents **N** (default **2**), one perspective per agent — the quality agent runs in addition to these and isn't counted in N. Trailing words are focus hints (e.g. `4 watch the sync path`). By default the skill implements the agreed improvements after presenting findings; add `--no-fix` to stop at the report. Add `--no-cleanup` to skip the quality read and run a pure adversarial bug-hunt (e.g. a tense pre-deploy pass where you only want defects). Add `loop` (e.g. `/areview loop` or `/areview loop 4`) to repeat the review→fix cycle until it converges.

Add a **method lens** — a reserved keyword (like `loop`/`--no-fix`/`--no-cleanup`, parsed out before the rest becomes focus hints) that makes *every* agent review with a specific method, orthogonal to its per-agent perspective: `codex` — e.g. `/areview codex 4`, `/areview loop codex`. See **Method lenses** below.

## Method lenses

Recognize a known method-lens keyword in the arguments. The N perspectives are *risk-domain* lenses — *what* to scrutinize (data-loss, races, failure-paths, ungrounded claims), one per agent, non-overlapping. A **method lens** is orthogonal — *how* to scrutinize — and applies to ALL agents at once (the quality agent included). Compose them: `/areview codex 4` = 4 adversarial agents on 4 distinct surfaces plus the quality agent, each reviewing with the codex method.

- **`codex`** (mechanical / empirical / ground-truth). Instruct every agent: verify per-statement and trace each branch with concrete values; ground **every** claim against the actual artifact — read the dependency source in `node_modules`, run the code, execute the SQL — never assert from memory; follow cross-references literally (does the cited section/symbol exist and say what the citing text claims?); for any tests, mutation-test them (would this test fail if the bug were reintroduced?); hunt edge values (empty / 0-byte / boundary / preexisting-state / unordered). Prefer "I ran it and observed X" over "this should…". This catches the implementation/contract bugs (pagination order, error-shape, idempotency, off-by-one, redaction) that thematic "is the abstraction sound?" review glosses over. For the quality agent, codex means proving the cleanup is real — the duplicated helper actually exists and matches, the "simpler" rewrite is genuinely behavior-preserving — not asserting it from a glance.

## The quality read (always on)

The adversarial agents answer "how does this break?"; the quality agent answers "what's heavier than it needs to be?" — logic duplicated from an existing helper, complexity that buys nothing, an abstraction pitched at the wrong altitude, an obvious efficiency win. Give it the **same self-contained brief** as the others — crucially including *what we were trying to do* — because the strongest cleanup findings are intent-aware ("you reinvented `aliasLookup`"), which a diff-only reviewer can't see. This is exactly why the quality read lives here and not only in `/code-review`: areview already holds the session's intent.

Keep it on the constructive axis only. It does **not** re-report bugs — that's the adversarial agents' job, and double-reporting muddies severity — and it flags cleanups it's confident about rather than padding the list with taste calls. For a large surface, run more than one quality agent in parallel, split by area.

## What matters

- **Pack a self-contained brief** — the agents have zero memory of this conversation. Scope comes from *this conversation* — what we actually worked on; git is just the artifact to hand over. Gather the matching diff (uncommitted `git diff` plus the commits you made this session — use `git log` to find them; if the base is unclear, the changes under discussion are the source of truth, not git), or the drafted artifact if it isn't in git yet. Include what we were trying to do and why, plus the invariants in play, and note anything you left out rather than truncating silently. If nothing was actually changed, say so and stop.
- **Pick N non-overlapping adversarial perspectives that fit what changed** — match the lens to where this work is most likely *wrong* (e.g. data loss for a migration, races for sync code, unhandled failure paths / ungrounded claims for a design doc), and include one aimed at its highest-stakes risk. Don't pad to hit N; if there are only K real angles, run K and say so. The quality agent is separate and always runs (unless `--no-cleanup`).
- **Use the Agent tool, parallel, read-only** — all calls in one message (the N adversarial agents plus the quality agent); agents review only, never edit. Tell the adversarial ones: cite file:line, give the concrete failure scenario + severity + fix, confirm against the code before claiming a bug, and say plainly when a lens turns up nothing rather than inventing nits. The quality agent gets the constructive brief from **The quality read** above and reports cleanups, not defects.
- **Synthesize into two separate sections** — first **Defects** (the adversarial findings): dedup (cross-agent agreement is a strong signal), drop false positives that don't hold against the code, rank by severity (blocker > major > minor > nit), surface real disagreements. Then, *below it and clearly partitioned*, **Cleanups** (the quality findings): reuse / simplification / efficiency / altitude, ordered by payoff, each with the concrete win. Never interleave the two or let a cleanup borrow defect-severity language — the whole point of keeping them apart is that the blocker signal stays loud. End with a bottom line that leads with the defect verdict.
- **Fix by default** — after presenting the report, implement the **defects** you stand behind, worst first: blocker, then major, then minor; nits are optional. Skip the ones you overruled as false positives, and note anything you deliberately left for the user. Apply **cleanups** as a *separate batch* (separate commit, or at least called out as their own group) and only the behavior-preserving ones you're confident in — leave anything that changes an interface, is large, or is a matter of taste for the user to greenlight rather than silently rewriting their abstractions. Then verify (`yarn run check`). With `--no-fix`, stop at the report instead and let the user act on it.
- **`loop` mode** — re-review the fixed state and fix again, repeating until a round surfaces no **defect** above a nit (no blocker/major/minor left). **Convergence is judged on defects only** — cleanups never keep the loop alive, since there's always something marginally simpler and you'd never terminate. Each round reviews the *current* state, not the original diff. Stop when converged, and also stop (reporting why) if you hit ~4 rounds, if findings stop making progress (the same issues recur or a fix spawns new ones), or if `yarn run check` fails and you can't resolve it — don't loop forever. `loop` always fixes; it ignores `--no-fix`.
